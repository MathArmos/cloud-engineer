## Conceitos

### 1. Hierarquia de Recursos no Google Cloud

**O quê.** Toda organização que usa GCP tem uma estrutura em árvore com 4 níveis:

```
Organization (raiz)
    └── Folder (opcional, pode aninhar várias camadas)
        └── Project (OBRIGATÓRIO — tudo vive dentro de um projeto)
            └── Resource (VMs, buckets, BDs, etc.)
```

**Por que importa.**
- **Organização lógica:** separar divisões/produtos/ambientes sem misturar recursos
- **Herança de políticas IAM:** uma policy concedida na Org desce para todos os Folders/Projects/Resources abaixo (princípio aditivo)
- **Billing isolado:** cada projeto pode estar em uma billing account diferente, facilitando atribuição de custo
- **Blast radius:** isolar `prod` de `dev` evita que um comando errado afete produção

**Exemplo Cymbal Superstore** (do vídeo):

```
Cymbal Superstore (Organization)
├── Folder: B2B Supply Chain        (divisão Operations)
│   └── Folder: Supply Chain App
│       ├── Project: supply-chain-dev
│       ├── Project: supply-chain-staging
│       └── Project: supply-chain-prod
├── Folder: Logistics               (divisão Logistics)
│   └── Folder: Transportation App
│       ├── Project: transport-dev
│       ├── Project: transport-staging
│       └── Project: transport-prod
└── Folder: Sales & Marketing       (divisão Sales & Marketing)
    └── Folder: Ecommerce App
        ├── Project: ecommerce-dev
        ├── Project: ecommerce-staging
        └── Project: ecommerce-prod
```

Cada app aterrissa em um compute diferente — alinhado ao caso de uso:
- **Supply Chain → Compute Engine (VMs):** provavelmente tem dependências de SO/middleware específicas, precisa controle total
- **Ecommerce → GKE:** containerizado, microsserviços, escala horizontal
- **Transportation → Cloud Run functions:** event-driven (rastrear localização de caminhões em tempo real, scale to zero)

**Quando usar Folders.**
- Empresa com >1 divisão/produto/ambiente que precisam de governança separada
- Necessidade de aplicar políticas IAM diferentes por divisão
- Para Cymbal faz sentido. Para conta pessoal de estudo, dispensável (vai direto Organization → Projects, ou só Project sem Org)

**Armadilhas de prova.**
- **Folder é opcional**, Project é obrigatório
- Hierarquia correta: **Organization → Folder → Project → Resource** (DQ 02). Cuidado com ordem invertida nas alternativas
- Políticas IAM são **aditivas e herdadas para baixo**. Você não consegue REMOVER uma permissão concedida em nível superior — só pode CONCEDER mais no nível inferior

### 2. Projetos: container fundamental + atributos

**O quê.** Um projeto é o container fundamental para todos os recursos GCP. Tudo precisa pertencer a um projeto — VMs, buckets, BDs, APIs habilitadas, billing.

**Três atributos identificadores:**

| Atributo | Mutável? | Único? | Exemplo |
|----------|----------|--------|---------|
| **Project ID** | ❌ NÃO | Globalmente único | `cymbal-ecommerce-prod-7821` |
| **Project Name** | ✅ SIM | Não precisa ser único | "Cymbal Ecommerce Production" |
| **Project Number** | ❌ NÃO | Globalmente único | `123456789012` (gerado pelo Google) |

**Por que importa.**
- IDs e Numbers viram parte de URLs, configs e referências em scripts — não dá pra mudar depois. **Escolha bem o ID na criação**
- Names podem ser corrigidos conforme produto é renomeado

**Armadilha de prova (DQ 03).**
- Pergunta clássica: "qual atributo PODE ser mudado?" → **Project Name** (única resposta correta)
- "Project Category" **não existe** — distrator puro

**Ambientes (dev/staging/prod).**
- Cymbal usa um projeto por ambiente — best practice de CI/CD
- Isolamento total: bugs em dev não vazam pra prod, IAM diferente, billing separado, quotas independentes
- Pipeline de deploy promove código/configs entre projetos

### 3. Habilitar APIs por projeto

**O quê.** Toda funcionalidade do GCP (GKE, Cloud SQL, Cloud Storage, BigQuery...) é exposta como uma API. Por padrão, **APIs ficam desabilitadas** em um projeto novo — você precisa habilitar explicitamente.

**Exemplo Cymbal Ecommerce** (do vídeo): precisa de
- **Kubernetes Engine API** (`container.googleapis.com`) — para criar clusters GKE
- **Cloud SQL Admin API** (`sqladmin.googleapis.com`) — para o backend DB

**Como habilitar:**
```bash
gcloud services enable container.googleapis.com
gcloud services enable sqladmin.googleapis.com

# Listar APIs habilitadas no projeto atual
gcloud services list --enabled
```

Ou via Console: APIs & Services → Library → Enable.

**Armadilhas de prova.**
- Erros tipo `API [X] not enabled` em comandos `gcloud` são comuns. A solução é habilitar a API antes de criar recursos
- **Habilitar API ≠ ter permissão IAM.** São duas barreiras independentes. Habilitar API só "liga" o serviço no projeto; você ainda precisa de role para usar

### 4. IAM — Identity and Access Management

**Conceito base.** "Quem (identidade) pode fazer o quê (role/permissão) em qual recurso (escopo)?"

**Identidades possíveis:**
- **Google Account** — humano com email Google individual
- **Google Workspace account** — humano em domínio gerenciado por Workspace (ex.: `usuario@cymbal.com`)
- **Cloud Identity** — Workspace sem o Workspace (só identidade gerenciada centralmente, sem Gmail/Drive)
- **Google Group** — coleção de identidades (recomendado para escalar!)
- **Service Account** — identidade para aplicação/recurso (NÃO humano) — ver Seção 5.2
- **`allAuthenticatedUsers`** — qualquer usuário Google (muito amplo, cuidado)
- **`allUsers`** — qualquer usuário da internet, mesmo anônimo (use só para conteúdo público)

**Tipos de roles:**

| Tipo | Granularidade | Quando usar |
|------|---------------|-------------|
| **Basic (primitivas)** — `roles/owner`, `roles/editor`, `roles/viewer` | MUITO ampla | Só em testes/sandbox. **Evite em produção** |
| **Predefinidas** — `roles/bigquery.dataViewer`, `roles/compute.viewer`, etc. | Granular, mantida pelo Google | **Padrão.** Cobre 95% dos casos |
| **Custom** | Você define exatamente quais permissões | Quando role predefinida é overkill ou underpowered |

**Princípio do least privilege.** Dar o **mínimo** de permissão necessária. Nunca usar `Owner` para alguém que só precisa ler. Nunca usar `Editor` quando uma predefinida específica resolve.

**Por que usar Groups em vez de usuários individuais (DQ 01).**
- Quando alguém entra/sai do time, você só ajusta o grupo no Cloud Identity/Workspace — não precisa mexer em IAM dos projetos
- Permissões ficam no grupo, não na pessoa — auditável, escalável
- Caso Cymbal: criar `data-analysts@cymbal.com` em vez de adicionar João, Maria, Pedro individualmente em cada projeto

**Exemplo do vídeo — data analysts no BigQuery:**

Os analistas precisam:
- **Ler** os datasets/tabelas → `roles/bigquery.dataViewer`
- **Rodar queries** (que são jobs) → permissão `bigquery.jobs.create`, que está dentro do `roles/bigquery.user`

Setup correto:
```bash
# Adicionar o grupo ao projeto com as roles necessárias
gcloud projects add-iam-policy-binding ecommerce-prod \
  --member="group:data-analysts@cymbal.com" \
  --role="roles/bigquery.dataViewer"

gcloud projects add-iam-policy-binding ecommerce-prod \
  --member="group:data-analysts@cymbal.com" \
  --role="roles/bigquery.user"
```

Se faltar uma delas:
- Sem `bigquery.dataViewer` → vê metadata dos datasets/tables mas não consegue ler os dados
- Sem `bigquery.user` (que dá `bigquery.jobs.create`) → não consegue submeter query nenhuma

**Armadilhas de prova consolidadas.**
- "Atribuir role direto à pessoa Stella" → ❌ ERRADO. Use Groups (DQ 01)
- "Atribuir Owner/Editor pra economizar tempo" → ❌ ERRADO. Viola least privilege (DQ 05)
- "Atribuir permissão individual em cada projeto" → ❌ ERRADO. Atribua no nível mais alto que cobre o escopo necessário (folder se forem todos os projetos da divisão; org se for cross-divisão) (DQ 04)
- **DQ 06 — Compute Engine Viewer (`roles/compute.viewer`):** dá permissões READ-ONLY tipo `compute.images.list`, `compute.images.get`, `compute.instances.list`, `compute.instances.get`. NÃO dá `create`, `setIamPolicy`, `update` (essas são de admin/editor)

### 5. Cloud Identity vs Google Workspace

**Cloud Identity.** Serviço (com tier gratuito) para gerenciar identidades de usuários e grupos quando você **não tem Google Workspace** (Gmail corporativo, Drive, etc.).

**Quando importa.**
- Cymbal pode usar Cloud Identity para o domínio `cymbal.com` sem necessariamente ter Gmail corporativo
- Permite criar Google Groups que aparecem como identidades válidas no IAM
- Centraliza onboarding/offboarding: deletar a identidade no Cloud Identity revoga TODOS os acessos GCP automaticamente

**Armadilha de prova.**
- "Onde gerencio usuários no GCP?" — depende: **Workspace** se a empresa usa, **Cloud Identity** se não. **Nunca gerencie usuários "dentro" do projeto GCP** — sempre na camada de identidade
- Service Accounts são gerenciadas DENTRO do projeto. Humanos são gerenciados FORA (Workspace/Cloud Identity)

### 7. Basic roles em detalhe (Viewer / Editor / Owner)

**As 3 basic roles do GCP**, em ordem crescente de privilégio:

| Basic role | Permissões | O que pode fazer | Caso de uso |
|------------|-----------|------------------|-------------|
| **Viewer** (`roles/viewer`) | Read-only de quase tudo no projeto | listar, ler recursos | Auditoria, debug sem modificar |
| **Editor** (`roles/editor`) | Tudo do Viewer **+ modificar estado** | criar, atualizar, deletar recursos | Operações dia-a-dia |
| **Owner** (`roles/owner`) | Tudo do Editor **+ gerenciar IAM + billing** | adicionar usuários, mudar billing, transferir propriedade | Dono do projeto |

Hierarquia: **Viewer < Editor < Owner**

Existe também `roles/browser` (visualização da hierarquia da organização), raramente cai em prova.

**Armadilhas críticas:**

1. **"Administrator" NÃO é uma basic role no GCP.** Quando aparecer "Administrator" como suposta basic role em alternativa de prova — distrator de cara. O nome "Admin" só aparece em **predefinidas específicas de serviço** (ex.: `roles/storage.objectAdmin`, `roles/compute.networkAdmin`)

2. **A basic role MAIS BAIXA que altera estado é Editor.** Viewer só lê. Owner também altera, mas é a mais alta — não a mais baixa. Editor é a fronteira read↔write

3. **Em produção: evite basic roles.** Use predefinidas/custom. Basic é muito amplo (Editor já dá poder pra quebrar quase tudo no projeto)

**Mnemônico:** **Viewer (Vê) → Editor (Edita) → Owner (Onipotente)**

### 8. Interfaces para interagir com o Google Cloud

O GCP oferece 5 interfaces principais. Cada uma com seu caso de uso:

| Interface | O que é | Onde roda | Caso de uso típico |
|-----------|---------|-----------|---------------------|
| **Google Cloud Console** | Web UI | console.cloud.google.com | Exploração visual, dashboards, configs pontuais |
| **Cloud Shell** | Shell Linux no browser, com `gcloud`/`gsutil`/`bq`/`kubectl` **pré-instalados e autenticados** | Browser (Google fornece a VM efêmera) | Scripting rápido sem instalar nada localmente |
| **Cloud SDK / gcloud CLI** | Conjunto de executáveis de linha de comando (`gcloud`, `gsutil`, `bq`, `kubectl`) instalados localmente | Sua máquina (Linux/Mac/Windows) | Automação local, CI/CD, scripts repetíveis |
| **Client Libraries / REST API** | Bibliotecas em Python/Go/Java/Node/etc. ou chamadas HTTP diretas | Sua aplicação | Apps que falam com GCP programaticamente |
| **Cloud Mobile App** | App nativo Android/iOS | Celular | Monitoramento on-the-go, alertas, ações rápidas |

**Pegadinhas de prova:**

- **Cloud Shell ≠ gcloud CLI.** Cloud Shell é o AMBIENTE; gcloud CLI é o conjunto de EXECUTÁVEIS. O Cloud Shell tem o gcloud CLI pré-instalado, autenticado e atualizado
- **"Set of command line executables"** aponta para gcloud CLI (Cloud SDK) OU Cloud Shell. Se gcloud CLI não estiver entre as opções, Cloud Shell é a resposta
- **REST API** é programática (chamadas HTTP) — **não é** "command line executable"
- **Cloud Console** é Web UI — sem scripting nativo

**Comandos úteis no Cloud Shell / gcloud CLI:**

```bash
gcloud auth list                    # contas autenticadas
gcloud config list                  # config atual (projeto, região)
gcloud config set project PROJECT_ID
gcloud projects list                # listar projetos
gsutil ls                           # listar buckets Cloud Storage
gsutil cp file.txt gs://bucket/     # copiar arquivo para bucket
bq query "SELECT 1"                 # executar SQL no BigQuery
kubectl get nodes                   # comando do Kubernetes
gcloud services list --enabled      # APIs habilitadas no projeto
gcloud services enable container.googleapis.com   # habilitar API
```

**Vantagem do Cloud Shell sobre instalar gcloud CLI local:**
- Zero setup (basta ter conta Google)
- Sempre atualizado (Google mantém)
- 5GB de persistent home directory grátis
- Editor de código integrado (Cloud Shell Editor, baseado em VS Code)

**Vantagem do gcloud CLI local sobre Cloud Shell:**
- Mais performance (sem latência de browser)
- Funciona offline pra ler config
- Integra com IDE/CI/CD da sua máquina

### 9. IAM Allow Policy vs IAM Deny Policy

**O quê.** Existem **dois tipos de policy IAM** que se aplicam a recursos. Até agora vimos só a allow policy implicitamente; agora o detalhe completo:

| Tipo | O que faz | Como combina com hierarquia |
|------|-----------|-----------------------------|
| **Allow policy** | Concede roles (e suas permissões) a um principal | **União aditiva** com policies herdadas dos pais. Se org dá `storage.admin` ao grupo, project herda. |
| **Deny policy** | Bloqueia permissões específicas a um principal | **Prevalece sobre allow** — bloqueia mesmo se principal tem a role concedida. |

**Por que importa.** Deny policy permite **exceções cirúrgicas** sem desmontar a estrutura de allow. Caso típico:
- Alice é `roles/owner` do projeto (precisa, mesmo)
- Mas você **não quer** que ela delete buckets de Cloud Storage
- Solução **errada**: tirar Owner e construir role custom sem `storage.buckets.delete` → muito trabalho
- Solução **correta**: manter Owner via allow, criar **deny policy** bloqueando `storage.buckets.delete` para Alice

**Armadilha de prova.**
- Pergunta "como impedir um Owner de fazer ação específica X sem revogar o Owner role?" → **deny policy**
- Allow é **aditivo e herdado**. Deny tem **prioridade absoluta** quando ambos se aplicam ao mesmo principal/permissão.
- Não confunda deny policy com "remover role" — são mecanismos diferentes (deny não tira a role, só nega a ação).

### 10. Billing acumula de baixo pra cima (fluxo bottom-up)

**O quê.** A hierarquia tem **dois fluxos opostos** acontecendo simultaneamente:

```
                    Organization
                         │
                    ┌────┴────┐
                    │ Folder  │
                    └────┬────┘
                         │
                    ┌────┴────┐
                    │ Project │
                    └────┬────┘
                         │
                    ┌────┴────┐
                    │Resource │
                    └─────────┘

  Policies IAM  ↓ (herdam de cima pra baixo — aditivas)
  Billing       ↑ (acumula de baixo pra cima)
```

**Como o consumo é medido.**
- **Quantidades** (GB armazenados no bucket, número de requests)
- **Rate of use** (CPU-hours, requests por segundo, vCPU-segundos do Cloud Run)
- **Tempo** (horas de VM ligada, segundos de execução de função)
- **Número de itens** (queries no BigQuery, mensagens publicadas em Pub/Sub)
- **Feature use** (uso de feature paga adicional, como backups do Spanner)

**Regras críticas do fluxo bottom-up.**
- 1 recurso → 1 projeto (sem dividir custo de recurso entre projetos)
- 1 projeto → 1 billing account (em um dado momento — pode trocar, mas é 1 por vez)
- 1 organization → contém TODAS as billing accounts dela

**Por que importa.** Pra fazer **chargeback** entre divisões/times, você desenha a hierarquia ANTES. Cymbal separa `ecommerce-dev` / `ecommerce-staging` / `ecommerce-prod` justamente pra cada ambiente ter seu próprio rollup financeiro. Sem essa separação na arquitetura, time de finanças depende de labels + queries no BigQuery export pra dividir custo — possível, mas mais frágil.

**Armadilha de prova.**
- "Posso ter um recurso em dois projetos?" → ❌ NÃO. Recurso pertence a exatamente 1 projeto. (Shared VPC compartilha *acesso*, não muda *posse*.)
- "Posso ter um projeto em duas billing accounts simultaneamente?" → ❌ NÃO. Um projeto está ligado a no máximo 1 billing account.

### 11. Classificação física dos recursos: Global, Regional, Zonal

**O quê.** Independente da hierarquia lógica (org/folder/project/resource), cada recurso GCP tem um **escopo físico** — onde ele vive e como você o referencia em comandos. São 3 categorias:

| Escopo | Significado | Exemplos | Implicação prática |
|--------|-------------|----------|---------------------|
| **Global** | Sem afinidade geográfica — visível de qualquer região | **Images**, **snapshots**, **VPC networks**, custom roles, IAM policies | Cria 1 vez, usa de qualquer lugar. Snapshot tirado em `us-east1` pode restaurar em `europe-west1`. |
| **Regional** | Vive em 1 região (várias zonas dentro) — sobrevive à queda de 1 zona | **External IP addresses (estáticos/reservados)**, regional persistent disks, **subnets**, regional MIGs, Cloud SQL regional, regional autoscaler | Acessível dentro da região. Falha de zona não derruba. |
| **Zonal** | Vive em 1 zona específica — morre se a zona cair | **Instances (VMs)**, **persistent disks (zonal)**, local SSDs, GKE nodes em cluster zonal | Acessível apenas naquela zona. Você é responsável por replicar pra HA. |

**Como aparece nos comandos `gcloud`:**
```bash
# Global — sem flag de localização
gcloud compute snapshots list
gcloud compute images list
gcloud compute networks list

# Regional — exige --region
gcloud compute addresses create my-ip --region=us-central1
gcloud compute addresses list --filter="region:us-central1"

# Zonal — exige --zone
gcloud compute instances create my-vm --zone=us-central1-a
gcloud compute instances list --filter="zone:us-central1-a"
```

**Por que importa.**
- **Zonal** → você sozinho é responsável por HA cross-zone (use MIG regional, persistent disk regional, ou replicação de aplicação)
- **Regional** → tolera queda de 1 zona automaticamente, mas se a região cair você precisava de cross-region
- **Global** → snapshot/image é restaurável em qualquer região; ótimo pra **DR** (disaster recovery) cross-region

**Armadilhas de prova críticas.**
- **External IP address.** **Estático/reservado é regional** (você reserva em uma região e ele só vale lá). **Ephemeral é zonal** (vem com a VM, morre com a VM). Cuidado, isso confunde.
- **Persistent disk** tem 2 sabores: **zonal** (padrão, mais barato, vive em 1 zona) e **regional** (cross-zone HA, mais caro, ~2x).
- **VPC network é global**; **subnets são regionais**. Numa pergunta tipo "qual recurso de network é global?" — VPC network é a resposta, subnet não.
- **Snapshot é global**, mesmo tendo sido tirado de um disco zonal — você pode restaurar em outra zona/região.
- **GKE cluster** pode ser **zonal** (control plane em 1 zona) ou **regional** (control plane replicado em 3 zonas) — afeta tanto disponibilidade quanto custo do control plane.

### 12. Quotas — limites por projeto

**O quê.** Todo projeto GCP está sujeito a **quotas** (limites) impostas pelo Google. Quotas são **guarda-rails** que protegem o usuário (de gastar demais por erro/ataque) e o Google (de um projeto consumir recursos infinitos). Quotas vivem **no nível do projeto**, não do usuário.

#### As 3 categorias de quota

| Categoria | O que limita | Exemplo |
|-----------|--------------|---------|
| **Quantos recursos por projeto** | Quantidade máxima de objetos do mesmo tipo | **15 VPC networks por projeto** (default) |
| **Rate limit de API** | Requests por segundo a uma API específica | **5 ações administrativas/seg** na API do Cloud Spanner |
| **Quota regional** | Recursos disponíveis dentro de uma região | **24 CPUs por região** (default em região nova) |

**Por que três categorias.** Cada uma protege contra um padrão de abuso diferente:
- **Por projeto** → evita "explosão" de objetos (10000 firewall rules num projeto)
- **Rate limit** → evita DDoS interno no plano de controle (create/destroy 1000 VMs/s)
- **Regional** → evita 1 projeto monopolizar capacidade física de uma região

#### Por que quotas existem (3 motivações canônicas)

1. **Prevenir runaway consumption** em caso de erro ou ataque
   - Você roda `gcloud` em loop, em vez de 10 VMs cria 100 → quota corta antes
2. **Prevenir spikes/surpresas de billing**
   - Mesmo mecanismo limita até onde a fatura pode crescer rapidamente
3. **Forçar sizing consideration e revisão periódica**
   - "Você precisa MESMO de uma VM de 96 cores, ou um `n2-standard-4` resolve?"

#### Como inspecionar e pedir aumento

```bash
# Quotas globais do projeto (CPUs globais, IPs, networks...)
gcloud compute project-info describe --project=PROJECT_ID

# Quotas regionais
gcloud compute regions describe us-central1

# Via API/IAM
gcloud alpha services quota list \
  --service=compute.googleapis.com \
  --consumer=projects/PROJECT_ID
```

**Pedir aumento:** Console → **IAM & Admin → Quotas** → filtra serviço/região/métrica → **Edit Quotas** → justifica → submete. Aprovação leva **1-2 dias úteis**.

**Quem aprova.** ACE normalmente **não tem permissão** — fluxo passa por `roles/serviceusage.quotaAdmin` ou Org Admin. ACE **identifica** o gargalo e **submete** o request.

#### Armadilhas de prova críticas

1. **Quota é MÁXIMO, NÃO é GARANTIA.**
   Tem quota de 100 local SSDs em `us-central1`? Ótimo — mas se a região não tem **capacidade física** disponível naquele momento, você **não consegue criar**, mesmo dentro da quota. Quota = teto autorizado; capacidade = teto físico. **Os dois** precisam estar OK.

2. **Quotas são por PROJETO, não por usuário.**
   Não importa se 1 ou 50 pessoas estão criando recursos no mesmo projeto — o teto é o mesmo.

3. **Aumento de quota NÃO é instantâneo.**
   Pergunta "preciso de 200 CPUs amanhã, o que faço?" → resposta inclui **abrir o request com antecedência** (1-2 dias úteis).

4. **Quota ≠ Budget alert.**
   - **Quota** limita **quantos recursos** você cria
   - **Budget alert/cap** limita/avisa sobre **quanto você gasta**
   - São controles diferentes e **complementares**. Quota previne "1000 VMs criadas por erro"; budget previne "10 VMs caras esquecidas ligadas o mês inteiro".

5. **Default quota varia com a idade/tipo da conta.**
   Conta trial nova tem quotas baixas (8 CPUs em vários lugares). Conta paga estabelecida sobe automaticamente com o tempo conforme uso real.

#### Cenário real — pré-checagem para promoção

Você lança promoção sexta-feira prevendo 10x o tráfego. App roda em GKE com autoscaler até 50 nodes. **Antes de sexta:**
- Quota de CPUs na região cabe 50 nodes × 4 CPUs = 200 CPUs?
- Quota de IPs externos comporta os nodes (se cada um tem IP público)?
- Quota de persistent disks suficiente?
- Se faltar quota → abrir request com **2-3 dias** de antecedência

Sem essa checagem: autoscaler tenta subir node, bate em quota, GKE para de escalar, app degrada na hora pior possível.

## Armadilhas de prova consolidadas

1. **Folder é opcional** na hierarquia — Project é o único obrigatório
2. **Hierarquia correta:** Organization → Folder → Project → Resource (DQ 02)
3. **Project Name** é o único atributo identificador mutável (DQ 03)
4. **Groups > usuários individuais** sempre que possível (DQ 01)
5. **Least privilege:** predefinidas/custom no lugar de basic (DQ 05)
6. **Atribuir no maior escopo que faz sentido** (folder em vez de cada projeto, se aplicável) (DQ 04)
7. **`roles/compute.viewer` = READ-ONLY:** `list`, `get`; NÃO tem `create`/`setIamPolicy`/`update` (DQ 06)
8. **Políticas IAM herdam descendentemente e são aditivas** — não conseguem ser revogadas em nível inferior
9. **Habilitar API ≠ ter IAM:** são duas barreiras independentes
10. **"Administrator" NÃO é basic role** — basic roles são apenas Viewer / Editor / Owner. "Admin" só aparece em predefinidas específicas (ex.: `storage.objectAdmin`)
11. **Editor é a basic role mais baixa que ALTERA estado** — Viewer só lê; Owner também altera mas é a mais alta
12. **Cloud Shell ≠ gcloud CLI** — Cloud Shell é o AMBIENTE (browser), gcloud CLI é o CONJUNTO DE EXECUTÁVEIS. O Cloud Shell tem o gcloud CLI pré-instalado e autenticado
13. **Allow vs Deny policy:** allow é aditivo + herdado; **deny PREVALECE sobre allow**. Pra impedir um Owner de uma ação específica sem tirar Owner, use deny policy
14. **Billing flui BOTTOM-UP** (resource → project → billing account). Recurso pertence a exatamente 1 projeto; projeto a exatamente 1 billing account
15. **Escopo físico:** Global (snapshots, images, VPC networks), Regional (subnets, IPs estáticos, regional PDs, Cloud SQL regional), Zonal (VMs, PDs zonais, local SSDs)
16. **External IP estático = REGIONAL; ephemeral = ZONAL.** VPC network é global, subnet é regional. Snapshot sempre é global, mesmo de disco zonal
17. **Quota é MÁXIMO, NÃO GARANTIA.** Pode ter quota e não ter capacidade física na região
18. **Quota é POR PROJETO**, não por usuário. Quota ≠ Budget alert (quota limita recursos, budget limita gasto)
19. **Aumento de quota leva 1-2 dias úteis** — abra request com antecedência. Approval via `roles/serviceusage.quotaAdmin`
