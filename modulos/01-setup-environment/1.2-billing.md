## Conceitos

### 1. Billing Account no Google Cloud

**O quê.** Uma "conta de pagamento". Define **quem paga** pelo uso de recursos GCP — método de pagamento (cartão de crédito, invoice mensal, etc.) e contatos financeiros.

**Regras importantes (decoram, caem em prova):**
- 1 billing account → **N projetos** (pode pagar por vários)
- 1 projeto → **só 1 billing account ATIVA por vez** (não dá pra dividir custos entre 2)
- Projeto sem billing account vinculada → **não consegue usar recursos pagos** (só free tier)
- Billing accounts **NÃO são projetos** — são entidades separadas, gerenciadas no Cloud Billing
- Uma billing account pode pagar projetos de **organizações diferentes**

**Por que importa.**
- **Atribuição de custo:** Cymbal separa billing por departamento porque cada um responde pela própria fatura. Operations não quer ser cobrada pelo BigQuery do Marketing
- **Migração CAPEX → OPEX:** ao mover pra cloud, gasto deixa de ser CAPEX (compra de servidor) e vira OPEX (mensal, variável). Cada departamento controla seu próprio OPEX — mais flexibilidade orçamentária

**Exemplo Cymbal** (do vídeo):

```
Cymbal Org
├── Billing Account: Operations Billing
│   └── Projects: supply-chain-dev / staging / prod
├── Billing Account: Logistics Billing
│   └── Projects: transport-dev / staging / prod
└── Billing Account: Sales & Marketing Billing
    └── Projects: ecommerce-dev / staging / prod
```

**Armadilha de prova (DQ 07).**

A pergunta é "Como as billing accounts são **aplicadas** aos projetos? (Pick two)". Resposta canônica = as duas metades da **cardinalidade N:1** (N projetos por 1 billing).

- ✅ Verdadeiro #1: "**Um projeto e seus recursos só podem ser vinculados a uma única billing account**" (1-para-1 do lado do projeto)
- ✅ Verdadeiro #2: "**Uma billing account pode estar ligada a um OU MAIS projetos**" (1-para-N do lado da billing)
- ❌ Distrator clássico: "Um projeto pode ser ligado a MAIS DE UMA billing account" — falso direto, inverte a regra
- ⚠️ Verdadeira mas NÃO é a resposta esperada: "Cloud Billing paga usage em GCP **e** Google Workspace" — frase verdadeira sobre o escopo do Cloud Billing, mas **não** responde "como APLICADAS a projetos". Em uma pergunta "pick two", o examinador espera as duas frases sobre cardinalidade, não esta sobre escopo
- ❌ (alternativa do workbook original, omitida em algumas versões do quiz) "Se só usa recursos free, não precisa de billing" — em contexto enterprise quase sempre precisa; mesmo em sandbox pessoal muitos serviços exigem billing account vinculada

### 2. Vincular billing a um projeto

```bash
# Vincular projeto a uma billing account
gcloud beta billing projects link PROJECT_ID \
  --billing-account=BILLING_ACCOUNT_ID

# Ver projetos linkados a uma billing account
gcloud beta billing projects list --billing-account=BILLING_ACCOUNT_ID

# Desvincular billing de um projeto (cuidado: serviços param)
gcloud beta billing projects unlink PROJECT_ID
```

Ou via Console: **Billing → Account management → Link a project**.

**Sem billing ativo:** muitos serviços (Compute Engine, GKE, Cloud SQL, BigQuery) ficam **indisponíveis** mesmo com APIs habilitadas. Você verá erros tipo `Billing must be enabled for the project`.

### 3. Budgets e Budget Alerts

**O quê.** Definir um **orçamento mensal** e ser **alertado** quando o gasto atinge certo % do budget. É a primeira linha de defesa contra surpresas de fatura.

**Configuração:**
- **Scope:** billing account inteira OU projetos/serviços/labels específicos
- **Amount:** valor fixo ($X por mês) ou baseado no último mês (auto-ajusta com crescimento)
- **Threshold rules:** alertar em 50%, 90%, 100% por padrão (configurável, pode adicionar mais)
- **Tipo de threshold (dois sabores):**
  - **Actual spend** — dispara quando o gasto REAL chega no %
  - **Forecasted spend** — dispara quando a **projeção** indica que vai ultrapassar até o fim do período (mesmo que o gasto atual ainda esteja baixo). Útil pra reagir cedo, sem esperar o estouro.
- **Destinatários do alerta:**
  - **Default** — Billing Admins e Billing Account Users automaticamente
  - **Custom email recipients** — você pode adicionar até **5 emails específicos** que não precisam ter role de billing
  - **Pub/Sub topic** — para automação programática

**Exemplo Cymbal** (do vídeo):
- Sales & Marketing tem budget mensal de $X para o projeto `ecommerce-prod`
- Particular preocupação com custo de BigQuery (data warehouse)
- Alertas configurados em 50%, 80%, 100% para `finance-marketing@cymbal.com`
- Lead do departamento (Jeffrey) também quer receber → adicionar como custom email recipient (mais sobre isso abaixo)

**⚠️ ARMADILHA CRÍTICA — decoreba certa de prova:**

> Budget alert **NÃO CORTA** o serviço quando o gasto estoura. É apenas notificação.

Se você quer auto-shutdown ao ultrapassar limite:
1. Budget envia alerta para um **Pub/Sub topic**
2. Cloud Run function (ou Cloud Function) consome o topic
3. A function executa lógica de cortar custo:
   - Desabilitar billing no projeto (`gcloud beta billing projects unlink`)
   - Deletar VMs/recursos caros
   - Reduzir capacidade

Isso vai cair em prova como cenário do tipo "como **impedir** que o gasto ultrapasse $X" — a resposta é a cadeia **Pub/Sub + Cloud Function**, NÃO o alerta sozinho.

**Armadilha de prova (DQ 08).**

Cenário: "Fiona é billing admin. Jeffrey (lead de marketing) quer receber emails de budget alerts, mas NÃO deve ganhar acesso a outras infos de billing."

Resposta correta: **Adicionar Jeffrey como custom email recipient no budget scope** (caixa de "custom email delivery" do budget).

Por que as outras estão erradas:
- **"Mudar threshold rules default para incluir Jeffrey"** ❌ — os defaults vão para Billing Admins/Users (roles), e Jeffrey não tem (nem deve ter) essa role
- **"Usar Cloud Monitoring notification channels"** ❌ — Billing alerts usam o canal próprio de email/Pub/Sub do Billing, não o de Cloud Monitoring
- **"Pub/Sub topic que Jeffrey assina"** ❌ — overkill quando o requisito é apenas receber email. Pub/Sub é para automação, não para um humano consumir diretamente

**Lição:** quando precisa dar visibilidade SEM dar acesso administrativo, use o mecanismo de notificação dedicado do recurso (custom recipients), não promova a pessoa a Admin/User.

### 4. Billing Export

**O quê.** Configurar export automático dos dados detalhados de billing para análise offline.

**Destinos suportados:**
- **BigQuery** — para análise via SQL (mais comum em enterprise)
- **Cloud Storage** — arquivos CSV/JSON em bucket (arquivamento, compliance)

**Por que importa.**
- Console de Billing dá visão geral, mas para análise profunda (custo por SKU, por label, por dia, tendência de 6 meses, projeção) você precisa dos dados raw em BigQuery
- Custom dashboards no Looker Studio consomem o export do BigQuery
- Compliance: alguns reguladores pedem retenção de N anos de dados de billing → export para GCS resolve com lifecycle policy barata

**Exemplo Cymbal** (do vídeo):
- Cada departamento configura seu próprio billing export
- Dataset BigQuery dedicado (ex.: `billing_marketing.cymbal_billing_export`)
- Looker Studio dashboard consome essa tabela e mostra custo do BigQuery por dia, por dataset, etc.
- Permite cada departamento responder por seus próprios gastos

**Custo do export:**
- O export em si é **gratuito**
- Você paga pelo **storage** no BigQuery/GCS dos dados exportados (geralmente alguns centavos a poucos dólares por mês, dependendo do volume)

**Habilitar:**
- Console: **Billing → Billing export → BigQuery export → Edit settings**
- Define dataset BigQuery de destino + tipo de export (standard usage, detailed usage, pricing)

### 5. Labels — chargeback dentro do mesmo projeto

**O quê.** Labels são pares **key:value** anexáveis a quase qualquer recurso GCP. O ponto crítico aqui em billing: **labels propagam para o BigQuery billing export**, o que permite agrupar custo por dimensões que **não** correspondem a project/folder/org.

**Por que isso resolve o que projetos não resolvem.**
- Projeto isola **billing inteiro** — bom para separar Marketing de Operations
- Mas dentro de um **mesmo projeto** com vários times/componentes, você não tem como dividir a fatura via hierarquia
- Labels (`team:X`, `component:Y`, `environment:Z`) → cada recurso ganha "etiquetas" que descem no export e permitem `GROUP BY` em SQL

**Onde label aparece no export do BigQuery:**

Cada linha de uso vira uma linha na tabela `gcp_billing_export_v1_XXXXX` com uma coluna `labels` (array de structs `{key, value}`).

```sql
-- Custo por time dentro de um único projeto
SELECT
  l.value AS team,
  SUM(cost) AS total_cost
FROM `myproj.billing_export.gcp_billing_export_v1_XXXXX`,
  UNNEST(labels) AS l
WHERE l.key = 'team'
  AND invoice.month = '202605'
GROUP BY team
ORDER BY total_cost DESC;
```

**Casos típicos para billing/governança:**

| Padrão | Exemplo | Resolve |
|--------|---------|---------|
| Time | `team:marketing` | Chargeback entre departamentos no mesmo projeto |
| Cost center | `cost-center:cc-4521` | Atribuição contábil mais granular que time |
| Componente | `component:frontend` | "Quanto custou nosso front-end vs backend?" |
| Ambiente | `environment:dev` | Só quando dev/prod compartilham projeto (ruim mas acontece) |
| Owner | `owner:gaurav` | Notificar quem deixou recurso caro ligado |
| Estado | `state:readyfordeletion` | Filtro pra script de cleanup automático |

**Limite e regras úteis.**
- Até **64 labels** por recurso
- Key: lowercase, max 63 chars, começa com letra
- Mudar label depois é OK — labels são **mutáveis** (diferente de Project ID)

**Comandos rápidos:**

```bash
# Adicionar labels
gcloud compute instances add-labels my-vm \
  --labels=team=marketing,environment=production \
  --zone=us-central1-a

# Filtrar por label
gcloud compute instances list --filter="labels.team=marketing"

# Bucket
gcloud storage buckets update gs://my-bucket --update-labels=team=marketing

# Dataset BigQuery
bq update --set_label team:marketing PROJECT_ID:DATASET
```

#### Labels vs Network Tags — armadilha de prova

| Aspecto | **Labels** | **Network Tags** |
|---------|-----------|------------------|
| Formato | **key:value** | string só |
| Aplica-se a | Quase todo recurso GCP | **Só instâncias** (VMs) |
| Função | Organizar, filtrar, **billing rollup** | **Firewall rules e routes** |
| Propaga em billing? | ✅ SIM | ❌ NÃO |

**Em pergunta de prova:**
- "Quero ver custo por time no mesmo projeto" → **labels**
- "Quero permitir tráfego HTTP só nos web servers" → **network tags** (`--target-tags=web-server`)
- "Aplicar tag pra firewall em bucket Cloud Storage" → **impossível** (tag é só de VM); bucket usa label, não tag

**Lições para billing.**
1. **Label é o principal mecanismo de billing rollup dentro de um projeto.** Sem label, o BigQuery export só permite agrupar por projeto/serviço/SKU.
2. **Plano-aware:** labels não substituem a separação em projetos. Para isolamento de IAM/quotas/blast radius, separar projetos. Labels resolvem só o eixo financeiro.
3. **Aplique labels na criação, não depois.** Recurso criado sem label retroativamente é difícil de rastrear — custo do dia anterior vai sem label no export. Política mandatória via IaC (Terraform/Deployment Manager) é o ideal.
4. **Atraso no export:** label nova aparece no BigQuery na próxima atualização (algumas horas), não em tempo real.

### 6. Loop de otimização de custo: Label → BigQuery → Looker Studio → Ação

**O quê.** O ferramental do GCP forma um **ciclo fechado de FinOps**:

```
   Labels nos recursos               Cloud Billing
   (team, region, component)             │
            │                            │ (export automático)
            └────────────────┐           ↓
                             ↓     BigQuery dataset
                       (gcp_billing_export_v1_XXXXX)
                             │
                             │ (SQL, GROUP BY labels.X)
                             ↓
                       Looker Studio dashboard
                       (custo por dimensão)
                             │
                             │ (decisão humana)
                             ↓
                       Ação: relocar VM, ligar Cloud CDN,
                       desligar recurso, mudar SKU
```

Cada peça resolve uma camada:

| Camada | Função |
|--------|--------|
| **Labels** | Adicionam dimensões além de project/serviço/SKU (`team`, `component`, `region-of-customer`...) |
| **Cloud Billing export → BigQuery** | Persiste o dado bruto, viabiliza análise em SQL com retenção customizada |
| **BigQuery** | Data warehouse que aguenta volume + queries ad-hoc rápidas (SQL standard) |
| **Looker Studio** | Camada de visualização (gratuita) que conecta no BigQuery e gera dashboards compartilháveis sem código |
| **Decisão humana** | Olha o dashboard, encontra a anomalia, decide a ação |

#### Exemplo concreto do vídeo — otimização de networking

**Cenário:** suas VMs estão em `us-central1`, mas o dashboard do Looker Studio mostra **custo de network egress** desproporcional. Cruzando com a label `customer-region:eu`, você descobre que a maioria do tráfego sai para Europa. Cobrança por **egress inter-continente é cara**.

**Caminho de análise:**
1. Label nas VMs: `app:catalog`, `customer-region:eu` (etiqueta aplicada via IaC na criação)
2. Export do billing rodando há semanas no BigQuery
3. Looker Studio mostra `Networking — Inter-Region Egress` puxando 40% do custo do app
4. Decisão: ou (a) **relocar VMs para `europe-west1`** (origem perto do usuário), ou (b) **ligar Cloud CDN** com cache na borda (Europa)
5. Em N dias, o mesmo dashboard valida se o custo caiu

**Por que isso importa pra prova.** Pergunta tipo "qual ferramenta usar pra analisar gasto detalhado e identificar otimizações?" → **Cloud Billing export → BigQuery → Looker Studio**. Não confunde com:
- **Cloud Monitoring** → métricas de saúde de recursos (CPU, latência), não custo
- **Recommender / Active Assist** → sugestões geradas pelo GCP automaticamente, complementar mas não substitui análise via dashboard
- **Console Reports** → bom pra visão rápida, ruim pra slicing custom

#### Outras decisões guiadas por labels

| Sinal no dashboard | Possível ação |
|--------------------|---------------|
| 1 time consome 70% do orçamento sem entregar valor proporcional | Conversar com o time, revisar arquitetura |
| Componente `redis` com custo crescendo linear | Avaliar Memorystore vs auto-managed VMs |
| `environment:dev` custando perto de `environment:prod` | Investigar VM dev esquecida ligada / dimensionada igual a prod |
| Spike em SKU específico (ex.: `BigQuery Streaming Insert`) | Considerar mudar para batch load |
| Egress alto em horário específico | Mover job pra dentro da mesma região / usar Pub/Sub interno |

**Por que importa.** O FinOps não é "olha a fatura grande". É **encontrar a dimensão** que explica o custo e **decidir a ação**. Sem labels você só tem projeto/serviço/SKU. Com labels você tem qualquer dimensão que o negócio precisar.

#### Armadilha de prova

- **Email de budget alert NÃO contém detalhamento por SKU/serviço.** Ele só diz "projeto X estourou Y% de Z $". Pra entender O QUÊ estourou, você precisa olhar o **export no BigQuery** ou Console → Billing → Reports. Pergunta tipo "como saber qual serviço causou o estouro?" — resposta NÃO é "olhar o email", e sim BigQuery export ou Reports.
- **Looker Studio é Google Cloud, mas é um produto separado** com sua própria autenticação/permission. Em pergunta "como compartilhar dashboard com finanças sem dar acesso ao GCP" → Looker Studio compartilha como Google Docs (link), não precisa de IAM no projeto.

### Lab Qwiklabs "Examine Billing Data with BigQuery" — concluído em 2026-05-15

**Objetivo do Google nesse lab.** Te ensinar a navegar o **billing export no BigQuery** com SQL básico-intermediário. É a primeira camada de FinOps: você sai de "olhar a fatura" pra "perguntar SQL específico sobre o gasto". O lab também reforça o entendimento do **schema do billing export** (RECORD aninhado, tipos: STRING, TIMESTAMP, FLOAT, RECORD, REPEATED).

#### Fluxo do lab

1. **Criar dataset** `billing_dataset` em região US, com expiração de tabela em 1 dia (limpa o ambiente Qwiklabs depois)
2. **Criar tabela `sampleinfotable`** importando o Avro `gs://cloud-training/archinfra/BillingExport-2020-09-18.avro`
3. **Examinar schema** (mesmo do billing export real: billing_account_id, service, sku, project, labels, location, cost, usage...) — neste sample são **415.602 linhas**
4. **Rodar 7 queries** evoluindo em complexidade

#### As 7 queries-chave (decorar os padrões)

```sql
-- Q1: filtro simples — quantas linhas têm custo > 0?
SELECT * FROM `billing_dataset.sampleinfotable`
WHERE Cost > 0
-- → 1.002 linhas (vs 415.602 total → 99,8% das linhas têm custo zero)

-- Q2: projeção com campos aninhados
SELECT billing_account_id, project.id, project.name, service.description,
       currency, currency_conversion_rate, cost,
       usage.amount, usage.pricing_unit
FROM `billing_dataset.sampleinfotable`
-- → 415.602 linhas; ensina dot notation (project.id, usage.amount)

-- Q3: top N mais recentes
SELECT service.description, sku.description, location.country, cost,
       project.id, project.name, currency, currency_conversion_rate,
       usage.amount, usage.unit
FROM `billing_dataset.sampleinfotable`
WHERE Cost > 0
ORDER BY usage_end_time DESC
LIMIT 100
-- ensina ORDER BY ts DESC LIMIT N — padrão "últimos N eventos"

-- Q4: filtro por valor maior
... WHERE cost > 10
-- ensina filtros por faixa de custo (excluir ruído)

-- Q5: histograma por serviço (volume)
SELECT service.description, COUNT(*) AS billing_records
FROM `billing_dataset.sampleinfotable`
GROUP BY service.description
ORDER BY billing_records DESC
-- → Compute Engine domina com 281.136 registros

-- Q6: histograma de serviço com filtro
SELECT service.description, COUNT(*) AS billing_records
FROM `billing_dataset.sampleinfotable`
WHERE cost > 1
GROUP BY service.description
ORDER BY billing_records DESC
-- → Cloud Storage lidera com 17 (volume com cost > $1 muda o ranking)

-- Q7: histograma por unidade de medida
SELECT usage.unit, COUNT(*) AS billing_records
FROM `billing_dataset.sampleinfotable`
WHERE cost > 0
GROUP BY usage.unit
ORDER BY billing_records DESC
-- → byte-seconds (storage com tempo domina)

-- Q8: custo agregado por serviço (a query mais importante de FinOps)
SELECT service.description, ROUND(SUM(cost), 2) AS total_cost
FROM `billing_dataset.sampleinfotable`
GROUP BY service.description
ORDER BY total_cost DESC
-- → Compute Engine $2.548,77 (volume + custo, geralmente lidera os dois)
```

#### Respostas das perguntas de progresso

| Pergunta | Resposta | Lição |
|----------|----------|-------|
| Total de linhas na tabela | **415.602** | Sample maior que o lab anterior (70.765) |
| Linhas com `Cost > 0` | **1.002** | 99,8% das linhas têm custo zero (free tier, créditos) — sempre filtrar antes de analisar |
| Produto com mais registros | **Compute Engine — 281.136** | GCE fatura em granularidade fina → infla volume |
| Produto com mais registros `cost > $1` | **Cloud Storage — 17** | Ranking muda quando filtra: GCE tem muita linha de centavo; GCS tem menos linhas mais caras |
| Unidade mais comum | **byte-seconds** | Storage com tempo (disco persistente, memória de VM ligada) domina |
| Maior custo agregado | **Compute Engine — $2.548,77** | GCE quase sempre lidera custo num projeto típico |

#### Lições que valem para feira e prova

1. **Schema do billing export é hierárquico** — `service`, `project`, `usage`, `location`, `sku` são todos RECORDs. Acessa com dot notation (`service.description`, não `service`).
2. **99% das linhas é ruído de custo zero.** Pra qualquer análise de gasto real, **sempre** filtra `WHERE cost > 0` (mínimo) ou `cost > 1` (pra remover sub-cent).
3. **Volume ≠ custo.** GCE produz mais linhas (volume), mas em outros projetos BigQuery ou Cloud Storage pode liderar custo. Sempre rode as duas queries.
4. **Stack canônica de FinOps no GCP** se monta com essas mesmas queries: `SUM(cost) GROUP BY service`, `GROUP BY UNNEST(labels)` para chargeback, `GROUP BY DATE_TRUNC(usage_start_time, MONTH)` para tendência.
5. **Avro como formato de carga** — mais compacto que CSV/JSON, preserva schema (BQ deduz tipos automaticamente).

#### Pegadinhas do lab

- A primeira tabela do lab anterior (70.765 linhas) NÃO é a mesma deste lab (415.602 linhas). Mesmo nome de arquivo, samples diferentes — o Qwiklabs prepara datasets pra cada lab.
- **`cost > 1` vs `cost > 0`** muda completamente o resultado de quem lidera em volume. Cuidado em pergunta de prova que mistura threshold com agregação.
- **Configuração do dataset:** localização **US** (multiregion barato) + expiração de 1 dia. Em produção, expiração de 1 dia é absurda — escolha do Qwiklabs pra limpar lixo.

## Armadilhas de prova consolidadas

1. **1 projeto = 1 billing account ATIVA por vez** | **1 billing account = N projetos** (DQ 07)
2. **Budget alert é APENAS notificação** — não corta serviço. Para auto-shutdown: Pub/Sub + Function
3. Para dar visibilidade de budget alert SEM dar billing access → **custom email recipients** no budget (DQ 08)
4. **Cloud Monitoring channels** não são canais de Billing alert — é o canal errado
5. Billing export é **grátis**; storage no destino é pago
6. Billing accounts e Projects são entidades **separadas** — billing não está "dentro" do projeto, e sim ligada a ele
7. **Labels propagam pro billing export** — usar `team:`/`component:`/`environment:` permite chargeback **dentro do mesmo projeto** via `GROUP BY` no BigQuery
8. **Label ≠ Network Tag.** Label = key:value, qualquer recurso, vai no billing. Network Tag = string, só VM, só firewall/route, NÃO vai no billing
9. **Label não restringe acesso** — não é mecanismo de segurança, IAM é
10. **Label nova entra no BigQuery export com atraso** (algumas horas) — efeito não é imediato
11. **Budget alert tem 2 sabores:** `actual spend` (gasto real chegou no %) e **`forecasted spend`** (projeção indica que vai estourar até o fim do período) — forecast permite reagir cedo
12. **Email de budget alert** só informa "X% de $Y estourado no projeto Z" — **NÃO** detalha por SKU/serviço. Pra descobrir o quê estourou, vá no **BigQuery export** ou Console → Billing → Reports
13. **Stack canônica de análise de custo:** Labels → Cloud Billing export → **BigQuery** → **Looker Studio**. Não confundir com Cloud Monitoring (saúde, não custo)
14. **Looker Studio compartilha como Google Docs** (link) — não precisa de IAM no projeto GCP pra ver o dashboard
