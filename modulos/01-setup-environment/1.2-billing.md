## Conceitos

### 1. Billing Account no Google Cloud

**O quê.** Uma "conta de pagamento". Define **quem paga** pelo uso de recursos GCP — método de pagamento (cartão de crédito, invoice mensal, etc.) e contatos financeiros.

**Regras importantes (decoram, caem em prova):**
- 1 billing account → **N projetos** (pode pagar por vários)
- 1 projeto → **só 1 billing account ATIVA por vez** (não dá pra dividir custos entre 2)
- Projeto sem billing account vinculada → **não consegue usar recursos pagos** (só free tier)
- Billing accounts **NÃO são projetos** — são entidades separadas, gerenciadas no Cloud Billing
- Uma billing account pode pagar projetos de **organizações diferentes**

**Por que importa.**
- **Atribuição de custo:** Cymbal separa billing por departamento porque cada um responde pela própria fatura. Operations não quer ser cobrada pelo BigQuery do Marketing
- **Migração CAPEX → OPEX:** ao mover pra cloud, gasto deixa de ser CAPEX (compra de servidor) e vira OPEX (mensal, variável). Cada departamento controla seu próprio OPEX — mais flexibilidade orçamentária

**Exemplo Cymbal** (do vídeo):

```
Cymbal Org
├── Billing Account: Operations Billing
│   └── Projects: supply-chain-dev / staging / prod
├── Billing Account: Logistics Billing
│   └── Projects: transport-dev / staging / prod
└── Billing Account: Sales & Marketing Billing
    └── Projects: ecommerce-dev / staging / prod
```

**Armadilha de prova (DQ 07).**

A pergunta é "Como as billing accounts são **aplicadas** aos projetos? (Pick two)". Resposta canônica = as duas metades da **cardinalidade N:1** (N projetos por 1 billing).

- ✅ Verdadeiro #1: "**Um projeto e seus recursos só podem ser vinculados a uma única billing account**" (1-para-1 do lado do projeto)
- ✅ Verdadeiro #2: "**Uma billing account pode estar ligada a um OU MAIS projetos**" (1-para-N do lado da billing)
- ❌ Distrator clássico: "Um projeto pode ser ligado a MAIS DE UMA billing account" — falso direto, inverte a regra
- ⚠️ Verdadeira mas NÃO é a resposta esperada: "Cloud Billing paga usage em GCP **e** Google Workspace" — frase verdadeira sobre o escopo do Cloud Billing, mas **não** responde "como APLICADAS a projetos". Em uma pergunta "pick two", o examinador espera as duas frases sobre cardinalidade, não esta sobre escopo
- ❌ (alternativa do workbook original, omitida em algumas versões do quiz) "Se só usa recursos free, não precisa de billing" — em contexto enterprise quase sempre precisa; mesmo em sandbox pessoal muitos serviços exigem billing account vinculada

### 2. Vincular billing a um projeto

```bash
# Vincular projeto a uma billing account
gcloud beta billing projects link PROJECT_ID \
  --billing-account=BILLING_ACCOUNT_ID

# Ver projetos linkados a uma billing account
gcloud beta billing projects list --billing-account=BILLING_ACCOUNT_ID

# Desvincular billing de um projeto (cuidado: serviços param)
gcloud beta billing projects unlink PROJECT_ID
```

Ou via Console: **Billing → Account management → Link a project**.

**Sem billing ativo:** muitos serviços (Compute Engine, GKE, Cloud SQL, BigQuery) ficam **indisponíveis** mesmo com APIs habilitadas. Você verá erros tipo `Billing must be enabled for the project`.

### 3. Budgets e Budget Alerts

**O quê.** Definir um **orçamento mensal** e ser **alertado** quando o gasto atinge certo % do budget. É a primeira linha de defesa contra surpresas de fatura.

**Configuração:**
- **Scope:** billing account inteira OU projetos/serviços/labels específicos
- **Amount:** valor fixo ($X por mês) ou baseado no último mês (auto-ajusta com crescimento)
- **Threshold rules:** alertar em 50%, 90%, 100% por padrão (configurável, pode adicionar mais)
- **Destinatários do alerta:**
  - **Default** — Billing Admins e Billing Account Users automaticamente
  - **Custom email recipients** — você pode adicionar até **5 emails específicos** que não precisam ter role de billing
  - **Pub/Sub topic** — para automação programática

**Exemplo Cymbal** (do vídeo):
- Sales & Marketing tem budget mensal de $X para o projeto `ecommerce-prod`
- Particular preocupação com custo de BigQuery (data warehouse)
- Alertas configurados em 50%, 80%, 100% para `finance-marketing@cymbal.com`
- Lead do departamento (Jeffrey) também quer receber → adicionar como custom email recipient (mais sobre isso abaixo)

**⚠️ ARMADILHA CRÍTICA — decoreba certa de prova:**

> Budget alert **NÃO CORTA** o serviço quando o gasto estoura. É apenas notificação.

Se você quer auto-shutdown ao ultrapassar limite:
1. Budget envia alerta para um **Pub/Sub topic**
2. Cloud Run function (ou Cloud Function) consome o topic
3. A function executa lógica de cortar custo:
   - Desabilitar billing no projeto (`gcloud beta billing projects unlink`)
   - Deletar VMs/recursos caros
   - Reduzir capacidade

Isso vai cair em prova como cenário do tipo "como **impedir** que o gasto ultrapasse $X" — a resposta é a cadeia **Pub/Sub + Cloud Function**, NÃO o alerta sozinho.

**Armadilha de prova (DQ 08).**

Cenário: "Fiona é billing admin. Jeffrey (lead de marketing) quer receber emails de budget alerts, mas NÃO deve ganhar acesso a outras infos de billing."

Resposta correta: **Adicionar Jeffrey como custom email recipient no budget scope** (caixa de "custom email delivery" do budget).

Por que as outras estão erradas:
- **"Mudar threshold rules default para incluir Jeffrey"** ❌ — os defaults vão para Billing Admins/Users (roles), e Jeffrey não tem (nem deve ter) essa role
- **"Usar Cloud Monitoring notification channels"** ❌ — Billing alerts usam o canal próprio de email/Pub/Sub do Billing, não o de Cloud Monitoring
- **"Pub/Sub topic que Jeffrey assina"** ❌ — overkill quando o requisito é apenas receber email. Pub/Sub é para automação, não para um humano consumir diretamente

**Lição:** quando precisa dar visibilidade SEM dar acesso administrativo, use o mecanismo de notificação dedicado do recurso (custom recipients), não promova a pessoa a Admin/User.

### 4. Billing Export

**O quê.** Configurar export automático dos dados detalhados de billing para análise offline.

**Destinos suportados:**
- **BigQuery** — para análise via SQL (mais comum em enterprise)
- **Cloud Storage** — arquivos CSV/JSON em bucket (arquivamento, compliance)

**Por que importa.**
- Console de Billing dá visão geral, mas para análise profunda (custo por SKU, por label, por dia, tendência de 6 meses, projeção) você precisa dos dados raw em BigQuery
- Custom dashboards no Looker Studio consomem o export do BigQuery
- Compliance: alguns reguladores pedem retenção de N anos de dados de billing → export para GCS resolve com lifecycle policy barata

**Exemplo Cymbal** (do vídeo):
- Cada departamento configura seu próprio billing export
- Dataset BigQuery dedicado (ex.: `billing_marketing.cymbal_billing_export`)
- Looker Studio dashboard consome essa tabela e mostra custo do BigQuery por dia, por dataset, etc.
- Permite cada departamento responder por seus próprios gastos

**Custo do export:**
- O export em si é **gratuito**
- Você paga pelo **storage** no BigQuery/GCS dos dados exportados (geralmente alguns centavos a poucos dólares por mês, dependendo do volume)

**Habilitar:**
- Console: **Billing → Billing export → BigQuery export → Edit settings**
- Define dataset BigQuery de destino + tipo de export (standard usage, detailed usage, pricing)

## Armadilhas de prova consolidadas

1. **1 projeto = 1 billing account ATIVA por vez** | **1 billing account = N projetos** (DQ 07)
2. **Budget alert é APENAS notificação** — não corta serviço. Para auto-shutdown: Pub/Sub + Function
3. Para dar visibilidade de budget alert SEM dar billing access → **custom email recipients** no budget (DQ 08)
4. **Cloud Monitoring channels** não são canais de Billing alert — é o canal errado
5. Billing export é **grátis**; storage no destino é pago
6. Billing accounts e Projects são entidades **separadas** — billing não está "dentro" do projeto, e sim ligada a ele
