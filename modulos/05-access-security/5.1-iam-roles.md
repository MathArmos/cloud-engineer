# 5.1 — Gerenciando IAM (Identity and Access Management)

> **Foco:** quem pode fazer o quê em qual nível da hierarquia. Roles administrativas de IAM por escopo (org / folder / project), princípio de menor privilégio aplicado, custom roles.

---

## Conceitos

### Os 4 tipos de Principal (member) que recebem acesso

Antes de pensar em role/permission, é preciso saber **a quem** você está concedendo. O PDF oficial lista 4 categorias de identidade (slide "Assign access to members using IAM"):

| Tipo | Formato típico | Quando usar |
|---|---|---|
| **Google Account** | `userid@gmail.com` (ou qualquer domínio Google-associated) | Humano individual com conta Google |
| **Service Account** | `1234@cloudservices.gserviceaccount.com` | Aplicações e recursos (M2M) — autentica com chaves, não senha |
| **Google Group** | `groupname@googlegroups.com` | Coleção de identidades. Permissões aplicadas ao grupo são herdadas por todos os membros — **forma escalável de gerenciar acesso** |
| **Cloud Identity / Google Workspace Domain** | `alias@example.com` | "Virtual group" representando **todos os usuários registrados da organização** — usado quando se concede acesso a toda a empresa |

> **Por que importa:** o pattern "Group > User" para conceder acesso já foi cobrado várias vezes (ver flashcards 1.1). Em vez de atribuir role ao João, ao Pedro e à Maria, você cria um grupo `compute-viewers@empresa.com`, atribui a role ao grupo, e gerencia entradas/saídas em um lugar só.

> **Workspace vs Cloud Identity (diferença sutil):** ambos gerenciam usuários da empresa, mas **Workspace** é o pacote completo (Gmail corporativo, Drive, Docs, Meet, etc.); **Cloud Identity** é só a parte de **gestão de identidade**, para empresas que **não usam** os apps colaborativos do Workspace. Em IAM, comportam-se do mesmo jeito.

### Roles administrativas de IAM por nível hierárquico

Para **gerenciar bindings IAM** (atribuir/remover roles) em um determinado escopo, você precisa de uma role administrativa específica daquele escopo. Decore esta tabela — ela é a base de várias perguntas:

| Escopo afetado | Role administrativa correta | Permission-chave |
|---|---|---|
| **Organização inteira** | `roles/resourcemanager.organizationAdmin` | `resourcemanager.organizations.setIamPolicy` |
| **Folder (e tudo dentro)** | `roles/resourcemanager.folderIamAdmin` | `resourcemanager.folders.setIamPolicy` |
| **Projeto único** | `roles/resourcemanager.projectIamAdmin` | `resourcemanager.projects.setIamPolicy` |

> **Princípio operacional:** desça na hierarquia o máximo possível até o **menor nó que ainda cobre todos os recursos que você precisa gerenciar**. Esse é o escopo correto pelo princípio de menor privilégio.

#### Exemplo aplicado (cenário Cymbal Superstore)

Você precisa gerenciar IAM dos projetos `ecommerce-dev` e `ecommerce-prod`, ambos dentro da folder `ecommerce` da organização Cymbal Superstore.

- ❌ `organizationAdmin` na org → cobre demais (todas as outras folders e projetos da Cymbal)
- ✅ `folderIamAdmin` na folder `ecommerce` → cobre exatamente os dois projetos e nada além
- ⚠️ `projectIamAdmin` em cada projeto → também respeita least privilege, mas exige **2 atribuições separadas** (mais trabalho, menos auditável). Use quando os projetos **não compartilham** uma folder comum

---

### Por que `iam.securityAdmin` raramente é a resposta certa em prova

`roles/iam.securityAdmin` é uma role genérica que **permite gerenciar IAM policies onde for atribuída**. Em prova, quando aparece junto com `folderIamAdmin` / `projectIamAdmin` / `organizationAdmin`, ela **perde** porque:

1. As roles `*IamAdmin` são **propósito-específicas** ao nível hierárquico (org/folder/project), então elas comunicam melhor a intenção e são auditavelmente mais limpas
2. Em listas de alternativas, geralmente `securityAdmin` aparece **sem escopo claro** ("in IAM") — escopo difuso = não-resposta

> **Regra prática:** se existe uma role com nome `<nível>IamAdmin` na lista de alternativas, ela vence `securityAdmin` para o cenário de "gerenciar IAM naquele nível".

---

### Permission ≠ Role (distinção que cai em prova)

GCP **nunca** atribui permissions diretamente a um principal. O fluxo é sempre:

```
Principal (user/group/SA)  ──[role]──>  Resource
                                Role contém permissions
```

Exemplo:
- `resourcemanager.projects.setIamPolicy` é uma **permission** (verbo concreto: "setar política IAM de projeto")
- `roles/resourcemanager.projectIamAdmin` é uma **role** que contém essa permission entre outras
- Você atribui **a role**, não a permission

> **Pegadinha clássica de prova:** alternativa que diz "atribua a permission `resourcemanager.projects.setIamPolicy` ao usuário" é **errada por construção** — não existe esse caminho em IAM. Sempre se atribui role.

---

### Diagnóstico: erro ao rodar `gcloud projects set-iam-policy`

Quando esse comando falha com `PERMISSION_DENIED`, o caminho de investigação é:

1. **Qual recurso você está modificando?** O projeto.
2. **Qual permission o comando exige?** `resourcemanager.projects.setIamPolicy`.
3. **Onde você precisa que essa permission seja efetiva?** No próprio projeto **ou** em qualquer nó ancestral (folder, org) — porque herança IAM desce.
4. **Qual a role com menor escopo que dá essa permission?**
   - Se você só vai mexer em **um projeto** → `projectIamAdmin` no projeto
   - Se vai mexer em **vários projetos sob a mesma folder** → `folderIamAdmin` na folder
   - Se vai mexer em **toda a org** → `organizationAdmin` na org

### Comandos `gcloud` relacionados

**Ver IAM policy de um nó:**
```bash
gcloud projects get-iam-policy meu-projeto
gcloud resource-manager folders get-iam-policy FOLDER_ID
gcloud organizations get-iam-policy ORG_ID
```

**Conceder role (binding):**
```bash
# No projeto:
gcloud projects add-iam-policy-binding meu-projeto \
  --member="user:joao@empresa.com" \
  --role="roles/resourcemanager.projectIamAdmin"

# Na folder:
gcloud resource-manager folders add-iam-policy-binding FOLDER_ID \
  --member="user:joao@empresa.com" \
  --role="roles/resourcemanager.folderIamAdmin"

# Na org:
gcloud organizations add-iam-policy-binding ORG_ID \
  --member="user:joao@empresa.com" \
  --role="roles/resourcemanager.organizationAdmin"
```

**Setar policy inteira (uso avançado, atomic):**
```bash
gcloud projects set-iam-policy meu-projeto policy.yaml
```
Esse é o comando do enunciado da DQ 02 — exige `*.setIamPolicy` no recurso ou ancestral.

### Atribuindo role pela interface IAM (UI) — passo a passo oficial

Sequência apresentada no PDF (slide "Assign roles in the IAM interface"):

1. **Vá para a página IAM**
2. **Selecione o nível** — projeto, folder ou organização
3. **Show info panel** se não estiver visível
4. **Click Permissions**
5. **Selecione ou adicione um principal:**
   - Se já existe → clique em "Add another role" para empilhar mais uma role
   - Se novo → clique em "Add" e informe o email do principal
6. **Selecione a role** a ser concedida (filtrar por serviço ajuda — ex.: "Storage" → ver só roles de GCS)
7. **Add a condition** (opcional) — IAM Conditions permite role só ser efetiva em determinadas circunstâncias (ex.: só nos últimos 7 dias, só para bucket cujo nome comece com `tmp-`)
8. **Click Save**

> **Por que cai em prova:** "Add a condition" é um passo opcional que existe especificamente para IAM Conditions (recurso avançado). Saber que esse passo existe pode aparecer numa pergunta de "qual recurso permite restringir role por contexto".

### Criação de custom roles — detalhes operacionais

#### Pré-requisito 1: descobrir as permissions disponíveis

Antes de criar a custom role, você precisa saber **quais permissions existem para o recurso-alvo**:

```bash
gcloud iam list-testable-permissions <FULL_RESOURCE_NAME>
```

Exemplo:
```bash
gcloud iam list-testable-permissions //cloudresourcemanager.googleapis.com/projects/meu-projeto
```

> Antes de criar uma custom role do zero, sempre **verifique se já existe uma predefinida** que cobre seu caso. Predefinida > Custom > Basic. O PDF reforça isso indiretamente: "you can also look at the permissions assigned to a specific role by looking at the role metadata".

#### Pré-requisito 2: quem pode criar custom role

Você precisa ter a permission `iam.roles.create`. Quem tem isso por padrão:
- **Owner** do grupo ou projeto
- **Organization Administrator** (`roles/resourcemanager.organizationAdmin`)
- **IAM Role Administrator** (`roles/iam.roleAdmin`)

#### Onde a custom role pode ser criada

| Nível | Comando | Quando faz sentido |
|---|---|---|
| **Project** | `gcloud iam roles create --project=PROJECT_ID` | Role usada só naquele projeto |
| **Organization** | `gcloud iam roles create --organization=ORG_ID` | Role compartilhada por vários projetos da org |

> Custom role **não pode** ser criada em nível de folder. Se quiser compartilhar entre vários projetos, suba para o nível org.

#### Fluxo recomendado para atualizar custom role (oficial do PDF)

> *"To update an existing role, you run roles.get(), update the role locally, and then run roles.patch()."*

Tradução em gcloud:
```bash
gcloud iam roles describe ROLE_ID --project=PROJECT_ID --format=yaml > role.yaml  # = roles.get()
# editar role.yaml localmente
gcloud iam roles update ROLE_ID --project=PROJECT_ID --file=role.yaml             # = roles.patch()
```

Esse é exatamente o caminho da DQ 03 — atualizar in-place preservando o ID e os bindings.

## Cenário real

> "Você é o engineer responsável pelo ambiente de ecommerce de uma empresa. Tem dois projetos: dev e prod. O time de plataforma vai automatizar a criação de bindings IAM via Terraform. Que role vocês precisam que a SA do Terraform tenha?"

**Resposta correta:** `roles/resourcemanager.folderIamAdmin` na folder `ecommerce`. Isso permite que o Terraform crie/altere bindings nos dois projetos via herança, sem precisar de privilégios elevados na organização. Auditavelmente limpo: se algo der errado, o blast radius está limitado à folder.

## Armadilhas de prova

| Alternativa | Por que erra |
|---|---|
| "Atribua a permission X ao usuário" | Permissions não são atribuídas diretamente; só via roles |
| "Use organizationAdmin" para gerenciar projetos de uma folder | Escopo demais — viola least privilege |
| "Use securityAdmin em IAM" quando existem opções *IamAdmin | Genérica perde para propósito-específica |
| "Use basic role Owner" para resolver setIamPolicy | Funciona, mas é o oposto de least privilege; basic roles têm escopo gigante |
| Repetir bindings em N projetos quando todos vivem sob uma mesma folder | Funciona, mas folderIamAdmin na folder é mais limpo |

## Questionários

> Numbering oficial das Diagnostic Questions (DQs) conforme o PDF do Skills Boost para a Seção 5. As perguntas 5.1 cobrem DQ 01–03; as 5.2 cobrem DQ 04–05.

### DQ 02 — Erro em set-iam-policy nos projetos dev e prod (folder ecommerce)

**Enunciado:** Tentando atribuir roles aos projetos dev e prod do app de ecommerce, recebendo erro ao rodar `set-iam-policy`. Os projetos estão organizados na folder `ecommerce` da org Cymbal Superstore. Best practice + least privilege.

**Alternativas:**
- A) `roles/resourcemanager.organizationAdmin` para Cymbal Superstore
- B) **`roles/resourcemanager.folderIamAdmin` para a folder ecommerce** ✅
- C) `roles/iam.securityAdmin` em IAM
- D) `resourcemanager.projects.setIamPolicy` para cada projeto

**Resposta:** B — a folder `ecommerce` é o menor nó que contém os dois projetos. Cobertura mínima, herança automática.

**Por que as outras erram:**
- A) Org-wide é escopo demais (cobre todas as outras folders/projetos da Cymbal)
- C) `securityAdmin` é genérica; `folderIamAdmin` é propósito-específica e vence
- D) `setIamPolicy` é **permission**, não role; e mesmo se fosse role, exigiria 2 atribuições separadas

**Regra:** desça a hierarquia até o menor nó que cobre todos os recursos necessários.

### DQ 03 — Adicionar permissions de Cloud Run a uma custom role existente

**Enunciado:** Custom role administra ambiente dev/test do app de transportation. Piloto migrando Cloud Functions → Cloud Run. Quer dar aos admins acesso às novas resources sem perder o que já existe.

**Alternativas:**
- A) Criar uma nova role com as permissions necessárias e migrar usuários
- B) **Deletar a custom role e recriar** ❌ (armadilha — perde TODOS os bindings)
- C) **Editar a custom role localmente e rodar update** ✅
- D) Copiar a role, adicionar permissions na cópia, deletar a antiga ❌ (mesmo problema do B)

**Resposta:** C — fluxo `describe → editar YAML → update` preserva o ID da role e mantém todos os bindings intactos.

**Por que B e D erram:**
- Deletar custom role **quebra todos os IAM bindings** que apontavam pra ela
- O ID da role deletada fica em **quarentena (7-37 dias)** — não pode reusar imediatamente
- Você teria que re-conceder a role manualmente a cada principal afetado

**Por que A erra:**
- Trabalho desnecessário: cria duas roles paralelas (velha + nova), migração binding a binding, janela de inconsistência onde alguns admins têm uma e outros têm outra
- Polui o catálogo de roles do projeto

**Comandos canônicos:**
```bash
# 1. Exportar definição atual
gcloud iam roles describe meuCustomRole --project=PROJECT_ID --format=yaml > role.yaml

# 2. Editar role.yaml localmente, adicionar permissions:
#    - run.services.get
#    - run.services.create
#    - run.services.update
#    - run.routes.get
#    etc.

# 3. Aplicar update atômico
gcloud iam roles update meuCustomRole --project=PROJECT_ID --file=role.yaml
```

**Mnemônico:** "Custom role = **update**, nunca delete. Delete quebra bindings; update preserva tudo."
