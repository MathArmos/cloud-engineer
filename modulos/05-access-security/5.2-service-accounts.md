# 5.2 — Service Accounts: criação, permissões e anexação à VM

> **Contexto do vídeo (Cymbal Superstore):** O app de supply chain roda em uma stack **LAMP** em VMs do **Compute Engine** e usa **Cloud SQL** como banco. Toda vez que o app atualiza estoque, **a VM precisa falar com o Cloud SQL**. Isso é uma comunicação **máquina-a-máquina (M2M)** — e o mecanismo correto para isso no GCP é a **Service Account**.

---

## Conceitos

### O que é uma Service Account (SA)

Uma Service Account é uma **identidade especial** que **não pertence a um humano** — ela pertence a uma aplicação, a uma VM, a um job, a um serviço gerenciado. Quando o código rodando nessa VM precisa chamar uma API do Google Cloud (ex.: `cloudsql.instances.connect`), ele **não usa seu email pessoal** — ele usa o email da Service Account anexada à VM.

**Detalhe crucial que o vídeo destaca (e cai em prova):**
> *"Service accounts are both identities and managed resources in Google Cloud."*

Isso significa duas coisas ao mesmo tempo:

1. **Identidade (sujeito de IAM)** — você concede roles **PARA** a service account, como faria para um usuário. Ex.: `roles/cloudsql.client` para `supply-chain-sa@meu-projeto.iam.gserviceaccount.com`.
2. **Recurso (objeto de IAM)** — a própria SA é um recurso do projeto. Você pode conceder roles **SOBRE** a SA para que outros usuários a usem (ex.: `roles/iam.serviceAccountUser` permite que um humano "use" a SA ao anexá-la a uma VM, e `roles/iam.serviceAccountTokenCreator` permite gerar tokens em nome dela).

> **Por que importa:** prova adora confundir os dois sentidos. Se a pergunta for *"Bob não consegue criar uma VM com a SA X anexada — o que falta?"*, a resposta é **dar a Bob `roles/iam.serviceAccountUser` SOBRE a SA X**, não nas permissões que a SA tem.

---

### Por que existem Service Accounts (o problema que resolvem)

Antes de SAs, máquinas se autenticavam guardando senhas/credenciais em arquivos no disco — péssimo do ponto de vista de segurança e rotação. Service Accounts resolvem isso porque:

- A VM **assume automaticamente** a identidade da SA anexada (via metadata server interno — `metadata.google.internal`)
- O Google **rotaciona os tokens curtos (1h)** sem intervenção sua
- **Não precisa armazenar nenhuma chave** se a SA estiver anexada ao recurso (essa é a forma recomendada)

> **Por que importa para ACE:** sempre que a prova mostrar uma VM acessando outro serviço GCP (Cloud SQL, GCS, Pub/Sub, etc.), a resposta **quase nunca** é "use uma chave de service account" ou "use a credencial do usuário X". É **"anexe uma service account com a role apropriada"**.

### O fluxo de 3 passos (Cymbal Superstore)

O vídeo apresenta o passo-a-passo canônico. Decore essa ordem — ela aparece em prova como pegadinha (ex.: "Por que a VM não conecta no Cloud SQL apesar da SA ter sido criada?" → faltou o passo 2 ou o passo 3).

#### Passo 1 — Criar a Service Account

- Vá ao projeto, menu **IAM & Admin → Service Accounts → Create Service Account**
- Dê nome (ex.: `supply-chain-sa`) e descrição
- O Google gera automaticamente o **email da SA** no formato:
  ```
  <nome>@<project-id>.iam.gserviceaccount.com
  ```
  Esse email é o **identificador único** da SA — anote-o, vai ser usado em todo binding de IAM.

**Equivalente gcloud:**
```bash
gcloud iam service-accounts create supply-chain-sa \
  --description="SA para o app de supply chain acessar Cloud SQL" \
  --display-name="Supply Chain App"
```

#### Passo 2 — Conceder permissões PARA a SA (autorização)

- Volte para **IAM** (não Service Accounts) → **Grant Access** → cole o email da SA no campo **Principal**
- Adicione a role apropriada. No exemplo do vídeo: **Cloud SQL Instance User** (`roles/cloudsql.instanceUser`) — permite conectar e operar em instâncias Cloud SQL.

> **Pegadinha de prova:** a tela de "Manage permissions" dentro do listing de Service Accounts gerencia permissões **SOBRE** a SA (quem pode usá-la), não as roles **DELA**. Para dar role a uma SA, você vai em **IAM** e a trata como principal normal — exatamente igual a um usuário humano.

**Equivalente gcloud:**
```bash
gcloud projects add-iam-policy-binding meu-projeto \
  --member="serviceAccount:supply-chain-sa@meu-projeto.iam.gserviceaccount.com" \
  --role="roles/cloudsql.instanceUser"
```

#### Passo 3 — Anexar a SA à VM (associação)

- Ao criar a VM (Compute Engine → Create Instance), expanda **Identity and API access**
- Selecione a SA criada no dropdown **Service account**
- Defina **Access scopes** — recomendação atual: **"Allow full access to all Cloud APIs"** e controle por IAM roles na SA (não por scopes legados)

> **Por que isso é a parte mágica:** depois de anexar, **qualquer processo rodando na VM** que use a Google Cloud Client Library encontrará credenciais automaticamente via Application Default Credentials (ADC), sem você configurar nada no código. A biblioteca consulta o **metadata server** (`http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token`) e recebe um token de acesso de curta duração.

**Equivalente gcloud (na criação da VM):**
```bash
gcloud compute instances create supply-chain-vm \
  --zone=us-central1-a \
  --service-account=supply-chain-sa@meu-projeto.iam.gserviceaccount.com \
  --scopes=cloud-platform
```

**Trocar a SA de uma VM existente** (precisa parar a VM antes):
```bash
gcloud compute instances stop supply-chain-vm --zone=us-central1-a
gcloud compute instances set-service-account supply-chain-vm \
  --zone=us-central1-a \
  --service-account=supply-chain-sa@meu-projeto.iam.gserviceaccount.com
gcloud compute instances start supply-chain-vm --zone=us-central1-a
```

### A frase-chave do vídeo (memorize)

> *"Attaching a service account allows the virtual machine and all the apps running on it to use the permissions assigned to the service account."*

Tradução prática: **a SA é "anexada à VM", mas as permissões são herdadas por TODO processo dentro da VM**. Isso significa que, se você roda dois apps na mesma VM, ambos ganham as MESMAS permissões. Se um deles só precisa ler GCS e o outro só precisa escrever no Pub/Sub, **o correto é VMs separadas com SAs separadas** (princípio do menor privilégio).

### Compute Engine Default Service Account vs SA customizada

Toda VM criada em um projeto, **se você não escolher nada**, recebe a **Compute Engine default service account**:
```
<project-number>-compute@developer.gserviceaccount.com
```

Ela vem com **Editor role** no projeto inteiro — o que é **assustadoramente permissivo**. Por isso a recomendação é sempre criar uma **SA dedicada por workload** com permissões mínimas. Em projetos novos criados em organizações com a constraint `iam.automaticIamGrantsForDefaultServiceAccounts` ativa, o Google já não atribui Editor automaticamente — mas em projetos antigos sim.

> **Armadilha clássica:** "Por que a VM consegue criar buckets, listar BigQuery, deletar VMs e mais um monte de coisa que ninguém pediu?" → SA default + role Editor + ninguém revisou.

### Relação VM ↔ Service Account (detalhe operacional do PDF)

Regra que o PDF oficial enfatiza no slide "Create, use, and assign service accounts":

> *"Multiple virtual machine instances can use the same service account, but a virtual machine can only have one service account identity. Service account changes will affect all virtual machine instances using the service account."*

Tradução prática:

| Relação | Permitido? | Implicação |
|---|---|---|
| 1 VM → 1 SA | ✅ Sempre exatamente uma | Não dá pra anexar duas SAs na mesma VM |
| N VMs → 1 SA | ✅ Permitido | Várias VMs compartilham identidade — útil para fleet/pool de mesmo workload |
| 1 VM → N SAs | ❌ Proibido | Se precisa de identidades diferentes, divida em VMs distintas |

> **Consequência forte:** se você alterar as permissions de uma SA compartilhada, **todas as VMs que a usam são afetadas instantaneamente**. Isso é poder e perigo ao mesmo tempo:
> - **Poder:** revogar acesso de uma frota inteira com uma única mudança
> - **Perigo:** errar uma permission da SA derruba todos os apps que dependem dela
>
> Por isso, em ambientes maduros, cada workload de finalidade distinta tem **sua própria SA dedicada**.

### Tipos de chaves de autenticação de uma SA

Quando uma SA precisa autenticar **fora do ambiente Google Cloud** (ex.: workload on-premises chamando GCP), ela usa **chaves**. O PDF (slide final do 5.2) diferencia dois tipos:

| Tipo | Quem armazena a chave privada | Você gerencia? | Quando usar |
|---|---|---|---|
| **User-managed keys** | Você (em arquivo JSON) | Sim — você cria, rotaciona e revoga | Servidor on-prem; integrações legadas. Google só guarda a parte pública |
| **Google-managed keys** | Google (interno) | Não — Google rotaciona automaticamente | Quando o código chama APIs do Google para "assinar" (signBlob) coisas em nome da SA, sem você ver a chave privada |

> **Pegadinha de prova:** "qual o método mais seguro de autenticar uma SA?" → **não usar chave nenhuma** (usar SA anexada ao recurso com metadata server). Se a alternativa "anexar a SA" não estiver lá, prefira **Google-managed key** sobre user-managed. Se for forçado a usar user-managed, **rotacione a cada 90 dias** e armazene em **Secret Manager**, nunca em Git.

### Conceitos avançados do 5.2 (cobertos no exam guide)

O PDF lista 6 considerations para o tópico 5.2. Além do básico, dois conceitos novos:

#### Service Account Impersonation

Permite que um **principal A** (humano ou outra SA) **assuma temporariamente** a identidade de uma SA B para fazer alguma operação. O principal A não precisa ter as permissions de B diretamente — ele só precisa da role `roles/iam.serviceAccountTokenCreator` **SOBRE** a SA B.

Comandos:
```bash
# Impersonar a SA pontualmente (token de curta duração)
gcloud auth print-access-token --impersonate-service-account=alvo-sa@projeto.iam.gserviceaccount.com

# Rodar comando gcloud impersonando
gcloud compute instances list --impersonate-service-account=alvo-sa@projeto.iam.gserviceaccount.com
```

**Quando usar:**
- Dev quer testar localmente o que a SA de produção pode fazer (sem ter as permissions diretamente, sem ter a chave dela)
- Pipeline CI/CD precisa fazer operações de prod sem que o runner precise ter `Editor`

**Por que é melhor que chave JSON:**
- Token impersonado é **de curta duração** (1h, configurável)
- Toda chamada fica **auditada** com o principal real que iniciou (em Cloud Audit Logs aparece "user X impersonated SA Y")
- Não cria secret de longa duração que pode vazar

#### Short-lived service account credentials

API que gera **credenciais temporárias** para uma SA — tokens com TTL configurável (1h por padrão, até 12h). Útil para:
- Distribuir credenciais para workloads externos sem entregar chave permanente
- Cenários de delegação cross-org

O fluxo é parte da Service Account Credentials API (`iamcredentials.googleapis.com`).

> **Padrão moderno geral:** o Google está empurrando todo mundo para parar de gerar chaves JSON e usar impersonation + short-lived credentials. Em prova nova, se aparecer alternativa "use chave JSON" vs "use impersonation", **impersonation vence**.

### Bloco "Service Accounts" (vídeo *Essential Core Services — IAM*)

#### Os 3 tipos de Service Account (taxonomia oficial)

O curso *Essential Core Services* classifica SAs em **três famílias**. Saber distinguir é essencial — perguntas de prova costumam misturar o jargão.

| # | Tipo | Email padrão | Quem criou | Role por padrão | Quando aparece |
|---|---|---|---|---|---|
| 1 | **User-created / Custom SA** | `<nome>@<project-id>.iam.gserviceaccount.com` | Você | Nenhuma (você atribui) | Sempre que você precisa de identidade específica para um workload. **Recomendado** |
| 2 | **Built-in: Compute Engine Default SA** | `<project-number>-compute@developer.gserviceaccount.com` | Google (criada automaticamente em todo projeto) | **Editor no projeto** ⚠️ | Default da VM se você não escolher outra na criação |
| 3 | **Built-in: Google APIs SA** | `<project-number>@cloudservices.gserviceaccount.com` | Google (criada automaticamente em todo projeto) | **Editor no projeto** | **Não é pra você usar** — designed para que Google rode processos internos em nome do seu projeto |

> **Pegadinha crítica:** o **Google APIs service account** (`@cloudservices.gserviceaccount.com`) **NÃO é a mesma** coisa que o Compute Engine default SA (`@developer.gserviceaccount.com`). Ambos têm Editor no projeto, mas:
> - **`cloudservices`** = Google usa internamente. Não anexe a VMs. Não mexa nas permissions dela sem saber o que faz (pode quebrar serviços do projeto)
> - **`developer.gserviceaccount.com`** = a SA que VMs assumem se você não escolher outra na criação

> **Mnemônico:** *"Cloudservices é deles, Developer é seu (mas você não devia querer essa)."*

#### Como uma SA é identificada — e o que escolher na criação da VM

Quando você cria uma VM via `gcloud` (ou Console) **sem especificar** SA, a Compute Engine default SA é anexada automaticamente. Para **sobrescrever** esse comportamento, dois caminhos:

```bash
# (a) Especificar SA customizada
gcloud compute instances create vm-app \
  --zone=us-central1-a \
  --service-account=app-sa@meu-projeto.iam.gserviceaccount.com \
  --scopes=cloud-platform

# (b) Desabilitar SA na VM (raro — fica sem identidade pra falar com APIs do GCP)
gcloud compute instances create vm-sem-sa \
  --zone=us-central1-a \
  --no-service-account \
  --no-scopes
```

#### Access Scopes — o mecanismo **legacy** que ainda confunde

Antes das **IAM roles** existirem, a única forma de limitar o que uma VM podia fazer eram **access scopes**: filtros aplicados ao **token OAuth** que o metadata server entrega.

```
SA tem role = "o que ela pode fazer no universo de APIs"
Scope da VM = "filtro: dessas permissões, quais podem ser usadas POR essa VM"
Permissão efetiva = INTERSEÇÃO (SA roles) ∩ (scopes da VM)
```

**Exemplo do vídeo:**
- App A → token com scope `storage-ro` → só lê GCS, mesmo se a SA tiver `storage.admin`
- App B → token com scope `storage-rw` → lê e escreve GCS

**Onde scopes ainda aparecem:**
- Criação de VM (`--scopes`): valor recomendado moderno é `cloud-platform` (= "todas as APIs habilitadas no projeto"). Controle de fato vem das **IAM roles** da SA, não dos scopes
- Antigos defaults restritivos: `default` scope deixava só `storage-ro` + `logging-write` + `monitoring-write` + algumas outras — esses defaults legacy ainda aparecem em VMs antigas

**Mudar scope de VM em execução:** **proibido**. Precisa parar a VM, alterar, reiniciar:

```bash
gcloud compute instances stop vm-app --zone=us-central1-a
gcloud compute instances set-service-account vm-app \
  --zone=us-central1-a \
  --service-account=app-sa@meu-projeto.iam.gserviceaccount.com \
  --scopes=cloud-platform
gcloud compute instances start vm-app --zone=us-central1-a
```

> **A frase exata do vídeo (memorize):** *"Access scopes are actually the legacy method of specifying permissions for your VM. Before the existence of IAM roles, access scopes were the only mechanism for granting permissions to service accounts. **For user-created service accounts, use IAM roles instead** to specify permissions."*

> **Pegadinha de prova:** alternativa "limite o acesso da VM ajustando os scopes do token" para uma SA customizada. **Errado** — em SA customizada, controle por **IAM roles**, deixe o scope em `cloud-platform`. Scope existe por compatibilidade.

#### Service Account User role — SA como **recurso** (quem pode "usar" a SA)

A frase-chave da segunda parte do vídeo:
> *"Roles for service accounts can also be assigned to groups or users."*

Tradução: além de atribuir roles **PARA** uma SA (tornando-a sujeito), você pode atribuir roles **SOBRE** ela (tratando-a como recurso). O caso clássico:

**Exemplo do vídeo:**
1. Você cria a SA `vm-admin-sa` e atribui a ela `roles/compute.instanceAdmin` (cria/modifica/deleta VMs e disks)
2. Você quer que um grupo `devops@empresa.com` possa **agir como** essa SA (ex.: criar VMs anexando-a)
3. Concede `roles/iam.serviceAccountUser` no recurso SA `vm-admin-sa` para o grupo

Agora qualquer usuário do grupo pode criar/modificar/deletar VMs **através da SA**, sem ter `compute.instanceAdmin` diretamente.

> **Aviso de segurança do vídeo (importante):** *"Users who are Service Account Users for a service account can access all the resources that the service account has access to. Therefore, be cautious when granting the Service Account User role."*

Dar `serviceAccountUser` é, na prática, **dar todas as permissions da SA**. Se a SA tem `roles/owner`, dar `serviceAccountUser` é equivalente a dar Owner.

#### Pattern de microsserviços com SAs por componente

O vídeo apresenta um padrão arquitetural canônico — slice de projeto em microsserviços, **cada um com sua SA**:

```
Project B
├── VMs do component_1 ──► SA-1 (tem roles/editor em project_b)
└── VMs do component_2 ──► SA-2 (tem roles/storage.objectViewer em bucket_1)
```

Resultado:
- VMs do `component_1` podem mexer em tudo dentro de `project_b` (editor) mas **não veem** `bucket_1` (não têm a role)
- VMs do `component_2` só veem `bucket_1` em modo leitura, **nada mais**

Sem SAs separadas, você teria que dar a TODAS as VMs as permissions do componente mais permissivo — quebrando least privilege.

> *"You assign the service accounts to the VMs when they are created, and you don't have to ensure that credentials are being managed correctly because Google Cloud manages security for you."*

A última parte é o **valor principal** das SAs anexadas: rotação de credenciais é problema do Google.

#### Service Account Keys — Google-managed vs User-managed (detalhe fino)

Toda SA tem **Google-managed key-pairs** automaticamente. **Você pode** criar **user-managed keys** ("external keys") por cima.

| | **Google-managed key** | **User-managed key** |
|---|---|---|
| Onde mora a **chave privada** | Em escrow no Google — **nunca acessível diretamente** | Você baixa o arquivo JSON. Google **só guarda a pública** |
| Rotação | Automática | Sua responsabilidade |
| Public key usável para signing por | **Máximo 2 semanas** | Sem limite (até você revogar/rotacionar) |
| Limite por SA | N/A (gerenciado pelo Google) | **Até 10 keys** por SA (para facilitar rotação overlap) |
| Onde criar/listar | Não aplicável | IAM API, `gcloud`, ou Console |
| Se você perde a chave | N/A | **Google não pode recuperar** — você fica fora |
| Quando faz sentido | Default. Sempre que SA está anexada a recurso GCP | Acesso de **fora** do GCP (on-prem, outro cloud) **como último recurso** |

**Listar keys de uma SA:**
```bash
gcloud iam service-accounts keys list \
  --iam-account=app-sa@meu-projeto.iam.gserviceaccount.com
```

**Criar user-managed key (use com parcimônia):**
```bash
gcloud iam service-accounts keys create chave.json \
  --iam-account=app-sa@meu-projeto.iam.gserviceaccount.com
```

> **A frase final do vídeo (decore):** *"User-managed keys should be used as a last resort. Consider the other alternatives, such as **short-lived service account credentials (tokens)**, or **service account impersonation**."*

Ordem de preferência **moderna**, do melhor para o pior:

```
1. Recurso dentro do GCP com SA anexada (metadata server)    ◄── faça SEMPRE que possível
2. Workload Identity Federation (workload externo sem chave)
3. Service Account impersonation (curto prazo, auditado)
4. Short-lived credentials via IAM Credentials API (TTL configurável)
5. User-managed JSON key                                      ◄── último recurso
```

> **Pegadinha:** prova mostra alternativas (a) "crie chave JSON", (b) "use Workload Identity Federation" para autenticar GitHub Actions no GCP. **(b) vence** — chave JSON é o anti-padrão moderno.

> **Por que máximo 2 semanas?** O vídeo cita: *"Each public key can be used for signing for a maximum of two weeks."* Isso é uma propriedade da Google-managed key: mesmo que a chave dure mais internamente, o **uso para assinar JWT** está limitado a uma janela curta — limitando o blast radius caso algo seja comprometido na cadeia de signing.

### Bloco "Best Practices de Service Accounts" (vídeo *Essential Core Services — IAM*)

O vídeo lista 3 best practices específicas de SA. Cada uma cobre uma falha clássica.

#### 1. Cuidado ao conceder `roles/iam.serviceAccountUser`

Já documentado acima — repetido aqui porque o vídeo enfatiza novamente como **a melhor forma de comprometer um ambiente é dar `serviceAccountUser` numa SA muito poderosa**.

> *"Be very careful when granting the Service Account User's role because it provides access to all the resources that the service account has access to."*

Antes de conceder, pergunte: **se essa pessoa quisesse fazer dano, o que ela conseguiria fazer agindo como esta SA?** Se a resposta envolve "deletar prod inteiro", essa SA não devia poder fazer isso — ou o `serviceAccountUser` não devia ser concedido.

#### 2. Display name + naming convention

> *"When you create a service account, give it a display name that clearly identifies its purpose, ideally using an established naming convention."*

Sem naming convention, daqui a 6 meses você terá `sa-1`, `test-sa`, `app-account`, `compute-sa-2024`, `arquivo-jose-talvez-deletar` — e ninguém saberá o que cada uma faz nem se é seguro deletar.

**Convenção recomendada (não é regra GCP, é prática comum):**

```
<workload>-<ambiente>-<finalidade>
```

Exemplos:
- `supply-chain-prod-app` — SA do app de supply chain em produção
- `etl-pipeline-staging-runner` — SA que roda jobs ETL em staging
- `backup-prod-storage-writer` — SA que escreve backups no GCS em prod

**Display name** (campo separado do email, livre):
> `Supply Chain App — Production runtime SA — owner: time-supply-chain`

Inclui:
- Propósito (o que ela faz)
- Ambiente (prod/staging/dev)
- Owner (time responsável — quem chamar quando der ruim)

> **Pegadinha de prova:** ao listar SAs com `gcloud iam service-accounts list`, o display name aparece — facilita auditoria. SAs sem display name claro são candidatas a "abandonadas" e devem ser revisadas.

#### 3. Key rotation + auditoria via `serviceAccount.keys.list`

> *"Establish key rotation policies and methods and audit keys with the serviceAccount.keys.list method."*

User-managed keys são o vetor #1 de comprometimento de SA. O vídeo aponta dois deveres:

**(a) Política de rotação** — defina **periodicidade** (sugestão usual: 90 dias) e **mecanismo** (manual ou automatizado via script/Terraform/Cloud Build).

**(b) Auditoria periódica** — listar keys existentes em todas as SAs e checar:
- Quantas keys cada SA tem (limite 10 — chegando perto = candidata a faxina)
- Quão antigas são (key > 90 dias sem rotação = bandeira vermelha)
- Se ainda são usadas (cruzar com Cloud Audit Logs de uso)

**Comando essencial:**
```bash
gcloud iam service-accounts keys list \
  --iam-account=app-sa@meu-projeto.iam.gserviceaccount.com \
  --format="table(name,validAfterTime,validBeforeTime,keyType)"
```

Saída diferencia:
- `keyType: SYSTEM_MANAGED` — chaves Google-managed (não preocupa, Google rotaciona)
- `keyType: USER_MANAGED` — chaves criadas por você (essas exigem auditoria)

**Script para listar todas as user-managed keys de todas as SAs de um projeto:**
```bash
for sa in $(gcloud iam service-accounts list --format="value(email)"); do
  echo "=== $sa ==="
  gcloud iam service-accounts keys list \
    --iam-account=$sa \
    --filter="keyType=USER_MANAGED" \
    --format="table(name,validAfterTime)"
done
```

> **Pegadinha de prova:** "como descobrir todas as user-managed keys que existem na minha org?" → resposta envolve `serviceAccount.keys.list` (ou equivalente `gcloud iam service-accounts keys list`) com filtro `keyType=USER_MANAGED`, executado para cada SA. **Não** Cloud Audit Logs (esses mostram **uso**, não **existência**).

### Authentication vs Authorization (o gancho final do vídeo)

O vídeo termina dizendo:
> *"This covers authorization. Authentication is another important aspect..."*

Distinção que SEMPRE cai:
- **Authentication (autenticação):** "quem é você?" → resolvida pelo **email da SA** + token assinado pelo Google (via metadata server quando a SA está anexada, ou via chave JSON quando ela não está)
- **Authorization (autorização):** "o que você pode fazer?" → resolvida pelos **roles IAM** atribuídos à SA

O fluxo Cymbal cobre os DOIS: passo 1 (criar) e passo 3 (anexar) resolvem autenticação; passo 2 (atribuir role) resolve autorização.

### Matriz oficial: os 4 tipos de credencial (slide "Types of authentication keys")

Este é o **quadro canônico** do PDF do Skills Boost — decisão de qual credencial usar baseada em **dois eixos**: (1) tipo de dado acessado e (2) ambiente de execução.

| # | Credencial | Para acessar | Quando usar |
|---|---|---|---|
| 1 | **API Key** | **Public data** | App identifica projeto para billing/quota, mas API consultada é pública (ex.: Maps, Translate em modo aberto). Não fornece identidade nem autoriza dado privado |
| 2 | **OAuth 2.0 Client** | **Private end-user data** | App acessa dados **de um usuário específico** em nome dele. Requer consentimento explícito do usuário (tela "Permitir que X acesse seu Y") |
| 3 | **Environment-provided Service Account** | **Recursos privados, código rodando dentro do GCP** | VM, Cloud Run, Cloud Function, GKE pod, etc. **Mais seguro** porque elimina gerenciamento manual de chaves — credencial vem via metadata server |
| 4 | **Service Account key (JSON)** | **Recursos privados, código rodando FORA do GCP** | Servidor on-premises ou outro cloud que precisa autenticar como uma SA. Necessário gerar chave porque não tem metadata server. **Armazenamento seguro é crítico** |

> **Critério de decisão em 2 perguntas:**
> 1. **Que tipo de dado?** público → API Key. Privado do usuário → OAuth. Privado da aplicação → SA.
> 2. **Onde o código roda?** Dentro do GCP → environment-provided SA. Fora do GCP → SA key.

> **Tendência atual (não está no PDF mas vale saber):** o item 4 (SA key) está sendo **gradualmente substituído** por **Workload Identity Federation** — permite que workloads externas (AWS, Azure, GitHub Actions, on-prem) impersonem uma SA **sem precisar de chave JSON**. Em prova nova de pós-2024, alternativa com "Workload Identity Federation" tende a vencer "SA key" para acesso externo.

## Cenário real

> "Você está consultoria/dev numa empresa. O time de backend criou um cron job em uma VM que precisa ler arquivos de um bucket GCS, processar, e gravar resultado em BigQuery."

**Resposta correta:**
1. Criar SA `cron-processor-sa`
2. Conceder a ela: `roles/storage.objectViewer` no bucket específico + `roles/bigquery.dataEditor` no dataset específico
3. Anexar essa SA à VM do cron

**Anti-padrões a evitar (cada um é uma pergunta errada de prova):**
- ❌ Usar a SA default do Compute Engine
- ❌ Gerar uma chave JSON da SA e colocar como variável de ambiente na VM
- ❌ Conceder `roles/editor` no projeto inteiro só porque "é mais rápido"
- ❌ Reusar a mesma SA do app de produção principal

## Armadilhas de prova

| Situação | Resposta-armadilha (errada) | Resposta correta |
|----------|------------------------------|------------------|
| VM precisa acessar Cloud SQL | "Coloque a senha do MySQL num arquivo na VM" | Anexar SA com `roles/cloudsql.client` ou `cloudsql.instanceUser` |
| App em VM, dev quer rodar localmente também com mesma identidade | "Crie uma chave JSON da SA e dê pro dev" | Dev usa **suas próprias credenciais** via `gcloud auth application-default login`; só a VM usa a SA |
| Bob precisa criar VMs anexando a SA `prod-sa` | "Dê `roles/owner` para o Bob" | `roles/iam.serviceAccountUser` em `prod-sa` para o Bob |
| App vai rodar em GKE | "Crie SA, gere chave JSON, monte como secret no pod" | **Workload Identity** — vincula KSA a GSA, sem chaves |
| Migrar VM para usar outra SA | "Edite enquanto a VM roda" | Precisa **parar a VM** antes de `set-service-account` |
| SA precisa de role em outro projeto | "Recrie a SA no outro projeto" | A SA vive no projeto onde foi criada; conceda binding no projeto-alvo usando o email da SA como principal |
| "VM da Cymbal usando a SA `<num>@cloudservices.gserviceaccount.com`" | "Tudo certo, é a SA padrão da VM" | **ERRADO.** Essa é a **Google APIs SA** (uso interno do Google). A SA padrão de VM é `<num>-compute@developer.gserviceaccount.com` |
| Limitar permissão de uma SA customizada usando access scope | "Use scope `storage-ro` e a VM só lê GCS" | Para SA customizada, **use IAM roles** — scope fica em `cloud-platform`. Access scope é legacy |
| Mudar o scope de uma VM em execução | "Edita o scope no Console com a VM ligada" | Precisa **parar a VM** primeiro — scope só muda com instance parada |
| Autenticar GitHub Actions no GCP | "Crie SA, gere chave JSON, salve como secret no GitHub" | **Workload Identity Federation** — sem chave persistente, sem secret no repositório |
| Rotacionar user-managed key sem downtime | "Delete a chave antiga, crie a nova" | Cria a **nova primeiro** (você pode ter até 10 keys por SA), aplica nos workloads, **só depois** revoga a antiga |
| "Public key vale para signing pra sempre" (Google-managed) | "Sim, fica ativa até a rotação" | **Errado** — máximo de 2 semanas para signing, independentemente de quando rotacionar |

## Questionários

> Numbering oficial das Diagnostic Questions (DQs) conforme o PDF do Skills Boost para a Seção 5. As 5.2 cobrem DQ 04–05. (As DQs 5.1 estão em [5.1-iam-roles.md](5.1-iam-roles.md).)

### DQ 01 — App em GKE precisa acessar Spanner. Que tipo de account usar?

(Apesar do número, esta é uma DQ do tópico 5.1 sobre "tipos de identidade". Documentada aqui por afinidade com o tema de service accounts.)

**Alternativas:**
- A) Google Workspace account
- B) Google account
- C) Cloud Identity account
- D) **Service account** ✅

**Resposta:** D — GKE pod chamando Spanner é M2M.

**Por que as outras erram:** A, B, C são todas identidades **humanas** (diferem só em quem gerencia o domínio: Workspace, Google geral, Cloud Identity). Nenhum humano roda dentro de um pod em produção.

**Os 4 tipos de principal no IAM (decore essa lista):**
1. Google account (humano em domínio Google geral)
2. Google Workspace account (humano em domínio Workspace)
3. Cloud Identity account (humano em domínio Cloud Identity)
4. **Service account (máquina)** — a única não-humana

**Produção em GKE:** usar **Workload Identity** (KSA ↔ GSA), nunca chave JSON em secret.

### DQ 04 — Em qual cenário usar uma Service Account?

**Alternativas:**
- A) For individual GKE pods ✅
- B) To directly access user data
- C) For interactive analysis
- D) For development environments

**Resposta:** A — GKE pods são processos M2M. Mecanismo moderno: **Workload Identity** (KSA ↔ GSA sem chaves).

**Por que as outras erram:**
- B) Dados de usuário → **OAuth 2.0 user credentials**, não SA
- C) "Interactive" = humano → credenciais humanas (`gcloud auth login`)
- D) Devs locais → `gcloud auth application-default login` com a própria conta; nunca distribuir chave de SA

**Regra-mãe:** humano no loop → credencial humana; código/container/VM/batch → Service Account.

> **Feedback oficial do PDF para "interactive analysis":** *"Service accounts should be used for **unattended work** that does not require user interaction."* Reforça: SA = workload automático, sem humano olhando.

### DQ 05 — Mobile app de tracking de entregas: como autenticar para ler Pub/Sub?

**Enunciado:** App mobile, end users finais, recebe dados privados (localização de caminhões) de um tópico Pub/Sub. Best practices do Google para **client-side application acessando dado privado GCP**.

**Alternativas:**
- A) **OAuth 2.0 user credentials** ✅
- B) API key
- C) Service account key
- D) Environment provided service account

**Resposta:** A — usuário final loga, consente, recebe token de curta duração. Sem secret embarcado no APK.

**Por que as outras erram (feedback oficial do PDF):**
- B) **API key** → "for quota/billing or public APIs; they offer no identity and cannot securely access private resources"
- C) **Service account key** → "Embedding a Service Account private key in a client app is a critical security risk, as the key can be extracted, granting permanent access to the resource"
- D) **Environment-provided SA** → "only for applications running inside Google Cloud infrastructure (e.g., Cloud Run, Compute Engine), not external mobile apps"

#### Nuance crítica — o fluxo recomendado de fato (importante!)

O feedback oficial do PDF revela um detalhe que generalizei demais antes:

> *"The app uses OAuth 2.0 to get a user token, which a **secure backend** validates and uses a **Service Account** to access Pub/Sub."*

Ou seja, o app mobile **não chama Pub/Sub diretamente** com o OAuth token do usuário. O fluxo correto é uma **arquitetura em duas etapas**:

```
[Mobile App]
   │
   │ 1. Login OAuth 2.0 do usuário
   ▼
[Google OAuth] ──> token do usuário (curto, com escopo limitado)
   │
   ▼
[Mobile App] ──── 2. chama backend com token do usuário ────►
                                                              [Backend seguro]
                                                                  │
                                                                  │ 3. valida o token
                                                                  │ 4. usa SA própria
                                                                  ▼
                                                              [Pub/Sub API]
```

**Por que esse padrão é o canônico:**
- Mobile só precisa autenticar o **usuário** — não tem chave de aplicação embarcada
- Backend é quem tem identidade GCP (SA com role Pub/Sub Subscriber) — está em ambiente seguro
- Backend pode aplicar lógica adicional: "esse usuário pode ver o tópico desse motorista específico?"
- Trocar permissions no backend não exige releasar versão nova do app

**Em prova:** a resposta da DQ continua sendo **OAuth 2.0 user credentials** porque é assim que o **app mobile** se autentica. Mas saiba que isso é parte de uma arquitetura maior — pode aparecer pergunta detalhando o backend (resposta: SA com role específica de Pub/Sub).

**Matriz mestre — onde o código roda determina a credencial:**

| Onde roda | Credencial |
|---|---|
| Celular/browser, acessa dado de usuário | OAuth 2.0 user credentials |
| Celular/browser, acessa API pública | API key |
| VM/GKE/Cloud Run/Cloud Function | Environment-provided SA (metadata server) |
| Servidor on-premises | Workload Identity Federation (ideal) ou chave JSON (legado) |

**Regra de ouro:** client-side **nunca** embarca secret de longa duração. Sempre token curto via OAuth.

### DQ 06 — App DENTRO do GCP autenticando em service APIs do GCP

**Enunciado:** "You are authenticating an application to service APIs. Both resources are internal to the Google Cloud environment. What type of credentials should you use?"

**Alternativas:**
- A) API keys
- B) User account credentials
- C) **Temporary credentials** ✅
- D) Locally stored keys

**Resposta:** C — tokens OAuth2 curtos (~1h) entregues pelo **metadata server** quando a SA está anexada ao recurso (GCE/GKE/Cloud Run/Cloud Functions/App Engine). Rotação automática, zero secret em disco, ADC busca sozinho.

> **Sinônimo crítico:** "Temporary credentials" nessa questão **é** o que o PDF oficial chama de **"Environment-provided service account"** (linha 3 da matriz dos 4 tipos). Terminologia muda entre Skills Boost / exam guide / blog posts — o conceito é único: metadata server → token de curta duração.

**Por que as outras estão erradas:**

- **D) Locally stored keys (armadilha principal)** — arquivo JSON de SA no disco. A frase "internal to Google Cloud environment" se refere a **onde a APLICAÇÃO roda**, não a onde a chave fica guardada. O próprio Google diz: *"avoid service account keys whenever possible"*. Chaves locais são fallback pra workloads **fora** do GCP (on-prem, AWS, dev laptop) que não falam com o metadata server. Usar dentro do GCP é piorar a segurança de graça: sem auto-rotação, vaza pra sempre se exposta, vira tua responsabilidade armazenar.

- **A) API keys** — só **identificam** o projeto (quota/billing), não **autenticam** principal. APIs públicas tipo Maps/Translate aceitam; APIs Cloud (Compute, Storage, BigQuery, IAM) **rejeitam**. Não dá pra anexar role IAM a uma API key — então não atende "autenticar app a service APIs".

- **B) User account credentials** — identidade de **humano** (tua conta Gmail/Workspace via `gcloud auth login`). Aplicação **nunca** deve usar em produção: (a) atrelada a uma pessoa específica — se ela sai, app quebra; (b) herda *todas* as permissions dela (menor privilégio quebrado); (c) exige fluxo interativo de consentimento. SA existe pra resolver exatamente esse problema.

**Pegadinha de linguagem:** "internal to Google Cloud environment" é a frase-gatilho para a resposta envolver o **metadata server**. Toda vez que aparecer "app rodando dentro do GCP / em GCE / GKE / Cloud Run / Cloud Functions" como contexto, a resposta certa é alguma forma de **credencial efêmera via SA anexada**.

## Labs

### Lab "Exploring IAM" (Skills Boost — 30 min)

Lab compartilhado com o tópico 5.1. Documentação completa em [5.1 — Lab "Exploring IAM"](5.1-iam-roles.md#lab-exploring-iam-skills-boost--30-min).

**Resumo do que esse lab ensina sobre Service Accounts (relevante para o 5.2):**

1. **Anexar SA à VM = VM herda permissions da SA.** Username 1 entra via SSH e os `gcloud` comandos rodam **como** a SA `read-bucket-objects`, não como Username 1. É o mecanismo central do 5.2 funcionando ao vivo.

2. **Service Account User na prática.** O lab concede `roles/iam.serviceAccountUser` sobre a SA para `altostrat.com` (domínio fictício de treino). Quem é membro pode anexar essa SA a VMs e SSH nelas. Combinado com `roles/compute.instanceAdmin.v1` no projeto, vira o pacote completo: pode criar VM + pode escolher a SA.

3. **Access scope filtra na borda, IAM filtra no backend.** O experimento `gcloud compute instances list` falha com **"insufficient authentication scopes"** porque a VM foi criada com scope só de Storage. Mesmo se a SA tivesse role `compute.viewer`, o scope bloquearia antes. Já o experimento de upload falha com **403 AccessDeniedException** porque o IAM da SA não tem `storage.objects.create`. **Duas camadas, dois erros diferentes.**

4. **Como diagnosticar bloqueio:** mensagem **"insufficient authentication scopes"** → arrume **scopes da VM** (parar VM + alterar). Mensagem **"403 AccessDeniedException"** → arrume **role da SA** (live, não precisa parar VM).
