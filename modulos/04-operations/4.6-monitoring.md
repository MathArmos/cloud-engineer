# 4.6 — Monitoring and Logging

> **DQ:** 09 · **Cursos:** Logging and Monitoring in Google Cloud · Observability in Google Cloud

---

## Visão geral — o que é "Cloud Observability"

**Google Cloud Observability** (ex-**Stackdriver**, rebatizado em 2020) é a **suíte unificada de telemetria** do GCP. Não é um produto único — é o guarda-chuva sob o qual vivem Cloud Monitoring, Cloud Logging, Error Reporting, Cloud Trace, Cloud Profiler e Managed Service for Prometheus.

**Os 3 pilares de observability** e onde cada um cai na suíte:

| Pilar | Pergunta que responde | Ferramenta |
|-------|----------------------|------------|
| **Metrics** | Está saudável agora? | **Cloud Monitoring** |
| **Logs** | O que aconteceu naquele momento? | **Cloud Logging** |
| **Traces** | Onde está o gargalo da request distribuída? | **Cloud Trace** |

**Diferenciais centrais** (e o que cada um significa na prática):

1. **Integração profunda GCP + suporte multicloud (AWS)** — serviços gerenciados emitem métricas/logs **sem configuração**. VMs em AWS entram via **Ops Agent + metrics scope**.
2. **Smart defaults** — dashboards básicos (CPU, memória, requests) aparecem prontos em minutos sem desenhar nada.
3. **Exportação para BigQuery / Looker Studio / Pub/Sub** — sem silo, pluga na stack existente.
4. **Pay-per-use** com free tier mensal — cobra por volume de logs ingeridos e métricas customizadas (métricas padrão são gratuitas).

**Por que "tudo numa suíte só" importa.** Em stack tradicional você teria ELK (logs) + Prometheus/Grafana (metrics) + Jaeger (traces) + Sentry (errors) — 4 UIs, 4 contas, correlação manual. No Observability, uma request com erro te leva do log → métrica → trace → exception com **link direto entre eles**. Correlação nativa, não plugin.

> **Armadilha de prova.** "Stackdriver" e "Cloud Observability" são o **mesmo serviço** — material antigo só não foi atualizado.

| Ferramenta | Para que serve |
|-----------|----------------|

## Alert Policies — estrutura completa

```
Alert Policy
├── WHAT: Conditions (até 6 por policy)
│   ├── Resource type (ex: VM instance)
│   ├── Metric (ex: CPU utilization)
│   └── Threshold (ex: > 0.60 por 5 min)
└── HOW: Notification Channels
    ├── Who: email, Slack, Pub/Sub
    ├── How: tipo de canal
    └── Documentation: contexto opcional para o on-call
```

---

## Configuração da Condition — campos críticos

| Campo | Opções | Pegadinha |
|-------|--------|-----------|
| **Resource type** | VM instance, GKE pod, Cloud Run, etc. | — |
| **Metric** | CPU **utilization** | NÃO é "CPU load" |
| **Trigger** | **any** time series / **all** time series | "each instance" = **any** |
| **Condition** | **above** / below threshold | CPU alto = **above** |
| **Threshold** | decimal para % | 60% → **0.60** |
| **Duration** | janela de violação | ex: 5 minutos |

---

## any vs all — diferença crítica

| Trigger | Quando dispara |
|---------|---------------|
| **any** time series | Qualquer **uma** das instâncias viola → alerta |
| **all** time series | Todas as instâncias precisam violar → alerta |

**"If CPU > 60% for each of your instances"** → trigger = **any**

## Cloud Logging — destinos de sink

| Destino | Quando usar |
|---------|-------------|
| **Log Buckets** (default) | Reter no GCP |
| **BigQuery** | Análise SQL, retenção longa |
| **Cloud Storage** | Arquivamento barato |
| **Pub/Sub** | Streaming para sistemas externos / SIEM |
| **On-premises (via Pub/Sub)** | Compliance externo |

```bash
# Criar sink para BigQuery
gcloud logging sinks create SINK_NAME \
  bigquery.googleapis.com/projects/PROJECT/datasets/DATASET \
  --log-filter='resource.type="gce_instance"'
```

## Tipos de logs

| Tipo | O que contém |
|------|-------------|
| **Admin Activity** | Mudanças na configuração de recursos (sempre ativo, sem custo) |
| **Data Access** | Leitura/escrita de dados (opcional, tem custo) |
| **System Event** | Ações do sistema Google |
| **Policy Denied** | Violações de VPC Service Controls |

## Cloud Logging — conceitos de prova

**Retenção padrão: 30 dias.** Depois disso, os logs somem — export via sink é obrigatório para reter mais.

**Destinos de sink e quando usar:**

| Destino | Caso de uso | Comando |
|---------|-------------|---------|
| **Cloud Storage** | Retenção > 30 dias, compliance, custo baixo | `gcloud logging sinks create SINK gs://BUCKET` |
| **BigQuery** | Análise SQL + Looker Studio, forense de rede, top IPs | `gcloud logging sinks create SINK bigquery.googleapis.com/...` |
| **Pub/Sub** | Streaming para SIEM externo (Splunk, Datadog) | `gcloud logging sinks create SINK pubsub.googleapis.com/...` |
| **Log Buckets** | Roteamento interno dentro do GCP | padrão |

**Fluxo Splunk (canônico de prova):**
```
Cloud Logging → sink → Pub/Sub → Dataflow (template) → Splunk
                                  └── pipeline secundário (failsafe)
```
Pub/Sub é o **buffer** que evita perda se o Splunk cair.

**Log-based metrics:** transformam padrão de texto em log em métrica numérica para alertar no Cloud Monitoring.

**Armadilhas de Logging:**

| Pegadinha | Resposta |
|-----------|----------|
| Retenção padrão | **30 dias** |
| Análise SQL de logs | Sink para **BigQuery** |
| Streaming para SIEM externo | Sink para **Pub/Sub** → Dataflow |
| Logs > 30 dias, custo mínimo | Sink para **Cloud Storage** |

## Error Reporting — conceitos de prova

**O quê:** agrega e deduplica stack traces em "classes de erro". 5.000 ocorrências da mesma exception = **1 card** com contador.

**Plataformas:** App Engine (std/flex), Apps Script, Compute Engine, Cloud Run, Cloud Run functions, GKE, **Amazon EC2** (AWS!).

**Linguagens:** Go, Java, .NET, Node.js, PHP, Python, Ruby.

**Alerta:** notificação em tempo real quando aparece um **novo** tipo de erro (não volume maior de erro já conhecido).

**Armadilhas:**

| Pegadinha | Resposta |
|-----------|----------|
| Erros **agrupados** por tipo | **Error Reporting** (não Cloud Logging) |
| Alerta de erro **novo** | Error Reporting (alerta em novo, não em volume) |
| EC2 é suportado? | **Sim** |

---

## Cloud Trace — conceitos de prova

**O quê:** rastreia latência **entre serviços** (distributed tracing). Responde "onde está o gargalo nessa request?".

**Fontes:** App Engine, global external Application LB, apps com Cloud Trace API.

**Diferencial:** detecta degradações de performance **automaticamente** comparando com histórico.

**Quando usar vs outros:**

| Pergunta | Ferramenta |
|----------|-----------|
| "A latência está alta?" | Cloud **Monitoring** (métrica) |
| "Qual serviço é o gargalo?" | Cloud **Trace** (caminho da request) |
| "O que aconteceu nessa request?" | Cloud **Logging** (logs brutos) |

---

## Cloud Profiler — conceitos de prova

**O quê:** analisa quais funções consomem mais **CPU ou memória** em produção, com overhead mínimo (técnicas estatísticas).

**Diferencial:** roda em **todas** as instâncias de produção simultaneamente; sem impacto perceptível; funciona fora do GCP (outras nuvens, on-prem).

**Linguagens:** Java, Go, Node.js, Python.

**Quando usar:** custo de CPU/memória alto sem causa óbvia; comparar perfil antes/depois de deploy.

## Ops Agent — para VMs

- Instala agente na VM para coletar métricas e logs além do padrão
- Coleta: CPU, memória, disco, processos, logs de aplicação
- Substitui o Stackdriver Agent (legado)

```bash
# Instalar Ops Agent via Cloud Shell / SSH na VM
curl -sSO https://dl.google.com/cloudagents/add-google-cloud-ops-agent-repo.sh
sudo bash add-google-cloud-ops-agent-repo.sh --also-install
```

## DQ 09 — Alertar quando CPU > 60% por mais de 5 min em cada instância

**Resposta:** Alert policy com:
- Metric: **CPU utilization** (não load)
- Condition: **above** 0.60
- Duration: 5 minutes
- Trigger: **any** time series

---

## Pegadinhas de prova

| Pegadinha | Resposta |
|-----------|----------|
| "CPU load vs CPU utilization" | **utilization** é a métrica correta no Cloud Monitoring |
| "60% em threshold" | **0.60** (decimal, não 60) |
| "Alerta por instância" | Trigger = **any** time series |
| "Alerta quando todas as instâncias violam" | Trigger = **all** time series |
| "CPU alto = above ou below?" | **above** |
| "Logs de mudança de configuração" | **Admin Activity** logs (sempre ativo, sem custo) |
| "Exportar logs para análise SQL" | Sink para **BigQuery** |
| "Agente para métricas extras em VM" | **Ops Agent** |
