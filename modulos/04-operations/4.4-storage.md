## Cloud Storage — Lifecycle Management

**Fluxo lógico:**
```
Conditions → aplicam-se a → Objects → quando satisfeitas → Actions
```

**Regras:**
- Aplica-se a **objetos atuais E futuros** no bucket
- Object metadata deve casar **TODAS** as condições (AND)
- Objeto que casa **múltiplas regras** → precedência:
  1. **Delete** vence sempre
  2. **SetStorageClass** com **menor preço** vence

---

## Conditions disponíveis

| Condition | Critério |
|-----------|----------|
| `age` | Idade em **dias** |
| `createdBefore` | Antes de uma **data específica** |
| `customTimeBefore` | Antes do custom time do objeto |
| `daysSinceCustomTime` | Dias desde custom time |
| `daysSinceNoncurrent` | Dias desde virou versão noncurrent |
| `isLive` | Versão viva ou noncurrent (requer versionamento) |
| `matchesStorageClass` | Filtra pela storage class **atual** do objeto |
| `noncurrentTimeBefore` | Antes do timestamp de virar noncurrent |
| `numberOfNewerVersions` | Quantidade de versões mais novas |

---

## Actions disponíveis

| Action | O que faz |
|--------|-----------|
| `SetStorageClass` | Move para Nearline / Coldline / Archive |
| `Delete` | Apaga o objeto |
| `AbortIncompleteMultipartUpload` | Cancela uploads incompletos |

---

## Exemplos de regras — como montar

| Cenário | Conditions | Action |
|---------|-----------|--------|
| Downgrade após 365 dias | `age: 365` | `SetStorageClass: COLDLINE` |
| Deletar antes de uma data | `createdBefore: 2025-01-01` | `Delete` |
| Manter só 3 versões | `numberOfNewerVersions: 3` | `Delete` |
| **Standard → Nearline após data X** | `createdBefore: X` + **`matchesStorageClass: STANDARD`** | `SetStorageClass: NEARLINE` |

**Por que precisa de `matchesStorageClass: STANDARD` no último exemplo?**  
Para garantir que a regra só converta objetos que **ainda são Standard**, ignorando os que já foram movidos para Nearline ou outra classe.

---

## Estimativa de custo de storage

| Componente | O que cobram |
|-----------|--------------|
| **Storage** | Por GB/mês por storage class |
| **Retrieval** | Nearline, Coldline, Archive cobram por GB lido |
| **Operations** | Class A (write) custa mais que Class B (read) |
| **Egress** | Saída de dados para a internet |

## Backup e restore — Cloud SQL

```bash
# Criar backup manual
gcloud sql backups create --instance=INSTANCE

# Listar backups
gcloud sql backups list --instance=INSTANCE

# Restaurar
gcloud sql backups restore BACKUP_ID --restore-instance=INSTANCE
```

## DQ 07 — Mover objetos Standard → Nearline após data específica

**Resposta:** usar lifecycle rule com `createdBefore` + `matchesStorageClass: STANDARD` → `SetStorageClass: NEARLINE`

O `matchesStorageClass` é necessário para não tentar converter objetos que já foram movidos (o que geraria erro ou ação desnecessária).

---

## Pegadinhas de prova

| Pegadinha | Resposta |
|-----------|----------|
| "Mover para Nearline após data X" | `createdBefore` + `matchesStorageClass: STANDARD` |
| "Mover após N dias de existência" | `age: N` |
| "Qual action vence quando duas regras casam?" | **Delete** vence; depois menor preço de storage class |
| "Lifecycle se aplica só a objetos futuros?" | ❌ Aplica-se a **atuais e futuros** |
| "Múltiplas conditions em uma regra" | Todas precisam ser **verdadeiras** (AND) |
