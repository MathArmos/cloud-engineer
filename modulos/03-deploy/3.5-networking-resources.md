# 3.5 — Deploying and implementing networking resources

> **Módulo 03, Subseção 3.5.** Implementação concreta de VPC, sub-redes, firewall rules e IPs em VMs.

---

## Conceitos

### Modos de VPC (Default vs Auto vs Custom)

No GCP, ao criar uma rede você escolhe entre três **modos de VPC**. Esses três modos são a resposta à pergunta clássica de prova "What are the three types of networks offered in Google Cloud?".

| Modo | Quem cria as sub-redes? | Ranges | Quando usar |
|------|--------------------------|--------|-------------|
| **Default** | Google cria automaticamente em todo projeto novo | Pré-fixados, uma subnet por região | Sandbox e aprendizado. Em produção, **deletar**. |
| **Auto-mode** | GCP cria automaticamente uma subnet por região | Fixos em `10.128.0.0/9` | Setup rápido, prototipagem, sem se preocupar com IPAM |
| **Custom-mode** | Você define manualmente cada subnet (região + CIDR) | Você escolhe | **Produção séria** — controle total, evita sobreposição com on-prem |

**Por que importa:**
- Default + auto-mode trazem regras de firewall pré-criadas (`default-allow-internal`, `default-allow-ssh`, `default-allow-rdp`, `default-allow-icmp`) que podem ser **inseguras** em produção.
- Custom-mode te força a desenhar IPAM (IP Address Management) consciente, o que é obrigatório quando vai existir interconexão com on-prem (VPN/Interconnect) — você não pode ter sobreposição de CIDR.

**Armadilha de prova — VPC é GLOBAL no GCP:**
A VPC em si é **global** (uma única VPC pode ter sub-redes em qualquer região do mundo). Só as **sub-redes** são regionais. Isso é diferente da AWS, onde VPC é regional. Pegadinha frequente quando o quiz contrasta "zonal / regional / global" como possíveis "tipos de rede" — esses são **escopos de recurso**, não modos de VPC.

**Exemplo prático:**
```bash
# Criar VPC custom-mode
gcloud compute networks create minha-vpc --subnet-mode=custom

# Criar sub-rede dentro dela
gcloud compute networks subnets create web-tier \
  --network=minha-vpc \
  --region=us-central1 \
  --range=10.10.0.0/24
```

### Firewall rules por tag vs por IP

Em GCP, firewall rules podem ser aplicadas usando três critérios principais:
- **Network tags** — labels arbitrários (strings) aplicados a VMs
- **Service accounts** — identidade IAM da VM
- **IP ranges** (CIDR de origem/destino)

**O ponto-chave das tags:**
Quando você associa uma firewall rule a uma tag (ex.: `web-server`), **qualquer VM** criada com essa tag herda a regra automaticamente — **independente do IP** que o GCP atribuir. Isso desacopla a regra do endereçamento.

**Por que importa:**
- IPs em GCP são frequentemente **efêmeros** (mudam quando a VM é stopada e re-iniciada, ou em auto-scaling). Manter allowlists por IP é insustentável.
- Em MIGs (Managed Instance Groups) com auto-scaling, novas VMs nascem e morrem o tempo todo — só funcionam regras por tag (ou por SA).
- A alternativa "service account" é até mais segura (porque tag é só uma string e qualquer um com permissão de editar VM pode adicionar), mas tag é o mecanismo mais comum em quizzes ACE.

**Quando usar IP/CIDR mesmo assim:**
- Quando a origem é **externa** à VPC (allowlist de IPs corporativos vindos de fora)
- Para regras de tráfego entre on-prem ↔ GCP via VPN

**Armadilha de prova:**
Distratores tipicamente sugerem que tags servem para "controlar IPs efêmeros" (não, tag não atribui IP), "billing" (firewall rules em GCP são **gratuitas** — só tráfego é cobrado), ou "network sniffing" (sniffing é captura, sem relação).

**Exemplo prático:**
```bash
# Aplicar tag a uma VM
gcloud compute instances add-tags minha-vm --tags=web-server --zone=us-central1-a

# Firewall rule que aplica a qualquer VM com a tag
gcloud compute firewall-rules create allow-https-web \
  --network=minha-vpc \
  --action=ALLOW \
  --direction=INGRESS \
  --rules=tcp:443 \
  --source-ranges=0.0.0.0/0 \
  --target-tags=web-server
```

### IPs de uma VM (interno obrigatório, externo opcional)

Uma VM no Compute Engine pode ter:

| Tipo de IP | Obrigatório? | Para que serve |
|------------|--------------|----------------|
| **IP interno (RFC 1918)** | **SIM, sempre** | Comunicação dentro da VPC. Atribuído pela sub-rede onde a VM nasce. |
| **IP externo** | **Não, opcional** | Conexões iniciadas **de** ou **para** a internet pública |
| **Alias IP ranges** | Não, opcional | Múltiplos serviços/pods por VM (usado pelo GKE para pods) |

**Por que isso é uma pegadinha clássica:**
O senso comum diz "uma VM precisa de internet para funcionar, logo precisa de IP externo". **Errado.** Em arquiteturas bem desenhadas, a **maioria** das VMs de produção **não tem IP externo**:

- **Acesso a pacotes/atualizações da internet:** roteia saída via **Cloud NAT** (NAT Gateway gerenciado). VM sem IP externo consegue fazer `apt update`, baixar imagens Docker, chamar APIs externas — tudo pela rota Cloud NAT.
- **Acesso administrativo (SSH/RDP):** use **IAP TCP Forwarding** (Identity-Aware Proxy). Você abre um túnel autenticado em vez de expor a porta 22 no IP público.
- **Comunicação entre serviços internos:** acontece via IP interno, sem precisar sair da VPC.

**Armadilha de prova:**
Sempre que a pergunta for *"minimum number of IP addresses a VM needs"*, a resposta é **um (1) — só o interno**. Distratores vão sugerir "dois (interno + externo)" para te pegar no senso comum.

**Quando uma VM realmente precisa de IP externo:**
- Bastion/jump host (mas IAP é melhor)
- VM servindo tráfego público diretamente sem load balancer (raro e geralmente desaconselhado)
- VM precisando ser alvo de tráfego inbound de origens fora do GCP que não passam por LB

**Exemplo prático:**
```bash
# VM sem IP externo (padrão de produção)
gcloud compute instances create vm-privada \
  --machine-type=e2-medium \
  --subnet=web-tier \
  --no-address \
  --zone=us-central1-a
```

A flag `--no-address` é a que importa: ela cria a VM **sem** IP externo. Em ambiente de produção, esse deveria ser o padrão e o IP externo a exceção.

### Cloud VPN — conectividade híbrida (on-premises ↔ GCP)

O Cloud VPN cria um túnel IPsec criptografado entre sua rede on-premises e uma VPC no GCP, permitindo comunicação segura pela internet pública. Existem dois tipos de gateway de Cloud VPN.

---

#### Classic VPN

**O que é:** gateway VPN regional que conecta on-prem à VPC via IPsec. Todo gateway criado antes da introdução do HA VPN é Classic VPN.

**Características principais:**

| Característica | Valor |
|---|---|
| SLA | **99.9%** de disponibilidade |
| Roteamento suportado | Estático e dinâmico (via Cloud Router / BGP) |
| Protocolos de cifra | IKEv1 e IKEv2 |
| Escopo do gateway | **Regional** (IP externo regional) |
| MTU máximo on-prem | **1460 bytes** (por causa da criptografia/encapsulamento) |

**Por que dois túneis?**
Para conectar dois gateways VPN você precisa criar **2 túneis** — cada um define a conexão da perspectiva do seu próprio gateway. Tráfego só passa quando o par de túneis está estabelecido.

**O que Classic VPN NÃO suporta:**
- "Dial-in" de computadores cliente com software VPN (client VPN / remote-access VPN). É exclusivamente **site-to-site**.

**Armadilha de prova:** Classic VPN é "useful for low-volume data connections" — baixo volume porque é via internet pública, sem a performance da rede privada do Google.

---

#### HA VPN (High Availability VPN)

**O que é:** solução de VPN de alta disponibilidade numa única região, com SLA superior ao Classic VPN.

**Diferenças-chave em relação ao Classic VPN:**

| Aspecto | Classic VPN | HA VPN |
|---|---|---|
| SLA | 99.9% | **99.99%** |
| Interfaces do gateway | 1 | **2** (fixo) |
| IPs externos | 1 | **2** (um por interface, escolhidos automaticamente de pools separados) |
| Roteamento | Estático ou BGP | **Somente BGP (dinâmico)** |
| Tunnels para garantir SLA | N/A | **2 ou 4 túneis** |

**Como atingir 99.99% com HA VPN:** você precisa configurar 2 ou 4 túneis do gateway HA VPN para o peer VPN (ou para outro gateway HA VPN). Com apenas 1 interface ativa você perde o SLA de 99.99%.

**Roteamento ativo/ativo vs ativo/passivo:**
Dependendo das prioridades de rota configuradas nos túneis, você obtém:
- **Ativo/ativo:** ambos os túneis carregam tráfego simultaneamente (mais throughput)
- **Ativo/passivo:** um túnel fica em standby, assume quando o primário falha (failover)

---

#### Topologias suportadas pelo HA VPN

**Topologia 1 — HA VPN gateway para dois dispositivos peer separados (`TWO_IPS_REDUNDANCY`):**

```
[HA VPN gateway GCP]          [Peer device A]  ← 1 IP
  interface 0  ──── túnel 1 ─→
  interface 1  ──── túnel 2 ─→ [Peer device B]  ← 1 IP
```

- Cada peer tem 1 interface e 1 IP externo
- Redundância no lado on-prem: se um dispositivo físico cair (ou precisar de manutenção), o outro assume
- Garante **99.99% SLA**

**Topologia 2 — HA VPN gateway para AWS:**
- Funciona com AWS transit gateway (suporta **ECMP — Equal-Cost Multipath**) ou virtual private gateway
- ECMP distribui tráfego igualmente entre os túneis ativos
- Configuração total de **4 túneis**: 2 da AWS VPG-A para interface 0 do HA VPN + 2 da AWS VPG-B para interface 1 do HA VPN

**Topologia 3 — HA VPN gateway para outro HA VPN gateway (VPC ↔ VPC):**
- Conecta duas VPCs do GCP entre si via VPN
- Interface 0 de um gateway → interface 0 do outro; interface 1 → interface 1
- Garante **99.99% SLA**

---

#### Cloud Router e BGP (roteamento dinâmico)

**Por que existe:** com roteamento estático, adicionar uma subnet nova exige reconfiguração manual dos túneis. Cloud Router resolve isso com BGP.

**O que é Cloud Router:** recurso gerenciado que mantém uma sessão BGP (Border Gateway Protocol) entre a VPC e o gateway VPN on-premises, trocando rotas dinamicamente.

**Vantagem prática:** novas subnets são **anunciadas automaticamente** entre as redes. Você cria a subnet "Staging" no GCP e ela já fica acessível para o on-premises sem tocar na configuração do túnel.

**Detalhe técnico dos IPs de BGP:**
- A sessão BGP requer um IP em cada ponta do túnel
- Esses IPs devem ser **link-local**: faixa `169.254.0.0/16`
- Eles **não fazem parte** do espaço de endereçamento de nenhuma das redes — existem exclusivamente para estabelecer a sessão BGP
- Armadilha: `169.254.x.x` pode parecer estranho numa pergunta, mas é o range correto e obrigatório para BGP em Cloud VPN

**HA VPN obriga BGP:** ao contrário do Classic VPN (que aceita rotas estáticas), HA VPN só funciona com roteamento dinâmico via Cloud Router.

---

#### Resumo comparativo para prova

| Critério | Classic VPN | HA VPN |
|---|---|---|
| SLA | 99.9% | **99.99%** |
| Interfaces | 1 | **2** |
| Roteamento | Estático ou BGP | **Só BGP** |
| Tunnels mínimos pro SLA | 2 | **2 ou 4** |
| IP externo | Regional, 1 | 2, de pools únicos |
| Caso de uso | Baixo volume, conexão simples | Produção, alta disponibilidade |

### Cloud Interconnect e Peering — visão geral

Além do Cloud VPN (que usa a internet pública com criptografia), o GCP oferece serviços de conectividade dedicada ou compartilhada com a rede do Google. A lógica de classificação é uma grade de **2 eixos**:

**Eixo 1 — Tipo de conexão:**

| | Dedicated (Dedicada) | Shared (Compartilhada) |
|---|---|---|
| **Definição** | Conexão física direta com a rede do Google | Conexão via parceiro/provedor |
| **Serviços** | Dedicated Interconnect | Partner Interconnect, Carrier Peering |

**Eixo 2 — Camada OSI:**

| | Layer 2 | Layer 3 |
|---|---|---|
| **Como funciona** | VLAN conecta diretamente ao ambiente GCP | Roteamento via IPs públicos |
| **Acessa** | IPs internos RFC 1918 (privados) | Google Workspace, YouTube, Cloud APIs (IPs públicos) |
| **Serviços** | Dedicated Interconnect, Partner Interconnect L2 | Direct Peering, Carrier Peering, Partner Interconnect L3 |

**Os quatro serviços:**

| Serviço | Tipo | Camada | O que oferece |
|---|---|---|---|
| **Dedicated Interconnect** | Dedicated | L2 | Conexão física direta, alta capacidade, IPs RFC 1918 |
| **Partner Interconnect** | Shared | L2 ou L3 | Via parceiro, mais flexível geograficamente |
| **Direct Peering** | Dedicated | L3 | Troca de rotas direta com o Google, acesso a serviços públicos |
| **Carrier Peering** | Shared | L3 | Acesso a serviços públicos do Google via provedor parceiro |

**Posição do Cloud VPN nesse contexto:**
Cloud VPN usa a internet pública com criptografia IPsec — está fora dessa grade de Interconnect/Peering. É um complemento útil para Direct Peering e Carrier Peering porque esses dois são Layer 3 (só acessam IPs públicos), enquanto Cloud VPN adiciona acesso a **IPs internos RFC 1918** sobre a internet pública.

**Segunda forma de organizar os serviços — Interconnect vs Peering:**

| | Interconnect services | Peering services |
|---|---|---|
| **Acesso** | IPs RFC 1918 (privados) | IPs públicos do Google |
| **SLA** | **Sim** | **Não** |
| **Serviços** | Dedicated Interconnect, Partner Interconnect, Cross-Cloud Interconnect | Direct Peering, Carrier Peering |

---

#### Dedicated Interconnect

**O que é:** conexão física direta entre seu datacenter e a rede do Google em uma **colocation facility** compartilhada. Você provisiona um cross-connect entre o roteador do Google e o seu, e configura BGP via Cloud Router para trocar rotas.

**Capacidade:**
- **10 Gbps ou 100 Gbps por link**
- Até 8 links de 10 Gbps (máximo 80 Gbps) ou até 2 links de 100 Gbps (máximo 200 Gbps)
- Mínimo: 10 Gbps (não existe opção abaixo disso)

**SLA:** 99.9% ou 99.99% dependendo da configuração de redundância.

**Requisito crítico:** seu datacenter precisa estar fisicamente próximo (ou dentro) de uma **colocation facility suportada pelo Google**. Se não estiver, use Partner Interconnect.

**Quando usar:** grandes volumes de dados, tráfego previsível e consistente, requisito de compliance que proíbe tráfego pela internet pública.

---

#### Partner Interconnect

**O que é:** conectividade entre seu datacenter e a VPC via um **service provider parceiro** que já tem conexão física com a rede do Google.

**Capacidade:** **50 Mbps a 50 Gbps** por conexão (varia por provedor).

**SLA:** 99.9% ou 99.99% entre Google e o service provider.

**Quando usar:**
- Seu datacenter não está perto de uma colocation facility do Google
- Sua necessidade de bandwidth não justifica um Dedicated Interconnect (que começa em 10 Gbps)
- Você quer o parceiro gerenciando a infraestrutura de conexão

**Layer 2 vs Layer 3 no Partner Interconnect:**
- **L2:** você mesmo configura o BGP peering entre Cloud Router e roteador on-prem (mais controle)
- **L3:** o parceiro gerencia o BGP por você (mais simples, menos controle)

---

#### Cross-Cloud Interconnect

**O que é:** conexão física dedicada de alta banda entre o Google Cloud e outro cloud provider. O Google provisiona e gerencia a conexão até o ponto de encontro com a rede do outro provider.

**Clouds suportadas:** AWS, Microsoft Azure, Oracle Cloud Infrastructure, Alibaba Cloud.

**Capacidade:** **10 Gbps ou 100 Gbps** (dois tamanhos apenas).

**Modelo de compra:** você compra portas primárias e redundantes no Google e no outro cloud provider.

**Limitação:** Google garante a conexão até a borda da outra nuvem. O que acontece dentro da outra nuvem não tem SLA do Google — você precisa abrir chamado no outro provider.

**Quando usar:** workloads multi-cloud que precisam de alta banda e roteamento gerenciado pelo Google entre nuvens.

---

#### Direct Peering

**O que é:** troca de rotas BGP diretamente com a rede do Google num **Edge PoP (Point of Presence)** do Google. Não passa por colocation facility — é peering de internet tradicional.

**Capacidade:** **10 Gbps por link**.

**SLA:** **nenhum** — é peering de internet, não um produto gerenciado.

**Acessa:** todos os serviços Google (Workspace, YouTube, Cloud APIs) via IPs públicos.

**Requisito:** você precisa satisfazer os requisitos de peering do Google (ter presença em um PoP do Google, volume de tráfego suficiente, etc.). O Google tem PoPs em mais de 90 internet exchanges e 100+ colocation facilities no mundo.

**Quando usar:** você quer conectividade de baixa latência com serviços públicos do Google (não RFC 1918) e consegue se encontrar com o Google num PoP.

---

#### Carrier Peering

**O que é:** acesso aos serviços públicos do Google via um **carrier parceiro** que já tem peering com o Google. Para quando você não satisfaz os requisitos de Direct Peering ou não está perto de um PoP.

**Capacidade e SLA:** variam conforme o provedor — **sem SLA do Google**.

**Quando usar:** precisa de acesso a Google Workspace, YouTube ou APIs públicas, mas não atende os requisitos de Direct Peering.

---

#### Tabela comparativa de capacidade e requisitos

| Serviço | Capacidade | SLA | Acessa | Requisito principal |
|---|---|---|---|---|
| **Cloud VPN** | 1.5–3 Gbps/túnel | 99.9% (Classic) / 99.99% (HA) | RFC 1918 | Dispositivo VPN on-prem |
| **Dedicated Interconnect** | 10 ou 100 Gbps/link | 99.9% ou 99.99% | RFC 1918 | Presença em colocation facility |
| **Partner Interconnect** | 50 Mbps – 50 Gbps | 99.9% ou 99.99% | RFC 1918 | Contrato com service provider |
| **Cross-Cloud Interconnect** | 10 ou 100 Gbps | Parcial (até a borda do outro cloud) | RFC 1918 | Portas no Google + outro cloud |
| **Direct Peering** | 10 Gbps/link | **Nenhum** | IPs públicos | Presença em PoP do Google |
| **Carrier Peering** | Varia | **Nenhum** | IPs públicos | Contrato com carrier |

> Google recomenda **Cloud Interconnect (Dedicated ou Partner) em vez de Direct/Carrier Peering** sempre que possível — Interconnect tem SLA e acessa RFC 1918.

---

#### Árvore de decisão — qual serviço escolher

```
Precisa acessar Workspace, YouTube ou Cloud APIs (IPs públicos)?
├── SIM → escolha um Peering service
│         ├── Atende requisitos de Direct Peering? → Direct Peering
│         └── Não atende → Carrier Peering
│
└── NÃO → quer conectar a outro cloud provider?
          ├── SIM → Cross-Cloud Interconnect
          │         (alta banda + roteamento gerenciado pelo Google)
          │         ou Cloud VPN (sem requisito de alta banda, criptografia Google)
          │
          └── NÃO → quer estender VPC para on-prem (RFC 1918)?
                    ├── Consegue chegar a uma colocation facility do Google?
                    │   ├── NÃO → Cloud VPN ou Partner Interconnect
                    │   │         ├── Baixa banda, curta duração, precisa de criptografia → Cloud VPN
                    │   │         └── Caso contrário → Partner Interconnect
                    │   │                              ├── Precisa de BGP próprio → L2
                    │   │                              └── Não precisa → L3
                    │   │
                    │   └── SIM → Dedicated Interconnect
                    │             Mas considere Cloud VPN ou Partner Interconnect se:
                    │             - Não pode prover criptografia própria para tráfego sensível
                    │             - 10 Gbps é grande demais para sua necessidade
                    │             - Precisa de acesso a múltiplos cloud providers
                    │
                    └── Quer criptografia gerenciada pelo Google sobre Interconnect?
                        → Cloud VPN over Interconnect
```

**Atalho mental para prova:**

| Cenário | Escolha |
|---|---|
| Baixo custo, baixa banda, teste/migração, criptografia | Cloud VPN |
| Enterprise, alta banda, on-prem → GCP | Dedicated ou Partner Interconnect |
| Multi-cloud, alta banda | Cross-Cloud Interconnect |
| Acesso a serviços públicos Google | Direct Peering ou Carrier Peering |
| Colocation disponível + >10 Gbps | Dedicated Interconnect |
| Sem colocation ou <10 Gbps | Partner Interconnect |

### Shared VPC e VPC Network Peering — compartilhamento de redes entre projetos

Duas formas de conectar projetos GCP diferentes via rede privada, com modelos de administração opostos.

---

#### Shared VPC

**O que é:** uma VPC de um projeto (o **host project**) é compartilhada com outros projetos (**service projects**). Os recursos dos service projects usam subnets da VPC do host project, mas a rede em si é controlada centralmente.

**Estrutura:**

```
[Organization]
    ├── Host Project  ← dona da VPC compartilhada (subnets, routes, firewall)
    │       └── Shared VPC Network
    ├── Service Project A  ← usa subnets do host project
    └── Service Project B  ← usa subnets do host project
```

**Por que usar:**
- Controle centralizado de rede (firewalls, rotas, subnets) pelo time de networking
- Times de aplicação (service projects) criam e gerenciam instâncias sem tocar na rede
- Separação de responsabilidades: **Network Admin** no host project, **Instance Admin** nos service projects

**Limitações importantes:**
- Só funciona **dentro da mesma organização**
- Só funciona **entre projetos diferentes** (não faz sentido dentro do mesmo projeto)
- É uma abordagem **centralizada**: política de rede num único lugar

---

#### VPC Network Peering

**O que é:** conectividade privada RFC 1918 entre duas VPCs independentes, sem compartilhar a rede em si. Cada VPC mantém seus próprios firewalls e tabelas de rota.

**Como funciona:**
- O Network Admin da VPC A faz peering com a VPC B
- O Network Admin da VPC B faz peering com a VPC A
- Quando **ambos os lados** criam o peering, a sessão fica **Active** e rotas são trocadas
- VMs nas duas VPCs passam a se comunicar via IPs internos

**Por que dois lados precisam aceitar:**
Assim como uma conexão TCP (handshake), o peering exige consentimento mútuo — garante que nenhuma VPC seja "adicionada" a um peering sem que o admin saiba.

**Vantagem sobre VPN e IPs externos:**
VPC Network Peering não tem a latência, risco de segurança e custo que surgem ao usar IPs externos ou VPN para conectar VPCs. Tráfego fica dentro da rede Google.

**Onde se aplica:**
- Projetos em **organizações diferentes** (ex.: consumidor e produtor de serviço)
- Projetos na **mesma organização** quando se quer administração descentralizada
- Até mesmo dentro do **mesmo projeto** (duas VPCs distintas)

---

#### Comparativo Shared VPC vs VPC Network Peering

| Critério | Shared VPC | VPC Network Peering |
|---|---|---|
| **Modelo de administração** | **Centralizado** — rede controlada pelo host project | **Descentralizado** — cada VPC mantém seus firewalls e rotas |
| **Escopo** | Somente dentro da **mesma organização** | **Mesma ou diferentes organizações** |
| **Nível** | Entre **projetos diferentes** | Entre projetos diferentes ou **mesmo projeto** |
| **Rede compartilhada?** | Sim — service projects usam subnets do host | Não — cada VPC é independente |
| **Quando usar** | Controle centralizado de rede, múltiplos times de app | Comunicação privada entre orgs distintas, administração separada |

**Armadilha de prova:**
- "Organizações diferentes" → **obrigatoriamente VPC Peering** (Shared VPC não alcança)
- "Controle centralizado de rede" → **Shared VPC**
- "Cada time administra sua própria rede" → **VPC Network Peering**

- Marquei: B ou A (incerto) ⚠️ — A é a correta

## Labs

### Lab — Configuring Google Cloud HA VPN

**Objetivo do Google:** demonstrar na prática como configurar HA VPN entre duas VPCs (uma simulando on-premises) com BGP dinâmico e verificar o failover automático de túnel.

**Arquitetura do lab:**

```
[vpc-demo]                          [on-prem]
  subnet1: 10.1.1.0/24 (us-west1)    subnet1: 192.168.1.0/24 (us-west1)
  subnet2: 10.2.1.0/24 (us-central1)
  vpc-demo-instance1 (us-west1-b)   on-prem-instance1 (us-west1-b)
  vpc-demo-instance2 (us-central1-a)

  vpc-demo-vpn-gw1 ──tunnel0──  on-prem-vpn-gw1
  (interface 0: IP-A)             (interface 0: IP-C)
  (interface 1: IP-B)  ──tunnel1── (interface 1: IP-D)

  vpc-demo-router1 (ASN 65001) ←BGP→ on-prem-router1 (ASN 65002)
```

---

**Task 1 — Criar vpc-demo (VPC global com 2 subnets):**

```bash
gcloud compute networks create vpc-demo --subnet-mode custom

gcloud compute networks subnets create vpc-demo-subnet1 \
  --network vpc-demo --range 10.1.1.0/24 --region us-west1

gcloud compute networks subnets create vpc-demo-subnet2 \
  --network vpc-demo --range 10.2.1.0/24 --region us-central1

# Firewall: tráfego interno (qualquer porta TCP/UDP/ICMP dentro de 10.0.0.0/8)
gcloud compute firewall-rules create vpc-demo-allow-custom \
  --network vpc-demo --allow tcp:0-65535,udp:0-65535,icmp \
  --source-ranges 10.0.0.0/8

# Firewall: SSH e ICMP de qualquer origem
gcloud compute firewall-rules create vpc-demo-allow-ssh-icmp \
  --network vpc-demo --allow tcp:22,icmp

gcloud compute instances create vpc-demo-instance1 \
  --machine-type=e2-medium --zone us-west1-b --subnet vpc-demo-subnet1

gcloud compute instances create vpc-demo-instance2 \
  --machine-type=e2-medium --zone us-central1-a --subnet vpc-demo-subnet2
```

---

**Task 2 — Criar on-prem (VPC que simula datacenter on-premises):**

```bash
gcloud compute networks create on-prem --subnet-mode custom

gcloud compute networks subnets create on-prem-subnet1 \
  --network on-prem --range 192.168.1.0/24 --region us-west1

gcloud compute firewall-rules create on-prem-allow-custom \
  --network on-prem --allow tcp:0-65535,udp:0-65535,icmp \
  --source-ranges 192.168.0.0/16

gcloud compute firewall-rules create on-prem-allow-ssh-icmp \
  --network on-prem --allow tcp:22,icmp

gcloud compute instances create on-prem-instance1 \
  --machine-type=e2-medium --zone us-west1-b --subnet on-prem-subnet1
```

---

**Task 3 — Criar gateways HA VPN e Cloud Routers:**

```bash
# Gateways HA VPN (cada um ganha 2 IPs automaticamente de pools distintos)
gcloud compute vpn-gateways create vpc-demo-vpn-gw1 --network vpc-demo --region us-west1
gcloud compute vpn-gateways create on-prem-vpn-gw1  --network on-prem  --region us-west1

# Verificar (confirmar que interface0 e interface1 têm IPs distintos)
gcloud compute vpn-gateways describe vpc-demo-vpn-gw1 --region us-west1
gcloud compute vpn-gateways describe on-prem-vpn-gw1  --region us-west1

# Cloud Routers com ASNs diferentes (obrigatório para BGP entre peers)
gcloud compute routers create vpc-demo-router1 \
  --region us-west1 --network vpc-demo --asn 65001

gcloud compute routers create on-prem-router1 \
  --region us-west1 --network on-prem --asn 65002
```

> Por que ASNs diferentes? BGP requer que cada peer tenha um número de sistema autônomo único. 65001 e 65002 estão na faixa privada (64512–65534), análoga aos IPs RFC 1918.

---

**Task 4 — Criar os 4 túneis VPN (interface0 ↔ interface0, interface1 ↔ interface1):**

```bash
# vpc-demo: tunnel0 sai da interface 0, aponta para on-prem-vpn-gw1
gcloud compute vpn-tunnels create vpc-demo-tunnel0 \
  --peer-gcp-gateway on-prem-vpn-gw1 --region us-west1 \
  --ike-version 2 --shared-secret [SHARED_SECRET] \
  --router vpc-demo-router1 --vpn-gateway vpc-demo-vpn-gw1 --interface 0

# vpc-demo: tunnel1 sai da interface 1
gcloud compute vpn-tunnels create vpc-demo-tunnel1 \
  --peer-gcp-gateway on-prem-vpn-gw1 --region us-west1 \
  --ike-version 2 --shared-secret [SHARED_SECRET] \
  --router vpc-demo-router1 --vpn-gateway vpc-demo-vpn-gw1 --interface 1

# on-prem: tunnel0 sai da interface 0, aponta para vpc-demo-vpn-gw1
gcloud compute vpn-tunnels create on-prem-tunnel0 \
  --peer-gcp-gateway vpc-demo-vpn-gw1 --region us-west1 \
  --ike-version 2 --shared-secret [SHARED_SECRET] \
  --router on-prem-router1 --vpn-gateway on-prem-vpn-gw1 --interface 0

# on-prem: tunnel1 sai da interface 1
gcloud compute vpn-tunnels create on-prem-tunnel1 \
  --peer-gcp-gateway vpc-demo-vpn-gw1 --region us-west1 \
  --ike-version 2 --shared-secret [SHARED_SECRET] \
  --router on-prem-router1 --vpn-gateway on-prem-vpn-gw1 --interface 1
```

> Regra crítica: interface0 de um gateway deve sempre conectar à interface0 do peer, e interface1 à interface1. Cruzar interfaces quebra o SLA de 99.99%.

---

**Task 5 — Configurar BGP peering em cada túnel:**

```bash
# --- vpc-demo-router1: peering para os dois túneis ---

# Interface BGP do tunnel0 (IP link-local do lado vpc-demo)
gcloud compute routers add-interface vpc-demo-router1 \
  --interface-name if-tunnel0-to-on-prem \
  --ip-address 169.254.0.1 --mask-length 30 \
  --vpn-tunnel vpc-demo-tunnel0 --region us-west1

# Peer BGP do tunnel0 (IP link-local do lado on-prem)
gcloud compute routers add-bgp-peer vpc-demo-router1 \
  --peer-name bgp-on-prem-tunnel0 \
  --interface if-tunnel0-to-on-prem \
  --peer-ip-address 169.254.0.2 --peer-asn 65002 --region us-west1

# Interface BGP do tunnel1
gcloud compute routers add-interface vpc-demo-router1 \
  --interface-name if-tunnel1-to-on-prem \
  --ip-address 169.254.1.1 --mask-length 30 \
  --vpn-tunnel vpc-demo-tunnel1 --region us-west1

# Peer BGP do tunnel1
gcloud compute routers add-bgp-peer vpc-demo-router1 \
  --peer-name bgp-on-prem-tunnel1 \
  --interface if-tunnel1-to-on-prem \
  --peer-ip-address 169.254.1.2 --peer-asn 65002 --region us-west1

# --- on-prem-router1: peering para os dois túneis (IPs espelhados) ---

gcloud compute routers add-interface on-prem-router1 \
  --interface-name if-tunnel0-to-vpc-demo \
  --ip-address 169.254.0.2 --mask-length 30 \
  --vpn-tunnel on-prem-tunnel0 --region us-west1

gcloud compute routers add-bgp-peer on-prem-router1 \
  --peer-name bgp-vpc-demo-tunnel0 \
  --interface if-tunnel0-to-vpc-demo \
  --peer-ip-address 169.254.0.1 --peer-asn 65001 --region us-west1

gcloud compute routers add-interface on-prem-router1 \
  --interface-name if-tunnel1-to-vpc-demo \
  --ip-address 169.254.1.2 --mask-length 30 \
  --vpn-tunnel on-prem-tunnel1 --region us-west1

gcloud compute routers add-bgp-peer on-prem-router1 \
  --peer-name bgp-vpc-demo-tunnel1 \
  --interface if-tunnel1-to-vpc-demo \
  --peer-ip-address 169.254.1.1 --peer-asn 65001 --region us-west1
```

> Padrão dos IPs link-local: tunnel0 usa `169.254.0.1/30` (vpc-demo) ↔ `169.254.0.2/30` (on-prem). Tunnel1 usa `169.254.1.1/30` ↔ `169.254.1.2/30`. O `/30` dá exatamente 2 IPs utilizáveis por par — suficiente para uma sessão BGP ponto-a-ponto.

**Task 6 — Verificar, liberar firewall e testar conectividade:**

```bash
# Inspecionar routers (conferir bgpPeers, interfaces e ASNs)
gcloud compute routers describe vpc-demo-router1 --region us-west1
gcloud compute routers describe on-prem-router1  --region us-west1

# Firewall: permitir tráfego entre as redes via VPN
gcloud compute firewall-rules create vpc-demo-allow-subnets-from-on-prem \
  --network vpc-demo --allow tcp,udp,icmp --source-ranges 192.168.1.0/24

gcloud compute firewall-rules create on-prem-allow-subnets-from-vpc-demo \
  --network on-prem --allow tcp,udp,icmp \
  --source-ranges 10.1.1.0/24,10.2.1.0/24

# Verificar status dos 4 túneis (esperar "Tunnel is up and running")
gcloud compute vpn-tunnels list
gcloud compute vpn-tunnels describe vpc-demo-tunnel0 --region us-west1
gcloud compute vpn-tunnels describe vpc-demo-tunnel1 --region us-west1
gcloud compute vpn-tunnels describe on-prem-tunnel0  --region us-west1
gcloud compute vpn-tunnels describe on-prem-tunnel1  --region us-west1

# SSH na instância on-prem e testar ping para vpc-demo-instance1
gcloud compute ssh on-prem-instance1 --zone us-west1-b
ping -c 4 10.1.1.2   # deve funcionar via tunnel
```

---

**Global routing mode — alcançar subnets de outras regiões:**

Por padrão, Cloud Router só anuncia/vê rotas da **região onde está implantado**. Para `on-prem-instance1` (us-west1) alcançar `vpc-demo-instance2` (us-central1), é preciso ativar roteamento global na VPC.

```bash
# Mudar BGP routing mode de REGIONAL para GLOBAL
gcloud compute networks update vpc-demo --bgp-routing-mode GLOBAL

# Verificar
gcloud compute networks describe vpc-demo
# routingConfig.routingMode deve mostrar GLOBAL

# Testar: de on-prem-instance1, pingar instância em us-central1
ping -c 2 10.2.1.2   # agora funciona
```

> `REGIONAL` (padrão): Cloud Router só vê/anuncia subnets da sua região. `GLOBAL`: Cloud Router vê todas as subnets da VPC, independente da região — permite que a rota de `10.2.1.0/24` seja anunciada para o on-prem via BGP mesmo estando em us-central1.

---

**Task 7 — Testar failover (prova do 99.99% SLA):**

```bash
# Derrubar tunnel0 do vpc-demo
gcloud compute vpn-tunnels delete vpc-demo-tunnel0 --region us-west1 --quiet

# on-prem-tunnel0 vai para estado "Handshake with peer broken"
gcloud compute vpn-tunnels describe on-prem-tunnel0 --region us-west1
# detailedStatus: Handshake with peer broken for unknown reason. Trying again soon.

# Mesmo assim, ping de on-prem-instance1 para vpc-demo-instance1 continua funcionando
ping -c 3 10.1.1.2
# 0% packet loss — tráfego migrou automaticamente para tunnel1
```

> Isso é o 99.99% SLA na prática: um túnel cai, o BGP reconverge e o tráfego flui pelo túnel sobrevivente sem intervenção manual.

---

**Observações do lab:**

- Os dois gateways HA VPN foram criados na **mesma região** (us-west1) porque o lab simula on-prem dentro do GCP. Em produção real, o gateway on-prem seria um dispositivo físico com IP público.
- O `--shared-secret` é a chave pré-compartilhada IKE. Em produção, use um segredo forte gerado aleatoriamente (não o default do console).
- `gcloud compute vpn-tunnels list` é o comando mais rápido para ver o status geral dos 4 túneis de uma vez.
- Status `ESTABLISHED` no campo `status` + `"Tunnel is up and running"` no `detailedStatus` = túnel saudável.

## Cenário real

**No trabalho como cloud engineer, você vai:**
- **Sempre** criar VPCs em **custom-mode** para projetos sérios. Auto-mode só em sandbox.
- Desenhar IPAM **antes** de criar qualquer subnet (alinhar com on-prem se houver VPN/Interconnect).
- Aplicar firewall rules **por tag ou service account**, nunca por IP da VM. IP só para origens externas fixas (ex.: range da matriz corporativa).
- Criar VMs **sem IP externo por padrão**. Cloud NAT para saída, IAP para admin. IP externo só quando há justificativa explícita (e mesmo assim, geralmente preferir LB na frente).

**Conversa típica de feira:**
*"No projeto X a gente saiu da default network e migrou pra VPC custom-mode porque tinha sobreposição de range com o on-prem. A gente parou de usar IP-based firewall e migrou tudo pra network tags + service accounts, o que destravou o auto-scaling — antes a gente atualizava allowlist manualmente cada vez que um IP mudava."*

---

## Armadilhas de prova

1. **"VM precisa de 2 IPs"** → não. Precisa de **1**, o interno.
2. **VPC zonal/regional** → não. VPC é **global**, só sub-rede é regional.
3. **Tags servem para billing/IP/sniffing** → não. Servem para **agrupar VMs e aplicar firewall independente do IP**.
4. **Firewall rules têm custo** → não. Regras são gratuitas, **só tráfego é cobrado** (egress).
5. **Default network é segura** → não. Tem regras `allow-ssh` e `allow-rdp` de `0.0.0.0/0` por padrão. Deletar em produção.
