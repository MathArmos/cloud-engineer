# 3.5 — Deploying and implementing networking resources

> **Módulo 03, Subseção 3.5.** Implementação concreta de VPC, sub-redes, firewall rules e IPs em VMs.

---

## Conceitos

### Modos de VPC (Default vs Auto vs Custom)

No GCP, ao criar uma rede você escolhe entre três **modos de VPC**. Esses três modos são a resposta à pergunta clássica de prova "What are the three types of networks offered in Google Cloud?".

| Modo | Quem cria as sub-redes? | Ranges | Quando usar |
|------|--------------------------|--------|-------------|
| **Default** | Google cria automaticamente em todo projeto novo | Pré-fixados, uma subnet por região | Sandbox e aprendizado. Em produção, **deletar**. |
| **Auto-mode** | GCP cria automaticamente uma subnet por região | Fixos em `10.128.0.0/9` | Setup rápido, prototipagem, sem se preocupar com IPAM |
| **Custom-mode** | Você define manualmente cada subnet (região + CIDR) | Você escolhe | **Produção séria** — controle total, evita sobreposição com on-prem |

**Por que importa:**
- Default + auto-mode trazem regras de firewall pré-criadas (`default-allow-internal`, `default-allow-ssh`, `default-allow-rdp`, `default-allow-icmp`) que podem ser **inseguras** em produção.
- Custom-mode te força a desenhar IPAM (IP Address Management) consciente, o que é obrigatório quando vai existir interconexão com on-prem (VPN/Interconnect) — você não pode ter sobreposição de CIDR.

**Armadilha de prova — VPC é GLOBAL no GCP:**
A VPC em si é **global** (uma única VPC pode ter sub-redes em qualquer região do mundo). Só as **sub-redes** são regionais. Isso é diferente da AWS, onde VPC é regional. Pegadinha frequente quando o quiz contrasta "zonal / regional / global" como possíveis "tipos de rede" — esses são **escopos de recurso**, não modos de VPC.

**Exemplo prático:**
```bash
# Criar VPC custom-mode
gcloud compute networks create minha-vpc --subnet-mode=custom

# Criar sub-rede dentro dela
gcloud compute networks subnets create web-tier \
  --network=minha-vpc \
  --region=us-central1 \
  --range=10.10.0.0/24
```

### Firewall rules por tag vs por IP

Em GCP, firewall rules podem ser aplicadas usando três critérios principais:
- **Network tags** — labels arbitrários (strings) aplicados a VMs
- **Service accounts** — identidade IAM da VM
- **IP ranges** (CIDR de origem/destino)

**O ponto-chave das tags:**
Quando você associa uma firewall rule a uma tag (ex.: `web-server`), **qualquer VM** criada com essa tag herda a regra automaticamente — **independente do IP** que o GCP atribuir. Isso desacopla a regra do endereçamento.

**Por que importa:**
- IPs em GCP são frequentemente **efêmeros** (mudam quando a VM é stopada e re-iniciada, ou em auto-scaling). Manter allowlists por IP é insustentável.
- Em MIGs (Managed Instance Groups) com auto-scaling, novas VMs nascem e morrem o tempo todo — só funcionam regras por tag (ou por SA).
- A alternativa "service account" é até mais segura (porque tag é só uma string e qualquer um com permissão de editar VM pode adicionar), mas tag é o mecanismo mais comum em quizzes ACE.

**Quando usar IP/CIDR mesmo assim:**
- Quando a origem é **externa** à VPC (allowlist de IPs corporativos vindos de fora)
- Para regras de tráfego entre on-prem ↔ GCP via VPN

**Armadilha de prova:**
Distratores tipicamente sugerem que tags servem para "controlar IPs efêmeros" (não, tag não atribui IP), "billing" (firewall rules em GCP são **gratuitas** — só tráfego é cobrado), ou "network sniffing" (sniffing é captura, sem relação).

**Exemplo prático:**
```bash
# Aplicar tag a uma VM
gcloud compute instances add-tags minha-vm --tags=web-server --zone=us-central1-a

# Firewall rule que aplica a qualquer VM com a tag
gcloud compute firewall-rules create allow-https-web \
  --network=minha-vpc \
  --action=ALLOW \
  --direction=INGRESS \
  --rules=tcp:443 \
  --source-ranges=0.0.0.0/0 \
  --target-tags=web-server
```

### IPs de uma VM (interno obrigatório, externo opcional)

Uma VM no Compute Engine pode ter:

| Tipo de IP | Obrigatório? | Para que serve |
|------------|--------------|----------------|
| **IP interno (RFC 1918)** | **SIM, sempre** | Comunicação dentro da VPC. Atribuído pela sub-rede onde a VM nasce. |
| **IP externo** | **Não, opcional** | Conexões iniciadas **de** ou **para** a internet pública |
| **Alias IP ranges** | Não, opcional | Múltiplos serviços/pods por VM (usado pelo GKE para pods) |

**Por que isso é uma pegadinha clássica:**
O senso comum diz "uma VM precisa de internet para funcionar, logo precisa de IP externo". **Errado.** Em arquiteturas bem desenhadas, a **maioria** das VMs de produção **não tem IP externo**:

- **Acesso a pacotes/atualizações da internet:** roteia saída via **Cloud NAT** (NAT Gateway gerenciado). VM sem IP externo consegue fazer `apt update`, baixar imagens Docker, chamar APIs externas — tudo pela rota Cloud NAT.
- **Acesso administrativo (SSH/RDP):** use **IAP TCP Forwarding** (Identity-Aware Proxy). Você abre um túnel autenticado em vez de expor a porta 22 no IP público.
- **Comunicação entre serviços internos:** acontece via IP interno, sem precisar sair da VPC.

**Armadilha de prova:**
Sempre que a pergunta for *"minimum number of IP addresses a VM needs"*, a resposta é **um (1) — só o interno**. Distratores vão sugerir "dois (interno + externo)" para te pegar no senso comum.

**Quando uma VM realmente precisa de IP externo:**
- Bastion/jump host (mas IAP é melhor)
- VM servindo tráfego público diretamente sem load balancer (raro e geralmente desaconselhado)
- VM precisando ser alvo de tráfego inbound de origens fora do GCP que não passam por LB

**Exemplo prático:**
```bash
# VM sem IP externo (padrão de produção)
gcloud compute instances create vm-privada \
  --machine-type=e2-medium \
  --subnet=web-tier \
  --no-address \
  --zone=us-central1-a
```

A flag `--no-address` é a que importa: ela cria a VM **sem** IP externo. Em ambiente de produção, esse deveria ser o padrão e o IP externo a exceção.

### Quiz "Virtual Networks" (2026-05-12) — 3 questões

**Q1 — Benefit of applying firewall rules by tag rather than by address?**
- Marquei: B ✅
- Correta: **B — VM criada com tag correspondente recebe a regra independente do IP atribuído**
- Conceito-chave: tags desacoplam firewall do endereçamento. Essencial para MIGs/auto-scaling, onde IPs são voláteis.

**Q2 — Minimum number of IP addresses a VM needs?**
- Marquei: B (interno + externo) ❌
- Correta: **A — One: only an internal IP address**
- Conceito-chave: externo é **opcional**. Em produção, a maioria das VMs vive só com interno e sai para internet via Cloud NAT, com acesso admin via IAP.
- **Erro registrado em [flashcards.md](../../flashcards.md).**

**Q3 — Three types of networks offered in Google Cloud?**
- Marquei: B ou A (incerto) ⚠️ — A é a correta
- Correta: **A — Default, auto, custom**
- Conceito-chave: três **modos de VPC**. Não confundir com "zonal/regional/global" (escopo de recurso) nem com "unicast/multicast" (tipo de tráfego — GCP nem suporta multicast nativo).

## Cenário real

**No trabalho como cloud engineer, você vai:**
- **Sempre** criar VPCs em **custom-mode** para projetos sérios. Auto-mode só em sandbox.
- Desenhar IPAM **antes** de criar qualquer subnet (alinhar com on-prem se houver VPN/Interconnect).
- Aplicar firewall rules **por tag ou service account**, nunca por IP da VM. IP só para origens externas fixas (ex.: range da matriz corporativa).
- Criar VMs **sem IP externo por padrão**. Cloud NAT para saída, IAP para admin. IP externo só quando há justificativa explícita (e mesmo assim, geralmente preferir LB na frente).

**Conversa típica de feira:**
*"No projeto X a gente saiu da default network e migrou pra VPC custom-mode porque tinha sobreposição de range com o on-prem. A gente parou de usar IP-based firewall e migrou tudo pra network tags + service accounts, o que destravou o auto-scaling — antes a gente atualizava allowlist manualmente cada vez que um IP mudava."*

## Armadilhas de prova

1. **"VM precisa de 2 IPs"** → não. Precisa de **1**, o interno.
2. **VPC zonal/regional** → não. VPC é **global**, só sub-rede é regional.
3. **Tags servem para billing/IP/sniffing** → não. Servem para **agrupar VMs e aplicar firewall independente do IP**.
4. **Firewall rules têm custo** → não. Regras são gratuitas, **só tráfego é cobrado** (egress).
5. **Default network é segura** → não. Tem regras `allow-ssh` e `allow-rdp` de `0.0.0.0/0` por padrão. Deletar em produção.
