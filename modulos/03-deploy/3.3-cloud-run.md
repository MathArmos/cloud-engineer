## Conceitos

### Bloco 1 — Overview de Cloud Run

Cloud Run é uma **plataforma de computação totalmente gerenciada (serverless)** que executa containers na infraestrutura do Google. Qualquer linguagem é suportada — basta compilar para **Linux 64-bit** e empacotar em uma imagem de container.

**Por que existe:** elimina o overhead operacional de gerenciar VMs/clusters quando você só quer rodar um container que responde a HTTP. O Google cuida de provisionamento, scaling, HTTPS, balanceamento, logging.

**Integração:** funciona bem com Cloud SQL, Cloud Storage, Firestore, Cloud Logging, IAM, Pub/Sub, Eventarc.

#### Services vs Jobs — a distinção fundamental

| Tipo | Quando usar | Comportamento |
|------|-------------|---------------|
| **Service** | Código que **responde a web requests** ou eventos | Roda continuamente, escuta em porta TCP, recebe HTTP |
| **Job** | Código que **executa uma tarefa e termina** (script, batch) | Executado via gcloud CLI, Cloud Scheduler ou workflow; pode ser **Array Job** (múltiplas instâncias paralelas) |

**Analogia:** Service é como um servidor web sempre ligado; Job é como um cron job que dispara e morre.

**Exemplo Array Job:** processar 1000 imagens do Cloud Storage em paralelo → 1000 task instances em paralelo, cada uma com seu container.

#### HTTPS no Cloud Run (Services)

- Endpoint HTTPS automático em subdomínio único `*.run.app`
- Provisiona **certificado TLS** automaticamente
- Suporta **WebSockets, HTTP/2, gRPC**
- Domínio customizado configurável
- Responsabilidade do dev: escutar em uma porta TCP e tratar HTTP requests

#### Features que caem em prova

- **Serverless** — sem provisionamento de infra
- **Auto-scale rápido baseado em requisições** — até **1000 container instances** por service (default)
- Remove containers ociosos automaticamente
- **Cada deploy cria uma revisão imutável** → permite **rollback** e **traffic splitting** (gradual rollout)
- Service pode ser público (internet) ou privado
- Acesso a recursos VPC via **Serverless VPC Access Connector**
- Logging e error reporting automáticos
- **Pay-per-use** pricing
- Suporta CI/CD com GitHub, Bitbucket, Cloud Source Repositories

#### Formas de invocar um Cloud Run Service

1. **HTTPS** — RESTful API, microservice, middleware, app web
2. **gRPC** — comunicação entre microservices internos (até **7x mais rápido que REST**, usa protocol buffers)
3. **WebSockets** — sem config adicional
4. **Pub/Sub** — push messages como HTTP requests (ex: transformar dados após upload no Cloud Storage)
5. **Cloud Scheduler** — execução agendada (similar a cron)
6. **Cloud Tasks** — tarefas assíncronas seguras
7. **Eventarc** — eventos de fontes GCP (Cloud Storage, BigQuery via Audit Logs)

### Bloco 2 — Resource Model

#### Criação de um service

- Primeiro deploy → cria **service** + **Revision 1**
- Deploy via: Console, gcloud CLI, YAML ou Terraform
- Cada service tem **URL HTTPS única e permanente** em subdomínio de `run.app`
- Service é o recurso principal de interação (deploy, rollback, env vars, scaling)

```bash
gcloud run deploy my-service \
  --image=gcr.io/my-project/my-image \
  --region=us-central1
```

#### Tratamento de requests

```
Cliente → endpoint HTTPS (*.run.app) → Cloud Run → Container Instance
```

Cloud Run inicia containers sob demanda e encaminha requests para eles.

### Bloco 3 — Container Lifecycle

Os **5 estados** do container no Cloud Run:

| Estado | Descrição |
|--------|-----------|
| **Starting** | Materializa imagem e inicia aplicação |
| **Serving requests** | Container processa web requests |
| **Idle** | Sem requisições (entra após 100 ms sem tráfego) |
| **Shutting down** | Encerramento gracioso (se SIGTERM tratado) |
| **Stopped** | Estado final |

#### Starting — 4 passos

1. **Downloads/materializa** a imagem (cria root file system)
2. **Runs** o entrypoint da aplicação
3. **Waits** conexões TCP — sonda continuamente a **porta 8080** (configurável). Pode-se configurar **health checks e liveness probes** via YAML (HTTP, TCP ou gRPC)
4. **Forwards** as requests para o container

**Regra crítica:** sua aplicação só deve abrir a porta quando estiver de fato pronta para tratar requests.

#### De onde vem a imagem de container?

Existem **dois momentos** em que o Cloud Run puxa a imagem, de **fontes diferentes**:

| Evento | Fonte | Detalhe |
|--------|-------|---------|
| **Deploy** (primeira vez) | **Artifact Registry** | Cloud Run puxa e **copia** para storage interno |
| **Start** (cada container novo) | **Internal storage** do Cloud Run | Cloud Run puxa da cópia interna |

**Benefícios do storage interno:**
- Otimizado: imagens grandes carregam tão rápido quanto pequenas
- **Isolamento de falhas** do Artifact Registry (se a imagem for removida lá, o service continua funcionando porque Cloud Run tem a cópia interna)

#### Idle

- Container ocioso após **100 ms sem requests**
- **Não atende requests, não gera custo** (default)
- **CPU throttled para ~zero** (app fica muito lenta)
- Pode ser encerrado a qualquer momento
- Background tasks NÃO funcionam confiavelmente quando idle → usar **Cloud Tasks**
- Requests para terceiros provavelmente falham em estado idle
- Pode configurar **CPU always allocated** (cobra ciclo de vida inteiro — útil para async tasks)

#### Idle ↔ Serving (transição)

- Container alterna entre idle e serving várias vezes
- Transição de idle → serving: CPU é destravada instantaneamente, sem lag perceptível
- Cloud Run mantém containers idle por **até 15 minutos** para mitigar cold start
- Setting **`min-instances`** garante N containers sempre prontos

#### Shutting down (SIGTERM)

- Sem tratamento de SIGTERM → container apenas desaparece
- Com tratamento de SIGTERM → **10 segundos** para limpeza graciosa

**Checklist de graceful shutdown:**
1. Fechar TCP/file descriptors/conexões de DB (evita "max connections" downstream)
2. Flush de buffers (telemetria, dados em batch)
3. Escrever log de debugging

#### Stopped — situações de parada forçada

Cloud Run **nunca para um container que está atendendo requests** em condições normais. Mas pode parar abruptamente se:

| Causa | Detalhe |
|-------|---------|
| **Aplicação falha** (exit/erro) | Erro no código da aplicação |
| **Memory limit excedido** | Default: **512 MiB** por instância; máximo configurável: **32 GiB** |

Quando isso acontece, **todas requests in-flight falham com erro**. Enquanto Cloud Run sobe um substituto, novas requests podem ter que esperar.

### Bloco 4 — Autoscaling

#### Como funciona

- Todo tráfego passa por um **Internal Application Load Balancer**
- Cloud Run adiciona containers quando todos estão ocupados
- Quando demanda cai, para de enviar tráfego e encerra containers
- Container pode receber **múltiplos requests simultâneos** (controlado por `concurrency`)

#### Fatores que influenciam o número de instâncias

1. Taxa de requests
2. **Utilização de CPU** das instâncias existentes — target **60%**
3. Setting `maximum concurrency`
4. Settings `min-instances` e `max-instances`

#### Números-chave (decoram para a prova)

| Setting | Valor |
|---------|-------|
| **Limite hard de instâncias por service** | **1.000** (default, quota request para mais) |
| **Max instances (default)** | **100** |
| **Concurrency (default)** | **80 requests/container** |
| **Concurrency (máximo)** | **1.000 requests/container** |
| **Concurrency (mínimo)** | **1** |
| **CPU utilization target** | **60%** |

#### Scale to zero

- Sem requests → última instância encerrada → custo zero
- Próximo request paga preço do **cold start** (queue + tempo de inicialização)
- Mitigado com `min-instances`

#### Request queuing (cold start)

- Após scale to zero, primeiros requests ficam **em fila** esperando container subir
- Mitigação: `min-instances` mantém containers idle prontos

#### Minimum instances

```bash
gcloud run services update my-service --min-instances 3
```

- Mantém **N containers sempre rodando** (mesmo idle)
- **Instâncias idle do min-instances GERAM COBRANÇA** (diferença crítica vs idle "normal" que não cobra)
- Reduz latência eliminando cold start

#### Maximum instances

```bash
gcloud run services update my-service --max-instances 3
```

Razões para limitar:
1. **Controle de custo**
2. **Proteção de downstream services** (ex: limite de conexões de DB)
3. **Compatibilidade com recursos externos**

**Cuidado:** setar muito baixo = Cloud Run não absorve picos → requests falham.

#### Maximum concurrency

```bash
gcloud run services update my-service --concurrency 80   # default
gcloud run services update my-service --concurrency 1    # 1 request/container
```

**Quando setar concurrency = 1:**
- Cada request consome CPU/memória quase total
- App não é multi-thread safe
- App depende de **estado global não-compartilhável**

**Trade-offs:**

| Concurrency baixa (ex: 1) | Concurrency alta (ex: 80+) |
|--------------------------|---------------------------|
| Mais containers → mais custo | Menos containers → menor custo |
| OK para apps single-thread/stateful | Risco de sobrecarregar downstream em picos |
| Menos memória/container | Mais memória total/container |

#### Cheat sheet de decisão

| Situação | Setting |
|----------|---------|
| Latência crítica, não pode cold start | `--min-instances N` |
| Proteção de DB downstream | `--max-instances N` baixo |
| App single-thread (Python sync, etc.) | `--concurrency 1` |
| App I/O bound multi-thread | `--concurrency 80+` |
| Custo é prioridade #1 | Scale to zero (sem min-instances) |

### Bloco 5 — Access Control

#### Google Cloud = coleção de APIs

Toda interação com GCP é via **chamadas de API**:
- **Console web** (`console.cloud.google.com`)
- **gcloud CLI** (terminal)
- **Terraform** (IaC)
- **Client libraries** (no código)

Exemplo: `gcloud run deploy` → chamada para `https://run.googleapis.com`.

#### IAM — anatomia da Policy

```
IAM Policy (anexada a um recurso)
└── Policy Binding
    ├── Member (Identity)
    └── Role
        └── Permissions
```

**Regras:**
- Policy **sempre anexada a um recurso**
- Policy binding: 1 member → 1 role
- 1 member pode ter **múltiplos bindings** (múltiplas roles)
- 1 role contém múltiplas permissions

#### Tornar service público

**Combo necessário:**
- Member: **`allUsers`**
- Role: **`roles/run.invoker`**

```bash
# Via add-iam-policy-binding
gcloud run services add-iam-policy-binding my-service \
  --member="allUsers" \
  --role="roles/run.invoker"

# Via deploy (atalho)
gcloud run deploy my-service --allow-unauthenticated
```

**Pré-requisito:** ser Project Owner ou ter `roles/run.admin`.

**Para tirar acesso público** (lab GSP723 task 2):

```bash
gcloud run services remove-iam-policy-binding my-service \
  --region=us-central1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

A flag `--no-allow-unauthenticated` **só funciona em `deploy`**, não em `update`. Em `update`, use `remove-iam-policy-binding`.

#### Comandos-chave de IAM

| Ação | Comando |
|------|---------|
| Adicionar role em service | `gcloud run services add-iam-policy-binding my-service --member=M --role=R` |
| Remover role em job | `gcloud run jobs remove-iam-policy-binding my-job --member=M --role=R` |
| Project-level (todos services) | `gcloud projects add-iam-policy-binding my-project --member=M --role=R` |

#### Ingress Settings — controle de rede (independente do IAM)

| Setting | Permite |
|---------|---------|
| **All** (default) | Internet via `*.run.app` ou domínio custom |
| **Internal and Cloud Load Balancing** | Internal + External HTTP(S) LB (não internet direto) |
| **Internal** (mais restritivo) | Internal HTTP(S) LB, VPC SC perimeter, VPC mesmo projeto, Cloud Tasks/Eventarc/Pub-Sub/Workflows |

**VPC Service Controls (VPC SC):** perímetro que protege contra **data exfiltration**. Aplica-se ao `run.app` URL e domínios custom.

Em `Internal`, tráfego de serviços GCP no mesmo projeto/perímetro **fica dentro da rede do Google** mesmo via `run.app`. Internet NUNCA atinge `run.app` nem domínio custom.

#### Serverless VPC Access (para acessar recursos internos)

**Por que:** Cloud Run é serverless e fica fora da sua VPC. Para acessar VMs/Memorystore/Cloud SQL via **IP interno**, precisa de um **Serverless VPC Access Connector**.

**Setup (4 passos):**
1. Habilitar Serverless VPC Access API
2. Criar connector
3. Anexar a VPC + região
4. Configurar service/job para usar o connector

```bash
gcloud run deploy my-service --image my-image \
  --vpc-connector my-connector
```

**Requisitos críticos:**

| Item | Regra |
|------|-------|
| **Região** | Connector tem que estar na **mesma região** do service |
| **CIDR** | **/28 não usado** ou non-overlapping /28 |
| **Subnet** | Usada **exclusivamente pelo connector** |

Para internal services, setar **todo o egress** para usar o VPC connector. Pode restringir o connector via firewall rules.

### Bloco 6 — Service Identity and Authentication

#### Tipos de identidades no IAM

| Categoria | Tipo | Uso |
|-----------|------|-----|
| **Human** | Google Account | login pessoal |
| **Human** | Group / Domain | grupos Workspace |
| **Machine** | **Service Account** | apps, VMs, Cloud Run, etc. |
| **All users** | `allUsers` | acesso público (anônimo) |

#### Service Account — anatomia

| Atributo | Valor |
|----------|-------|
| Identificador | **email único** |
| Senha | NÃO tem |
| Login browser | IMPOSSÍVEL |
| Impersonation | Outros podem agir em seu nome |
| Workspace domain | NÃO faz parte (mas pode ir em grupos) |

#### Service Accounts em Cloud Run

- **TODA service/job tem uma SA** = **service identity**
- **Default:** **default Compute Engine service account** com role **Editor** (muito amplo)
- **Best practice:**
  - **User-managed SA** dedicada
  - **Uma SA por service**
  - **Permissões mínimas** (least privilege)

#### Tokens — diferença crítica

| Token | Uso | Cenário |
|-------|-----|---------|
| **OAuth 2.0 Access Token** | Chamadas para **APIs Google** (Pub/Sub, BQ, Storage...) | App → API Google |
| **OIDC ID Token** | **Service-to-service Cloud Run** | Cloud Run A → Cloud Run B |

**OpenID Connect (OIDC):** protocolo de identidade baseado em OAuth 2.0. Verifica identidade do cliente.

#### Fluxo: Cloud Run → API Google (ex: Pub/Sub)

```
App + Client Library
    ↓ usa
Service Account (runtime)
    ↓ obtém
OAuth 2.0 Access Token
    ↓ API call com token
IAM verifica token + policy do Pub/Sub Topic
    ↓
Permite OU Rejeita
```

A client library **automaticamente** obtém o access token via SA — você não gerencia credenciais.

#### Fluxo: Cloud Run → Cloud Run (síncrono, service-to-service)

```
Calling Service (com SA)
    ↓ obtém
ID Token (OIDC) assinado pelo Google
    ↓ HTTP request com Authorization: Bearer <ID_TOKEN>
Receiving Service
    ↓ IAM valida token
    ↓ Verifica policy: SA tem roles/run.invoker no receiving?
    ↓
Permite OU Rejeita
```

**Setup:**
```bash
gcloud run services add-iam-policy-binding RECEIVING_SERVICE \
  --member='serviceAccount:CALLING_SERVICE_IDENTITY' \
  --role='roles/run.invoker'
```

#### Async vs Sync entre Cloud Run services

| Tipo | Ferramentas |
|------|-------------|
| **Assíncrona** | **Cloud Tasks, Pub/Sub, Cloud Scheduler, Eventarc** |
| **Síncrona** | HTTP direto + IAM (`roles/run.invoker` + OIDC ID token) |

### Bloco 7 — Resource Hierarchy

```
Organization (raiz, opcional)
    ↓
Folder (opcional, aninhável)
    ↓
Project (base obrigatória)
    ↓
Resources (Cloud Run, Pub/Sub topic, VM, etc.)
```

| Nível | Característica |
|-------|---------------|
| **Organization** | Nó raiz. Visibilidade/controle central. Não tem parent. |
| **Folder** | Opcional. Aninhável. Mapeia departamentos/times/BUs. |
| **Project** | **Base obrigatória.** Necessário para: criar recursos, usar APIs, gerenciar permissões, billing. |
| **Resource** | Recursos individuais (todo recurso pertence a 1 project). |

**Regra:** cada recurso tem **exatamente 1 parent** (exceto Organization).

#### Herança de policy bindings (conceito-chave)

**Bindings em nível superior são herdadas pelos níveis inferiores.**

Exemplo: `roles/pubsub.publisher` no project → user pode publicar em **TODOS os topics** desse project.

#### Effective IAM Policy

```
Effective Policy do recurso =
    bindings do recurso
    + bindings do project
    + bindings do folder
    + bindings da organization
```

**Regra de ouro:** permissões herdadas de cima **NÃO podem ser removidas em níveis inferiores**. Se quer restringir, **sobe na hierarquia** e remove lá (ou usa IAM Deny policies, recurso avançado).

#### Quando adicionar binding em cada nível

| Nível | Use quando |
|-------|------------|
| **Organization** | Permissões globais (admins, security, billing) |
| **Folder** | Permissões por BU/time |
| **Project** | Permissões padrão da aplicação inteira |
| **Recurso individual** | Granularidade fina, least privilege |

### Bloco 8 — Principle of Least Privilege

#### Tipos de IAM Roles

| Tipo | Características | Exemplos | Quando usar |
|------|----------------|----------|-------------|
| **Basic** | Muito amplas, todos services | Owner, Editor, Viewer | NUNCA em produção |
| **Predefined** | Granulares, service-específicas, gerenciadas pelo Google | Cloud Run Admin, Pub/Sub Publisher, Cloud Tasks Enqueuer | **Default recomendado** |
| **Custom** | Lista definida por você | Suas próprias | Quando predefined não serve |

#### O perigo da Default Service Account

Cloud Run sem `--service-account` usa **Default Compute Engine SA** com role **Editor** no project.

**Risco:** por causa da **herança**, Editor no project = acesso read/write em **TODOS** os recursos:
- Cloud SQL instances
- Cloud Storage buckets
- Outros Cloud Run services
- Etc.

Se o código tiver vulnerabilidade → atacante destrói/exfiltra tudo do project.

#### Aplicando least privilege — 3 passos

1. Criar nova SA para o Cloud Run service
2. Configurar essa SA como service identity (`--service-account` no deploy)
3. Adicionar policy bindings com roles predefined/custom **APENAS nos recursos necessários**

#### SA nova começa SEM permissões

Recém-criada → nenhum binding → toda chamada IAM rejeitada → precisa adicionar bindings explicitamente **no recurso destino**.

```bash
gcloud pubsub topics add-iam-policy-binding my-topic \
  --member="serviceAccount:my-sa-email" \
  --role="roles/pubsub.publisher"
```

#### Default vs User-managed (comparação)

| | **Default Compute Engine SA** | **User-managed (per-service)** |
|---|------------------------------|--------------------------------|
| Role | Editor (basic, ampla) | Predefined/custom (granular) |
| Escopo | TODO o project | Apenas recursos específicos |
| Risco | Alto | Baixo |
| Quando | Dev/protótipo | **PRODUÇÃO** |

#### Onde configurar a SA no Cloud Run

Via Console, gcloud, YAML ou Terraform no create/update/deploy:

```bash
gcloud run deploy my-service --image=IMAGE \
  --service-account=MY_SA_EMAIL
```

### Bloco 9 — Secrets and Environment Variables

#### Environment Variables

- **Key-value pairs** injetadas no container, lidas em runtime
- Setáveis em: deploy, create/update, nova revisão
- Via: Console, gcloud, YAML, Terraform
- Default do `ENV` no Dockerfile é **sobrescrito** pelo valor setado no Cloud Run
- Existem **vars reservadas** (container runtime contract)

```bash
gcloud run deploy my-service --image=IMG \
  --update-env-vars FOO=bar,BAZ=boo
```

**Leitura no código:**
| Linguagem | Como |
|-----------|------|
| Python | `os.environ.get("key")` |
| Node.js | `process.env.key` |
| Java | `System.getenv("key")` |

#### Secrets via Secret Manager

Para info sensível (API keys, senhas, tokens), use Secret Manager — **nunca hardcode em env var**.

**Estrutura:**
- Metadados (replication locations, labels, permissions)
- **Secret versions** = armazenam o dado real (string ou binary blob)

#### Duas formas de acessar secret (DIFERENÇA CRÍTICA p/ prova)

| Método | Comportamento | Quando usar |
|--------|---------------|-------------|
| **Volume mount** | Arquivo no container, re-lê do Secret Manager a cada read | **`latest` OK** |
| **Environment variable** | Resolvida no startup do container | **PINAR versão específica** (não usar `latest`) |

```bash
# Como volume (arquivo)
gcloud run deploy my-service --image=IMG \
  --update-secrets=SECRET_FILE_PATH=my_secret:VERSION

# Como env var
gcloud run deploy my-service --image=IMG \
  --update-secrets=ENV_VAR_NAME=my_secret:VERSION
```

#### Permissão para acessar secrets

SA do service precisa da role **`roles/secretmanager.secretAccessor`** no secret:

```bash
gcloud secrets add-iam-policy-binding my-secret-id \
  --member="serviceAccount:sa-email" \
  --role="roles/secretmanager.secretAccessor"
```

### Cloud Run Functions

| Capability | Detalhe |
|-----------|---------|
| **Tipo** | Serverless function execution |
| **Trigger** | Eventos do Cloud (HTTPS, Pub/Sub, Cloud Storage, Eventarc) |
| **Scaling** | Por número de eventos |
| **Estado** | **Stateless** — persistir externamente (Datastore, Cloud Storage) |
| **Concorrência** | **1 request por instância** — escala criando novas instâncias |
| **Cold start** | Min instances elimina cold start (paga mesmo idle) |
| **Base image** | Provido pelo Cloud Run Functions, auto-patched |

#### Cloud Storage Triggers — evento correto

```bash
gcloud functions deploy NAME \
  --trigger-event google.storage.object.finalize \
  --trigger-resource BUCKET_NAME \
  --runtime python311
```

| Event | Válido? | O que dispara |
|-------|---------|---------------|
| **`google.storage.object.finalize`** | ✅ Único válido | Write para Cloud Storage **completo** |
| `google.storage.object.create` | ❌ | Não existe |
| `google.storage.object.change` | ❌ | Não existe |
| `google.storage.object.add` | ❌ | Não existe |

### App Engine — Standard vs Flexible (comparação)

| Característica | **Standard** | **Flexible** |
|----------------|--------------|--------------|
| Runtime | Sandbox com **linguagem específica** | Docker containers em Compute Engine VMs |
| Scale to zero | ✅ | ❌ |
| Native code | ❌ | ✅ |
| Modificar runtime | ❌ | ✅ |
| Startup | **Segundos** | **Minutos** |
| Custom packages | ❌ | ✅ |
| Use case | Scaling rápido | Native code / customização |

#### Comparação rápida — quando usar cada um

| Critério | Cloud Run | App Engine Standard | App Engine Flexible | Cloud Run Functions |
|----------|-----------|--------------------|--------------------|---------------------|
| Container custom | ✅ | ❌ | ✅ | ❌ |
| Scale to zero | ✅ | ✅ | ❌ | ✅ |
| Pay-per-request | ✅ | ✅ | ❌ | ✅ |
| Custom packages | ✅ | ❌ | ✅ | ❌ |
| Requests concorrentes/instância | Múltiplas | Múltiplas | Múltiplas | **1** |

## Questionários

### DQ 04 — Container web app, serverless, pay-per-request, custom packages

**Resposta: C — Cloud Run**

| Opção | Por que não? |
|-------|--------------|
| App Engine Flexible | ❌ Não escala para zero |
| App Engine Standard | ❌ Não permite custom packages |
| **Cloud Run** ✅ | Serverless, container, custom packages, pay-per-request |
| Cloud Run Functions | ❌ Não usa containers completos para lógica |

### DQ 05 — Trigger de Cloud Storage para função

**Resposta:** `google.storage.object.finalize` — único evento real. Os outros não existem.

### Quiz Módulo 1 — Fundamentals of Cloud Run

Pontos de atenção do quiz:
- **Revisões no Cloud Run são imutáveis** — não dá para "modificar uma revision", você deploya uma nova
- **Cloud Run services são regionais** (não globais nem zonais)
- **Cada deploy cria uma nova revision** (essa imutabilidade é o que permite rollback/traffic splitting)
- **Idle container não cobra por default** (mas com `min-instances` sim)
- **CPU throttled em idle** → background tasks não-confiáveis
- Cloud Run **reduz instâncias** quando demanda cai (não só aumenta)

Lab prático que valida toda a teoria de IAM/SA/least privilege em Cloud Run.

#### Task 1-2 — Remover acesso público

Cenário: existe `billing-service` deployado com `--allow-unauthenticated` (combo `allUsers` + `roles/run.invoker`). Time de segurança pediu para remover.

**Pegadinha:** `--no-allow-unauthenticated` SÓ existe em `gcloud run deploy`, NÃO em `update`. Solução em `update`:

```bash
gcloud run services remove-iam-policy-binding billing-service \
  --region=us-central1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

#### Task 3 — Redeploy seguro

```bash
gcloud run services delete billing-service
gcloud run deploy billing-service \
  --image gcr.io/qwiklabs-resources/gsp723-parking-service \
  --region $LOCATION \
  --no-allow-unauthenticated
```

Curl sem autenticação → **HTTP 403 Forbidden**. Esperado.

#### Task 4 — Invocar com autenticação (OIDC ID token)

Passos:

1. Criar SA `Billing Initiator` no Console com role **Cloud Run Invoker** (no project!)
2. Capturar email da SA:
   ```bash
   BILLING_INITIATOR_EMAIL=$(gcloud iam service-accounts list \
     --filter="Billing Initiator" --format="value(EMAIL)")
   ```
3. Gerar key file:
   ```bash
   gcloud iam service-accounts keys create key.json \
     --iam-account=${BILLING_INITIATOR_EMAIL}
   ```
4. Ativar a SA no gcloud:
   ```bash
   gcloud auth activate-service-account --key-file=key.json
   ```
5. Invocar com ID token:
   ```bash
   curl -i -X POST -H "Content-Type: application/json" \
     -H "Authorization: Bearer $(gcloud auth print-identity-token)" \
     $BILLING_SERVICE_URL -d '{"userid": "1234", "minBalance": 500}'
   ```
   → **HTTP 204 No Content** (sucesso, app Express não retorna body)

#### Task 5 — Least privilege na prática

Lab demonstra que **dar role no nível do project é privilégio EXCESSIVO**:

1. Deploy de um segundo service `billing-service-2`
2. Mesma SA `Billing Initiator` invoca o segundo service também → funciona por **herança** do binding no project
3. Solução: **mover o binding do project para o service específico**:

```bash
# Remover no project
gcloud projects remove-iam-policy-binding $GOOGLE_CLOUD_PROJECT \
  --member=serviceAccount:${BILLING_INITIATOR_EMAIL} \
  --role=roles/run.invoker

# Adicionar apenas no billing-service
gcloud run services add-iam-policy-binding billing-service \
  --region=$LOCATION \
  --member=serviceAccount:${BILLING_INITIATOR_EMAIL} \
  --role=roles/run.invoker \
  --platform=managed
```

Resultado:
- `billing-service` → **HTTP 204** (SA tem binding direto)
- `billing-service-2` → **HTTP 403** (SA não tem mais binding no project, nem no service-2)

#### Demonstrações poderosas do least privilege

A SA `Billing Initiator` (depois de ajustada) NÃO consegue:
- Listar service accounts (`iam.serviceAccounts.list` negado)
- Listar Cloud Run services (`run.services.list` negado)
- Invocar `billing-service-2`

Mas CONSEGUE:
- Invocar `billing-service` (única permissão real)

Isso é **least privilege absoluto** — SA literalmente useless para qualquer coisa além da tarefa específica.

#### Conceitos validados pelo lab

| Conceito | Como apareceu |
|----------|---------------|
| `--allow-unauthenticated` = `allUsers + roles/run.invoker` | Removido via `remove-iam-policy-binding` |
| Service privado precisa de OIDC ID token | `gcloud auth print-identity-token` |
| Herança em nível de project = privilégio excessivo | SA invocou service-2 sem permissão explícita |
| Binding em nível de recurso = least privilege | Mover do project para o service |
| SAs no Cloud Run usam OIDC para auth, não OAuth | `print-identity-token`, não `print-access-token` |

## Cenário real

**Caso típico no trabalho:**

Você está montando um sistema de microservices em Cloud Run para um cliente de e-commerce:
- `frontend-service` — público (allow-unauthenticated)
- `cart-service` — privado, invocado só pelo frontend
- `payment-service` — privado, invocado só pelo cart
- `analytics-job` — Cloud Run **Job** disparado por Cloud Scheduler diariamente

**Como aplicar tudo que aprendemos:**

1. **Cada service tem sua user-managed SA** (não usar a default Compute Engine SA)
2. **frontend-service SA** → role `roles/run.invoker` no `cart-service` (binding no recurso, não no project!)
3. **cart-service SA** → role `roles/run.invoker` no `payment-service`
4. **payment-service SA** → role `roles/secretmanager.secretAccessor` no secret com a API key do gateway
5. **analytics-job SA** → roles em BigQuery (`roles/bigquery.dataEditor`) e Cloud Storage (`roles/storage.objectViewer`)
6. **Cloud Scheduler** que dispara o job tem sua própria SA com `roles/run.invoker` no job
7. Secrets via **volume mount** (sempre latest, sem precisar redeploy)
8. Env vars apenas para config não-sensível (URLs, feature flags)
9. **Ingress = Internal and Cloud Load Balancing** em todos os services privados
10. **Serverless VPC Connector** se algum service precisar acessar Cloud SQL via IP interno

Sem fazer nada disso, alguém com acesso ao default SA podia deletar Cloud SQL inteiro porque tem Editor herdado no project.

## Armadilhas de prova

| Pegadinha | Resposta |
|-----------|----------|
| "Container + serverless + custom packages" | **Cloud Run** (não App Engine) |
| "App Engine sem scale to zero" | **Flexible** |
| "App Engine sem custom packages" | **Standard** |
| "Trigger Cloud Storage para função" | `google.storage.object.finalize` |
| "Cloud Run Functions: concorrência" | **1 request/instância** — escala criando novas instâncias |
| "Cloud Run: concorrência" | **Múltiplas** requests por instância (default 80) |
| "Tornar service público" | `allUsers` + `roles/run.invoker` (ou `--allow-unauthenticated` no deploy) |
| "`--no-allow-unauthenticated` em update" | NÃO existe — use `remove-iam-policy-binding` |
| "Service-to-service auth" | OIDC ID token + `roles/run.invoker` no receiver |
| "Cloud Run → API Google (Pub/Sub etc.)" | OAuth 2.0 access token (gerenciado pela client library) |
| "Default SA em Cloud Run" | Default Compute Engine SA com role **Editor** (perigo!) |
| "Acessar VM por IP interno" | **Serverless VPC Access Connector** (/28, mesma região) |
| "Min-instances cobra?" | **SIM** — instâncias idle do min-instances são cobradas |
| "Idle normal cobra?" | **NÃO** — só o min-instances |
| "Concurrency default / máximo" | **80 / 1000** |
| "Max instances default / hard limit" | **100 / 1000** |
| "Memory default / máximo" | **512 MiB / 32 GiB** |
| "Tempo até idle" | **100 ms** sem requests |
| "Tempo após SIGTERM até parar" | **10 segundos** |
| "Porta default de probing" | **8080** (configurável) |
| "Service revision pode ser modificada?" | **NÃO** — revisões são imutáveis. Cada mudança cria nova revision |
| "Cloud Run services são regionais?" | **SIM** (não globais nem zonais) |
| "Secret latest em env var é OK?" | **NÃO** — env vars resolvem só no startup; use volume mount para latest |
| "Volume mount lê quando?" | A cada **read** (sempre da SM, suporta latest) |
| "Role para SA acessar Secret Manager" | `roles/secretmanager.secretAccessor` |
| "Permissão herdada do project pode ser removida no filho?" | **NÃO** — sobe na hierarquia ou usa Deny policy |
| "Tipos de roles, qual usar em produção?" | **Predefined** primeiro, Custom se não tiver. NUNCA Basic (Owner/Editor/Viewer) |
| "Async entre Cloud Run services" | Cloud Tasks, Pub/Sub, Cloud Scheduler, Eventarc |
| "Sync entre Cloud Run services" | HTTP direto + OIDC ID token + `roles/run.invoker` |
