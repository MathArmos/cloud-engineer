## Conceitos

### Bloco 1 — Overview de Cloud Run

Cloud Run é uma **plataforma de computação totalmente gerenciada (serverless)** que executa containers na infraestrutura do Google. Qualquer linguagem é suportada — basta compilar para **Linux 64-bit** e empacotar em uma imagem de container.

**Por que existe:** elimina o overhead operacional de gerenciar VMs/clusters quando você só quer rodar um container que responde a HTTP. O Google cuida de provisionamento, scaling, HTTPS, balanceamento, logging.

**Integração:** funciona bem com Cloud SQL, Cloud Storage, Firestore, Cloud Logging, IAM, Pub/Sub, Eventarc.

#### Services vs Jobs — a distinção fundamental

| Tipo | Quando usar | Comportamento |
|------|-------------|---------------|
| **Service** | Código que **responde a web requests** ou eventos | Roda continuamente, escuta em porta TCP, recebe HTTP |
| **Job** | Código que **executa uma tarefa e termina** (script, batch) | Executado via gcloud CLI, Cloud Scheduler ou workflow; pode ser **Array Job** (múltiplas instâncias paralelas) |

**Analogia:** Service é como um servidor web sempre ligado; Job é como um cron job que dispara e morre.

**Exemplo Array Job:** processar 1000 imagens do Cloud Storage em paralelo → 1000 task instances em paralelo, cada uma com seu container.

#### HTTPS no Cloud Run (Services)

- Endpoint HTTPS automático em subdomínio único `*.run.app`
- Provisiona **certificado TLS** automaticamente
- Suporta **WebSockets, HTTP/2, gRPC**
- Domínio customizado configurável
- Responsabilidade do dev: escutar em uma porta TCP e tratar HTTP requests

#### Features que caem em prova

- **Serverless** — sem provisionamento de infra
- **Auto-scale rápido baseado em requisições** — até **1000 container instances** por service (default)
- Remove containers ociosos automaticamente
- **Cada deploy cria uma revisão imutável** → permite **rollback** e **traffic splitting** (gradual rollout)
- Service pode ser público (internet) ou privado
- Acesso a recursos VPC via **Serverless VPC Access Connector**
- Logging e error reporting automáticos
- **Pay-per-use** pricing
- Suporta CI/CD com GitHub, Bitbucket, Cloud Source Repositories

#### Formas de invocar um Cloud Run Service

1. **HTTPS** — RESTful API, microservice, middleware, app web
2. **gRPC** — comunicação entre microservices internos (até **7x mais rápido que REST**, usa protocol buffers)
3. **WebSockets** — sem config adicional
4. **Pub/Sub** — push messages como HTTP requests (ex: transformar dados após upload no Cloud Storage)
5. **Cloud Scheduler** — execução agendada (similar a cron)
6. **Cloud Tasks** — tarefas assíncronas seguras
7. **Eventarc** — eventos de fontes GCP (Cloud Storage, BigQuery via Audit Logs)

### Bloco 2 — Resource Model

#### Criação de um service

- Primeiro deploy → cria **service** + **Revision 1**
- Deploy via: Console, gcloud CLI, YAML ou Terraform
- Cada service tem **URL HTTPS única e permanente** em subdomínio de `run.app`
- Service é o recurso principal de interação (deploy, rollback, env vars, scaling)

```bash
gcloud run deploy my-service \
  --image=gcr.io/my-project/my-image \
  --region=us-central1
```

#### Tratamento de requests

```
Cliente → endpoint HTTPS (*.run.app) → Cloud Run → Container Instance
```

Cloud Run inicia containers sob demanda e encaminha requests para eles.

### Bloco 3 — Container Lifecycle

Os **5 estados** do container no Cloud Run:

| Estado | Descrição |
|--------|-----------|
| **Starting** | Materializa imagem e inicia aplicação |
| **Serving requests** | Container processa web requests |
| **Idle** | Sem requisições (entra após 100 ms sem tráfego) |
| **Shutting down** | Encerramento gracioso (se SIGTERM tratado) |
| **Stopped** | Estado final |

#### Starting — 4 passos

1. **Downloads/materializa** a imagem (cria root file system)
2. **Runs** o entrypoint da aplicação
3. **Waits** conexões TCP — sonda continuamente a **porta 8080** (configurável). Pode-se configurar **health checks e liveness probes** via YAML (HTTP, TCP ou gRPC)
4. **Forwards** as requests para o container

**Regra crítica:** sua aplicação só deve abrir a porta quando estiver de fato pronta para tratar requests.

#### De onde vem a imagem de container?

Existem **dois momentos** em que o Cloud Run puxa a imagem, de **fontes diferentes**:

| Evento | Fonte | Detalhe |
|--------|-------|---------|
| **Deploy** (primeira vez) | **Artifact Registry** | Cloud Run puxa e **copia** para storage interno |
| **Start** (cada container novo) | **Internal storage** do Cloud Run | Cloud Run puxa da cópia interna |

**Benefícios do storage interno:**
- Otimizado: imagens grandes carregam tão rápido quanto pequenas
- **Isolamento de falhas** do Artifact Registry (se a imagem for removida lá, o service continua funcionando porque Cloud Run tem a cópia interna)

#### Idle

- Container ocioso após **100 ms sem requests**
- **Não atende requests, não gera custo** (default)
- **CPU throttled para ~zero** (app fica muito lenta)
- Pode ser encerrado a qualquer momento
- Background tasks NÃO funcionam confiavelmente quando idle → usar **Cloud Tasks**
- Requests para terceiros provavelmente falham em estado idle
- Pode configurar **CPU always allocated** (cobra ciclo de vida inteiro — útil para async tasks)

#### Idle ↔ Serving (transição)

- Container alterna entre idle e serving várias vezes
- Transição de idle → serving: CPU é destravada instantaneamente, sem lag perceptível
- Cloud Run mantém containers idle por **até 15 minutos** para mitigar cold start
- Setting **`min-instances`** garante N containers sempre prontos

#### Shutting down (SIGTERM)

- Sem tratamento de SIGTERM → container apenas desaparece
- Com tratamento de SIGTERM → **10 segundos** para limpeza graciosa

**Checklist de graceful shutdown:**
1. Fechar TCP/file descriptors/conexões de DB (evita "max connections" downstream)
2. Flush de buffers (telemetria, dados em batch)
3. Escrever log de debugging

#### Stopped — situações de parada forçada

Cloud Run **nunca para um container que está atendendo requests** em condições normais. Mas pode parar abruptamente se:

| Causa | Detalhe |
|-------|---------|
| **Aplicação falha** (exit/erro) | Erro no código da aplicação |
| **Memory limit excedido** | Default: **512 MiB** por instância; máximo configurável: **32 GiB** |

Quando isso acontece, **todas requests in-flight falham com erro**. Enquanto Cloud Run sobe um substituto, novas requests podem ter que esperar.

### Bloco 4 — Autoscaling

#### Como funciona

- Todo tráfego passa por um **Internal Application Load Balancer**
- Cloud Run adiciona containers quando todos estão ocupados
- Quando demanda cai, para de enviar tráfego e encerra containers
- Container pode receber **múltiplos requests simultâneos** (controlado por `concurrency`)

#### Fatores que influenciam o número de instâncias

1. Taxa de requests
2. **Utilização de CPU** das instâncias existentes — target **60%**
3. Setting `maximum concurrency`
4. Settings `min-instances` e `max-instances`

#### Números-chave (decoram para a prova)

| Setting | Valor |
|---------|-------|
| **Limite hard de instâncias por service** | **1.000** (default, quota request para mais) |
| **Max instances (default)** | **100** |
| **Concurrency (default)** | **80 requests/container** |
| **Concurrency (máximo)** | **1.000 requests/container** |
| **Concurrency (mínimo)** | **1** |
| **CPU utilization target** | **60%** |

#### Scale to zero

- Sem requests → última instância encerrada → custo zero
- Próximo request paga preço do **cold start** (queue + tempo de inicialização)
- Mitigado com `min-instances`

#### Request queuing (cold start)

- Após scale to zero, primeiros requests ficam **em fila** esperando container subir
- Mitigação: `min-instances` mantém containers idle prontos

#### Minimum instances

```bash
gcloud run services update my-service --min-instances 3
```

- Mantém **N containers sempre rodando** (mesmo idle)
- **Instâncias idle do min-instances GERAM COBRANÇA** (diferença crítica vs idle "normal" que não cobra)
- Reduz latência eliminando cold start

#### Maximum instances

```bash
gcloud run services update my-service --max-instances 3
```

Razões para limitar:
1. **Controle de custo**
2. **Proteção de downstream services** (ex: limite de conexões de DB)
3. **Compatibilidade com recursos externos**

**Cuidado:** setar muito baixo = Cloud Run não absorve picos → requests falham.

#### Maximum concurrency

```bash
gcloud run services update my-service --concurrency 80   # default
gcloud run services update my-service --concurrency 1    # 1 request/container
```

**Quando setar concurrency = 1:**
- Cada request consome CPU/memória quase total
- App não é multi-thread safe
- App depende de **estado global não-compartilhável**

**Trade-offs:**

| Concurrency baixa (ex: 1) | Concurrency alta (ex: 80+) |
|--------------------------|---------------------------|
| Mais containers → mais custo | Menos containers → menor custo |
| OK para apps single-thread/stateful | Risco de sobrecarregar downstream em picos |
| Menos memória/container | Mais memória total/container |

#### Cheat sheet de decisão

| Situação | Setting |
|----------|---------|
| Latência crítica, não pode cold start | `--min-instances N` |
| Proteção de DB downstream | `--max-instances N` baixo |
| App single-thread (Python sync, etc.) | `--concurrency 1` |
| App I/O bound multi-thread | `--concurrency 80+` |
| Custo é prioridade #1 | Scale to zero (sem min-instances) |

### Bloco 5 — Access Control

#### Google Cloud = coleção de APIs

Toda interação com GCP é via **chamadas de API**:
- **Console web** (`console.cloud.google.com`)
- **gcloud CLI** (terminal)
- **Terraform** (IaC)
- **Client libraries** (no código)

Exemplo: `gcloud run deploy` → chamada para `https://run.googleapis.com`.

#### IAM — anatomia da Policy

```
IAM Policy (anexada a um recurso)
└── Policy Binding
    ├── Member (Identity)
    └── Role
        └── Permissions
```

**Regras:**
- Policy **sempre anexada a um recurso**
- Policy binding: 1 member → 1 role
- 1 member pode ter **múltiplos bindings** (múltiplas roles)
- 1 role contém múltiplas permissions

#### Tornar service público

**Combo necessário:**
- Member: **`allUsers`**
- Role: **`roles/run.invoker`**

```bash
# Via add-iam-policy-binding
gcloud run services add-iam-policy-binding my-service \
  --member="allUsers" \
  --role="roles/run.invoker"

# Via deploy (atalho)
gcloud run deploy my-service --allow-unauthenticated
```

**Pré-requisito:** ser Project Owner ou ter `roles/run.admin`.

**Para tirar acesso público** (lab GSP723 task 2):

```bash
gcloud run services remove-iam-policy-binding my-service \
  --region=us-central1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

A flag `--no-allow-unauthenticated` **só funciona em `deploy`**, não em `update`. Em `update`, use `remove-iam-policy-binding`.

#### Comandos-chave de IAM

| Ação | Comando |
|------|---------|
| Adicionar role em service | `gcloud run services add-iam-policy-binding my-service --member=M --role=R` |
| Remover role em job | `gcloud run jobs remove-iam-policy-binding my-job --member=M --role=R` |
| Project-level (todos services) | `gcloud projects add-iam-policy-binding my-project --member=M --role=R` |

#### Ingress Settings — controle de rede (independente do IAM)

| Setting | Permite |
|---------|---------|
| **All** (default) | Internet via `*.run.app` ou domínio custom |
| **Internal and Cloud Load Balancing** | Internal + External HTTP(S) LB (não internet direto) |
| **Internal** (mais restritivo) | Internal HTTP(S) LB, VPC SC perimeter, VPC mesmo projeto, Cloud Tasks/Eventarc/Pub-Sub/Workflows |

**VPC Service Controls (VPC SC):** perímetro que protege contra **data exfiltration**. Aplica-se ao `run.app` URL e domínios custom.

Em `Internal`, tráfego de serviços GCP no mesmo projeto/perímetro **fica dentro da rede do Google** mesmo via `run.app`. Internet NUNCA atinge `run.app` nem domínio custom.

#### Serverless VPC Access (para acessar recursos internos)

**Por que:** Cloud Run é serverless e fica fora da sua VPC. Para acessar VMs/Memorystore/Cloud SQL via **IP interno**, precisa de um **Serverless VPC Access Connector**.

**Setup (4 passos):**
1. Habilitar Serverless VPC Access API
2. Criar connector
3. Anexar a VPC + região
4. Configurar service/job para usar o connector

```bash
gcloud run deploy my-service --image my-image \
  --vpc-connector my-connector
```

**Requisitos críticos:**

| Item | Regra |
|------|-------|
| **Região** | Connector tem que estar na **mesma região** do service |
| **CIDR** | **/28 não usado** ou non-overlapping /28 |
| **Subnet** | Usada **exclusivamente pelo connector** |

Para internal services, setar **todo o egress** para usar o VPC connector. Pode restringir o connector via firewall rules.

### Bloco 6 — Service Identity and Authentication

#### Tipos de identidades no IAM

| Categoria | Tipo | Uso |
|-----------|------|-----|
| **Human** | Google Account | login pessoal |
| **Human** | Group / Domain | grupos Workspace |
| **Machine** | **Service Account** | apps, VMs, Cloud Run, etc. |
| **All users** | `allUsers` | acesso público (anônimo) |

#### Service Account — anatomia

| Atributo | Valor |
|----------|-------|
| Identificador | **email único** |
| Senha | NÃO tem |
| Login browser | IMPOSSÍVEL |
| Impersonation | Outros podem agir em seu nome |
| Workspace domain | NÃO faz parte (mas pode ir em grupos) |

#### Service Accounts em Cloud Run

- **TODA service/job tem uma SA** = **service identity**
- **Default:** **default Compute Engine service account** com role **Editor** (muito amplo)
- **Best practice:**
  - **User-managed SA** dedicada
  - **Uma SA por service**
  - **Permissões mínimas** (least privilege)

#### Tokens — diferença crítica

| Token | Uso | Cenário |
|-------|-----|---------|
| **OAuth 2.0 Access Token** | Chamadas para **APIs Google** (Pub/Sub, BQ, Storage...) | App → API Google |
| **OIDC ID Token** | **Service-to-service Cloud Run** | Cloud Run A → Cloud Run B |

**OpenID Connect (OIDC):** protocolo de identidade baseado em OAuth 2.0. Verifica identidade do cliente.

#### Fluxo: Cloud Run → API Google (ex: Pub/Sub)

```
App + Client Library
    ↓ usa
Service Account (runtime)
    ↓ obtém
OAuth 2.0 Access Token
    ↓ API call com token
IAM verifica token + policy do Pub/Sub Topic
    ↓
Permite OU Rejeita
```

A client library **automaticamente** obtém o access token via SA — você não gerencia credenciais.

#### Fluxo: Cloud Run → Cloud Run (síncrono, service-to-service)

```
Calling Service (com SA)
    ↓ obtém
ID Token (OIDC) assinado pelo Google
    ↓ HTTP request com Authorization: Bearer <ID_TOKEN>
Receiving Service
    ↓ IAM valida token
    ↓ Verifica policy: SA tem roles/run.invoker no receiving?
    ↓
Permite OU Rejeita
```

**Setup:**
```bash
gcloud run services add-iam-policy-binding RECEIVING_SERVICE \
  --member='serviceAccount:CALLING_SERVICE_IDENTITY' \
  --role='roles/run.invoker'
```

#### Async vs Sync entre Cloud Run services

| Tipo | Ferramentas |
|------|-------------|
| **Assíncrona** | **Cloud Tasks, Pub/Sub, Cloud Scheduler, Eventarc** |
| **Síncrona** | HTTP direto + IAM (`roles/run.invoker` + OIDC ID token) |

### Bloco 7 — Resource Hierarchy

```
Organization (raiz, opcional)
    ↓
Folder (opcional, aninhável)
    ↓
Project (base obrigatória)
    ↓
Resources (Cloud Run, Pub/Sub topic, VM, etc.)
```

| Nível | Característica |
|-------|---------------|
| **Organization** | Nó raiz. Visibilidade/controle central. Não tem parent. |
| **Folder** | Opcional. Aninhável. Mapeia departamentos/times/BUs. |
| **Project** | **Base obrigatória.** Necessário para: criar recursos, usar APIs, gerenciar permissões, billing. |
| **Resource** | Recursos individuais (todo recurso pertence a 1 project). |

**Regra:** cada recurso tem **exatamente 1 parent** (exceto Organization).

#### Herança de policy bindings (conceito-chave)

**Bindings em nível superior são herdadas pelos níveis inferiores.**

Exemplo: `roles/pubsub.publisher` no project → user pode publicar em **TODOS os topics** desse project.

#### Effective IAM Policy

```
Effective Policy do recurso =
    bindings do recurso
    + bindings do project
    + bindings do folder
    + bindings da organization
```

**Regra de ouro:** permissões herdadas de cima **NÃO podem ser removidas em níveis inferiores**. Se quer restringir, **sobe na hierarquia** e remove lá (ou usa IAM Deny policies, recurso avançado).

#### Quando adicionar binding em cada nível

| Nível | Use quando |
|-------|------------|
| **Organization** | Permissões globais (admins, security, billing) |
| **Folder** | Permissões por BU/time |
| **Project** | Permissões padrão da aplicação inteira |
| **Recurso individual** | Granularidade fina, least privilege |

### Bloco 8 — Principle of Least Privilege

#### Tipos de IAM Roles

| Tipo | Características | Exemplos | Quando usar |
|------|----------------|----------|-------------|
| **Basic** | Muito amplas, todos services | Owner, Editor, Viewer | NUNCA em produção |
| **Predefined** | Granulares, service-específicas, gerenciadas pelo Google | Cloud Run Admin, Pub/Sub Publisher, Cloud Tasks Enqueuer | **Default recomendado** |
| **Custom** | Lista definida por você | Suas próprias | Quando predefined não serve |

#### O perigo da Default Service Account

Cloud Run sem `--service-account` usa **Default Compute Engine SA** com role **Editor** no project.

**Risco:** por causa da **herança**, Editor no project = acesso read/write em **TODOS** os recursos:
- Cloud SQL instances
- Cloud Storage buckets
- Outros Cloud Run services
- Etc.

Se o código tiver vulnerabilidade → atacante destrói/exfiltra tudo do project.

#### Aplicando least privilege — 3 passos

1. Criar nova SA para o Cloud Run service
2. Configurar essa SA como service identity (`--service-account` no deploy)
3. Adicionar policy bindings com roles predefined/custom **APENAS nos recursos necessários**

#### SA nova começa SEM permissões

Recém-criada → nenhum binding → toda chamada IAM rejeitada → precisa adicionar bindings explicitamente **no recurso destino**.

```bash
gcloud pubsub topics add-iam-policy-binding my-topic \
  --member="serviceAccount:my-sa-email" \
  --role="roles/pubsub.publisher"
```

#### Default vs User-managed (comparação)

| | **Default Compute Engine SA** | **User-managed (per-service)** |
|---|------------------------------|--------------------------------|
| Role | Editor (basic, ampla) | Predefined/custom (granular) |
| Escopo | TODO o project | Apenas recursos específicos |
| Risco | Alto | Baixo |
| Quando | Dev/protótipo | **PRODUÇÃO** |

#### Onde configurar a SA no Cloud Run

Via Console, gcloud, YAML ou Terraform no create/update/deploy:

```bash
gcloud run deploy my-service --image=IMAGE \
  --service-account=MY_SA_EMAIL
```

### Bloco 9 — Secrets and Environment Variables

#### Environment Variables

- **Key-value pairs** injetadas no container, lidas em runtime
- Setáveis em: deploy, create/update, nova revisão
- Via: Console, gcloud, YAML, Terraform
- Default do `ENV` no Dockerfile é **sobrescrito** pelo valor setado no Cloud Run
- Existem **vars reservadas** (container runtime contract)

```bash
gcloud run deploy my-service --image=IMG \
  --update-env-vars FOO=bar,BAZ=boo
```

**Leitura no código:**
| Linguagem | Como |
|-----------|------|
| Python | `os.environ.get("key")` |
| Node.js | `process.env.key` |
| Java | `System.getenv("key")` |

#### Secrets via Secret Manager

Para info sensível (API keys, senhas, tokens), use Secret Manager — **nunca hardcode em env var**.

**Estrutura:**
- Metadados (replication locations, labels, permissions)
- **Secret versions** = armazenam o dado real (string ou binary blob)

#### Duas formas de acessar secret (DIFERENÇA CRÍTICA p/ prova)

| Método | Comportamento | Quando usar |
|--------|---------------|-------------|
| **Volume mount** | Arquivo no container, re-lê do Secret Manager a cada read | **`latest` OK** |
| **Environment variable** | Resolvida no startup do container | **PINAR versão específica** (não usar `latest`) |

```bash
# Como volume (arquivo)
gcloud run deploy my-service --image=IMG \
  --update-secrets=SECRET_FILE_PATH=my_secret:VERSION

# Como env var
gcloud run deploy my-service --image=IMG \
  --update-secrets=ENV_VAR_NAME=my_secret:VERSION
```

#### Permissão para acessar secrets

SA do service precisa da role **`roles/secretmanager.secretAccessor`** no secret:

```bash
gcloud secrets add-iam-policy-binding my-secret-id \
  --member="serviceAccount:sa-email" \
  --role="roles/secretmanager.secretAccessor"
```

### Cloud Run Functions

| Capability | Detalhe |
|-----------|---------|
| **Tipo** | Serverless function execution |
| **Trigger** | Eventos do Cloud (HTTPS, Pub/Sub, Cloud Storage, Eventarc) |
| **Scaling** | Por número de eventos |

```bash
gcloud functions deploy NAME \
  --trigger-event google.storage.object.finalize \
  --trigger-resource BUCKET_NAME \
  --runtime python311
```

| Event | Válido? | O que dispara |
|-------|---------|---------------|
| **`google.storage.object.finalize`** | ✅ Único válido | Write para Cloud Storage **completo** |
| `google.storage.object.create` | ❌ | Não existe |
| `google.storage.object.change` | ❌ | Não existe |
| `google.storage.object.add` | ❌ | Não existe |

#### 10.1 — Adequação ao Cloud Run (fit check)

Antes de partir pra implementação, o curso pergunta: **sua app é uma boa candidata pra Cloud Run?**

Sua aplicação serve se **TODOS** os critérios bater:

| Critério | Detalhe |
|----------|---------|
| **Protocolo** | HTTP, HTTP/2, WebSockets, gRPC — ou job que executa até completar |
| **File system** | Não exige persistente local; aceita **efêmero in-memory** OU **network FS** (Filestore/FUSE) |
| **Concorrência** | Lida com **múltiplas instâncias simultâneas** (não pode ter estado local crítico) |
| **Recursos** | Máx **8 vCPU + 32 GiB** por instância |
| **Linguagem** | Containerizada **OU** Go/Java/Node.js/Python/.NET (Buildpacks) |

**Armadilha de prova:** se a app precisa de **disco persistente local** (DB local, FS read-write entre requests) → **não cabe em Cloud Run**. Vai pra Compute Engine ou GKE Standard com PV.

#### 10.2 — Dev workflow: as 2 rotas até o deploy

**Rota A — Container-first (você builda):**
```
Source → Dockerfile → docker build → docker push → gcloud run deploy --image=...
```

**Rota B — Source-based (Buildpacks builda pra você):**
```
Source → gcloud run deploy --source .
         └─ se tem Dockerfile, usa; senão Buildpacks decide
```

**Armadilha:** Buildpacks só suporta **Go, Java, Node.js, Python, .NET Core, Ruby**. Outras linguagens → precisa Dockerfile.

#### 10.3 — Container runtime contract (regras pro container rodar)

Coisas que o container **DEVE** respeitar pra funcionar em Cloud Run:

1. **Linux 64-bit executável** dentro da imagem (não roda nada compilado pra Windows/arm32)
2. **Formatos aceitos**: Docker Manifest V2 (Schema 1 e 2) + **OCI**
3. **Service:** escutar na porta `$PORT` (default 8080) — responder dentro do **request timeout** (máx **1h**)
4. **Job:** sair com **exit code 0** (sucesso) ou **non-zero** (falha). **NÃO escutar porta**
5. **NÃO implementar TLS** no container — Cloud Run termina HTTPS/gRPC e proxia HTTP/1 ou gRPC pra você. Pra HTTP/2 use **HTTP/2 cleartext**

**Cenário real:** muito desenvolvedor traz container Go que escuta `:80` ou hardcoded `:3000`. No Cloud Run, **leia da env `PORT`** sempre — Cloud Run pode mudar e você quebra.

```javascript
const port = process.env.PORT || 8080;  // ✅ correto
app.listen(port);
```

#### 10.4 — Execution environments (1st vs 2nd gen)

Cloud Run tem **dois execution environments** rodando services/jobs:

| Critério | 1st generation | 2nd generation |
|----------|----------------|----------------|
| Default para | **Services** (configurável) | **Jobs** (fixo, não muda) |
| Cold start | **Mais rápido** | Mais lento |
| Compatibilidade Linux | **Emulação** (maioria das syscalls) | **Linux completo** (todas syscalls, namespaces, cgroups) |
| Network FS | ❌ | ✅ |
| CPU/network | Padrão | **Mais rápido**, melhor com packet loss |
| Memória < 512 MiB | ✅ recomendado | OK também |
| CPU-intensive | ⚠️ | ✅ |

**Quando trocar pra 2nd gen num service:**
- Você usa **system calls** que o 1st gen não emula (ex: `io_uring`, namespaces específicos)
- Precisa de **network file system** (Filestore, NFS)
- Workload **CPU-bound**

```bash
gcloud run deploy my-service --execution-environment=gen2
```

**Armadilha:** **jobs SEMPRE rodam em 2nd gen**, não tem como trocar.

#### 10.5 — File system e armazenamento

```
┌──────────────────────────┐         ┌─────────────────────┐
│      Container           │         │  Cloud data storage │
│  ┌────────────────────┐  │ persist │  Cloud Storage      │
│  │  Application code  │ ─┼────────►│  Memorystore        │
│  └────────────────────┘  │         │  Cloud SQL          │
│  ┌────────────────────┐  │         │  Firestore          │
│  │  In-memory FS      │ ─┼─eph────►│  Spanner            │
│  │  (writable, RAM)   │  │         │  BigQuery           │
│  └────────────────────┘  │         │  Filestore (NFS)    │
└──────────────────────────┘         └─────────────────────┘
```

**Regras:**
- **Escrever em arquivo dentro do container = consome memória** alocada (porque é tmpfs)
- **Dados NÃO persistem** após instance encerrar
- Use o FS in-memory como **cache**, **dados descartáveis por request**, **config temporária**

**Persistência além da instância:**
- **Filestore** (NFS gerenciado) ou self-managed NFS → exige **2nd gen execution environment**
- **Cloud Storage FUSE** → monta bucket como FS no container
- **Client libraries** → conecta direto em Firestore/SQL/Spanner/etc (jeito mais simples e recomendado)

#### 10.9 — Onde fica a imagem (registry)

```
┌──────────┐  push  ┌──────────────────────┐  pull  ┌──────────┐
│Developer │ ─────► │  Artifact Registry   │ ────►  │ Cloud Run│
└──────────┘        │  (Docker repository) │        └──────────┘
                    └──────────────────────┘
```

**Registries suportados:**
- ✅ **Artifact Registry** — recomendado (`*-docker.pkg.dev/...`)
- ✅ **Docker Hub**
- ✅ Externos (JFrog, Nexus, GitHub Container Registry) **via AR remote repository** (proxy)
- ✅ Imagem em **outro projeto GCP** se tiver IAM correto

**Artifact Registry é universal** — armazena Docker, npm, Maven, PyPI, apt, yum no mesmo serviço.

**Armadilha de prova:** Container Registry (`gcr.io`) **está deprecated**. URL nova: `REGION-docker.pkg.dev/PROJECT/REPO/IMAGE`.

**Característica crítica:** **Cloud Run copia a imagem pra storage interno** no deploy. Consequências:
1. **Tamanho da imagem não afeta cold start** — 2GB sobe tão rápido quanto 100MB
2. Se você **apagar a imagem do AR depois do deploy**, o service **continua rodando** com a cópia
3. Mas perde a capacidade de **redeploy** dessa revision específica

#### 10.10 — Criar/atualizar um service — IAM necessário

Pra fazer deploy você precisa de **uma** das combinações:
- `Owner`
- `Editor`
- `Cloud Run Admin` **+** `Service Account User` (combinados)
- Custom role com permissões equivalentes

**Por que `Service Account User`?** Porque o deploy "anexa" uma SA ao service (que vira a identidade do service) — você precisa de permissão pra **act as** aquela SA.

**Armadilha clássica:** ter só `Cloud Run Admin` sem `Service Account User` → deploy falha com `Permission iam.serviceAccounts.actAs denied`.

**Regra clássica:** **1 service = 1 container image** (sem sidecar no modelo original; hoje Cloud Run suporta sidecars, mas o curso ensina o modelo base).

#### 10.11 — Service configuration — 8 elementos que disparam revision

Mudar **qualquer um** desses cria nova revision (**mesmo sem mudar a imagem**):

| # | Configuração | Flag/exemplo |
|---|--------------|--------------|
| 1 | Container image URL | `--image us-central1-docker.pkg.dev/.../v2` |
| 2 | Entrypoint + args | `--command="/app/server" --args="--port=8080"` |
| 3 | Secrets + env vars | `--set-env-vars FOO=bar`, `--set-secrets DB_PASS=db-pass:latest` |
| 4 | Request timeout | `--timeout=300` (máx 3600s/1h) |
| 5 | Concurrency | `--concurrency=80` |
| 6 | CPU/memory limits | `--cpu=2 --memory=1Gi` |
| 7 | Scaling boundaries | `--min-instances=1 --max-instances=50` |
| 8 | GCP config | `--service-account=...`, `--vpc-connector=...` |

**Comportamento:** novas revisions **herdam** o config da anterior. Você passa só o que mudou.

**Armadilha de prova:** "só mudei a memória, por que criou revision?" → porque **toda mudança de config cria revision**, não só mudança de imagem. Isso é proposital — permite rollback de qualquer alteração, não só de código.

#### 10.12 — Revision = cópia imutável (imagem + config)

```
┌──────────┐ watch  ┌────────┐  makes immutable copy   ┌─────────────────────┐
│ Service  │changes │Cloud   │ ────────────────────►   │      Revision       │
│ resource │ ─────► │ Run    │                         │ ┌─────────────────┐ │
└──────────┘        └────────┘                         │ │ Container image │ │
                                                       │ └─────────────────┘ │
                                                       │         +           │
                                                       │ ┌─────────────────┐ │
                                                       │ │ Service config  │ │
                                                       │ └─────────────────┘ │
                                                       └─────────────────────┘
```

**Imutável** = não dá pra editar uma revision existente; só adicionar novas.

**Por que é poderoso:**
- **Rollback trivial** — aponta tráfego pra revision antiga
- **Auditoria** — sabe exatamente o que estava rodando em cada momento histórico
- **Canary** — manda 10% pra revision nova sem mexer na antiga

#### 10.13 — Serving traffic — fluxo de transição

**Fase 1 — Nova revision subindo:**
```
┌──────────┐ Starting   ◄── Cloud Run escala a nova até match
│ Revision │            ◄── espera ficar ready
└──────────┘
┌─────────────────────┐
│ ▓▓▓ Serving traffic │  ◄── Revision atual ainda atende
└─────────────────────┘
```

**Fase 2 — Tráfego migra:**
```
┌─────────────────────┐
│ ▓▓▓ Serving traffic │  ◄── Nova revision ativa
└─────────────────────┘
┌─────────┐ Idle       ◄── Antiga vira idle
└─────────┘
   ▼ scale to zero
```

**Default:** quando a nova fica ready, **100% imediato** vai pra ela (zero downtime — antiga continua até a nova estar pronta).

**Gradual rollout — `--no-traffic`:**
```bash
gcloud run deploy my-svc --image=... --no-traffic
gcloud run services update-traffic my-svc --to-revisions=my-svc-00042=10  # 10% canary
gcloud run services update-traffic my-svc --to-revisions=my-svc-00042=50  # 50%
gcloud run services update-traffic my-svc --to-latest                     # promover 100%
```

---

#### 10.14 — Pinning, tagging e splitting

**Pinning** — fixa 100% do tráfego em uma revision específica:
```bash
gcloud run services update-traffic my-svc --to-revisions=my-svc-00041=100
```
Com pinning ativo, **novas revisions deployadas NÃO recebem tráfego automaticamente** — precisa mudar routing explicitamente.

**Tag** — apelido + URL própria pra testar revision sem expor pra produção:
```bash
gcloud run deploy my-svc --image=... --tag=green --no-traffic
# URL: https://green---my-svc-xyz-uc.a.run.app  (note os 3 hifens)
```

**Splitting** — canary com percentuais:
```bash
gcloud run services update-traffic my-svc \
  --to-revisions=my-svc-00043=10,my-svc-00042=90
```

**Comportamento crítico do split:** mudanças **não são instantâneas** — requests em andamento terminam onde começaram (graceful, sem drop).

**Session affinity:** por default, mesmo cliente pode ir pra container instances diferentes. Pra forçar stickiness (best-effort):
```bash
gcloud run services update my-svc --session-affinity
```

**Armadilha:** split + session affinity = cliente fica "preso" tanto à revision quanto à instance. Pode desviar do percentual exato (50/50 vira 60/40 na prática).

#### 10.15 — Integração com serviços Google Cloud (l3)

```
┌──────────────────────────┐    API call    ┌─────────────┐
│  Cloud Run Service       │ ─────────────► │ Google Cloud│
│  ┌────────────────────┐  │                │ Pub/Sub     │
│  │ Container          │  │                │ Firestore   │
│  │  Application code  │  │  Access/ID     │ Memorystore │

│  └────────────────────┘  │                │ etc.        │
```

**Como funciona:** client library pega a SA do service automaticamente → gera token transparente → faz a chamada autenticada. Você não armazena chave no código.

#### 10.16 — Conectar a Memorystore (Redis)

**Problema:** Cloud Run vive fora da VPC; Redis tem IP privado dentro da VPC. **Como conversam?** → **Serverless VPC Access connector**.

```
┌────────────┐   ┌─────────────────────┐   ┌──────────┐
│ Cloud Run  │──►│ Serverless VPC      │──►│ Redis    │
│ (serverless)│   │ Access connector    │   │ (VPC)    │
└────────────┘   │ (/28, mesma region) │   └──────────┘
                 └─────────────────────┘
```

**3 passos:**
1. Descobrir a **VPC autorizada** do Redis
2. Criar **Serverless VPC Access connector** na **mesma region** do service
3. **Anexar** o connector à VPC do Redis

```bash
gcloud run deploy my-svc \
  --image=my-image \
  --region=us-central1 \
  --vpc-connector=my-connector \
  --set-env-vars REDISHOST=10.0.0.5,REDISPORT=6379
```

**Cloud Run Integrations** = atalho UI/gcloud que automatiza esses 3 passos. Atualmente suporta:
- Mapear **custom domain** → service
- Conectar service → **Memorystore for Redis**

#### 10.17 — Trigger por Pub/Sub (push subscription)

```
Client ──publish──► Pub/Sub Topic ──► Push subscription
                                            │
                                            ▼ HTTPS request (com OIDC ID token)
                                  ┌────────────────────┐
                                  │ Cloud Run Service  │
                                  └────────────────────┘
```

**5 passos pra integrar:**
1. Criar Pub/Sub **topic**
2. Adicionar código no service pra **extrair mensagem** do HTTP body
3. Responder com **status code apropriado** (200/204 = ack; 4xx/5xx = retry)
4. Criar **SA com `roles/run.invoker`** sobre o service
5. Criar **push subscription** apontando pro endpoint do service, com a SA do passo 4

**Regras críticas:**
- **Ack deadline: 600s** (10min máx) — sem resposta = retry automático
- Service precisa ser **idempotente** (duplicate delivery é regra, não exceção)
- Endpoint **NÃO precisa ser público** — pode ser privado com `--no-allow-unauthenticated`, a SA da push subscription cuida da auth via OIDC ID token

**Códigos HTTP que importam:**
| Code | Significado pro Pub/Sub |
|------|------------------------|
| **200, 204** | Mensagem processada → ack (não reentrega) |
| **400, 500** | Falha → retry (reentrega com backoff) |
| Sem resposta em 600s | Timeout → retry |

#### 10.18 — Conectar a Cloud SQL — 2 caminhos

**Caminho A — IP público (default):**
```
┌────────────┐ Cloud SQL Auth Proxy (TLS + IAM)  ┌──────────┐
│ Cloud Run  │ ──────────────────────────────►   │ Cloud SQL│
│ Service    │                                    │ (pub IP) │
└────────────┘                                    └──────────┘
```

```bash

  --add-cloudsql-instances=PROJECT:REGION:INSTANCE
```

SA precisa: **`roles/cloudsql.client`** (ou `cloudsql.admin`).

**Caminho B — IP privado (recomendado prod):**
```
┌────────────┐ Serverless VPC Access  ┌──────────┐
│ Cloud Run  │ ──────────────────────►│ Cloud SQL│
│ Service    │                         │ (priv IP)│
└────────────┘                         └──────────┘
```

#### 10.19 — Boas práticas de conexão Cloud SQL

1. **Credenciais sempre via Secret Manager** — nunca env var hardcoded
   ```bash
   gcloud secrets create db-pass --data-file=-
   gcloud run deploy my-svc --set-secrets=DB_PASS=db-pass:latest
   ```
2. **Connection pool obrigatório** — limite hard de **100 conexões/service** ao DB
3. **Reconexão automática** via biblioteca cliente (pg-pool, HikariCP, SQLAlchemy)
4. **Cold start + pool** = atenção: 50 instances novas = 50 pools × N connections. Configure `max` por pool em função de `--max-instances`

### Quiz Módulo 1 — Fundamentals of Cloud Run

Pontos de atenção do quiz:

### Lab GSP723 — Service Identity and Authentication

Lab prático que valida toda a teoria de IAM/SA/least privilege em Cloud Run.

#### Task 1-2 — Remover acesso público

Cenário: existe `billing-service` deployado com `--allow-unauthenticated` (combo `allUsers` + `roles/run.invoker`). Time de segurança pediu para remover.

**Pegadinha:** `--no-allow-unauthenticated` SÓ existe em `gcloud run deploy`, NÃO em `update`. Solução em `update`:

```bash
gcloud run services remove-iam-policy-binding billing-service \
  --region=us-central1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

#### Task 3 — Redeploy seguro

```bash
gcloud run services delete billing-service
gcloud run deploy billing-service \
  --image gcr.io/qwiklabs-resources/gsp723-parking-service \
  --region $LOCATION \
  --no-allow-unauthenticated
```

Curl sem autenticação → **HTTP 403 Forbidden**. Esperado.

#### Task 4 — Invocar com autenticação (OIDC ID token)

Passos:

1. Criar SA `Billing Initiator` no Console com role **Cloud Run Invoker** (no project!)
2. Capturar email da SA:
   ```bash
   BILLING_INITIATOR_EMAIL=$(gcloud iam service-accounts list \
     --filter="Billing Initiator" --format="value(EMAIL)")
   ```
3. Gerar key file:
   ```bash
   gcloud iam service-accounts keys create key.json \
     --iam-account=${BILLING_INITIATOR_EMAIL}
   ```
4. Ativar a SA no gcloud:
   ```bash
   gcloud auth activate-service-account --key-file=key.json
   ```
5. Invocar com ID token:
   ```bash
   curl -i -X POST -H "Content-Type: application/json" \
     -H "Authorization: Bearer $(gcloud auth print-identity-token)" \
     $BILLING_SERVICE_URL -d '{"userid": "1234", "minBalance": 500}'
   ```
   → **HTTP 204 No Content** (sucesso, app Express não retorna body)

#### Task 5 — Least privilege na prática

Lab demonstra que **dar role no nível do project é privilégio EXCESSIVO**:

1. Deploy de um segundo service `billing-service-2`
2. Mesma SA `Billing Initiator` invoca o segundo service também → funciona por **herança** do binding no project
3. Solução: **mover o binding do project para o service específico**:

```bash
# Remover no project
gcloud projects remove-iam-policy-binding $GOOGLE_CLOUD_PROJECT \
  --member=serviceAccount:${BILLING_INITIATOR_EMAIL} \
  --role=roles/run.invoker

# Adicionar apenas no billing-service
gcloud run services add-iam-policy-binding billing-service \
  --region=$LOCATION \
  --member=serviceAccount:${BILLING_INITIATOR_EMAIL} \
  --role=roles/run.invoker \
  --platform=managed
```

Resultado:
- `billing-service` → **HTTP 204** (SA tem binding direto)
- `billing-service-2` → **HTTP 403** (SA não tem mais binding no project, nem no service-2)

#### Demonstrações poderosas do least privilege

A SA `Billing Initiator` (depois de ajustada) NÃO consegue:
- Listar service accounts (`iam.serviceAccounts.list` negado)
- Listar Cloud Run services (`run.services.list` negado)
- Invocar `billing-service-2`

Mas CONSEGUE:
- Invocar `billing-service` (única permissão real)

Isso é **least privilege absoluto** — SA literalmente useless para qualquer coisa além da tarefa específica.

#### Conceitos validados pelo lab

| Conceito | Como apareceu |
|----------|---------------|
| `--allow-unauthenticated` = `allUsers + roles/run.invoker` | Removido via `remove-iam-policy-binding` |
| Service privado precisa de OIDC ID token | `gcloud auth print-identity-token` |
| Herança em nível de project = privilégio excessivo | SA invocou service-2 sem permissão explícita |
| Binding em nível de recurso = least privilege | Mover do project para o service |
| SAs no Cloud Run usam OIDC para auth, não OAuth | `print-identity-token`, não `print-access-token` |

### Lab GSP724 — Pub/Sub + Cloud Run (Critter Junction)

**Cenário:** time de dev da Critter Junction quer integrar microservices com **mensageria assíncrona resiliente**. Tentativas anteriores falharam por **alto acoplamento** (services precisavam conhecer uns aos outros) e por **perda de mensagens** quando algum service caía.

**Requisitos das user stories:**
| Ref | User Story |
|-----|------------|
| 1 | Mensageria resiliente — operações se recuperam sem intervenção manual |
| 2 | Services escalam sem causar instabilidade |
| 3 | Services rodam sem supervisão (gerenciados) |

**Decisão Pub/Sub vs Cloud Tasks:**
- **Pub/Sub** ✅ — pattern push para distribuição de eventos com pouco controle de timing
- **Cloud Tasks** ❌ — usado quando o produtor precisa **deferir/controlar timing** de webhook ou RPC específica

Critter Junction escolheu **Pub/Sub** (não precisa controlar timing fino, só distribuir eventos).

#### Arquitetura MVP

```
                      ┌─────────────────┐
   public internet ──►│  store-service  │  (produtor, --allow-unauthenticated)
                      │   Cloud Run     │
                      └────────┬────────┘
                               │ publish
                               ▼
                      ┌─────────────────┐
                      │  ORDER_PLACED   │  (Pub/Sub topic)
                      └────────┬────────┘
                               │ push subscription
                               │ (com OIDC token)
                               ▼
                      ┌─────────────────┐
                      │  order-service  │  (consumidor, --no-allow-unauthenticated)
                      │   Cloud Run     │
                      └─────────────────┘
```

#### Task 1 — Habilitar a Pub/Sub API

Lab pede um "ciclo disable → enable" pra validar que a API funciona. Console: Navigation menu → APIs & Services → Library → Pub/Sub API → Disable → Confirm → Enable.

**Pegadinha:** ao desabilitar uma API com **APIs dependentes**, o GCP pergunta confirmação dupla porque vai derrubar tudo que depende. Em produção isso é **destrutivo** — não faça em projeto real.

Equivalente gcloud:
```bash
gcloud services disable pubsub.googleapis.com --force  # --force pra cascade nas deps
gcloud services enable pubsub.googleapis.com
```

#### Task 2.1 — Deploy do producer (store-service, público)

```bash
gcloud services enable run.googleapis.com
LOCATION=us-east1
gcloud config set compute/region $LOCATION

gcloud run deploy store-service \
  --image gcr.io/qwiklabs-resources/gsp724-store-service \
  --region $LOCATION \
  --allow-unauthenticated
```

**Anatomia:**
| Token | O que faz |
|-------|-----------|
| `gcloud run deploy store-service` | Cria service `store-service` (se não existir) + Revision 1 |
| `--image gcr.io/qwiklabs-resources/...` | Imagem pré-pronta do Qwiklabs (não buildamos nada) |
| `--region us-east1` | Cloud Run é **regional** — escolhe onde rodar |
| `--allow-unauthenticated` | Combo automático: adiciona `allUsers` ao role `roles/run.invoker` → qualquer um na internet invoca |

#### Task 2.2 — Deploy do consumer (order-service, privado)

```bash
gcloud run deploy order-service \
  --image gcr.io/qwiklabs-resources/gsp724-order-service \
  --region $LOCATION \
  --no-allow-unauthenticated
```

**Diferença crítica:** `--no-allow-unauthenticated` → service **privado**, só invocável por accounts com `roles/run.invoker` explicitamente concedido.

#### Conceitos Pub/Sub recapitulados

| Campo | Definição |
|-------|-----------|
| **Topic** | Recurso nomeado pra onde publishers mandam mensagens |
| **Subscription** | Stream de mensagens de um topic específico, entregue ao subscriber |
| **Message** | Data + attributes (opcional) enviada pelo publisher |
| **Message attribute** | Par key-value que o publisher anexa (ex: `iana.org/language_tag=en`) |

**Use cases canônicos:**
- Balanceamento de workload (fila distribuída entre workers)
- Workflows assíncronos (order processing)
- Notificações de eventos (cadastro novo → vários downstream)
- Invalidação de cache distribuído
- Logging multi-destino
- Streaming de dados de sensores/devices
- Recuperação de falha zonal (subscribers em outras zonas pegam o que faltou)

#### Task 3 — Criar o Pub/Sub topic

```bash
gcloud pubsub topics create ORDER_PLACED
```

**Observação importante:** mensagens trafegam em Pub/Sub **base64-encoded** — o consumer precisa **decodificar** o `data` antes de processar.

```javascript
// Exemplo no order-service:
const pubsubMessage = req.body.message;
const data = Buffer.from(pubsubMessage.data, 'base64').toString();
const order = JSON.parse(data);
```

#### Task 4.1 — Criar a Service Account "Order Initiator"

```bash
gcloud iam service-accounts create pubsub-cloud-run-invoker \
  --display-name "Order Initiator"

# Confirmar:
gcloud iam service-accounts list --filter="Order Initiator"
```

Nesse ponto a SA existe mas é **inútil** — sem nenhum role atribuído.

#### Task 4.2 — Binding `Cloud Run Invoker` no order-service (least privilege!)

```bash
gcloud run services add-iam-policy-binding order-service \
  --region $LOCATION \
  --member=serviceAccount:pubsub-cloud-run-invoker@$GOOGLE_CLOUD_PROJECT.iam.gserviceaccount.com \
  --role=roles/run.invoker \
  --platform managed
```

**Por que no service e não no project:** **escopo mínimo** — essa SA só consegue invocar **um service específico**, nada mais. Validamos isso no GSP723.

#### Task 4.3 — Token Creator pra SA interna do Pub/Sub

```bash
PROJECT_NUMBER=$(gcloud projects list \
  --filter="qwiklabs-gcp" \
  --format='value(PROJECT_NUMBER)')

gcloud projects add-iam-policy-binding $GOOGLE_CLOUD_PROJECT \
  --member=serviceAccount:service-$PROJECT_NUMBER@gcp-sa-pubsub.iam.gserviceaccount.com \
  --role=roles/iam.serviceAccountTokenCreator
```

**O que está acontecendo aqui — passo conceitualmente mais difícil do lab:**

- `service-PROJECT_NUMBER@gcp-sa-pubsub.iam.gserviceaccount.com` é a **Google-managed SA** que o Pub/Sub usa internamente
- Pra **assinar OIDC ID tokens em nome de** `pubsub-cloud-run-invoker`, a SA interna do Pub/Sub precisa do role **`roles/iam.serviceAccountTokenCreator`** sobre o projeto
- Sem isso: a push subscription **não consegue gerar o token** que vai no header `Authorization: Bearer ...` da chamada HTTPS → order-service responde **403**

**Armadilha de prova:** muita gente lembra de dar `roles/run.invoker` à SA, mas **esquece** do `tokenCreator` na SA gerenciada do Pub/Sub. Sem os **dois bindings**, a integração quebra.

#### Task 5 — Criar a push subscription

```bash
ORDER_SERVICE_URL=$(gcloud run services describe order-service \
  --region $LOCATION \
  --format="value(status.address.url)")

gcloud pubsub subscriptions create order-service-sub \
  --topic ORDER_PLACED \
  --push-endpoint=$ORDER_SERVICE_URL \
  --push-auth-service-account=pubsub-cloud-run-invoker@$GOOGLE_CLOUD_PROJECT.iam.gserviceaccount.com
```

**Anatomia:**
| Flag | O que faz |
|------|-----------|
| `--topic ORDER_PLACED` | Subscribe a esse topic |
| `--push-endpoint=$ORDER_SERVICE_URL` | URL HTTPS do order-service — Pub/Sub vai fazer POST aqui |
| `--push-auth-service-account=...` | **SA que assina os tokens OIDC**. Pub/Sub gera token em nome dela e manda no header `Authorization` |

#### Task 6 — Teste end-to-end

`test.json` com payload de pedido (billing + shipping + items).

```bash
STORE_SERVICE_URL=$(gcloud run services describe store-service \
  --region $LOCATION \
  --format="value(status.address.url)")

curl -X POST -H "Content-Type: application/json" -d @test.json $STORE_SERVICE_URL
# → {"status":"success","order_id":"6pa5mmh"}
```

**Fluxo completo do que aconteceu:**

1. `curl` chega no `store-service` (público, sem auth)
2. Store gera `order_id`, monta mensagem, **publica em `ORDER_PLACED`** via client library do Pub/Sub
3. Pub/Sub recebe, persiste a mensagem no topic
4. **Push subscription** acorda, pega a SA `pubsub-cloud-run-invoker`, **gera OIDC ID token** (graças ao `tokenCreator`)
5. Faz **POST com `Authorization: Bearer <token>`** no `$ORDER_SERVICE_URL`
6. Cloud Run **valida o token**, checa se a SA tem `roles/run.invoker` no order-service → sim
7. `order-service` recebe o request, **decodifica `data` base64**, processa o order, responde **200/204**
8. Pub/Sub vê o ack → marca a mensagem como entregue

**Validação nos logs (Console):**
- `store-service` → filtro `ORDER ID` → mostra o ID gerado
- `order-service` → filtro `Order Placed` → mostra o mesmo ID, comprovando a entrega

#### Conceitos validados pelo lab

| Conceito | Como apareceu |
|----------|---------------|
| Service público vs privado | `--allow-unauthenticated` vs `--no-allow-unauthenticated` |
| Pub/Sub desacopla services | Store não conhece o order; só sabe do topic |
| Push subscription = HTTP POST com OIDC | `--push-auth-service-account` é a chave |
| **Dois bindings** necessários | `roles/run.invoker` na SA + `roles/iam.serviceAccountTokenCreator` na SA gerenciada do Pub/Sub |
| Mensagens são base64 | Consumer precisa `Buffer.from(data, 'base64')` |
| Idempotência implícita | Lab não trata, mas Pub/Sub pode duplicar — em prod, salvar `order_id` com unique constraint |
| Resiliência | Se order-service cair, Pub/Sub retém a mensagem e retry (até ack deadline + retry policy) |

#### Cheatsheet de comandos GSP724

```bash
# Setup
gcloud services enable run.googleapis.com pubsub.googleapis.com
LOCATION=us-east1
gcloud config set compute/region $LOCATION

# Producer público
gcloud run deploy store-service \
  --image gcr.io/qwiklabs-resources/gsp724-store-service \
  --region $LOCATION --allow-unauthenticated

# Consumer privado
gcloud run deploy order-service \
  --image gcr.io/qwiklabs-resources/gsp724-order-service \
  --region $LOCATION --no-allow-unauthenticated

# Topic
gcloud pubsub topics create ORDER_PLACED

# SA + bindings
gcloud iam service-accounts create pubsub-cloud-run-invoker --display-name "Order Initiator"

gcloud run services add-iam-policy-binding order-service \
  --region $LOCATION \
  --member=serviceAccount:pubsub-cloud-run-invoker@$GOOGLE_CLOUD_PROJECT.iam.gserviceaccount.com \
  --role=roles/run.invoker --platform managed

PROJECT_NUMBER=$(gcloud projects list --filter="qwiklabs-gcp" --format='value(PROJECT_NUMBER)')
gcloud projects add-iam-policy-binding $GOOGLE_CLOUD_PROJECT \
  --member=serviceAccount:service-$PROJECT_NUMBER@gcp-sa-pubsub.iam.gserviceaccount.com \
  --role=roles/iam.serviceAccountTokenCreator

# Push subscription
ORDER_SERVICE_URL=$(gcloud run services describe order-service --region $LOCATION --format="value(status.address.url)")
gcloud pubsub subscriptions create order-service-sub \
  --topic ORDER_PLACED \
  --push-endpoint=$ORDER_SERVICE_URL \
  --push-auth-service-account=pubsub-cloud-run-invoker@$GOOGLE_CLOUD_PROJECT.iam.gserviceaccount.com

# Teste
STORE_SERVICE_URL=$(gcloud run services describe store-service --region $LOCATION --format="value(status.address.url)")
curl -X POST -H "Content-Type: application/json" -d @test.json $STORE_SERVICE_URL
```

#### Armadilhas observadas

| Sintoma | Causa | Solução |
|---------|-------|---------|
| Push subscription criada, mas mensagens não chegam | Faltou `roles/iam.serviceAccountTokenCreator` na SA gerenciada do Pub/Sub | Adicionar o binding no PROJECT |
| `order-service` retorna 403 pras pushes | SA `pubsub-cloud-run-invoker` não tem `roles/run.invoker` no service | `add-iam-policy-binding` no service específico |
| Mensagem chega mas o JSON está estranho | Esqueceu de decodificar base64 | `Buffer.from(msg.data, 'base64').toString()` |
| Mensagem é entregue 2x ou mais | Service demorou >600s pra responder OU retornou 5xx | Idempotência via `order_id` + unique constraint |
| `curl` direto no order-service: 403 | Service é privado | Esperado — só Pub/Sub (SA com role) invoca |

## Cenário real

**Caso típico no trabalho:**

Você está montando um sistema de microservices em Cloud Run para um cliente de e-commerce:
- `frontend-service` — público (allow-unauthenticated)
- `cart-service` — privado, invocado só pelo frontend
- `payment-service` — privado, invocado só pelo cart
- `analytics-job` — Cloud Run **Job** disparado por Cloud Scheduler diariamente

**Como aplicar tudo que aprendemos:**

1. **Cada service tem sua user-managed SA** (não usar a default Compute Engine SA)
2. **frontend-service SA** → role `roles/run.invoker` no `cart-service` (binding no recurso, não no project!)
3. **cart-service SA** → role `roles/run.invoker` no `payment-service`
4. **payment-service SA** → role `roles/secretmanager.secretAccessor` no secret com a API key do gateway
5. **analytics-job SA** → roles em BigQuery (`roles/bigquery.dataEditor`) e Cloud Storage (`roles/storage.objectViewer`)
6. **Cloud Scheduler** que dispara o job tem sua própria SA com `roles/run.invoker` no job
7. Secrets via **volume mount** (sempre latest, sem precisar redeploy)
8. Env vars apenas para config não-sensível (URLs, feature flags)
9. **Ingress = Internal and Cloud Load Balancing** em todos os services privados
10. **Serverless VPC Connector** se algum service precisar acessar Cloud SQL via IP interno

Sem fazer nada disso, alguém com acesso ao default SA podia deletar Cloud SQL inteiro porque tem Editor herdado no project.

## Armadilhas de prova

| Pegadinha | Resposta |
|-----------|----------|
| "Container + serverless + custom packages" | **Cloud Run** (não App Engine) |
| "App Engine sem scale to zero" | **Flexible** |
| "App Engine sem custom packages" | **Standard** |
| "Trigger Cloud Storage para função" | `google.storage.object.finalize` |
| "Cloud Run Functions: concorrência" | **1 request/instância** — escala criando novas instâncias |
| "Cloud Run: concorrência" | **Múltiplas** requests por instância (default 80) |
| "Tornar service público" | `allUsers` + `roles/run.invoker` (ou `--allow-unauthenticated` no deploy) |
| "`--no-allow-unauthenticated` em update" | NÃO existe — use `remove-iam-policy-binding` |
| "Service-to-service auth" | OIDC ID token + `roles/run.invoker` no receiver |
| "Cloud Run → API Google (Pub/Sub etc.)" | OAuth 2.0 access token (gerenciado pela client library) |
| "Default SA em Cloud Run" | Default Compute Engine SA com role **Editor** (perigo!) |
| "Acessar VM por IP interno" | **Serverless VPC Access Connector** (/28, mesma região) |
| "Min-instances cobra?" | **SIM** — instâncias idle do min-instances são cobradas |
| "Idle normal cobra?" | **NÃO** — só o min-instances |
| "Concurrency default / máximo" | **80 / 1000** |
| "Max instances default / hard limit" | **100 / 1000** |
| "Memory default / máximo" | **512 MiB / 32 GiB** |
| "Tempo até idle" | **100 ms** sem requests |
| "Tempo após SIGTERM até parar" | **10 segundos** |
| "Porta default de probing" | **8080** (configurável) |
| "Service revision pode ser modificada?" | **NÃO** — revisões são imutáveis. Cada mudança cria nova revision |
| "Cloud Run services são regionais?" | **SIM** (não globais nem zonais) |
| "Secret latest em env var é OK?" | **NÃO** — env vars resolvem só no startup; use volume mount para latest |
| "Volume mount lê quando?" | A cada **read** (sempre da SM, suporta latest) |
| "Role para SA acessar Secret Manager" | `roles/secretmanager.secretAccessor` |
| "Permissão herdada do project pode ser removida no filho?" | **NÃO** — sobe na hierarquia ou usa Deny policy |
| "Tipos de roles, qual usar em produção?" | **Predefined** primeiro, Custom se não tiver. NUNCA Basic (Owner/Editor/Viewer) |
| "Async entre Cloud Run services" | Cloud Tasks, Pub/Sub, Cloud Scheduler, Eventarc |
| "Sync entre Cloud Run services" | HTTP direto + OIDC ID token + `roles/run.invoker` |
| "Container Registry (`gcr.io`)" | **Deprecated** — usar Artifact Registry (`*-docker.pkg.dev`) |
| "Cloud Run apaguei a imagem do AR, derrubou o service?" | **Não** — Cloud Run copia internamente, service segue rodando |
| "Tamanho da imagem afeta cold start?" | **Não** — storage interno é otimizado, 2GB sobe como 100MB |
| "Roles mínimos pra fazer deploy" | Owner OU Editor OU **(Cloud Run Admin + Service Account User)** |
| "Por que precisa de SA User pra deploy?" | Você "anexa" SA ao service → `iam.serviceAccounts.actAs` |
| "1 service = quantas imagens?" | **1** (modelo clássico; sidecars vieram depois) |
| "Mudei só `--memory`, criou revision?" | **Sim** — qualquer dos 8 configs cria revision |
| "Service tem downtime no deploy?" | **Não** — nova revision escala primeiro, depois migra |
| "Como deploy sem expor pra produção?" | `--no-traffic` + tag + URL `tag---svc.run.app` (3 hifens) |
| "Migrar 10% pra revision nova" | `update-traffic --to-revisions=NAME=10` |
| "Pinning ativo, deployei nova revision" | Nova **NÃO recebe tráfego automaticamente** |
| "Session affinity é garantido?" | **Não** — best-effort apenas |
| "Job em 1st gen execution env" | **Impossível** — jobs sempre 2nd gen |
| "Service quer network FS" | Mudar pra **2nd gen execution environment** |
| "App precisa de disco persistente local" | **Não cabe em Cloud Run** — vai pra GCE/GKE |
| "Pub/Sub push subscription quebra com 403" | Faltou `tokenCreator` na SA gerenciada `service-NUM@gcp-sa-pubsub` |
| "Mensagem Pub/Sub chega com data estranho" | Não decodificou **base64** (`Buffer.from(msg.data, 'base64')`) |
| "Pub/Sub ack deadline máximo" | **600s** (10 min) — depois disso, retry automático |
| "Cloud SQL: limite de conexões/service" | **100 por service** ao banco |
| "Cloud SQL: como passar credencial?" | Secret Manager via `--set-secrets`, NUNCA env var direta |
| "Cloud SQL público: como conecta?" | `--add-cloudsql-instances=PROJ:REGION:INST` + role `cloudsql.client` na SA |
| "Cloud Tasks vs Pub/Sub" | **Cloud Tasks** = controle de timing/webhook específico; **Pub/Sub** = distribuição de eventos genérica |
| "Cloud Run Integrations cobre o quê?" | Custom domain + Memorystore for Redis (atalho de setup) |
| "Default port" | **8080** — sempre leia de `$PORT`, é configurável |
| "Cloud Run espera o quê do container?" | Linux 64-bit, escuta porta, **SEM TLS no container** (Cloud Run termina) |
