## Conceitos

### Visão geral do módulo (Overview video)

**O que é Compute Engine?**

Compute Engine é o serviço do Google Cloud para criar e gerenciar **Virtual Machines (VMs)**. É a peça fundamental de infraestrutura — tudo mais (GKE, Dataproc, Cloud SQL) roda, no fundo, sobre Compute Engine.

Uma VM é similar a um computador físico, mas não idêntica. Ela abstrai o hardware real e oferece:
- **vCPU** — CPU virtual (pode ser compartilhada com outras VMs ou dedicada)
- **Memória RAM** — configurável de forma independente da CPU
- **Disco de armazenamento** — vários tipos e tamanhos
- **Endereço IP** — interno (sempre) e externo (opcional)

**Por que importa?** Toda vez que a prova mencionar controle de OS, dependências específicas, lift-and-shift ou customização profunda de ambiente, a resposta é Compute Engine. É o único serviço do GCP que dá acesso completo ao sistema operacional.

---

### O que é uma "instância" no GCP (Compute Engine VM instance)

Quando o GCP fala em **"instância"**, ele está falando em **uma única VM em execução** dentro do serviço Compute Engine. O termo é uma herança de virtualização: o "molde" é a definição (machine type + imagem + disco), e cada VM concreta criada a partir desse molde é uma **instância** desse molde.

Mais precisamente:

> Uma **instância** é uma máquina virtual hospedada na infraestrutura do Google, identificada por um nome único dentro de uma **zona**, rodando uma cópia de uma **imagem de SO** sobre uma combinação específica de **vCPU + RAM + disco(s) + interface(s) de rede**.

Três coisas que a definição esconde, mas a prova cobra:

1. **Instância é zonal.** Ela vive em **uma zona** (ex.: `us-central1-a`). Não existe VM "regional" — o que existe é grupo de VMs distribuídas em zonas (MIG regional). Se a zona cai, a instância cai com ela.
2. **Instância tem identidade.** Ela tem nome, ID numérico, e — por padrão — uma **service account** anexada. A identidade dela é o que define o que ela pode chamar via API (IAM).
3. **Instância é efêmera por design.** O Google explicitamente projetou o serviço para que você possa destruir e recriar VMs sem medo. Estado importante mora no disco persistente (que sobrevive à VM) ou em serviços externos (Cloud SQL, GCS, etc.).

---

### Anatomia de uma instância

Toda instância é a composição de **cinco partes obrigatórias** mais algumas opcionais. Entender essa composição é o que destrava as perguntas de prova.

```
INSTÂNCIA
├── 1. Machine type      (vCPU + RAM)            ← define o "tamanho"
├── 2. Boot disk         (imagem do SO)          ← define o "que roda"
├── 3. Network interface (VPC + subnet + IPs)    ← define o "onde se conecta"
├── 4. Service account   (identidade IAM)        ← define o "o que pode fazer"
├── 5. Zona              (us-central1-a, etc.)   ← define o "onde mora"
│
├── (opcional) Discos adicionais (persistent, local SSD)
├── (opcional) GPU/TPU anexada
├── (opcional) Metadata (startup-script, ssh-keys, labels)
├── (opcional) Tags de rede (para firewall)
└── (opcional) Availability policy (Spot, preemptible, scheduling)
```

**Por que isso importa para a prova:** quase toda DQ de Compute Engine pergunta sobre **uma dessas dimensões**. Se a questão fala em "qual o impacto de mudar X", traduza X para qual das 5+ dimensões está sendo tocada.

### Ciclo de vida de uma instância

Uma instância não está apenas "ligada" ou "desligada". Ela passa por estados bem definidos, e cada transição tem implicações de custo e de comportamento:

```
PROVISIONING  →  STAGING  →  RUNNING  →  STOPPING  →  TERMINATED
   (alocando      (subindo    (rodando    (desligando   (parada,
   recursos)      o SO)       normal)     ordenado)     mas existe)
                                                            │
                                                            └─ DELETED (sumiu)
```

| Estado | O que acontece | Você paga? |
|---|---|---|
| **PROVISIONING** | Google aloca CPU/RAM no host físico | Não (transitório) |
| **STAGING** | Boot disk anexado, SO carregando | Não (transitório) |
| **RUNNING** | VM ativa, respondendo | ✅ **Sim** — CPU/RAM + disco + rede |
| **STOPPING** | Shutdown em curso | Não (transitório) |
| **TERMINATED** | VM existe (config, disco, IP estático) mas não roda | ❌ **Não** paga CPU/RAM, mas **paga disco e IP estático reservado** |

**Insight crítico:** *Parar* uma VM (`gcloud compute instances stop`) ≠ *deletar* (`...instances delete`). Parar economiza CPU/RAM mas mantém disco, IP, configuração. Deletar apaga tudo a menos que o boot disk esteja com `--keep-disk` (ou auto-delete=false).

> **Armadilha de prova:** se a pergunta diz "como reduzir custos enquanto a VM não está em uso, mantendo configuração e dados" → resposta é **STOP** (não delete, não snapshot). Se for "queremos que ela suba só em horário comercial" → **scheduled instance start/stop** ou **Cloud Scheduler + gcloud**.

### Famílias de machine type (o "tamanho" da instância)

Machine type é a combinação **vCPU + RAM** que define o tamanho da instância. O GCP organiza em **famílias**, cada uma com um propósito:

| Família | Prefixo | Propósito | Quando usar |
|---|---|---|---|
| **General-purpose** | `E2`, `N2`, `N2D`, `N4`, `T2D` | Carga balanceada CPU/RAM | Default. Web servers, app servers, bancos pequenos. **`E2` = mais barato**. |
| **Compute-optimized** | `C2`, `C2D`, `C3`, `C4` | Mais CPU por vCPU | Games servers, HPC, simulações, encoding de mídia. |
| **Memory-optimized** | `M1`, `M2`, `M3`, `X4` | Muita RAM por vCPU | SAP HANA, bancos in-memory, analytics em RAM. |
| **Accelerator-optimized** | `A2`, `A3`, `G2` | GPU/TPU acoplada | Treino ML, inferência pesada, renderização. |
| **Storage-optimized** | `Z3` | Local SSD massivo | Bancos OLTP de alta IOPS, scratch para HPC. |

**Dentro de cada família, há tamanhos:**
- `standard` — proporção balanceada vCPU:RAM (geralmente 1:4 GB)
- `highmem` — mais RAM por vCPU (1:8 GB)
- `highcpu` — menos RAM por vCPU (1:1 ou 1:2)
- `mega-mem` / `ultramem` — extremos de RAM (M2/M3)

**Exemplo de nome completo:** `n2-standard-4`
- `n2` = família General-purpose, geração 2
- `standard` = razão balanceada
- `4` = 4 vCPUs (então: 4 vCPU + 16 GB RAM)

**Custom machine type:** se nenhum predefinido encaixa, você pode definir **N vCPUs + M GB RAM** sob medida (`--custom-cpu=6 --custom-memory=20GB`). Disponível nas famílias `E2`, `N1`, `N2`, `N2D`.

> **Armadilha de prova:**
> - "Workload precisa do **menor custo possível** e tolera CPU compartilhada" → **`e2-micro/small/medium`** (shared-core, mais barato).
> - "Workload precisa de **muita RAM** (banco in-memory)" → família **M** (memory-optimized).
> - "Treino de modelo ML grande" → família **A** (com GPU) ou TPU dedicada.
> - "Banco que precisa de IOPS altíssimo em disco local" → família **Z** (Z3 com Local SSD).

### Relação instância ↔ imagem ↔ disco

Esses três conceitos são frequentemente confundidos:

| Conceito | O que é | Persistente? | Reutilizável? |
|---|---|---|---|
| **Imagem (Image)** | "Foto" de um SO + software pré-configurado, somente leitura | ✅ (no Cloud Storage interno) | ✅ Cria N discos a partir dela |
| **Disco (Persistent Disk)** | Volume de dados anexado à VM | ✅ Sobrevive à VM | ⚠️ Anexa em 1 VM por vez (read-write) ou várias (read-only) |
| **Instância (VM)** | Hardware virtual em execução | ❌ Efêmera | — |

**Fluxo de criação de uma VM:**

```
1. Você escolhe uma IMAGEM        (ex.: debian-12, ubuntu-22-04-lts, windows-2022)
        ↓
2. Compute Engine cria um BOOT DISK copiando a imagem
        ↓
3. Anexa o boot disk a uma nova INSTÂNCIA
        ↓
4. Instância faz boot a partir do disco
```

**Tipos de imagem que a prova cobra:**
- **Public images** — Google fornece (Debian, Ubuntu, CentOS, RHEL, Windows Server). Gratuitas ou com licença embutida.
- **Custom images** — você cria a partir de um disco existente. Útil para padronizar baseline (apps + libs + config).
- **Marketplace images** — terceiros (Bitnami, etc.).

**Insight de prova:** se a questão pergunta "como garantir que todas as VMs criadas tenham o mesmo software base, mesmas configs de segurança e patches X aplicados" → resposta é **custom image** (não startup script — startup script roda toda vez que a VM sobe e é mais lento).

### Relação instância ↔ rede

Toda instância **precisa** de pelo menos uma **network interface (NIC)**, e essa NIC vive dentro de uma **subnet** de uma **VPC**.

```
INSTÂNCIA
   └── NIC (nic0)
         ├── VPC          (ex.: default ou minha-vpc)
         ├── Subnet       (ex.: subnet-us-central1, range 10.0.0.0/20)
         ├── IP interno   (ex.: 10.0.0.5) — obrigatório
         └── IP externo   (ex.: 34.123.45.67) — opcional
              └── pode ser efêmero ou estático (reservado)
```

- **IP interno:** atribuído automaticamente do range da subnet. Pode ser estático se você reservar.
- **IP externo:** opcional. Sem IP externo, a VM **não tem acesso direto à internet** (precisa de Cloud NAT para sair, ou IAP para entrar).
- Uma VM pode ter **múltiplas NICs** (`nic0`, `nic1`, ...) cada uma em uma **VPC diferente**. Útil para gateways/firewalls entre redes.

> **Armadilha de prova:** "VM precisa acessar a internet mas **sem ter IP público** (por segurança)" → resposta é **Cloud NAT** + VM sem IP externo, ou Private Google Access para acessar APIs Google.

### Acesso a VMs: quem pode entrar e como

**Princípio:** quem **criou a instância** tem privilégios **root** completos sobre ela. A partir daí o acesso muda conforme o SO.

#### Linux — SSH

- O criador da VM já tem capacidade de SSH automaticamente.
- O criador pode usar a **Cloud Console** (ou `gcloud`) para **conceder SSH capability** a outros usuários.
- Mecanismo por baixo: a chave pública SSH do usuário é injetada via **metadata** (instance-level ou project-level) ou via **OS Login** (que integra com IAM — preferido em ambientes com muita gente).

#### Windows — RDP

- O criador usa a Console para **gerar um username + senha** específicos para a VM Windows.
- Qualquer pessoa que conheça esse par usuário/senha consegue conectar via cliente **RDP (Remote Desktop Protocol)**.
- Não há "OS Login" para Windows do mesmo jeito que para Linux — o controle é por credenciais geradas.

#### Firewall rules necessárias

| Protocolo | Porta TCP padrão | Para que serve |
|---|---|---|
| **SSH** | `22` | Acesso Linux |
| **RDP** | `3389` | Acesso Windows |

> **Atalho importante:** se a VM está na **rede `default`**, **você NÃO precisa criar essas regras** — a rede default já vem com `default-allow-ssh` (TCP:22) e `default-allow-rdp` (TCP:3389) liberadas de qualquer origem (`0.0.0.0/0`). Em VPC customizada você cria as regras manualmente.

> **Armadilha de prova:** "VM Linux criada em VPC customizada e não consigo SSH" → verifique se há **firewall rule permitindo TCP:22** da sua origem para a tag/SA da VM. Esse é o erro mais comum em DQs.

### Estados da VM — ciclo de vida COMPLETO

A versão expandida do diagrama (incluindo estados que o curso só agora introduziu):

```
                    ┌──────────────┐
   create   ──────► │ PROVISIONING │ (reservando CPU/RAM/disco)
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐
                    │   STAGING    │ (anexando IPs, bootando imagem)
                    └──────┬───────┘
                           ▼
                    ┌──────────────┐ ──── reset ───┐
            ┌──────►│   RUNNING    │ ◄──────────────┘
            │       └──┬────────┬──┘
            │          │        │
            │   stop   │        │  suspend
            │          ▼        ▼
            │   ┌──────────┐ ┌────────────┐
            │   │ STOPPING │ │ SUSPENDING │
            │   └─────┬────┘ └──────┬─────┘
            │         ▼              ▼
            │   ┌────────────┐  ┌───────────┐
            │   │ TERMINATED │  │ SUSPENDED │
            │   └────┬───────┘  └─────┬─────┘
            │        │ start          │ resume
            └────────┴────────────────┘
                     │
                     │ delete
                     ▼
                  (sumiu)


   Em paralelo, a partir de RUNNING (ou outros), pode ir para:
                    ┌──────────────┐
                    │  REPAIRING   │ ── volta a RUNNING se conseguir
                    └──────────────┘
```

#### PROVISIONING
Google está **reservando recursos** (CPU, memória, disco) no host físico. A VM ainda não roda. **Transitório** — você não paga.

#### STAGING
Recursos já adquiridos. Compute Engine está:
- Anexando endereços IP
- Inicializando a imagem do sistema
- Inicializando o SO

Ainda transitório — não paga.

#### RUNNING
VM ativa. Roda **startup scripts pré-configurados** e habilita acesso SSH/RDP. **Paga CPU + RAM + disco + rede**.

Coisas que dá pra fazer com VM em **RUNNING**:
- **Live migrate** para outro host na mesma zona (sem rebootar)
- **Mover** a VM para outra zona
- **Snapshot** do persistent disk
- **Exportar** a system image
- **Reconfigurar metadata** (ex.: trocar startup script, adicionar SSH keys)
- **Reset** (ver abaixo)

Coisas que **exigem parar** a VM antes:
- **Mudar machine type** (upgrade de CPU/RAM)
- **Trocar boot disk**
- **Mudar a imagem** (na verdade, isso exige recriação)

#### STOPPING
Transitório. A VM está executando os **shutdown scripts pré-configurados** e desligando ordenadamente. Logo vai para `TERMINATED`.

#### TERMINATED
VM parada mas **ainda existe** (config, disco anexado, IP estático reservado permanecem). Daqui você pode:
- **Restart** → volta a `PROVISIONING` → ... → `RUNNING`
- **Delete** → some

**Cobrança em TERMINATED:**
- ❌ Não paga CPU / RAM
- ✅ Paga **discos anexados** (boot + adicionais)
- ✅ Paga **IPs estáticos reservados** (se não estiver usado por outra VM, GCP cobra como "unused static IP")

#### REPAIRING (estado novo do vídeo)
A VM entrou em erro interno OU o **host físico subjacente está indisponível** por manutenção. Durante REPAIRING:
- A VM está **inutilizável** (não responde)
- **Você NÃO é cobrado** enquanto está em repair
- **SLA não cobre** o tempo em REPAIRING
- Se o reparo **deu certo**, a VM volta para algum dos estados anteriores (normalmente RUNNING)
- Se não deu, pode ir pra TERMINATED

> **Armadilha de prova:** "minha VM não responde, mas o billing parou de cobrar dela" → provavelmente está em **REPAIRING** (não TERMINATED — em TERMINATED você ainda paga disco).

#### SUSPENDING / SUSPENDED (estado novo do vídeo)
**Suspender ≠ parar.** Suspender é mais parecido com "hibernar" um laptop:
- O **estado da memória** (RAM) é salvo em **disco persistente**
- A VM para de consumir CPU
- Quando você dá **resume**, a VM volta exatamente onde estava — processos rodando, conexões abertas, etc.

Fluxo: `RUNNING` → `SUSPENDING` (transitório) → `SUSPENDED` → `resume` → volta a `RUNNING` (ou `delete`).

> **Quando usar suspend em vez de stop:**
> - Você quer **economizar** mas **manter o estado em RAM** (ex.: pausar um job longo no meio).
> - **Custo:** você paga apenas pelo storage usado pra salvar a memória — mais barato que RUNNING, mais caro que TERMINATED.

> **Armadilha de prova:** suspend **não é** o mesmo que stop. Stop encerra processos; suspend preserva o estado em memória.

### Reset: o "Ctrl+Alt+Del" da VM

`gcloud compute instances reset` é equivalente a apertar o **botão reset** no PC:

- **Limpa o conteúdo da memória** da máquina
- **Reseta a VM para o estado inicial** (faz um boot duro)
- A VM permanece em `RUNNING` durante o reset (não passa por `STOPPING`/`TERMINATED`)
- **Não executa shutdown scripts** (porque não é um shutdown ordenado — é um power cycle forçado)

> **Quando usar reset:** VM travou e SSH não responde. Você precisa do equivalente a um power cycle sem chegar a destruir a VM.

> **Armadilha de prova:** reset **não passa por TERMINATED**. Se a questão pede "trazer a VM de volta ao estado inicial **sem desligá-la**" → **reset**. Se pede "reduzir custos enquanto não estiver em uso" → **stop**.

### Live migration — manutenção sem downtime

Quando o Google precisa fazer **manutenção no host físico** onde sua VM mora (patch de hypervisor, reboot do host, troca de hardware), por padrão a VM é **migrada ao vivo** para outro host na **mesma zona** — sem reboot, sem perda de conexão de rede.

- Esse mecanismo é o que permite o Google manter a infraestrutura atualizada **sem interromper seus workloads**.
- Funciona transferindo o estado completo da VM (memória, registers de CPU, conexões) para outro host de forma transparente.

**Limitações:** algumas VMs **não suportam live migration** (ex.: VMs com **GPUs**, **Spot/Preemptible VMs**, certos machine types com Local SSD em condições específicas) — nesses casos a VM é **terminada** durante a manutenção.

### Availability policy: como a VM reage à manutenção

Toda VM tem uma **availability policy** com duas opções configuráveis:

| Opção | Valores | Default | O que controla |
|---|---|---|---|
| **`onHostMaintenance`** | `MIGRATE` ou `TERMINATE` | `MIGRATE` | O que fazer quando o host precisa de manutenção |
| **`automaticRestart`** | `true` ou `false` | `true` | Se a VM reinicia sozinha após crash ou manutenção que a terminou |

**Pode ser configurado:**
- **Na criação** (`gcloud compute instances create ... --maintenance-policy=MIGRATE --restart-on-failure`)
- **Enquanto a VM roda** (não precisa parar — é uma das ações editáveis em RUNNING)

**Quando você forçaria `TERMINATE` em vez de `MIGRATE`?**
- VMs com **dependências de hardware específico** que não toleram troca de host (raro)
- VMs com **licenças por host** (lift-and-shift de software com licenciamento estrito)
- VMs com **GPUs** — não suportam migrate; vão terminar de qualquer jeito

> **Armadilha de prova:** se a questão fala "queremos que a VM nunca seja migrada, mesmo durante manutenção do Google" → `onHostMaintenance=TERMINATE`. Combine com `automaticRestart=true` se quer que ela suba de novo automaticamente depois.

### Timing de shutdown — 90 segundos (normal) vs 30 segundos (preemptible)

Quando você dá `stop`, `delete` ou `reboot` numa VM:

| Tipo de VM | Tempo de graça pro shutdown ordenado | O que acontece depois |
|---|---|---|
| **VM normal (Standard)** | **~90 segundos** | Compute Engine força desligamento |
| **VM preemptible / Spot** | **~30 segundos** | Envia sinal **ACPI G3 "Mechanical Off"** ao SO (corte abrupto) |

**Implicações práticas:**
- **Shutdown scripts em VMs normais** têm ~90s pra rodar (drenar conexões, fechar processos, salvar estado).
- **Shutdown scripts em preemptibles** têm apenas ~30s — **sejam rápidos e idempotentes**. Salvar checkpoint, dar flush em buffer, marcar job como interrompido. Nada de operação longa.

> **Armadilha de prova clássica:** "estamos perdendo dados quando preemptible VMs são encerradas" → causa quase sempre é shutdown script que tenta fazer trabalho longo em mais de 30s. Solução: simplificar o script para fazer apenas checkpoint mínimo.

### OS patch management

Manter o SO atualizado é parte da operação de qualquer infra. O Compute Engine tem um serviço dedicado: **OS patch management**.

**Por que o GCP gerencia isso:**
- Quando você usa uma **premium image** (Windows Server, RHEL com licença), o **custo da imagem já inclui patch management**.
- Patching manual é "labor intensive" e arriscado — VMs antigas viram brechas de segurança.

**Dois componentes principais:**

#### 1. Patch compliance reporting
- Te dá **visibilidade** do status de patches de todas as VMs (Windows e Linux)
- Mostra **recomendações** de quais VMs precisam de atualização

#### 2. Patch deployment
- **Automatiza** a aplicação de patches
- Você cria um **patch deployment** → ele agenda **patch jobs** → patch jobs rodam nas VMs alvo

**Tarefas que você pode configurar:**

| Tarefa | O que faz |
|---|---|
| **Patch approvals** | Selecionar quais patches aplicar (do conjunto completo do SO) |
| **Flexible scheduling** | Agendar one-time ou recorrente (ex.: toda 2ª terça do mês às 3h) |
| **Advanced patch config** | Scripts **pré** e **pós** patching (ex.: parar serviço antes, validar depois) |
| **Centralized management** | Gerenciar todos os jobs em um único lugar |

> **Armadilha de prova:** "queremos aplicar patches de segurança em uma frota de VMs Linux + Windows de forma centralizada, com janela de manutenção definida" → resposta é **OS Patch Management** (não scripts customizados, não Ansible, não soluções de terceiros).

### O que dá (e o que NÃO dá) pra fazer no estado TERMINATED

Quando a VM está parada (TERMINATED), você **pode**:

- ✅ **Mudar machine type** (upgrade de CPU/RAM)
- ✅ **Anexar/desanexar discos persistentes**
- ✅ **Mudar tags de rede**
- ✅ **Mudar service account**
- ✅ **Mudar metadata** (startup script, SSH keys)
- ✅ **Mudar availability policy**
- ✅ **Adicionar/remover GPU** (em famílias que suportam)
- ✅ **Iniciar de novo** (start)
- ✅ **Deletar**

O que você **NÃO pode**:

- ❌ **Trocar a imagem** do boot disk (a imagem virou disco no momento da criação — pra mudar, recrie a VM ou o boot disk a partir de outra imagem)
- ❌ **Mudar a zona** diretamente (precisa do processo "move instance", que recria)

**E o que pode ser feito SEM parar (VM em RUNNING):**

- ✅ Mudar **availability policy** (on-host-maintenance, automatic restart)
- ✅ Mudar **metadata** (incluindo SSH keys)
- ✅ Tirar **snapshot** do persistent disk
- ✅ Anexar **discos adicionais** (hot-attach)
- ✅ Mudar **labels**
- ✅ Trocar **firewall tags** (afeta regras imediatamente)

> **Armadilha de prova:**
> - "preciso aumentar a CPU da VM, ela pode ficar fora do ar?" → **sim, precisa parar** (mudar machine type exige TERMINATED).
> - "Preciso mudar quantos discos a VM tem?" → **NÃO precisa parar** (hot-attach/detach é suportado pra discos persistentes).
> - "Preciso trocar a imagem do SO" → **não dá em VM existente**; recrie.

---

### Resumo prático em 1 frase por tópico

- **Acesso:** criador é root; Linux via SSH+chave (porta 22), Windows via RDP+credencial (porta 3389); rede `default` já libera essas portas.
- **Provisioning/Staging:** transitórios, sem custo.
- **Running:** paga tudo (CPU+RAM+disco+rede).
- **Stopping → Terminated:** shutdown ordenado, ~90s; em TERMINATED você paga só disco e IP estático.
- **Repairing:** Google está consertando; sem custo, sem SLA.
- **Suspending/Suspended:** "hibernar" — memória salva em disco; paga só pelo storage.
- **Reset:** power cycle forçado; sem shutdown scripts; permanece em RUNNING.
- **Live migration:** transparente; pode ser desabilitada via `onHostMaintenance=TERMINATE`.
- **Shutdown timing:** 90s normal, 30s preemptible (depois ACPI G3).
- **OS Patch Management:** compliance reporting + patch deployment com pré/pós scripts.
- **Imagem é imutável:** não dá pra trocar imagem de VM parada; tudo o mais é editável.

> Esta seção é o aprofundamento do vídeo de **pricing** do módulo Compute Engine. Cobre: modelo de cobrança (minuto mínimo + segundos), resource-based pricing, **Sustained Use Discounts (SUD)**, **Committed Use Discounts (CUD)**, **Preemptible vs Spot VMs**, **VM sizing recommendations** e Free Usage Limits.

---

### Modelo de cobrança — minuto mínimo, depois segundos

Compute Engine cobra **vCPUs, GPUs e GB de memória** com a seguinte granularidade:

| Tempo de uso | Como é cobrado |
|---|---|
| **< 1 minuto** | Cobra **1 minuto inteiro** (mínimo) |
| **≥ 1 minuto** | Cobra em **incrementos de 1 segundo** |

**Exemplo prático:** VM rodando 30 segundos → você paga 1 minuto. VM rodando 90 segundos → você paga 1 min e 30 seg.

> **Por que isso importa:** scripts que sobem/derrubam VMs frequentemente (CI ephemeral runners, jobs curtos) pagam o "minuto mínimo" várias vezes. Se você sobe a VM toda hora por 20s, você paga 60 minutos no dia (não 6 min e 40s).

---

### Resource-based pricing — vCPU e RAM cobrados separadamente

Mesmo quando você escolhe um machine type predefinido (ex.: `n1-standard-4`), **a fatura mostra vCPU e memória separados**.

```
Você pediu:    n1-standard-4         (4 vCPU + 15 GB RAM)
Fatura mostra: 4 × n1-vCPU           (preço por vCPU × tempo)
              + 15 GB × n1-memoria   (preço por GB × tempo)
```

**Por que o Google faz isso?**
- Permite calcular **descontos no nível de recurso individual**, não no machine type inteiro
- Trata predefined e custom machine types de forma uniforme (custom já era cobrado assim)
- Facilita a aplicação **agregada** de Sustained Use Discounts (ver abaixo)

---

### Sustained Use Discounts (SUD) — automático, por uso prolongado no mês

**Princípio:** se você roda um recurso (vCPU, RAM, GPU) por **uma parte significativa do mês de faturamento**, o GCP **automaticamente** dá desconto crescente. Sem precisar pedir, sem contrato.

#### A escala de desconto

| Uso no mês | Effective discount |
|---|---|
| Até **25%** do mês | **0%** (preço normal) |
| Acima de **25%** | Desconto começa a aplicar nas horas incrementais |
| **50%** do mês | **~10%** effective |
| **75%** do mês | **~20%** effective |
| **100%** do mês | **~30%** effective (máximo) |

**Mecânica:** o desconto é aplicado **a cada minuto incremental** acima do threshold de 25%. Quanto mais tempo a VM roda no mês, maior o desconto efetivo sobre o total.

#### Detalhe crítico de timing

> **Os descontos resetam no início de cada mês.** Pra ter o **30% máximo**, a VM precisa estar rodando **desde o dia 1**. Subir uma VM dia 15 e deixar rodando 100% do tempo a partir daí ≠ 30% — você só vai ter ~50% de uso no mês corrente, então ~10%.

#### Como SUD é calculado: por **recurso**, por **região**, por **categoria**

O Google **não** olha cada VM isoladamente. Ele agrega:

1. **Por região** (`us-central1` é separado de `europe-west1`)
2. **Por categoria** (predefined machine types **separado de** custom machine types)
3. **Decompondo em vCPU + RAM individuais**

#### Exemplo do vídeo (clássico de prova)

Você roda na **us-central1** durante o mês:
- **1ª metade**: `n1-standard-4` (4 vCPU, 15 GB)
- **2ª metade**: `n1-standard-16` (16 vCPU, 60 GB)

O Compute Engine **NÃO** trata isso como "dois machine types diferentes, pouca SUD em cada um". Ele **reorganiza** assim:

```
Decomposição:
├── 4 vCPU + 15 GB    →  rodando o MÊS INTEIRO  (SUD máximo!)
└── 12 vCPU + 45 GB   →  rodando METADE do mês  (SUD parcial)
```

**Resultado:** a parte que se sobrepõe entre os dois machine types (4 vCPU + 15 GB) ganha o desconto cheio porque ficou ligada o mês todo, mesmo que tenha sido "n1-standard-4 numa metade e parte de n1-standard-16 na outra".

> **Armadilha de prova:** "redimensionei a VM no meio do mês, perdi todos os descontos acumulados?" → **não**. O SUD é calculado em cima de **recursos individuais agregados**, não do machine type específico. A parte sobreposta mantém o desconto.

---

### Committed Use Discounts (CUD) — compromisso de 1 ou 3 anos

**Quando usar:** workload **estável e previsível** (ex.: produção 24/7 que você sabe que vai rodar pelos próximos 1-3 anos).

**Como funciona:** você **compromete** a comprar uma quantidade específica de **vCPUs + memória** numa região por **1 ano ou 3 anos**, e ganha desconto pesado em troca.

| Categoria | Desconto máximo |
|---|---|
| **Most machine types** (general-purpose, compute-optimized) + custom | **até 57%** |
| **Memory-optimized** (M-series, X4) | **até 70%** |

**Importante:** CUD é **compromisso financeiro** — você paga mesmo se não usar a capacidade. Por isso só vale pra cargas estáveis.

> **Armadilha de prova:** "carga de produção previsível 24/7 nos próximos 3 anos, queremos máximo desconto" → **CUD**. "Carga em batch que roda só algumas horas por dia" → **SUD** (ou Spot se tolerar interrupção).

---

### Os descontos NÃO podem ser combinados

> **Regra que cai em prova:** SUD, CUD e Preemptible/Spot **não se combinam**. Você escolhe **um** modelo de desconto por recurso.

- VM com CUD não recebe SUD adicional em cima.
- VM preemptible/Spot **já vem com desconto embutido** — não soma SUD/CUD.
- Free Tier também não soma.

---

### Preemptible VMs vs Spot VMs

Os dois são **VMs de excess capacity** (capacidade ociosa do Google) com preço muito mais baixo. O Google pode **terminar** ("preempt") essas VMs **a qualquer momento** se precisar dos recursos para clientes pagantes normais.

**Diferença crítica:**

| | Preemptible VM | Spot VM |
|---|---|---|
| **Custo** | Muito menor que VM normal | Muito menor que VM normal (geralmente mais barato que preemptible) |
| **Pode ser terminada a qualquer momento?** | ✅ Sim | ✅ Sim |
| **Runtime máximo** | **24 horas** (Google encerra de qualquer jeito) | **Sem limite** — roda até o Google precisar dos recursos |
| **Aviso antes do encerramento** | 30 segundos (ACPI G3) | 30 segundos (ACPI G3) |
| **Disponibilidade** | Varia com uso do GCP na região | Varia com uso do GCP na região |
| **Geração** | Modelo legado | **Substituto recomendado** (geração mais nova) |

**Quando usar:**
- ✅ Workloads **tolerantes a falha** (batch, render, fault-tolerant analytics)
- ✅ Pipelines que fazem **checkpoint** periodicamente
- ✅ Jobs onde "perdi a VM no meio, recomeço" não custa nada
- ❌ Bancos de dados, web servers de produção, qualquer coisa stateful sensível

**Spot é o caminho moderno.** Preemptible existe por compatibilidade. Se você está começando agora, use Spot.

> **Armadilha de prova:**
> - "queremos VMs descartáveis pra batch noturno de até 4 horas, máximo desconto" → **Spot VM** (ou Preemptible, mas Spot é mais moderno).
> - "queremos VMs descartáveis mas precisamos que rodem mais de 24h sem interrupção forçada" → **Spot VM** (preemptible te força a parar em 24h).
> - "Convertemos a VM normal em preemptible pra economizar" → **errado**, preemptibility é decisão de criação.

---

### Custom machine types — extensão natural do pricing

Já vimos custom machine type na anatomia da instância. No contexto de preço, o ponto é: como vCPU e RAM são cobrados **separadamente** mesmo em predefined, escolher custom não tem penalidade de preço — você paga só pelo que pediu.

**Implicações:**
- Se sua app precisa de 3 vCPU + 7 GB, **escolha custom** em vez de saltar pra `e2-standard-4` (4 vCPU + 16 GB) e desperdiçar.
- Custom recebe SUD e CUD normalmente (em categoria separada de predefined).

### VM Sizing Recommendations (rightsizing)

Compute Engine **observa** o uso de CPU e RAM das suas VMs e **sugere automaticamente** machine types mais apropriados (geralmente menores) quando detecta superprovisionamento.

**Como funciona:**
- Recomendações aparecem **24 horas após a criação** da instância (precisa coletar dados)
- Aparecem no Console (na lista de VMs e na página de detalhes da VM)
- O Recommender também tem API: `gcloud recommender recommendations list`

**Use caso típico:** você criou `n2-standard-8` "no chute". Depois de uma semana, o Recommender vê que a VM usou no máximo 20% CPU e 40% RAM, e sugere descer pra `n2-standard-4` (corta o custo pela metade).

> **Armadilha de prova:** "minimizar custo sem trocar manualmente cada VM" → resposta inclui **VM sizing recommendations** (ou Recommender em geral).

### Free Usage Limits

O Free Tier do Google Cloud inclui uma quantidade gratuita de Compute Engine **por mês**, com restrições específicas (1 `e2-micro` por mês em algumas regiões dos EUA, com limite de disco, etc.). Detalhes mudam — o que importa pra prova é **saber que existe**, não decorar números.

### Resumo prático em 1 frase por tópico

- **Mínimo 1 minuto** + incrementos de 1 segundo depois.
- **Resource-based:** vCPU e RAM cobrados separados, mesmo em predefined.
- **SUD:** automático, até 30% pra VM rodando o mês inteiro, reseta dia 1 do mês.
- **SUD agrega por região + categoria** (predefined ≠ custom); recursos sobrepostos entre VMs ganham desconto cheio.
- **CUD:** 1 ou 3 anos de compromisso, até 57% (most) ou 70% (memory-optimized).
- **Descontos não combinam** — escolha um.
- **Preemptible:** máximo 24h, modelo legado.
- **Spot:** sem limite de runtime, modelo recomendado, mais barato.
- **Sizing recommendations:** 24h após criação, sugere downsize de VM superprovisionada.
- **Free Tier existe**, com limites específicos.

### Flexibilidade que o hardware físico não tem

Compute Engine vai além do que um servidor físico consegue fazer. Dois exemplos importantes:

#### 1. Micro VMs (shared-core)
- Uma VM micro **compartilha a CPU física** com outras VMs no mesmo host
- Resultado: você consegue uma VM com custo muito baixo para workloads que não precisam de CPU constante (ex.: servidor de desenvolvimento, apps de baixo tráfego)
- Tipos shared-core: `e2-micro`, `e2-small`, `e2-medium`, `f1-micro`, `g1-small`

#### 2. Burst capability
- Algumas VMs conseguem **usar mais CPU do que sua cota nominal** por um período curto, quando há CPU ociosa disponível no host físico
- Funciona como um turbo: o rated capacity é o mínimo garantido; o burst é o máximo possível em momentos de pico
- Muito útil para workloads com picos ocasionais que não justificam provisionar CPU dedicada full-time

> **Analogia:** É como um plano de celular pré-pago com bônus de dados. Você tem um limite base, mas pode usar mais quando a rede estiver ociosa.

### TPU — Tensor Processing Unit

**O problema que motivou a criação:**
CPUs e GPUs pararam de escalar rápido o suficiente para a demanda de Machine Learning. Em 2016, o Google criou os TPUs para resolver isso.

**O que são:**
- **ASICs** (Application-Specific Integrated Circuits) desenvolvidos pelo Google
- Hardware desenhado **especificamente** para acelerar workloads de ML
- São "domain-specific hardware" em oposição ao hardware de propósito geral (CPU/GPU)

**Por que são mais eficientes para ML?**
- A arquitetura é otimizada para a operação matemática central do ML: **multiplicação de matrizes**
- Isso resulta em maior throughput para esse tipo de cálculo com menos energia

**Vantagens vs CPU/GPU:**
- Geralmente **mais rápidos** que GPUs e CPUs para aplicações de IA
- **Significativamente mais eficientes em energia**

**Quando usar Cloud TPU:**
- Modelos que **treinam por longos períodos**
- Modelos **grandes** com **batch sizes grandes**
- Não vale a pena para inferência simples ou modelos pequenos

> **Armadilha de prova:** TPU é para **treinar modelos de ML**. CPU/GPU são para propósito geral. Se a questão menciona "treino de modelo de ML de larga escala" → TPU. Se menciona "GPU para renderização gráfica" → GPU (não TPU).

---

### vCPU: o que é na prática

Na Compute Engine, cada **vCPU** é implementada como um único **hardware hyper-thread** em uma das plataformas de CPU disponíveis.

> **Analogia:** Em um servidor físico com um core que suporta hyperthreading, esse core apresenta 2 threads simultâneas para o OS. Cada uma dessas threads é uma vCPU no mundo do Google Cloud.

Isso é diferente do que você pode estar acostumado em on-premises, onde "core físico" e "thread" são conceitos separados.

---

### Throughput de rede: regra dos 2 Gbps por vCPU

A capacidade de rede da sua VM depende diretamente do número de vCPUs:

| vCPUs | Throughput de rede |
|---|---|
| 1 vCPU | 2 Gbps |
| **2 ou 4 vCPUs** | **até 10 Gbps** (exceção à regra) |
| N vCPUs (geral) | N × 2 Gbps |
| **176 vCPUs** (série C3) | **até 200 Gbps** (máximo teórico) |

**A regra:**
```
Throughput = vCPUs × 2 Gbps
Exceção: 2 ou 4 vCPUs → máximo de 10 Gbps
```

> **Armadilha de prova:** A exceção de 2 e 4 vCPUs recebendo até 10 Gbps (não 4 Gbps e 8 Gbps respectivamente) é contra-intuitiva. O Google faz isso para garantir que instâncias menores ainda tenham largura de banda competitiva.

---

### Tipos de disco: Standard vs SSD vs Local SSD

Os três tipos de disco têm o mesmo **tamanho máximo** de persistent disk, mas diferem em **performance**, **custo** e **persistência de dados**:

| Tipo | Tecnologia | Ponto forte | Ponto fraco | Persistência |
|---|---|---|---|---|
| **Standard (HDD)** | Spinning hard disk | Maior **capacidade** por dólar | Menor IOPS | ✅ Persiste (é um Persistent Disk) |
| **SSD Persistent Disk** | Flash memory | Maior **IOPS** por dólar | Mais caro por GB | ✅ Persiste |
| **Local SSD** | Flash, físico no host | **Maior throughput + menor latência** | Dados perdidos ao parar/deletar VM | ❌ Temporário |

**Detalhes importantes:**

- **Standard vs SSD persistent disk:** ambos têm o mesmo **tamanho máximo** (até **257 TB por instância**). A escolha é sobre performance vs custo.
- **Performance escala com o tamanho:** mais GB alocado = mais IOPS disponíveis (para persistent disks)
- **Local SSD** é físicamente conectado ao hardware do host → latência mínima. Mas: se a VM parar ou for deletada, **os dados somem**. Uso típico: swap disk, cache temporário, ramdisk de alta capacidade.

> **Analogia de Local SSD:** É como a memória RAM — ultra-rápida, mas volátil. Se o servidor desligar, perdeu.

> **Armadilha de prova:** Se a questão menciona "dados precisam persistir após reinicialização" → Local SSD está fora. Se menciona "máxima performance com dados temporários" → Local SSD.

---

### Load Balancing: sem necessidade de pre-warming

No Google Cloud, **load balancers não são dispositivos de hardware** — são **conjuntos de regras de engenharia de tráfego** aplicadas pela VPC.

Isso tem uma implicação importante: **não é necessário pre-warming**.

**Por quê em clouds tradicionais/on-premises é necessário pre-warming?**
Um hardware load balancer precisa "aprender" o padrão de tráfego antes de distribuí-lo eficientemente. Se chega um pico repentino, o dispositivo pode não aguentar.

**No GCP:**
- O load balancer é software puro na rede do Google
- As regras ficam na VPC e são aplicadas globalmente
- Tráfego repentino é absorvido sem necessidade de preparação prévia

**Tipos disponíveis (vistos no módulo anterior):**
- **Application Load Balancing** (Camada 7 — HTTP/HTTPS)
- **Network Load Balancing** (Camada 4 — TCP/UDP)

### Preemptible VMs — detalhes operacionais

Já vimos na seção de pricing que **Preemptible VM** é uma instância com desconto enorme (**60% a 91%**) que o Google pode terminar a qualquer momento. Esse vídeo acrescenta os detalhes **operacionais** que caem em prova:

**1. "Sem cobrança no primeiro minuto"**
- Se o Google preemptar sua VM **dentro do primeiro minuto** de execução, você **não paga nada** por ela.
- Por quê? Porque preemptible só existe como "deal com risco" — se o Google te interrompeu antes mesmo de você usar, ele assume o custo.
- Implicação prática: se você está testando se há capacidade preemptible disponível, o "custo de tentativa" é zero.

**2. Limite duro de 24 horas**
- Mesmo que o Google **não precise** dos recursos, a VM preemptible **vai ser encerrada após 24h**. Não tem como prorrogar.
- Comportamento esperado: você precisa de pipeline com **checkpoint** ou job que termine em menos de 24h.

**3. Notificação de preempt = 30 segundos**
- O Google manda um **shutdown signal ACPI G3** para a VM com 30 segundos de antecedência.
- Se você tem **shutdown script**, ele tem 30 segundos para salvar estado, drenar conexões, etc.
- Compare com VM normal: 90 segundos. Preemptible é **3x mais curto**.

**4. Sem live migration**
- VMs preemptible **NÃO** suportam live migration. Se há um evento de manutenção no host físico, a VM é **encerrada**, não migrada.
- Faz sentido: se o Google já trata preemptible como "descartável", não vai gastar recurso movendo essas VMs.

**5. Sem automatic restart**
- Se a VM preemptible for encerrada (por preempção ou por maintenance), o Compute Engine **não vai reiniciá-la automaticamente**.
- **Pegadinha:** a opção "Automatic restart" no Console fica **bloqueada** quando você marca a VM como preemptible.

**6. Como fazer "auto-restart" externo (idiomatic GCP)**
- Apesar do Compute Engine não reiniciar preemptibles, você pode usar:
  - **Managed Instance Groups (MIGs)** com health checks → o MIG cria uma nova preemptible quando a antiga morre
  - **Cloud Monitoring** alertando + Cloud Functions/Workflows recriando a VM
  - **Load balancer** com health check → tira a VM morta do pool e o MIG por trás repõe
- **Por que isso é importante na prova:** se a questão diz "queremos preemptibles + tolerância a falha", a resposta é geralmente **MIG + autoscaling/autohealing**, não "use VM normal".

**7. Caso de uso clássico: batch processing**
> "Se algumas instâncias terminam durante o processamento, o job desacelera mas não para completamente."

- Render farms, processamento de imagens, ETL noturno, treino de ML pouco-sensível-a-interrupção, fault-tolerant analytics.
- A ideia: **paralelize** o batch entre N preemptibles. Se 3 morrem, as outras N-3 continuam, e o MIG repõe.

> **Armadilha de prova:**
> - "Web server de produção em preemptible para economizar" → ❌ errado, preemptible não serve para stateful/sensível a downtime.
> - "Job de 36 horas em preemptible" → ❌ vai ser encerrado em 24h. Use **Spot** se não puder dividir o job.
> - "Como reiniciar preemptible automaticamente?" → use **MIG**, não automatic restart (que não existe pra preemptible).

---

### Spot VMs — a geração nova do preemptible

**O conceito de "spot provisioning model"** vem do mercado: você "dá um lance" pela capacidade ociosa do provedor, e ela pode ser reclamada.

**Spot VM = a versão moderna de Preemptible VM.**

| Aspecto | Preemptible (legado) | Spot (moderno) |
|---|---|---|
| Status | Mantido por compatibilidade | **Recomendado para uso novo** |
| Preço | 60–91% off | Mesmo modelo de pricing |
| Runtime máximo | **24h forçado** | **Sem limite** — roda até o Google precisar |
| Aviso de preempt | 30s | 30s |
| Live migration | ❌ | ❌ |
| Automatic restart | ❌ | ❌ |
| Disponibilidade | Capacidade ociosa | **Capacidade ociosa + backup** do Google Cloud |

**Por que Spot é "novo" se o preço é igual?**
Spot trouxe **features** que preemptible não tem:
- **Sem runtime máximo** — pode rodar dias, semanas, meses, até o Google precisar dos recursos.
- Mesmo modelo de pricing, mas pode ser **encarado como recurso "first-class"** com APIs e integrações mais ricas.

**Probabilidade de preempção:**
- "Geralmente baixa", mas **varia por zona e por dia** conforme a demanda interna do Google.
- Spot vem de **excess capacity + backup capacity** — então em horários de pico, a probabilidade de preempção aumenta.

**Best practice — escolha tamanhos menores:**
- Capacidade Spot **é mais fácil de conseguir para machine types menores** (menos vCPU, menos RAM).
- Se você precisa de muita capacidade Spot, **divida o workload em muitas VMs pequenas** em vez de poucas grandes.
- Analogia: é mais fácil achar 10 carros pequenos disponíveis num estacionamento cheio do que 10 vagas grandes para SUVs.

**Comportamento idêntico ao Preemptible em:**
- ❌ Sem live-migrate para "virar VM normal" enquanto roda
- ❌ Sem automatic restart em maintenance event
- ✅ Mesma necessidade de **MIG + autohealing** se você quer fault tolerance

**Spot VMs são "recurso finito"**
- Mesmo que você esteja disposto a pagar pelo Spot, pode **não haver capacidade disponível** naquele momento na zona.
- Em prova, se a questão diz "precisamos garantir capacidade", Spot é a resposta **errada**. Use VM normal ou CUD para garantir.

> **Armadilha de prova:**
> - "Workload que roda por mais de 24h e tolera interrupção" → **Spot** (Preemptible não dá conta).
> - "Quero usar preemptible pra economizar mas precisamos garantir que sempre tenhamos VMs disponíveis" → ❌ contradição; Spot/Preemptible **não garantem disponibilidade**. Use normal + CUD.
> - "Como diminuir probabilidade de não conseguir capacidade Spot?" → use **machine types menores**.

---

### Sole-tenant nodes — servidor físico dedicado ao seu projeto

**O problema que resolve:** alguns workloads têm **requisitos de compliance** (PCI-DSS para processamento de pagamentos, HIPAA para dados de saúde, regulações governamentais) que exigem **isolamento físico** — ou seja, suas VMs **não podem compartilhar o mesmo hardware físico** com VMs de outros clientes.

**O que é:**
- Um **sole-tenant node** é um **servidor Compute Engine físico inteiro**, dedicado **só ao seu projeto**.
- Você ainda cria VMs dentro dele (várias, de tamanhos diferentes), mas **todas as VMs daquele node pertencem ao mesmo projeto**.

**Comparação visual (do vídeo):**

| Host normal (multi-tenant) | Sole-tenant node |
|---|---|
| VM-A (cliente 1) | VM-A (seu projeto) |
| VM-B (cliente 2) | VM-B (seu projeto) |
| VM-C (cliente 3) | VM-C (seu projeto) |
| Hardware **compartilhado** | Hardware **dedicado a você** |

**Casos de uso:**
1. **Compliance / regulação** — "nossas VMs não podem rodar em hardware compartilhado com outros tenants" (ex.: payment processing, dados de saúde).
2. **Performance previsível** — sem "noisy neighbors" disputando CPU/disco/rede no host.
3. **Agrupar VMs do mesmo projeto no mesmo host** — útil para latência ultra-baixa entre VMs.
4. **Bring Your Own License (BYOL)** — se você tem licenças de SO (Windows Server, RHEL, Oracle) que **só permitem rodar em hardware dedicado por core físico**, sole-tenant deixa você trazer essas licenças para o GCP.

**In-place restart (feature destacada no vídeo):**
- Para minimizar uso de cores físicos quando você traz BYOL, sole-tenant suporta **in-place restart** — a VM é reiniciada **no mesmo node físico**, sem realocação. Isso preserva a aderência da licença ao hardware específico.

**Flexibilidade:**
- Você pode encher o node com **VMs de tamanhos diferentes**, incluindo **custom machine types** e **extended memory**.
- Não precisa "preencher" o node — pode rodar 1 VM grande ou 10 VMs pequenas dentro dele.

**Trade-off:**
- 💰 **Mais caro** que VM normal — você paga pelo **node inteiro**, não só pelas VMs que rodam nele.
- Faz sentido só quando há requisito real de compliance ou BYOL.

> **Armadilha de prova:**
> - "Precisamos rodar workload de processamento de pagamentos com compliance que exige isolamento físico" → **Sole-tenant nodes**.
> - "Queremos garantir performance previsível sem noisy neighbors" → Sole-tenant é **uma** das respostas; CUD + machine type maior também ajuda.
> - "Compartilhamos licenças de Oracle que exigem hardware dedicado" → **Sole-tenant + BYOL**.
> - **Nunca** confunda sole-tenant com **Confidential VM** (são problemas diferentes — isolamento físico vs criptografia em uso).

### Shielded VMs — integridade verificável do boot e do kernel

**O problema que resolve:** como você prova que sua VM **não foi comprometida** por malware de boot, rootkit em kernel, ou modificação no firmware? Você não consegue inspecionar fisicamente o host. Você precisa de **integridade verificável**.

**O que é:**
- **Shielded VM** é uma VM com três proteções de baixo nível ativadas:
  1. **Secure Boot** — só permite que firmware/bootloader/kernel **assinados digitalmente** sejam executados. Bloqueia rootkits que tentam se carregar antes do SO.
  2. **vTPM (virtual Trusted Platform Module) + Measured Boot** — um TPM virtual mede cada estágio do boot (firmware → bootloader → kernel → drivers) e gera **hashes verificáveis**. Se algo mudou desde a última boot, dá pra detectar.
  3. **Integrity monitoring** — o Compute Engine compara as medidas atuais com uma baseline conhecida e **alerta** quando há desvio.

**Analogia:**
> É como uma caixa-forte com **selo de fábrica** + **câmera filmando 24h** + **alarme automático** se alguém abrir. Você não confia apenas em "está trancado" — você confia em "está trancado E sei provar que ninguém mexeu".

**Faz parte da Shielded Cloud Initiative:**
- Iniciativa mais ampla do Google Cloud para oferecer **fundação mais segura** em toda a plataforma.
- Inclui features como **vTPM shielding/sealing** que ajudam a prevenir **exfiltração de dados** — chaves podem ser "seladas" ao estado da máquina e só liberadas se o boot for íntegro.

**Como ativar:**
- Você precisa selecionar uma **shielded image** ao criar a VM.
- Imagens shielded estão disponíveis para várias distros Linux e Windows (vamos ver imagens no próximo bloco do módulo).
- No Console, isso aparece como uma seção "Shielded VM" com checkboxes para Secure Boot, vTPM e Integrity Monitoring.

**Custo extra:** zero. Shielded VM **não custa mais** que VM normal — você paga o mesmo, mas ganha as proteções.

> **Armadilha de prova:**
> - "Precisamos garantir que o boot da VM não foi adulterado" → **Shielded VM** (Secure Boot + vTPM + Integrity Monitoring).
> - "Como prevenir rootkit em produção?" → **Shielded VM** (Secure Boot bloqueia código não assinado).
> - "Preciso de criptografia da RAM em uso" → ❌ isso é **Confidential VM**, não Shielded.
> - Shielded VM **não criptografa memória** — só garante integridade do boot.

---

### Confidential VMs — criptografia da memória durante o processamento

**O problema que resolve:** dados em **trânsito** já são criptografados (TLS). Dados em **repouso** já são criptografados no disco (CMEK, default encryption). Mas e quando os dados estão **carregados na RAM**, sendo processados pelo CPU? **Em uso = em claro na memória.**

Para workloads ultra-sensíveis (financeiro, governo, healthcare, multi-party computation), isso é uma exposição inaceitável — alguém com acesso à RAM do host (físico ou via exploit do hypervisor) veria seus dados.

**Confidential VM = criptografia de memória em uso, sem mudar o código.**

**Como funciona:**
- É um **tipo específico de VM N2D** (machine family N2D).
- Roda em hosts com processadores **AMD EPYC de 2ª geração ("Rome")**.
- Usa **AMD SEV (Secure Encrypted Virtualization)** — feature do processador que criptografa a memória **na hora** que dados são gravados, com uma chave que **fica dentro do processador** e que **nem o Google tem acesso**.
- A criptografia é **inline** — feita pelo hardware, durante leitura/escrita normal de memória. Penalidade de performance é **muito pequena**.

**Diferenciais do vídeo:**
1. **Zero mudança de código** — sua aplicação roda exatamente igual. A criptografia é transparente.
2. **Sem grande penalidade de performance** — AMD SEV é otimizado para isso.
3. **Otimizado para workloads enterprise** com **alta memória** e **alto throughput**.
4. **Compute-heavy + paralelo** — AMD Rome é forte em workloads paralelos pesados.
5. **Google não tem as chaves** — a chave de criptografia é gerada e mantida **dentro do processador AMD**. Nem engenheiros do Google conseguem ler a memória da sua Confidential VM.
6. **Colaboração segura** — você pode rodar análises com parceiros sobre dados sensíveis sem expor o conteúdo.

**Como criar:**
- Console Cloud → seleciona "Confidential VM service" ao criar a VM.
- API Compute Engine → flag específica.
- CLI: `gcloud compute instances create` com `--confidential-compute` (e tipo N2D).

**Diferença crítica vs Shielded VM:**

| | Shielded VM | Confidential VM |
|---|---|---|
| **Protege contra** | Adulteração no boot/kernel | Acesso à memória durante processamento |
| **Como funciona** | Secure Boot + vTPM + Integrity Monitoring | AMD SEV + chave no processador |
| **Onde a proteção age** | Antes da VM rodar (boot) + monitoramento contínuo | Durante a execução (RAM em uso) |
| **Tipo de máquina** | Várias famílias suportadas | **Só N2D (AMD Rome)** |
| **Muda o código?** | Não | Não |
| **Custo extra** | Não | Sim (CPU específica AMD; pode ter pequeno overhead) |

> **Armadilha de prova:**
> - "Precisamos processar dados sensíveis sem expor em memória nem para o provedor" → **Confidential VM**.
> - "Preciso garantir que o boot não foi adulterado" → **Shielded VM** (não Confidential).
> - "Confidential VM em N2 standard" → ❌ **só funciona em N2D**.
> - "Confidential VM exige reescrita do código?" → ❌ não, é transparente.

---

### Resumo prático em 1 frase por tópico

- **Preemptible VM:** descarte-e-recomeço; 60–91% off; **máx 24h**; 30s de aviso; sem live migration; sem auto-restart; combine com **MIG** para resiliência.
- **Spot VM:** versão moderna do preemptible; **sem limite de runtime**; mesma economia; capacity vem de excess + backup; prefira **machine types menores** para conseguir capacity.
- **Sole-tenant node:** servidor físico **dedicado ao seu projeto**; resolve **compliance** (isolamento físico) e **BYOL** com licenças por core.
- **Shielded VM:** boot íntegro e verificável; Secure Boot + vTPM + Integrity Monitoring; protege contra **rootkit e malware de boot**; sem custo extra.
- **Confidential VM:** criptografa **RAM em uso** com AMD SEV; **só em N2D**; Google **não tem a chave**; zero mudança de código.

> **Mnemônico para diferenciar Shielded vs Confidential:**
> - **Shielded** = "escudo" no **boot** (parou ataque antes da VM ligar direito)
> - **Confidential** = "confidencial" na **memória** (ninguém vê o que está sendo processado)

### O boot disk — cada VM nasce com um

Toda VM no Compute Engine **nasce com um disco**: o **root persistent disk**, ou **boot disk**.

**Por quê?** Porque você escolheu uma **imagem base** (Linux/Windows) ao criar a VM. Essa imagem precisa estar **gravada em algum lugar** para a VM iniciar. Esse "lugar" é o boot disk.

**Duas propriedades importantes do boot disk:**
1. **Bootable** — você pode anexar ele a uma VM e dar boot a partir dele.
2. **Durable** — ele **sobrevive ao término da VM**, *se você quiser*.

**A pegadinha do "Delete boot disk when instance is deleted":**
- Por padrão, quando você **deleta** uma VM, o boot disk dela é **deletado junto**.
- Para preservar o boot disk após deleção da VM, você precisa **desmarcar** a opção `Delete boot disk when instance is deleted` nas propriedades da instância.
- Caso de uso: você quer manter o disco com a configuração toda salva, deletar a VM (por questão de custo), e mais tarde criar uma VM nova **a partir desse disco**.

> 🎯 **Pegadinha de prova:** "Deletamos a VM mas precisamos preservar os dados do disco" → a resposta envolve **desmarcar a opção de deletar boot disk junto** (ou ter feito **snapshot** antes).

---

### Persistent Disk — o conceito-chave

O **Persistent Disk (PD)** é o tipo de armazenamento principal do Compute Engine. Entender o que ele **é** e o que ele **não é** muda tudo.

**O PD NÃO é um disco físico conectado à sua VM.** Ele é:
- Um **dispositivo de bloco virtual**
- Anexado à VM **através da interface de rede**
- Fisicamente armazenado em vários discos físicos do datacenter (com redundância)

**Implicação fundamental dessa separação:**

```
[ VM ]  ←──── rede ────→  [ Persistent Disk ]
```

Como o disco está **separado** da máquina via rede, ele:
- ✅ **Sobrevive** se a VM for terminada
- ✅ Pode ser **destacado** de uma VM e **reanexado** em outra
- ✅ Pode ser **redimensionado dinamicamente** enquanto a VM roda (sem reboot)
- ✅ Pode ser **anexado em modo read-only a múltiplas VMs ao mesmo tempo**
- ✅ Tem **snapshots incrementais** nativos
- ✅ É **automaticamente criptografado** at-rest
- ✅ Tem **redundância** built-in (os bits estão espalhados em vários discos físicos)

**Read-only multi-attach — feature econômica que cai em prova:**
- Você pode anexar **o mesmo PD em modo read-only** a **várias VMs simultaneamente**.
- Caso de uso: **dados estáticos compartilhados** (ex.: dataset de referência, assets de jogo, modelo de ML para servir inferência).
- **Mais barato** que replicar os dados em discos separados por VM.

> 🎯 **Pegadinha:** "Como compartilhar dados estáticos entre várias VMs sem replicar o storage?" → **PD read-only multi-attach**. Não confunda com Filestore (NFS, leitura+escrita).

---

### Zonal vs Regional Persistent Disk — a dimensão da disponibilidade

Persistent Disks vêm em **duas variantes** baseadas em **escopo geográfico**:

| Aspecto | **Zonal PD** | **Regional PD** |
|---|---|---|
| Escopo | Vive em **1 zona** | Replicado em **2 zonas** da mesma região |
| Replicação | Dentro da zona | **Síncrona** entre 2 zonas (active-active) |
| Disponibilidade | Cai se a zona cair | **Sobrevive à queda de 1 zona** |
| Custo | Mais barato | ~2x o preço (você paga as 2 cópias) |
| Caso de uso | Apps tolerantes a zona / não-críticos | **HA databases, enterprise apps críticos** |

**O detalhe técnico do Regional PD:**
- A replicação é **síncrona** — cada write só é confirmado depois que **ambas as zonas** receberam os dados.
- Resultado: latência um pouco maior por write, mas **zero perda de dados** em falha de zona.
- "Active-active" = ambas as cópias estão sempre sincronizadas e prontas para uso.

> 🎯 **Pegadinha:** "Banco de dados com alta disponibilidade entre zonas com replicação síncrona" → **Regional PD**. "Database HA com replicação assíncrona" → tipicamente é replicação no nível da aplicação, **não** Regional PD.

---

### Os 4 tipos de Persistent Disk

Quando você cria um PD (zonal ou regional), escolhe **um dos 4 tipos**:

| Tipo | Tecnologia | IOPS | Preço por GB | Caso de uso |
|---|---|---|---|---|
| **Standard (pd-standard)** | HDD (spinning) | Baixo | 💰 Mais barato | Workloads de **grandes volumes** com I/O **sequencial** (ETL, data lake bruto, logs) |
| **Balanced (pd-balanced)** | SSD | Médio-alto | 💰💰 Médio | **Uso geral** — escolha default para a maioria dos apps |
| **Performance SSD (pd-ssd)** | SSD | Alto | 💰💰💰 Mais caro | **Apps enterprise e DBs** que precisam de **latência baixa + IOPS alto** |
| **Extreme (pd-extreme)** | SSD | **Você provisiona IOPS desejado** | 💰💰💰💰 Premium | **DBs de alta performance** (SAP HANA, MongoDB grande, Cassandra) |

**Detalhes que caem em prova:**

1. **Balanced vs Performance SSD** — confusão comum:
   - **Mesmo IOPS máximo** que Performance SSD
   - **Menor IOPS por gigabyte**
   - Para **a maioria das VMs (exceto as muito grandes)**, Balanced entrega performance suficiente para apps de uso geral, **a preço menor** que Performance SSD
   - **Default recomendado** quando você não sabe qual escolher

2. **Extreme PD** — único onde você **provisiona o IOPS desejado**:
   - Nos outros tipos, IOPS escala automaticamente com o tamanho do disco
   - No Extreme, você diz: "quero 100k IOPS"
   - Workloads que cabem aqui: high-end OLTP, NoSQL massivo, analytics tempo-real

3. **Standard (HDD)** é a única opção **não-SSD**:
   - Otimizado para **I/O sequencial** (ler/escrever blocos grandes em ordem)
   - Ruim para **I/O random** (banco de dados típico, por exemplo)
   - Use para data lakes, backups, logs.

> 🎯 **Regra simples de prova:**
> - "Volume grande + sequencial + custo baixo" → **Standard**
> - "Uso geral, default razoável" → **Balanced**
> - "DB enterprise com IOPS alto" → **Performance SSD**
> - "DB high-end com IOPS provisionado" → **Extreme**

### Encryption at-rest — os 3 níveis de controle

**Toda data em PD é criptografada at-rest por padrão.** Não tem opção de desligar. A diferença é **quem controla a chave**:

| Modelo | Quem gerencia a chave? | Quando usar |
|---|---|---|
| **Google-managed (padrão)** | Google cuida de tudo automaticamente | Default — funciona pra 90% dos casos |
| **CMEK (Customer-Managed Encryption Keys)** | Você cria/gerencia chave no **Cloud KMS** (serviço do Google) | Compliance que exige **controle da chave** com auditoria |
| **CSEK (Customer-Supplied Encryption Keys)** | Você gera e fornece a chave **fora do Google** a cada operação | Compliance que exige **chave NUNCA dentro do Google** |

**Diferença crítica CMEK vs CSEK:**
- **CMEK** — a chave **existe no Cloud KMS** (dentro do Google), mas com você no controle (rotação, revogação, audit log). O Google **usa** a chave em seu nome.
- **CSEK** — a chave **nunca fica armazenada no Google**. Você manda a chave a **cada operação** (read/write), o Google usa, descarta. Se você perder a chave, **perdeu os dados**.

> 🎯 **Pegadinha de prova:**
> - "Quero controlar minhas chaves de criptografia com audit e rotação" → **CMEK** (Cloud KMS)
> - "Compliance exige que a chave **nunca** esteja armazenada no Google" → **CSEK**
> - "Quero criptografar mas sem trabalho" → **default Google-managed** (já vem ligado)

---

### Local SSD — o "disco físico" do hospedeiro

Diferente do Persistent Disk, o **Local SSD** é **fisicamente conectado** ao servidor físico onde sua VM roda.

**Características-chave:**

| Aspecto | Local SSD |
|---|---|
| **Onde está fisicamente** | No host físico que hospeda a VM |
| **Conexão** | Direta (não pela rede) |
| **Performance** | **A mais alta** entre os discos persistentes — IOPS muito altos, latência mínima |
| **Tamanho de cada partição** | **375 GB** (tamanho fixo) |
| **Quantidade máxima** | Até **24 partições** = **9 TB** por instância |
| **Sobrevive a `reset`?** | ✅ Sim |
| **Sobrevive a `stop` / `terminate`?** | ❌ **Não** — os dados **somem** |
| **Pode ser reanexado a outra VM?** | ❌ Não — dados são amarrados ao host físico |
| **Snapshot?** | ❌ Não suporta |

**Por que Local SSD é mais rápido?**
- Sem hop de rede entre VM e disco
- Conexão PCIe direta com o host
- Sem replicação síncrona pra outras zonas (sem custo de redundância)

**O preço: você perde os dados se a VM parar.**
- `reset` (reboot da VM, fica no mesmo host) → dados preservados ✅
- `stop` ou `terminate` (VM solta o host) → dados perdidos ❌
- Manutenção de host com live migration → **dados podem ser perdidos** (porque Local SSD não migra)

**Casos de uso típicos:**
- **Cache** de banco de dados (Redis, Memcached) que pode ser repopulado
- **Swap** / **scratch space** para jobs de processamento
- **Buffer temporário** em pipelines de dados (ex.: shuffle space do Spark)
- Tudo que cabe em "se eu perder, eu reconstruo"

> 🎯 **Pegadinha:** "Dados precisam sobreviver após `stop` da VM" → ❌ Local SSD **não serve**. "Máxima performance de IOPS para dados temporários" → ✅ Local SSD.

---

### RAM disk — o nível mais extremo de performance (e volatilidade)

Se você precisa de **performance ainda maior** que Local SSD e os dados podem ser totalmente voláteis, você pode usar **tmpfs** (filesystem em memória) — a famosa "RAM disk".

**Como funciona:**
- Você monta um filesystem em **RAM** (tmpfs no Linux)
- Operações de I/O ficam **na velocidade da memória** (nanossegundos)
- Os dados **somem** quando a VM desliga (ou quando você desmonta o tmpfs)

**Recomendação do Google (do vídeo):**
- Use uma **VM high-memory** para ter RAM sobrando
- **Faça backup dos dados num Persistent Disk** — porque a RAM é totalmente volátil

**Quando usar:**
- Pequenas estruturas de dados que precisam de **acesso ultra-rápido**
- Cache em memória que não precisa ser inicializado fora do processo
- Buffer de operação muito frequente

---

### Resumo de persistência e redundância

| Tipo | Performance | Redundância (dados em múltiplos discos físicos) | Sobrevive reboot | Sobrevive stop/terminate |
|---|---|---|---|---|
| **Persistent Disk (HDD)** | Baixa | ✅ Built-in | ✅ | ✅ |
| **Persistent Disk (SSD/Balanced/Extreme)** | Média a alta | ✅ Built-in | ✅ | ✅ |
| **Local SSD** | **Muito alta** | ❌ **Sem redundância** | ✅ (reset) | ❌ |
| **RAM disk (tmpfs)** | **Máxima** | ❌ | ❌ | ❌ |

> **Regra mental:** *mais rápido = menos durável*. Você escolhe a posição que faz sentido para o workload.

---

### Limites — quantos discos cabem numa VM?

A **quantidade de discos** que você pode anexar a uma VM **depende do machine type**:

| Machine type | Limite de Persistent Disks |
|---|---|
| **Shared-core** (`e2-micro`, `e2-small`, `e2-medium`, `f1-micro`, `g1-small`) | **16 discos** |
| **Standard, High-Memory, High-CPU, Memory-Optimized, Compute-Optimized** | **128 discos** |

**Implicação:** numa VM "grande", você consegue criar **capacidade massiva** num único host (128 discos × tamanho de cada um).

---

### A pegadinha do bandwidth compartilhado

> ⚠️ **Lembra da regra "2 Gbps por vCPU" de throughput de rede?** Ela vale também para **Disk I/O**.

A bandwidth disponível na VM é **compartilhada entre tráfego de rede e Disk I/O**.

**Implicação prática:**
- Se sua VM tem **2 vCPUs (até 10 Gbps)** e você satura o disco lendo a 8 Gbps, **sobra pouca banda para tráfego de rede**.
- Se você anexa **muitos discos** numa VM pequena, esperando alta vazão agregada, vai **bater no teto da bandwidth do host** e degradar tudo.

**Solução:** se você precisa de **alto Disk I/O E alto Network I/O simultaneamente**, escolha uma VM com **mais vCPUs** (a banda escala com vCPU).

> 🎯 **Pegadinha de prova:** "Por que minha VM não atinge o IOPS esperado mesmo com SSD grande?" → pode ser **bandwidth shared** entre disk e rede, ou **vCPUs insuficientes** (IOPS também escala com vCPU em alguns casos).

---

### Persistent Disk vs disco físico — a diferença filosófica

Esse é o ponto que o vídeo termina destacando e que vale ler com atenção:

| Operação | Disco físico (on-premises) | Persistent Disk (Cloud) |
|---|---|---|
| **Particionar** | Você precisa criar partições manualmente | Não precisa — é um volume virtual |
| **Crescer disco** | Repartition + (às vezes) reformat | **Resize on-the-fly**, sem reboot |
| **Redundância** | Você monta **RAID** manualmente | **Built-in** — Google espalha bits em vários discos físicos |
| **Encryption** | Você criptografa arquivos por aplicação | **Automática** at-rest (com opção de CMEK/CSEK) |
| **Snapshot** | Você instala um software de backup | **Built-in** — `gcloud compute disks snapshot` |
| **Atachar a outra máquina** | Desmonta + monta fisicamente | **Detach + Reattach** via API/Console |
| **Compartilhar (read-only) entre máquinas** | Complicado (NFS, SAN) | **Multi-attach read-only** nativo |

**A lição:** quando você passa de on-premises pra Cloud, **muita complexidade de gerenciamento de storage desaparece**. Você ganha tempo para focar na aplicação.

---

### Resumo prático em 1 frase por tópico

- **Boot disk:** todo VM tem um root PD; desmarque "Delete boot disk on instance delete" para preservá-lo.
- **Persistent Disk:** virtual, em rede, sobrevive à VM, redimensionável on-the-fly, snapshotável, criptografado.
- **Zonal vs Regional PD:** Regional replica **síncrono em 2 zonas** (HA + ~2x preço).
- **4 tipos de PD:** Standard (HDD, sequencial), Balanced (SSD, default), Performance SSD (DB enterprise), Extreme (IOPS provisionado).
- **Encryption:** sempre ligada; CMEK = chave no Cloud KMS; CSEK = chave fora do Google.
- **Local SSD:** 375 GB cada, até **24 partições = 9 TB**; ultra-rápido; **sobrevive reset, não sobrevive stop/terminate**.
- **RAM disk (tmpfs):** mais rápido de todos, totalmente volátil; combine com PD para backup.
- **Limite de discos:** 16 (shared-core) ou **128** (todos os outros).
- **Bandwidth:** Disk I/O **compartilha** banda com rede — escale vCPUs se precisar dos dois.
- **PD ≠ disco físico:** sem partition manual, redundância built-in, resize on-the-fly, snapshot nativo.

- ❌ **Zone**

- ✅ Firewall rules associadas

> 2. Parar a VM e criar a imagem com o disco ainda intacto.
>

**Por que custom machine type existe:**

Os predefinidos (`e2-standard-N`, `e2-highmem-N`, `e2-highcpu-N`) seguem proporções fixas:
- standard: **1 vCPU : 4 GB**
- highmem: **1 vCPU : 8 GB**
- highcpu: **1 vCPU : 1 GB**

Se sua app precisa de **2 vCPU + 4 GB** (que é 1:2, sem predefinido E2 que case), você usa custom. Você paga **só pelo que escolheu** — sem desperdiçar capacidade de um predefinido maior.

**Famílias que suportam custom:** E2, N1, N2, N2D.

**Comandos rodados via SSH (e o que cada um revela):**

```bash
free
```
Mostra **memória usada/livre** e swap. Pra confirmar que a VM realmente tem ~4 GB de RAM (sai com `Mem: total ~4G`).

```bash
sudo dmidecode -t 17
```
Lê **DMI / SMBIOS type 17** = "Memory Device". Mostra detalhes de cada slot de RAM "instalado". Em VM o GCP **emula** isso — você vê os pentes virtuais que o hypervisor apresenta.

```bash
nproc
```
Conta o **número de unidades de processamento** que o SO enxerga. Output: `2` (porque pedimos 2 cores).

```bash
lscpu
```
Detalha **CPU**: arquitetura, modelo, threads por core, sockets, cache, flags de virtualização. Útil pra ver qual plataforma de CPU o Google alocou (Intel Broadwell, Skylake, Cascade Lake, AMD Rome, etc.).

**Por que esses comandos no lab:** o Google quer que você **prove pra você mesmo** que o que você pediu na criação (2 vCPU + 4 GB) é o que a VM realmente entrega. Custom machine types às vezes geram dúvida — esses comandos eliminam.

```bash
exit
```
Encerra a sessão SSH. A VM continua rodando.

### Metadata Server e Scripts

**O que é:**
Cada instância no Compute Engine tem acesso a um **metadata server** (servidor de metadados) local que armazena informações sobre a VM (como IP externo, chaves SSH, project ID, etc.). 

**Por que / Quando usar:**
É extremamente útil em combinação com **startup scripts e shutdown scripts**. 
- Um script pode consultar o metadata server internamente (sem precisar de autenticação adicional) para descobrir seu próprio IP e configurar um banco de dados, por exemplo.
- Como as chaves de metadados padrão são as mesmas para todas as VMs, você pode escrever um script **genérico e reutilizável** em várias instâncias diferentes.
- **Boa prática recomendada:** armazene os startup e shutdown scripts no **Cloud Storage** em vez de embuti-los diretamente na configuração da VM, para facilitar o versionamento e a manutenção.

> **Armadilha de prova:** se a questão pergunta como uma VM recém-criada pode descobrir dinamicamente configurações sobre o projeto ou sobre si mesma sem usar service accounts extras → a resposta é consultar o **Metadata Server**.

### Mover instâncias entre zonas ou regiões

**O que é:**
Mudar fisicamente a VM de um data center (zona) para outro, seja por minimizar latência para usuários, razões geográficas ou porque a zona está sendo descontinuada.

**Como funciona:**
O processo manual envolve:
1. Desligar a VM de origem (`stop`).
2. Criar uma **machine image** (imagem da máquina) a partir da VM de origem.
3. Criar uma **nova VM** na zona de destino a partir dessa machine image.
4. Atualizar referências (IPs, target pools, load balancers) porque propriedades geradas pelo servidor (como IP efêmero) podem mudar.

> **Armadilha de prova:** "Preciso mover minha VM de `us-central1-a` para `us-east1-b` sem mudar nenhuma propriedade gerada pelo servidor" → Impossível fazer live migration entre zonas, e IPs efêmeros vão mudar. A VM **precisa ser desligada**.

---

### Persistent Disk Snapshots

**O que é:**
Um backup point-in-time dos dados de um disco. 
- Snapshots são armazenados automaticamente de forma durável no **Cloud Storage**.
- Eles são **incrementais** e **automaticamente comprimidos**, o que significa que fazer snapshots regulares é muito mais barato e rápido do que criar full images do disco repetidas vezes.
- Importante: snapshots suportam apenas **persistent disks** (HDD/SSD/Balanced), **NÃO suportam Local SSDs**.

**Quando usar:**
1. **Backup:** automatizado via **snapshot schedules** para recuperação de desastres (DR) e disponibilidade.
2. **Migração zonal de dados:** transferir dados de uma zona para outra restaurando o snapshot num novo disco na zona de destino (minimiza latência localizando o disco perto da computação).
3. **Upgrade de performance:** restaurar um snapshot de um disco HDD (Standard) para um novo disco SSD para melhorar o I/O instantaneamente.

> **Armadilha de prova:**
> - "Como fazer backup regular e barato de um disco de VM" → **Snapshot schedule** (porque é incremental e comprimido).
> - "Preciso de um backup dos dados que estão no Local SSD" → Snapshots **não suportam** Local SSD. Você precisará copiar manualmente os dados a nível de SO (ex: rsync para o Cloud Storage).
> - **Snapshot vs Imagem:** Snapshot é primariamente para **backup periódico de dados**. Custom Image é para criar instâncias-molde para escalar.

---

### Redimensionar Discos Persistentes (Resize)

**O que é:**
Aumentar a capacidade de armazenamento de um disco existente.

**Por que / Quando usar:**
- Além do óbvio ganho de espaço, discos no GCP têm sua performance (IOPS e Throughput) vinculada ao seu tamanho. Aumentar o tamanho do disco **melhora a performance de I/O**.
- Pode ser feito **enquanto o disco está anexado a uma VM em execução** e **sem precisar criar snapshot** antes (hot-resize).

> **Armadilha de prova:**
> - Você **só pode crescer** um disco, **nunca encolhê-lo**. Se a prova pedir para diminuir um disco existente para economizar dinheiro, não é possível fazer in-place. Terá que provisionar um menor e migrar os dados no nível do SO.
> - "Minha VM com Persistent Disk precisa de mais IOPS no banco de dados, o que fazer?" → Aumente o tamanho do disco.

## Armadilhas de prova

1. **TPU ≠ GPU:** TPU é para treinar modelos de ML de larga escala. GPU é para processamento gráfico e também serve para ML, mas TPU é mais eficiente e específico para isso.
2. **Exceção de throughput:** 2 e 4 vCPUs recebem até **10 Gbps**, não 4 e 8 Gbps. Memorize essa exceção.
3. **Local SSD é temporário:** dados somem quando a VM para ou é deletada. Não use para dados que precisam persistir.
4. **Standard vs SSD:** ambos são persistent disks do mesmo tamanho máximo (257 TB). A diferença é IOPS por dólar (SSD) vs capacidade por dólar (Standard).
5. **Load balancer no GCP não precisa de pre-warming:** é software, não hardware. Absorve picos sem preparação.
6. **Stop ≠ Suspend ≠ Reset ≠ Delete:** Stop encerra processos e libera CPU/RAM (paga só disco+IP); Suspend salva RAM em disco (hibernar — paga só storage da memória); Reset é power cycle sem shutdown script (fica em RUNNING); Delete some.
7. **REPAIRING não tem SLA e não cobra:** se sua VM "sumiu" e o billing parou, ela pode estar em REPAIRING (não TERMINATED — em TERMINATED você ainda paga disco).
8. **Shutdown de preemptible tem ~30s, não 90s:** depois disso vem ACPI G3 (corte abrupto). Shutdown scripts em preemptible precisam ser rápidos e idempotentes.
9. **Imagem não pode ser trocada em VM existente:** a imagem virou disco no momento da criação. Pra mudar, recrie a VM ou crie novo boot disk a partir de outra imagem.
10. **Machine type exige VM parada; metadata e SSH keys não:** mudar CPU/RAM só com a VM em TERMINATED. Já SSH keys, snapshot, anexar disco — tudo isso é hot.
11. **Rede `default` libera SSH (22) e RDP (3389) automaticamente:** em VPC custom você precisa criar firewall rules manualmente. Causa #1 de "não consigo SSH na minha VM".
12. **`onHostMaintenance=TERMINATE` para VMs que não toleram migração:** GPUs já forçam isso. Pra licenciamento por host ou hardware dependency específica, configure manualmente.
13. **Não dá pra criar imagem de boot disk anexado a VM rodando:** pra criar uma system image a partir do boot disk, **pare a VM** primeiro OU desabilite "Delete boot disk when instance is deleted" e delete a VM (o disco sobrevive). Esse é um detalhe operacional clássico de prova.
14. **Preemptibility é decidida na criação, não dá pra converter depois:** VM normal **não vira** preemptible e vice-versa. Se a questão diz "convertemos a VM para preemptible pra economizar" — está errado por definição.
15. **Custom machine type só em famílias específicas:** E2, N1, N2, N2D. Famílias como C2, C3, M-series **não suportam custom** (você escolhe entre os predefinidos).
16. **Image family vs Image fixo:** `--image-family=debian-12` sempre pega a versão mais recente da família (você fica auto-atualizado). `--image=debian-12-bookworm-v20240101` trava na versão exata (reproduzibilidade). Em prova, prefira `--image-family` pra "sempre na última" e `--image` pra "exatamente essa versão".
17. **Minuto mínimo na cobrança:** VM rodando 20 segundos → você paga 1 minuto. Importante pra workloads que sobem/derrubam VMs muito rápido.
18. **SUD reseta no dia 1 do mês:** pra ganhar 30% máximo, a VM tem que estar ligada desde o início do mês — não adianta subir dia 20 e querer SUD máximo do mês corrente.
19. **SUD agrega recursos sobrepostos entre VMs:** redimensionar a VM (n1-standard-4 → n1-standard-16) NÃO zera o SUD acumulado da parte sobreposta (4 vCPU + 15 GB). O Google decompõe em vCPU + RAM individuais antes de calcular.
20. **SUD predefined ≠ SUD custom:** as duas categorias são pools separados pra cálculo. Não confunda.
21. **Descontos não combinam:** CUD não soma SUD; Preemptible/Spot já têm desconto embutido e não somam outros. Em DQ que oferece "combinar preemptible + sustained use", é pegadinha.
22. **CUD memory-optimized vai até 70%:** o resto vai até 57%. Se a questão menciona "máximo desconto para SAP HANA" → memory-optimized + CUD 3 anos = 70%.
23. **Preemptible tem cap de 24h; Spot não tem:** se precisa rodar batch mais longo que 24h sem ser forçado a parar → Spot.
24. **VM sizing recommendations aparece 24h depois da criação:** se a questão menciona "recém-criada, sem recomendação ainda" — é esperado.
25. **Metadata Server para dados dinâmicos:** Para uma VM descobrir coisas sobre si mesma (IP externo, etc.) sem auth extra, consulte o Metadata Server.
26. **Mover VM entre zonas:** Exige downtime. Processo envolve criar machine image e subir nova VM. IPs efêmeros mudam.
27. **Snapshots são incrementais e comprimidos:** Muito mais eficientes e baratos que custom images para backup periódico de discos.
28. **Snapshots NÃO suportam Local SSD:** Apenas Persistent Disks podem ter snapshots.
29. **Redimensionar Persistent Disk:** Melhora performance (IOPS/Throughput). Pode ser feito a quente (com a VM rodando). **Só pode crescer, nunca encolher**.
