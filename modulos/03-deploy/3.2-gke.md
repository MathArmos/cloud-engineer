### Sub-bloco 1 — Cloud computing: definição NIST-style (5 traços)

**Por que importa.** Esses 5 traços compõem a definição padrão que o Google (e a maioria dos provedores) usa para diferenciar "cloud" de "datacenter terceirizado". Cai literal em quiz introdutório.

| # | Traço | Significado prático |
|---|-------|---------------------|
| 1 | **On-demand & self-service** | Provisiona via web UI / API, sem ticket pra humano |
| 2 | **Acesso pela rede** | De qualquer lugar com conexão (não amarrado a hardware no escritório) |
| 3 | **Resource pooling** | Provider compra em escala → economia repassada; cliente não sabe (nem precisa saber) onde fica o hardware exato |
| 4 | **Elasticidade** | Escala **pra cima E pra baixo**, rápido — não só pra cima |
| 5 | **Pay-per-use** | Cobrança por consumo/reserva; parou de usar, parou de pagar |

**Armadilha do quiz (Q2 — verificado):** "elasticidade" sem mencionar reduzir recursos é resposta **incompleta**. A definição inclui obrigatoriamente os dois sentidos (scale-up E scale-down).

### Sub-bloco 2 — As 5 opções de compute do GCP (lente do curso GKE)

**Por que importa.** O curso apresenta GKE no contexto de **quando NÃO usar GKE**. Saber discriminar as 5 opções é o que separa um engenheiro júnior ("usa GKE pra tudo") de um sênior ("escolhe pela carga").

| Serviço | Modelo | Granularidade | Caso canônico |
|---------|--------|---------------|---------------|
| **Compute Engine** | IaaS | VM | Lift-and-shift, controle total do SO, mix de SOs, workloads com requisitos específicos de kernel |
| **GKE** | CaaS (managed K8s) | Cluster + Pods | Orquestração de containers, multi-tenant, microservices com requisitos avançados (HPA, network policies, sidecars) |
| **App Engine** | PaaS | App / serviço | Quer focar em código, sem gerenciar infra; web apps e APIs em linguagens suportadas (Java, Node, Python, PHP, C#/.NET, Ruby, Go) |
| **Cloud Run** | Serverless container | Container stateless | Container que escala **de zero**, request-driven, billing em 100ms |
| **Cloud Run functions** | FaaS | Função | Trigger event-driven (objeto no Cloud Storage, mensagem em Pub/Sub); free tier perpétuo |

**Pontos-chave de cada uma que o curso destacou:**

**Compute Engine**
- VMs até **416 vCPUs / 12+ TB RAM** (cap citado pelo curso à época da gravação — checar `gcloud compute machine-types list` para valor atual)
- Block storage: **Persistent Disks** (até 257 TB, snapshots, mobilidade) ou **Local SSDs** (alto IOPS, ephemeral)
- Autoscaling: **Managed Instance Groups + Global LB**
- Billing: **per-second** após o 1º minuto
- **Preemptible VMs** (e Spot VMs, sucessoras) — desconto agressivo para workloads tolerantes a interrupção

**App Engine**
- Integração nativa com **Cloud Monitoring, Logging, Profiler, Error Reporting**
- **Versionamento** e **traffic splitting** built-in (canary/blue-green sem mais nada)
- Bom para: **websites, backends de mobile/game, APIs RESTful**

**Cloud Run**
- Base **Knative** (API aberta, portável entre GKE / outro Knative / GCP managed)
- Escala **de zero a N quase instantaneamente**
- Billing em **100ms** — nunca paga por idle/overprovisioning

**Cloud Run functions** (antigamente Cloud Functions / Cloud Functions 2nd gen)
- Linguagens: Node.js, Python, Go, Java, .NET Core, Ruby, PHP
- **Free tier perpétuo** — muitos casos de uso ficam de graça
- Casos de uso: microservices, processar arquivos em buckets em tempo real, ETL leve, integração com APIs externas, backends de IoT/mobile

**Armadilha do quiz (Q3 — verificado):** "managed compute platform com suporte **nativo** a containers" = **GKE**. Cloud Run também roda containers, mas a "lente" da pergunta enfatiza orquestração nativa de containers — GKE é a resposta canônica do quiz.

> **Referência detalhada por serviço:** [3.1-compute-engine.md](3.1-compute-engine.md) (Compute Engine), [3.3-cloud-run.md](3.3-cloud-run.md) (Cloud Run/App Engine/Cloud Run functions), o resto deste arquivo (GKE).

### Sub-bloco 3 — Infraestrutura global (rede + regions + zones)

**Por que importa.** A rede do Google é o "produto invisível" que torna LB global, replicação multi-region e cache em edge possíveis. Entender isso explica decisões de pricing (egress, multi-region storage) e arquitetura (Premium vs Standard tier).

**Dados que o curso citou:**
- **~40% do tráfego mundial da internet** trafega na rede Google.
- **100+ content caching nodes** distribuídos globalmente (edge POPs).
- **7 macro-regiões geográficas:** North America, South America, Europe, Africa, Middle East, Asia, Australia.
- À época da gravação: **41 regions / 124 zones**. Valor atualizado em `cloud.google.com/about/locations`.

**Hierarquia geográfica:**

```
Multi-Region (ex: "US", "EU")
  └── Region (ex: us-east1, europe-west2)
        └── Zone (ex: us-east1-c, europe-west2-a)
              └── Resources (VMs, Persistent Disks, etc.)
```

| Nível | Failure domain | Exemplo de uso |
|-------|----------------|----------------|
| **Zone** | Falha de DC/rack | VM única, Local SSD |
| **Region** | Falha regional inteira (raríssimo, mas existe) | Regional MIG, Cloud SQL HA, Regional PD |
| **Multi-Region** | Resiliência continental | Cloud Storage multi-region, Spanner multi-region |

> Para detalhes de HA/DR multi-zona vs multi-region, **ver [01-setup-environment/1.1-resource-hierarchy-iam.md](../01-setup-environment/1.1-resource-hierarchy-iam.md)** seção "Classificação global/regional/zonal".

### Sub-bloco 5 — Billing + Quotas

**O quê:** billing fica no nível **Project**; um Project tem **1 billing account ativa** ou **0** (só free tier); uma billing account pode estar linkada a **N Projects**; **billing subaccounts** servem para resellers separarem por cliente.

**4 ferramentas de controle de gasto:**

| Ferramenta | Onde aplica | Função |
|------------|-------------|--------|
| **Budget** | Billing account OU Project | Define limite (fixo ou % do mês anterior) |
| **Alert** | Vinculado a um budget | Notifica em % do limite (default 50/90/100, customizável) |
| **Reports** | Console | Visualização de gastos por project/service |
| **Quota** | Project | Limita uso de recursos (anti-erro/anti-abuso) |

**Quota — 2 tipos (cai em prova):**

| Tipo | O que limita | Reseta? | Exemplo |
|------|--------------|---------|---------|
| **Rate quota** | Chamadas/operações por janela de tempo | ✅ sim, após a janela | 3.000 calls/100s na API GKE |
| **Allocation quota** | Quantidade simultânea de recursos | ❌ não, é capacidade | Máx 5 VPCs por project |

**Aumento de quota:** via **Google Cloud Support request** (nem todas são aumentáveis — depende da quota).

**Armadilha do quiz (Q1 — verificado):** "ferramentas para controlar gasto **antes** que se torne excessivo" = **Budgets + Alerts no nível do projeto**. "Configurar billing account na pasta" é falacioso (billing é no project, não na folder). "Quotas por folder" também é falacioso (quotas são no project, não na folder, e limitam recursos, não custo).

### Sub-bloco 7 — Vídeo "What is a container?" — a evolução do deploy

**Por que importa.** O argumento "container existe porque VM tem limites" só convence se você entende **quais** limites. Esse vídeo monta a escada **bare metal → VM → container** mostrando o problema específico que cada degrau resolve e o que ainda fica em aberto. Cai em quiz introdutório, mas o uso real é em entrevista: explicar **por que** containers, não **o que** são.

**A escada completa:**

```
┌─────────────────────────────────────────────────────────────┐
│ Degrau 1 — DEPLOY EM HARDWARE FÍSICO                        │
│ 1 app por máquina (DB, web server, CDN em hosts separados)  │
│ Problema: desperdício de recurso, lento pra escalar         │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 2 — VIRTUALIZAÇÃO (hypervisor)                       │
│ N VMs compartilham 1 hardware via hypervisor (ex.: KVM)     │
│ Ganho: deploy rápido, portabilidade (image de VM)           │
│ Problemas remanescentes:                                    │
│   - VM carrega OS completo (boot demorado)                  │
│   - Difícil mover entre hypervisors diferentes              │
│   - Apps no mesmo OS disputam deps e recursos               │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 2.5 — VM POR APP (tentativa ingênua de isolar)       │
│ 1 VM por aplicação → cada uma com seu kernel completo       │
│ Funciona, mas N kernels rodando = overhead absurdo          │
│ Não escala pra centenas/milhares de apps                    │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 3 — CONTAINERS                                       │
│ Abstrai SÓ o user space (não a máquina nem o OS inteiro)    │
│ Kernel é compartilhado; app + deps ficam isolados           │
│ Lightweight, start/stop instantâneo (vira um processo OS)   │
└─────────────────────────────────────────────────────────────┘
```

**O que é "user space" (palavra crítica do vídeo):**

> User space = **tudo que roda acima do kernel** — aplicação + bibliotecas + dependências.
> O container virtualiza **só** essa camada. Kernel continua sendo o do host (compartilhado entre todos os containers).

**Por isso o container é "lightweight":**

| Comparativo | VM | Container |
|-------------|-----|-----------|
| O que está empacotado | App + deps + **OS completo** + kernel | App + deps **só** |
| Boot | Inicializa OS (segundos a minutos) | Inicia processo OS (milissegundos) |
| Footprint | GB | MB |
| Portabilidade entre hypervisors/runtimes | Difícil | Trivial (container engine padrão) |
| Isolamento entre apps no mesmo host | Total (kernels separados) | Forte (mesmo kernel, mas namespaces/cgroups isolam) |

**Os 3 problemas específicos que apps-no-mesmo-VM têm (e container resolve):**

1. **Starvation** — app A consome todo CPU/RAM, app B trava.
2. **Dependency hell** — upgrade da lib X pro app A quebra o app B que dependia da versão antiga.
3. **Lockdown dependências não escala** — você até consegue congelar versões, mas hora alguma vai precisar atualizar, e aí volta o problema.

**Tentativas "via processo" que NÃO funcionam (e por isso container venceu):**

- ❌ Rigorous engineering policies (não permitir upgrades) → impede evolução
- ❌ Integration tests pesados → lento, e ainda assim aparecem falhas exóticas em produção

**Os 3 benefícios canônicos que o vídeo enumera (decora isso pra prova):**

| # | Benefício | O que significa na prática |
|---|-----------|----------------------------|
| 1 | **Code-centric delivery** | Foco em entregar código, não em provisionar infra de baixo nível |
| 2 | **HW/SW reliability via Linux kernel** | Mesmo kernel-base no laptop, na CI e em produção → "works on my machine" desaparece |
| 3 | **Habilita microservices** | Loosely-coupled, fine-grained components que escalam e atualizam isoladamente |

**Analogia útil pra feira de trabalho:**

> VM é como **alugar uma casa inteira mobiliada** pra cada inquilino — privacidade total, mas cada casa tem cozinha/banheiro/fundação duplicados. Container é como **república com quartos individuais**: cada um tem sua privacidade (namespaces), sua cota de geladeira (cgroups), mas dividem a estrutura da casa (kernel). Mais barato, mais rápido pra entrar e sair.

### Sub-bloco 8 — Vídeo "Containers and container images" — anatomia técnica

**Por que importa.** Esse vídeo é onde a maior parte das **armadilhas de prova** mora: image ≠ container, Linux namespace ≠ K8s namespace, container layer é efêmero, multi-stage build é best practice. Também onde aparecem os serviços GCP relevantes (Artifact Registry e Cloud Build) — esses são alvo direto da ACE.

#### Image vs Container — distinção que cai literal

| Conceito | Definição | Analogia |
|----------|-----------|----------|
| **Container image** | Aplicação **+ todas as suas dependências** empacotadas | Receita de bolo |
| **Container** | **Instância em execução** de uma image | Bolo assado (rodando) a partir da receita |

**Implicação prática:** uma image gera N containers idênticos. Container morre, a image continua intacta.

#### Por que existem Docker E Kubernetes (e por que o curso usa Cloud Build em vez de Docker)

| Ferramenta | O que faz | Onde para |
|------------|-----------|-----------|
| **Docker** | Cria + roda containers em **um host** | ❌ Não orquestra containers entre hosts |
| **Kubernetes** | **Orquestra** containers (scheduling, scaling, networking, self-healing) entre N hosts | ✅ É a peça que falta |
| **Cloud Build** | Build managed pelo Google que **gera images Docker-formatted** | Substitui o `docker build` local no fluxo GCP |

> Pro curso (e pra prova): **você vai usar Cloud Build para criar a image** e GKE para rodar os containers. Docker continua sendo o formato da image.

#### As 4 tecnologias Linux que viabilizam containers

Container não é magia — é uma combinação de 4 features do kernel Linux. Saber **qual feature isola o quê** é diferencial em entrevista.

| # | Tecnologia | Isola/Controla | Exemplo prático |
|---|------------|----------------|-----------------|
| 1 | **Linux process** | Memória virtual isolada por processo | Cada container roda como um (ou poucos) processos do host |
| 2 | **Linux namespaces** | **O que o app PODE VER** (PIDs, dirs, IPs, hostname, users) | Container só "vê" seus próprios processos, não os do host |
| 3 | **Linux cgroups** | **O que o app PODE USAR** (CPU, RAM, I/O, network bandwidth) | Container limitado a 0.5 CPU e 256 MB RAM |
| 4 | **Union file systems** | Empacotamento em **camadas** read-only + 1 layer writable | Image fica como cebola de layers; container ganha 1 layer extra ao subir |

**Regra mnemônica:** **namespaces = visão**, **cgroups = consumo**. (Eu vejo o quê × eu posso usar quanto.)

**⚠️ Armadilha de prova clássica:**

> **Linux namespace ≠ Kubernetes namespace.** São conceitos diferentes que compartilham o nome.
> - **Linux namespace** = isolamento de recursos do kernel (PID, network, mount, etc.) — feature do kernel Linux usada por containers.
> - **Kubernetes namespace** = agrupamento lógico de recursos K8s dentro de um cluster (`kubectl get pods -n kube-system`) — abstração de organização.
>
> Se a questão fala em "isolate which processes a container can see", é **Linux** namespace. Se fala em "agrupar recursos dentro do cluster", é **K8s** namespace.

#### Anatomia da image — camadas (layers)

Uma image é uma **pilha de layers read-only**. Cada instrução no Dockerfile gera uma layer. Quando o container sobe, o runtime adiciona **mais uma layer no topo — writable e efêmera** (a "container layer").

```
   ┌──────────────────────────────────────────────┐
   │ Container layer (writable, EFÊMERA)          │ ← morre quando o container morre
   ├──────────────────────────────────────────────┤
   │ Layer 4: CMD ["./app"]                       │ ┐
   ├──────────────────────────────────────────────┤ │
   │ Layer 3: RUN make /app                       │ │ Read-only
   ├──────────────────────────────────────────────┤ │ (compartilhada
   │ Layer 2: COPY . /app                         │ │  entre N containers
   ├──────────────────────────────────────────────┤ │  da mesma image)
   │ Layer 1: FROM ubuntu:20.04                   │ ┘
   └──────────────────────────────────────────────┘
```

**Por que isso é importante (3 consequências):**

1. **Container é stateless por design.** Tudo que o container escrever na container layer (logs, arquivos temporários, uploads) **some quando o container é deletado**. Dados persistentes precisam ir pra **volumes / Persistent Disks / Cloud Storage** — fora da image.
2. **Image é diff-friendly.** Próxima versão da image cria uma nova layer com **só o que mudou** (kilobytes), não a image inteira. Push e pull ficam absurdamente rápidos comparados a VM image.
3. **N containers da mesma image compartilham as layers read-only.** Não há cópia em memória — economia gigante de footprint quando você sobe 100 réplicas.

#### Dockerfile — anatomia comentada

O Dockerfile é o **manifest** que descreve como construir a image. Cada comando = uma layer.

```dockerfile
FROM ubuntu:20.04        # Layer 1 — image base puxada de um registry público
COPY . /app              # Layer 2 — copia o código do build context pro /app dentro da image
RUN make /app            # Layer 3 — compila/builda dentro da image
CMD ["./app"]            # Layer 4 — comando default quando o container sobe
```

**Comandos principais (apareceram no vídeo):**

| Comando | Função |
|---------|--------|
| `FROM` | Define a image base (puxa do registry — Docker Hub, Artifact Registry, etc.) |
| `COPY` | Copia arquivos do **build context** (diretório local do `docker build`) pra dentro da image |
| `RUN` | Executa um comando durante o build (compila, instala pacotes, etc.) |
| `CMD` | Define o comando default quando o container é executado |

**Best practice de ordenação de layers (mencionada explicitamente no vídeo):**

> Layers **estáveis no topo** (FROM, dependências do sistema), layers **voláteis embaixo** (código do app).
>
> **Por quê:** Docker reusa layers por hash. Se sua primeira layer muda toda hora (código), todas as layers seguintes precisam rebuild → cache vai pro lixo. Se as primeiras layers raramente mudam (image base, deps), só as últimas rebuilam → build voa.

#### Multi-stage build — best practice moderna

O Dockerfile do vídeo é declarado como **oversimplified**. Por quê?

**Problema do single-stage build:**
- Você builda e roda no **mesmo container**.
- Container final carrega ferramentas de build (compilador, SDKs, libs de dev) que **não são necessárias em produção**.
- Resultado: image inchada (200 MB → 1 GB) **e** ataque amplificado (cada ferramenta extra = mais superfície de ataque).

**Solução: multi-stage build.**

```
┌─────────────────────────────────────┐
│ Stage 1 — BUILD container           │
│   - JDK, Maven, gcc, etc.           │
│   - Compila o app                   │
│   - Gera o binário                  │
└─────────────────────────────────────┘
                  │
                  │  (copia SÓ o binário final)
                  ▼
┌─────────────────────────────────────┐
│ Stage 2 — RUNTIME container         │
│   - Image base mínima (Alpine, etc.)│
│   - Só o binário copiado            │
│   - Só o que precisa em produção    │
└─────────────────────────────────────┘
```

**Ganho:** image final enxuta + menor superfície de ataque + cold start mais rápido.

#### Onde guardar images — Artifact Registry

| Opção | Quando usar |
|-------|-------------|
| **Artifact Registry** (`pkg.dev`) | **Default no GCP.** Privado por padrão, integrado com IAM, suporta Docker images + outros formatos (Maven, npm, etc.). Substituto do antigo Container Registry. |
| **Docker Hub** | Images **públicas** (image base, projetos open source) |
| **GitLab Container Registry** | Quando o CI/CD já está no GitLab |

**Por que Artifact Registry é a resposta canônica da prova:**

1. **Privado por padrão** — suas images não vazam pra internet.
2. **IAM-integrated** — `roles/artifactregistry.reader` controla quem pode pull.
3. **Cohabita com outros artefatos** — Maven/npm/Python no mesmo serviço.
4. **Próximo dos clusters GKE** — pull com latência baixíssima.

#### Cloud Build — managed build de images

Cloud Build é o **`docker build` gerenciado** do GCP. Em vez de rodar o build local ou num runner self-managed, você submete o source code e o Cloud Build executa as etapas em containers Docker.

**Fluxo típico:**

```
Source code (Cloud Source Repos / GitHub / Bitbucket)
         │
         ▼
Cloud Build (executa N build steps, cada step roda num container Docker)
         │
         ├── Step 1: fetch deps (npm install / mvn dependency:resolve)
         ├── Step 2: compile + test
         ├── Step 3: docker build (gera a image)
         └── Step 4: push pra Artifact Registry
         │
         ▼
Deploy para → GKE / Cloud Run / App Engine / Cloud Run functions
```

**Pontos-chave que o vídeo enfatizou:**

1. **Integrado com IAM** — controle fino de quem pode disparar build.
2. **Cada build step roda num container Docker** — você pode usar qualquer imagem como ferramenta de build (`gcr.io/cloud-builders/docker`, `gcr.io/cloud-builders/mvn`, etc.).
3. **Suporta múltiplos source repos** — Cloud Source Repos, GitHub, Bitbucket (qualquer git-compatible).
4. **Entrega multi-destino** — a mesma image pode ir pra GKE, App Engine ou Cloud Run functions.

**Comando `gcloud` típico:**

```bash
# Submete o source do diretório atual e roda Cloud Build
gcloud builds submit --tag us-central1-docker.pkg.dev/PROJECT/REPO/IMAGE:TAG

# Equivalente declarativo via cloudbuild.yaml
gcloud builds submit --config cloudbuild.yaml
```

### Sub-bloco 9 — Vídeo "What is Kubernetes?" — a ponte container → orquestrador

**Por que importa.** Os dois vídeos anteriores explicaram o **tijolo** (container). Este vídeo introduz o **andaime** (Kubernetes) — e mais importante, **a dor que justifica o andaime**: quando você tem 100 containers em vez de 10 VMs, "rodar `docker run` na mão" deixa de ser viável. K8s é a resposta. Este sub-bloco é a **definição canônica de prova** + os conceitos-âncora que aparecem em quase toda DQ de GKE: declarativo vs imperativo, node, workload types, resource requests/limits.

#### A dor que Kubernetes resolve

O vídeo abre com um cenário concreto:

> "Sua organização adotou containers. Como containers são leves, seus colegas estão criando-os em **números que excedem a contagem de VMs** que vocês tinham antes. As aplicações nos containers precisam **se comunicar pela rede**, mas vocês não têm um **network fabric** que deixa containers se encontrarem."

**Os 3 problemas que aparecem nessa escala:**

1. **Descoberta / networking** — container A precisa achar container B; IPs efêmeros mudam o tempo todo.
2. **Scheduling** — em qual host (de N) cada container deve rodar? E se um host morrer?
3. **Lifecycle** — como fazer rolling update sem downtime? Como reverter (rollback) se quebrar?

Docker sozinho **não resolve nenhum dos três** em escala (Docker opera num host). É aí que entra K8s.

#### Definição canônica (memorize palavra por palavra para a prova)

> **Kubernetes** é uma **open-source platform** para **gerenciar workloads e serviços containerizados**.

Três capacidades-âncora que sempre aparecem como descrição:

| Capacidade | O que significa |
|------------|-----------------|
| **Orquestrar muitos containers em muitos hosts** | Distribui containers entre N nodes automaticamente (scheduling) |
| **Escalar como microservices** | Sobe/desce réplicas conforme demanda; cada microservice escala independente |
| **Rollout e rollback fáceis** | Atualizações progressivas + reversão se algo quebrar |

#### Arquitetura — o esqueleto

Em alto nível, K8s é **um conjunto de APIs** que você usa para deployar containers num **cluster**. O cluster se divide em dois grupos:

```
┌─────────────────────────────────────────────────────────┐
│ Cluster Kubernetes                                      │
│                                                         │
│  ┌──────────────────────────────────────────────┐       │
│  │ CONTROL PLANE (primary components)           │       │
│  │  - API server (a porta de entrada)           │       │
│  │  - Scheduler                                 │       │
│  │  - Controller manager                        │       │
│  │  - etcd (banco de estado)                    │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │ Node 1      │  │ Node 2      │  │ Node 3      │      │
│  │ (containers)│  │ (containers)│  │ (containers)│      │
│  └─────────────┘  └─────────────┘  └─────────────┘      │
└─────────────────────────────────────────────────────────┘
```

> **No GKE**, o **control plane é gerenciado pelo Google** (você não enxerga as VMs). Você só vê os **nodes** (que são VMs do Compute Engine). Em K8s self-hosted, você gerencia os dois.

#### ⚠️ Armadilha de terminologia — "node"

O vídeo destaca isso explicitamente:

> **Em Kubernetes, "node" = computing instance** (uma máquina, podendo ser VM ou bare metal) que roda containers.
> **No Google Cloud genérico**, "node" muitas vezes aparece como sinônimo de **VM do Compute Engine**.

São **conceitos relacionados** (no GKE, um node K8s **é** uma VM do GCE), mas **não são intercambiáveis** em todo contexto. Se a questão fala em "GKE node", é a VM que roda containers. Se fala em "compute node" sem contexto K8s, é uma instância genérica.

#### Declarativo × Imperativo — conceito-pilar de K8s

Esse é **o conceito que diferencia quem entende K8s de quem só usou**. Vai aparecer em **muitas** DQs, direta ou indiretamente.

| Modelo | O que você faz | Exemplo |
|--------|----------------|---------|
| **Declarativo** | Descreve o **estado desejado** (em YAML); K8s descobre como chegar lá | `kubectl apply -f deployment.yaml` (com `replicas: 3` no manifest) |
| **Imperativo** | Emite **comandos** que mudam o estado passo a passo | `kubectl scale deployment my-app --replicas=3` |

**Por que declarativo vence (3 razões):**

1. **Auto-healing** — K8s **monitora continuamente** se o estado real bate com o desejado. Pod morreu? K8s recria. Node caiu? K8s reagenda em outro. Sem você fazer nada.
2. **Estado documentado** — o YAML *é* a documentação. Quem ler o repo sabe o que está em produção (idealmente).
3. **Menos erros operacionais** — não tem comando para esquecer ou rodar fora de ordem.

**Quando imperativo ainda é útil:**

- **Fix rápido temporário** — `kubectl scale --replicas=10` enquanto você abre PR pro YAML.
- **Exploração** — `kubectl run nginx --image=nginx` pra testar antes de escrever o manifest.
- **Ferramenta para construir declarativo** — `kubectl create deployment --dry-run=client -o yaml > deployment.yaml` gera o YAML pra você editar.

**Regra prática de quem é experiente** (vídeo cita literal):

> "Administradores experientes usam configuração imperativa **apenas** para fixes rápidos temporários e como ferramenta para construir configuração declarativa."

#### Workload types suportados

K8s não roda só "app web stateless". Suporta 4 categorias canônicas:

| Tipo | Característica | Exemplo | Recurso K8s típico |
|------|----------------|---------|-------------------|
| **Stateless** | Sem estado persistente, descartável | Nginx, Apache, API REST | `Deployment` |
| **Stateful** | Mantém estado (sessão, dados) entre restarts | Banco de dados, Redis com persistência | `StatefulSet` |
| **Batch jobs** | Roda até completar, depois encerra | Processamento em lote, ETL, geração de relatório | `Job` / `CronJob` |
| **Daemon tasks** | Roda **um pod por node** | Coletor de log/métrica, agente de segurança | `DaemonSet` |

> Esses 4 nomes (`Deployment`, `StatefulSet`, `Job`, `DaemonSet`) são **resources nativos** de K8s — vão voltar nos próximos módulos do curso.

#### Auto-scaling — capacidade nativa

Kubernetes pode **escalar containers automaticamente in/out com base em utilização de recurso** (CPU, RAM, métricas customizadas). Mecanismos relevantes:

| Scaler | O que escala | Critério |
|--------|--------------|----------|
| **Horizontal Pod Autoscaler (HPA)** | Número de **réplicas** de um workload | CPU, RAM, métricas customizadas |
| **Vertical Pod Autoscaler (VPA)** | **Tamanho** dos pods (CPU/RAM requests) | Recomendação baseada em uso histórico |
| **Cluster Autoscaler** | Número de **nodes** no cluster | Pods pending por falta de recurso |

> Esses 3 também são jargão obrigatório — vão aparecer no Module de scaling do curso.

#### Resource requests e limits — controle fino

K8s permite (e recomenda) que cada container declare:

| Campo | O que significa |
|-------|-----------------|
| **Resource requests** | "Eu preciso de **no mínimo** X CPU e Y RAM pra rodar". K8s usa isso pra **schedular** (escolher node que tenha o recurso disponível) |
| **Resource limits** | "Eu **nunca** posso consumir mais que X CPU e Y RAM". K8s mata o container (OOMKilled) se exceder RAM, throttle se exceder CPU |

**Por que isso melhora a performance geral do cluster** (vídeo cita explicitamente):

- Sem requests, K8s não sabe **onde** colocar o pod → vai pra um node lotado → vizinhos sofrem.
- Sem limits, um pod pode **vazar memória** e derrubar tudo no node.
- Com ambos, K8s otimiza **bin-packing** (encaixa pods nos nodes como Tetris) e **isola falhas**.

#### Extensibilidade — Custom Resource Definitions (CRDs)

K8s é extensível por **plugins** e **addons**. O mecanismo mais importante:

> **Custom Resource Definitions (CRDs)** — permitem que desenvolvedores **definam novos tipos de recursos** que podem ser criados, gerenciados e usados em Kubernetes.

**Exemplo real.** Antes, K8s não conhecia "Certificate". O **cert-manager** define um CRD `Certificate`; agora você roda `kubectl apply -f cert.yaml` e o K8s entende. **Operators** (padrão para gerenciar apps complexas em K8s — bancos de dados, Kafka, Elastic) são todos baseados em CRDs.

**Por que isso importa pra prova.** Quando a questão menciona "extending Kubernetes", "custom resources" ou "operators", a resposta envolve **CRDs**.

#### Portabilidade — "no vendor lock-in"

K8s é **open-source** e **portátil**. Roda em:

- On-premises (bare metal ou VMs próprias)
- Outro cloud provider (AKS no Azure, EKS na AWS)
- **GKE no Google Cloud** (foco do curso)

> **Implicação prática.** O **YAML** que você escreve para GKE roda quase intacto em EKS ou AKS. **Ferramentas em torno** (Cloud Build, Artifact Registry, IAM) é que mudam. Isso é argumento forte de venda do K8s: você não fica preso a um provider.

**⚠️ Nuance da prova.** Se a questão pergunta "qual benefício do GKE vs Compute Engine", a resposta canônica inclui "**evita vendor lock-in**" porque K8s é open-source — diferente de soluções proprietárias como App Engine standard.

### Armadilhas de prova consolidadas (Module 2)

| Pegadinha | Resposta correta |
|-----------|------------------|
| "Linux namespace e Kubernetes namespace são a mesma coisa" | ❌ **Não.** Linux namespace = isolamento do kernel (PID, network, mount). K8s namespace = agrupamento lógico de recursos no cluster. |
| "Image e container são sinônimos" | ❌ **Não.** Image é o **artefato estático** (receita); container é a **instância em execução** (bolo assado). |
| "Container preserva dados após restart/delete" | ❌ **Não.** A container layer é **efêmera**. Dados persistentes vão pra volume / Persistent Disk / Cloud Storage. |
| "Docker substitui Kubernetes" | ❌ **Não.** Docker cria/roda em **um host**. K8s orquestra entre N hosts. São complementares. |
| "namespaces controlam consumo de CPU/RAM do container" | ❌ **Não.** Quem controla consumo é **cgroups**. Namespaces controlam **visão** (o que o container enxerga). |
| "Builda e roda no mesmo container é best practice" | ❌ **Não.** Multi-stage build separa build container (com toolchain) de runtime container (mínimo). |
| "Container Registry é o serviço atual do GCP pra images" | ❌ **Não.** **Artifact Registry** é o serviço atual (Container Registry foi descontinuado). |
| "Cada build step do Cloud Build é uma função serverless" | ❌ **Não.** Cada build step **roda num container Docker** que você escolhe (ou usa um pré-construído do Google). |
| "Node em Kubernetes e node no Google Cloud são a mesma coisa" | ⚠️ **Relacionados, não idênticos.** Em K8s, **node = computing instance**. No GCP, "node" sem contexto pode ser uma VM do GCE. No GKE especificamente, **node K8s = VM do GCE** que roda containers. |
| "Configuração imperativa é o jeito recomendado de usar K8s" | ❌ **Declarativo** é o recomendado (auto-healing, estado documentado, menos erros). Imperativo só pra fixes rápidos e exploração. |
| "K8s só serve para apps stateless" | ❌ Suporta 4 tipos: stateless (Deployment), stateful (StatefulSet), batch (Job/CronJob), daemon (DaemonSet). |
| "Resource requests = resource limits" | ❌ **Request** = mínimo (usado pelo scheduler). **Limit** = máximo (usado pelo runtime para matar/limitar). |
| "GKE me prende ao Google Cloud (vendor lock-in)" | ❌ K8s é **open-source e portátil**. Seu YAML roda em GKE, EKS, AKS ou on-prem com mudanças mínimas. |

## Autopilot vs Standard

| Característica | **Autopilot** | **Standard** |
|----------------|---------------|--------------|
| Provisionamento | Fully-managed pelo Google | Você define e gerencia |
| Cobrança | Por **recursos dos pods** | Por **nodes** |
| Localização | **Regional sempre** | **Zonal ou regional** |
| Gerenciamento | Nível de pod | Nível de cluster |
| Image type | Apenas **Container-Optimized OS** | Ubuntu, COS, etc. |
| Custom packages | ❌ | ✅ |
| Flexibilidade de arquitetura | ❌ Limitada | ✅ Total |

**Regra mnemônica:** Autopilot = Auto**pilot** = você não pilota nada = só regional, só COS, sem customização.
