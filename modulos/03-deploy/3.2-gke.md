### Sub-bloco 1 — Cloud computing: definição NIST-style (5 traços)

**Por que importa.** Esses 5 traços compõem a definição padrão que o Google (e a maioria dos provedores) usa para diferenciar "cloud" de "datacenter terceirizado". Cai literal em quiz introdutório.

| # | Traço | Significado prático |
|---|-------|---------------------|
| 1 | **On-demand & self-service** | Provisiona via web UI / API, sem ticket pra humano |
| 2 | **Acesso pela rede** | De qualquer lugar com conexão (não amarrado a hardware no escritório) |
| 3 | **Resource pooling** | Provider compra em escala → economia repassada; cliente não sabe (nem precisa saber) onde fica o hardware exato |
| 4 | **Elasticidade** | Escala **pra cima E pra baixo**, rápido — não só pra cima |
| 5 | **Pay-per-use** | Cobrança por consumo/reserva; parou de usar, parou de pagar |

**Armadilha do quiz (Q2 — verificado):** "elasticidade" sem mencionar reduzir recursos é resposta **incompleta**. A definição inclui obrigatoriamente os dois sentidos (scale-up E scale-down).

### Sub-bloco 2 — As 5 opções de compute do GCP (lente do curso GKE)

**Por que importa.** O curso apresenta GKE no contexto de **quando NÃO usar GKE**. Saber discriminar as 5 opções é o que separa um engenheiro júnior ("usa GKE pra tudo") de um sênior ("escolhe pela carga").

| Serviço | Modelo | Granularidade | Caso canônico |
|---------|--------|---------------|---------------|
| **Compute Engine** | IaaS | VM | Lift-and-shift, controle total do SO, mix de SOs, workloads com requisitos específicos de kernel |
| **GKE** | CaaS (managed K8s) | Cluster + Pods | Orquestração de containers, multi-tenant, microservices com requisitos avançados (HPA, network policies, sidecars) |
| **App Engine** | PaaS | App / serviço | Quer focar em código, sem gerenciar infra; web apps e APIs em linguagens suportadas (Java, Node, Python, PHP, C#/.NET, Ruby, Go) |
| **Cloud Run** | Serverless container | Container stateless | Container que escala **de zero**, request-driven, billing em 100ms |
| **Cloud Run functions** | FaaS | Função | Trigger event-driven (objeto no Cloud Storage, mensagem em Pub/Sub); free tier perpétuo |

**Pontos-chave de cada uma que o curso destacou:**

**Compute Engine**
- VMs até **416 vCPUs / 12+ TB RAM** (cap citado pelo curso à época da gravação — checar `gcloud compute machine-types list` para valor atual)
- Block storage: **Persistent Disks** (até 257 TB, snapshots, mobilidade) ou **Local SSDs** (alto IOPS, ephemeral)
- Autoscaling: **Managed Instance Groups + Global LB**
- Billing: **per-second** após o 1º minuto
- **Preemptible VMs** (e Spot VMs, sucessoras) — desconto agressivo para workloads tolerantes a interrupção

**App Engine**
- Integração nativa com **Cloud Monitoring, Logging, Profiler, Error Reporting**
- **Versionamento** e **traffic splitting** built-in (canary/blue-green sem mais nada)
- Bom para: **websites, backends de mobile/game, APIs RESTful**

**Cloud Run**
- Base **Knative** (API aberta, portável entre GKE / outro Knative / GCP managed)
- Escala **de zero a N quase instantaneamente**
- Billing em **100ms** — nunca paga por idle/overprovisioning

**Cloud Run functions** (antigamente Cloud Functions / Cloud Functions 2nd gen)
- Linguagens: Node.js, Python, Go, Java, .NET Core, Ruby, PHP
- **Free tier perpétuo** — muitos casos de uso ficam de graça
- Casos de uso: microservices, processar arquivos em buckets em tempo real, ETL leve, integração com APIs externas, backends de IoT/mobile

**Armadilha do quiz (Q3 — verificado):** "managed compute platform com suporte **nativo** a containers" = **GKE**. Cloud Run também roda containers, mas a "lente" da pergunta enfatiza orquestração nativa de containers — GKE é a resposta canônica do quiz.

> **Referência detalhada por serviço:** [3.1-compute-engine.md](3.1-compute-engine.md) (Compute Engine), [3.3-cloud-run.md](3.3-cloud-run.md) (Cloud Run/App Engine/Cloud Run functions), o resto deste arquivo (GKE).

### Sub-bloco 3 — Infraestrutura global (rede + regions + zones)

**Por que importa.** A rede do Google é o "produto invisível" que torna LB global, replicação multi-region e cache em edge possíveis. Entender isso explica decisões de pricing (egress, multi-region storage) e arquitetura (Premium vs Standard tier).

**Dados que o curso citou:**
- **~40% do tráfego mundial da internet** trafega na rede Google.
- **100+ content caching nodes** distribuídos globalmente (edge POPs).
- **7 macro-regiões geográficas:** North America, South America, Europe, Africa, Middle East, Asia, Australia.
- À época da gravação: **41 regions / 124 zones**. Valor atualizado em `cloud.google.com/about/locations`.

**Hierarquia geográfica:**

```
Multi-Region (ex: "US", "EU")
  └── Region (ex: us-east1, europe-west2)
        └── Zone (ex: us-east1-c, europe-west2-a)
              └── Resources (VMs, Persistent Disks, etc.)
```

| Nível | Failure domain | Exemplo de uso |
|-------|----------------|----------------|
| **Zone** | Falha de DC/rack | VM única, Local SSD |
| **Region** | Falha regional inteira (raríssimo, mas existe) | Regional MIG, Cloud SQL HA, Regional PD |
| **Multi-Region** | Resiliência continental | Cloud Storage multi-region, Spanner multi-region |

> Para detalhes de HA/DR multi-zona vs multi-region, **ver [01-setup-environment/1.1-resource-hierarchy-iam.md](../01-setup-environment/1.1-resource-hierarchy-iam.md)** seção "Classificação global/regional/zonal".

### Sub-bloco 5 — Billing + Quotas

**O quê:** billing fica no nível **Project**; um Project tem **1 billing account ativa** ou **0** (só free tier); uma billing account pode estar linkada a **N Projects**; **billing subaccounts** servem para resellers separarem por cliente.

**4 ferramentas de controle de gasto:**

| Ferramenta | Onde aplica | Função |
|------------|-------------|--------|
| **Budget** | Billing account OU Project | Define limite (fixo ou % do mês anterior) |
| **Alert** | Vinculado a um budget | Notifica em % do limite (default 50/90/100, customizável) |
| **Reports** | Console | Visualização de gastos por project/service |
| **Quota** | Project | Limita uso de recursos (anti-erro/anti-abuso) |

**Quota — 2 tipos (cai em prova):**

| Tipo | O que limita | Reseta? | Exemplo |
|------|--------------|---------|---------|
| **Rate quota** | Chamadas/operações por janela de tempo | ✅ sim, após a janela | 3.000 calls/100s na API GKE |
| **Allocation quota** | Quantidade simultânea de recursos | ❌ não, é capacidade | Máx 5 VPCs por project |

**Aumento de quota:** via **Google Cloud Support request** (nem todas são aumentáveis — depende da quota).

**Armadilha do quiz (Q1 — verificado):** "ferramentas para controlar gasto **antes** que se torne excessivo" = **Budgets + Alerts no nível do projeto**. ### Sub-bloco 7 — Vídeo "What is a container?" — a evolução do deploy

**Por que importa.** O argumento "container existe porque VM tem limites" só convence se você entende **quais** limites. Esse vídeo monta a escada **bare metal → VM → container** mostrando o problema específico que cada degrau resolve e o que ainda fica em aberto. Cai em quiz introdutório, mas o uso real é em entrevista: explicar **por que** containers, não **o que** são.

**A escada completa:**

```
┌─────────────────────────────────────────────────────────────┐
│ Degrau 1 — DEPLOY EM HARDWARE FÍSICO                        │
│ 1 app por máquina (DB, web server, CDN em hosts separados)  │
│ Problema: desperdício de recurso, lento pra escalar         │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 2 — VIRTUALIZAÇÃO (hypervisor)                       │
│ N VMs compartilham 1 hardware via hypervisor (ex.: KVM)     │
│ Ganho: deploy rápido, portabilidade (image de VM)           │
│ Problemas remanescentes:                                    │
│   - VM carrega OS completo (boot demorado)                  │
│   - Difícil mover entre hypervisors diferentes              │
│   - Apps no mesmo OS disputam deps e recursos               │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 2.5 — VM POR APP (tentativa ingênua de isolar)       │
│ 1 VM por aplicação → cada uma com seu kernel completo       │
│ Funciona, mas N kernels rodando = overhead absurdo          │
│ Não escala pra centenas/milhares de apps                    │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│ Degrau 3 — CONTAINERS                                       │
│ Abstrai SÓ o user space (não a máquina nem o OS inteiro)    │
│ Kernel é compartilhado; app + deps ficam isolados           │
│ Lightweight, start/stop instantâneo (vira um processo OS)   │
└─────────────────────────────────────────────────────────────┘
```

**O que é "user space" (palavra crítica do vídeo):**

> User space = **tudo que roda acima do kernel** — aplicação + bibliotecas + dependências.
> O container virtualiza **só** essa camada. Kernel continua sendo o do host (compartilhado entre todos os containers).

**Por isso o container é "lightweight":**

| Comparativo | VM | Container |
|-------------|-----|-----------|
| O que está empacotado | App + deps + **OS completo** + kernel | App + deps **só** |
| Boot | Inicializa OS (segundos a minutos) | Inicia processo OS (milissegundos) |
| Footprint | GB | MB |
| Portabilidade entre hypervisors/runtimes | Difícil | Trivial (container engine padrão) |
| Isolamento entre apps no mesmo host | Total (kernels separados) | Forte (mesmo kernel, mas namespaces/cgroups isolam) |

**Os 3 problemas específicos que apps-no-mesmo-VM têm (e container resolve):**

1. **Starvation** — app A consome todo CPU/RAM, app B trava.
2. **Dependency hell** — upgrade da lib X pro app A quebra o app B que dependia da versão antiga.
3. **Lockdown dependências não escala** — você até consegue congelar versões, mas hora alguma vai precisar atualizar, e aí volta o problema.

**Tentativas "via processo" que NÃO funcionam (e por isso container venceu):**

- ❌ Rigorous engineering policies (não permitir upgrades) → impede evolução
- ❌ Integration tests pesados → lento, e ainda assim aparecem falhas exóticas em produção

**Os 3 benefícios canônicos que o vídeo enumera (decora isso pra prova):**

| # | Benefício | O que significa na prática |
|---|-----------|----------------------------|
| 1 | **Code-centric delivery** | Foco em entregar código, não em provisionar infra de baixo nível |
| 2 | **HW/SW reliability via Linux kernel** | Mesmo kernel-base no laptop, na CI e em produção → "works on my machine" desaparece |
| 3 | **Habilita microservices** | Loosely-coupled, fine-grained components que escalam e atualizam isoladamente |

**Analogia útil pra feira de trabalho:**

> VM é como **alugar uma casa inteira mobiliada** pra cada inquilino — privacidade total, mas cada casa tem cozinha/banheiro/fundação duplicados. Container é como **república com quartos individuais**: cada um tem sua privacidade (namespaces), sua cota de geladeira (cgroups), mas dividem a estrutura da casa (kernel). Mais barato, mais rápido pra entrar e sair.

### Sub-bloco 8 — Vídeo "Containers and container images" — anatomia técnica

**Por que importa.** Esse vídeo é onde a maior parte das **armadilhas de prova** mora: image ≠ container, Linux namespace ≠ K8s namespace, container layer é efêmero, multi-stage build é best practice. Também onde aparecem os serviços GCP relevantes (Artifact Registry e Cloud Build) — esses são alvo direto da ACE.

#### Image vs Container — distinção que cai literal

| Conceito | Definição | Analogia |
|----------|-----------|----------|
| **Container image** | Aplicação **+ todas as suas dependências** empacotadas | Receita de bolo |
| **Container** | **Instância em execução** de uma image | Bolo assado (rodando) a partir da receita |

**Implicação prática:** uma image gera N containers idênticos. Container morre, a image continua intacta.

#### Por que existem Docker E Kubernetes (e por que o curso usa Cloud Build em vez de Docker)

| Ferramenta | O que faz | Onde para |
|------------|-----------|-----------|
| **Docker** | Cria + roda containers em **um host** | ❌ Não orquestra containers entre hosts |
| **Kubernetes** | **Orquestra** containers (scheduling, scaling, networking, self-healing) entre N hosts | ✅ É a peça que falta |
| **Cloud Build** | Build managed pelo Google que **gera images Docker-formatted** | Substitui o `docker build` local no fluxo GCP |

> Pro curso (e pra prova): **você vai usar Cloud Build para criar a image** e GKE para rodar os containers. Docker continua sendo o formato da image.

#### As 4 tecnologias Linux que viabilizam containers

Container não é magia — é uma combinação de 4 features do kernel Linux. Saber **qual feature isola o quê** é diferencial em entrevista.

| # | Tecnologia | Isola/Controla | Exemplo prático |
|---|------------|----------------|-----------------|
| 1 | **Linux process** | Memória virtual isolada por processo | Cada container roda como um (ou poucos) processos do host |
| 2 | **Linux namespaces** | **O que o app PODE VER** (PIDs, dirs, IPs, hostname, users) | Container só "vê" seus próprios processos, não os do host |
| 3 | **Linux cgroups** | **O que o app PODE USAR** (CPU, RAM, I/O, network bandwidth) | Container limitado a 0.5 CPU e 256 MB RAM |
| 4 | **Union file systems** | Empacotamento em **camadas** read-only + 1 layer writable | Image fica como cebola de layers; container ganha 1 layer extra ao subir |

**Regra mnemônica:** **namespaces = visão**, **cgroups = consumo**. (Eu vejo o quê × eu posso usar quanto.)

**⚠️ Armadilha de prova clássica:**

> **Linux namespace ≠ Kubernetes namespace.** São conceitos diferentes que compartilham o nome.
> - **Linux namespace** = isolamento de recursos do kernel (PID, network, mount, etc.) — feature do kernel Linux usada por containers.
> - **Kubernetes namespace** = agrupamento lógico de recursos K8s dentro de um cluster (`kubectl get pods -n kube-system`) — abstração de organização.
>
> Se a questão fala em "isolate which processes a container can see", é **Linux** namespace. Se fala em "agrupar recursos dentro do cluster", é **K8s** namespace.

#### Anatomia da image — camadas (layers)

Uma image é uma **pilha de layers read-only**. Cada instrução no Dockerfile gera uma layer. Quando o container sobe, o runtime adiciona **mais uma layer no topo — writable e efêmera** (a "container layer").

```
   ┌──────────────────────────────────────────────┐
   │ Container layer (writable, EFÊMERA)          │ ← morre quando o container morre
   ├──────────────────────────────────────────────┤
   │ Layer 4: CMD ["./app"]                       │ ┐
   ├──────────────────────────────────────────────┤ │
   │ Layer 3: RUN make /app                       │ │ Read-only
   ├──────────────────────────────────────────────┤ │ (compartilhada
   │ Layer 2: COPY . /app                         │ │  entre N containers
   ├──────────────────────────────────────────────┤ │  da mesma image)
   │ Layer 1: FROM ubuntu:20.04                   │ ┘
   └──────────────────────────────────────────────┘
```

**Por que isso é importante (3 consequências):**

1. **Container é stateless por design.** Tudo que o container escrever na container layer (logs, arquivos temporários, uploads) **some quando o container é deletado**. Dados persistentes precisam ir pra **volumes / Persistent Disks / Cloud Storage** — fora da image.
2. **Image é diff-friendly.** Próxima versão da image cria uma nova layer com **só o que mudou** (kilobytes), não a image inteira. Push e pull ficam absurdamente rápidos comparados a VM image.
3. **N containers da mesma image compartilham as layers read-only.** Não há cópia em memória — economia gigante de footprint quando você sobe 100 réplicas.

#### Dockerfile — anatomia comentada

O Dockerfile é o **manifest** que descreve como construir a image. Cada comando = uma layer.

```dockerfile
FROM ubuntu:20.04        # Layer 1 — image base puxada de um registry público
COPY . /app              # Layer 2 — copia o código do build context pro /app dentro da image
RUN make /app            # Layer 3 — compila/builda dentro da image
CMD ["./app"]            # Layer 4 — comando default quando o container sobe
```

**Comandos principais (apareceram no vídeo):**

| Comando | Função |
|---------|--------|
| `FROM` | Define a image base (puxa do registry — Docker Hub, Artifact Registry, etc.) |
| `COPY` | Copia arquivos do **build context** (diretório local do `docker build`) pra dentro da image |
| `RUN` | Executa um comando durante o build (compila, instala pacotes, etc.) |
| `CMD` | Define o comando default quando o container é executado |

**Best practice de ordenação de layers (mencionada explicitamente no vídeo):**

> Layers **estáveis no topo** (FROM, dependências do sistema), layers **voláteis embaixo** (código do app).
>
> **Por quê:** Docker reusa layers por hash. Se sua primeira layer muda toda hora (código), todas as layers seguintes precisam rebuild → cache vai pro lixo. Se as primeiras layers raramente mudam (image base, deps), só as últimas rebuilam → build voa.

#### Multi-stage build — best practice moderna

O Dockerfile do vídeo é declarado como **oversimplified**. Por quê?

**Problema do single-stage build:**
- Você builda e roda no **mesmo container**.
- Container final carrega ferramentas de build (compilador, SDKs, libs de dev) que **não são necessárias em produção**.
- Resultado: image inchada (200 MB → 1 GB) **e** ataque amplificado (cada ferramenta extra = mais superfície de ataque).

**Solução: multi-stage build.**

```
┌─────────────────────────────────────┐
│ Stage 1 — BUILD container           │
│   - JDK, Maven, gcc, etc.           │
│   - Compila o app                   │
│   - Gera o binário                  │
└─────────────────────────────────────┘
                  │
                  │  (copia SÓ o binário final)
                  ▼
┌─────────────────────────────────────┐
│ Stage 2 — RUNTIME container         │
│   - Image base mínima (Alpine, etc.)│
│   - Só o binário copiado            │
│   - Só o que precisa em produção    │
└─────────────────────────────────────┘
```

**Ganho:** image final enxuta + menor superfície de ataque + cold start mais rápido.

#### Onde guardar images — Artifact Registry

| Opção | Quando usar |
|-------|-------------|
| **Artifact Registry** (`pkg.dev`) | **Default no GCP.** Privado por padrão, integrado com IAM, suporta Docker images + outros formatos (Maven, npm, etc.). Substituto do antigo Container Registry. |
| **Docker Hub** | Images **públicas** (image base, projetos open source) |
| **GitLab Container Registry** | Quando o CI/CD já está no GitLab |

**Por que Artifact Registry é a resposta canônica da prova:**

1. **Privado por padrão** — suas images não vazam pra internet.
2. **IAM-integrated** — `roles/artifactregistry.reader` controla quem pode pull.
3. **Cohabita com outros artefatos** — Maven/npm/Python no mesmo serviço.
4. **Próximo dos clusters GKE** — pull com latência baixíssima.

#### Cloud Build — managed build de images

Cloud Build é o **`docker build` gerenciado** do GCP. Em vez de rodar o build local ou num runner self-managed, você submete o source code e o Cloud Build executa as etapas em containers Docker.

**Fluxo típico:**

```
Source code (Cloud Source Repos / GitHub / Bitbucket)
         │
         ▼
Cloud Build (executa N build steps, cada step roda num container Docker)
         │
         ├── Step 1: fetch deps (npm install / mvn dependency:resolve)
         ├── Step 2: compile + test
         ├── Step 3: docker build (gera a image)
         └── Step 4: push pra Artifact Registry
         │
         ▼
Deploy para → GKE / Cloud Run / App Engine / Cloud Run functions
```

**Pontos-chave que o vídeo enfatizou:**

1. **Integrado com IAM** — controle fino de quem pode disparar build.
2. **Cada build step roda num container Docker** — você pode usar qualquer imagem como ferramenta de build (`gcr.io/cloud-builders/docker`, `gcr.io/cloud-builders/mvn`, etc.).
3. **Suporta múltiplos source repos** — Cloud Source Repos, GitHub, Bitbucket (qualquer git-compatible).
4. **Entrega multi-destino** — a mesma image pode ir pra GKE, App Engine ou Cloud Run functions.

**Comando `gcloud` típico:**

```bash
# Submete o source do diretório atual e roda Cloud Build
gcloud builds submit --tag us-central1-docker.pkg.dev/PROJECT/REPO/IMAGE:TAG

# Equivalente declarativo via cloudbuild.yaml
gcloud builds submit --config cloudbuild.yaml
```

### Sub-bloco 9 — Vídeo "What is Kubernetes?" — a ponte container → orquestrador

**Por que importa.** Os dois vídeos anteriores explicaram o **tijolo** (container). Este vídeo introduz o **andaime** (Kubernetes) — e mais importante, **a dor que justifica o andaime**: quando você tem 100 containers em vez de 10 VMs, "rodar `docker run` na mão" deixa de ser viável. K8s é a resposta. Este sub-bloco é a **definição canônica de prova** + os conceitos-âncora que aparecem em quase toda DQ de GKE: declarativo vs imperativo, node, workload types, resource requests/limits.

#### A dor que Kubernetes resolve

O vídeo abre com um cenário concreto:

> "Sua organização adotou containers. Como containers são leves, seus colegas estão criando-os em **números que excedem a contagem de VMs** que vocês tinham antes. As aplicações nos containers precisam **se comunicar pela rede**, mas vocês não têm um **network fabric** que deixa containers se encontrarem."

**Os 3 problemas que aparecem nessa escala:**

1. **Descoberta / networking** — container A precisa achar container B; IPs efêmeros mudam o tempo todo.
2. **Scheduling** — em qual host (de N) cada container deve rodar? E se um host morrer?
3. **Lifecycle** — como fazer rolling update sem downtime? Como reverter (rollback) se quebrar?

Docker sozinho **não resolve nenhum dos três** em escala (Docker opera num host). É aí que entra K8s.

#### Definição canônica (memorize palavra por palavra para a prova)

> **Kubernetes** é uma **open-source platform** para **gerenciar workloads e serviços containerizados**.

Três capacidades-âncora que sempre aparecem como descrição:

| Capacidade | O que significa |
|------------|-----------------|
| **Orquestrar muitos containers em muitos hosts** | Distribui containers entre N nodes automaticamente (scheduling) |
| **Escalar como microservices** | Sobe/desce réplicas conforme demanda; cada microservice escala independente |
| **Rollout e rollback fáceis** | Atualizações progressivas + reversão se algo quebrar |

#### Arquitetura — o esqueleto

Em alto nível, K8s é **um conjunto de APIs** que você usa para deployar containers num **cluster**. O cluster se divide em dois grupos:

```
┌─────────────────────────────────────────────────────────┐
│ Cluster Kubernetes                                      │
│                                                         │
│  ┌──────────────────────────────────────────────┐       │
│  │ CONTROL PLANE (primary components)           │       │
│  │  - API server (a porta de entrada)           │       │
│  │  - Scheduler                                 │       │
│  │  - Controller manager                        │       │
│  │  - etcd (banco de estado)                    │       │
│  └──────────────────────────────────────────────┘       │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │ Node 1      │  │ Node 2      │  │ Node 3      │      │
│  │ (containers)│  │ (containers)│  │ (containers)│      │
│  └─────────────┘  └─────────────┘  └─────────────┘      │
└─────────────────────────────────────────────────────────┘
```

> **No GKE**, o **control plane é gerenciado pelo Google** (você não enxerga as VMs). Você só vê os **nodes** (que são VMs do Compute Engine). Em K8s self-hosted, você gerencia os dois.

#### ⚠️ Armadilha de terminologia — "node"

O vídeo destaca isso explicitamente:

> **Em Kubernetes, "node" = computing instance** (uma máquina, podendo ser VM ou bare metal) que roda containers.
> **No Google Cloud genérico**, "node" muitas vezes aparece como sinônimo de **VM do Compute Engine**.

São **conceitos relacionados** (no GKE, um node K8s **é** uma VM do GCE), mas **não são intercambiáveis** em todo contexto. Se a questão fala em "GKE node", é a VM que roda containers. Se fala em "compute node" sem contexto K8s, é uma instância genérica.

#### Declarativo × Imperativo — conceito-pilar de K8s

Esse é **o conceito que diferencia quem entende K8s de quem só usou**. Vai aparecer em **muitas** DQs, direta ou indiretamente.

| Modelo | O que você faz | Exemplo |
|--------|----------------|---------|
| **Declarativo** | Descreve o **estado desejado** (em YAML); K8s descobre como chegar lá | `kubectl apply -f deployment.yaml` (com `replicas: 3` no manifest) |
| **Imperativo** | Emite **comandos** que mudam o estado passo a passo | `kubectl scale deployment my-app --replicas=3` |

**Por que declarativo vence (3 razões):**

1. **Auto-healing** — K8s **monitora continuamente** se o estado real bate com o desejado. Pod morreu? K8s recria. Node caiu? K8s reagenda em outro. Sem você fazer nada.
2. **Estado documentado** — o YAML *é* a documentação. Quem ler o repo sabe o que está em produção (idealmente).
3. **Menos erros operacionais** — não tem comando para esquecer ou rodar fora de ordem.

**Quando imperativo ainda é útil:**

- **Fix rápido temporário** — `kubectl scale --replicas=10` enquanto você abre PR pro YAML.
- **Exploração** — `kubectl run nginx --image=nginx` pra testar antes de escrever o manifest.
- **Ferramenta para construir declarativo** — `kubectl create deployment --dry-run=client -o yaml > deployment.yaml` gera o YAML pra você editar.

**Regra prática de quem é experiente** (vídeo cita literal):

> "Administradores experientes usam configuração imperativa **apenas** para fixes rápidos temporários e como ferramenta para construir configuração declarativa."

#### Workload types suportados

K8s não roda só "app web stateless". Suporta 4 categorias canônicas:

| Tipo | Característica | Exemplo | Recurso K8s típico |
|------|----------------|---------|-------------------|
| **Stateless** | Sem estado persistente, descartável | Nginx, Apache, API REST | `Deployment` |
| **Stateful** | Mantém estado (sessão, dados) entre restarts | Banco de dados, Redis com persistência | `StatefulSet` |
| **Batch jobs** | Roda até completar, depois encerra | Processamento em lote, ETL, geração de relatório | `Job` / `CronJob` |
| **Daemon tasks** | Roda **um pod por node** | Coletor de log/métrica, agente de segurança | `DaemonSet` |

> Esses 4 nomes (`Deployment`, `StatefulSet`, `Job`, `DaemonSet`) são **resources nativos** de K8s — vão voltar nos próximos módulos do curso.

#### Auto-scaling — capacidade nativa

Kubernetes pode **escalar containers automaticamente in/out com base em utilização de recurso** (CPU, RAM, métricas customizadas). Mecanismos relevantes:

| Scaler | O que escala | Critério |
|--------|--------------|----------|
| **Horizontal Pod Autoscaler (HPA)** | Número de **réplicas** de um workload | CPU, RAM, métricas customizadas |
| **Vertical Pod Autoscaler (VPA)** | **Tamanho** dos pods (CPU/RAM requests) | Recomendação baseada em uso histórico |
| **Cluster Autoscaler** | Número de **nodes** no cluster | Pods pending por falta de recurso |

> Esses 3 também são jargão obrigatório — vão aparecer no Module de scaling do curso.

#### Resource requests e limits — controle fino

K8s permite (e recomenda) que cada container declare:

| Campo | O que significa |
|-------|-----------------|
| **Resource requests** | "Eu preciso de **no mínimo** X CPU e Y RAM pra rodar". K8s usa isso pra **schedular** (escolher node que tenha o recurso disponível) |
| **Resource limits** | "Eu **nunca** posso consumir mais que X CPU e Y RAM". K8s mata o container (OOMKilled) se exceder RAM, throttle se exceder CPU |

**Por que isso melhora a performance geral do cluster** (vídeo cita explicitamente):

- Sem requests, K8s não sabe **onde** colocar o pod → vai pra um node lotado → vizinhos sofrem.
- Sem limits, um pod pode **vazar memória** e derrubar tudo no node.
- Com ambos, K8s otimiza **bin-packing** (encaixa pods nos nodes como Tetris) e **isola falhas**.

#### Extensibilidade — Custom Resource Definitions (CRDs)

K8s é extensível por **plugins** e **addons**. O mecanismo mais importante:

> **Custom Resource Definitions (CRDs)** — permitem que desenvolvedores **definam novos tipos de recursos** que podem ser criados, gerenciados e usados em Kubernetes.

**Exemplo real.** Antes, K8s não conhecia "Certificate". O **cert-manager** define um CRD `Certificate`; agora você roda `kubectl apply -f cert.yaml` e o K8s entende. **Operators** (padrão para gerenciar apps complexas em K8s — bancos de dados, Kafka, Elastic) são todos baseados em CRDs.

**Por que isso importa pra prova.** Quando a questão menciona "extending Kubernetes", "custom resources" ou "operators", a resposta envolve **CRDs**.

#### Portabilidade — "no vendor lock-in"

K8s é **open-source** e **portátil**. Roda em:

- On-premises (bare metal ou VMs próprias)
- Outro cloud provider (AKS no Azure, EKS na AWS)
- **GKE no Google Cloud** (foco do curso)

> **Implicação prática.** O **YAML** que você escreve para GKE roda quase intacto em EKS ou AKS. **Ferramentas em torno** (Cloud Build, Artifact Registry, IAM) é que mudam. Isso é argumento forte de venda do K8s: você não fica preso a um provider.

**⚠️ Nuance da prova.** Se a questão pergunta "qual benefício do GKE vs Compute Engine", a resposta canônica inclui "**evita vendor lock-in**" porque K8s é open-source — diferente de soluções proprietárias como App Engine standard.

### Armadilhas de prova consolidadas (Module 2)

| Pegadinha | Resposta correta |
|-----------|------------------|
| "Linux namespace e Kubernetes namespace são a mesma coisa" | ❌ **Não.** Linux namespace = isolamento do kernel (PID, network, mount). K8s namespace = agrupamento lógico de recursos no cluster. |
| "Image e container são sinônimos" | ❌ **Não.** Image é o **artefato estático** (receita); container é a **instância em execução** (bolo assado). |
| "Container preserva dados após restart/delete" | ❌ **Não.** A container layer é **efêmera**. Dados persistentes vão pra volume / Persistent Disk / Cloud Storage. |
| "Docker substitui Kubernetes" | ❌ **Não.** Docker cria/roda em **um host**. K8s orquestra entre N hosts. São complementares. |
| "namespaces controlam consumo de CPU/RAM do container" | ❌ **Não.** Quem controla consumo é **cgroups**. Namespaces controlam **visão** (o que o container enxerga). |
| "Builda e roda no mesmo container é best practice" | ❌ **Não.** Multi-stage build separa build container (com toolchain) de runtime container (mínimo). |
| "Container Registry é o serviço atual do GCP pra images" | ❌ **Não.** **Artifact Registry** é o serviço atual (Container Registry foi descontinuado). |
| "Cada build step do Cloud Build é uma função serverless" | ❌ **Não.** Cada build step **roda num container Docker** que você escolhe (ou usa um pré-construído do Google). |
| "Node em Kubernetes e node no Google Cloud são a mesma coisa" | ⚠️ **Relacionados, não idênticos.** Em K8s, **node = computing instance**. No GCP, "node" sem contexto pode ser uma VM do GCE. No GKE especificamente, **node K8s = VM do GCE** que roda containers. |
| "Configuração imperativa é o jeito recomendado de usar K8s" | ❌ **Declarativo** é o recomendado (auto-healing, estado documentado, menos erros). Imperativo só pra fixes rápidos e exploração. |
| "K8s só serve para apps stateless" | ❌ Suporta 4 tipos: stateless (Deployment), stateful (StatefulSet), batch (Job/CronJob), daemon (DaemonSet). |
| "Resource requests = resource limits" | ❌ **Request** = mínimo (usado pelo scheduler). **Limit** = máximo (usado pelo runtime para matar/limitar). |
| "GKE me prende ao Google Cloud (vendor lock-in)" | ❌ K8s é **open-source e portátil**. Seu YAML roda em GKE, EKS, AKS ou on-prem com mudanças mínimas. |

**Por que importa.** Os sub-blocos 7-9 explicaram **container + K8s genérico**. Este vídeo introduz **o produto GKE especificamente**: o que ele entrega de "managed" em cima do K8s open-source, e por que vale escolher GKE em vez de auto-hospedar K8s numa frota de Compute Engine. É a transição natural para os módulos de arquitetura e operações que vêm em seguida.

**Definição operacional do curso:**

> **GKE = Kubernetes gerenciado rodando na infra do Google.** Você fica com a API; o Google fica com o sysadmin.

#### O que "fully managed" significa em GKE

| Aspecto | O que o Google faz |
|---------|--------------------|
| **Provisionamento da infra** | Não precisa criar VMs/redes/discos manualmente |
| **SO dos nodes** | **Container-Optimized OS (COS)** — mantido pelo Google, otimizado para escala rápida e footprint mínimo |
| **Upgrades do K8s** | `auto-upgrade` mantém o cluster na **última versão estável** |
| **Reparo de nodes** | `node auto-repair` — health checks periódicos; node ruim é **drenado e recriado** (não consertado in-place) |

#### As 4 features automáticas que caem em prova

| Feature | O que faz | Não confundir com |
|---------|-----------|--------------------|
| **Auto-upgrade** | Mantém K8s na última versão estável | Não é o autoscaler |
| **Node auto-repair** | Detecta node unhealthy → **dreno + recreate** | Não conserta in-place |
| **Cluster autoscaler** | Adiciona/remove **nodes** conforme demanda | ≠ HPA (que escala pods) |
| **HPA** (Horizontal Pod Autoscaler) | Escala **pods** conforme métricas | Vem do K8s open-source, GKE só plugga |

> ⚠️ **Pegadinha:** "Node auto-repair recupera o mesmo node danificado" — **falso**. Ele **drena e recria**, não conserta in-place. Idem para `auto-upgrade`: cria novos nodes com a versão nova, drena os antigos.

#### GKE Autopilot (intro do curso — visão geral)

**Definição do curso:** modo de operação onde o GKE **gerencia a configuração do cluster** — nodes, scaling, security, e diversas configs pré-definidas.

> Comparação detalhada Autopilot vs Standard está mais abaixo neste arquivo, na seção [`## Autopilot vs Standard`](#autopilot-vs-standard). Aqui, guarde: **Autopilot = você não pensa em nodes; Standard = você pilota**.

#### As 5 integrações nativas com GCP (memorize)

| Serviço | Função no contexto GKE |
|---------|------------------------|
| **Cloud Build** | CI/CD — build de containers e deploy automatizado (já documentado em [lab-cloud-build-quickstart.md](lab-cloud-build-quickstart.md)) |
| **Artifact Registry** | Armazena **imagens privadas seguras** (sucessor do Container Registry, suporta Docker, npm, Maven, etc.) |
| **IAM** | Controle de acesso (accounts + role permissions) |
| **Cloud Observability** (Monitoring + Logging + Trace + Profiler) | Visibilidade de performance e saúde |
| **VPC** | Rede, **load balancers**, **ingress** |

**Bonus:** **Cloud Console** é o dashboard nativo do GKE — mais poderoso que o K8s dashboard open-source (que é trabalhoso pra securizar — geralmente requer proxy reverso + auth bolted on).

| # | Conceito | Definição |

### Sub-bloco 2 — Anatomia de um Kubernetes Object

**Definição:** entidade **persistente** (vive no `etcd`) que representa o estado de algo no cluster.

Cada objeto tem **2 elementos**:

| Elemento | Definido por | Representa |
|----------|--------------|------------|
| **`spec`** | **Você** (manifest YAML) | **Desired state** — o que você quer |
| **`status`** | **Control plane** | **Current state** — como está agora |

```yaml
apiVersion: v1
kind: Pod              # ← o "tipo" (kind) do objeto
metadata:
  name: nginx
spec:                  # ← VOCÊ define
  containers:
  - name: nginx
    image: nginx:1.25
status:                # ← K8s preenche automaticamente
  phase: Running
  podIP: 10.0.0.42
```

> ⚠️ **Pegadinha:** você **nunca escreve `status`** num manifest. O control plane preenche.

### Sub-bloco 3 — O Watch Loop (alma do K8s)

```
1. Você declara DESIRED STATE no manifest (kubectl apply)
            │
            ▼
2. Control plane persiste no etcd via apiserver
            │
            ▼
3. Controllers comparam DESIRED vs CURRENT
            │
       Match?
        /    \
      SIM    NÃO
       │      │
       │      ▼
       │  4. Toma ação (cria/deleta/edita)
       │      │
       └──────┴──── Loop infinito ──────────────────┐
```

**Exemplo concreto do curso:**
- **Desired state:** 3 pods nginx sempre rodando
- **Current state inicial:** 0 pods
- **Ação:** K8s cria 3 pods
- **Em runtime:** se 1 morrer → cria outro; se editar pra 5 → cria mais 2; pra 1 → mata 2

> **Reconciliação contínua = resiliência do K8s.** Não é mágica, é loop infinito comparando 2 documentos.

**Imperativo vs Declarativo (recap com exemplo concreto):**

| Estilo | Como você fala | Quem decide o "como" | Exemplo |
|--------|----------------|----------------------|---------|
| **Imperativo** | "Crie 3 pods nginx agora" | Você | `kubectl run nginx-1`, `kubectl run nginx-2`, `kubectl run nginx-3` |
| **Declarativo** | "**Sempre existam** 3 pods nginx" | K8s reconcilia | `kubectl apply -f deployment.yaml` (replicas: 3) |

> **Por que declarativo vence (recap):** se um pod morre, imperativo não recria. Declarativo sim, porque desired diz "3" e current baixou pra 2.

### Sub-bloco 4 — Pod (building block fundamental)

| Propriedade | Detalhe |
|-------------|---------|
| **Definição** | Menor objeto deployable do K8s |
| **Regra de ouro** | **Todo container rodando em K8s está dentro de algum Pod** — nunca solto |
| **Containers por pod** | 1 ou mais (geralmente 1; múltiplos = "sidecar pattern") |
| **Quando ter 2+ containers** | Apenas se forem **tightly coupled** (compartilhar rede e/ou storage) |
| **IP** | **Cada pod tem 1 IP único** atribuído pelo K8s |
| **Network namespace** | **Compartilhado** entre containers do mesmo pod |
| **Comunicação intra-pod** | Via **localhost / 127.0.0.1** + portas distintas |
| **Storage volumes** | Declarados no pod, montáveis em múltiplos containers do mesmo pod |

```
┌─────────────────── Pod (IP: 10.0.0.42) ───────────────────┐
│                                                            │
│   ┌────────────────┐         ┌────────────────┐           │
│   │  Container A   │         │  Container B   │           │
```

### Sub-bloco 6 — Os 5 componentes do Control Plane

#### 1. `kube-apiserver` — a porta única

| Aspecto | Detalhe |
|---------|---------|
| Função | Aceita comandos pra **ver/mudar** o estado do cluster |
| Quem interage | `kubectl`, outros componentes, controllers, dashboards |
| Responsabilidades extras | **Autenticação** (quem é você?) + **Autorização** (pode fazer?) + **Admission control** (requisição válida?) |
| **Único componente** com interação direta | ✅ |

#### 2. `etcd` — o banco de dados do cluster

| Aspecto | Detalhe |
|---------|---------|
| Função | Armazena **com confiabilidade** o estado do cluster |
| O que guarda | Config do cluster + nodes ativos + pods desejados + onde devem rodar |
| Você toca direto? | ❌ **Nunca** — só via apiserver |
| Tipo | Key-value store distribuído (consenso Raft) |

> ⚠️ Se etcd corromper, o cluster vira pó. Por isso o apiserver é o único cliente "abençoado". Em prod sempre 3/5 réplicas; em GKE, Google cuida disso.

#### 3. `kube-scheduler` — o despachante

| Aspecto | Detalhe |
|---------|---------|
| Função | Decide **em qual node** cada pod vai rodar |
| **O que NÃO faz** | **Não lança o pod** — só escreve o nome do node no objeto Pod (pegadinha clássica!) |
| Como decide | State dos nodes + constraints + affinity/anti-affinity + recursos disponíveis |

**Critérios de decisão:**

| Critério | Exemplo |
|----------|---------|
| **Recursos** | "Esse pod precisa 4 GB RAM — qual node tem livre?" |
| **Hardware/SW constraint** | "Esse pod só roda em node com GPU / SO X" |
| **Affinity** | "Esse pod **deve** rodar **junto** com o pod X" (mesma node) |
| **Anti-affinity** | "Esse pod **NÃO pode** rodar com o pod X" (espalhamento HA) |
| **Taints/tolerations** | "Esse node é restrito; só pods com 'permissão' podem ir" |

#### 4. `kube-controller-manager` — motor do watch loop

| Aspecto | Detalhe |
|---------|---------|
| Função | Loop infinito comparando current vs desired state |
| Composição | Vários **controllers** (loops de código), cada um com responsabilidade específica |

**Tipos de controllers:**

| Categoria | Exemplos | Função |
|-----------|----------|--------|
| **Workload** | `Deployment`, `ReplicaSet`, `StatefulSet`, `DaemonSet` | Mantém X pods rodando, faz rolling update |
| **System** | `Node Controller` | Detecta node offline e reage |
| **Outros** | `Job`, `CronJob`, `ServiceAccount` | Cada um com sua responsabilidade |

> 💡 O exemplo "3 nginx pods sempre rodando" é gerenciado pelo **Deployment controller** dentro do controller-manager.

#### 5. `kube-cloud-manager` — ponte com a cloud

| Aspecto | Detalhe |
|---------|---------|
| Função | Controllers que **falam com o provedor de cloud** (GCP, AWS, Azure) |
| Exemplo no GCP | Provisionar **Load Balancers** quando você cria um Service `type: LoadBalancer`; criar **Persistent Disks** para PVCs |
| Em on-prem | Pode não existir ou ser substituído por implementação específica |

### Sub-bloco 7 — Os 3 componentes do Node

#### 1. `kubelet` — agente no node

| Aspecto | Detalhe |
|---------|---------|
| Função | "Embaixador" do K8s em cada node — recebe ordens do apiserver |
| Workflow | Apiserver: "rode esse pod aqui" → kubelet usa o **container runtime** pra criar containers |
| Monitoramento | Executa **liveness/readiness/startup probes**, reporta status ao apiserver |

**Probes (cai em prova):**

| Probe | Pergunta que responde | Ação se falhar |
|-------|----------------------|----------------|
| **Liveness** | "Está vivo?" | Reinicia o container |
| **Readiness** | "Pronto pra receber tráfego?" | Remove do Service (não recebe requests) |
| **Startup** | "Já terminou de iniciar?" | Adia liveness/readiness até startup OK |

#### 2. Container Runtime — quem roda o container

| Aspecto | Detalhe |
|---------|---------|
| Função | Software que **lança containers a partir de imagens** |
| Opções no K8s | containerd, CRI-O, Docker (deprecated em 1.24+) |
| GKE usa | **`containerd`** (o miolo do Docker, sem a CLI) |

> ⚠️ "K8s removeu suporte ao Docker em 1.24" = não usa mais o **daemon** do Docker como runtime. Imagens Docker continuam funcionando porque `containerd` (que era parte do Docker) roda elas perfeitamente.

#### 3. `kube-proxy` — a rede do node

| Aspecto | Detalhe |
|---------|---------|
| Função | Mantém conectividade de rede entre pods do cluster |
| Mecanismo (K8s open-source) | **iptables** do kernel Linux (firewalling/NAT) |
| Implementa | Regras do **Service** (load balancing entre pods) |

### Sub-bloco 8 — GKE como abstração do control plane

**Do ponto de vista do usuário, GKE é MUITO mais simples que K8s self-managed:**

| Componente | Self-managed K8s | GKE |
|------------|------------------|-----|
| **Control plane** | Você provisiona VMs, instala apiserver/etcd/scheduler/controller-mgr, gerencia upgrades, certs, HA... | **Google gerencia 100%**. Você só vê **1 IP do apiserver** |
| **etcd** | Você cuida de backup, snapshot, replicação, restauração | Google cuida |
| **Upgrades do control plane** | Manual ou via tooling próprio | Auto-upgrade (release channels Rapid/Regular/Stable) |
| **HA do control plane** | Você desenha (3+ control planes + etcd cluster) | Built-in (regional cluster = replicação em zonas) |
| **Nodes (Autopilot)** | N/A | **Google gerencia totalmente** — você só pensa em pods |
| **Nodes (Standard)** | Você gerencia | **Você gerencia** (config, node pools, scaling, upgrades) |

**Quando Standard ainda faz sentido (vs Autopilot):**

| Caso | Por quê |
|------|---------|
| DaemonSet privilegiado (agente de monitoring custom) | Autopilot restringe pods privilegiados |
| Machine type específico / GPU / TPU customizada | Autopilot abstrai parcialmente |
| Controle fino sobre node pools (tainting custom) | Standard expõe isso |
| Apps que precisam acesso ao SO do node (hostPath, hostNetwork) | Autopilot bloqueia |

### Sub-bloco 11 — Vídeo "Autopilot vs Standard" (deep dive nos 2 modos de operação)

> **Por que este sub-bloco está aqui.** Sub-bloco 8 já comparou GKE vs self-managed K8s e mencionou Autopilot/Standard em alto nível. Este vídeo do Module 3 dedica-se **exclusivamente aos 2 modos**, explicando os 4 pilares de benefício do Autopilot, suas restrições técnicas (com o **porquê**), e quando Standard se justifica. Este é o material canônico do curso sobre o tema — a seção `## Autopilot vs Standard` mais abaixo é apenas o resumo de bolso.

#### 11.1 — Comparação em uma frase

| Aspecto | **Autopilot** | **Standard** |
|---------|---------------|--------------|
| Filosofia | **Hands-off** — Google decide tudo | **Hands-on** — você configura tudo |
| Overhead de management | Mínimo | Alto |
| Opções de configuração | **Menos** (restrito a perfis seguros) | **Muitas** (fine-grained) |
| Modelo de cobrança | **Pay-per-pod** — só paga o que os pods consomem | **Pay-per-node** — paga toda a infra provisionada (mesmo ociosa) |

> **Analogia.** Autopilot é como **alugar um Uber Black** — você paga pela corrida real, não tem volante, motorista escolhe a rota seguindo regras profissionais. Standard é como **comprar um carro** — você dirige, você abastece, você troca pneu, paga seguro mesmo se ficar parado na garagem.

#### 11.2 — Os 4 pilares de benefício do Autopilot

O vídeo organiza Autopilot em **4 razões para escolhê-lo**:

##### Pilar 1 — Production-optimized (otimizado pra produção)

| Característica | Detalhe |
|----------------|---------|
| **Battle-tested best practices** | Google aplica as configs hardened de cluster que ele mesmo opera internamente (security, networking, defaults sãos) |
| **Machine type auto-definido** | Você não escolhe `e2-medium` vs `n2-standard-4`. Autopilot **olha os workloads** (pods declarados, suas requests de CPU/memory) e provisiona a máquina **certa pra eles** |
| **Adapta a mudanças** | Se workloads mudam ao longo do tempo, o substrato adapta — você não fica preso a uma decisão de capacity planning de 6 meses atrás |
| **Otimização de custo e uso** | Bin packing automático — menos waste de CPU/memory ociosa |
| **Time-to-production reduzido** | Sem decisões de "que node pool, que machine type, quantos nodes mínimos" — você só joga pods e vai |

**Por que importa.** Em Standard, decisões erradas de node pool são caras: nodes superdimensionados queimam dinheiro 24/7, subdimensionados causam `Pending` pods e contenção. Autopilot elimina essa categoria de erro.

##### Pilar 2 — Strong security posture (postura de segurança forte)

| Característica | Detalhe |
|----------------|---------|
| **Cluster nodes + infra securizados pelo Google** | Você não configura SELinux, AppArmor, audit logging, certs de nodes — Google já fez |
| **Elimina infra security management** | Patching de OS, hardening de kernel, atualizações de runtime — tudo no controle do Google |
| **Locked-down nodes** | Acesso direto ao node é **bloqueado** (sem SSH, sem login no host) — reduz **superfície de ataque** |
| **Reduz config mistakes contínuos** | Você não tem como, por engano, deixar `hostNetwork: true` ou `privileged: true` num pod (Autopilot recusa o admit) — **prevenção em vez de detecção** |

**Por que importa.** A maioria dos compromissos de cluster K8s na vida real vem de: pod privilegiado escapando pro host, secret vazada via env var, node com kernel desatualizado, IAM de service account mal configurado. Autopilot **remove a maioria desses vetores antes deles existirem** (não dá pra cometer o erro porque a API rejeita).

##### Pilar 3 — Operational efficiency (eficiência operacional)

| Característica | Detalhe |
|----------------|---------|
| **Monitoring abrangente** | Google monitora **control plane + worker nodes + core K8s system components** (kubelet, runtime, kube-proxy, etc.) — não só "o cluster está acessível", mas saúde fina de cada peça |
| **Pod-scheduling SLA** | Google **garante contratualmente** que pods declarados são agendados (desde que respeitem as constraints de Autopilot) — não vira "Pending" por capacidade |
| **Updates contínuos** | Google mantém o cluster sempre numa versão suportada — sem patch-fatigue do operador |
| **Update windows configuráveis** | Você define **janelas de manutenção** (ex.: 02:00-06:00 UTC sábado e domingo) — disrupção em horários planejados |

**Por que importa.** Pod-scheduling SLA é o **diferencial técnico**. Em K8s normal, se o cluster autoscaler demora ou não tem quota, pods ficam `Pending`. Em Autopilot, o ato de criar o pod **já garante** que vai rodar — Google opera o substrato para isso ser verdade.

##### Pilar 4 — Pricing (pague só por pods)

| Modelo Autopilot | Modelo Standard |
|------------------|------------------|
| Cobra **CPU/memória/ephemeral storage solicitados pelos pods** (vCPU-hora, GiB-hora) + cluster management fee (a partir do 1º cluster Autopilot do projeto na hora) | Cobra **VMs do node pool** + Persistent Disks + cluster management fee |
| Se pod pede 250 mCPU + 512 MiB, você paga **250 mCPU + 512 MiB** mesmo se o node físico tiver mais sobra | Se node é `n2-standard-4` (4 vCPU, 16 GiB) e roda 1 pod pequeno, **você paga os 4 vCPUs e 16 GiB de qualquer jeito** |
| **Bin packing é responsabilidade do Google** — ele escolhe quantos pods empilhar por node | Bin packing é responsabilidade sua (request/limit tuning, node pool sizing) |

> **Armadilha de prova.** "GKE Autopilot é mais barato" — **depende**. Para workloads **bem dimensionados em Standard** (alto utilization, requests calibrados), Standard pode sair mais barato. Para workloads **com utilization baixo/inconsistente** ou times sem expertise em capacity planning, Autopilot é vitória clara. A frase do vídeo é: "you only pay for what you use" — comparar com "pay for provisioned" do Standard.

#### 11.3 — Restrições do Autopilot (o preço da abstração)

O vídeo é explícito: **Autopilot é mais restritivo que Standard**. O motivo técnico é que ele **assina um pod-scheduling SLA** — pra cumprir essa garantia, ele tem que **bloquear configs** que poderiam impedir o agendamento ou comprometer a malha gerenciada.

##### Restrição 1 — Acesso a node objects bloqueado

| O que é bloqueado | Por quê |
|-------------------|---------|
| **SSH no node** | Node é "caixa preta" gerenciada pelo Google. SSH abre vetor de modificação manual que quebraria as garantias |
| **Login na máquina** | Idem |
| **Modificar configs do node** | Google define machine type, OS image (sempre **Container-Optimized OS — COS**), kernel params |

##### Restrição 2 — Privilege escalation removida

| O que é bloqueado | Por quê |
|-------------------|---------|
| `securityContext.privileged: true` | Pod privilegiado pode acessar todo o host — quebra isolamento. Inaceitável num substrato compartilhado e gerenciado |
| `allowPrivilegeEscalation: true` | Mesmo motivo, num escopo menor |
| **`hostNetwork: true`** | Pod usar a network namespace do node fura a malha de rede gerenciada |
| **`hostPID: true` / `hostIPC: true`** | Acesso aos namespaces do host quebra isolamento |
| **`hostPath` volumes** | Pod tocar disco do node vaza dado entre pods, prende pod a node específico |

> **Recap.** Esses bloqueios são o **inverso** da lista "quando Standard ainda faz sentido" do Sub-bloco 8: qualquer workload que precisa de **algum** desses é exatamente o caso onde Autopilot **não cabe** e Standard é obrigatório.

##### Restrição 3 — Limitações em node affinity e host access

| Conceito | Em Autopilot |
|----------|--------------|
| **Node affinity** (regra: "agendar este pod só em nodes com label X") | Limitado — Autopilot suporta algumas affinities (workload separation, GPU/TPU nodes Autopilot-managed), mas não affinity arbitrária baseada em labels custom de node pools que você não pode criar |
| **Anti-affinity** | Suportada (espalhar replicas entre zonas, por exemplo) |
| **Taints + tolerations custom** | Restrito — você não cria node pools com taints arbitrárias |
| **Host access (hostPath, hostNetwork, hostPID)** | **Bloqueado** (visto acima) |

##### Restrição 4 — Todo pod = Guaranteed QoS class

O vídeo diz: **"all pods in GKE Autopilot are scheduled with a Guaranteed class Quality of Service"**. Isso requer **uma pequena mudança de config**. Pra entender o que isso significa, precisa do **conceito-base de QoS classes em K8s** (que você não viu ainda explicitamente):

**As 3 QoS classes do Kubernetes:**

| Classe | Critério | Comportamento sob pressão de recurso |
|--------|----------|---------------------------------------|
| **Guaranteed** | **Requests == Limits** em **todos** os containers do pod, **para CPU e memória** | **Último a ser despejado** (evicted) sob memory pressure. Maior prioridade. SLA mais forte |
| **Burstable** | Pelo menos 1 container tem **request < limit** (ou só uma das duas definida) | Pode "estourar" o request até o limit, mas é candidato a eviction antes do Guaranteed |
| **BestEffort** | **Nenhum** request ou limit definido | **Primeiro a ser despejado**. Sem garantia nenhuma |

**O que muda em Autopilot:** todo pod **automaticamente vira Guaranteed** — a "minor configuration change" do vídeo é que você precisa **declarar requests E limits iguais** (ou o Autopilot ajusta os limits pra igualar os requests no admission). Resultado:

- Pods de Autopilot têm **prioridade máxima** sob pressão
- Mas você **paga o limit cheio** (não tem "burstable" pra apertar custo)
- Não dá pra rodar pod BestEffort experimental em Autopilot — mesmo um job de teste tem que ter requests/limits

> **Por que isso liga ao SLA.** Pra Google garantir scheduling, ele precisa **conhecer com precisão o footprint de cada pod**. BestEffort/Burstable têm consumo imprevisível — Guaranteed é o único modelo onde requests == realidade, que é o que o bin packing automático precisa.

#### 11.4 — Standard mode — quando vale a pena

> Citação direta do vídeo: **"The GKE Standard mode has the same functionality as Autopilot, but you're responsible for the configuration, management, and optimization of the cluster. Unless you require the specific level of configuration control offered by GKE standard, it's recommended that you use Autopilot mode."**

Standard é a escolha certa quando você precisa de **alguma** das coisas que Autopilot bloqueia. Recap consolidado:

| Razão pra escolher Standard | Exemplo concreto |
|------------------------------|-------------------|
| **Workload privilegiado** | Agente de monitoring custom que precisa `securityContext.privileged: true` ou ler `/proc` do host |
| **DaemonSet que toca o node** | Coletor de logs que monta `/var/log` do host via `hostPath` |
| **Machine type específico** | Workload de ML que exige GPU **A100** numa configuração Autopilot ainda não suporta, ou TPU custom |
| **Affinity custom complexa** | Pin de pods a node pool específico com labels que você define |
| **Pods BestEffort/Burstable** | Jobs experimentais sem requests/limits, ou apps legados que estouram limits sob carga e dependem de burst |
| **Acesso ao node** | Necessidade real de SSH no node pra debug profundo (raro em prod, mais comum em research/legacy) |
| **Otimização extrema de custo** | Você tem time experiente em K8s e tunou requests/limits ao milímetro — Standard custa menos nesse caso |
| **OS diferente de COS** | Ubuntu nodes (ex.: dependências de pacote `.deb` não disponíveis em COS) |

#### 11.5 — Decision tree resumida (mental model)

```
Você precisa de SSH no node, privileged pods, hostPath/hostNetwork, ou GPUs/TPUs raras?
├── SIM → Standard (obrigatório)
└── NÃO
    │
    Você tem time com expertise em K8s e quer extrair último centavo de bin packing manual?
    ├── SIM → Standard (mais barato se bem operado)
    └── NÃO
        │
        Workload típico (stateless services, jobs, APIs)?
        ├── SIM → Autopilot (Google recomenda por default)
        └── ?    → Autopilot (na dúvida, Autopilot)
```

> **A regra do curso:** *"Unless you require the specific level of configuration control offered by GKE standard, it's recommended that you use Autopilot mode."* — em prova ACE, se a questão não dá nenhum motivo específico pra Standard, a resposta certa **provavelmente é Autopilot**. A exceção clássica é a DQ 03 desta seção, que pede explicitamente "controle total de arquitetura" e "Ubuntu" — aí a resposta é Standard.

#### 11.7 — Mnemônico do Sub-bloco 11

- **"Autopilot = Auto-tudo + Guaranteed + sem SSH."** Auto-machine, auto-monitor, auto-update; QoS sempre Guaranteed; sem porta dos fundos pro node.

#### 12.1 — Identidade de um objeto K8s (Name + UID + Label)

Todo objeto K8s tem **3 formas de identificação**, cada uma com função distinta:

| # | Identificador | Quem define | Escopo de unicidade | Para que serve |
|---|---------------|-------------|---------------------|----------------|
| 1 | **Name** | **Você** (no manifest) | Único **por namespace** + **por kind** num momento | Endereço humano. Como você se refere ao objeto em comandos `kubectl` |
| 2 | **UID** | **Kubernetes** (auto-gerado) | Único **na vida inteira do cluster** | Endereço para a máquina. Distingue "este Pod" de outro com o mesmo nome criado depois |
| 3 | **Labels** | **Você** (no manifest, ou depois) | Não único (vários objetos podem ter o mesmo label) | Tags para **agrupar e selecionar** objetos |

##### Regras do Name

| Regra | Detalhe |
|-------|---------|
| Tamanho | **Até 253 caracteres** (string única) |
| Caracteres permitidos | **Números, letras, hífens, pontos** |
| Caracteres proibidos | Underscores (`_`), espaços, `@`, `/`, etc. |
| Caso | Lowercase (DNS-1123 subdomain rules na maioria dos recursos) |
| Unicidade | **Um nome por kind por namespace por vez** — não pode ter 2 Pods `nginx` no namespace `default` simultaneamente |
| Reuso após delete | **Pode** — depois de deletar o Pod `nginx`, você pode criar outro com o mesmo nome (mas terá **UID diferente**) |

> **Armadilha de prova.** "Posso ter um Pod `nginx` e um Service `nginx` no mesmo namespace?" **Sim** — unicidade é por **kind + namespace**, não global. Mas dois Pods `nginx` no mesmo namespace simultaneamente, **não**.

##### O papel do UID

O UID é gerado **uma única vez** quando o objeto é criado pela primeira vez. Se você deleta e recria com o mesmo nome, o **UID é diferente** — isso permite ao K8s distinguir "a versão antiga já reconciliada" da "nova versão que aparece agora", evitando confusão em controllers que ainda têm referência ao objeto antigo.

```yaml
metadata:
  name: nginx
  uid: 5b6e8a4c-12ab-4cde-9876-abcdef012345   # gerado pelo K8s, você nunca define
  namespace: default
```

##### Labels — o sistema de tags

Labels são **key-value pairs** que você pluga em objetos pra agrupá-los logicamente. Exemplos canônicos:

```yaml
metadata:
  name: nginx-frontend-7d4
  labels:
    app: nginx
    tier: frontend
    env: prod
    version: v1.2.3
```

**Por que importa:** Services, Deployments, NetworkPolicies, etc. **não conhecem pods por nome** — eles **selecionam por label**. É a cola que junta tudo em K8s.

##### Label selectors via kubectl

```bash
# Equality-based — todos os pods com label app=nginx
kubectl get pods -l app=nginx

# Inequality
kubectl get pods -l app!=nginx

# Set-based — label app está dentro do conjunto {nginx, apache}
kubectl get pods -l 'app in (nginx,apache)'

# Notin
kubectl get pods -l 'app notin (legacy,deprecated)'

# Existência de label (sem checar valor)
kubectl get pods -l env

# Combinação — AND implícito
kubectl get pods -l app=nginx,tier=frontend,env=prod
```

> **Pegadinha.** Selectors usam **AND implícito** entre vírgulas — não OR. Para OR de valores, use set-based (`in (a,b,c)`).

#### 12.2 — Manifest YAML — anatomia completa

Você declara objetos em arquivos texto YAML (ou JSON — o curso usa YAML). O K8s lê o manifest e **persiste o objeto no etcd** com o `spec` que você declarou. A partir daí, o watch loop reconcilia.

##### Os 4 campos obrigatórios (de topo)

| Campo | O que é | Exemplo |
|-------|---------|---------|
| **`apiVersion`** | Versão da API do K8s usada para **criar** o objeto. Define qual schema vale | `v1` (core), `apps/v1` (Deployments), `batch/v1` (Jobs), `networking.k8s.io/v1` (Ingress) |
| **`kind`** | Tipo do objeto | `Pod`, `Deployment`, `Service`, `ConfigMap`, etc. |
| **`metadata`** | Identificação — nome, UID (auto), namespace (opcional), labels, annotations | `metadata: { name: nginx, namespace: default }` |
| **`spec`** | **Estado desejado** — o que você quer (containers, replicas, volumes, etc.). Cada `kind` tem um schema próprio de `spec` | `spec: { containers: [...] }` |

> **Recap do Sub-bloco 2.** `spec` é o que **você escreve**. `status` é o que o **K8s te dá** (você não escreve status). Esses 2 campos + os 2 obrigatórios (`apiVersion`, `kind`, `metadata`) formam a estrutura canônica de **todo** objeto K8s.

##### Exemplo: Pod nginx mínimo

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.27
      ports:
        - containerPort: 80
```

**O que cada linha faz:**

- `apiVersion: v1` — schema do core K8s API group (onde mora Pod, Service, ConfigMap, etc.)
- `kind: Pod` — quero um Pod
- `metadata.name: nginx-pod` — nome humano
- `metadata.labels.app: nginx` — tag para Services/Deployments selecionarem depois
- `spec.containers[]` — lista de containers que rodam neste pod (lembre: pod pode ter N containers, sub-bloco 4)
- `spec.containers[0].image: nginx:1.27` — imagem (com tag — boa prática evitar `latest`)
- `spec.containers[0].ports[0].containerPort: 80` — porta que o container expõe

##### Best practices de manifest

| Prática | Por quê |
|---------|---------|
| **Objetos relacionados no mesmo YAML** (separados por `---`) | Facilita gerenciar como unidade lógica (ex.: Deployment + Service + ConfigMap de uma mesma app) |
| **YAML em version control** (Git) | Estado declarado vira **histórico auditável** — rollback é um `git revert + kubectl apply` |
| **Cloud Source Repositories** (ou GitHub/GitLab/Bitbucket) | O curso menciona Cloud Source Repos especificamente; qualquer Git serve. CSR integra nativo com Cloud Build pra CI/CD |
| **Tag específica em images** (não `:latest`) | `:latest` muda silenciosamente — quebra rollback determinístico |

> **Por que `kubectl apply -f file.yaml` é poderoso.** Como o manifest descreve o **estado desejado**, aplicar o mesmo manifest 2 vezes é **idempotente** — se o objeto já está naquele estado, nada acontece. Diferente do `kubectl create`, que falha se já existe.

#### 12.3 — Por que Pod solto é uma armadilha (recap aprofundado)

O vídeo recaptura o exemplo dos **3 nginx web servers**. A pergunta: "como mantenho **sempre** 3 instâncias rodando?"

**Solução ingênua — 3 manifests de Pod soltos:**

```yaml
apiVersion: v1
kind: Pod
metadata: { name: nginx-1, labels: { app: nginx } }
spec: { containers: [{ name: nginx, image: nginx }] }
---
apiVersion: v1
kind: Pod
metadata: { name: nginx-2, labels: { app: nginx } }
spec: { containers: [{ name: nginx, image: nginx }] }
---
# ... idem para nginx-3
```

**Por que isso quebra:**

| Problema | Detalhe |
|----------|---------|
| Pods são **ephemeral** | Sub-bloco 4: pod é "casa temporária". Se o node morre, o pod desaparece — não é recriado |
| Pods **não se auto-curam** | Sub-bloco 9, pegadinha: "Pod solto é recriado se morrer" → **falso**. Só se gerenciado por controller |
| Pods **não são feitos pra rodar pra sempre** | Design intencional do K8s — ciclo de vida curto, identidade descartável |
| **3 manifests separados** = **3 vezes mais YAML pra editar**, version, sincronizar (atualizar imagem? alterar nos 3 lugares) |

##### A solução correta: Controller (Deployment)

Em vez de 3 Pods soltos, você declara **1 Deployment** que diz "quero **3 réplicas** do template Pod abaixo":

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
  labels:
    app: nginx
spec:
  replicas: 3                          # ← número desejado
  selector:
    matchLabels:
      app: nginx                       # ← quais pods o Deployment "possui"
  template:                            # ← TEMPLATE do pod (não o pod em si)
    metadata:
      labels:
        app: nginx                     # ← labels casam com selector acima
    spec:
      containers:
        - name: nginx
          image: nginx:1.27
          ports:
            - containerPort: 80
```

**Resultado:** 1 arquivo YAML cria **3 pods** + garante que **sempre** vai ter 3. Se um morre, outro nasce.

#### 12.4 — A árvore Deployment → ReplicaSet → Pod

O vídeo é explícito sobre **como** o Deployment faz o trabalho. Não é o Deployment **direto** que cria os pods — ele cria um **ReplicaSet filho**, e o ReplicaSet cria os pods.

##### Diagrama da hierarquia parent-child

```
Deployment (você declara)
    │  cria e gerencia
    ▼
ReplicaSet (criado pelo Deployment Controller)
    │  cria e gerencia
    ▼
Pod, Pod, Pod, ... (réplicas)
    │  cada um agendado num node
    ▼
Container(s) dentro de cada Pod (rodando no container runtime do node)
```

**Por que essa indireção (Deployment → ReplicaSet → Pod, em vez de Deployment → Pod direto)?**

| Razão | Detalhe |
|-------|---------|
| **Rolling updates** | Quando você atualiza a imagem do Deployment, ele cria um **novo ReplicaSet** (com a nova imagem) e gradualmente escala o antigo pra 0 enquanto sobe o novo. Os 2 ReplicaSets coexistem durante a transição |
| **Rollback** | O Deployment guarda histórico de ReplicaSets antigos. `kubectl rollout undo deployment/X` re-escala o ReplicaSet anterior pra `replicas: N` e o novo pra `0` — instantâneo |
| **Separação de responsabilidades** | ReplicaSet é "burro" — só conta pods e cria/mata. Deployment é "esperto" — orquestra ReplicaSets entre si |

##### O fluxo real de criação (passo a passo)

1. Você roda `kubectl apply -f deploy.yaml`.
2. **kubectl** valida o YAML e envia request HTTP para o **kube-apiserver**.
3. **apiserver** autentica → autoriza → admite → persiste o objeto `Deployment` no **etcd**.
4. O **Deployment Controller** (parte do `kube-controller-manager`) faz watch no apiserver e vê o novo Deployment.
5. Controller cria um **ReplicaSet** filho (também via apiserver → etcd).
6. O **ReplicaSet Controller** vê o novo ReplicaSet, conta os pods existentes (0), e cria os 3 pods que faltam.
7. Cada Pod é persistido no etcd com `spec.nodeName` **vazio**.
8. O **kube-scheduler** vê pods com `nodeName` vazio, escolhe um node, escreve `nodeName: <node>` via apiserver.
9. O **kubelet** daquele node vê o pod atribuído, pede ao **container runtime** (containerd) pra puxar a imagem e rodar.
10. Pods entram em `Running`. ReplicaSet vê `currentReplicas == desiredReplicas == 3`. ✅

**Se um pod morre (passo 11):**

11. Kubelet do node afetado reporta o pod como `Failed` (via apiserver → etcd).
12. **ReplicaSet Controller** vê `currentReplicas: 2`, `desiredReplicas: 3` → cria um pod novo.
13. Volta pro passo 8 (scheduler escolhe node, kubelet roda).

> **Recap do Sub-bloco 3 (Watch Loop) + Sub-bloco 6 (control plane):** todo esse fluxo é o **watch loop em ação**, com 3 controllers diferentes (Deployment, ReplicaSet, Scheduler) e o kubelet do node fazendo seu papel. **Tudo passa pelo apiserver** (Sub-bloco 5 e 6).

#### 12.5 — Os 4 tipos de controller (recap do Sub-bloco 9 do Module 2)

O vídeo cita 4 controllers que gerenciam pods. Já vimos em Sub-bloco 6, mas vale recapturar com o ângulo "object management":

| Controller | Quando usar | Exemplo |
|------------|-------------|---------|
| **Deployment** | Workloads **stateless long-lived** — web servers, APIs, microservices stateless | Frontend nginx, backend Node.js |
| **StatefulSet** | Workloads **stateful** — identidade estável, ordem, storage persistente | Cassandra cluster, Postgres com réplicas |
| **DaemonSet** | **Um pod por node** — agentes de infra | fluentd (logs), node-exporter (metrics), CNI plugins |
| **Job / CronJob** | Workloads **finitos** — rodam e terminam | Batch ETL noturno, migration de DB |

> **Quando o vídeo diz "Deployments are a great choice for long-lived software components like web servers"** — é exatamente essa distinção: web servers = stateless = Deployment. Banco com identidade fixa = StatefulSet. Agente em cada node = DaemonSet. Tarefa finita = Job.

#### 12.6 — O que vai dentro do `spec` do Deployment

O vídeo menciona **3 elementos canônicos** do spec do Deployment:

| Elemento | Função |
|----------|--------|
| **`replicas`** | Número de pods desejados |
| **`template.spec.containers`** | Quais containers cada pod roda (image, ports, resources, env, etc.) |
| **`template.spec.volumes`** | Quais volumes os pods montam (PV/PVC, ConfigMap, Secret, etc.) |

Mais o **`selector`** (não citado explicitamente no vídeo mas essencial):

```yaml
spec:
  replicas: 3
  selector:                    # ← Deployment usa isto pra achar "seus" pods
    matchLabels:
      app: nginx
  template:                    # ← molde do pod
    metadata:
      labels:
        app: nginx             # ← TEM que casar com selector.matchLabels
    spec:
      containers: [...]
      volumes: [...]
```

> **Armadilha de prova clássica.** `selector.matchLabels` **tem que casar** com `template.metadata.labels`. Se você muda o label do template sem mudar o selector (ou vice-versa), o Deployment perde a referência dos pods e cria **novos pods**, deixando os antigos órfãos. Isso é fonte clássica de "por que eu tenho 6 pods se declarei replicas: 3?".

#### 12.7 — Pegadinhas de prova consolidadas (Sub-bloco 12)

| Afirmação | Verdade |
|-----------|---------|
| "Posso ter um Pod `nginx` e um Service `nginx` no mesmo namespace" | ✅ Unicidade é por **kind + namespace** — diferentes kinds podem repetir nome |
| "Depois de deletar um Pod posso reusar o nome" | ✅ Sim — mas o **UID** será diferente |
| "Manifest pode estar em YAML ou JSON" | ✅ Ambos suportados. YAML é o convencional |
| "O `status` faz parte do manifest que eu escrevo" | ❌ Você só escreve **spec** — status vem do K8s |
| "Pods soltos (sem controller) são recriados se morrerem" | ❌ **Nunca** — só pods gerenciados por controller (Deployment/StatefulSet/etc.) |
| "Deployment cria pods diretamente" | ❌ Cria **ReplicaSet**, que cria pods |
| "kubectl apply do mesmo arquivo 2 vezes cria 2 vezes os objetos" | ❌ `apply` é **idempotente** — só atualiza diff. Quem cria sempre é `create` (que falha se existe) |
| "Labels servem só pra organização visual no Console" | ❌ Labels são **funcionais** — Services/Deployments/NetworkPolicies selecionam pods por label |
| "Label selectors fazem OR por default" | ❌ Vírgula = **AND**. Para OR, use set-based `in (a,b)` |
| "Rolling update reusa o mesmo ReplicaSet" | ❌ Cria **um novo ReplicaSet** com a nova versão; escala antigo pra 0 |
| "Rollback é manual — você precisa reaplicar o YAML antigo" | ❌ `kubectl rollout undo` re-escala o ReplicaSet anterior — instantâneo |
| "DaemonSet é como Deployment mas com replicas configuráveis" | ❌ DaemonSet **não tem replicas** — é "**um pod por node**" automaticamente |
| "Job é o controller para web servers" | ❌ Job é pra workloads **finitos** (que terminam). Web server = Deployment |
| "Selector do Deployment é opcional" | ❌ É **obrigatório** e tem que casar com labels do template |
| "`apiVersion: v1` cobre todos os kinds" | ❌ Cobre só o **core group** (Pod, Service, ConfigMap, Namespace, etc.). Deployment é `apps/v1`, Job é `batch/v1`, Ingress é `networking.k8s.io/v1` |
| "Pode usar underscore no nome do objeto" | ❌ Só **letras, números, hífens, pontos** (DNS-1123) |

#### 12.8 — Mnemônicos do Sub-bloco 12

- **"N-U-L: Name, UID, Label."** As 3 formas de identidade. Name pra humano, UID pra máquina, Label pra grupo.
- **"Pod = cattle, não pet."** Pods são descartáveis — você nunca cria pod solto em produção; sempre via controller.
- **"Deployment → ReplicaSet → Pod."** A árvore canônica de workload stateless. Decore essa hierarquia.
- **"`apply` é idempotente, `create` é uma vez só."** Em prova ACE, se a pergunta envolve "aplicar o mesmo manifest várias vezes sem erro", a resposta é `apply`.
- **"4 obrigatórios no manifest: `apiVersion`, `kind`, `metadata`, `spec`."** Mnemônico **AKMS** ("á-kê-éme-éssi") — toda manifest tem esses 4 no topo.
- **"Selector tem que casar com template labels."** Esquece isso = pods órfãos.
- **"Workloads: Stateless → Deployment, Stateful → StatefulSet, Per-node → DaemonSet, Finito → Job/CronJob."**

### Sub-bloco 2 — Vídeo "The kubectl command" (deep dive na ferramenta)

#### 2.1 — O que é `kubectl` e por que importa

**Definição precisa:** `kubectl` é uma **utility de linha de comando** que administradores usam pra controlar clusters Kubernetes. Mecânica: ele **transforma comandos CLI em chamadas HTTPS pra API do kube-apiserver** do control plane.

| Camada | Papel |
|--------|-------|
| Você digita | `kubectl get pods` |
| `kubectl` faz | Lê config local → monta request HTTP `GET /api/v1/namespaces/default/pods` → autentica → envia HTTPS pro apiserver |
| `kube-apiserver` faz | Auth + authz + query no `etcd` |
| Resposta | JSON do apiserver → `kubectl` renderiza tabela bonita no terminal |

> **Analogia.** `kubectl` é como o **navegador**: você digita uma URL (comando), ele monta o HTTP request, recebe o JSON, e te mostra HTML/tabela. Não é "mágica" — é só um cliente HTTP especializado pra falar K8s API.

##### Fluxo end-to-end de `kubectl get pod`

```
1. Você digita: kubectl get pod
       │
       ▼
2. kubectl lê ~/.kube/config → encontra endpoint + credenciais do cluster atual
       │
       ▼
3. kubectl monta request HTTPS: GET https://<apiserver-ip>/api/v1/namespaces/default/pods
       │   Header: Authorization: Bearer <token>
       ▼
4. kube-apiserver recebe → auth + authz + admission
       │
       ▼
5. apiserver consulta o etcd (key-value store, Sub-bloco 6 do Module 3)
       │
       ▼
6. etcd devolve a lista de Pods em JSON
       │
       ▼
7. apiserver devolve o JSON via HTTPS pro kubectl
       │
       ▼
8. kubectl renderiza:
       NAME          READY   STATUS    RESTARTS   AGE
       my-app-7d4    1/1     Running   0          2m
```

> **Recap Module 3 Sub-bloco 9 (pegadinha):** "kubectl fala direto com o etcd" → ❌ **Não**. `kubectl` fala com **apiserver**, que é a **única coisa** que toca o etcd.

#### 2.2 — Configuração do kubectl (o arquivo `~/.kube/config`)

`kubectl` precisa saber **(a) onde está o cluster** e **(b) como autenticar**. Essas duas coisas vivem em um arquivo YAML:

| Atributo | Valor padrão |
|----------|--------------|
| **Localização do arquivo** | `$HOME/.kube/config` (pasta oculta `.kube` no home) |
| **Formato** | YAML |
| **Conteúdo** | Lista de **clusters** (endpoint + CA cert), **users** (credenciais), e **contexts** (combinação de cluster + user + namespace default) |

##### Estrutura do `~/.kube/config` (exemplo simplificado)

```yaml
apiVersion: v1
kind: Config
clusters:                    # ← onde estão os clusters
  - name: gke_proj_us-central1_my-cluster
    cluster:
      server: https://34.123.45.67
      certificate-authority-data: LS0tLS1CRUdJTi...
users:                       # ← credenciais
  - name: gke_proj_us-central1_my-cluster
    user:
      auth-provider:
        name: gcp
contexts:                    # ← combinações que você pode "switchar"
  - name: gke_proj_us-central1_my-cluster
    context:
      cluster: gke_proj_us-central1_my-cluster
      user: gke_proj_us-central1_my-cluster
      namespace: default
current-context: gke_proj_us-central1_my-cluster   # ← o ativo
```

##### `kubectl config view` — meta-comando importante

Aqui mora uma **distinção sutil mas vital**:

| Comando | O que mostra |
|---------|--------------|
| `kubectl config view` | A configuração do **próprio kubectl** (clusters/users/contexts do arquivo `~/.kube/config`) |
| `kubectl get pods` / `kubectl describe deployment X` / etc. | Configurações de **objetos dentro do cluster** (pods, deployments, services) |

> **Armadilha de prova.** "Como vejo o estado dos pods do cluster?" → **`kubectl get pods`** (não `kubectl config view`). `config view` é sobre **o kubectl** (qual cluster ele aponta), não sobre **objetos** no cluster.

#### 2.3 — De onde vêm as credenciais? (`gcloud container clusters get-credentials`)

**GKE fornece as credenciais via comando `gcloud`.** O passo canônico pra conectar `kubectl` a um cluster GKE:

```bash
gcloud container clusters get-credentials CLUSTER_NAME \
  --zone=ZONE_OR_REGION \
  --project=PROJECT_ID
```

**O que esse comando faz por baixo dos panos:**

1. Pede ao **GKE control plane** (via Google API) o endpoint e o CA cert do cluster.
2. Escreve uma **entrada nova** (ou atualiza a existente) no `~/.kube/config`:
   - Novo entry em `clusters:` (endpoint do apiserver)
   - Novo entry em `users:` (credencial via `gcp` auth provider — usa seu login `gcloud`)
   - Novo entry em `contexts:` ligando os dois
3. Define o **`current-context`** pra esse cluster.

**Características importantes:**

| Característica | Detalhe |
|----------------|---------|
| **One-time per cluster (no Cloud Shell)** | Como `~/.kube/config` persiste no `$HOME` do Cloud Shell, você roda 1 vez e fica configurado |
| **Idempotente** | Rodar de novo no mesmo cluster sobrescreve sem quebrar |
| **Múltiplos clusters** | Rodar em N clusters diferentes adiciona N entries no config — você pode switchar entre eles |
| **Cloud Shell tem tudo pré-instalado** | `gcloud` e `kubectl` já vêm no Cloud Shell — não precisa instalar nada |

> **Pegadinha.** "Get-credentials rerodado pra outro cluster apaga o anterior" → ❌ **Não apaga**, **adiciona**. O que ele faz é mudar o `current-context` pro novo cluster. Os outros continuam acessíveis via `kubectl --context=<nome-do-outro>` ou `kubectl config use-context <outro>`.

#### 2.4 — `gcloud` vs `kubectl` — a divisão de responsabilidades

**Esta é uma das pegadinhas mais cobradas em prova ACE.** Os dois CLIs fazem coisas **completamente diferentes**:

| Aspecto | `gcloud` | `kubectl` |
|---------|----------|------------|
| **Escopo** | Google Cloud todo (qualquer serviço GCP) | **Só Kubernetes** (qualquer cluster K8s, não só GKE) |
| **Auth** | Google login (`gcloud auth login`, ADC, SAs) | Via `~/.kube/config` (que pode usar `gcp` auth provider) |
| **Cria/destrói clusters GKE** | ✅ Sim — `gcloud container clusters create/delete` | ❌ **Não** |
| **Muda forma do cluster** (node pools, machine types, versão) | ✅ Sim — `gcloud container clusters resize/upgrade` | ❌ **Não** |
| **Gerencia objetos dentro do cluster** (pods, deployments, services) | ❌ Não | ✅ Sim — `kubectl apply/get/describe/delete` |
| **Configura `kubectl`** | ✅ `gcloud container clusters get-credentials` | ❌ Não |
| **Onde roda** | Localmente, Cloud Shell, scripts CI | Onde tiver `~/.kube/config` configurado |

##### Diagrama mental

```
┌──────────────────────────────────────┐
│  gcloud — porta de entrada do GCP    │
│  (cria/redimensiona/destrói cluster) │
└─────────────┬────────────────────────┘
              │
              │ get-credentials
              ▼
┌──────────────────────────────────────┐
│  ~/.kube/config (credencial + URL)   │
└─────────────┬────────────────────────┘
              │
              │ usado por
              ▼
┌──────────────────────────────────────┐
│  kubectl — dentro do cluster         │
│  (pods, deployments, services, ...)  │
└──────────────────────────────────────┘
```

> **Pegadinha clássica.** "Posso usar `kubectl` pra criar um cluster GKE novo" → ❌ **Não**. `kubectl` só gerencia o **conteúdo** de um cluster existente. Criar cluster é trabalho do `gcloud` (ou Console, ou Terraform).

#### 2.5 — Sintaxe do `kubectl`: 4 partes

> **Citação do vídeo:** *"kubectl's syntax is composed of four parts: the command, the type, the name, and optional flags."*

```
kubectl  <COMMAND>  <TYPE>  [NAME]  [FLAGS]
   │         │         │       │        │
   │         │         │       │        └─ opcional, modificadores
   │         │         │       └────────── opcional, nome específico do objeto
   │         │         └────────────────── tipo de objeto (pod, deployment, ...)
   │         └──────────────────────────── ação (get, describe, logs, ...)
   └────────────────────────────────────── a ferramenta
```

##### Parte 1 — Command (a ação)

| Categoria | Comandos típicos |
|-----------|------------------|
| **Inspeção (read-only)** | `get`, `describe`, `logs`, `top`, `explain`, `config view` |
| **Mutação** | `apply`, `create`, `delete`, `edit`, `patch`, `scale`, `rollout`, `label`, `annotate` |
| **Acesso direto** | `exec`, `port-forward`, `cp`, `attach`, `proxy` |
| **Outros** | `version`, `cluster-info`, `api-resources` |

##### Parte 2 — Type (qual tipo de objeto)

Os mais comuns:

| Forma curta | Forma longa | Tipo |
|-------------|-------------|------|
| `po` | `pods` / `pod` | Pod |
| `deploy` | `deployments` / `deployment` | Deployment |
| `rs` | `replicasets` / `replicaset` | ReplicaSet |
| `svc` | `services` / `service` | Service |
| `cm` | `configmaps` / `configmap` | ConfigMap |
| `ns` | `namespaces` / `namespace` | Namespace |
| `no` | `nodes` / `node` | Node |
| `ing` | `ingresses` / `ingress` | Ingress |
| `pv` / `pvc` | `persistentvolume` / `persistentvolumeclaim` | Volume |

> **Dica.** `kubectl api-resources` lista **tudo** que o cluster suporta (incluindo CRDs).

##### Parte 3 — Name (opcional)

| Com nome | Sem nome |
|----------|----------|
| `kubectl get pod my-test-app` → info **só do pod `my-test-app`** | `kubectl get pods` → **lista todos** os pods do namespace atual |

**Comando que requer nome:** `describe`, `logs`, `exec`, `delete` (na maioria dos usos).
**Comando que opcionalmente usa nome:** `get`.

##### Parte 4 — Flags (modificadores)

As mais úteis no dia a dia:

| Flag | O que faz | Exemplo |
|------|-----------|---------|
| `-o=yaml` / `-o yaml` | Saída em YAML — mostra o **estado completo** do objeto (spec + status) | `kubectl get pod nginx -o yaml` — útil pra exportar e recriar em outro cluster |
| `-o=json` | Saída em JSON | Idem, formato JSON pra parsear com `jq` |
| `-o=wide` | Tabela **com colunas extras** — incluindo o **node** em que o pod roda, IP do pod, etc. | `kubectl get pods -o wide` |
| `-o=name` | Só o nome (útil pra scripts) | `kubectl get pods -o name \| xargs -I{} kubectl delete {}` |
| `-n NAMESPACE` | Roda no namespace especificado | `kubectl get pods -n kube-system` |
| `--all-namespaces` / `-A` | Todos os namespaces | `kubectl get pods -A` |
| `-l KEY=VALUE` | Filtra por label selector | `kubectl get pods -l app=nginx` |
| `--kubeconfig=FILE` | Usa outro arquivo de config (não `~/.kube/config`) | Útil em CI/CD com config dedicado |
| `--context=NAME` | Usa outro contexto dentro do mesmo config | Switch rápido entre clusters |
| `-f FILE.yaml` | Especifica um manifest (usado com `apply`, `create`, `delete`) | `kubectl apply -f deploy.yaml` |
| `--watch` / `-w` | Fica observando mudanças em tempo real | `kubectl get pods -w` (vê pods nascendo/morrendo) |

#### 2.6 — Exemplos práticos do vídeo (e expansão)

```bash
# Listar todos os pods do namespace atual
kubectl get pods

# Filtrar por nome
kubectl get pod my-test-app

# Pegar o YAML completo de um pod (spec + status preenchido pelo K8s)
kubectl get pod my-test-app -o=yaml

# Listar pods com coluna NODE (em que máquina rodam)
kubectl get pods -o=wide

# Tudo do cluster, em qualquer namespace
kubectl get pods -A

# Filtrar por label
kubectl get pods -l app=nginx,env=prod

# Apontar pra outro cluster sem mudar o current-context global
kubectl --context=prod-cluster get pods

# Usar config alternativo
kubectl --kubeconfig=./prod-kubeconfig.yaml get pods
```

> **`-o yaml` é uma porta dos fundos útil.** O vídeo destaca: *"telling kubectl to produce a YAML output can be helpful for other tasks related to Kubernetes objects, for example, recreating an object in another cluster."* — fluxo real: `kubectl get deploy X -o yaml > backup.yaml` → editar → `kubectl apply -f backup.yaml` em outro cluster.

### Sub-bloco 3 — Vídeo "Introspection" (debugging com kubectl: get, describe, exec, logs)

> **Por que este sub-bloco está aqui.** Sub-bloco 2 ensinou a **falar** com o cluster. Este vídeo ensina a **ouvir** — quando algo está errado, quais comandos extraem o **estado real** do cluster pra você descobrir o porquê. Em produção, 80% do uso diário de kubectl é introspection: pod travado, app caindo, network não funciona — você sempre começa olhando, não modificando.

#### 3.1 — O que é introspection?

> **Citação do vídeo:** *"Introspection... is the act of gathering information about the containers, pods, services, and other engines that run within the cluster."*

É o **subconjunto read-only** dos comandos kubectl. Você não muda nada — só **extrai informação** pra entender o estado atual. As **4 ferramentas-âncora** do vídeo:

| Comando | Em uma frase | Quando usar |
|---------|--------------|--------------|
| `kubectl get` | Lista objetos com **status resumido** | "Está rodando?" |
| `kubectl describe` | Mostra **tudo** sobre um objeto + **Events** | "Por que está nesse estado?" |
| `kubectl logs` | Mostra **stdout/stderr** do app dentro do container | "O que o app está dizendo?" |
| `kubectl exec` | **Executa comando dentro do container** (ou shell interativo) | "Eu preciso entrar e tocar" |

> **Ordem natural de troubleshooting:** `get` → `describe` → `logs` → (se necessário) `exec`. Da menor invasão pra maior.

#### 3.2 — Pod Phases (os 6 estados de alto nível)

Quando você roda `kubectl get pods`, a coluna **STATUS** mostra o **phase** do Pod. **São 6 estados** (5 do K8s open-source + 1 derivado clássico do GKE):

| Phase | O que significa | O que está acontecendo por baixo |
|-------|------------------|----------------------------------|
| **Pending** | K8s **aceitou** o Pod, mas ele **ainda não está rodando** | Scheduler ainda decidindo node, OU node escolhido mas **imagem sendo puxada** (image pull), OU PVC não bound, OU quota apertada |
| **Running** | Pod **anexou num node** e **todos os containers foram criados** | Pelo menos 1 container está rodando, iniciando, ou reiniciando |
| **Succeeded** | **Todos os containers terminaram com sucesso** e **não vão reiniciar** | Típico de Jobs/CronJobs que rodam até concluir |
| **Failed** | **Pelo menos 1 container terminou com erro** e **não vai reiniciar** | Exit code ≠ 0 e `restartPolicy: Never` (ou Job que falhou irremediavelmente) |
| **Unknown** | K8s **não consegue saber** o estado | Comunicação **control plane ↔ kubelet** está quebrada (node fora do ar, network particionada) |
| **CrashLoopBackOff** | Container **morreu pelo menos uma vez** e está em **backoff exponencial** entre tentativas de restart | Reinicia → crasha → espera (10s, 20s, 40s, ... até 5min) → tenta de novo. **Erro mais comum em produção** |

##### Diagrama do ciclo de vida típico

```
                   ┌──────────┐
                   │ Pending  │  ← apiserver persistiu o pod
                   └────┬─────┘     scheduler ainda decidindo
                        │           ou imagem sendo puxada
                        ▼
                  ┌──────────┐
                  │ Running  │  ← agendado + imagem puxada + container started
                  └────┬─────┘
        ┌──────────────┼──────────────┐
        │              │              │
        ▼              ▼              ▼
  ┌──────────┐  ┌────────────┐  ┌──────────────────┐
  │ Succeeded│  │   Failed   │  │ CrashLoopBackOff │
  │ (Job ✅) │  │ (exit ≠ 0  │  │ (crasha + tenta  │
  │          │  │  + Never)  │  │  de novo + ...)  │
  └──────────┘  └────────────┘  └──────────────────┘

                          ┌──────────┐
                          │ Unknown  │  ← pod existe mas kubelet
                          └──────────┘     não responde
```

> **A pegadinha de prova mais comum:** "CrashLoopBackOff significa que o container está rodando agora" → ❌ **Não**. Significa que ele **morreu, está em pausa, e vai tentar reiniciar**. É um estado **cíclico**, não estável.

##### CrashLoopBackOff — causas típicas (extra do vídeo)

| Causa | Como confirmar |
|-------|----------------|
| App crashou no startup (env var faltando, config inválida) | `kubectl logs POD --previous` (logs do container anterior) |
| Image errada (entrypoint não existe) | `kubectl describe pod POD` — vê erro no Events |
| Liveness probe muito agressivo (mata antes do app subir) | `describe` mostra "Liveness probe failed" |
| Recursos insuficientes (OOMKilled) | `describe` mostra `Last State: Terminated, Reason: OOMKilled` |
| Config errada (Secret/ConfigMap referenciado não existe) | `describe` mostra `Failed to mount volume` |

#### 3.3 — `kubectl get` — status de alto nível

Já visto em Sub-bloco 2. Aqui o foco é **debugging** — `get` é seu primeiro look-around:

```bash
# Visão geral — quem está com problema?
kubectl get pods                  # NAMES + READY + STATUS + RESTARTS + AGE
kubectl get pods -o wide          # ... + IP + NODE  (descobre se concentrou em 1 node)
kubectl get pods -A               # tudo, todos namespaces
kubectl get pods --watch          # vê em tempo real (pods nascendo/morrendo)

# Cruzar com Deployment/ReplicaSet
kubectl get deploy,rs,pods -l app=nginx       # vê a árvore inteira
```

**Colunas que importam pra troubleshooting:**

| Coluna | Sinal |
|--------|-------|
| `READY` | `0/1` ou `0/2` = container não está pronto (liveness/readiness falhando) |
| `STATUS` | `CrashLoopBackOff`, `ImagePullBackOff`, `Pending`, `Error` = problema |
| `RESTARTS` | `> 0` e crescente = pod está caindo repetidamente |
| `AGE` | Pod **recém-criado** (segundos) + restarts = crash no startup |

#### 3.4 — `kubectl describe pod NAME` — o canivete suíço do troubleshooting

> **Citação do vídeo:** *"This command provides information about a Pod and its containers such as labels, resource requirements, and volumes. It also details the status information about the Pod and container."*

O `describe` é o **comando mais útil** pra investigar um Pod. Saída tem ~6 seções:

| Seção | Conteúdo | Por que importa |
|-------|----------|------------------|
| **Metadata** | Name, Namespace, Node, Labels, Annotations | Vê em qual node o pod foi parar; labels pra confirmar Service matching |
| **Status** | Phase, IP, QoS Class, Start Time | Vê o estado high-level + IP atribuído |
| **Containers** (pra cada container) | Image, Image ID, Port, Host Port, State (Waiting/Running/Terminated), Last State, Ready, Restart Count, Resource Requests/Limits, Environment, Mounts | **A maioria do diagnóstico mora aqui** — image errada? OOMKilled? Volume não montou? |
| **Conditions** | PodScheduled, Initialized, ContainersReady, Ready | True/False — dá pra ver onde travou (ex.: PodScheduled=False = scheduler não achou node) |
| **Volumes** | Lista de volumes montados + tipo (configMap, secret, PVC, emptyDir, ...) | Confirma que Secret referenciado existe |
| **Events** ⭐ | **Histórico de eventos do K8s pro pod** — Scheduled, Pulling, Pulled, Created, Started, Killing, FailedScheduling, FailedMount, BackOff, OOMKilled, Unhealthy (probe failed), etc. | **O ouro da depuração.** 9 de 10 problemas são óbvios olhando os Events |

##### Eventos típicos e o que significam

| Event Reason | Tradução |
|--------------|----------|
| `Scheduled` | Scheduler escolheu o node |
| `Pulling` / `Pulled` | Image puxada com sucesso |
| `Failed to pull image` | Registry inacessível, tag errada, ou credenciais (`ImagePullSecrets`) faltando |
| `ImagePullBackOff` | Após N falhas de pull, K8s entra em backoff |
| `Created` / `Started` | Container subiu |
| `Killing` | K8s está matando o container (graceful) |
| `OOMKilled` | Container excedeu o **memory limit** — kernel oom-killer matou |
| `Unhealthy` | Liveness/readiness/startup probe falhou |
| `BackOff` | Container morreu, K8s espera antes de reiniciar |
| `FailedScheduling` | Scheduler não achou node compatível (recursos, taints, affinity) |
| `FailedMount` | Volume não montou (Secret/PVC inexistente, permissão, ...) |

##### Container State (3 estados runtime)

> **Diferença Pod Phase vs Container State.** Pod Phase é **alto nível** (Pending/Running/...). Container State é **fine-grained** (cada container individual no pod):

| Container State | Sub-info |
|-----------------|----------|
| **Waiting** | `Reason: ImagePullBackOff` / `CrashLoopBackOff` / `ContainerCreating` |
| **Running** | `Started: TIMESTAMP` |
| **Terminated** | `Exit Code: N`, `Reason: Completed` / `Error` / `OOMKilled`, `Started/Finished` timestamps |

> **Dica de troubleshooting:** `Last State` (estado anterior) é tão útil quanto o `State` atual. Se `State: Waiting (CrashLoopBackOff)` e `Last State: Terminated (Error, Exit 137)` → 137 = killed (provavelmente OOMKilled).

#### 3.5 — `kubectl logs POD` — o que o app está dizendo

> **Citação do vídeo:** *"The logs contain both the standard output and standard error messages that the applications within the container have generated."*

**Mecânica:** containers escrevem em `stdout` e `stderr`. O **container runtime** (containerd no GKE) captura essas streams em arquivos no node. O kubelet expõe esses arquivos via apiserver, e o `kubectl logs` puxa via HTTPS. **Não é mágica** — é só leitura de arquivo, intermediada pelo apiserver.

```bash
# Logs do container atual
kubectl logs POD

# Pod com múltiplos containers → especifica qual
kubectl logs POD -c CONTAINER_NAME

# Pod que está em CrashLoopBackOff — quero os logs do container ANTERIOR (que crashou)
kubectl logs POD --previous            # ⭐ flag salva-vidas

# Follow (tipo `tail -f`) — vê logs ao vivo
kubectl logs POD -f

# Últimas N linhas
kubectl logs POD --tail=100

# Logs desde N tempo atrás
kubectl logs POD --since=10m

# Combinação típica de debug
kubectl logs POD -c app --tail=200 -f
```

> **A flag mais importante na vida real é `--previous`.** Em CrashLoopBackOff, o container atual ainda nem subiu (está esperando o backoff) — `kubectl logs POD` sem `--previous` retorna **vazio ou erro**. Você precisa dos logs do container **morto** pra saber por que ele morreu.

#### 3.6 — `kubectl exec` — entrar no container

> **Citação do vídeo:** *"Single command execution... lets you run a single command inside a container and view the results in your own command shell."*

`exec` permite **executar comando dentro do container** — útil quando logs não bastam e você precisa **interagir**.

##### Mecânica (em camadas)

```
kubectl exec → apiserver (HTTPS) → kubelet do node → container runtime (containerd)
              → spawn um processo dentro do PID namespace do container → stream de I/O de volta
```

##### Modos de uso

```bash
# Comando único — vê o resultado e sai
kubectl exec POD -- ls /app
kubectl exec POD -- env
kubectl exec POD -- cat /etc/nginx/nginx.conf

# Pod com múltiplos containers → especifica qual
kubectl exec POD -c CONTAINER -- ls /app

# Shell interativo
kubectl exec -it POD -- /bin/bash    # ou /bin/sh em containers minimalistas (Alpine)
```

##### O que `-i` e `-t` significam (a sutileza do vídeo)

> **Citação do vídeo:** *"The -i argument tells kubectl to pass the terminal's standard input to the container, and the -t argument tells kubectl that the input is a TTY."*

| Flag | Significa | Sem ele... |
|------|-----------|------------|
| **`-i`** (`--stdin`) | **STDIN** do seu terminal → enviado pro processo no container | Comando roda mas você **não consegue digitar** dentro dele |
| **`-t`** (`--tty`) | Aloca um **pseudo-TTY** (terminal virtual) | Shell roda mas **sem cores, sem `clear`, sem bash prompt visível** |

**O par `-it` (ou `-ti`)** é o jeito canônico de abrir um **shell interativo**. Sem o par:

```bash
kubectl exec POD -- /bin/bash
# Sai imediatamente — o bash não tem TTY pra interagir, encerra
```

> **Pegadinha de prova:** "`kubectl exec POD -- bash` abre um shell interativo" → ❌ **Não**. Sem `-it`, o comando executa e retorna imediatamente. Pra shell interativo é **sempre `-it`**.

##### Quando NÃO usar exec

Os usuários frequentemente "instalam coisas" dentro do container via exec — o vídeo é explícito que isso é **anti-pattern**:

| ❌ Tentação | ✅ Caminho certo |
|------------|------------------|
| `kubectl exec -it POD -- apt install vim` | Adicionar `RUN apt install vim` no Dockerfile, rebuild, redeploy |
| `kubectl exec -it POD -- pip install lib` | Adicionar `lib` no `requirements.txt`, rebuild, redeploy |
| Editar arquivo de config no container | Mudar ConfigMap/Secret, K8s remonta |
| "Hot patch" código no container | Build nova imagem com a correção, `kubectl rollout` |

**Por que isso é ruim** (Sub-bloco 8 do Module 2, recap):
- A **writable layer** do container é **ephemeral** — quando o pod morre/reagenda, **tudo o que você instalou some**
- Se o ReplicaSet criar um novo pod, ele vai **sem** suas mudanças → inconsistência
- Quebra reprodutibilidade — você não consegue recriar o estado deliberadamente
- Quebra GitOps/CD — o estado real do cluster diverge do manifest

> **Citação chave do vídeo:** *"The interactive shell will allow you to figure out what needs to be changed to solve a problem, but you should then integrate those changes into your container images and redeploy them."* — exec é uma **ferramenta de investigação**, não de remediação.

#### 3.7 — Workflow canônico de troubleshooting

A ordem que você roda os comandos é quase sempre:

```
1. kubectl get pods                          ← quem está com problema?
       │
       ▼
2. kubectl describe pod POD-COM-PROBLEMA     ← Events + Container State revelam 80% dos casos
       │
       ▼
3. kubectl logs POD --previous               ← se descreve não bastou (crash silencioso, race)
       │
       ▼
4. kubectl exec -it POD -- /bin/sh           ← último recurso, só pra investigar
       │
       ▼
5. (corrigir no manifest/Dockerfile/ConfigMap, redeployar — NÃO no exec)
```

##### Cenários típicos

| Sintoma | Comando que revela |
|---------|---------------------|
| Pod stuck em `Pending` há minutos | `describe` → Events → "FailedScheduling: insufficient cpu" |
| Pod em `ImagePullBackOff` | `describe` → Events → "Failed to pull image: 401 Unauthorized" → faltou `imagePullSecrets` |
| Pod em `CrashLoopBackOff` | `logs --previous` → app stack trace mostrando NullPointer |
| Pod `Running` mas app não responde | `exec -it POD -- curl localhost:8080` (testa de dentro), `describe` checa readiness probe |
| OOMKilled recorrente | `describe` → `Last State: Terminated, Reason: OOMKilled` → aumentar `resources.limits.memory` |
| Volume não montou | `describe` → Events → "FailedMount" → Secret/PVC inexistente |

#### 3.8 — Pegadinhas de prova consolidadas (Sub-bloco 3)

| Afirmação | Verdade |
|-----------|---------|
| "Pod em `CrashLoopBackOff` está rodando" | ❌ Está **entre restarts**, em backoff exponencial |
| "Pod em `Pending` está rodando esperando tráfego" | ❌ `Pending` é **antes** de subir — scheduler ou image pull |
| "`Failed` e `CrashLoopBackOff` são o mesmo estado" | ❌ `Failed` = terminou e **não vai reiniciar**. CLBO = vai reiniciar continuamente |
| "`kubectl logs POD` mostra logs de todos os containers do pod" | ❌ Só do **primeiro/único** container. Multi-container → use `-c CONTAINER` |
| "`kubectl logs POD` em pod CrashLoopBackOff sempre mostra a causa do crash" | ❌ Pode estar **vazio** — use `--previous` pra ver logs do container morto |
| "`kubectl describe pod` muda configuração do pod" | ❌ É **read-only** — só mostra |
| "`kubectl exec POD -- bash` abre shell interativo" | ❌ Precisa **`-it`** (sem TTY + stdin, bash encerra imediatamente) |
| "Instalar pacote via `exec` é solução de produção" | ❌ Anti-pattern — writable layer é ephemeral, novo pod vem sem |
| "Section Events do `describe` mostra logs do app" | ❌ Mostra **eventos do K8s** sobre o pod (scheduled, pulled, killed, OOMKilled). Logs do app = `kubectl logs` |
| "`OOMKilled` aparece em `kubectl logs`" | ❌ Aparece em **`describe`** (`Last State: Terminated, Reason: OOMKilled`), não em logs do app — o app foi morto pelo kernel sem chance de logar |
| "Phase `Unknown` significa que o pod foi deletado" | ❌ Significa que o **kubelet não responde** — control plane perdeu contato. Pod pode estar rodando, ou não |
| "`kubectl logs --follow` é o mesmo que `tail -f` no node" | ⚠️ **Análogo**, mas via apiserver — stream HTTPS, não acesso direto ao arquivo |
| "Pra debugar pod multi-container preciso entrar em todos juntos" | ❌ `exec` entra em **um por vez** com `-c CONTAINER`. Cada container é um shell separado |
| "Exit code 137 é genérico" | ❌ É **especificamente** SIGKILL (128 + 9) — normalmente OOMKilled ou eviction |
| "Container `Waiting` está rodando lentamente" | ❌ `Waiting` é **antes de rodar** — pulling image, ou aguardando dependência |

#### 3.9 — Mnemônicos do Sub-bloco 3

- **"4 ferramentas de introspection: GDLE"** — **G**et + **D**escribe + **L**ogs + **E**xec. Da menor invasão pra maior.
- **"6 phases: PR-SF-UC"** — **P**ending → **R**unning → **S**ucceeded / **F**ailed / **U**nknown / **C**rashLoopBackOff.
- **"Events em describe = ouro da depuração."** Sempre olhe a seção Events primeiro.
- **"CrashLoopBackOff → `logs --previous`."** Container atual está morto/esperando — logs estão no anterior.
- **"`-i` é stdin, `-t` é TTY. Pra shell interativo, sempre os dois."**
- **"exec investiga, manifest corrige."** Nunca conserte com exec — é anti-pattern; corrija o Dockerfile/ConfigMap/manifest e redeploye.
- **"OOMKilled mora em `describe`, não em `logs`."** App não consegue logar quando o kernel mata.
- **"Pod Phase = alto nível. Container State = fine-grained."** Pod phase é o resumo; container state mora dentro de cada container.

#### Task 1 — Criar cluster Autopilot

```bash
gcloud container clusters create-auto $my_cluster --region $my_region
```

**Anatomia do comando:**

| Token | O que é | Por que importa |
|-------|---------|-----------------|
| `gcloud container clusters` | Grupo de comandos do GKE | "container" é o nome histórico — GKE nasceu como Google Container Engine antes de ser renomeado. O alias continua. |
| `create-auto` | Variante que cria **Autopilot** | `create` (sem `-auto`) criaria **Standard**. Não há flag `--autopilot` no `create` regular — é subcomando separado. |
| `$my_cluster` | Nome do cluster | Posicional (1º arg). Vira parte do `current-context` do kubeconfig depois (`gke_PROJECT_REGION_NAME`). |
| `--region $my_region` | **Region** (não zone) | Autopilot **exige** region. Control plane fica replicado em 3 zonas dessa region automaticamente. |

**O que está acontecendo por trás:**
1. GCP provisiona o **control plane regional** (3 réplicas do `kube-apiserver`, `etcd`, `scheduler`, `controller-manager` em 3 zonas distintas da `us-east4`). Você não vê isso — é o que "managed" significa.
2. GCP **NÃO provisiona nodes**. Autopilot só sobe node quando há Pod pedindo. Esse é o motivo do `Pending` que você viu na Task 4.
3. Network: cria sub-redes, range de IPs para Pods e Services (alias IP, VPC-native default), firewall rules necessárias.
4. Habilita add-ons obrigatórios: metrics-server, DNS (kube-dns), GLBC, Workload Identity, etc.

**Tempo esperado:** 3–7 minutos. O `gcloud` fica bloqueado mostrando spinner.

**Avisos amarelos que aparecem:** vários `WARNING` sobre defaults novos (release channel = Regular, log/metrics habilitados, etc.) — são informativos, não erros.

**Equivalente declarativo (Terraform):** `google_container_cluster` com `enable_autopilot = true`.

**Cenário real:** ninguém cria cluster prod via `gcloud` manual. Vai em **Terraform** ou **Config Connector**. Mas saber o comando ajuda em PoC, debug e para entender o exam.

#### Task 2 — Conectar `kubectl` ao cluster


**O que faz:**
1. Chama API do GKE pra pegar o **endereço do control plane** + **certificado CA** do cluster.

#### Task 4 — Deploy de Pods (a parte densa do lab)

##### 4.1 — Criar deployment imperativo

```bash
kubectl create deployment --image nginx nginx-1
```

**Anatomia:**

| Token | Significado |
|-------|-------------|
| `kubectl create` | Verbo imperativo — cria recurso na hora. **Anti-pattern em prod** (sem versionamento). |
| `deployment` | Tipo de recurso (não Pod direto — Deployment cria ReplicaSet que cria Pod). Resultado: você ganha self-healing automático. |
| `--image nginx` | Imagem do container. Sem tag → vira `nginx:latest`. **Anti-pattern em prod** (versão muda sob seus pés). |
| `nginx-1` | Nome do Deployment (posicional). Pods herdam esse nome + sufixo aleatório: `nginx-1-<replicaset-hash>-<pod-hash>`. |

**O que aparece no terminal:**

```
Warning: autopilot-default-resources-mutator:Autopilot updated Deployment default/nginx-1:
         defaulted unspecified 'cpu' resource for containers [nginx]
         (see http://g.co/gke/autopilot-defaults).
deployment.apps/nginx-1 created
```

**Esse warning é assinatura do Autopilot e cai em prova:**
- Você não definiu `--requests`/`--limits` → no Standard, Pod sairia sem nada (QoS `BestEffort`).
- No Autopilot, um **MutatingWebhookConfiguration** intercepta a criação e injeta defaults: `cpu=500m`, `memory=2Gi`, `ephemeral-storage=1Gi`.
- Resultado: `Limits == Requests` → **QoS Guaranteed** (todo Pod Autopilot é Guaranteed).
- Cobrança Autopilot é **por recursos requisitados** — por isso ele força ter requests.

##### 4.2 — Status `Pending` — por quê?

```bash
kubectl get pods
```

```
NAME                       READY   STATUS    RESTARTS   AGE
nginx-1-98f744bf7-5qjr6    0/1     Pending   0          28s
```

**Por que Pending (e não Running)?**

1. Autopilot **não pré-provisiona nodes**. Cluster recém-criado tem **0 nodes**.
2. Você pediu um Pod → scheduler tenta agendar → **FailedScheduling** (`0/0 nodes available: 0 Insufficient cpu, 0 Insufficient memory`).
3. **Cluster Autoscaler** detecta o Pod pendente e dispara scale-up: provisiona um node novo (em alguma zona da region).
4. ~1–2 min depois, o node fica `Ready`.
5. Scheduler agenda o Pod → kubelet do node faz `Pulling image "nginx"` → `Pulled` → `Created` → `Started`.
6. `kubectl get pods` agora mostra `Running 1/1`.

**Tempo total típico:** 2–4 minutos. Vá tomar café.

##### 4.3 — `kubectl top node` falhando

```bash
kubectl top node
```

```
error: Metrics API not available
```

**Por que:**
- O `top` consulta o **metrics-server** (add-on que coleta CPU/memória dos nodes a cada ~15s).
- Metrics-server **existe no cluster** (Autopilot vem com ele), mas:
  - Precisa de **um node** rodando pra coletar dados (e você está em scale-up).
  - Mesmo depois do node subir, leva ~30–60s pra começar a popular as métricas.
- **Solução:** aguarde o Pod sair de Pending. Re-rode `kubectl top node` 1–2 min depois — o erro some sozinho.

Saída esperada quando funciona:

```
NAME                                       CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
gk3-autopilot-cluster-1-pool-2-XXX-q4q7s   1971m        102%   1803Mi          30%
```

> CPU% > 100% acontece em node compartilhado quando há **bursting** acima do allocatable — não é erro, é o cgroup permitindo uso temporário acima do request.

##### 4.6 — Injetar arquivo no container com `kubectl cp`

```bash
nano ~/test.html
# conteúdo:
# <header><title>Este é o título</title></header>
# Olá mundo

kubectl cp ~/test.html $my_nginx_pod:/usr/share/nginx/html/test.html
```

**Anatomia do `kubectl cp`:**

| Parte | Significado |
|-------|-------------|
| `~/test.html` | Origem (path local no Cloud Shell). |
| `$my_nginx_pod:/usr/share/nginx/html/test.html` | Destino — `<pod>:<path-dentro-do-container>`. |
| Por padrão usa o **primeiro container** do Pod | Pra Pods multi-container: `-c <nome-container>`. |

**`/usr/share/nginx/html/` é o webroot do nginx** (definido no `nginx.conf` da imagem oficial). Tudo nessa pasta é servido em `http://host/`.

**⚠️ Anti-pattern em produção:**
- Mudança não está na **imagem** — se o Pod morrer e renascer, perde tudo.
- Em prod: rebuild da imagem com `COPY test.html /usr/share/nginx/html/` no Dockerfile, OU monta `ConfigMap`/`PersistentVolume`.
- `kubectl cp` é só pra **debug/lab**. Mesmo princípio do `kubectl exec` (Sub-bloco 3.6 do Module 4).

##### 4.7 — Expor o Pod via Service LoadBalancer

```bash
kubectl expose pod $my_nginx_pod --port 80 --type LoadBalancer
```

| Token | Significado |
|-------|-------------|
| `kubectl expose pod` | Cria um **Service** que aponta pro Pod via label selector. |
| `--port 80` | Porta do Service (lado do cliente). |
| `--type LoadBalancer` | Tipo do Service — provisiona um **Network Load Balancer externo do GCP** automaticamente. |

**O que acontece por trás:**
1. K8s cria objeto Service tipo `LoadBalancer`.
2. **GKE cloud-controller-manager** vê isso → chama API do GCP pra criar Forwarding Rule + Backend Service + Target Pool (NLB passthrough L4).
3. GCP aloca um **IP externo público**.
4. Service ganha esse IP no campo `EXTERNAL-IP`.

```bash
kubectl get services
```

Primeiras vezes:
```
NAME             TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
kubernetes       ClusterIP      10.11.240.1    <none>        443/TCP        1h
nginx-1-...wsc   LoadBalancer   10.11.240.87   <pending>     80:31695/TCP   3s
```

`<pending>` significa: GCP ainda provisionando o NLB. Leva 30–90s. Re-rode até aparecer IP.

**Pegadinha de prova:** o tipo `LoadBalancer` **só funciona em cloud provider integrado** (GCP, AWS, Azure). Em K8s on-premise ou bare metal, fica `<pending>` pra sempre — precisa de MetalLB.

**Por que existe um Service `kubernetes` no cluster?**
- É criado automaticamente pelo control plane no namespace `default`.
- Permite Pods acessarem o `kube-apiserver` via DNS (`kubernetes.default.svc.cluster.local`).
- Não mexa nele.

Aqui o lab **muda de filosofia**: sai do imperativo (`kubectl create`) e entra no **declarativo** (`kubectl apply -f`).

##### 5.2 — Anatomia do manifest

`new-nginx-pod.yaml`:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: new-nginx
  labels:
    name: new-nginx
spec:
  containers:
  - name: new-nginx
    image: nginx
    ports:
    - containerPort: 80
```

| Campo | Significado |
|-------|-------------|
| `apiVersion: v1` | Pod é recurso "core" (sem grupo). Já um Deployment seria `apps/v1`. |
| `kind: Pod` | Cria Pod **solto** — sem ReplicaSet, sem Deployment. Se morrer, **não renasce**. Ver Sub-bloco 12.3 do Module 3 (armadilha). |
| `metadata.name: new-nginx` | Nome único dentro do namespace. |
| `metadata.labels.name: new-nginx` | Label arbitrário. Useful pra um Service mais tarde apontar via `selector: { name: new-nginx }`. |
| `spec.containers[0].image: nginx` | Mesma imagem do exemplo imperativo. |
| `spec.containers[0].ports[0].containerPort: 80` | **Apenas documentação informativa** — não abre porta, não cria firewall. O nginx escuta na 80 por causa do `nginx.conf`, não desse campo. |

##### 5.3 — Aplicar o manifest

```bash
kubectl apply -f ./new-nginx-pod.yaml
```

- `apply` é **idempotente**: se o recurso não existe, cria. Se existe, faz **patch** das diferenças (3-way merge).
- `-f ./new-nginx-pod.yaml` — caminho do manifest. Pode ser arquivo, diretório (aplica tudo recursivo com `-R`), ou URL.
- Resultado: `pod/new-nginx created`.

**Por que `apply` > `create`:**

| Aspecto | `create -f` | `apply -f` |
|---------|------------|------------|
| Recurso já existe | ❌ Erro `already exists` | ✅ Faz patch |
| Versionamento (GitOps) | ❌ Não rastreia diff | ✅ Compara com `last-applied-configuration` annotation |
| Re-rodar o mesmo arquivo | ❌ Quebra na 2ª vez | ✅ No-op se nada mudou |

**Regra de bolso:** **sempre `apply`** em prod. `create` só pra one-shot rápido.

##### 5.4 — Por que Pod solto é armadilha (recap)

Esse manifest cria **Pod solto**, não Deployment. Consequências:
- Sem self-healing: Pod morre? Não volta.
- Sem rolling update: pra trocar imagem, precisa deletar + recriar (downtime).
- Sem replicas: 1 Pod = 1 ponto de falha.

**Pra prod**, o YAML certo é Deployment (ver Sub-bloco 12.4 do Module 3). Esse lab usa Pod só pra simplificar a introspecção. **Não copie esse padrão pro trabalho.**

## Autopilot vs Standard

> **Resumo de bolso.** Para o **deep dive completo** (4 pilares de benefício, restrições técnicas com porquê, QoS classes, decision tree), veja o **[Sub-bloco 11](#sub-bloco-11--vídeo-autopilot-vs-standard-deep-dive-nos-2-modos-de-operação)** do Module 3 acima.

| Característica | **Autopilot** | **Standard** |
|----------------|---------------|--------------|
| Provisionamento | Fully-managed pelo Google | Você define e gerencia |
| Cobrança | Por **recursos dos pods** (pay-per-pod) | Por **nodes** provisionados (mesmo ociosos) |
| Localização | **Regional sempre** | **Zonal ou regional** |
| Gerenciamento | Nível de pod | Nível de cluster (node pools, machine types) |
| Image type | Apenas **Container-Optimized OS** | Ubuntu, COS, etc. |
| Custom packages | ❌ | ✅ |
| SSH no node / `hostPath` / `privileged` | ❌ Bloqueado | ✅ Permitido |
| QoS class dos pods | **Sempre Guaranteed** | Qualquer (Guaranteed / Burstable / BestEffort) |
| Pod-scheduling SLA | ✅ Contratual | ❌ Best-effort do cluster autoscaler |
| Flexibilidade de arquitetura | ❌ Limitada | ✅ Total |

**Regra mnemônica:** Autopilot = Auto**pilot** = você não pilota nada = só regional, só COS, sem customização. **Na dúvida, Autopilot** (recomendação default do Google).

## Availability — Zonal vs Regional

| Tipo | Control Plane | Nodes |
|------|---------------|-------|
| **Zonal** | Único, em uma zona | Pode distribuir em múltiplas zonas |
| **Regional** | Réplicas em múltiplas zonas | Replicados em **3 zonas** (configurável) |

---

## Network Routing

| Tipo | Como funciona | Quando usar |
|------|---------------|-------------|
| **VPC-native** | Pods usam **alias IPs** | Obrigatório para Internal HTTP(S) LB |
| **Routes-based** | Usa Google Cloud Routes | Legacy |

---

## Network Isolation

| Tipo | Característica |
|------|----------------|
| **Public GKE** | Nodes e pods acessíveis por IPs públicos |
| **Private GKE** | Pods/nodes usam IPs internos — isolados de redes públicas |

## Versões e upgrades

- Você escolhe versão específica **ou** release channel: **Rapid, Regular, Stable**
- Sem escolha → versão default atual
- **Best practice**: habilitar **auto-upgrade** de nodes e cluster
