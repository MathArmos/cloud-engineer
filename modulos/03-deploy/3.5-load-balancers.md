#### O que é Cloud Load Balancing

**Cloud Load Balancing** é o serviço gerenciado de balanceamento de carga do GCP. Ele distribui requisições entre recursos computacionais (VMs, containers, buckets, etc.), tanto **dentro de uma única região** quanto **entre várias regiões**, para atender requisitos de **alta disponibilidade** e **escala global**.

**Características-chave (cai em prova):**

| Característica | O que significa |
|---|---|
| **Fully distributed** | Não tem um "appliance" central que precise ser provisionado/dimensionado — está espalhado pela infraestrutura do Google |
| **Software-defined** | Implementado em software, não em hardware dedicado — você não compra/configura "um load balancer" físico |
| **Managed service** | Google opera, escala, atualiza e troubleshoot — você só configura |
| **Não é instance- ou device-based** | Diferente da AWS (ALB/NLB rodam em VMs gerenciadas com IP regional). No GCP **não há instâncias** representando o LB |
| **Single anycast IP** | Você expõe **um único IP global** que o Google anuncia em vários PoPs ao redor do mundo. O cliente sempre conecta no PoP mais próximo |
| **Mais de 1 milhão de QPS** | Escala horizontal massiva sem você configurar nada |

**Por que importa:**
- Você não precisa "subir um LB", configurar HA do próprio LB, gerenciar capacidade do LB. Isso é vendido como uma das grandes vantagens do GCP versus competidores — o LB é parte da infraestrutura, não um recurso que você opera.
- O **anycast IP** é a peça que permite "serve content as close as possible to your users" — o pacote do cliente entra na rede do Google no PoP mais próximo dele (Google tem ~150+ PoPs globais), e dentro da rede do Google segue até o backend.

**Analogia:**
Pense no LB do GCP como uma "função" da rede do Google — está em todo lugar, sempre on, sem instância física. Compare com AWS ALB: lá existe uma VM gerenciada com IP regional. No GCP, o IP é global e o "LB" é uma config, não uma máquina.

#### Os dois grandes grupos: Application LB vs Network LB

O GCP organiza seus load balancers em **dois grupos primários**, baseados na camada do modelo OSI em que operam:

| | **Application Load Balancer** | **Network Load Balancer** |
|---|---|---|
| **Camada OSI** | Layer 7 (Application) | Layer 4 (Transport) |
| **Roteia por** | HTTP(S) headers, cookies, URL paths | IP e porta |
| **Protocolos** | HTTP, HTTPS, HTTP/2, gRPC | TCP, UDP |
| **Features** | SSL/TLS termination, session affinity, **content-based routing** (URL maps) | TCP/UDP load balancing, health checks, baixa latência, alta vazão |
| **Use case típico** | Apps web, APIs, microsserviços com rotas (`/api`, `/static`) servidos por backends diferentes | Cargas TCP/UDP brutas: jogos, telefonia, streaming, qualquer coisa que precise de **latência mínima** ou que **não seja HTTP** |

---

##### Application Load Balancer (Layer 7)

**O quê:** balanceia tráfego HTTP(S) tomando decisões com base no **conteúdo da requisição** — pode olhar dentro do pacote (URL, header `Host`, cookie, método, etc.).

**Por que isso importa:**
- **Content-based routing (URL maps):** `/api/*` vai para um MIG de backend; `/static/*` vai para um Cloud Storage bucket; `/admin/*` vai para outro grupo de VMs. Tudo isso com **um único IP/hostname público**.
- **SSL/TLS termination:** o LB decripta o HTTPS no edge do Google, e fala HTTP simples (ou re-encriptado) com os backends. Centraliza certificados, livra os backends do custo de CPU do TLS.
- **Session affinity:** garante que o mesmo cliente caia no mesmo backend (útil para apps stateful tipo carrinho de compras em memória). Pode ser por client IP ou cookie.

**Quando usar:** qualquer app HTTP/HTTPS — site, API REST, GraphQL, gRPC. **Default** para apps web.

---

##### Network Load Balancer (Layer 4)

**O quê:** balanceia tráfego com base apenas em **IP de origem/destino + porta + protocolo**. Não olha dentro do pacote — não sabe o que é HTTP, não sabe que existe um header `Host`.

**Por que isso importa:**
- **Baixa latência:** menos processamento — não precisa parsear HTTP, não termina SSL. O pacote chega, o LB decide pra qual backend mandar, fim.
- **Alta vazão:** sem terminação de TLS no LB, o backend recebe o pacote praticamente como veio do cliente (preserva IP de origem em algumas variantes).
- **TCP/UDP genérico:** funciona para qualquer protocolo sobre TCP/UDP — não precisa ser HTTP. Servidor de jogo (UDP), gateway VoIP (SIP+RTP), banco que aceita conexões TCP diretas, etc.
- **Health checks:** verifica saúde dos backends mesmo sem entender o protocolo da aplicação (pode usar TCP check, HTTP check, HTTPS check ou SSL check).

**Quando usar:**
- Cargas **não-HTTP** (UDP, TCP brutos)
- Cargas que precisam **preservar IP do cliente** sem usar X-Forwarded-For
- Cargas onde **latência adicional do parsing L7 é inaceitável**
- Quando a aplicação faz seu próprio TLS e você não quer terminar no LB

#### O que é um Managed Instance Group

**MIG** = coleção de **VMs idênticas** controladas como **uma única entidade** a partir de um **instance template**.

A palavra-chave aqui é **idênticas**: todas as VMs do grupo nascem do **mesmo template**, com mesma imagem, mesmo machine type, mesmas tags, mesmo disco, mesmo startup script. Isso é o que permite tratá-las como **gado e não como animal de estimação** (cattle vs pets) — você não cuida de cada VM individualmente, cuida do grupo.

**Analogia:** instance template é a **fôrma**; MIG é o **lote de bolos** assados nessa fôrma. Você não decora cada bolo individualmente — você manda assar 10, pinta o lote inteiro com um novo glacê (rolling update), ou pede pra fazer mais 5 quando o cliente quer (autoscale).

---

#### Os 4 superpoderes que o MIG entrega

| Feature | O que faz | Por que importa |
|---|---|---|
| **Rolling updates** | Especificar novo template → MIG troca as VMs uma a uma (ou em lotes) sem derrubar o serviço | Deploy de nova versão sem downtime; rollback fácil voltando ao template anterior |
| **Autoscaling** | Cresce/diminui o número de VMs com base em métricas (CPU, requests/s, Pub/Sub queue, métrica custom) | Paga só pelo que usa; absorve picos sem você acordar de madrugada |
| **Load balancing integration** | MIG vira **backend service** de um LB; LB distribui tráfego entre as VMs do grupo | É **o** padrão de arquitetura para apps web no GCP: LB → MIG → autoscale |
| **Auto-healing** | Se uma VM parar, crashar ou for deletada (por algo que **não** seja comando do MIG), o MIG **recria** automaticamente — **mesmo nome, mesmo template** | A app continua de pé sem intervenção humana; "self-healing infrastructure" |

**Sobre auto-healing — sutileza importante:**
- "Pelas ações que não sejam comandos do instance group" significa: se **você** rodar `gcloud compute instance-groups managed resize --size=2` para diminuir o grupo, o MIG **não** vai recriar. Ele só recria quando a remoção não foi intencional (crash, delete manual fora do MIG, falha de hardware, instância marcada unhealthy).
- A nova instância tem **o mesmo nome** da que sumiu — útil porque IPs internos costumam vir do nome em algumas configurações, e logs/monitoring mantêm continuidade.
- O MIG decide "unhealthy" via **health check** que você configura (HTTP, HTTPS, TCP ou SSL).

#### Regional MIG vs Zonal MIG

| | **Zonal MIG** | **Regional MIG** |
|---|---|---|
| Onde as VMs ficam | 1 zona só | Espalhadas em **3+ zonas** dentro da região |
| Falha zonal | App fica fora | App continua servindo de outra zona |
| Quando usar | Casos raros — testes, workloads que **precisam** estar numa zona específica | **Default recomendado pelo Google** para qualquer carga de produção |
| Gestão | Você precisaria criar 1 MIG por zona se quiser HA = complexidade | **1 MIG só** cobre múltiplas zonas |

**Por que regional é a recomendação default:**
1. **Resiliência:** se a zona `us-central1-a` cair (falha de hardware, manutenção não-planejada, evento de capacidade), o MIG continua servindo de `us-central1-b` e `us-central1-c`.
2. **Sem gestão multi-MIG:** se você quisesse a mesma resiliência com zonal MIGs, teria que criar 3 MIGs (um por zona) e configurar LB pra balancear entre eles → 3× mais trabalho.
3. **Espalhamento automático de carga:** o regional MIG distribui as VMs ~igualmente entre as zonas, então o tráfego do LB já fica naturalmente balanceado.

**Armadilha de prova:**
A frase exata do curso é "**regional MIGs are generally recommended over zonal MIGs**". Se um cenário de prova falar "I need high availability against zone failures" e oferecer zonal MIG vs regional MIG → resposta é **regional**.

---

#### Como criar — fluxo em duas etapas

```
Instance template  →  Managed Instance Group  →  N VMs idênticas
   (a fôrma)             (o pedido de lote)        (o lote pronto)
```

**Etapa 1 — criar o instance template:**
- No console, o diálogo é **idêntico** ao de criar uma VM (machine type, imagem, disco, network, startup script, tags, SA).
- Diferença única: as escolhas são **gravadas** em vez de criar uma VM imediatamente. O template fica disponível para uso repetido.
- `gcloud compute instance-templates create <nome> --machine-type=... --image-family=... --boot-disk-size=...`

**Etapa 2 — criar o MIG referenciando o template:**
- Você define `N` (size inicial) e o template.
- O **instance group manager** popula automaticamente — cria N VMs usando o template.
- `gcloud compute instance-groups managed create <nome> --size=3 --template=<template> --zone=us-central1-a` (zonal)
- `gcloud compute instance-groups managed create <nome> --size=3 --template=<template> --region=us-central1` (regional — preferido)

#### As 6 decisões ao criar um MIG (cheklist de prova)

Quando o console pede pra configurar um MIG, ele te força a decidir 6 coisas — **decora essa ordem**, porque a prova pode perguntar "qual dessas opções **não** faz parte da criação de um MIG":

| # | Decisão | Opções |
|---|---|---|
| 1 | **Tipo de workload** | **Stateless** (front-end web, batch, processamento de fila) OU **Stateful** (banco de dados, app legado com estado em disco local) |
| 2 | **Nome** | Identificador do grupo |
| 3 | **Single-zone vs multi-zone** | Zonal MIG vs Regional MIG + localização |
| 4 | **Port name mapping** (opcional) | Mapear nome simbólico ("http") para porta (8080). Usado pelo LB para saber qual porta atacar nas VMs |
| 5 | **Instance template** | Qual template usar |
| 6 | **Autoscaling + condições** | Habilitar/desabilitar; se sim, qual métrica e thresholds |
| 7 | **Health check** (recomendado) | Quais verificações decidem "VM saudável"; usado por auto-healing e LB |

**Stateless vs Stateful (decisão #1) — entender:**
- **Stateless MIG:** todas as VMs são intercambiáveis. Não importa qual cliente cai em qual VM. Reciclar uma VM é trivial. Use para front-end web, workers de fila, processamento de imagem. **Default na maioria dos casos.**
- **Stateful MIG:** algumas VMs têm **identidade preservada** — disco persistente atribuído a uma instância específica, nome estável, IP estável. Reciclar não é trivial porque a VM **tem estado**. Use para apps legadas que esperam um disco em um caminho fixo, bancos não-gerenciados, qualquer coisa que não foi escrita pra ser cattle.
- Stateful é a fuga de emergência — quando você não consegue tornar a app stateless, ainda dá pra ter os benefícios de MIG (auto-healing, rolling update controlado) preservando estado por VM.

**Port name mapping (decisão #4) — por que existe:**
O LB precisa saber em qual **porta** das VMs ele deve mandar o tráfego. Em vez de hardcodear 8080 em vários lugares, você dá um **nome** ("http") para a porta no MIG, e o LB referencia o nome. Se um dia migrar de 8080 para 9000, muda só no MIG.

```bash
# Mapeia "http" → 8080 nas VMs do MIG
gcloud compute instance-groups managed set-named-ports meu-mig \
  --named-ports=http:8080 --region=us-central1
```

**Health check (decisão #7) — papel duplo:**
O **mesmo** health check (ou um dedicado) serve para:
1. **Auto-healing** — MIG recria a VM se ela falhar o check repetidas vezes.
2. **Load balancing** — LB para de mandar tráfego para a VM enquanto ela não responde ao check.

Pegadinha: você pode configurar **dois health checks diferentes**, um para healing (mais conservador, evita reciclar à toa) e outro para LB (mais agressivo, tira de circulação rápido). Em prova, basta saber que health check é o mecanismo comum aos dois.

---

#### Armadilha de prova — MIGs

1. **"Auto-healing recria quando você deleta a VM"** → depende. Se deletou via comando do MIG (resize, abandon), **não recria**. Se deletou manualmente fora do MIG, **recria**.
2. **"Recriada com mesmo nome e mesmo template"** — sim, ambos. Isso é o que dá continuidade ao monitoring e ao DNS interno.
3. **Zonal MIG não dá HA por si só** — se a zona cair, app cai. Use regional MIG.
4. **Instance template é imutável** — não dá pra "editar" um template. Pra mudar config, cria **novo template** e roda **rolling update** apontando para ele. Em prova, "edit the instance template" é distrator.
5. **Stateless ≠ sem disco persistente** — pode ter disco persistente no template, mas se a VM for substituída, o disco também é (boot disk efêmero ao longo das reciclagens). Stateful é o oposto: disco **preservado** por VM específica.

#### Autoscaling — como funciona

**Autoscaling** = adicionar/remover VMs do MIG automaticamente com base na **carga medida**, sem você intervir.

**Os dois objetivos simultâneos:**
1. **Absorver picos graciosamente** — quando a carga sobe, novas VMs aparecem antes do app degradar.
2. **Cortar custo no vale** — quando a carga cai, VMs são desligadas e você para de pagar por elas.

**O modelo é declarativo:**
- Você define a **policy** (qual métrica vigiar + qual o target).
- O **autoscaler** faz o trabalho — mede, compara com o target, e cria/destrói VMs sem você precisar fazer nada.

---

##### As 4 categorias de policy de autoscaling

| Categoria | Como funciona | Exemplo de uso |
|---|---|---|
| **CPU utilization** | "Mantém CPU média do grupo abaixo de X%" — métrica nativa, mais comum | Backend web normal: target 60-75% de CPU |
| **Load balancing capacity** | "Mantém o backend service abaixo de X% da capacidade serving" — usa a métrica do LB (requests/s, max RPS por VM, max utilization) | App atrás de Application LB com tráfego HTTP irregular |
| **Monitoring metrics** | Qualquer métrica do **Cloud Monitoring** — incluindo métricas custom da sua app | Memory usage, queue depth interna, latência p95 |
| **Queue-based (Pub/Sub) ou Schedule** | Tamanho de fila Pub/Sub, **ou** janela de horário/duração/recorrência fixa | Worker que processa fila: escala com mensagens pendentes. **Schedule:** sobe 20 VMs todo dia 8h–18h para batch noturno previsível |

**Análise do exemplo do slide (importante de entender):**

> "2 VMs em 100% e 85% de CPU. Target CPU utilization = 75%. Autoscaler **adiciona** outra instância."

- Média atual do grupo = (100 + 85) / 2 = **92.5%** → acima do target 75%.
- Adicionar 1 VM espalha a carga, baixando a média para o alvo.
- O mecanismo é: **media de carga sobre N+1 instâncias estimada < target?** Se sim, sobe pra N+1.

**E na descida:**
- Se a carga total cair muito, autoscaler **remove** instâncias — mas **só enquanto a média continuar abaixo do target após remover**. Ele não remove se isso fosse fazer a média estourar o target imediatamente.

**Armadilha de prova:**
1. **Target é uma média do grupo, não um teto por VM** — pode existir VM em 100% se a média do grupo continua abaixo do target. O autoscaler não promete "nenhuma VM passa de X%"; promete "a média fica abaixo de X%".
2. **Cooldown period** (não mencionado no vídeo mas presente na prova) — após cada decisão de scale, o autoscaler espera um intervalo antes de medir de novo, pra não oscilar (anti-flapping).
3. **Min/max instances** são limites obrigatórios — define o piso e o teto de quanto o MIG pode encolher ou crescer. Sem isso, escalaria infinitamente.

---

##### Monitorar a utilização para calibrar a policy

- Console: clica no MIG **ou** numa VM individual → aba de métricas.
- **Default:** CPU utilization da **última 1h**.
- Pode trocar timeframe (1h → 6h → 24h → 7d → custom) e ver outras métricas: **disk** I/O, **network** in/out.
- **Cloud Monitoring** estende isso: alertas (notificação via email, Slack, PagerDuty) quando alguma métrica passa de threshold. **Usar antes de criar a policy** — você precisa **medir** primeiro pra saber qual target faz sentido.

**Fluxo recomendado (cenário real):**
1. Sobe o MIG sem autoscaling (size fixo).
2. Roda carga real / sintética e olha os gráficos de CPU/disco/rede em diferentes horários.
3. Identifica o ponto onde a latência da app começa a degradar (ex.: CPU > 80%).
4. Define target = ponto degradação **menos folga** (ex.: 65%).
5. Liga autoscaling com esse target.

#### Health checks — anatomia detalhada

**Health check** = mecanismo que decide se uma VM do MIG está **saudável**. Determina se ela:
- Continua recebendo tráfego do **LB**.
- É **recriada** pelo auto-healing do MIG.

**Analogia com Cloud Monitoring uptime check:** estrutura parecida (protocolo + porta + critério), só que o resultado é consumido pelo MIG/LB, não por alertas. Você pode até reutilizar a mesma lógica.

---

##### Configuração — protocolo, porta, critérios

**Protocolo + porta:**
- **HTTP / HTTPS** — útil quando o backend é um web service. Pode bater num endpoint específico tipo `/healthz` e exigir resposta `200 OK`.
- **TCP / SSL** — pra workloads não-HTTP (banco, jogo). Só checa se o socket abre.

**Critérios de saúde — os 4 parâmetros (decora):**

| Parâmetro | O que define | Exemplo |
|---|---|---|
| **Check interval** | A cada quanto tempo o health check é executado | `5s` — checa a cada 5 segundos |
| **Timeout** | Quanto tempo esperar pela resposta antes de considerar essa tentativa como falha | `5s` — se não respondeu em 5s, falhou |
| **Healthy threshold** | Quantas verificações **consecutivas com sucesso** para considerar a VM como saudável | `2` — precisa passar 2 checks em seguida |
| **Unhealthy threshold** | Quantas verificações **consecutivas falhas** para considerar a VM como unhealthy | `2` — falha 2 checks → unhealthy |

**Análise do exemplo do slide:**

> "Health check teria que falhar **duas vezes** ao longo de **15 segundos** antes da instância ser considerada unhealthy."

Como chegamos em 15s? Se check interval = 5s e timeout = 5s, então:
- Tentativa 1: bate em t=0, falha em t=5 (timeout)
- Tentativa 2: bate em t=10, falha em t=15 (timeout)
- Unhealthy threshold = 2 → t=15: marca unhealthy.

**Por que os thresholds existem (não usar threshold = 1):**
- Um único timeout pode ser um glitch de rede transitório, GC pausa, latência momentânea.
- Threshold = 2 ou 3 evita reciclar/desviar tráfego à toa por falsos positivos.
- Trade-off: threshold alto = mais resiliente a falsos positivos, mas demora mais pra reagir a falha real. Padrão típico em prod: `unhealthy_threshold=2-3`, `healthy_threshold=2`.

**Armadilha de prova:**
- "Quanto tempo até a VM ser tirada de circulação?" = `unhealthy_threshold × check_interval` (aproximado, considerando timeout).
- Health check do **MIG** ≠ health check do **LB** — podem ser **diferentes objetos**, mesmo conceito. Você pode ter um mais agressivo para LB (tira tráfego rápido) e outro mais conservador para healing (não recicla à toa). Cobri isso no bloco anterior.

#### Stateful IPs em MIGs

**O problema que resolve:**
Por padrão, quando o MIG **reciclou** uma VM (auto-healing, rolling update, scale-in), a nova VM recebe um **IP diferente** (interno ou externo) — porque o IP era **ephemeral** (atribuído na criação, liberado ao destruir).

Isso quebra cenários onde **o IP precisa ser estável**:
- App **depende de IP estático** após atribuído (ex.: licença vinculada a IP, firewall em outro sistema com allowlist de IPs).
- **Configuração da app** referencia IPs específicos (ex.: peer-to-peer com IPs hardcoded).
- **Outras apps acessam via IP dedicado** — clientes externos com a URL apontando para `35.X.Y.Z`.
- **Migração de workload existente** — você quer trazer a app sem refazer config de rede (DNS, firewall, monitoring).

---

##### Como funciona

**Stateful IP policy no MIG** garante que IPs sejam **preservados** através de eventos de:
- **Auto-healing** (VM crashou e foi recriada)
- **Update** (rolling update para novo template)
- **Recreation** (você forçou recriação manualmente)

**Granularidade:**
- Funciona para **IPv4 interno** e **IPv4 externo** (ambos podem ser preservados).
- Cada VM do MIG pode ter **seu próprio IP estável** — você atribui automaticamente (autoscaler escolhe IPs livres) ou manualmente (você diz qual IP para qual instância).

---

##### Operações de stateful policy num MIG existente

| Operação | Efeito |
|---|---|
| **Configurar IPs como stateful para todas as instâncias atuais E futuras** | Promove os IPs **ephemeral** das VMs existentes para **static** (reserva no projeto), e garante que futuras VMs do MIG também tenham IP estável |
| **Atualizar config stateful existente** | Mudar de "automático" para "manual", trocar mapeamento VM → IP, etc. |

**Promoção ephemeral → static é importante de entender:**
- IP ephemeral: vive enquanto a VM vive. VM morre, IP volta pro pool.
- IP static: reservado no seu projeto (você paga por reserva quando **não está em uso**), permanente até você liberar.
- Stateful policy faz a promoção **transparentemente** — você não precisa criar manualmente cada IP static. O MIG faz isso para todas as VMs do grupo de uma vez.

**Pegadinha de custo:**
- IPs **externos** static **ociosos** (reservados mas não anexados a VM) custam ~$0.01/h cada. Em um MIG com 50 VMs externas e stateful policy, se você diminuir o grupo para 10, vai ter 40 IPs reservados parados sendo cobrados.

---

##### Cenários reais onde stateful IP é certo

1. **Banco legado em VM** com licença atrelada a IP — auto-healing não pode mudar o IP, ou a licença para de funcionar.
2. **Cluster Kafka/Zookeeper** com peers descobertos via IPs estáticos no `server.properties`.
3. **Migração lift-and-shift** de on-premises onde firewalls externos já têm allowlist de IPs específicos — manter os mesmos IPs após migração economiza semanas de coordenação com a equipe de rede.
4. **App que se anuncia via IP** (raro hoje com DNS, mas existe em sistemas legados).

---

#### Armadilha de prova — autoscaling + health check + stateful IP

1. **Autoscaling olha a média do grupo**, não cada VM individual. Resposta certa para "uma VM em 100% deveria triggar scale" → não necessariamente; depende da **média**.
2. **As 4 policies de autoscaling:** CPU, LB capacity, Monitoring metric, queue/schedule. Decora porque "memória" sozinha não é uma das 4 — ela só entra **via** Cloud Monitoring metric.
3. **Health check com 4 parâmetros:** check interval, timeout, healthy threshold, unhealthy threshold. Saber para que serve cada um — prova adora trocar nomes.
4. **Stateful IP ≠ Stateful MIG.** São coisas **diferentes**:
   - **Stateful MIG** preserva **identidade da VM** (nome, disco, IP). Configuração no nível do grupo.
   - **Stateful IP policy** preserva **só o IP** entre reciclagens, mesmo num MIG stateless. É uma config independente que você adiciona.
   - Pode ter MIG stateless **com** stateful IP — preserva IP mas não disco.
5. **Stateful IP funciona para IPv4 interno E externo** — não é só externo.

### Bloco "Application Load Balancer — arquitetura detalhada"

> Esse vídeo destrincha o **Application LB (HTTPS)**: onde opera, quais modos existem, qual é a **stack de componentes** (forwarding rule → target proxy → URL map → backend service → backend → instance group) e como **balancing mode + capacity scaler** governam o uso dos backends.

---

#### O que é (e onde opera)

**Application LB** = balanceador de carga **Layer 7 (Application)** do OSI. Ele **abre o pacote** e olha o conteúdo HTTP — URL, headers, cookies — para tomar decisões de roteamento. Diferente do Network LB (Layer 4), que só conhece IP + porta.

**Para que tipo de tráfego:** apenas **HTTP, HTTPS, HTTP/2, gRPC**. Se a sua aplicação não é HTTP, este balanceador não te serve.

**Backends suportados (a lista é mais ampla do que parece):**

| Backend | Caso típico |
|---|---|
| **Compute Engine** (instance group) | VMs rodando seu app server |
| **Google Kubernetes Engine (GKE)** | Pods expostos via Ingress |
| **Cloud Storage bucket** | Conteúdo estático (imagens, JS, HTML) — **só no Global external** |
| **Cloud Run** | Serverless containers HTTP |
| **External backends via internet** | Servidor on-premises ou em outra cloud, acessível pela internet |
| **External backends via conectividade híbrida** | Servidor on-premises atrás de VPN/Interconnect |

Esses dois últimos são importantes pra prova — Application LB consegue ter como backend **algo fora do GCP**, o que destrava casos de migração gradual ou multi-cloud.

---

#### Deployment modes — External vs Internal

| Modo | Onde fica o IP | Quem alcança | Quando usar |
|---|---|---|---|
| **External** | Internet pública | Qualquer um na web | Site público, API pública |
| **Internal** | IP privado dentro da VPC | Só recursos na mesma rede (VMs, GKE pods, on-prem via VPN) | API entre microsserviços, dashboard interno |

**O vídeo foca em External agora; Internal vem depois no módulo.**

---

#### Implementação — GFEs vs Envoy

Aqui entra um detalhe técnico do GCP que cai em prova: **o que faz o load balancing fisicamente?**

**Duas tecnologias diferentes** rodam por baixo dependendo da variante:

##### Google Front Ends (GFEs)
- **Quem usa:** Global external Application LB **e** Classic Application LB.
- **O que são:** **front-ends distribuídos globalmente** na borda da rede do Google (em centenas de PoPs no mundo).
- **Como funcionam:** trabalham juntos via a **rede global e o control plane** do Google. Cliente conecta no PoP mais próximo (anycast IP). GFE termina HTTPS lá mesmo, perto do usuário.
- **Vantagem (Premium tier):** **multi-region load balancing** — direciona automaticamente para o **backend saudável mais próximo que tem capacidade**, atravessando a rede privada do Google em vez da internet pública.
- **Por que importa:** termina o HTTPS o mais perto possível do usuário → menor latência percebida. O tráfego entre GFE e backend trafega pela backbone do Google, não pela internet.

##### Envoy proxy (open source)
- **Quem usa:** Global external Application LB **e** Regional external Application LB.
- **O que é:** o proxy **Envoy** (mesmo projeto open source usado em Istio, AWS App Mesh, etc.) operado como **managed proxy** pelo Google.
- **Vantagem:** **advanced traffic management** — features mais ricas como traffic splitting (canary), retries, circuit breaking, header transformations, rewriting de URL etc.

**Observação importante:** o **Global external Application LB** moderno **suporta os dois** (GFE para terminação edge + Envoy para advanced features). O **Classic** só GFE. O **Regional** só Envoy.

##### Os 3 modos resumidos

| Modo | Implementação | Escopo de IP | Use case |
|---|---|---|---|
| **Global external** | GFE + Envoy | IP global anycast | App mundial com features avançadas |
| **Classic external** | Só GFE | IP global anycast | Variante legada — features mais simples, ainda existe por compatibilidade |
| **Regional external** | Só Envoy | IP regional | Carga concentrada numa região, precisa de features Envoy |

**Armadilha de prova:**
- Se um cenário fala "multi-region failover automático para backend mais próximo" → **Global**, não Regional.
- Se um cenário fala "controle fino de tráfego para canary deployment" → **Envoy-based** (Global ou Regional, não Classic).

---

#### Anatomia — a stack de componentes

A grande sacada do Application LB do GCP é que ele **não é um objeto único**. É uma **stack de 5 componentes** que você compõe. Cada um tem sua função e seu CRUD próprio. Entender essa stack é **obrigatório para prova** — eles adoram perguntar "qual recurso faz X?".

```
Cliente
  │
  ▼ HTTP(S) request
External Forwarding Rule  ← define IP externo + porta + aponta para o proxy
  │
  ▼
Target HTTP(S) Proxy      ← recebe a request, faz terminação TLS, consulta o URL Map
  │
  ▼ (URL Map decide qual backend service)
URL Map                   ← roteamento por caminho/host (ex.: /api → backend-api, /static → backend-bucket)
  │
  ▼
Backend Service           ← contém health check, session affinity, timeout, lista de backends
  │
  ▼
Backend                   ← instance group + balancing mode + capacity scaler
  │
  ▼
Instance Group            ← MIG (com/sem autoscaling) ou unmanaged
  │
  ▼
VMs (Compute Engine)      ← onde a request finalmente é processada
```

---

##### 1. External Forwarding Rule

**O quê:** ponto de entrada. Especifica:
- **IP externo** (anycast global ou regional, dependendo do modo)
- **Porta** (443 para HTTPS, 80 para HTTP, ou qualquer outra)
- **Target proxy** para onde encaminhar

**Analogia:** é como a entrada do prédio — você fala "endereço Rua X, número 100" e a rua leva ao prédio. A forwarding rule é o endereço público.

**Por que existe separado:** o mesmo target proxy pode ser apontado por **múltiplas** forwarding rules (ex.: uma para porta 80 que redireciona pra 443, outra para 443 direto). Você reusa a lógica e só muda o IP/porta de entrada.

---

##### 2. Target HTTP(S) Proxy

**O quê:** o coração L7. Recebe a request do cliente, e:
- **Termina TLS/SSL** (se for HTTPS) — descriptografa usando **SSL certificates** anexados ao proxy.
- **Avalia a request** consultando o **URL Map** para decidir pra onde mandar.
- **Re-emite** a request para o backend service escolhido.

**Os certificados SSL ficam aqui**, não no backend service. Você anexa um ou mais certs ao target HTTPS proxy. Isso permite **um único proxy** servir múltiplos domínios via SNI.

---

##### 3. URL Map

**O quê:** tabela de roteamento L7. Define **regras** tipo:

```
host = api.exemplo.com  →  backend-service-api
host = www.exemplo.com, path = /static/*  →  backend-bucket-static
host = www.exemplo.com, path = /*           →  backend-service-web
```

**Por que isso é poderoso:** com **um único IP público + um único certificado**, você roteia para múltiplos serviços diferentes baseado no host e na URL. Isso é o que torna possível "um site que mistura Cloud Run, Cloud Storage e VMs sob o mesmo domínio".

---

##### 4. Backend Service

**O quê:** o objeto que **agrupa backends** e define as regras de **como distribuir** entre eles. Contém:

| Atributo | Função |
|---|---|
| **Health check** | Polla os backends em intervalos. **Healthy** → recebe tráfego; **Unhealthy** → não recebe |
| **Session affinity** | Override do round-robin default. Tenta mandar **o mesmo cliente para a mesma VM** |
| **Timeout** | Default **30 segundos**. Quanto esperar pelo backend antes de marcar a request como falha. **Fixed timeout, não idle.** Se a request demora mais que isso, falha |
| **One or more backends** | Lista de backends (cada um aponta para um instance group) |

**Algoritmo de distribuição default: round-robin.**
- Sem session affinity, requests são distribuídas em ordem entre as VMs disponíveis.
- Com session affinity (por cookie, IP de cliente, header), o LB tenta manter o cliente "grudado" numa VM específica.

**Sobre o timeout — pegadinha de prova:**
- "Fixed, não idle" significa: **conta o tempo total da request**, do início até a resposta completa. Não é "tempo sem receber nada". 
- Se sua app tem uploads grandes que demoram 45s, **default de 30s vai falhar todos eles**. Você precisa aumentar.
- Se você quer **streaming SSE** ou **WebSocket-like** que duram horas, esse timeout é **inadequado** — você teria que setar valor enorme (ex.: 86400 = 24h), mas idealmente usaria outro tipo de LB para casos extremos.

**Sobre o health check do backend service — relação com auto-healing:**
- O mesmo conceito do bloco anterior (4 parâmetros: check interval, timeout, healthy/unhealthy threshold).
- Health check do **backend service** é o que controla quem recebe **tráfego do LB**.
- Health check do **MIG** é o que controla **auto-healing** (recriação de VM).
- Podem ser o **mesmo objeto** ou **dois objetos diferentes** — escolha sua.

---

##### 5. Backend (dentro do backend service)

Cada backend **aponta para um instance group** (não para VMs individuais). Tem 3 campos:

| Campo | O que faz |
|---|---|
| **Instance group** | MIG (com/sem autoscaling) **ou** unmanaged instance group — o conjunto de VMs |
| **Balancing mode** | Como decidir que esse backend está **at full usage** (cheio) — duas opções: **CPU utilization** ou **Requests per Second (RPS)** |
| **Capacity scaler** | Controle adicional que **interage com o balancing mode** (explicação abaixo) |

**Cross-region failover automático (importante):**
- Se **todos** os backends de uma região atingem capacidade máxima, o Application LB **automaticamente** roteia novas requests para a **região mais próxima que ainda tem capacidade**.
- Esse é o grande valor do **Global** external Application LB: failover regional gratuito, sem configuração extra.

---

#### Balancing mode + Capacity scaler — entender o jogo

Esse par é a parte mais confusa do LB do GCP. Os dois trabalham **juntos** para definir **quanto cada backend deve receber** antes de o LB considerar mandar pra outro.

##### Balancing mode

Define **o critério de "cheio"** — quando o LB decide que o backend não aguenta mais:

- **CPU utilization** — backend considerado "cheio" quando média de CPU do grupo atinge X% (ex.: 80%).
- **RPS (Requests Per Second)** — backend considerado "cheio" quando atinge N requests/s por VM (ex.: 100 RPS por VM).

##### Capacity scaler

**Multiplicador** que diz **qual fração** da capacidade definida pelo balancing mode você quer usar **agora**.

- Default = **1.0** (= 100%) → usar tudo que o balancing mode permite.
- 0.5 → usar **metade**.
- 0.0 → **drenar** completamente esse backend (não manda mais nada, mas mantém configurado — útil para manutenção).

**Os 2 exemplos do vídeo, esmiuçados:**

**Exemplo A — "operar a no máximo 80% CPU"**
- Balancing mode = **80% CPU utilization**
- Capacity scaler = **100%** (1.0)
- Resultado: backend é considerado cheio quando CPU média = 80%. Aceita tráfego até esse ponto.

**Exemplo B — "cortar utilização pela metade"**
- Balancing mode = **80% CPU utilization** (não muda)
- Capacity scaler = **50%** (0.5)
- Resultado: backend é considerado cheio quando CPU média = 80% × 50% = **40%**. Acima disso, novo tráfego vai para outros backends/regiões.

**Por que separar em dois conceitos:**
- O **balancing mode** define a **capacidade absoluta** do backend ("esta máquina aguenta isso").
- O **capacity scaler** é a **alavanca operacional do dia-a-dia** ("quero usar X% dessa capacidade hoje, sem mexer na config base"). É o botão para **drenar gradualmente**, fazer **canary**, ou reduzir tráfego para uma região durante manutenção.

**Armadilha de prova:**
- Capacity scaler = 0.0 **não desabilita o backend**, só **drena**. Ele segue health-checkeado e configurado.
- Capacity scaler é por **backend**, não global. Você pode escalar diferente cada região.

---

#### Propagação de mudanças

> "Mudanças no backend service não são instantâneas — pode levar **vários minutos** pra propagar pela rede."

**Por que demora:**
- O Application LB do GCP é **globalmente distribuído** (centenas de GFEs em PoPs no mundo). Quando você muda config, o control plane precisa propagar para todos.
- Mudanças típicas (adicionar backend, mudar capacity scaler, trocar URL map) levam de **30 segundos a alguns minutos** para refletir em todos os pontos do mundo.

**Implicação prática:**
- Não fique surpreso se um teste imediato após uma mudança ainda mostrar comportamento antigo.
- Em prova: pergunta tipo "o usuário mudou a URL map mas o tráfego ainda vai pro backend antigo, o que pode ser?" → **propagação ainda em andamento**.
- Em produção: não pendure deploys críticos esperando que mudanças sejam instantâneas — sempre janela de tolerância.

---

#### Armadilha de prova — Application LB

1. **Forwarding rule, target proxy, URL map, backend service, backend são objetos separados.** Cada um tem seu `gcloud` próprio (`gcloud compute forwarding-rules`, `gcloud compute target-https-proxies`, `gcloud compute url-maps`, `gcloud compute backend-services`). Prova pergunta "qual recurso faz X?" → tem que saber qual.
2. **SSL certificate fica no target HTTPS proxy**, não no backend service.
3. **URL map é o roteamento L7** — qualquer cenário "rote `/api` pra X e `/static` pra Y" envolve URL map.
4. **Timeout default = 30s e é fixed (não idle).**
5. **Backend bucket** (Cloud Storage como backend) só funciona no **Global external** Application LB, não nos outros modos.
6. **Round-robin é o default**; session affinity sobrescreve.
7. **Cross-region failover é automático** no Global external — não precisa configurar nada além do mesmo backend service apontando para backends em múltiplas regiões.
8. **Capacity scaler = 0.0 drena, não desabilita.**
9. **Backend aponta para instance group**, não para VM individual.
10. **Propagação leva minutos** — não é instantânea.

### Bloco "Network Load Balancers — Proxy e Passthrough"

> Este vídeo detalha os **Network Load Balancers (Layer 4)**: os dois tipos (Proxy vs Passthrough), modos de deploy, arquitetura e a diferença entre Backend Service e Target Pool.

---

#### Visão geral — Layer 4

**Network Load Balancers (NLB)** operam na **camada 4 (Transport)** do OSI. Conhecem apenas **IP + porta + protocolo** — não abrem o pacote para ver se é HTTP. Suportam TCP, UDP, ESP, GRE, ICMP e ICMPv6.

Existem dois tipos fundamentalmente diferentes:

| | **Proxy NLB** | **Passthrough NLB** |
|---|---|---|
| **É proxy?** | Sim — reverse proxy | Não — passa pacote direto |
| **Termina a conexão?** | Sim — no LB | Não — no backend |
| **Preserva IP do cliente?** | Não | **Sim** |
| **Resposta do backend** | Volta pelo LB | **Direto ao cliente (DSR)** |
| **Escopo** | Global ou Regional | **Regional apenas** |
| **Protocolos** | TCP (com ou sem SSL) | TCP, UDP, ESP, GRE, ICMP, ICMPv6 |
| **Quando usar** | Backends on-prem/multi-cloud, controle avançado | Preservar IP, DSR, multi-protocolo, baixa latência |

---

#### Proxy Network Load Balancer

**Como funciona:** reverse proxy Layer 4. O tráfego é **terminado** no LB e uma **nova conexão** é estabelecida ao backend mais próximo — igual ao Application LB, mas operando em L4 (TCP puro, sem olhar dentro do pacote HTTP).

**Tipos de target proxy:**
- **Target TCP Proxy** — para tráfego TCP puro.
- **Target SSL Proxy** — para tráfego TCP com SSL. O LB pode terminar o SSL (e reencriptar ao falar com o backend — recomendado usar SSL no trecho LB → backend também).

**Fluxo (exemplo TCP Proxy):**
```
Cliente (Iowa + Boston)
  │
  ▼ TCP
External Proxy NLB  ←  termina a conexão aqui
  │
  ▼ nova conexão TCP/SSL
Backend mais próximo (us-central para Iowa, us-east para Boston)
```

**Modos de deploy:**
- **Global** — distribui para backend mais próximo entre regiões.
- **Regional** — restringe ao backends numa região.
- **Classic** — variante legada.

**Implementação:** GFEs ou Envoy proxies (mesma base do Application LB).

**Armadilha de prova:** para HTTP/HTTPS, **sempre use Application LB** — o Proxy NLB não entende HTTP, só TCP/SSL.

---

#### Passthrough Network Load Balancer

**Como funciona:** **não é proxy**. O pacote chega ao backend com os IPs e portas **originais preservados**. A resposta vai **direto do backend para o cliente**, sem passar pelo LB — isso é chamado de **Direct Server Return (DSR)**.

**Implementação:** Andromeda virtual networking + **Google Maglev**.

**Características:**

| Característica | Detalhe |
|---|---|
| Escopo | **Regional apenas** |
| IP do cliente preservado | **Sim** |
| Resposta | **DSR — bypass do LB** |
| Protocolos | TCP, UDP, ESP, GRE, ICMP, ICMPv6 |
| Terminação da conexão | No **backend** |

**Quem alcança:**
- Qualquer cliente na internet (External).
- VMs no GCP com IP externo ou via Cloud NAT.
- Independente de Network Service Tier.

---

#### Backend: Backend Service vs Target Pool

O Passthrough NLB externo suporta dois modelos de backend. **Recomendação do Google: usar Backend Service** para novas implantações.

| | **Backend Service (recomendado)** | **Target Pool (legado)** |
|---|---|---|
| IPv4 + IPv6 | **Sim** | IPv4 apenas |
| Protocolos | TCP, UDP, ESP, GRE, ICMP, ICMPv6 | **TCP e UDP apenas** |
| Instance Groups | Managed e Unmanaged | Sim |
| Zonal NEGs (GCE_VM_IP) | Sim | Não |
| Health checks | Non-legacy (TCP, SSL, HTTP, HTTPS, HTTP/2) | **Apenas 1 health check por pool** |
| Failover policies | Sim | Não |
| Distribuição | Controles granulares | Hash de src IP:port + dst IP:port |
| Limite por projeto | — | **50 target pools** |

**O que é um Target Pool:**
- Grupo de instâncias que recebe tráfego de forwarding rules.
- Distribuição baseada em **hash de 4 tuplas** (src IP + src port + dst IP + dst port).
- Restrição: todas as instâncias devem estar na **mesma região** (igual ao Passthrough NLB).
- É possível migrar um target pool existente para usar backend service.

**Por que existe o limite de 50 target pools:** é um recurso legado com limitações estruturais; o backend service foi criado para superar essas restrições.

---

#### Quando usar cada NLB

| Situação | Use |
|---|---|
| Preservar IP original do cliente | **Passthrough NLB** |
| Direct Server Return (DSR) | **Passthrough NLB** |
| UDP, ESP, GRE, ICMP | **Passthrough NLB** |
| Backends on-premises ou multi-cloud | **Proxy NLB** |
| Controles avançados de tráfego | **Proxy NLB** |
| Global, backends mais próximos | **Proxy NLB** (global) |
| HTTP/S | **Application LB** — não NLB |

---

#### Armadilhas de prova — NLB

1. **Passthrough NLB é sempre regional** — Proxy NLB pode ser global/regional.
2. **DSR (Direct Server Return)** é exclusivo do Passthrough — a resposta não passa pelo LB.
3. **Target Pool é legado** — se a questão mencionar "migrar de target pool para backend service", é sobre modernizar um Passthrough NLB externo.
4. **Passthrough NLB não é proxy** — não termina conexão, não vê dentro do pacote.
5. **50 target pools por projeto** — limite hard do recurso legado; backend service não tem esse limite.
6. **Target Pool suporta apenas TCP e UDP** — para qualquer outro protocolo (ESP, GRE, ICMP), obrigatoriamente backend service.
7. **Health check do Target Pool** — só pode ter **1** por pool. Backend service suporta health checks mais ricos e flexíveis.

### Bloco "Choosing the right load balancer — decision tree" (encerramento da trilha)

> Bloco final da trilha de Load Balancing do curso **Elastic Google Cloud Infrastructure: Scaling and Automation**. Consolida tudo o que foi visto e responde à pergunta prática: **"qual LB eu uso para este cenário?"** — é o slide que mais cai em prova.

---

#### Como escolher — fluxo em 3 perguntas

A escolha do LB no GCP segue um **decision tree de 3 perguntas**, nessa ordem:

```
PERGUNTA 1: Qual o tipo de tráfego?
   ├── HTTP(S) com feature set rico (URL maps, SSL termination, session affinity)
   │       → Application Load Balancer
   │
   ├── TLS offload, TCP proxy, OU backends externos em múltiplas regiões
   │       → Proxy Network Load Balancer
   │
   └── Preservar IP de origem do cliente, evitar overhead de proxy,
       OU UDP/ESP/ICMP, OU expor o IP do cliente para a aplicação
           → Passthrough Network Load Balancer

PERGUNTA 2: O LB é externo (internet-facing) ou interno (dentro da VPC)?
   ├── External  → variante externa do tipo escolhido
   └── Internal  → variante interna do tipo escolhido

PERGUNTA 3: Os backends estão em múltiplas regiões (global) ou em uma só (regional)?
   ├── Global   → variante global
   └── Regional → variante regional
```

**Multiplicando isso dá ~6 variantes principais:**

| | Application LB | Proxy Network LB | Passthrough Network LB |
|---|---|---|---|
| **External global** | Global external Application LB | Global external proxy NLB | (não existe — passthrough é regional) |
| **External regional** | Regional external Application LB | Regional external proxy NLB | External passthrough NLB |
| **Internal regional** | Internal Application LB | Internal proxy NLB | Internal passthrough NLB |
| **Cross-region internal** | Cross-region internal Application LB | Cross-region internal proxy NLB | — |

> **Pegadinha:** **Passthrough é sempre regional.** Não existe passthrough global porque o modelo de passthrough preserva o IP de origem e entrega o pacote direto à VM — isso só funciona quando o backend está próximo (mesma região). Não dá pra ter anycast global passthrough.

---

#### Quando cada um — em uma frase (regra geral do curso)

| LB | Regra geral do slide |
|---|---|
| **Application LB** | Quando você precisa de um **feature set flexível** para apps com tráfego **HTTP(S)** |
| **Proxy NLB** | Para implementar **TLS offload**, **TCP proxy**, ou para **external load balancing para backends em múltiplas regiões** |
| **Passthrough NLB** | Para **preservar source IP do cliente**, evitar o **overhead de proxies**, suportar **protocolos adicionais (UDP, ESP, ICMP)**, ou se você precisa **expor IP do cliente para sua aplicação** |

**Exemplos colados nos use cases:**

- **App SaaS com `/api`, `/static`, `/admin` em backends diferentes** → Application LB (URL map resolve isso).
- **App que termina TLS no LB e fala TCP plain entre regiões** → Proxy NLB.
- **Servidor de jogo UDP, gateway VoIP (SIP+RTP), DNS authoritative** → Passthrough NLB (UDP não passa por proxy).
- **App legada que loga "IP do cliente" sem entender `X-Forwarded-For`** → Passthrough NLB (preserva IP real).
- **Cluster de banco TCP interno (Redis sentinel, Patroni)** → Internal passthrough NLB.

---

#### Refinando com external/internal e global/regional

Depois de escolher a categoria, **2 perguntas-filtro** reduzem para a variante específica:

| Pergunta | Opções | O que muda |
|---|---|---|
| **External vs Internal** | Internet pública vs IP privado dentro da VPC | Acessibilidade. Internal só serve recursos na mesma VPC (ou conectados via VPN/peering) |
| **Global vs Regional** | Backends espalhados no mundo vs backends em uma única região | Latência percebida pelo cliente + complexidade de roteamento + custo |

> **O slide explicitamente recomenda a tabela-resumo em vez do fluxograma** — porque a tabela cruza as 3 dimensões em um único olhar e é mais fácil de bater contra um cenário de prova.

---

#### Load-balancing scheme — o atributo que materializa a escolha

A escolha do decision tree vira um valor concreto chamado **load-balancing scheme**, presente em **dois recursos**:

- **Forwarding rule** (define o ponto de entrada)
- **Backend service** (define o grupo de backends)

**O que esse atributo indica:**
1. Se o LB pode ser usado para tráfego **interno ou externo**.
2. Se ele é **MANAGED** (gerenciado por GFE/Envoy) ou não.

**Os valores que aparecem na CLI:**

| `--load-balancing-scheme` | Significa | Implementado em |
|---|---|---|
| `EXTERNAL_MANAGED` | LB externo gerenciado — Application LB externos e Proxy NLB externos modernos | GFE / Envoy |
| `INTERNAL_MANAGED` | LB interno gerenciado — Application LB internos e Proxy NLB internos | Envoy |
| `EXTERNAL` | Legacy — Classic Application LB, External passthrough NLB | Maglev (data plane Google) |
| `INTERNAL` | Internal passthrough NLB | Andromeda (data plane da VPC) |

**O que MANAGED significa exatamente (frase exata do slide):**
- "MANAGED indica que o LB é implementado como **um serviço gerenciado** rodando em **Google Front Ends** ou em **Envoy proxy** (open source)."
- "Em um load-balancing scheme MANAGED, as requests são roteadas **para o GFE ou para o Envoy proxy**."
- O cliente **nunca conecta direto à VM**. Sempre passa por GFE ou Envoy quando o scheme é MANAGED.

**E em um scheme não-managed (passthrough)?**
- Não tem GFE/Envoy no caminho. O LB é apenas uma regra de roteamento implementada no Andromeda (data plane da rede).
- O pacote vai **direto do cliente para a VM**, sem terminação intermediária. **Por isso preserva o IP de origem.**

**Resumindo a divisão mental:**

```
MANAGED scheme       →  GFE / Envoy intercepta  →  conexão termina no LB
                       →  proxy mode             →  NÃO preserva IP do cliente

Não-MANAGED scheme   →  só regra de roteamento  →  pacote vai direto pra VM
                       →  passthrough mode       →  PRESERVA IP do cliente
```

#### Network Service Tiers — referência rápida

O slide encerra apontando para a documentação de **Network Service Tiers**. Em uma linha:

- **Premium Tier** (default para LBs globais): tráfego viaja pela **backbone privada do Google** desde o PoP mais próximo do cliente até a região do backend. Latência menor, qualidade superior, custo maior.
- **Standard Tier**: tráfego viaja pela **internet pública** até chegar à região do backend, e só ali entra na rede do Google. Mais barato, latência variável.

**Implicação direta para LBs:**
- LBs **globais** (Application LB Global, Proxy NLB global) **só funcionam em Premium Tier** — dependem da backbone do Google para anycast global e cross-region failover.
- LBs **regionais** podem funcionar em Premium **ou** Standard — você escolhe pelo trade-off latência vs custo.

#### Armadilha de prova — decision tree do LB

1. **A primeira pergunta é sempre "qual é o tráfego?"** — não comece por external vs internal. Se é UDP, já elimina Application LB e Proxy NLB de cara, antes de saber se é externo ou interno.
2. **Passthrough é sempre regional.** Não existe passthrough global.
3. **Backend bucket (Cloud Storage)** só serve no **Global external Application LB**. Nas outras variantes, esse tipo de backend não é opção.
4. **"Multi-region failover automático"** → Global Application LB (GFE em Premium Tier). Regional não tem essa feature out-of-the-box.
5. **"Preservar IP do cliente real, sem `X-Forwarded-For`"** → Passthrough NLB. Application LB e Proxy NLB **sempre** terminam a conexão e re-emitem (proxy mode), então o IP de origem que o backend vê é do LB, não do cliente.
6. **"TLS offload em conexão TCP que não é HTTP"** → Proxy NLB (TCP proxy mode). Não é Application LB (esse é só HTTP/HTTPS) nem Passthrough NLB (esse não termina TLS).
7. **MANAGED no scheme = GFE ou Envoy.** Se um cenário pergunta "qual tecnologia executa o LB?", a resposta depende: Application Global/Classic usa **GFE**; Application Regional usa **Envoy**; Application Global moderno usa **os dois juntos**.
8. **Premium Tier é obrigatório para LBs globais.** Se o cenário diz "Standard Tier" e oferece um LB global como opção → distrator.
9. **`EXTERNAL_MANAGED` ≠ `EXTERNAL`.** O segundo é **legacy** (Classic Application LB, passthrough NLB externo). O moderno é `EXTERNAL_MANAGED`. Em prova, comando `gcloud` com flag errada de scheme é distrator clássico.
10. **Forwarding rule e backend service têm que combinar no scheme.** Não dá pra ter forwarding rule `EXTERNAL_MANAGED` apontando para backend service `INTERNAL_MANAGED` — o `gcloud` reclama.

## Cenário real

**Onde isso aparece no trabalho de cloud engineer:**

- **App web SaaS com tráfego global:** External Application LB com URL maps roteando `/api/*` para Cloud Run, `/static/*` para Cloud Storage, `/app/*` para MIGs em 3 regiões. Cliente sempre cai no PoP mais próximo (anycast).
- **API interna entre microsserviços na mesma VPC:** Internal Application LB regional — não precisa atravessar internet, IP privado, latência baixa.
- **Servidor de jogo multiplayer (UDP):** External passthrough Network LB — preserva IP do cliente, latência mínima, não interpreta o protocolo do jogo.
- **Cluster de banco TCP (ex.: Redis sentinel):** Internal passthrough Network LB — distribui conexões TCP entre nós sem entender o protocolo.
- **Gateway TCP para backends on-premises via Interconnect:** External proxy NLB com target TCP proxy — tráfego terminado no GCP, nova conexão ao on-prem. Não precisa preservar IP do cliente porque o backend já espera conexão do GCP.
- **VoIP gateway com ESP/ICMP:** Passthrough NLB com backend service — único NLB que suporta protocolos além de TCP/UDP.

## Armadilhas de prova

Compilado das armadilhas espalhadas pelos blocos — visão rápida pré-prova:

**LB em si (genérico):**
- "Você precisa redimensionar seu load balancer" → distrator. No GCP, LB escala sozinho.
- "Configure alta disponibilidade do load balancer" → distrator. O LB já é HA por construção.
- Application LB ≠ Network LB pela camada, mas dentro de cada categoria existem variantes regional/global e internal/external que mudam o use case.

**Decision tree (escolhendo o LB):**
- **Tráfego antes de tudo.** UDP elimina Application e Proxy NLB sem precisar saber se é external ou internal.
- **Passthrough é sempre regional** — não existe passthrough global.
- **Backend bucket (Cloud Storage)** só no **Global external Application LB**.
- **Preservar IP de origem do cliente** → só Passthrough NLB. Application LB e Proxy NLB sempre re-emitem (proxy mode).
- **TLS offload em TCP não-HTTP** → Proxy NLB (não Application, não Passthrough).
- **Multi-region failover automático** → Global Application LB em Premium Tier.

**Scheme + implementação:**
- **MANAGED = GFE ou Envoy.** Não-MANAGED = passthrough direto pela rede.
- **`EXTERNAL_MANAGED` ≠ `EXTERNAL`** — o segundo é legacy.
- **LBs globais exigem Premium Tier.** Standard Tier não suporta LB global.
- Forwarding rule e backend service têm que ter o **mesmo scheme** — o `gcloud` rejeita mistura.

**MIG + autoscaling + health check:**
- Auto-healing só recria se a remoção **não foi comando do MIG**.
- Regional MIG é o **default recomendado** sobre zonal.
- Instance template é **imutável** — pra mudar config, novo template + rolling update.
- Autoscaling olha **média do grupo**, não cada VM.
- **4 policies de autoscaling:** CPU, LB capacity, Monitoring metric, queue/schedule.
- **Health check do MIG ≠ health check do LB** — podem ser objetos diferentes.
- **Stateful IP ≠ Stateful MIG** — preservar só IP é independente de preservar disco/nome.

**Application LB (stack):**
- **5 componentes separados** (forwarding rule, target proxy, URL map, backend service, backend).
- **SSL cert fica no target HTTPS proxy**, não no backend service.
- **Timeout default = 30s, fixed** (não idle).
- **Capacity scaler = 0.0 drena, não desabilita.**
- **Propagação de mudanças leva minutos** — não é instantânea.
