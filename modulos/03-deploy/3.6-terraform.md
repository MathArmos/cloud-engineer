## Bloco "Introdução — Automating the Deployment of GCP Infrastructure"
> Fonte: *Elastic Google Cloud Infrastructure: Scaling and Automation* — vídeo 1 (intro do módulo de automação)

### O ponto de partida: por que automatizar?
Já vimos um monte de serviços do Google Cloud (Compute, GKE, Cloud Run, VPC, LB...). Criar tudo isso na mão no Console **não escala**: ambientes de dev/test/prod precisam ser idênticos, recriáveis, versionáveis. A pergunta natural vira "**como automatizo a criação dessa infra?**" — e existem dois caminhos possíveis. O vídeo apresenta os dois e justifica por que o segundo (Terraform) ganha.

### Caminho 1 — Chamar a Cloud API direto do código (imperativo)
- **O quê:** escrever um programa (Python, Go, Node, etc.) que usa o SDK do Google Cloud para fazer chamadas à **Cloud API** criando recursos um a um (`compute.instances().insert(...)`, `compute.networks().insert(...)` etc.).
- **Funciona? Sim** — é poderoso, você consegue criar qualquer coisa que a API expõe.
- **Por que dói:** a **manutenibilidade da infra** vira refém da **qualidade do software**. Se o programa estiver mal escrito, sua infra também está.
- **Cenário do vídeo (importante):** imagine um programa com **dezenas de chamadas à API criando VMs**, espalhadas em funções/módulos diferentes. Quando precisar arrumar a definição de **uma** VM específica (mudar disco, mudar zona, mudar tag), você primeiro tem que **caçar qual das dezenas de chamadas criou aquela VM** — só depois consegue editar. Não dá pra "ler a infra" do código sem rastrear toda a lógica.
- **Software muda rápido** → cada deploy novo exige manutenção do código de infra, mais bugs, mais drift entre "o que tá no código" e "o que tá rodando no projeto GCP".

> **Analogia:** é como construir uma casa lendo 50 post-its grudados pela parede — cada um com uma instrução isolada ("furar parede X", "colocar tomada Y"). Funciona, mas quando o cliente pede pra mover uma janela, você precisa varrer todos os post-its pra entender o que muda.

### Caminho 2 — Terraform (declarativo, com camada de abstração)
- **O quê:** Terraform usa **templates altamente estruturados + arquivos de configuração** (`.tf`) que **documentam a infraestrutura num formato legível e compreensível** — você lê o arquivo e *vê* a infra que vai existir.
- **A grande sacada:** Terraform **esconde as chamadas reais à Cloud API**. Você não escreve `compute.instances().insert(...)` — você escreve `resource "google_compute_instance" "web" { ... }`. O Terraform cuida de traduzir isso nas chamadas certas.
- **Você foca em DEFINIR a infra**, não em escrever o código que a cria. Esse é o salto **imperativo → declarativo**: você diz **o quê** (estado final desejado), não **como** (sequência de chamadas).
- **Consequência prática:** os arquivos `.tf` viram a **fonte da verdade** da infra. Git, code review, diff entre ambientes, rollback — tudo de graça.

> **Analogia:** Terraform é a **planta arquitetônica da casa** num único documento. Quando o cliente pede pra mover a janela, você edita a planta e o builder reconstrói só o que mudou.

### O que esse módulo vai cobrir
1. **Terraform** para automatizar o deploy de infra GCP
2. **Google Cloud Marketplace** para lançar soluções de infra prontas (stack inteiro com 1 clique — WordPress, LAMP, soluções de parceiros etc.)
3. **Lab da seção:** usar Terraform para criar **VPC network + firewall rule + VM instances** (exatamente o que o vídeo prometeu — perfeito porque cobre os 3 pilares: rede, segurança, compute).

### Imperativo vs Declarativo — resumo de uma frase
| Modelo | Você escreve | Exemplo |
|--------|--------------|---------|
| **Imperativo** | Sequência de **ações** | Código chamando Cloud API: "crie VM A, depois crie VM B, depois conecte..." |
| **Declarativo** | **Estado final** desejado | Arquivo `.tf`: "devem existir VM A e VM B na rede X" — Terraform decide a ordem |

### Por que isso importa pra prova (e pra feira)
- A prova ACE pode perguntar **"qual a vantagem de IaC sobre scripts ou Console?"** — a resposta correta gravita em torno de **repetibilidade, versionamento, declarativo, idempotência**.
- Na feira, saber explicar **por que ninguém deveria criar infra com scripts ad-hoc chamando a API** é um diferencial — é o argumento que vende DevOps/SRE pra qualquer time que ainda tá no clique-clique.

## Bloco "Terraform: a ferramenta, HCL e o fluxo init→plan→apply"
> Fonte: *Elastic Google Cloud Infrastructure: Scaling and Automation* — vídeo 2 (Introduction to Terraform)

### Da escada: Console → Cloud Shell → Terraform
O vídeo posiciona Terraform como o **próximo passo natural** depois que você já domina os outros dois caminhos.

| Ferramenta | Quando usar (segundo o curso) |
|------------|-------------------------------|
| **Console (UI)** | Você é novo no serviço, quer explorar, prefere clicar |
| **Cloud Shell (`gcloud`)** | Já domina o serviço, quer rapidez via CLI, ações pontuais |
| **Terraform (IaC)** | Quer **provisionar e remover infra inteira sob demanda**, versionada e repetível |

### IaC — benefícios (refinando o que vimos no vídeo 1)
- **Provisionamento on-demand** — sobe/derruba ambientes inteiros com 1 comando
- **Integra com CI/CD** — encaixa direto num pipeline de continuous deployment
- **Complexidade do deploy mora no código** — não na cabeça de quem clicou ontem
- **Flexibilidade** — requisitos mudaram? edita o `.tf`, aplica, pronto. Mudanças concentradas **num lugar só**
- **Dev/test = prod** — replica o ambiente de produção fielmente, e **deleta quando não tá em uso** (economia gigante)

### Ferramentas IaC suportadas pelo GCP (oficialmente)
Esses 5 são citados nominalmente no vídeo (decora a lista — pode cair em prova de "qual NÃO é uma ferramenta IaC suportada pelo GCP"):

| Ferramenta | Tipo |
|------------|------|
| **Terraform** | Declarativo, multi-cloud — **foco deste curso** |
| **Chef** | Configuration management (procedural) |
| **Puppet** | Configuration management (declarativo) |
| **Ansible** | Configuration management (procedural via YAML) |
| **Packer** | Construção de imagens (não provisiona infra rodando) |

> **Armadilha:** Packer **não** cria recursos rodando — ele constrói *imagens* (AMIs, GCP custom images). Já Chef/Puppet/Ansible são mais voltados a **configurar máquinas existentes** do que a provisionar a infra do zero. Terraform é o único que **provisiona infra** de ponta a ponta.

### Por que Terraform especificamente
- **Declarativo** — você descreve o estado final, o Terraform descobre os passos
- **Deploy em paralelo** ⚠️ — diferente do `gcloud` no Cloud Shell, que é **sequencial**. Terraform monta um grafo de dependências e cria tudo que pode em paralelo, respeitando só as dependências reais.
- **Usa as APIs subjacentes do GCP** — não é uma camada mágica; por baixo, são as mesmas chamadas que o `gcloud` faria
- **Multi-cloud** — funciona com AWS, Azure, on-prem, etc. (mesma sintaxe, providers diferentes)
- **Já vem instalado no Cloud Shell** — não precisa instalar nada pra começar

### HCL — HashiCorp Configuration Language
Terraform usa **HCL**, uma linguagem de configuração própria. A sintaxe tem **3 elementos**:

| Elemento | O que é | Exemplo |
|----------|---------|---------|
| **Block** | Representa um objeto. Pode ter **0 ou mais labels** e um **body** com argumentos e blocos aninhados | `resource "google_compute_instance" "web" { ... }` (2 labels: tipo + nome) |
| **Argument** | Atribui um **valor a um nome** dentro do body | `machine_type = "e2-medium"` |
| **Expression** | Representa um **valor** que pode ser atribuído a um identificador | `"us-central1-a"`, `var.region`, `google_compute_network.vpc.self_link` |

### Anatomia de um `main.tf` — os 3 tipos de bloco que sempre aparecem
O vídeo destaca **3 blocos** que estruturam quase todo arquivo Terraform pra GCP:

```hcl
# 1. PROVIDER — quem é a cloud + config global (região, projeto, credenciais)
provider "google" {
  project = "meu-projeto"
  region  = "us-central1"
}

# 2. RESOURCE — o recurso que você quer criar
resource "google_compute_instance" "web" {
  name         = "web-vm"
  machine_type = "e2-medium"
  zone         = "us-central1-a"
  # ... boot_disk, network_interface etc
}

# 3. OUTPUT — variáveis de saída do módulo
output "instance_ip" {
  value = google_compute_instance.web.network_interface[0].access_config[0].nat_ip
}
```

| Bloco | Pra que serve |
|-------|---------------|
| **`provider`** | Diz **qual cloud** vai receber os recursos + configs globais (projeto, região, credencial) |
| **`resource`** | Declara **um recurso de infra** (VM, network, firewall, bucket...). Cada um vira uma chamada de API |
| **`output`** | Expõe **valores computados** após o apply (IP da VM, URL do LB, nome do bucket gerado) — útil pra encadear módulos ou ler depois |

> Existem outros tipos importantes que vão aparecer mais pra frente: `variable` (entrada), `data` (consulta de recurso existente), `module` (reusar bloco), `terraform` (config do próprio Terraform). Esses 3 acima são o **mínimo funcional**.

### Exemplo guiado do vídeo: VPC auto-mode + firewall HTTP
O vídeo mostra um `main.tf` que cria **uma VPC auto-mode + uma firewall rule** liberando TCP 80 e 8080. Esquema do arquivo:

```hcl
provider "google" {
  project = "meu-projeto"
  region  = "us-central1"
}

resource "google_compute_network" "vpc_network" {
  name                    = "terraform-network"
  auto_create_subnetworks = true   # subnet automática em CADA região
  mtu                     = 1460   # MTU default do GCP
}

resource "google_compute_firewall" "allow_http" {
  name    = "allow-http"
  network = google_compute_network.vpc_network.name  # ← referência ao recurso acima

  allow {
    protocol = "tcp"
    ports    = ["80", "8080"]
  }
}
```

**Pontos a observar:**
- `auto_create_subnetworks = true` reproduz o comportamento da "**auto mode VPC**" que vimos em 3.5 — uma subnet por região, ranges fixos do GCP
- `mtu = 1460` é o MTU default do GCP (já estudado em 3.5)
- `network = google_compute_network.vpc_network.name` é uma **referência cruzada** — Terraform usa isso pra **montar o grafo de dependências** e saber que a firewall só pode ser criada depois da rede

### O fluxo `init → plan → apply` (agora com detalhes do vídeo)

#### `terraform init`
- Roda **na pasta** que contém o `main.tf`
- Garante que o **plugin do provider Google** está **baixado e instalado** num subdiretório (`.terraform/`) do working directory
- Cria também **vários arquivos de bookkeeping** (lock file, etc.)
- Mensagem visível: **"Initializing provider plugins"**
- Terraform detecta que você tá num projeto Google e baixa os recursos do provider Google

#### `terraform plan`
- Faz um **refresh** primeiro (a menos que você desabilite com `-refresh=false`) — ou seja, **lê o estado real** dos recursos no GCP
- Depois **determina as ações necessárias** pra chegar no estado desejado (o que tá no `.tf`)
- **Não muda nada** — é só preview
- Serve pra **confirmar que o plano bate com sua expectativa** antes de aplicar

#### `terraform apply`
- **Cria/atualiza a infra** descrita no `.tf`
- Antes de executar, mostra o plano novamente e pede confirmação (`yes`)
- Depois disso, os recursos existem no projeto GCP

### Armadilhas de prova / pontos sutis do vídeo
| Armadilha | Verdade |
|-----------|---------|
| "Terraform deploya recursos em sequência" | **Não.** Deploya em **paralelo** (respeitando dependências). Cloud Shell é sequencial. |
| "`terraform plan` lê só o arquivo `.tf`" | **Não.** Faz um **refresh do estado real** antes de calcular o diff. |
| "Packer é uma alternativa ao Terraform" | **Não exatamente** — Packer constrói **imagens**, não provisiona infra rodando. |
| "Provider block é opcional" | **Não.** Sem provider, Terraform não sabe pra que cloud ir nem quais credenciais usar. |
| "Output block cria um recurso" | **Não.** Output só **expõe valores** após o apply (útil pra ler IP, URL etc). |
| "Terraform substitui o `gcloud`" | **Não.** Terraform **usa as APIs do GCP por baixo** — mesma camada que `gcloud` chama. São ferramentas complementares com casos de uso diferentes. |

## Lifecycle de comandos

```
terraform init  →  terraform plan  →  terraform apply
                                            ↓
                                  terraform destroy (quando descartar)
```

| Comando | O que faz |
|---------|-----------|
| **`terraform init`** | Baixa a última versão do **provider** |
| **`terraform plan`** | Verifica sintaxe + preview do que será criado/alterado |
| **`terraform apply`** | **Cria/configura recursos** declarados no config file |
| **`terraform destroy`** | Destrói todos os recursos do config file |

## DQ 10 — O que o `terraform apply` faz?

**Resposta: D — Sets up resources requested in the terraform config file**

| Comando | Confusão comum |
|---------|----------------|
| `init` | Baixa **provider** (não cria recursos) |
| `plan` | **Preview** (não cria recursos) |
| **`apply`** ✅ | **Cria/configura recursos** |
| `destroy` | Remove recursos |
