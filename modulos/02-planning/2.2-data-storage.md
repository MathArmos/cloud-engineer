# 2.2 — Planning and configuring data storage options

> **Fonte:** Google Cloud — *Preparing for Your Associate Cloud Engineer Journey*, Module 2 Lesson 7 (Section 2)
> **DQs do workbook cobertas:** 05, 06, 07
> **Considerations da prova:** escolha do produto (Cloud SQL, BigQuery, Firestore, Spanner, Bigtable); escolha da storage class (zonal/regional Persistent Disk; Standard, Nearline, Coldline, Archive).

---

## Conceitos

### Mapa mental — 4 famílias de storage no Google Cloud

| Família | Serviços | O que armazena | Caso típico |
|---|---|---|---|
| **Relational (SQL)** | Cloud SQL, Spanner | Dados estruturados em tabelas com schema fixo, transacional | Apps web, e-commerce, sistemas operacionais |
| **Non-relational (NoSQL)** | Firestore, Bigtable | Dados sem schema rígido | Apps mobile, séries temporais, IoT |
| **Object** | Cloud Storage | Blobs (arquivos), dados binários | Imagens, vídeos, backups, logs |
| **Warehouse (analytical)** | BigQuery | Tabelas otimizadas para queries analíticas em massa | Dashboards, BI, analytics histórico |

**Princípio fundamental:** "qual workload?" responde a família, e dentro da família o produto sai de outras pistas (escala, latência, transacionalidade, distribuição global).

---

### Workloads: transacional vs analítico

Antes de escolher o banco, identifique o tipo de workload na pergunta:

| Característica | Transacional (OLTP) | Analítico (OLAP) |
|---|---|---|
| Padrão de uso | Muitas escritas e atualizações pequenas | Muitas leituras em massa, escritas raras |
| Latência alvo | Milissegundos por operação | Segundos a minutos por query |
| Volume por operação | Pequeno (registros individuais) | Grande (varredura de milhões de linhas) |
| Exemplos | Pedido de e-commerce, transferência bancária | Relatório de vendas trimestrais, dashboard |
| Produtos GCP | Cloud SQL, Spanner, Firestore | BigQuery, Bigtable (parcialmente) |

> **Atalho de prova:** se a pergunta menciona "**historical data**", "**dashboards**", "**analyze**", "**analyst running SQL queries**" → quase sempre é **BigQuery**. Se menciona "**reads and writes**", "**transactional**", "**user orders**", "**credentials**" → Cloud SQL ou Spanner.

---

### Relational SQL — Cloud SQL e Spanner

#### Cloud SQL

**O que é:** Versão **gerenciada** dos bancos relacionais que você já conhece — MySQL, PostgreSQL, SQL Server. Roda em VMs (com várias opções de tamanho e HA), mas você não as gerencia.

**Quando usar:**
- Apps web tradicionais com banco relacional
- E-commerce com pedidos transacionais
- Sistemas que **já rodam** em MySQL/Postgres on-prem (migração quase direta)
- Você quer um banco "normal" sem se preocupar com servidor, backup, patching

**Limite mental:** Cloud SQL é **regional**. Se você precisa de banco relacional **global** com baixa latência em qualquer região → Spanner.

#### Spanner

**O que é:** Banco relacional **horizontalmente escalável e globalmente consistente**, criado pelo Google. SQL-compatível, mas é uma arquitetura própria (não é MySQL/Postgres por baixo).

**Quando usar:**
- Aplicações de **grande escala** (> ~2 TB) com necessidade de SQL
- Necessidade de **consistência global** (transações distribuídas em vários continentes)
- Alta disponibilidade (99,999%) sem trade-off de consistência
- Cargas de **alto I/O** (AdTech, FinTech, MarTech)

**Trade-off:** mais caro que Cloud SQL. Use só quando justificar.

**Atalho lexical:** "high I/O", "global consistency", "globally available" → Spanner.

---

### Non-relational NoSQL — Firestore e Bigtable

#### Firestore

**O que é:** Banco NoSQL **de documentos**. Você guarda entidades (documentos JSON-like) com atributos (chave-valor). Foco em **transacional**: sync com clients mobile/web, ACID em documento.

**Quando usar:**
- Apps mobile/web com sincronização de estado entre clients
- Perfis de usuário com atributos variáveis
- Estado de jogo (game state)
- Dados hierárquicos (documentos aninhados)

**Casos típicos:** mobile, web, server development.

#### Bigtable

**O que é:** Banco NoSQL **wide-column** (família de colunas). Os dados são organizados em **row keys** otimizadas para a query que você precisa fazer. Petabyte-scale, **baixa latência** para reads/writes massivos.

**Quando usar:**
- **Séries temporais** (time-series): leituras históricas indexadas por tempo
- Workloads de **muitas escritas + muitas leituras** simultâneas
- AdTech, FinTech, IoT (telemetria)
- Dados analíticos que **não cabem** em BigQuery (escala extrema) ou que precisam de latência mais previsível por linha

**Diferença pra BigQuery:** BigQuery é para queries SQL analíticas ad-hoc (analista escreve SELECT). Bigtable é para padrões de acesso programáticos (app lê linha por row key).

**Atalho lexical:** "time-based data", "historical time-series", "heavy read + write" → Bigtable (combinado com BigQuery em alguns cenários).

---

### Object storage — Cloud Storage

**O que é:** Bucket de objetos (blobs). Não é banco; é **armazenamento de arquivos** com API HTTP. Cada objeto é imutável (você sobe um novo para "atualizar").

**Quando usar:**
- Imagens, vídeos, áudio
- Backups
- Logs grandes em arquivos
- CSVs e arquivos com schema implícito (consumidos por jobs de processamento)
- Servir conteúdo estático para web

**Não usar para:** dados queryáveis transacionais. Não é um banco.

---

### Cloud Storage — Storage Classes (decisão de custo vs latência)

O Cloud Storage tem **4 classes**, todas com **disponibilidade imediata** (não há "tempo de retrieval" como em Glacier da AWS), mas com pricing/restrições diferentes baseadas em **frequência de acesso** e **duração mínima de armazenamento**.

| Classe | Acesso esperado | Duração mínima | Custo armazenamento | Custo de retrieval | Caso típico |
|---|---|---|---|---|---|
| **Standard** | Frequente / hot | Nenhuma | Mais alto | Sem custo de retrieval | Dados "quentes", computação intensiva, dados sendo gerados ou consumidos agora |
| **Nearline** | ~1x/mês | 30 dias | Médio-baixo | Taxa por operação | Backup, conteúdo multimídia long-tail |
| **Coldline** | ~1x/trimestre (90 dias) | 90 dias | Baixo | Taxa maior | DR, dados financeiros para compliance |
| **Archive** | ~1x/ano (365 dias) | 365 dias | Mais baixo | Taxa mais alta | Arquivo de longo prazo, online backup, DR remoto |

**Regra de ouro:**
- Acessa **muito** → Standard
- Acessa **às vezes** → Nearline
- Acessa **raramente** → Coldline
- Quase **nunca** → Archive

**Armadilha de prova:** "frequently analyzes large amounts of data" / "dashboards" / "data-intensive computation" → **Standard**. Não há "tempo de retrieval" como AWS Glacier; o gatilho é o **custo de acesso** e a **duração mínima**.

**Pegadinha de pricing:** Se você acessar dado em Nearline/Coldline/Archive **antes** do tempo mínimo, você paga **taxa de acesso antecipado**. Não é "não pode", é "pode mas vai custar caro".

---

### Localização: Regional, Dual-region, Multi-region

Independente da classe, você escolhe onde o bucket reside:

- **Regional:** uma região (ex.: `us-central1`). Menor latência se sua compute está na mesma região; mais barato.
- **Dual-region:** duas regiões específicas (ex.: `us-central1` + `us-east1`). Latência baixa numa delas; **resiliência** se uma cair.
- **Multi-region:** geografia ampla (`US`, `EU`, `ASIA`). Latência consistente para usuários globais; redundância máxima; mais caro.

**Regra:** coloque o storage **na mesma região** dos serviços que consomem o dado. Evita custo de network egress e reduz latência.

---

### Tabela-resumo final (decisão de banco)

| Pista no enunciado | Resposta provável |
|---|---|
| "Analytical", "SQL", "historical data", "dashboards" | **BigQuery** |
| "Time-series", "heavy read/write", "events", "AdTech/IoT" | **Bigtable** |
| "Mobile/web app", "user profiles", "game state", "documents" | **Firestore** |
| "Web framework", "OLTP", "user credentials", "customer orders" | **Cloud SQL** |
| "Global", "high I/O", "horizontal scale", "SQL+scale" | **Spanner** |
| "Images", "videos", "backups", "binary files" | **Cloud Storage** |

## Questionários

### DQ 05 — Analisar projeção de vendas trimestrais com SQL
**Cenário:** Cymbal precisa analisar se atingiu projeções de vendas trimestrais. Analistas são familiares com SQL.

**Resposta correta: A — BigQuery**

**Por quê:**
- **"Quarterly sales"** = dado **histórico** + **analítico** (não transacional)
- **"Familiar with SQL"** = precisa de SQL engine
- BigQuery = **data warehouse** com query engine SQL distribuído

**Por que as outras estão erradas:**
- **B (Cloud SQL):** otimizado para transações operacionais, não para queries analíticas em massa
- **C (Spanner):** SQL-compatible, mas projetado para **OLTP** global, não para analytics
- **D (Firestore):** NoSQL de documentos — não há SQL nativo nem é analítico

**Armadilha:** "SQL + análise histórica + grande volume" = **sempre BigQuery**. Não é Cloud SQL — apesar do nome "SQL", Cloud SQL é OLTP.

### DQ 06 — Storage class para análise frequente de grandes volumes
**Cenário:** Supply chain frequentemente analisa grandes volumes de dados para alimentar processos de negócio e dashboards operacionais.

**Resposta correta: D — Standard**

**Por quê:**
- **"Frequently analyzes"** = acesso constante (hot data)
- **"Dashboards"** = dado precisa estar imediatamente disponível, sem custo de retrieval
- Standard storage = sem taxa de recuperação, sem duração mínima, ideal para hot data

**Por que as outras estão erradas:**
- **A (Archive):** acesso < 1x/ano — incompatível com "frequently"
- **B (Coldline):** acesso a cada 90 dias — dashboards precisam de dado atual
- **C (Nearline):** acesso a cada 30 dias — ainda muito pouco para "frequently"

**Armadilha:** "Frequently accessed" = Standard. **Sempre.** As classes Nearline/Coldline/Archive **cobram retrieval fee** que destrói o custo de workloads frequentes.

**Padrão geral:** quanto mais frequente o acesso, mais barato o **retrieval** mas mais caro o **storage**. Standard tem retrieval **grátis**, e por isso é a classe de hot data.

### DQ 07 — Dashboards históricos time-based (escolha 2)
**Cenário:** Popular dashboards visuais com dado histórico time-based. Caso de uso analítico. **Escolher 2.**

**Resposta correta: A — BigQuery + E — Bigtable**

**Por quê:**

**BigQuery:**
- Data warehouse otimizado para queries SQL históricas
- Pode rodar queries federadas contra fontes externas (Cloud Storage, Bigtable)
- Ideal para dashboards visuais (Looker, Looker Studio)

**Bigtable:**
- Petabyte-scale NoSQL wide-column
- Row keys otimizadas para queries específicas
- **Especificamente projetado para dados históricos time-based**

**Por que as outras estão erradas:**
- **B (Cloud Storage):** object store, não é queryável, não é transacional nem analítico
- **C (Firestore):** NoSQL de documentos, transacional, não analítico
- **D (Cloud SQL):** OLTP transacional regional, não para análise histórica em massa

**Armadilha sutil:** A questão pede **duas** soluções. BigQuery cobre o caso "analytical/SQL" e Bigtable cobre o caso "time-series em escala". As duas se complementam: Bigtable armazena os eventos brutos (telemetria), BigQuery agrega/analisa.

## Armadilhas de prova

1. **"SQL" não significa Cloud SQL.** SQL é uma linguagem. Cloud SQL é OLTP. BigQuery, Spanner e até alguns conectores de Bigtable usam SQL. **Cloud SQL ≠ SQL.**
2. **"Historical / analytical / dashboards"** → BigQuery. Quase determinístico.
3. **"Time-series / time-based"** → Bigtable. Especialmente se a escala for grande.
4. **"Global / horizontal scale / strong consistency global"** → Spanner. Cloud SQL é regional.
5. **"Frequently accessed"** = **Standard storage class**. Não Nearline.
6. **"Once a year / archive / regulatory"** = **Archive** class.
7. **Spanner vs Cloud SQL:** se o cenário não menciona escala absurda ou distribuição global, presuma **Cloud SQL** (mais simples e barato).
8. **"Mobile / sync / user profile / game state"** → **Firestore**. NoSQL document.
9. **"Images / videos / backups / binary"** → **Cloud Storage**. Não banco.
10. **Não confundir Bigtable com BigQuery.** Bigtable é NoSQL operacional (alto throughput), BigQuery é data warehouse analítico. Mesmo prefixo, propósitos diferentes.

## Cenário real

**Onde eu usaria isso no trabalho?**

- **Cloud SQL (Postgres)** → backend da Portaria/TaxIA — dados transacionais de usuário, mensagens, configs. Já uso Postgres on-prem; Cloud SQL seria migração quase direta.
- **Firestore** → se eu fosse fazer um mobile app com sync offline (notificações, estado de UI). Atualmente uso Postgres porque o app é stateless do lado do client.
- **BigQuery** → quando precisasse fazer **analytics** sobre o histórico de mensagens, padrões de uso, métricas de negócio. Hoje faria isso ad-hoc em SQL contra o Postgres, mas em escala isso quebraria.
- **Cloud Storage** → armazenar **mídia** (imagens enviadas via WhatsApp, PDFs de relatórios) e **backups** do banco.
- **Spanner / Bigtable** → escala que eu não tenho hoje. Spanner para banco relacional global; Bigtable para telemetria massiva (IoT, logs de sensor).

**Armadilha que cometeria no trabalho real:** usar Standard storage class para tudo, mesmo dados que ficam meses sem ser tocados. Lifecycle policies do Cloud Storage podem mover automaticamente para classes mais baratas após X dias.
