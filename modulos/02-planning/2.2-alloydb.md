# 2.2 — AlloyDB for PostgreSQL

> **Considerações da prova:** o ACE **não cobra AlloyDB com profundidade**, mas pode citar como opção em pergunta de "qual serviço escolher" quando o cenário envolve **PostgreSQL + HTAP + analítico em tempo real**. Saber **quando AlloyDB vence Cloud SQL** e **quando vence BigQuery** é suficiente.

---

## Conceitos

### Bloco "O problema que AlloyDB resolve — HTAP de verdade"

O contexto que já apareceu no overview (`2.2-storage-database-overview.md`): o padrão clássico para juntar OLTP + analytics é manter **dois bancos separados** com ETL no meio.

```
   App OLTP                  ETL noturno              Warehouse
 (Cloud SQL ou           ─────────────────►        (BigQuery /
  Postgres on-prem)       (Dataflow, Datastream,     Snowflake /
                          Spark, scripts)            Redshift)
        ▲
        |
   Escrita rápida                                  Análise lenta
   Latência baixa                                  Defasagem 24h
```

**Problemas dessa arquitetura:**
- **Defasagem** — analista vê dado de ontem (ou de 1h atrás, no melhor caso com CDC).
- **Custo do ETL** — infra extra (Dataflow, Datastream, scripts), monitoramento, escala.
- **Discrepâncias** — schema desincronizado entre os dois bancos vira fonte de bugs.
- **Dois conjuntos de privilégios** para administrar.

**AlloyDB elimina o ETL:** OLTP e analytics rodam **no mesmo banco**, com performance honesta nos dois.

> *"AlloyDB for PostgreSQL is a fully managed, PostgreSQL-compatible database service that's designed for demanding workloads such as hybrid transactional and analytical processing."*

#### Quando esse problema vale a pena resolver?

- Quando o negócio precisa de **decisões em tempo real** sobre dado fresco (detecção de fraude, BI operacional, dashboards executivos com latência de segundos).
- Quando o cliente final **paga** pelo dado em tempo real (analytics-as-a-service embarcado em produto SaaS).
- Quando o time **odeia gerenciar pipeline de ETL** e está disposto a pagar pelo banco mais caro pra eliminar essa complexidade.

**Quando NÃO vale a pena:**
- Workload puramente OLTP, sem necessidade analítica → Cloud SQL é mais barato.
- Workload puramente analítico, batch, sobre dado estável → BigQuery é incomparável em custo/performance pra esse cenário.

### Bloco "Arquitetura — o que torna AlloyDB diferente"

> *"AlloyDB pairs a Google-built database engine with a cloud-based, multi-node architecture to deliver enterprise-grade performance, reliability, and availability."*

Pontos centrais (sem cair na engenharia profunda, que está fora do escopo ACE):

1. **PostgreSQL-compatible, mas não é o Postgres upstream puro.** Compatibilidade de **wire protocol** e de **SQL** — você conecta com qualquer driver Postgres padrão e suas queries funcionam. **O engine por baixo** foi modificado pela Google para HTAP.

2. **Storage e compute desacoplados.** Diferente do Postgres clássico (onde storage e compute estão no mesmo nó), AlloyDB separa:
   - Camada de **compute** (nodes primários + read pool).
   - Camada de **storage** (separada, intelligent, com cache regional).
   - Isso permite **escalar leitura horizontalmente** adicionando nodes que compartilham o mesmo storage.

3. **Columnar engine embutido.** Para queries analíticas, o AlloyDB mantém uma representação **colunar** dos dados quentes (em memória/SSD), permitindo varreduras analíticas rápidas sem ETL externo.

4. **Multi-node architecture.** Você tem:
   - **Primary instance** — recebe escrita.
   - **Read pool instances** — múltiplos nodes para leitura, com load balancing automático.
   - **Standby** — failover automático.

#### O que "managed" cobre

> *"AlloyDB automates administrative tasks, such as backups, replication, patching, and capacity management."*

- **Backups** automatizados + on-demand + PITR.
- **Replicação** entre primary e read pool, e entre regiões.
- **Patches** do engine e do SO.
- **Capacity management** — incluindo armazenamento que cresce sozinho conforme necessário.

#### Adaptive algorithms e ML embutido

> *"AlloyDB also uses adaptive algorithms and machine learning for PostgreSQL vacuum management, storage and memory management, data tiering, and analytics acceleration."*

Quatro áreas onde algoritmos adaptativos atuam:

1. **Vacuum management** — o Postgres tradicional precisa de `VACUUM` para limpar tuplas mortas (resultado do MVCC). Gerenciar `autovacuum` é dor de cabeça em produção. AlloyDB usa ML para decidir **quando** rodar vacuum, **em que tabelas**, **com que agressividade**.

2. **Storage e memory management** — caching automático de blocos quentes, decisão de o que manter em RAM vs SSD vs storage frio.

3. **Data tiering** — mover dado frio para storage mais barato sem o DBA precisar configurar.

4. **Analytics acceleration** — reconhece padrões de query analítica e ativa o engine colunar para acelerar.

**Tradução prática:** o DBA tradicional gastaria 30% do tempo cuidando dessas coisas no Postgres puro. AlloyDB libera esse tempo.

### Bloco "Performance — os números que o vídeo afirma"

> *"AlloyDB provides fast transactional processing, more than 4 times faster than standard PostgreSQL for transactional workloads."*

> *"AlloyDB provides real-time business insights and is up to 100 times faster than standard PostgreSQL for analytical queries."*

| Workload          | Speedup vs Postgres padrão |
|-------------------|----------------------------|
| **Transacional**  | **≥4x**                    |
| **Analítico**     | **até 100x**               |

**Como ler esses números (e onde NÃO se deixar enganar):**

- "Standard PostgreSQL" no comparativo do Google = Postgres **upstream sem otimização**. Não compare com Postgres bem-tunado com extensões (Citus, TimescaleDB) — aí o gap é menor.
- O **4x transacional** vem principalmente de cache mais inteligente, vacuum sem stalls, e a separação storage/compute.
- O **100x analítico** vem do **columnar engine**. Em query OLAP típica (agregação sobre poucas colunas de milhões de linhas), processamento colunar é ordens de magnitude mais rápido que row-oriented.

**Casos em que AlloyDB brilha:**

> *"It's suitable for demanding enterprise workloads, including workloads that require high transaction throughput, large data sizes, or multiple read replicas."*

- **Alto throughput transacional** (milhares de TPS sustentados).
- **Datasets grandes** (TB+ em um único banco — Cloud SQL até 64 TB; AlloyDB escala melhor nesse range).
- **Múltiplos read replicas** servindo análise em tempo real sobre o mesmo dado operacional.

### Bloco "Disponibilidade — SLA e HA"

> *"AlloyDB provides high-availability and a 99.99% uptime SLA, inclusive of maintenance."*

| Métrica                 | Valor                                         |
|-------------------------|-----------------------------------------------|
| **SLA de uptime**       | **99,99%** (inclui janelas de manutenção)     |
| **HA**                  | Built-in, com standby em zona diferente       |
| **Replicação**          | Síncrona dentro da região, opção cross-region |

**"Inclusive of maintenance"** é uma frase importante: muitos serviços excluem janelas de manutenção do SLA (você pode ficar fora do ar 4h pra patch e isso "não conta"). O AlloyDB **conta** — então 99,99% é honesto.

**Comparação com Cloud SQL Postgres:**

| Característica            | Cloud SQL Postgres   | AlloyDB                     |
|---------------------------|----------------------|-----------------------------|
| SLA com HA                | ~99,95%              | **99,99%**                  |
| Inclui manutenção?        | Não (varia)          | **Sim**                     |
| Engine                    | Postgres upstream    | Postgres-compatible (custom)|
| Storage/compute desacopla?| Não                  | Sim                         |
| Engine colunar embutido?  | Não                  | Sim                         |
| Vacuum gerenciado por ML? | Não                  | Sim                         |
| Custo                     | Menor                | **Maior**                   |

### Bloco "Vertex AI integration"

> *"Built-in integration with Vertex AI, Google's artificial intelligence platform, lets you call machine learning models."*

**O que isso significa:** dentro de uma query SQL no AlloyDB, você pode invocar **modelos do Vertex AI** (ML hospedados no GCP) e usar o resultado como qualquer coluna.

**Exemplo conceitual:**

```sql
SELECT
  order_id,
  customer_id,
  amount,
  ml_predict.fraud_score(amount, customer_id, country) AS fraud_score
FROM orders
WHERE amount > 1000;
```

**Casos de uso:**
- **Scoring em tempo real** dentro de transação (fraude, risco de crédito, churn).
- **Embeddings** para busca semântica (similaridade de texto/imagem em coluna).
- **Classificação automática** de entidades (tagging, categorização).

**Por que isso importa estrategicamente:** elimina o padrão clássico de "exportar dado pra serviço de ML, receber resposta, atualizar banco". Tudo acontece dentro do banco, com latência mínima.

**Armadilha de prova (improvável, mas):** se aparecer "PostgreSQL + chamar modelo ML inline em query" → AlloyDB. Cloud SQL não tem essa integração nativa.

### Bloco "Quando usar AlloyDB — decision points"

#### Use AlloyDB quando:

| Pista no enunciado                                              | Por quê                                       |
|-----------------------------------------------------------------|-----------------------------------------------|
| "PostgreSQL-compatible" + "analytical and transactional"        | Definição de HTAP                             |
| "Real-time business insights on operational data"               | Definição de HTAP em outras palavras          |
| "PostgreSQL + escala que Cloud SQL não dá"                      | AlloyDB tem teto maior + storage desacoplado  |
| "4x faster transactional / 100x faster analytical than PostgreSQL" | Marketing oficial do AlloyDB               |
| "99.99% SLA including maintenance, PostgreSQL-compatible"       | Diferencial vs Cloud SQL                      |
| "Invoke ML models inline in SQL queries"                        | Integração Vertex AI                          |
| "Migrar Postgres on-prem com requisitos enterprise"             | Versão "premium" gerenciada                   |

#### NÃO use AlloyDB quando:

| Pista                                                           | Use isso ao invés                              |
|-----------------------------------------------------------------|------------------------------------------------|
| "Aplicação Postgres pequena/regional, sem analytics"            | **Cloud SQL Postgres** (mais barato)           |
| "Workload puramente OLTP global"                                | **Spanner**                                    |
| "Workload puramente analítico ad-hoc sobre dado estável"        | **BigQuery**                                   |
| "Não relacional / document model / mobile sync"                 | **Firestore**                                  |
| "Time-series / IoT escala massiva"                              | **Bigtable**                                   |
| "Cache de aplicação"                                            | **Memorystore**                                |

### Bloco "Comparativo definitivo Cloud SQL Postgres vs AlloyDB vs Spanner"

Esse é o triângulo que confunde — todos têm SQL, gerenciados, "premium". Memorizar:

| Eixo                    | Cloud SQL Postgres        | AlloyDB                              | Spanner                               |
|-------------------------|---------------------------|--------------------------------------|---------------------------------------|
| **Compatibilidade**     | Postgres upstream         | Postgres-compatible (custom engine)  | Google SQL ou PG dialect              |
| **Escopo**              | Regional                  | Regional (multi-region read replica) | Regional **ou global**                |
| **Escala de escrita**   | Vertical                  | Vertical (mais ampla)                | **Horizontal**                        |
| **HTAP**                | ❌                        | ✅                                   | Limitado (não otimizado pra OLAP)     |
| **Engine colunar**      | ❌                        | ✅                                   | ❌                                    |
| **Consistência global** | N/A (regional)            | N/A (regional)                       | ✅ **forte global**                   |
| **SLA típico (HA)**     | 99,95%                    | 99,99% (inclui manutenção)           | 99,99% regional / 99,999% multi-region|
| **Use case**            | App Postgres comum        | Postgres + analytics tempo real      | Global, mission-critical, escala      |
| **Custo relativo**      | $                         | $$$                                  | $$$$                                  |

**Heurística pra prova:**
1. Cenário tem "global" + relacional → **Spanner**.
2. Cenário tem "Postgres + analytics tempo real / HTAP" → **AlloyDB**.
3. Cenário tem "Postgres regional / app web comum" → **Cloud SQL**.

## Labs

### Comandos `gcloud` críticos (referência)

#### Criar um cluster AlloyDB

```bash
# 1. Criar o cluster (regional)
gcloud alloydb clusters create app-cluster \
  --region=us-central1 \
  --password=trocar-em-prod

# 2. Criar a primary instance dentro do cluster
gcloud alloydb instances create primary-instance \
  --cluster=app-cluster \
  --region=us-central1 \
  --instance-type=PRIMARY \
  --cpu-count=4

# 3. (opcional) Criar read pool instance
gcloud alloydb instances create read-pool \
  --cluster=app-cluster \
  --region=us-central1 \
  --instance-type=READ_POOL \
  --cpu-count=4 \
  --read-pool-node-count=3
```

**O que está acontecendo:**
- O **cluster** é o agrupamento lógico (define região, rede, configuração de storage).
- A **primary instance** é onde escrita acontece.
- O **read pool** é o conjunto de nodes para análise + leitura escalada.

#### Conectar

```bash
# Conexão pela AlloyDB Auth Proxy (similar ao Cloud SQL Auth Proxy)
./alloydb-auth-proxy projects/SEU_PROJETO/locations/us-central1/clusters/app-cluster/instances/primary-instance

# Depois, conecta como Postgres normal
psql -h 127.0.0.1 -U postgres
```

#### Backup on-demand

```bash
gcloud alloydb backups create manual-backup-$(date +%Y%m%d) \
  --cluster=app-cluster \
  --region=us-central1
```

## Cenário real

### Onde AlloyDB faria sentido?

**Não nos projetos atuais (Portaria / TaxIA), por hora.** Cloud SQL Postgres atende com folga, e a complexidade/custo do AlloyDB seria desperdício.

**Quando viraria a chave:**
- Se o Portaria começasse a oferecer **dashboards analíticos para os condomínios em tempo real** (ocupação, fluxo, alertas) sobre os mesmos dados operacionais, e a query analítica começasse a competir com transação OLTP no mesmo Postgres.
- Se o TaxIA fosse migrar para uma **arquitetura analítica embarcada** (dashboards de uso por cliente em tempo real, segmentação inline em SQL).

**Cenários canônicos fora dos projetos:**
- **Plataformas de adtech**: matching de impressions em tempo real + relatórios analíticos sobre as mesmas tabelas.
- **Logística**: tracking de rotas (OLTP) + dashboards de SLA por região (OLAP), com a mesma fonte de dados.
- **FinTech regional** (não global — aí seria Spanner): OLTP de transações + scoring/fraud em tempo real, com chamadas Vertex AI inline.

### O argumento "AlloyDB elimina o pipeline de ETL"

Para um time pequeno, eliminar um pipeline `Cloud SQL → Datastream → BigQuery` pode justificar o custo do AlloyDB **se o time gastaria mais tempo mantendo o pipeline do que pagando a diferença de licença**.

**Conta típica:**
- Cloud SQL Postgres + BigQuery + ETL: ~$X/mês infra + 0,3 FTE de manutenção.
- AlloyDB sozinho: ~$2X/mês infra + 0 FTE de manutenção.

Se o FTE custa caro e o time é pequeno, AlloyDB ganha. Se o time já tem time de data engineering, talvez não.

## Armadilhas de prova

1. **AlloyDB não está no exam guide oficial do ACE.** Pode aparecer em multiple choice mas raramente como tema central. Não invista tempo desproporcional.

2. **AlloyDB ≠ Cloud SQL Postgres.** Mesma compatibilidade SQL, mas **engine diferente** (Google reescreveu partes), **storage desacoplado**, **engine colunar**, **integração ML nativa**.

3. **AlloyDB é regional.** Não escala "globalmente" como o Spanner. Multi-region read replicas existem, mas escrita é regional. Se o cenário pede "global strong consistency relacional", **Spanner**, não AlloyDB.

4. **HTAP é a pista determinística.** Se o enunciado tem "HTAP", "hybrid transactional/analytical", "real-time analytics on operational data" + "PostgreSQL" → AlloyDB.

5. **AlloyDB não substitui BigQuery.** Para queries analíticas **ad-hoc** sobre **TB-PB** de dado **estável**, BigQuery ainda é melhor (preço por TB processado, scaling automático, sem capacidade fixa).

6. **AlloyDB não substitui Cloud SQL pra apps simples.** Cobre custo extra **só vale** se houver necessidade real de HTAP, escala, ou SLA mais alto.

7. **"99,99% inclusive of maintenance" é o tagline oficial.** Diferencial vs Cloud SQL nesse aspecto.

8. **"4x transacional, 100x analítico"** comparado a **Postgres padrão**. Memorizar os números, eles aparecem em material oficial.

9. **Vertex AI integration é diferencial AlloyDB.** Postgres puro / Cloud SQL não têm essa integração de fábrica.

10. **AlloyDB ≠ AlloyDB Omni.** Existe uma versão "AlloyDB Omni" que roda fora do GCP (on-prem, outras clouds). Para o exame ACE, só o "AlloyDB for PostgreSQL" gerenciado importa.

11. **Quando a pergunta lista as opções "Cloud SQL / AlloyDB / Spanner / BigQuery":** verifique se há pista clara — se for ambígua entre Cloud SQL e AlloyDB, **prefira Cloud SQL** (mais simples, mais barato), a menos que o cenário enfatize HTAP/analytics/escala.
