## Conceitos

### Bloco "O que é Filestore — definição precisa"

**Definição:** Filestore é o serviço **gerenciado de file storage** do Google Cloud para aplicações que precisam de **interface de file system** e/ou **file system compartilhado** entre múltiplos clientes.

Em palavras diretas: Filestore é o **NAS (Network-Attached Storage) do GCP**. Você cria uma instância Filestore, ela expõe um endpoint, e várias VMs (ou pods do GKE) **montam** aquele endpoint como se fosse um disco compartilhado.

**Citação do vídeo:**
> *"Filestore gives users a simple native experience for standing up managed network attached storage with either Compute Engine or Google Kubernetes Engine instances."*

#### Características-chave (memorize)

| Característica                                   | Detalhe                                                                                  |
|--------------------------------------------------|------------------------------------------------------------------------------------------|
| **Interface**                                    | File system (POSIX), via **NFSv3**                                                        |
| **Compatibilidade**                              | Qualquer **cliente NFSv3** funciona — sem plugins, sem driver custom                       |
| **Capacity**                                     | Até **centenas de terabytes** por instância                                                |
| **File locking**                                 | Nativo (semântica POSIX completa, com `flock`/`fcntl`)                                     |
| **Scale-out performance**                        | Performance cresce horizontalmente conforme a capacity/tier escolhido                       |
| **Capacity e performance ajustáveis independentemente** | Você dimensiona storage e throughput separadamente                              |
| **Native a GCE e GKE**                           | Mount direto via NFS; no GKE existe driver CSI                                              |

#### Por que o "ajuste independente de capacity e performance" importa

Esta é uma diferença sutil mas importante. Em muitos sistemas de storage tradicional (incluindo Persistent Disk), **performance é função do tamanho**: discos maiores são mais rápidos. Se você precisa de mais IOPS mas o dado é pequeno, **paga storage que não usa**.

No Filestore, dependendo do tier, você pode:
- Ter pouco dado (pequena capacity) com **alta performance** (throughput configurado em cima).
- Ter muito dado com **performance moderada** (economiza no throughput).

Isso casa com workloads de pesquisa científica / engenharia onde o dataset é pequeno mas a leitura precisa ser **muito rápida** (e vice-versa).

### Bloco "Filestore vs Cloud Storage vs Persistent Disk — a confusão clássica"

Os três armazenam "arquivos", mas resolvem **problemas completamente diferentes**. Esta confusão cai na prova com frequência.

| Característica          | **Cloud Storage**                       | **Persistent Disk**                     | **Filestore**                                  |
|-------------------------|------------------------------------------|-----------------------------------------|------------------------------------------------|
| Modelo                  | Object storage (blob)                    | Disco em bloco                          | File system NFS                                |
| Interface               | HTTPS / API                              | Disco local na VM (`/dev/sdX`)          | NFS montado em diretório (`/mnt/share`)        |
| Semântica POSIX         | ❌ Não                                   | ✅ Sim (filesystem local em cima)        | ✅ Sim (NFSv3)                                 |
| Compartilhamento        | Acesso simultâneo por API                | **Read-write em 1 VM** (read-only em N)  | **Read-write em N VMs** simultâneas            |
| File locks (`flock`)    | ❌                                       | ✅ (local)                               | ✅ (distribuído)                                |
| Latência típica         | Dezenas de ms                            | Sub-ms (SSD)                            | Single-digit ms                                |
| Caso emblemático        | Imagens, vídeos, backups, data lake       | Disco-raiz e disco-de-dados de uma VM    | NAS compartilhado, render farm, HPC            |
| Pricing model           | Por GB armazenado + ops + egress         | Por GB provisionado + IOPS opcionais     | Por GB do tier escolhido (throughput incluso)  |

**Regras-âncora pra prova:**

- "Vários clientes precisam **acessar simultaneamente o mesmo arquivo**" → **Filestore**.
- "Aplicação espera `open()`, `seek()`, `flock()`, `chmod()`" → **Filestore** (Cloud Storage não tem; Persistent Disk só serve uma VM).
- "Quero servir o arquivo pela internet via HTTP" → **Cloud Storage**.
- "Quero disco-raiz ou disco-de-dados de uma VM" → **Persistent Disk** (ou Hyperdisk).
- "Lift-and-shift de app que assume NFS on-prem" → **Filestore**.

### Bloco "Use cases canônicos (os 6 do vídeo)"

O vídeo lista exatamente 6 casos de uso. Todos pelo mesmo motivo: **múltiplas VMs acessando o mesmo conjunto de arquivos com semântica POSIX**.

#### 1. Lift-and-shift de apps enterprise on-prem

Muitos apps legados foram escritos assumindo que existe um **share NFS** em algum servidor da rede interna. Reescrever esses apps pra usar Cloud Storage seria projeto de meses. Filestore permite que você **monte o share na nuvem** e o app continue rodando sem mudança de código.

**Quem cai nesse caso:** ERPs antigos, CMSes legados, sistemas de gestão hospitalar/jurídica, qualquer app monolítico antigo.

#### 2. Media rendering (VFX, animação 3D)

Renderização de imagem/vídeo distribuída exige que **dezenas a milhares de máquinas** leiam o mesmo conjunto de assets (texturas, modelos, cenas) e escrevam **frames de saída** num lugar central.

> *"As rendering workflows typically run across fleets of Compute machines, all of which mount a shared file system."*

**Por que Cloud Storage não atende:** as ferramentas de render (Blender, Maya, Houdini, Nuke) esperam **paths de filesystem** (`/proj/scene01/textures/...`) e fazem **random access** com `seek`. Cloud Storage não suporta isso. Filestore sim.

#### 3. EDA — Electronic Design Automation

Ferramentas que projetam chips e PCBs (Cadence, Synopsys, Mentor) rodam **batch jobs em milhares de cores**. Cada core lê e escreve em arquivos compartilhados (netlists, layouts, simulações). Sem NAS performático, o cluster fica idle esperando IO.

**Citação:**
> *"EDA is all about data management. It requires the ability to batch workloads across thousands of cores and has large memory needs."*

#### 4. Data analytics latency-sensitive

Workloads como **modelos financeiros complexos** ou **análise de dados ambientais** que dependem de baixa latência por arquivo (não por bucket). Diferente de analytics em BigQuery — aqui é simulação científica/quantitativa com código próprio (R, Python, C++) que abre arquivos como filesystem.

> *"Filestore offers low latency for file operations, and as capacity or performance needs change, you can easily grow or shrink your instances as needed."*

**Benefício extra mencionado no vídeo:** evita o **load/offload** de dado entre client drives e storage — o dado já está **persistente e compartilhável** desde o início.

#### 5. Genome sequencing

Sequenciamento genético gera **bilhões de data points por pessoa**. As pipelines (BWA, GATK, etc.) leem arquivos enormes (`.bam`, `.vcf`) e escrevem intermediários. Múltiplas etapas em máquinas diferentes precisam compartilhar resultados.

> *"This type of analysis requires speed, scalability, and security."*

#### 6. Web hosting (WordPress e similares)

WordPress, Joomla, Drupal e parecidos guardam **uploads de mídia** num diretório `wp-content/uploads/`. Se você quiser **rodar várias instâncias** do WordPress atrás de um load balancer, todas precisam ver o mesmo `uploads/`. Filestore é o jeito direto de resolver isso.

> Alternativa: reescrever o site pra subir mídia em Cloud Storage (mais limpo, mais barato em escala). Mas isso exige plugin/modificação. Filestore é a opção sem mexer no app.

### Bloco "Tiers do Filestore"

> O vídeo não entra em tiers, mas a prova ACE pode mencionar. Resumo do que existe atualmente.

O Filestore tem múltiplos **service tiers** que afetam capacity, performance e disponibilidade. Os principais (oferta vigente):

| Tier               | Capacidade (range) | Performance        | Resiliência                 | Caso típico                                    |
|--------------------|--------------------|--------------------|-----------------------------|------------------------------------------------|
| **BASIC HDD**      | 1 TB – 64 TB       | Throughput moderado | **Zonal**, sem HA            | Storage barato, workloads pouco sensíveis a latência |
| **BASIC SSD**      | 2,5 TB – 64 TB     | Throughput alto, baixa latência | **Zonal**, sem HA  | Web hosting, dev/test                         |
| **ZONAL**          | Capacity e performance ajustáveis | Throughput escalável | **Zonal**         | Produção média; melhor relação preço-performance |
| **REGIONAL**       | Capacity e performance ajustáveis | Throughput escalável | **Regional** (HA entre zonas) | Produção crítica com tolerância a falha de zona |
| **ENTERPRISE**     | Alta capacidade + features avançadas | Alta performance | **Regional**           | Cargas mission-critical com requisitos de SLA  |

**Notas pra prova:**
- **Zonal** = barato, mas se a zona cair, o file share fica indisponível.
- **Regional** / **Enterprise** = replica entre zonas, mais caro, sobrevive a falha de zona.
- Critério de escolha = mesmo padrão do resto do GCP: **criticidade do workload vs orçamento**.

### Bloco "Como Filestore se encaixa nos workloads do GCP"

#### Montagem em Compute Engine (Linux)

```bash
# 1. Cria a instância Filestore
gcloud filestore instances create nfs-render \
  --zone=us-central1-a \
  --tier=ZONAL \
  --file-share=name="vol1",capacity=2TB \
  --network=name="default"

# 2. Obtém o IP do file share (saída do describe)
gcloud filestore instances describe nfs-render --zone=us-central1-a \
  --format="value(networks.ipAddresses[0])"

# 3. Na VM Linux, instala cliente NFS e monta
sudo apt-get install -y nfs-common
sudo mkdir -p /mnt/render
sudo mount -t nfs <FILESTORE_IP>:/vol1 /mnt/render
```

Após montar, qualquer leitura/escrita em `/mnt/render/` é **vista por todas as VMs** que também montaram o mesmo share.

#### Em GKE — via CSI driver

GKE tem **driver CSI nativo** pro Filestore. Você cria um `PersistentVolumeClaim` com a storage class do Filestore e ganha **`ReadWriteMany`** — o "santo graal" de volumes compartilhados em Kubernetes.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: filestore-shared
spec:
  accessModes: [ReadWriteMany]
  storageClassName: standard-rwx  # provisiona Filestore
  resources:
    requests:
      storage: 1Ti
```

**Por que isso é grande:** Persistent Disk **não** suporta `ReadWriteMany` (RWX). Quase todo pod que precisa de RWX no GKE acaba sentado em cima de Filestore. Esse é, na prática, o caso mais comum de Filestore em produção.

### Bloco "Quando NÃO usar Filestore"

Filestore é caro comparado a Cloud Storage. Antes de escolhê-lo, verifique se o caso **realmente** exige semântica POSIX. Antipatterns:

1. **Servir mídia pela internet** → Cloud Storage + Cloud CDN. Filestore aqui é exagero (e nem tem CDN nativo).
2. **Banco de dados em Filestore** → antipattern. Use Cloud SQL, AlloyDB ou Persistent Disk diretamente. Bancos em NFS sofrem com locks, fsync e performance imprevisível.
3. **Disco-raiz de VM** → use Persistent Disk / Hyperdisk. Filestore é shared storage, não disco bootável.
4. **Compartilhar arquivos com poucos clientes esporádicos** → Cloud Storage com SDK. Filestore só vale a pena com IO contínuo e múltiplos clientes simultâneos.
5. **Backup** → Cloud Storage com lifecycle. Filestore não tem classes frias; pagar Filestore por dado morto é desperdício.

**Regra prática:** se você consegue reescrever o app pra usar Cloud Storage SEM dor, faça isso. Filestore é o caminho do **lift-and-shift** ou de workloads onde POSIX é requisito **não-negociável**.

## Cenário real

**Onde eu usaria Filestore nos meus produtos:**

- **Portaria/TaxIA hoje:** **não usaria**. Imagens do WhatsApp e PDFs ficam ótimos em Cloud SQL/Cloud Storage. Não preciso de NFS.
- **Sistema web de ocorrências (Produto 2 do cenário da empresa):** **não usaria** como default. Mídia em Cloud Storage com Signed URLs é mais barato e escala melhor. **Exceção:** se o app web fosse **legacy PHP** com múltiplos servidores rodando atrás de load balancer **e** o código não suportasse Cloud Storage, Filestore seria caminho de lift-and-shift.
- **Pipeline de processamento de vídeo de ocorrência:** se eu tivesse um job de transcoder/análise de vídeo rodando em fleet de VMs (várias VMs processando frames do mesmo vídeo em paralelo), Filestore faria sentido como buffer temporário. Mas hoje **Transcoder API** + Cloud Storage resolve sem precisar.
- **GKE com pods stateful compartilhando volume RWX:** se algum dia rodar Jenkins/Airflow/CMS no GKE com `ReadWriteMany`, Filestore vira candidato natural — não há alternativa boa.

**Armadilha que cometeria na vida real:** subir Filestore "por garantia" no dia 1 porque "talvez no futuro vamos precisar". Filestore custa **caro** mesmo ocioso. Se o caso de uso não exige POSIX agora, espere. Migração de Cloud Storage pra Filestore depois é trabalhosa, mas começar com Filestore desnecessariamente queima caixa.

## Armadilhas de prova

1. **"Shared file system" → Filestore.** Determinístico. Nunca Cloud Storage.
2. **"NFSv3" / "NFS" → Filestore.** Mesma coisa.
3. **"File locking" → Filestore.** Cloud Storage não tem.
4. **"POSIX semantics" → Filestore.** Cloud Storage não tem.
5. **Múltiplas VMs com `ReadWriteMany` (RWX) → Filestore.** Persistent Disk não suporta RWX em modo escrita.
6. **Lift-and-shift de app legado com NFS on-prem → Filestore.** Não tente reescrever pra Cloud Storage no contexto da DQ.
7. **Capacity e performance escalam independentemente.** Não é como Persistent Disk (onde performance é função do tamanho).
8. **Filestore NÃO é object storage.** Não tem URL pública, não tem storage classes (Standard/Nearline/Coldline/Archive), não tem lifecycle policy de classe.
9. **Filestore NÃO é Persistent Disk.** PD é bloco para 1 VM (RW); Filestore é file system para N VMs.
10. **Filestore Basic vs Zonal vs Regional vs Enterprise** — Regional/Enterprise são HA entre zonas; Basic e Zonal são zonais. Em DQ que exige HA, escolha Regional/Enterprise.
11. **Cloud Storage FUSE (`gcsfuse`) NÃO substitui Filestore.** Monta bucket como filesystem, mas sem semântica POSIX completa. Se a DQ exige locking, transações em filesystem, ou compatibilidade NFS real → Filestore.
12. **Use cases canônicos memorizados:**
    - Migração de enterprise app legado
    - Media rendering / VFX
    - EDA (chip design)
    - Data analytics latency-sensitive
    - Genome sequencing
    - WordPress / web hosting multi-instância
