# 2.3 — VPC Fundamentals (Virtual Networks)

> **Fonte:** Essential Google Cloud Infrastructure: Foundation — M2 (Virtual Networks)

---

## Rede global do Google Cloud

- **SDN** sobre fibra global + cabos submarinos
- **Region** = local geográfico; a maioria tem 3 zonas. Exceção: **us-central1 (Iowa) = 4 zonas** (a/b/c/f)
- **PoPs** = pontos onde a rede do Google encontra a internet

---

## VPC — visão geral

VPC = conjunto de objetos gerenciados pelo Google para conectar, isolar e definir políticas de rede.

**Objetos:** Projects · Networks · Subnetworks · Regions/Zones · IPs · VMs · Routes · Firewall Rules.

---

## Projects → Networks

- **Project** organiza recursos e billing. **Quota default: 15 networks/project**.
- Networks são **globais** (atravessam regions). **Subnetworks são regionais** (cruzam zonas dentro da region).

### Três tipos de Network

| Tipo | Característica |
|---|---|
| **Default** | Vem com o project. É uma auto mode. Subnet por region + firewall rules pré (ICMP/RDP/SSH + tráfego interno). |
| **Auto mode** | Cria 1 subnet por region automaticamente. Ranges `/20` (expansível a `/16`) dentro do CIDR **`10.128.0.0/9`**. Novas regions ganham subnet auto. |
| **Custom mode** | Você controla subnets, regions e ranges. Sem sobreposição. Suporta **IPv6 (dual-stack)**. |

**Auto → Custom**: permitido. **Custom → Auto**: ❌ one-way.

---

## Subnetworks

- Subnet = range de IPs regional. Cruza zonas → uma firewall rule serve ambas.
- **4 IPs reservados** em todo range:
  - `.0` network · `.1` gateway · penúltimo reservado · último broadcast
  - Primeira VM recebe `.2`

### Expansão de subnet (sem downtime)

- Só dá pra **aumentar** o range (prefix menor). Não dá pra encolher (one-way).
- Não pode sobrepor outras subnets da mesma VPC.
- Auto mode: `/20` até `/16`. Pra ir além → converter para custom.
- Evitar subnets grandes demais → risco de colisão CIDR em peering/VPN.

### Comunicação entre VMs

- **Mesma network** → IPs internos, mesmo entre regions (usa fibra global).
- **Networks diferentes** → IPs externos (mesmo na mesma region). Tráfego vai por Google Edge routers, não internet pública — mas tem implicações de billing/segurança.

---

## IP Addresses

Cada VM tem até 2 IPs:

### Internal IP (obrigatório)
- Atribuído via **DHCP**
- Resolvido pelo **Internal DNS** (scoped à network)

### External IP (opcional)
| Tipo | Comportamento |
|---|---|
| **Ephemeral** | Do pool. **Muda em stop/start**. |
| **Static (reservado)** | Persistente entre restarts. |
| **None** | VM sem acesso externo. |

⚠️ **Static reservado e NÃO usado custa MAIS caro** que static em uso (penalidade anti-desperdício).

**BYOIP:** trazer seu próprio bloco `/24` ou maior como external IP do GCP.

### Stop/Start (comportamento prático)
- Stop → external IP some, internal **fica reservado**. Aviso de force após **90s** (tempo para shutdown scripts).
- Start → internal mantém; external ephemeral **muda**.

### Quotas
- ~4.000 IPs num `/20`, mas **quota = 15.000 instâncias/network**. Range grande ≠ poder criar tudo.

---

## External IP é invisível ao OS

`ifconfig` dentro da VM mostra **só o internal IP**. VPC mantém lookup table mapeando external ↔ internal de forma transparente.

---

## DNS

### Internal DNS — 2 tipos
- **Zonal DNS** ← **recomendado pelo Google** (isola falhas por zona)
- **Global DNS** (project-wide)

- Hostname = instance name. FQDN segue formato padrão.
- DNS name **persiste** ao recriar VM, mesmo que o IP mude.
- Cada VM tem um **metadata server** que age como DNS resolver local + repassa queries públicas pro DNS público do Google.

### External DNS
- Registros públicos **NÃO são publicados automaticamente** — admin precisa fazer (servidor próprio ou Cloud DNS).

### Cloud DNS
- DNS authoritative managed, escalável.
- **Anycast** global → baixa latência + alta disponibilidade.
- **SLA 100% uptime** para domínios configurados.
- UI / CLI / API.

---

## Alias IP Ranges

- Atribui um **range de IPs internos** como alias a uma única NIC.
- Útil quando múltiplos serviços/containers numa mesma VM precisam de IPs próprios sem criar NICs adicionais.
- Range tirado do **primary ou secondary CIDR** da subnet.

---

## Gotchas pra prova

1. **us-central1 = 4 zonas** (única região fora do padrão de 3).
2. **Default network = auto mode** (não é custom!).
3. **Auto → Custom é one-way.**
4. Quota default: **15 networks/project** · **15.000 VMs/network**.
5. **4 IPs reservados** em toda subnet (.0, .1, penúltimo, último).
6. **Auto mode CIDR = `10.128.0.0/9`**, subnets `/20` (até `/16`).
7. **Expansão de subnet é one-way** (não dá pra reduzir).
8. **Static external IP reservado mas não usado é mais caro** que em uso.
9. **External IP é invisível ao OS** (só lookup table do VPC).
10. **Zonal DNS > Global DNS** (recomendação oficial).
11. **Cloud DNS SLA = 100%**.
12. **Internal DNS é scoped à network** — não resolve VMs de outras networks.
13. **Ephemeral external IP muda em stop/start** — use static se precisa persistir.
14. **BYOIP exige bloco /24 ou maior**.
15. VMs em **networks diferentes** comunicam por IPs externos mesmo que estejam na mesma region.
