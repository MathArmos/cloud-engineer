# 2.2 — Spanner

> **DQs do workbook cobertas:** nenhuma diretamente. Spanner aparece como **resposta certa** sempre que o cenário tem "global", "horizontal scaling em relacional", "strong consistency worldwide", "outgrew sharded MySQL".
> **Considerações da prova:** o exame ACE raramente pede arquitetura interna do Spanner — mas pede **quando usar Spanner em vez de Cloud SQL**. Esse arquivo foca nessa diferença e nas pistas léxicas que viram a resposta.

---

## Conceitos

### Bloco "O problema que Spanner resolve"

Spanner nasce de uma tensão histórica entre dois mundos:

| Mundo                | Pontos fortes                                      | Pontos fracos                                         |
|----------------------|----------------------------------------------------|-------------------------------------------------------|
| **Relacional (SQL)** | Schema, joins, transações ACID, integridade        | Escala vertical — escrita é um único nó               |
| **NoSQL**            | Escala horizontal massiva, alta disponibilidade    | Sem schema rígido, sem transações cross-row globais, consistência fraca/eventual |

**Antes do Spanner**, quem tinha um app relacional que crescia além do limite de um único servidor era forçado a uma das três soluções abaixo — todas dolorosas:

1. **Sharding manual** — quebrar o banco em pedaços ("usuários A-M ficam no shard 1, N-Z no shard 2") e a aplicação sabe em qual shard ir. Funciona, mas vira um inferno operacional: joins entre shards, rebalanceamento, transações cross-shard.
2. **Read replicas** — espalhar leitura, mas escrita continua centralizada (gargalo).
3. **Migrar pra NoSQL** — perder schema, perder transações ACID, reescrever a aplicação.

**Spanner é a quarta saída:** banco **relacional de verdade** (schema, SQL, ACID) que **escala horizontalmente** sem o app saber, com **consistência forte global**.

> *"Spanner is a service built for the cloud, specifically to combine the benefits of relational database structure with non-relational horizontal scale."*

#### Onde Spanner se posiciona na árvore de decisão

```
                     Dado relacional?
                          | SIM
                          v
                 Precisa de HTAP?
                          | NÃO
                          v
               Precisa de escala horizontal
               OU distribuição global
               OU strong consistency mundial?
                          |
                  +-------+--------+
                  | SIM            | NÃO
                  v                v
              Spanner          Cloud SQL
```

A diferença Cloud SQL ↔ Spanner é **decidida por uma única pergunta**: você precisa que **escrita** escale horizontalmente, ou que dados estejam **fortemente consistentes em múltiplos continentes**? Se sim, Spanner. Se não, Cloud SQL — mais simples, mais barato.

### Bloco "Capacidades anunciadas no vídeo"

| Capacidade                                          | Detalhe                                                      |
|-----------------------------------------------------|--------------------------------------------------------------|
| **Petabytes de capacidade**                         | Cresce muito além do teto de Cloud SQL (64 TB por instância) |
| **Transactional consistency at global scale**       | Transações ACID atravessam regiões e continentes              |
| **Schemas, SQL**                                    | Linguagem SQL padrão (dialeto Google SQL + dialeto PostgreSQL) |
| **Automatic synchronous replication**               | Você não escreve nenhuma linha de código de replicação        |
| **High availability**                               | SLA até **99,999%** em multi-region (consultar docs para o número vigente) |

**Como decorar a lista para a prova:**

Spanner = **relacional clássico (schema + SQL + ACID) + horizontal não-relacional (HA + replicação + escala) + algo novo (consistência forte global)**.

Se a pergunta tem **todas essas pistas juntas**, é Spanner. Cloud SQL não dá horizontal de escrita, Bigtable não dá schema/SQL/joins, BigQuery não é OLTP, Firestore não dá joins relacionais.

#### SLA: regional vs multi-regional

> *"Depending on whether you create a multi-regional or regional instance, you'll have different monthly uptime SLAs. For up-to-date numbers, you should always refer to the documentation."*

**Forma do SLA (números podem variar — sempre confirmar nas docs):**

| Configuração      | SLA típico                | O que significa                                                |
|-------------------|---------------------------|----------------------------------------------------------------|
| **Regional**      | ~99,99%                   | ~52 minutos de downtime/ano permitido. Réplicas em zonas da mesma região. |
| **Multi-regional**| ~99,999% ("five nines")  | ~5 minutos de downtime/ano. Réplicas espalhadas em múltiplas regiões. |

**Trade-off:** multi-regional é mais caro e tem **latência de escrita ligeiramente maior** (precisa replicar entre regiões antes de commitar). Regional é mais barato e tem escrita mais rápida, mas perde a região inteira → indisponível.

**Armadilha de prova:** *"99,999% availability"* é pista textual quase determinística para **Spanner multi-regional**. Cloud SQL não chega lá nem com HA.

### Bloco "Comparativo Spanner vs Relacional vs NoSQL"

O instrutor coloca esse comparativo explícito no vídeo. Memorizar:

| Característica          | Banco relacional clássico | Banco NoSQL          | **Spanner**          |
|-------------------------|---------------------------|----------------------|----------------------|
| Schema                  | ✅                        | ❌ (em geral)        | ✅                   |
| SQL                     | ✅                        | ❌ (geralmente API)  | ✅                   |
| Consistência forte      | ✅ (dentro do nó)         | ❌ (eventual)        | ✅ **global**        |
| Alta disponibilidade    | Parcial (HA single-region)| ✅                   | ✅                   |
| **Escalabilidade horizontal** | ❌ (vertical only)  | ✅                   | ✅                   |
| Replicação configurável | Limitada                  | ✅                   | ✅                   |

> *"Spanner offers the best of the relational and non-relational worlds."*

**Tradução prática:** Spanner é o **único banco** que entrega **ACID + SQL + escala horizontal**. Por isso é caro (e por isso a Google montou esse produto inteiro — não é trivial).

#### Use cases canônicos

> *"These features allow for mission-critical use cases, such as building consistent systems for transactions and inventory management in the financial services and retail industries."*

- **Financial services / FinTech:** sistemas de transações onde "consistência forte global" não é luxo, é regulação. Não dá pra ter saldo eventualmente consistente.
- **Inventory / retail global:** estoque de e-commerce em múltiplos continentes. Se um produto está no estoque, **todo cliente em todo lugar tem que ver o mesmo número** — senão você vende o que não tem.
- **AdTech:** orçamento de campanha consumido por bidders distribuídos. Não pode estourar o budget porque dois bidders viram o mesmo valor.
- **Gaming global:** estado de jogo persistente compartilhado entre jogadores em diferentes continentes.
- **Marketplaces planetários:** Uber, Airbnb, etc. — eventos transacionais com latência baixa em qualquer lugar.

**Padrão comum:** **escala + consistência + dinheiro / inventário envolvido**. Quando errar dói (perda financeira ou regulatória), Spanner ganha a discussão.

### Bloco "Arquitetura — como Spanner consegue escalar e ser consistente"

Esse bloco vai um pouco além do vídeo, mas é o que faz a história fazer sentido. A prova ACE não cobra os internals, mas entender ajuda a memorizar **por que Spanner é caro e quando faz sentido**.

#### O esqueleto

> *"A Spanner instance replicates data in N cloud zones, which can be within one region or across several regions."*

- A unidade básica é um **Spanner instance** (não confundir com "instance" de Cloud SQL — aqui é mais um cluster gerenciado).
- Dentro da instância, o dado é dividido em **splits** (pedaços de tabela). Cada split é replicado em **N zonas**.
- As zonas podem estar **na mesma região** (configuração regional) ou **espalhadas em várias regiões** (configuração multi-regional).
- Você escolhe a configuração na criação. Mudar depois é possível mas trabalhoso.

```
Spanner instance "fintech-prod" (multi-regional nam-eur-asia1)
┌──────────────────────────────────────────────────────────┐
│   us-east1 (3 zonas)  ─────  eu-west1 (3 zonas)  ─────  asia-east1 (3 zonas)
│        Réplica A              Réplica B                    Witness
│                                                            (vota mas não serve)
│
│   Splits de tabela distribuídos automaticamente
│   Replicação síncrona via Paxos
└──────────────────────────────────────────────────────────┘
```

#### Replicação síncrona via fibra global da Google

> *"The replication of data will be synchronized across zones using Google's global fiber network."*

- Spanner usa **Paxos** (algoritmo de consenso) para garantir que toda escrita é confirmada por **maioria das réplicas** antes do commit.
- A rede que conecta as zonas é a **rede de fibra privada da Google** (a mesma que serve Search, Gmail, YouTube). Isso é o que torna a latência inter-regional **viável** — em uma cloud sem fibra própria, isso não funcionaria.

**Implicação:** Spanner é caro **porque essa infra é cara**. Você não está pagando só pelo software — está pagando pela rede global e pelos atomic clocks (próximo bloco).

#### TrueTime — o segredo da consistência global

> *"Using atomic clocks ensures atomicity whenever you are updating your data."*

O vídeo só toca de leve — mas isso é o coração técnico do Spanner. Vale entender.

**Problema:** se você tem servidores em New York e em Tóquio, e cada um tem seu próprio relógio, dois eventos podem aparentar acontecer "no mesmo segundo" mesmo estando milissegundos separados. Em transações financeiras, isso é catastrófico (qual transação aconteceu primeiro? Quem fez o débito antes do crédito?).

**Solução do Spanner:** **TrueTime** — um serviço de tempo distribuído baseado em **relógios atômicos** + **GPS** nos data centers da Google. Cada chamada de "que horas são?" retorna não um timestamp, mas um **intervalo** `[ts_min, ts_max]` com a garantia de que o tempo real está dentro do intervalo.

Quando o Spanner commita uma transação, ele espera o **intervalo de incerteza passar** (typically 1-7 ms). Isso permite ordenar transações globalmente com certeza — duas transações que parecem simultâneas são ordenadas pela TrueTime.

**Por que isso importa pra prova:**
- Você não vai ser cobrado em "explique o TrueTime", mas...
- Pode aparecer **"strong consistency at global scale"** como pista — só Spanner entrega isso entre os bancos do GCP.
- Bigtable, Firestore, BigQuery: nenhum oferece consistência forte **global**. Spanner é único.

**Analogia:** imagine dois cartórios em cidades diferentes registrando transações. Sem um relógio comum confiável, é impossível dizer qual transação foi "primeiro" entre eles. TrueTime é o "relógio universal compartilhado" entre todos os data centers da Google.

#### Database placement configurável

> *"The database placement is configurable, meaning you can choose which region to put your database in."*

- Você escolhe a **configuração** (regional ou multi-regional) e **quais regiões** participam.
- Configurações multi-regionais pré-definidas pela Google (ex.: `nam-eur-asia1` cobre América do Norte, Europa, Ásia).
- Trade-off: mais regiões = mais resiliência + maior latência de escrita.

### Bloco "Decision tree do final do vídeo"

> *"If you've outgrown any relational database, are sharding your databases for throughput / high performance, need transactional consistency, global data and strong consistency, or just want to consolidate your database, consider using Spanner."*

#### Os 5 gatilhos canônicos para Spanner

| Gatilho                                                            | Tradução de cenário                                              |
|--------------------------------------------------------------------|------------------------------------------------------------------|
| **"Outgrew any relational database"**                              | Cloud SQL / Postgres / MySQL chegou no teto de uma instância     |
| **"Sharding for throughput / high performance"**                   | Já está fazendo sharding manual e quer parar de sofrer            |
| **"Transactional consistency"**                                    | Precisa de ACID e a alternativa "NoSQL eventual" não serve        |
| **"Global data and strong consistency"**                           | Multi-região com leitura/escrita consistente em qualquer lugar    |
| **"Consolidate your database"**                                    | Tem 5 Cloud SQLs por região, quer unificar num só banco global    |

**Se NENHUM desses gatilhos está presente** → não é Spanner. Provavelmente é Cloud SQL (se relacional) ou Firestore/Bigtable (se NoSQL).

> *"If you don't need any of these, nor full relational capabilities, consider a NoSQL service, such as Firestore, which we will cover next."*

#### Quando NÃO usar Spanner

- **Aplicação regional simples** → Cloud SQL é **muito** mais barato.
- **Volume pequeno (<10 GB de dado, <1000 QPS)** → Cloud SQL ou Postgres em VM resolvem. Spanner cobra por nó, e o piso ainda é considerável.
- **Você não precisa de schema/SQL** → Firestore ou Bigtable são mais simples e baratos.
- **OLAP / analítico ad-hoc** → BigQuery.
- **HTAP** → AlloyDB.
- **Não pode pagar a complexidade de modelagem do Spanner** — modelar interleaved tables, evitar hotspots de PK, etc. requer expertise.

**Armadilha frequente:** times pequenos sendo seduzidos pela arquitetura do Spanner e indo direto pra ele "porque é a tecnologia da Google". Resultado: gastam 10x mais do que precisariam, com complexidade extra que não usam.

#### Migração de MySQL → Spanner

> *"If you're now convinced that using Spanner as a managed service is better than using or re-implementing your existing MySQL solution, refer to the documentation for a solution on how to migrate from MySQL to Spanner."*

- Spanner aceita **dialeto Google SQL** (default) ou **dialeto PostgreSQL** (`POSTGRESQL_DIALECT`).
- Para migração, ferramentas oficiais: **Database Migration Service**, **Harbourbridge** (open source da Google), **Datastream** + Dataflow.
- Não é "lift-and-shift" no sentido literal — você precisa adaptar schema (PKs distribuídas, interleaved tables, evitar AUTO_INCREMENT) e re-validar queries.

**A prova ACE não cobra detalhes de migração**, mas pode citar que **Spanner suporta Postgres dialect** — isso ajuda quando o time já tem expertise Postgres.

### Bloco "Comparativo direto Cloud SQL vs Spanner"

Esse comparativo é a única coisa que **precisa estar absolutamente clara** para o ACE.

| Eixo                          | Cloud SQL                          | Spanner                                  |
|-------------------------------|------------------------------------|------------------------------------------|
| **Tipo**                      | Relacional gerenciado clássico     | Relacional distribuído proprietário      |
| **Engines suportados**        | MySQL, Postgres, SQL Server        | Google SQL (dialect) ou Postgres dialect |
| **Escopo**                    | Regional (HA = multi-zonal)        | Regional **ou** multi-regional / global  |
| **Escala de escrita**         | Vertical (1 nó primary)            | Horizontal (N nodes)                     |
| **Escala de leitura**         | Vertical + read replicas (async)   | Horizontal (replicação síncrona)         |
| **Capacidade max**            | 64 TB / instância                  | Petabytes                                |
| **Consistência**              | Forte dentro do nó                 | Forte **globalmente**                    |
| **SLA típico**                | ~99,95% (HA)                       | ~99,99% regional / ~99,999% multi-region |
| **Pricing model**             | Por instância (CPU/RAM/disk)       | Por **node** (unidade de capacidade)     |
| **Custo de entrada**          | Baixo                              | **Alto** (piso de node)                  |
| **Use case típico**           | App web regional, ERP, e-commerce regional | FinTech global, AdTech, inventário global |
| **Quando faz sentido**        | Não preciso global nem horizontal  | Preciso global, horizontal ou ambos       |

#### Heurística pra responder rápido no exame

Leia o cenário e pergunte na ordem:

1. **Tem palavra "global" / "worldwide" / "multi-region"?** → Spanner.
2. **Tem "horizontal scale of writes" / "sharded"?** → Spanner.
3. **Tem "strong consistency at scale"?** → Spanner.
4. **Tem só "regional" / "small to medium"?** → Cloud SQL.
5. **Não tem nenhuma pista de relacional?** → não é nenhum dos dois — provavelmente Bigtable, Firestore, BigQuery ou Memorystore.

### Comandos `gcloud` críticos (referência)

#### Criar uma instância Spanner regional

```bash
gcloud spanner instances create fintech-prod \
  --config=regional-us-central1 \
  --description="Fintech production database" \
  --nodes=1
```

**Flags importantes:**
- `--config=regional-us-central1` → configuração **regional** (HA multi-zonal dentro da região).
- `--config=nam-eur-asia1` → exemplo de configuração **multi-regional** (3 continentes).
- `--nodes=1` → 1 node de capacidade. Cada node atende ~10.000 QPS de leitura, ~2.000 QPS de escrita, e suporta ~2 TB de storage (números aproximados — confirmar nas docs).
- Alternativa moderna: `--processing-units=100` (PU = 1/1000 de um node — granularidade mais fina para cargas pequenas).

#### Criar um banco com dialeto Postgres

```bash
gcloud spanner databases create app-db \
  --instance=fintech-prod \
  --database-dialect=POSTGRESQL
```

**Por que importa:** se o time tem expertise Postgres, escolher esse dialeto reduz a curva de aprendizado.

#### Listar configurações disponíveis

```bash
gcloud spanner instance-configs list
```

**Retorna:** `regional-us-central1`, `regional-europe-west1`, `nam3` (multi-region North America), `nam-eur-asia1` (3 continentes), etc.

#### Executar uma query

```bash
gcloud spanner databases execute-sql app-db \
  --instance=fintech-prod \
  --sql="SELECT * FROM users LIMIT 10"
```

#### Backup e restore

```bash
# Backup on-demand
gcloud spanner backups create users-backup-2026-05 \
  --instance=fintech-prod \
  --database=app-db \
  --retention-period=30d

# Restore para um novo banco
gcloud spanner databases restore \
  --source-backup=users-backup-2026-05 \
  --source-instance=fintech-prod \
  --destination-database=app-db-restored \
  --destination-instance=fintech-prod
```

## Cenário real

### Quando você usaria Spanner nos seus projetos?

**Portaria / TaxIA:** hoje **não justifica**. São apps regionais, com volume típico de SaaS B2B brasileiro. Cloud SQL Postgres atende com folga e a um custo muito menor.

**Quando viraria a chave para Spanner:**

- Se o produto Portaria fosse adotado por **redes hoteleiras multinacionais**, com hóspedes acessando portarias em vários países simultaneamente, e a operação precisasse de **estado consistente global** (ex.: um hóspede tem direito a 1 entrada por dia, e não pode usar duas portarias em continentes diferentes ao mesmo tempo).
- Se o TaxIA precisasse rodar **transações financeiras reais** (não só conversas IA) — pagamentos, débitos, créditos com consistência forte regulamentada.

**Cenário canônico fora dos seus projetos:**

- **Uber/iFood-style:** estado de corrida/pedido distribuído globalmente. Dois motoristas/entregadores não podem "aceitar" o mesmo pedido — Spanner garante.
- **Bancos digitais com presença internacional:** Nubank, Revolut, Wise. Saldo e transações precisam ser globalmente consistentes.
- **AdTech (Google Ads é o caso da casa):** budget de campanha consumido por bidders no mundo todo, sem estouro.

### O erro clássico do "Spanner por hype"

Times com 10 GB de dado relacional e 100 QPS escolhendo Spanner "porque escala". Resultado:

- Pagam **$650+/mês** no piso de 1 node (ou $65+/mês com 100 processing units — ainda muito mais caro que Cloud SQL pra mesmo volume).
- Complexidade de modelagem que não usam (interleaved tables, primary keys distribuídas, evitar hotspots).
- Acabam usando 5% das capacidades do Spanner.

**A regra:** vá pro Spanner **quando o problema literalmente exige** — não como flex tecnológico.

### Padrão de combinação Spanner + outros

- **Spanner + BigQuery:** OLTP no Spanner, export periódico (ou streaming via Datastream / Pub/Sub) pro BigQuery para análise histórica.
- **Spanner + Memorystore:** Spanner é forte mas tem latência típica de transação distribuída (poucos ms até dezenas de ms). Memorystore na frente serve leitura quente sub-ms.
- **Spanner + Cloud CDN:** dados públicos derivados (catálogos, leaderboards) cacheados na borda para latência mínima ao cliente final.

## Armadilhas de prova

1. **Palavra "global" + "relacional" + "strong consistency" → Spanner.** Determinístico. Cloud SQL não cobre isso, BigQuery não é transacional, Bigtable não tem schema/SQL relacional.

2. **"Horizontal scaling of writes" → Spanner.** Cloud SQL escala leitura via read replicas mas **não escreve em paralelo**. Spanner sim.

3. **"Sharding manual / outgrew relational" → Spanner.** O caso de uso clássico do produto.

4. **99,999% ("five nines") availability → Spanner multi-regional.** Cloud SQL não chega lá.

5. **Spanner ≠ Cloud SQL "premium".** Não é "Cloud SQL maior". É arquitetura diferente, pricing diferente, dialeto SQL diferente (mesmo com a opção Postgres dialect, há diferenças).

6. **Spanner cobra por NODE (ou processing units), não por CPU/RAM da instância.** Pricing model é distinto. Se a pergunta foca em "low cost for small workload", **Cloud SQL** vence.

7. **Spanner regional ≠ Spanner multi-regional.**
   - Regional = HA dentro de uma região (multi-zonal), latência baixa, mais barato.
   - Multi-regional = sobrevive a falha regional, SLA maior, latência de escrita ligeiramente maior, **muito mais caro**.

8. **Replicação no Spanner é síncrona via Paxos.** Não é "eventual" como em alguns NoSQL. Por isso "strong consistency" é garantida.

9. **TrueTime / atomic clocks** raramente caem no exame por nome, mas a **consequência** (consistência ordenada global) é o que diferencia Spanner.

10. **Spanner NÃO é a primeira escolha "default" para SQL.** O default é Cloud SQL. Spanner aparece quando o cenário **explicitamente** pede escala/global.

11. **Spanner aceita SQL, mas o dialeto importa.** Google SQL dialect (default) e PostgreSQL dialect. Se você migrar de Postgres e quer minimizar mudança, escolha PG dialect.

12. **Migração de MySQL → Spanner não é lift-and-shift.** Requer adaptar schema (sem AUTO_INCREMENT, PKs distribuídas, interleaved tables). Se a pergunta diz "minimum effort migration from MySQL", **Spanner não é a resposta** — Cloud SQL MySQL é.

13. **Spanner não substitui BigQuery para análise pesada.** Tem SQL, mas é OLTP. Queries analíticas ad-hoc com varreduras gigantes pertencem ao BigQuery.

14. **Spanner ≠ AlloyDB.** Os dois têm "SQL gerenciado", mas:
    - **AlloyDB** = Postgres-compatible + HTAP, **regional** (não global).
    - **Spanner** = SQL distribuído global + escala horizontal, não é Postgres por baixo.

15. **"Consolidate multiple regional databases into one global database" → Spanner.** Esse é um dos 5 gatilhos canônicos.

16. **Spanner não tem AUTO_INCREMENT clássico.** Use UUID ou bit-reversed sequential keys para evitar hotspots de escrita. A prova não cobra isso em detalhe mas pode aparecer como pista de "design difference".
