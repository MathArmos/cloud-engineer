## Conceitos

### Bloco "O que é Bigtable — definição precisa"

> *"Bigtable is a fully managed NoSQL database with petabyte-scale and very low latency."*

Bigtable é o NoSQL **wide-column** do Google. Algumas características que importam pra prova:

| Característica          | Detalhe                                                            |
|-------------------------|--------------------------------------------------------------------|
| **Tipo**                | NoSQL wide-column store (não é document, não é key-value puro)     |
| **Escala**              | **Petabytes** (muito além de Firestore e Cloud SQL)                |
| **Latência**            | **<10 ms** read/write consistente                                  |
| **Consistência**        | **Forte** dentro de um cluster                                     |
| **Throughput**          | Linear: cada node adiciona ~10.000 ops/s. Centenas de nodes ⇒ milhões de ops/s |
| **Modelo de cobrança**  | Pay-per-node (você paga pelos nodes provisionados, **não escala a zero**) |
| **API aberta**          | **HBase API** compatible — facilita migração de quem já usa HBase  |
| **Pedigree**            | É o **mesmo banco** que roda Google Search, Analytics, Maps, Gmail |

#### A frase mais importante do vídeo

> *"Bigtable is actually the same database that powers many of Google's core services, including Search, Analytics, Maps, and Gmail."*

Isso não é só marketing. O Bigtable do GCP é literalmente a versão gerenciada do mesmo storage interno do Google. Por isso ele aguenta os volumes que aguenta — foi testado em produção há mais de 15 anos antes de virar produto público.

#### Casos de uso canônicos do vídeo

> *"Bigtable is a great choice for both operational and analytical applications, including IoT, user analytics, and financial data analysis, because it supports high read and write throughput at low latency."*

- **IoT** — milhões de sensores escrevendo telemetria por segundo.
- **User analytics** — clickstream, events, behavior tracking em volume massivo.
- **Financial data analysis** — séries temporais de cotações, transações, posições.
- **Machine Learning** — *"It's also a great storage engine for machine learning applications."* — feature store, vetores de embeddings, treinamento offline.
- **Ad-tech** (não dito explicitamente mas é o caso clássico) — bidding logs, attribution data.

**Padrão comum em todos:** muito dado, muita escrita, lookup por chave previsível (ID + tempo, geralmente), e **sem necessidade de joins relacionais**.

### Bloco "Storage model — wide-column explicado"

> *"Bigtable stores data in massively scalable tables, each of which is a sorted key/value map."*

A imagem do vídeo (rede social fictícia de presidentes dos EUA):

```
                          Column family "Follows"
              ┌─────────────────────────────────────────────────────┐
              │                                                     │
Row Key       │ gwashington │ jadams │ tjefferson │ wmckinley       │
──────────────┼─────────────┼────────┼────────────┼──────────────────
gwashington   │             │   1    │            │                 │
jadams        │      1      │        │     1      │                 │
tjefferson    │      1*     │   1    │    1*      │       1         │
wmckinley     │             │        │     1      │                 │
                    ↑                       ↑
                    multiple versions (cells com timestamps diferentes)
```

#### Conceitos centrais

| Conceito           | Definição                                                                              |
|--------------------|----------------------------------------------------------------------------------------|
| **Row**            | Uma entidade (no exemplo, um presidente). Indexada por **uma única** row key.          |
| **Row key**        | A chave primária — única, ordenada, é **o único índice** do Bigtable.                  |
| **Column family** | Grupo lógico de colunas relacionadas (ex.: `Follows`). Definida no schema da tabela.   |
| **Column qualifier**| Nome único dentro da column family (ex.: `gwashington`). **Pode ser criado dinamicamente**. |
| **Column**         | Combinação `family:qualifier` (ex.: `Follows:gwashington`).                            |
| **Cell**           | Interseção row × column. Pode conter **múltiplas versões** (timestamps).               |
| **Sparse**         | Células vazias **não ocupam espaço**. Pode ter milhões de colunas sem custo.           |

#### O que o vídeo destaca sobre o design do exemplo

> *"The table contains one column family, the 'follows' family. This family contains multiple column qualifiers. Column qualifiers are used as data. This design choice takes advantage of the sparseness of Bigtable tables, and the fact that new column qualifiers can be added as your data changes."*

**Padrão idiomático em Bigtable:** usar o **nome do column qualifier** para representar dados que em SQL seriam linhas em uma tabela de relacionamento. No exemplo:
- Em SQL relacional, faria `follows(follower_id, followed_id)` — uma linha por relacionamento.
- Em Bigtable, faz `Follows:<username_followed> = 1` — uma coluna por relacionamento, dentro da row do follower.

**Vantagem:** lookup ultrarrápido — buscar todos os "follows" de um usuário é **uma leitura de row**, sem joins. Adicionar novo follow = adicionar nova coluna esparsa.

**Quando isso vence vs schema fixo:**
- Quando o número de "colunas" potenciais é grande mas a maioria é esparsa.
- Quando você sempre lê **tudo de uma row** (não filtra por valor de coluna específica).
- Quando o padrão de acesso é determinístico pela row key.

#### Row key — o conceito MAIS importante

> *"The username is used as the row key. Assuming usernames are evenly spread across the alphabet, data access will be reasonably uniform across the entire table."*

**Por que isso importa:**
- Bigtable distribui **rows ordenadas** entre nodes. Rows próximas ficam no mesmo tablet (próximo bloco).
- Se sua row key é **mal escolhida** (ex.: timestamp puro, ou sequência incremental), todas as escritas novas vão para o **mesmo node** → **hotspot** → throughput cai a 1/N do potencial.
- Row keys **bem distribuídas** = throughput linear com número de nodes.

**Anti-padrões clássicos:**
- `timestamp` como prefixo da row key → todas as escritas novas no mesmo tablet.
- `auto_increment` numérico → mesmo problema.
- Hash MD5 do dado → distribui mas perde a propriedade de "rows próximas no espaço lógico ficam juntas" (você quebra range scans).

**Padrões bons:**
- `<userId>#<reverseTimestamp>` → distribuído por usuário + ordenado por tempo dentro do usuário (clássico para séries temporais).
- `<deviceId>#<event_type>#<timestamp>` → para IoT.
- `<hashed_prefix>#<real_key>` → quando precisa distribuir um row key naturalmente sequencial.

> **Aviso do exame:** o ACE **não cobra design profundo de row key** (isso é tema de Data Engineering). Mas a **noção** de que row key influencia distribuição/hotspots pode aparecer em pergunta de "why is my Bigtable cluster slow?".

#### Multi-version cells

> *"Each row/column intersection can contain multiple cells, or versions, at different timestamps, providing a record of how the stored data has been altered over time."*

Cada célula guarda **histórico**. Você pode:
- Ler a versão mais recente (padrão).
- Ler uma versão específica por timestamp.
- Ler todas as versões.
- Configurar **GC policy** (manter últimas N versões, ou versões dos últimos X dias).

**Casos canônicos:** auditoria, "ver estado do mundo em tal momento", change feed.

### Bloco "Arquitetura — processing separado de storage"

> *"Processing, which is done through a frontend server pool and nodes, is handled separately from the storage."*

```
        Clients (apps, jobs, BigQuery, Dataflow, etc.)
            │       │       │       │       │       │
            └──┬────┴───┬───┴───┬───┴───┬───┴────┬──┘
               │        │       │       │        │
              (qualquer client → qualquer node — não há "primary")
               │        │       │       │        │
            ┌──┴────────┴───────┴───────┴────────┴──┐
            │   Bigtable nodes (processing layer)    │
            │                                        │
            │   Node A     Node B     Node C ...     │
            └─────┬──────────┬──────────┬────────────┘
                  │          │          │
                  ▼          ▼          ▼
            ┌────────────────────────────────────┐
            │       Colossus file system          │
            │   (SSTables — dados persistidos)    │
            └────────────────────────────────────┘
```

#### Os 3 conceitos-chave dessa arquitetura

##### 1. Tablets — como a tabela é dividida

> *"A Bigtable table is sharded into blocks of contiguous rows, called tablets, to help balance the workload of queries."*

- A tabela é particionada em **tablets** — pedaços de **rows contíguas** (ordenadas pela row key).
- Cada tablet é uma "fatia" do espaço de row keys (ex.: tablet 1 = keys A-G, tablet 2 = H-N, ...).
- Cada node "possui" um conjunto de tablets em dado momento.

**Por que isso é importante:** distribuir bem a carga = distribuir bem os tablets = depende da distribuição da row key.

> *"Tablets are similar to HBase regions, for those of you who have used the HBase API."*

##### 2. Colossus + SSTable — como o dado é guardado

> *"Tablets are stored on Colossus, which is Google's file system, in SSTable format."*

> *"An SSTable provides a persistent, ordered, immutable map from keys to values, where both keys and values are arbitrary byte strings."*

- **Colossus** é o filesystem distribuído interno do Google (sucessor do GFS — Google File System).
- O Bigtable armazena os dados como **SSTables** (Sorted String Tables): arquivos imutáveis, ordenados por chave.
- Imutável = uma vez escrito, **não muda**. Updates são feitos escrevendo nova SSTable e mergeando depois (LSM tree).

**Implicações:**
- Writes são **rápidos** (sempre append em memória + flush para SSTable nova).
- Reads podem precisar consultar várias SSTables, mas Bigtable consolida com **compaction** em background.
- Tudo é **byte string** — Bigtable não interpreta o conteúdo das células. Tipos são responsabilidade da aplicação.

##### 3. Rebalance sem mover dado — o truque essencial

A terceira imagem do vídeo mostra: um node está sobrecarregado, e o Bigtable **realoca tablets** sem copiar os bytes.

```
Antes (Node B sobrecarregado, tem tablets C+D):
   Node A (tablets A, B)    Node B (C, D — overload!)    Node C (E)

Depois (rebalance):
   Node A (A, B)            Node B (D)                   Node C (C, E)
                                                          ↑
                                         O "ponteiro" do tablet C foi
                                         reatribuído a Node C — os bytes
                                         de C continuam no Colossus, no
                                         mesmo lugar. Só mudou quem é
                                         "dono" lógico daquele tablet.
```

> *"If a certain Bigtable node is frequently accessing a certain subset of data, Bigtable will update the indexes so that other nodes can distribute that workload evenly."*

**Por que isso é genial:**
- Em bancos clássicos (Postgres, Cassandra), rebalance = copiar bytes de um nó para outro = lento + dispendioso em rede.
- Em Bigtable, **os dados estão no Colossus**, e os nodes são apenas "interfaces" que servem queries sobre aqueles dados. Reatribuir um tablet a outro node é **só atualizar metadata**.
- Resultado: rebalances são **rápidos** (segundos), não interrompem o serviço, e o Bigtable faz **automaticamente** conforme o padrão de acesso muda.

> *"Bigtable learns to adjust to specific access patterns."*

##### 4. Linear scaling

> *"That throughput scales linearly, so for every single node that you do add, you're going to see a linear scale of throughput performance, up to hundreds of nodes."*

- 1 node ≈ 10.000 ops/s.
- 10 nodes ≈ 100.000 ops/s.
- 100 nodes ≈ 1.000.000 ops/s.
- (sujeito a row key bem distribuída — hotspot quebra a linearidade).

**Trade-off:** **você paga pelos nodes mesmo se não usar.** *"You pay for those nodes while they are operational, whether your application is using them or not."*

### Bloco "Integração com o ecossistema"

> *"Bigtable integrates easily with popular big data tools like Hadoop, Dataflow, and Dataproc."*

- **Dataflow** (managed Apache Beam) — pipelines de streaming que escrevem/leem Bigtable.
- **Dataproc** (managed Hadoop/Spark) — jobs MapReduce/Spark sobre dados Bigtable.
- **BigQuery** — consegue ler tabelas Bigtable como **external table** (federation), permitindo análise SQL sem ETL.
- **Vertex AI** — feature store backed by Bigtable.

> *"Plus, Bigtable supports the open source industry standard HBase API, which makes it easy for your development teams to get started."*

**Por que HBase importa:**
- HBase é o NoSQL wide-column do mundo Hadoop (open source, inspirado no paper Bigtable original).
- Times migrando de Hadoop on-prem podem trocar HBase por Bigtable **sem reescrever código**.
- Para a prova: "compatible with HBase API" = pista determinística para Bigtable.

### Bloco "Decision points — quando usar Bigtable"

#### Os 5 gatilhos canônicos do vídeo

> *"If you need to store more than 1 TB of structured data, have very high volume of writes, need read/write latency of less than 10 milliseconds along with strong consistency, or need a storage service that is compatible with the HBase API, consider using Bigtable."*

| Gatilho                                          | Pista léxica no exame                              |
|--------------------------------------------------|----------------------------------------------------|
| **>1 TB de dado estruturado**                    | "petabyte-scale", "terabytes of data"              |
| **Alto volume de escrita**                       | "high write throughput", "thousands of writes/sec" |
| **<10 ms latência**                              | "low latency", "single-digit ms", "real-time"      |
| **Strong consistency**                           | "strong consistency" (dentro do cluster)           |
| **HBase API**                                    | "HBase compatibility", "migrating from HBase"      |

**Se NENHUM desses está presente → não é Bigtable.** Provavelmente é Firestore (apps gerais) ou Cloud SQL (relacional).

#### Quando NÃO usar Bigtable

> *"If you don't need any of these and are looking for a storage service that scales down well, consider using Firestore."*

| Pista                                                 | Use isso ao invés                              |
|-------------------------------------------------------|------------------------------------------------|
| Dado < 1 TB, baixo volume                             | **Firestore** (escala melhor para baixo, é serverless) |
| Precisa de transações ACID multi-row                  | **Firestore** ou **Spanner**                   |
| Precisa de joins / SQL relacional                     | **Cloud SQL** ou **Spanner**                   |
| Workload analítico ad-hoc sobre dado estável          | **BigQuery**                                   |
| Sync real-time mobile/web                             | **Firestore Native**                           |
| Cache de aplicação (sub-ms)                           | **Memorystore**                                |
| Workload muito intermitente (scale to zero importa)   | **Firestore** (Bigtable não escala a zero)     |

#### O detalhe de custo

> *"The smallest Bigtable cluster you can create has three nodes and can handle 30,000 operations per second."*

> *"Remember that you pay for those nodes while they are operational, whether your application is using them or not."*

- **Piso:** 3 nodes (em produção; SSD; HDD permite 1 node em dev).
- **3 nodes = 30.000 ops/s** garantidos.
- **Custo de 3 nodes ≈ $1.500-$2.000/mês** (sem contar storage). Caro pra workload pequeno.
- Firestore para o mesmo volume pequeno custa **dezenas de dólares** porque é pay-per-operation.

**Tradução pra prova:** se o cenário enfatiza **baixo volume + economia + scale to zero**, **NÃO escolha Bigtable** mesmo que seja "NoSQL" — Firestore vence.

### Bloco "Bigtable vs Firestore vs BigQuery — o triângulo NoSQL/analítico"

Esse comparativo confunde porque os três aparecem em cenários NoSQL/analytics.

| Eixo                       | Bigtable                          | Firestore                            | BigQuery                              |
|----------------------------|-----------------------------------|--------------------------------------|---------------------------------------|
| **Categoria**              | NoSQL wide-column                 | NoSQL document                       | Data warehouse (analytic)             |
| **Escala**                 | Petabytes                         | Terabytes                            | Petabytes                             |
| **Latência típica**        | <10 ms                            | 10s-100s ms                          | Segundos (query analítica)            |
| **Transações ACID**        | Atômico por row apenas            | ACID multi-document (até 500)        | Não é OLTP                            |
| **Schema flexível**        | Wide-column (estrutura semi-rígida)| ✅ totalmente                        | Schema definido                       |
| **Modelo de cobrança**     | Pay-per-node                      | Pay-per-operation (serverless)       | Pay-per-TB-processed (serverless)     |
| **Scale to zero**          | ❌                                | ✅                                   | ✅                                    |
| **Real-time mobile sync**  | ❌                                | ✅                                   | ❌                                    |
| **SQL**                    | ❌ (API HBase-like)               | Limitado                             | ✅ ANSI SQL                           |
| **Caso primário**          | IoT, AdTech, séries temporais     | App mobile/web, NoSQL geral          | BI, analytics histórico               |

**Heurística pra prova:**
1. Cenário tem "mobile/web sync, ACID, schema flexível, scale to zero" → **Firestore**.
2. Cenário tem ">1 TB, high write throughput, <10 ms, HBase, time-series, IoT" → **Bigtable**.
3. Cenário tem "SQL ad-hoc analytics, dashboards, dado estável" → **BigQuery**.

#### Bigtable vs BigQuery (a confusão mais comum)

Ambos escalam a PB, ambos aparecem em "analytics". A diferença:

| Aspecto                  | Bigtable                                | BigQuery                              |
|--------------------------|------------------------------------------|---------------------------------------|
| **Lookup por chave**     | Sub-10 ms — caso primário                | Lento (não é o caso de uso)           |
| **Query SQL ad-hoc**     | Não suporta SQL nativo                  | Caso primário                         |
| **Update / Insert**      | Alta frequência (operacional)           | DML existe mas com cache; caro/lento  |
| **Modelo mental**        | "NoSQL gigante, lookup rápido por row"  | "Warehouse SQL gigante, ler históricos" |

**Pista decisiva:** se a pergunta enfatiza **alta taxa de update + low latency lookup por chave** → Bigtable. Se enfatiza **SQL ad-hoc + dado que não muda muito + dashboards** → BigQuery.

**Padrão real comum:** Bigtable guarda evento bruto (latência baixa + atualização rápida), e periodicamente exporta para BigQuery (analytics histórico SQL). Os dois trabalham juntos.

## Labs

> O vídeo não menciona lab específico de Bigtable neste módulo.

### Comandos `gcloud` críticos (referência)

#### Criar uma instância Bigtable

```bash
gcloud bigtable instances create iot-prod \
  --display-name="IoT Production" \
  --cluster=iot-cluster-1 \
  --cluster-zone=us-central1-b \
  --cluster-num-nodes=3 \
  --cluster-storage-type=SSD \
  --instance-type=PRODUCTION
```

**Flags importantes:**
- `--cluster-num-nodes=3` → piso de produção; cada node = 10k ops/s.
- `--cluster-storage-type=SSD` → SSD para latência baixa (ou `HDD` para arquivo barato).
- `--instance-type=PRODUCTION` → versus `DEVELOPMENT` (single node, sem SLA, mais barato).

#### Adicionar um cluster replicado (multi-region replication)

```bash
gcloud bigtable clusters create iot-cluster-2 \
  --instance=iot-prod \
  --zone=europe-west1-b \
  --num-nodes=3
```

**O que acontece:** o Bigtable replica os dados entre clusters. Você ganha **DR cross-region + leitura local** para clients europeus. Trade-off: replicação assíncrona introduz potencial de **eventual consistency** entre clusters (dentro de cada cluster continua strong).

#### Criar tabela e column family (via `cbt` CLI)

```bash
# Instalar cbt: gcloud components install cbt

# Criar tabela
cbt -instance=iot-prod createtable sensors

# Criar column family
cbt -instance=iot-prod createfamily sensors readings

# Escrever uma cell
cbt -instance=iot-prod set sensors device001#1700000000 readings:temp=22.5

# Ler row
cbt -instance=iot-prod read sensors prefix=device001
```

#### Listar e ver utilização

```bash
gcloud bigtable instances list
gcloud bigtable clusters list --instances=iot-prod
```

## Cenário real

### Onde Bigtable encaixa nos seus projetos?

**Não nos projetos atuais.** Portaria e TaxIA têm volume muito menor — Cloud SQL Postgres + Cloud Storage + Memorystore cobrem tudo.

**Quando viraria o switch para Bigtable:**

- **Portaria evoluindo para IoT pesado** — se cada portaria começasse a enviar telemetria contínua (sensores de presença, câmeras com analytics, etc.) e o volume saísse de "milhares de eventos/dia" para "milhões/hora", o Postgres deixa de servir. Aí Bigtable + dashboards Looker / BigQuery por cima.
- **Plataforma de análise de uso do TaxIA** — se viralizasse e cada interação com a IA virasse um evento (cada tokenização, cada chamada, cada feedback de qualidade) em volume massivo, com necessidade de query rápida por usuário, Bigtable é candidato.

### Casos canônicos fora dos projetos

- **AdTech (DSP / SSP):** logs de bid requests + impressions em volume insano, com necessidade de lookup por user_id (cookie / device ID) em <10 ms para personalização em tempo real.
- **FinTech / mercado financeiro:** ticks de mercado (preço de cada ação a cada millissegundo), com row key `<ticker>#<reverse_timestamp>` para range scans rápidos por janela temporal.
- **IoT industrial:** turbinas eólicas, plataformas de óleo, sensores ambientais — cada device escrevendo telemetria.
- **Monitoring de larga escala:** Stackdriver / Prometheus internamente usam wide-column stores.
- **Personalização / recomendação:** feature store para ML.

### Padrão "Bigtable + BigQuery em conjunto"

Esse é **o padrão que mais aparece em entrevistas técnicas**:

```
Sensores IoT → Pub/Sub → Dataflow ┬→ Bigtable    (escrita rápida, lookup recente)
                                  └→ BigQuery    (analytics histórico SQL)
```

- **Bigtable** atende dashboards operacionais ("últimos 5 min de leituras desse sensor").
- **BigQuery** atende análises históricas ("consumo médio por região nos últimos 12 meses").
- Cada um faz o que é bom — e Dataflow alimenta os dois em paralelo.

### Erro clássico no mundo real

**Escolher Bigtable para um workload pequeno** "porque é o que o Google usa". Resultado:
- 3 nodes mínimos = $1.500/mês para 30k ops/s que você nunca atinge.
- Modelagem de row key é difícil — você gasta semanas escolhendo e ainda assim cria hotspots.
- Falta de transações multi-row vira bug constante.

**Regra prática:** se o volume é < 100 GB e < 1000 QPS, Bigtable é **overkill**. Fica com Firestore ou Cloud SQL.

## Armadilhas de prova

1. **Bigtable = NoSQL wide-column, petabyte-scale, <10 ms, sem ACID multi-row, sem SQL nativo.** Memorize essa stack — é o que define.

2. **5 gatilhos canônicos do vídeo:** >1 TB, alto throughput de escrita, <10 ms latência, strong consistency dentro do cluster, HBase API. Se algum desses aparece no enunciado, Bigtable é candidato.

3. **Sem nenhum dos gatilhos → não é Bigtable.** Firestore (NoSQL geral) ou Cloud SQL (relacional) vencem.

4. **"Same database as Google Search/Maps/Gmail"** — pista improvável mas determinística para Bigtable.

5. **Bigtable NÃO escala a zero.** Você paga 3 nodes mínimos sempre. *"Pay for those nodes whether your application is using them or not."* — Se a pergunta enfatiza "scale to zero" ou "intermittent workload", **Firestore**, não Bigtable.

6. **3 nodes = 30.000 ops/s** = piso de produção. Decorar.

7. **Bigtable não tem joins nem SQL ANSI.** API é estilo HBase / Cloud Bigtable client libraries (Java, Go, Python, etc.).

8. **HBase API é pista determinística** para Bigtable. Nenhum outro serviço GCP suporta HBase API.

9. **Row key é a única chave indexada.** Não há "secondary index" como em SQL ou Firestore. Queries são todas baseadas em row key (lookup direto ou range scan).

10. **Hotspot de row key** é o erro de design mais comum. Timestamp/sequencial como prefixo = todas as escritas novas vão pro mesmo node → throughput cai.

11. **Strong consistency dentro de um cluster.** Replicação multi-cluster (multi-region) introduz **eventual consistency** entre clusters. Se a pergunta diz "strong consistency across multiple regions" com Bigtable como opção, **suspeite** — Spanner é mais provavelmente a resposta.

12. **Transações: atomicidade só em row única.** Operações cross-row não são atômicas. Cenário com "ACID transactions across multiple entities" → Firestore ou Spanner.

13. **Bigtable vs Firestore (a confusão mais comum):**
    - Bigtable: PB scale, <10 ms, sem ACID multi-row, sem mobile SDK, pay-per-node.
    - Firestore: TB scale, ACID, mobile SDK, scale to zero, pay-per-operation.

14. **Bigtable vs BigQuery (a segunda confusão):**
    - Bigtable: operacional, lookup por chave rápido, alta taxa de update.
    - BigQuery: analítico, SQL ad-hoc, dado estável / append-only.

15. **Bigtable é o storage do mesmo "Bigtable paper" de 2006** — base do HBase, Cassandra (parcialmente), e muitos outros. Não confundir com o produto open-source que se inspirou nele.

16. **Storage = Colossus + SSTable.** Imutável, ordenado, distribuído. Não precisa saber detalhes pro ACE, mas saber que "processing está separado de storage" é o que explica os rebalances rápidos.

17. **Rebalance não move bytes.** Só reatribui ownership de tablets — por isso é rápido e contínuo.

18. **Throughput linear até centenas de nodes.** Triplicar nodes = triplicar throughput, **se a row key estiver bem distribuída**.

19. **Tablets são análogos a "regions" do HBase.** Pode aparecer essa equivalência em pergunta.

20. **Sparse tables = células vazias não custam.** Você pode ter milhões de colunas com 99% delas vazias sem problema de custo/performance.
