> **Considerations da prova:** Load Balancing; disponibilidade de locations na rede; Network Service Tiers.

---

## Conceitos

### Por que existem tantos load balancers no Google Cloud?

O Google Cloud tem um catálogo grande de load balancers (LB) porque cada combinação de **3 eixos de decisão** vira um produto diferente. Entender os eixos é mais útil que decorar nomes.

**Os 3 eixos:**

1. **Camada OSI** — Application (L7) ou Network (L4)
2. **Escopo** — Global ou Regional ou Cross-region
3. **Visibilidade** — External (internet) ou Internal (VPC)

Mais um **sub-eixo** dentro de Network LB:

4. **Comportamento** — Proxy (termina e abre nova conexão) ou Passthrough (encaminha sem alterar)

E um eixo de **performance/custo** transversal:

5. **Network Service Tier** — Premium (rede Google global) ou Standard (sai pra internet pública mais cedo)

---

### Eixo 1 — Camada OSI: L7 (Application) vs L4 (Network)

#### Application Load Balancer (L7, HTTP/HTTPS)

**O que faz:**
- Inspeciona o conteúdo da requisição HTTP/HTTPS
- Roteia baseado em **headers, URL paths, cookies**
- Faz **terminação SSL/TLS** (descriptografa para inspecionar)
- Atua como **reverse proxy**

**Use quando:**
- Tráfego é **HTTP ou HTTPS** (web, APIs)
- Você precisa de **roteamento baseado em path** (`/api/*` vai pra um backend, `/static/*` vai pra outro)
- Quer SSL terminado no LB
- Precisa de features avançadas: Cloud Armor, Cloud CDN

**Backends suportados:** Compute Engine, GKE, Cloud Storage, Cloud Run, e até backends externos via Hybrid Connectivity.

#### Network Load Balancer (L4, TCP/UDP/outros)

**O que faz:**
- Opera no nível de **TCP/UDP** (não inspeciona conteúdo HTTP)
- Roteia baseado em **IP + porta + protocolo**
- Mais rápido (menos overhead) que L7

**Use quando:**
- Protocolo **não é HTTP** (TCP raw, UDP, outros)
- Precisa de **performance bruta** sem inspeção de aplicação
- Você quer preservar o **IP de origem** (passthrough)

**Atalho mental:** Application = L7. Network = L4. Sempre.

---

### Eixo 2 — Escopo: Global vs Regional vs Cross-region

| Escopo | O que significa | Quando usar |
|---|---|---|
| **Global** | Frontend único anycast, distribui pra backends no mundo todo | App com usuários no mundo todo, latência ótima |
| **Regional** | Tudo numa única região | App regional, todos os recursos na mesma região |
| **Cross-region** | Internal LB que atravessa regiões dentro da VPC | Comunicação interna entre regiões |

**Premium vs Standard tier:**
- **Global LB** está disponível **somente em Premium tier** (usa a rede global do Google)
- **Standard tier** força você a usar **Regional LB** (tráfego entra na região mais cedo)

---

### Eixo 3 — Visibilidade: External vs Internal

- **External:** o LB recebe tráfego da **internet pública**. Tem IP público.
- **Internal:** o LB só recebe tráfego de **dentro da VPC** (ou de redes conectadas via VPN/Interconnect). IP privado.

**Use external quando:** sua app é exposta à internet (site, API pública).
**Use internal quando:** comunicação entre tiers da sua app (web → backend → DB) dentro da VPC.

---

### Eixo 4 — Proxy vs Passthrough (só pra Network LB)

#### Proxy Load Balancer

- Funciona como **reverse proxy**: termina a conexão do cliente, abre uma nova conexão pro backend
- O backend vê o IP do **load balancer**, não do cliente original (precisa ler header `X-Forwarded-For` se quiser saber o IP real)
- Permite features avançadas: traffic management, backends mistos (cloud + on-prem)
- Tipos: Global external proxy NLB, Regional external proxy NLB, Regional internal proxy NLB, Cross-region internal proxy NLB

#### Passthrough Load Balancer

- **Não termina** a conexão. Encaminha o pacote IP direto pro backend
- O backend vê o **IP original do cliente** (Direct Server Return)
- Mais simples, melhor para protocolos que não sejam HTTP/TCP comuns
- Tipos: Regional external passthrough NLB, Regional internal passthrough NLB
- **Sem versão global passthrough** (passthrough é sempre regional)

**Atalho:** quer **preservar IP do cliente** → passthrough. Quer **inspecionar/modificar tráfego** ou **distribuir global** → proxy.

---

### Eixo 5 — Network Service Tiers

| Tier | Como funciona | Custo | Quando usar |
|---|---|---|---|
| **Premium** | Tráfego entra na rede privada do Google no PoP mais perto do cliente, atravessa o backbone do Google até a região do backend | Mais caro | Tráfego global, latência crítica, qualidade de rede |
| **Standard** | Tráfego sai da rede do Google mais cedo (perto do destino), atravessa internet pública até o cliente | Mais barato | Tráfego regional, custo é prioridade, latência aceita variação |

**Implicação prática:**
- Premium tier desbloqueia **LBs globais**.
- Standard tier força você a usar **LBs regionais**.

**Atalho de prova:** "low-cost + local traffic + single region" → **Standard tier**. "Global users + lowest latency" → **Premium tier**.

### Padrão de prova: arquitetura típica de app web

Para uma aplicação web de **3 tiers** (frontend → web → backend), o padrão é:

```
Internet
    ↓
[External Application LB]  ← frontend (recebe tráfego da internet)
    ↓
[Web tier — Compute/GKE/Cloud Run]
    ↓
[Internal Application LB]  ← entre tiers (dentro da VPC)
    ↓
[Backend tier — bancos, serviços internos]
```

**Por que essa combinação:**
- External ALB no frontend → recebe HTTPS público, termina TLS, distribui pra web tier
- Internal ALB entre tiers → tráfego HTTP/HTTPS interno, sem expor backend à internet

**Se o app for regional + low-cost:** ambos os LBs são **regionais** + **Standard tier**.

## Questionários

### DQ 08 — Load balancing seguro + low-cost para app regional de 3 tiers
**Cenário:** App de e-commerce piloto, 3 tiers, **tráfego local** de Minneapolis (us-central1), recursos dedicados em us-central1. Precisa de arquitetura **segura** e **low-cost**.

**Resposta correta: C — Standard tier regional external Application LB (frontend) + regional internal Application LB (entre web e backend)**

**Por quê:**
- **Standard tier:** tráfego local — não precisa da rede global do Google
- **Regional:** todos os recursos estão em us-central1 — global seria desperdício
- **External ALB no frontend:** recebe tráfego da internet, termina HTTPS (segurança)
- **Internal ALB entre tiers:** comunicação interna, sem expor backend
- **Application LB (L7):** apropriado pra ecommerce (HTTP/HTTPS)

**Por que as outras estão erradas:**
- **A (Premium global ALB):** tráfego é local, Premium global é caro e desnecessário
- **B (Global proxy NLB + premium passthrough NLB):** TCP/Network LB **não é HTTPS-aware** — perda de segurança HTTP; passthrough premium é global, não cabe entre tiers regionais
- **D (Internal proxy NLB no frontend):** o frontend precisa receber tráfego da **internet** — não pode ser **internal**. Erro grosseiro.

**Armadilha:** A combinação "low-cost + tráfego local" sempre traz "Standard tier + regional". Não caia no "premium global" que parece "melhor".

### DQ 09 — Qual LB roda em Layer 7 do TCP stack?

**Resposta correta: A — Global Application Load Balancer**

**Por quê:** "Application" no nome sempre = **Layer 7**. ALB inspeciona HTTP headers, paths e cookies — capacidades de camada de aplicação.

**Por que as outras estão erradas:**
- **B (Global proxy NLB):** Layer 4. Proxy reverso de TCP, sem inspeção de aplicação.
- **C (Regional passthrough NLB):** Layer 4. Encaminha pacotes IP/TCP/UDP.
- **D (Regional internal proxy NLB):** Layer 4. Proxy de TCP regional interno.

**Armadilha:** **Application** = L7. **Network** = L4. Esse é o mnemônico mais útil dessa seção.

## Armadilhas de prova

1. **Application LB = L7. Network LB = L4.** Sempre. Esse mnemônico isolado vale 2-3 questões.
2. **Standard tier ⇒ regional.** Não existe LB global em Standard tier.
3. **Global LB ⇒ Premium tier.** Caminho único.
4. **"Local / regional / mesma região / low-cost"** → Standard tier + regional LB.
5. **"Global users / worldwide / lowest latency"** → Premium tier + global LB.
6. **Frontend de app web público** → **External** ALB. Nunca internal.
7. **Comunicação entre tiers dentro da VPC** → **Internal** ALB.
8. **Passthrough preserva IP do cliente.** Proxy não (precisa de `X-Forwarded-For`).
9. **HTTP/HTTPS** → Application LB. Nunca Network LB. Mesmo que "funcione", você perde features de aplicação e o exam considera erro.
10. **Passthrough NLB é sempre regional.** Não existe passthrough global.

## Cenário real

**Onde eu usaria isso no trabalho?**

- **Backend-portaria / TaxIA-backend** hoje: provavelmente um **Regional external Application LB** seria suficiente. Tráfego brasileiro, recursos em uma região (`southamerica-east1`), Standard tier basta.
- **Internal ALB** entre o BFF e o serviço de IA: comunicação privada dentro da VPC, sem expor IA na internet.
- **Cloud Armor + Application LB:** se eu precisasse de WAF (proteção contra OWASP top 10) ou rate limiting, anexaria Cloud Armor ao ALB. Network LB não permite.
- **Network LB passthrough:** se eu hospedasse um serviço **não-HTTP** (game server UDP, MQTT broker TCP) — aí ALB não serve, NLB passthrough é o caminho.

**Erro comum que cometeria sem estudar:** escolher Global LB Premium por achar que "é melhor". Se meus usuários são todos no Brasil, é desperdício enorme. **Regional Standard tier** entrega a mesma latência local por uma fração do custo.
