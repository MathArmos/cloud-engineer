## Conceitos

### Bloco "O que é Firestore — definição precisa"

> *"Firestore is a fast, fully managed, serverless, cloud-native, NoSQL, document database that simplifies storing, syncing and querying data for your mobile, web, and IoT apps at global scale."*

Vamos quebrar cada adjetivo, porque cada um responde a uma pergunta de prova diferente:

| Adjetivo            | O que significa                                                              | Implicação pra prova                          |
|---------------------|------------------------------------------------------------------------------|-----------------------------------------------|
| **Fast**            | Latência baixa (poucos ms para read/write)                                   | Apto a app real-time                          |
| **Fully managed**   | Você não cuida de SO, patch, replicação                                      | Pista pra escolher Firestore em vez de "rodar Mongo numa VM" |
| **Serverless**      | Sem provisioning de instância; escala automática **incluindo scale to zero** | Pista decisiva: "scale to zero" → Firestore   |
| **Cloud-native**    | Built for GCP, integração nativa com Auth, Functions, etc.                   | -                                             |
| **NoSQL**           | Sem schema rígido; modelo de documento (JSON-like)                           | Não tem joins relacionais                     |
| **Document**        | Dados em documentos, organizados em coleções                                 | Modelo mental: pasta (collection) com arquivos JSON (documents) |
| **Global scale**    | Replicação multi-região nativa                                               | Pista para Firestore vs alternativas regionais |
| **Mobile, web, IoT**| Casos de uso primários, com SDKs específicos                                 | Pista léxica "mobile app" / "real-time sync"  |

### Bloco "Modelo de dados — collections e documents"

O modelo do Firestore não é tabela/linha como SQL, nem chave/valor puro como Memorystore.

```
Database
  └── Collection "users"
        └── Document "user_123"
              ├── name: "Alice"
              ├── email: "alice@example.com"
              ├── created_at: 2026-05-14T...
              └── Subcollection "orders"   ← um documento pode conter subcollections
                    └── Document "order_001"
                          ├── product_id: "p42"
                          └── amount: 199.90
```

**Conceitos centrais:**

- **Document** = unidade básica; tem ID, campos chave-valor, e pode conter **subcollections**.
- **Collection** = container de documents, sem schema fixo entre eles (cada doc pode ter campos diferentes).
- **Subcollection** = collection dentro de um document — permite hierarquia natural.
- **Field** = chave-valor dentro de um document; tipos: string, number, boolean, array, map (objeto aninhado), timestamp, geopoint, reference, etc.

**Limites práticos** (não cobrados de cor no ACE, mas bom saber):
- Documento ≤ **1 MB**.
- Até **20.000 campos** por documento.
- Profundidade de subcollections ≤ **100 níveis**.
- ~1 escrita por segundo por documento (com burst tolerável).

**Modelagem prática:** documents grandes que mudam o tempo todo não funcionam. Modelo correto = muitos documents pequenos.

### Bloco "Features de destaque do vídeo"

#### 1. Live sync + offline support

> *"Its client libraries provide live synchronization and offline support."*

**Live sync:** SDKs Firebase para mobile/web mantêm **listeners** em queries. Quando o dado muda no servidor, o client recebe **push automático** — sem polling.

**Offline support:** o SDK mantém cache local. Se o dispositivo perde rede, ele:
1. Continua respondendo leituras do cache.
2. Aceita escritas e as enfileira localmente.
3. Sincroniza quando a rede volta, resolvendo conflitos com last-write-wins por default.

**Por que isso é grande:**
- Apps mobile **funcionam offline sem código custom** de sincronização.
- Apps web colaborativos (tipo Google Docs) ficam **simples** — basta inscrever no document, mudanças aparecem automáticas.

**Casos canônicos:** Slack-like chat, Trello-like board, dashboard ao vivo, apps de delivery / ride-sharing, IoT command/state.

#### 2. Integração com Firebase e Google Cloud

> *"It's security features and integrations with Firebase and Google Cloud accelerate building truly serverless apps."*

- **Firebase Authentication** integrado nativamente.
- **Security Rules** — você escreve regras declarativas (parecidas com JSON+sintaxe própria) que controlam quem pode ler/escrever cada documento. **Avaliadas no servidor**, então clientes não conseguem burlar.
- **Cloud Functions** podem disparar em **eventos do Firestore** (`onCreate`, `onUpdate`, `onDelete`) — base do padrão event-driven.
- **App Hosting / Cloud Run** pode usar Firestore como backend sem precisar gerenciar conexão (não tem connection pool — é HTTP/gRPC, sem persistência de conexão).

#### 3. ACID transactions

> *"Firestore also supports ACID transactions, so if any of the operations in the transaction fail and cannot be retried, the whole transaction will fail."*

Diferente de muitos NoSQL (que oferecem só "operações atômicas em chave única"), Firestore tem **transações ACID multi-document** de verdade:
- Atômicas: tudo ou nada.
- Consistentes: respeita security rules e tipos.
- Isoladas: outras transações não veem estado parcial.
- Duráveis: commit é persistido em múltiplas zonas.

**Limite prático:** transação pode tocar até **500 documents**. Suficiente pra maioria dos casos de app; não pra batch jobs.

**Pegadinha do vídeo:** *"if any of the operations in the transaction fail and cannot be retried, the whole transaction will fail."* — Firestore **retenta automaticamente** transações que falharam por conflito de versão. Você só vê a falha se ela for irrecuperável.

#### 4. Replicação multi-região + strong consistency

> *"With automatic multi-region replication and strong consistency, your data is safe and available even when disasters strike."*

Firestore tem dois tipos de localização ao criar o banco:
- **Multi-region** (`nam5` = EUA, `eur3` = Europa, etc.): replicação em **múltiplas regiões**, sobrevive a falha regional, **strong consistency**.
- **Regional** (`us-central1`, `europe-west1`, etc.): replicado em **múltiplas zonas** da região; mais barato, latência local menor, mas não sobrevive a falha regional.

**Decisão da localização é definitiva — não dá pra mudar depois.**

**Strong consistency** = depois que uma escrita commita, **qualquer leitura subsequente** (de qualquer lugar) **vê** o novo valor. Isso vale para Firestore Native e Datastore mode (ver bloco seguinte).

#### 5. Queries sofisticadas

> *"Firestore even allows you to run sophisticated queries against your NoSQL data without any degradation in performance."*

Tipos de query suportados:
- Filtros por campo (`where`).
- Composições (`AND`, `OR` desde 2023).
- Ordenação por múltiplos campos.
- Paginação (`startAt`, `endAt`, `limit`).
- Queries em arrays (`array-contains`, `array-contains-any`).
- Queries com subcollection groups (todas as subcollections com mesmo nome, agregadas).

**Sem performance degradation:** o Firestore mantém **índices automáticos** em todos os campos por padrão. Queries combinatórias precisam de **índice composto** (criado uma vez via Console ou `firestore.indexes.json`).

**Limites:**
- Sem joins relacionais (precisa denormalizar).
- Sem agregações arbitrárias estilo `SUM`, `GROUP BY` (existe `count`, `sum`, `avg` agora em queries de agregação, mas ainda limitado).

### Bloco "Datastore vs Firestore — a parte que cai na prova"

Esse é o bloco mais importante do vídeo para o ACE.

> *"Firestore is actually the next generation of Datastore."*

**Histórico em uma linha:** o Google tinha um NoSQL chamado **Datastore** (que era o backend default do App Engine). Em 2019, lançou **Firestore** como sucessor, com modelo melhor e features extras (real-time, offline, mobile SDKs).

Para não quebrar apps existentes, o Firestore **roda em dois modos**:

#### Firestore in Native mode

> *"Firestore in Native mode introduces new features such as: A new, strongly consistent storage layer. A collection and document data model. Real-time updates. And mobile and web client libraries."*

| Característica                  | Firestore Native             |
|---------------------------------|------------------------------|
| Storage layer                   | **Novo**, strongly consistent|
| Modelo de dados                 | Collections + documents (com subcollections) |
| Real-time updates               | ✅                           |
| Mobile/Web client libraries     | ✅                           |
| Backward-compatible com Datastore? | ❌ (modelo diferente)     |

**Use Native mode quando:** você está fazendo um **app novo mobile ou web** que precisa das features modernas (real-time sync, offline, SDKs do Firebase).

#### Firestore in Datastore mode

> *"By creating a Firestore database in Datastore mode, you can access Firestore's improved storage layer while keeping Datastore system behavior."*

| Característica                  | Firestore Datastore mode     |
|---------------------------------|------------------------------|
| Storage layer                   | **Novo Firestore** (melhor)  |
| Modelo de dados                 | **Entities/kinds** (modelo legado Datastore) |
| Real-time updates               | ❌                           |
| Mobile/Web client libraries     | ❌                           |
| Backward-compatible com Datastore? | ✅ (APIs antigas funcionam) |

**Datastore mode remove 3 limitações antigas do Datastore:**

> *"This removes the following Datastore limitations: Queries are no longer eventually consistent; instead, they are all strongly consistent. Transactions are no longer limited to 25 entity groups. And writes to an entity group are no longer limited to 1 per second."*

1. **Antes:** queries eram **eventualmente consistentes**. **Agora:** strongly consistent.
2. **Antes:** transações limitadas a 25 entity groups. **Agora:** sem esse limite.
3. **Antes:** ~1 escrita/segundo por entity group. **Agora:** sem esse limite.

**Use Datastore mode quando:** você tem um **servidor existente** que já usa Datastore APIs (App Engine, GCE, GKE) e quer **continuar usando** sem reescrever, mas ganhando a infra moderna.

#### Resumo prático (a regra do vídeo)

> *"A general guideline is to use Firestore in Datastore mode for new server projects, and Native mode for new mobile and web apps."*

| Tipo de projeto novo                | Modo recomendado          |
|-------------------------------------|---------------------------|
| **Mobile app / Web app**            | **Firestore Native mode** |
| **Server / backend só**             | **Datastore mode**        |
| **App legado com Datastore APIs**   | Datastore mode (compat)   |

**Por que server prefere Datastore mode:** servers não precisam das features mobile (real-time sync, offline). Datastore mode dá APIs mais maduras e schema de entidade ligeiramente diferente.

#### A regra de ouro do vídeo

> *"To access all of the new Firestore features, you must use Firestore in Native mode."*

Se a pergunta enfatizar **mobile sync**, **real-time**, **offline**, **client libraries** → **Native mode**.

Se a pergunta enfatizar **migração de Datastore**, **server-side**, **compatibilidade com APIs antigas** → **Datastore mode**.

#### Restrições importantes (cair certo)

- Você **escolhe o modo na criação do banco**.
- Um projeto GCP pode ter **apenas um Firestore database** (essa era a limitação histórica; recentemente o Google permitiu múltiplos databases por projeto, mas o curso ainda reflete a regra antiga — para prova, assuma "um banco por projeto").
- **Não dá pra mudar o modo depois.** Native → Datastore não é conversão; é criar outro projeto/banco e migrar.

### Bloco "Decision tree do final do vídeo"

> *"If your schema might change and you need an adaptable database, you need to scale to zero, or you want low maintenance overhead scaling up to terabytes, consider using Firestore. Also, if you don't require transactional consistency, you might want to consider Bigtable, depending on the cost or size."*

#### As 3 pistas léxicas para Firestore

| Pista                                                       | Por quê                                       |
|-------------------------------------------------------------|-----------------------------------------------|
| "Schema might change / adaptable database"                  | NoSQL document = schema-flexível              |
| "Scale to zero"                                             | Serverless puro, paga só pelo que usar         |
| "Low maintenance overhead scaling up to terabytes"          | Fully managed, sem tuning                     |

#### Firestore vs Bigtable — o trade-off

| Característica            | Firestore                          | Bigtable                              |
|---------------------------|------------------------------------|---------------------------------------|
| Transações ACID           | ✅                                 | ❌ (operações atômicas por row apenas)|
| Schema flexível           | ✅                                 | Wide-column (estrutura mais rígida)   |
| Real-time sync mobile     | ✅                                 | ❌                                    |
| Escala                    | Até **TB**                         | **PB**                                |
| Latência                  | Vários ms                          | **Sub-ms** consistente                |
| Custo em escala alta      | Mais caro                          | Mais barato em PB                     |
| Use case                  | App geral, mobile, web             | IoT, AdTech, FinTech high-throughput  |

> *"If you don't require transactional consistency, you might want to consider Bigtable, depending on the cost or size."*

**Tradução:** Firestore é o **default NoSQL** para apps gerais. Bigtable só vira a resposta quando **escala massiva (PB), latência sub-ms, ou throughput crítico** entram em cena — e você **abre mão** de transações ACID em troca.

### Comandos `gcloud` e console (referência)

#### Criar um Firestore database

**Importante:** Firestore não tem comando `gcloud firestore databases create` super amigável pra todas as configurações — o jeito mais comum é Console ou Terraform.

```bash
# Listar databases existentes
gcloud firestore databases list

# Criar database (Native mode, multi-region nam5)
gcloud firestore databases create \
  --location=nam5 \
  --type=firestore-native

# Ou Datastore mode regional
gcloud firestore databases create \
  --location=us-central1 \
  --type=datastore-mode
```

#### Exportar e importar (backup manual)

```bash
# Exportar tudo para um bucket
gcloud firestore export gs://meu-bucket-backups/firestore/$(date +%Y-%m-%d)

# Importar de volta
gcloud firestore import gs://meu-bucket-backups/firestore/2026-05-14
```

#### Criar índice composto

```bash
gcloud firestore indexes composite create \
  --collection-group=orders \
  --field-config field-path=user_id,order=ascending \
  --field-config field-path=created_at,order=descending
```

#### Security Rules (snippet declarativo, não gcloud)

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
    match /public/{document=**} {
      allow read: if true;
      allow write: if false;
    }
  }
}
```

**Para o ACE:** entender que Security Rules existem e são avaliadas no servidor é o suficiente. Sintaxe profunda não cai.

## Cenário real

### Onde Firestore encaixa nos seus projetos

**Portaria / TaxIA:**

- **Não é o banco principal.** Você usa Postgres → Cloud SQL faria sentido na nuvem, não Firestore.
- **Pode ser auxiliar:** se o app mobile da Portaria precisar de **real-time sync** de eventos (visitante chegou, alerta gerado), Firestore Native é forte candidato. Empurra mensagens pro mobile sem implementar WebSocket/Push manual.
- **Configurações em tempo real:** flags de feature, configurações por condomínio que mudam e refletem instantaneamente no app — Firestore brilha aqui.

**Cenários canônicos fora dos projetos:**

- **Chat / mensageria** (WhatsApp-like): conversas como collections, mensagens como documents, real-time listeners no SDK.
- **Trello-like / Notion-like**: board/page como document, edições aparecem ao vivo para todos.
- **Apps de delivery / ride-sharing**: estado do pedido/corrida sincronizado entre app do cliente, motorista e admin.
- **Profile + preferences** de usuário em apps mobile.
- **State de jogo** em jogos multiplayer não-competitivos (turn-based, casual).

### O erro clássico no mundo real

**Modelar Firestore como banco relacional.** Tentar fazer joins entre collections, esperar `GROUP BY` complexo, normalizar tudo "3FN". Resultado: queries chatas, performance ruim, código com mil round-trips.

**Modelagem correta NoSQL:**
- **Denormalize.** Repita dado entre documents quando vale evitar join.
- **Modele para a query, não para o dado.** Se a tela mostra "últimos 10 pedidos do usuário com nome do produto", guarda o nome do produto **redundante** no documento de pedido.
- **Use subcollections** para hierarquia natural (user → orders → items), mas evite profundidade excessiva.

### Limite real que te derruba

**1 escrita/segundo por document** — se a sua modelagem coloca um contador global em um document só, o Firestore não escala. Solução = **shard counters** (vários documents, cada um com pedaço do contador, somados no read).

## Armadilhas de prova

1. **Firestore = NoSQL document, ACID, scale to zero, mobile/web SDKs.** Memorize essa stack — é o que diferencia das alternativas.

2. **Firestore Native ≠ Firestore Datastore mode.** O vídeo dedica metade do tempo a isso porque cai na prova:
   - Native = novo modelo, real-time, mobile/web SDKs.
   - Datastore mode = backward-compatible com Datastore, server-side.

3. **"Real-time sync" / "mobile app" / "offline support" → Firestore Native.** Determinístico.

4. **"Migrar Datastore existente" / "server-side, manter APIs antigas" → Firestore Datastore mode.** Não Native.

5. **"Scale to zero" é pista forte para Firestore.** Cloud SQL, Spanner, AlloyDB **não** escalam a zero — sempre tem instância rodando. Firestore sim.

6. **Firestore tem strong consistency (multi-region e regional).** Não confunda com NoSQL "eventually consistent" (estilo Cassandra puro).

7. **Limite ~1 write/sec por document** — se a pergunta fala em "high-frequency updates to a single counter", Firestore não é a melhor escolha (a menos que use shard counters).

8. **Firestore não tem joins relacionais.** Se o cenário enfatiza joins, integridade referencial, schema fixo → não é Firestore.

9. **Firestore Native introduz "novo storage layer".** Datastore mode usa o mesmo storage novo, mas mantém o modelo antigo de entidades.

10. **Datastore mode removeu 3 limitações:** queries strongly consistent (não eventual), sem limite de 25 entity groups por transação, sem limite de 1 write/sec por entity group.

11. **Compatibilidade:** Firestore é compatível com **todas** as APIs e client libraries do Datastore (em Datastore mode). Native mode **não** é backward-compat.

12. **Multi-region location é definitiva.** Não dá pra trocar de regional para multi-region depois.

13. **Apenas um banco por projeto (assumindo o curso).** Recentemente o GCP permite múltiplos, mas a prova ACE costuma seguir a regra antiga.

14. **Firestore vs Bigtable:** Firestore tem ACID, Bigtable não. Bigtable escala mais (PB), Firestore TB. Se o cenário pede transações ACID + NoSQL → Firestore. Se pede throughput massivo IoT/AdTech sem ACID → Bigtable.

15. **Firestore vs Memorystore:** Memorystore é cache in-memory (sub-ms, volátil). Firestore é storage primário durável. Não são intercambiáveis — Firestore não substitui cache, e Memorystore não substitui storage.

16. **Security Rules são avaliadas no servidor.** Clientes não burlam. Pista de "client-side security" → suspeita do erro; sempre confirme rules no server.

17. **Firestore é serverless / pay-per-use.** Você paga por operação (read/write/delete) + storage + network. Não há "instância rodando" cobrando mensal. Vantagem para apps com tráfego irregular.
