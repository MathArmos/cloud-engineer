## Conceitos

### Bloco "Por que Cloud SQL existe — managed vs self-managed"

**Pergunta que abre o vídeo:** *"Por que usar um serviço Google Cloud para SQL se eu poderia subir um SQL Server numa VM Compute Engine?"*

Essa pergunta é a fundação do mindset do exame: **escolher entre rolar o seu** (IaaS — instalar Postgres numa VM) **ou usar um serviço gerenciado** (PaaS — Cloud SQL).

#### O que "gerenciado" significa na prática (Cloud SQL faz por você)

- **Patches do SO** subjacente.
- **Patches e upgrades de versão** do engine (MySQL / Postgres / SQL Server).
- **Backups automatizados** + point-in-time recovery.
- **Replicação** (read replicas + standby de HA).
- **Failover** automático em HA.
- **Monitoring básico** (Cloud Monitoring integrado).
- **Encryption at rest** por default.
- **Storage auto-scaling** (cresce sozinho até o teto que você definir).

#### O que **você ainda administra** (mesmo com Cloud SQL)

> *"You still have to administer MySQL users with the native authentication tools that come with these databases."* — instrutor.

- **Usuários e privilégios do banco** (`GRANT`, `CREATE USER`, roles do Postgres, logins do SQL Server).
- **Schema** (tabelas, índices, views, procedures).
- **Performance tuning de query** (planos, `EXPLAIN ANALYZE`, índices).
- **Configurações de parâmetros do banco** (parameter flags do Cloud SQL — só os que o Google expõe).

**Tradução prática:** Cloud SQL elimina o trabalho de **infra do banco**, mas você continua sendo **DBA lógico**.

#### Quando NÃO usar Cloud SQL (a contraprova)

- Precisa de **escala horizontal** ou **distribuição global** → **Spanner**.
- Precisa de **HTAP** (OLTP + analytics em tempo real no mesmo banco) → **AlloyDB**.
- Workload puramente **analítico** com SQL ad-hoc sobre dado histórico → **BigQuery**.
- Engine que o Cloud SQL não suporta (Oracle, MariaDB nativo, Db2, etc.) → **VM no Compute Engine** com o engine instalado manualmente.
- Customizações profundas que o Cloud SQL não permite (extensões Postgres não whitelisted, plugin específico do MySQL, etc.) → **VM** ou outra solução.

### Bloco "Engines e versões suportadas"

Cloud SQL é **fully managed** para três engines:

| Engine          | Versões suportadas (no vídeo)                     | Equivalente on-prem            |
|-----------------|---------------------------------------------------|--------------------------------|
| **MySQL**       | 5.6, 5.7, 8.0                                     | MySQL Community / Enterprise   |
| **PostgreSQL**  | 9.6, 10, 11, 12, 13, 14, 15                       | PostgreSQL upstream            |
| **SQL Server**  | 2017, 2019 — edições Web, Express, Standard, Enterprise | Microsoft SQL Server          |

**Armadilha de prova:** o exame **pode** perguntar "qual desses engines o Cloud SQL suporta?" e listar opções como Oracle, MariaDB, Db2 — todos **não suportados**. Decorar a lista das três famílias resolve.

> **Observação real:** as versões evoluem. A documentação oficial muda mais rápido que os cursos. Para a prova ACE, **saber que esses três engines são os suportados** é suficiente. Saber a versão exata raramente cai.

#### Clientes / drivers compatíveis

> *"Cloud SQL supports many clients, such as Cloud Shell, App Engine, and Google Workspace scripts. It also supports other applications and tools that you might be used to like SQL Workbench, Toad, and other external applications using standard MySQL drivers."*

**Mensagem-chave:** Cloud SQL **não inventa um protocolo novo**. Ele aceita os drivers padrão de cada engine. Se o seu app fala MySQL hoje, ele fala Cloud SQL MySQL amanhã **sem mudar nenhuma linha de código** — só muda a string de conexão.

### Bloco "Limites de performance e escala (números do vídeo)"

| Recurso                    | Limite por instância              |
|----------------------------|-----------------------------------|
| Storage                    | Até **64 TB**                     |
| IOPS                       | Até **60.000 IOPS**               |
| RAM                        | Até **624 GB**                    |
| CPUs (vCPU)                | Até **96 cores**                  |
| Scale out (leitura)        | **Read replicas**                 |
| Scale up (CPU/RAM)         | Requer **machine restart**        |

**Como ler esses números:**

- 624 GB de RAM e 96 vCPUs são **muito** para um banco transacional regional. Cabe a maioria absoluta dos workloads OLTP de empresas grandes.
- 60.000 IOPS é teto generoso para banco de aplicação. Se você precisa de mais, **provavelmente o problema está em outro lugar** (deveria estar particionando dado ou indo pra Spanner / Bigtable, dependendo do caso).
- 64 TB de storage é o teto duro de um shard. Se o seu dado relacional passa disso, **não é mais cenário Cloud SQL** — vai pra Spanner ou um pattern de sharding aplicativo.

**Armadilha de prova:** o exame não cobra os números exatos. Cobra a **forma do limite**:
- Scale up tem **restart** → indisponibilidade momentânea.
- Scale out é **só leitura** (read replicas) — escrita continua única.
- Se precisa de **escrita escalando horizontalmente** → Cloud SQL não serve. Vai pra **Spanner**.

> **Pista léxica decisiva:** o vídeo deixa explícito: *"if you are concerned about horizontal scalability, you'll want to consider Spanner."*

### Bloco "High Availability (HA) — como funciona"

HA é provavelmente o **tópico mais cobrado** sobre Cloud SQL no ACE. Vale entender o mecanismo em detalhe.

#### Setup

- A instância é **regional**.
- Dentro da região, ficam **duas zonas**:
  - **Primary instance** (zona A) — recebe todas as escritas e leituras "padrão".
  - **Standby instance** (zona B) — espelho passivo.
- Cada instância tem seu próprio **persistent disk**.

#### Como a replicação acontece

> *"Through synchronous replication to each zone's persistent disk, all writes made to the primary instance are replicated to disks in both zones before a transaction is reported as committed."*

**Replicação síncrona**. Isso significa:

```
App → INSERT INTO orders ...
        |
        v
   Cloud SQL primary (zona A)
        |
        +-- escreve no PD da zona A
        +-- escreve no PD da zona B (síncrono)
        |
        v
   só agora o COMMIT é confirmado para o app
```

**Implicação:** **zero perda de dado** em falha de zona (RPO = 0). Em troca, **cada escrita paga o custo de ida-e-volta entre zonas** — pequeno aumento de latência de gravação comparado a instância single-zone.

#### Falha e failover

Quando a zona A (primária) cai ou a instância falha:

1. O Google detecta a falha.
2. O **persistent disk** da zona B já tem **todos os dados commitados** (por causa da replicação síncrona).
3. A **standby** vira nova primary.
4. As conexões dos clientes são **roteadas para o novo primary** (mesma string de conexão — Cloud SQL usa um **endereço estável** por trás).
5. Esse processo todo é chamado de **failover**.

**Tempos típicos:** 60-120 segundos. Não é "zero downtime" — é "downtime mínimo automatizado".

#### O que HA NÃO protege

- **Falha regional completa.** HA de Cloud SQL é **multi-zona**, não multi-região. Se a região inteira cai, sua instância cai junto. Para sobreviver a isso, você usa **cross-region read replicas** + promove uma manualmente em desastre.
- **Erros lógicos.** Se você rodou `DROP TABLE orders;` por engano, a replicação síncrona **replicou esse drop fielmente** para o standby. A salvaguarda contra erro humano é **backup + PITR**, não HA.

**Armadilha de prova:**

- **HA = multi-zona regional**, não multi-região.
- **Read replicas ≠ standby.** Read replicas atendem **leitura**, não viram primary automaticamente. Standby é dedicada ao failover.
- **Failover é automático.** Você não precisa scriptar — Cloud SQL faz.

### Bloco "Backups e recuperação"

Cloud SQL oferece três mecanismos complementares:

#### 1. Backups automatizados

- Configuráveis por uma **janela diária** ("backup window").
- Retenção configurável (default ~7 dias; pode chegar a centenas com configurações apropriadas).
- Não impactam significativamente performance (snapshot do disco, não cópia integral).
- Armazenados de forma redundante pela Google.

#### 2. Backups on-demand

- Disparados manualmente via Console / `gcloud` / API.
- Útil antes de operações de risco (migração de versão, alter table grande, deploy crítico).
- Não expira por janela — fica até você apagar (ou até o limite da política).

#### 3. Point-in-time recovery (PITR)

> *"Cloud SQL also provides automated and on-demand backups with point-in-time recovery."*

- Permite restaurar o banco para **um momento específico** (não só para um snapshot).
- Mecanismo por trás: **binary logs / WAL** (write-ahead log) habilitados, combinados com o último backup.
- **Pré-requisito:** PITR **precisa estar habilitado** na instância (não vem ligado em todos os tiers / configurações).

**Quando usar cada um:**

| Cenário                                                  | Mecanismo                  |
|----------------------------------------------------------|----------------------------|
| Compliance / DR (recuperar dado de ontem)                | Backup automatizado        |
| Vou rodar uma migração arriscada agora                   | Backup on-demand           |
| Acabei de apagar 1.000 pedidos por engano às 14:32       | PITR para 14:31            |
| Quero clonar a base de produção pra ambiente de staging  | Restaurar um backup numa nova instância |

#### Import / Export

> *"You can import and export databases using mysqldump, or import and export CSV files."*

- **mysqldump** (MySQL) / **pg_dump** (Postgres) / **bacpac** (SQL Server) — formato nativo, preserva schema completo.
- **CSV** — bom para tabelas isoladas, intercâmbio com outras ferramentas.
- Os dumps são tipicamente armazenados em **Cloud Storage** (bucket). O Cloud SQL lê / escreve direto do bucket.

**Armadilha de prova:** export para Cloud Storage requer **permissão IAM da service account do Cloud SQL no bucket** (objectAdmin ou similar). Errar isso é causa comum de export falhar.

### Bloco "Scaling — vertical, horizontal, e o limite duro"

#### Scale up (vertical)

- Aumentar **CPU / RAM** da instância.
- **Requer restart** da máquina → janela de indisponibilidade (curta, mas existe).
- Storage cresce **sem restart** (auto-grow se habilitado).
- Útil quando o gargalo é uma única consulta pesada, ou volume agregado de transações.

#### Scale out (horizontal — leitura)

- **Read replicas:** instâncias somente-leitura replicadas (assíncrono) a partir da primary.
- Aplicação roteia leituras pesadas (relatórios, dashboards) para replicas e mantém escritas na primary.
- Trade-off: **lag de replicação** — leituras na replica podem ver um estado ligeiramente atrasado.

#### O que Cloud SQL **não faz**: scale out de escrita

> *"if you are concerned about horizontal scalability, you'll want to consider Spanner."*

**Pista léxica fundamental:** o exame usa **"horizontal scaling"**, **"scale writes"**, **"globally distributed"** como sinais para **Spanner**. Cloud SQL é **vertical-first** com **horizontal-só-pra-leitura**.

### Bloco "Connection types — segurança e performance"

Esse é um dos pontos onde Cloud SQL pode parecer confuso. Quatro opções; entender **quando** usar cada uma é o ponto da prova.

#### Visão geral

```
                      App quer conectar no Cloud SQL
                                  |
              +-------------------+-------------------+
              | mesmo projeto + mesma região?         |
              v                                       v
            SIM                                      NÃO
              |                                       |
              v                                       v
        PRIVATE IP                            (3 opções abaixo)
   (mais performático, sem                 1. Cloud SQL Auth Proxy ← recomendado
   internet pública envolvida)             2. SSL com certificados manuais
                                           3. IP externo autorizado (sem encriptação)
```

#### 1. Private IP (VPC peering)

**O quê:** a instância Cloud SQL recebe um IP **interno** dentro da sua VPC (via Service Networking / VPC peering). VMs da mesma VPC conectam por esse IP privado.

**Quando usar:** *"if you're connecting an application that is hosted within the same Google Cloud project as your Cloud SQL instance, and it's collocated in the same region"*.

**Por que é a melhor opção quando aplicável:**
- **Não passa pela internet pública** — *"traffic is never exposed to the public internet."*
- **Latência mais baixa** (caminho mais curto, rede da Google).
- **Segurança por default** — só quem está na VPC alcança.

**Observação técnica do vídeo:** *"connecting to the Cloud SQL Private IP address from VMs in the same region is only a performance-based recommendation and not a requirement."* Ou seja, **dá pra acessar Private IP de outra região via VPC**, mas a latência fica pior. A regra "mesma região" é **performance**, não conectividade.

**Armadilha de prova:** se o cenário diz "menor latência possível, mesma VPC, mesmo projeto" → **Private IP**. Nunca Auth Proxy nesse caso (Proxy é overhead extra).

#### 2. Cloud SQL Auth Proxy

> *"In this case, I recommend using the Cloud SQL Auth Proxy, which handles authentication, encryption, and key rotation for you."*

**O quê:** um **proxy local** (binário fornecido pela Google ou sidecar container) que roda **no host do cliente** (GCE em outra região, GKE, máquina do dev, Cloud Run, etc.) e abre um túnel autenticado e criptografado para o Cloud SQL.

**O que o proxy faz por você:**
- **Autenticação via IAM** (usa as credenciais da sua service account / usuário).
- **Encriptação TLS automática.**
- **Rotação de chaves automática** (você não gerencia certificado).
- **Mantém a string de conexão simples** — o app conecta em `localhost:5432` (ou similar) e o proxy roteia.

**Quando usar:**
- App **fora do GCP** (sua máquina, on-prem, outra cloud).
- App em **outra região** ou **outro projeto** dentro do GCP.
- Workloads serverless (Cloud Run, Cloud Functions) — usam o proxy embutido.
- Você quer **a forma mais segura** sem se preocupar com certificados.

**Armadilha de prova:** "secure, no manual SSL management, app outside the same project" → **Cloud SQL Auth Proxy**. É a opção "default" para qualquer cenário onde Private IP não rola.

#### 3. SSL com certificados auto-gerenciados

> *"If you need manual control over the SSL connection, you can generate and periodically rotate the certificates yourself."*

**O quê:** você gera certificados (client + server CA) no Console / `gcloud`, distribui pro app, e conecta usando TLS direto na porta pública do Cloud SQL.

**Quando usar:**
- Política de segurança exige **certificado gerenciado pela empresa** (custom CA, validação de quem assinou).
- Você quer **controle total** sobre rotação (data específica, processo de auditoria).
- Aplicação legada que **não roda o Auth Proxy** mas suporta TLS nativo do driver.

**Custo:** **trabalho operacional** — você precisa rotacionar manualmente (90 dias, 180, 1 ano, conforme política).

#### 4. IP externo autorizado (sem encriptação)

> *"Otherwise, you can use an unencrypted connection by authorizing a specific IP address to connect to your SQL server over its external IP address."*

**O quê:** você libera um IP público específico (ou range) na configuração da instância. O tráfego vai por **internet pública sem TLS**.

**Quando usar:**
- **Quase nunca em produção.**
- Útil para **debug rápido**, teste de conectividade, prova de conceito.
- Casos legados onde o driver não fala TLS e a rede entre app e Cloud SQL é privada por outro meio (raríssimo).

**Armadilha de prova:** se o exame oferece "unencrypted IP-allowlist" como opção pra um cenário de **app de produção** — **errado**. Sempre prefira Auth Proxy (ou Private IP).

#### Comparativo rápido

| Opção                       | Segurança | Performance | Esforço operacional | Use quando…                                |
|-----------------------------|-----------|-------------|---------------------|--------------------------------------------|
| **Private IP**              | Alta      | Máxima      | Baixo               | Mesmo projeto + mesma região (ou VPC)      |
| **Cloud SQL Auth Proxy**    | Alta      | Boa         | Baixo (binário roda) | App fora da VPC / outro projeto / serverless |
| **SSL manual**              | Alta      | Boa         | **Alto** (rotação)  | Compliance que exige CA própria             |
| **IP externo sem TLS**      | Baixa     | OK          | Médio (allowlist)   | Quase nunca em prod                         |

O instrutor reapresenta a árvore de decisão filtrada para o ramo relacional:

```
                  Precisa de in-memory data store
                  (microssegundos, spikes de tráfego)?
                              |
                  +-----------+-----------+
                  | SIM                   | NÃO
                  v                       v
              Memorystore          Workload analítico
                                   sobre dado relacional?
                                          |
                                  +-------+-------+
                                  | SIM           | NÃO
                                  v               v
                              BigQuery       Spanner vs Cloud SQL
                                                   |
                                          Precisa de horizontal
                                          scaling ou global?
                                                   |
                                           +-------+-------+
                                           | SIM           | NÃO
                                           v               v
                                       Spanner         Cloud SQL
                                                       (cost-effective)
```

**A fala do vídeo:** *"If you don't need horizontal scaling or a globally available system, Cloud SQL is a cost-effective solution."*

**Como ler "cost-effective" aqui:** Spanner cobra por **node** (unidade de capacidade), e o piso é caro. Cloud SQL cobra por **instância dimensionada ao seu tamanho real**. Para cargas pequenas/médias, Cloud SQL é **ordens de magnitude mais barato** que Spanner.

**Armadilha de prova:** se o cenário diz "**small startup, regional, SQL, gerenciado**" → **Cloud SQL**, mesmo que ofereçam Spanner como opção. "Spanner para tudo relacional" é overkill na maioria absoluta dos casos.

### Comandos `gcloud` críticos (referência para o lab e prova)

#### Criar uma instância Cloud SQL (Postgres)

```bash
gcloud sql instances create minha-instancia \
  --database-version=POSTGRES_15 \
  --tier=db-custom-2-7680 \
  --region=us-central1 \
  --storage-size=20GB \
  --storage-type=SSD \
  --availability-type=REGIONAL \      # ← REGIONAL = HA habilitado (multi-zona)
  --backup-start-time=03:00 \
  --enable-point-in-time-recovery
```

**Pistas léxicas dos flags:**
- `--availability-type=REGIONAL` → **HA ligado** (primary + standby em zonas diferentes da região).
- `--availability-type=ZONAL` → instância single-zone, sem standby.
- `--enable-point-in-time-recovery` → PITR ligado (precisa também ter binlog/WAL).

#### Criar um usuário do banco

```bash
gcloud sql users create app-user \
  --instance=minha-instancia \
  --password=trocar-em-prod
```

**Lembrar:** o instrutor disse que **usuários do banco continuam sendo sua responsabilidade**. Cloud SQL gerencia infra, não modelagem nem usuários.

#### Configurar Private IP

```bash
# 1. Aloca um range na VPC para Service Networking (uma vez por VPC)
gcloud compute addresses create google-managed-services-default \
  --global \
  --purpose=VPC_PEERING \
  --prefix-length=16 \
  --network=default

# 2. Cria o peering com o Service Networking da Google
gcloud services vpc-peerings connect \
  --service=servicenetworking.googleapis.com \
  --ranges=google-managed-services-default \
  --network=default

# 3. Cria a instância com Private IP (sem IP público)
gcloud sql instances create app-db \
  --database-version=POSTGRES_15 \
  --tier=db-custom-2-7680 \
  --region=us-central1 \
  --network=projects/SEU_PROJETO/global/networks/default \
  --no-assign-ip       # ← desabilita IP público
```

#### Rodar o Cloud SQL Auth Proxy localmente

```bash
# Linux/macOS
./cloud-sql-proxy SEU_PROJETO:us-central1:minha-instancia

# Depois, o app conecta como se fosse local:
psql -h 127.0.0.1 -U app-user -d mydb
```

**O que está acontecendo:** o proxy abre uma porta em `127.0.0.1:5432`, autentica via IAM com a service account do ambiente, encripta o túnel, e fala com o Cloud SQL pela rede da Google. Você nunca lida com certificado.

#### Failover manual de HA (teste)

```bash
gcloud sql instances failover minha-instancia
```

**Quando usar:** simular falha em teste, ou intervir se a Google demorar a fazer o failover automático em incidente real.

#### Criar uma read replica

```bash
gcloud sql instances create minha-replica \
  --master-instance-name=minha-instancia \
  --tier=db-custom-2-7680 \
  --region=us-central1
```

**Observação:** a réplica **pode estar em outra região** (cross-region read replica) — útil para DR e para servir leitura regional baixa-latência a usuários geograficamente distantes.

## Cenário real

### Onde Cloud SQL encaixa nos seus projetos

**Portaria / TaxIA:**
- Você usa Postgres pra dados transacionais (usuários, conversas, transações, configurações de conta). **Cloud SQL Postgres é a tradução direta** se migrar pra GCP — mantém o mesmo Postgres, ganha backups, HA, patches.
- Como o projeto é regional (clientes no Brasil), **Cloud SQL é a escolha**. Spanner seria overkill (mais caro, mais complexo, sem ganho).
- HA (`REGIONAL`) faria sentido pra produção real. Em dev/staging, `ZONAL` economiza.

**Quando você migraria para AlloyDB ao invés de Cloud SQL:**
- Quando começar a rodar **dashboards analíticos em cima do mesmo dado transacional** (ex.: BI em tempo real do uso dos clientes) e o Postgres começar a sofrer com a mistura de cargas. Aí AlloyDB elimina a necessidade de ETL pra um warehouse.

**Quando você iria pra Spanner:**
- Cliente enterprise que opera em múltiplos continentes com SLA de consistência forte (ex.: marketplace global, plataforma financeira multinacional). Hoje, **nenhum dos seus projetos justifica**.

### Padrões comuns que combinam Cloud SQL com outros serviços

- **Cloud SQL + Memorystore Redis:** cache de leituras quentes. Postgres atende escrita + consulta inicial; Redis atende leitura repetida da mesma chave em sub-milissegundos.
- **Cloud SQL + Cloud Storage:** arquivos binários (imagens, PDFs) ficam no Cloud Storage; metadados/relações ficam no Cloud SQL.
- **Cloud SQL + BigQuery:** dado transacional fresco fica no Cloud SQL; export noturno pra BigQuery alimenta dashboards históricos.
- **Cloud SQL + Cloud Run / GKE:** Cloud Run conecta via Auth Proxy; GKE pode usar Private IP via VPC.

### Erro comum no mundo real

Subir **um Postgres numa VM Compute Engine** "porque parece mais barato" e descobrir 6 meses depois que:
- Você esqueceu de configurar backup automatizado e perdeu 3 dias de dado num crash.
- Você não tem HA porque nunca configurou replicação.
- Os patches de segurança do Postgres ficaram desatualizados.
- O custo da operação manual (seu tempo) é maior que o Cloud SQL.

**Lição:** managed compensa quase sempre. Self-managed só faz sentido se você tem **necessidade técnica específica** que o managed não cobre (engine não suportado, extensão exótica, requisito regulatório raríssimo).

## Armadilhas de prova

1. **"Horizontal scaling" / "scale writes" / "globally distributed" → Spanner, NÃO Cloud SQL.** Cloud SQL escala leitura via read replicas, mas escrita é vertical.

2. **"Patches automáticos" ≠ "tudo automático".** Você ainda gerencia **usuários, schema, queries**. Se a pergunta listar "manage user authentication" como o que Cloud SQL faz por você — **errado**. Authentication tools são nativos do engine, você administra.

3. **HA = multi-zona dentro da região.** Não é multi-região. Cenário "survive a regional outage" → não é HA do Cloud SQL, é **cross-region read replica + promote**.

4. **Failover é automático.** Não precisa scriptar. A pergunta pode oferecer "configure a script to detect failure and switch" — errado, o Cloud SQL faz isso por você quando `--availability-type=REGIONAL`.

5. **Read replicas ≠ standby.** Read replicas servem leitura (com lag assíncrono); a **standby** de HA é dedicada ao failover (replicação síncrona). Não são intercambiáveis.

6. **Replicação síncrona de HA tem custo de latência de escrita.** Não é "grátis". Você ganha durabilidade multi-zona em troca de uma RTT extra por commit. Cenário "minimum write latency" → **ZONAL** pode ser preferido em troca de menos disponibilidade.

7. **Engines suportados: MySQL, PostgreSQL, SQL Server.** Não Oracle, não MariaDB nativo, não Db2. Se a prova listar "Cloud SQL for Oracle" — não existe.

8. **Conexão "mesma região, mesma VPC" → Private IP é a recomendação.** Se a opção "Cloud SQL Auth Proxy" aparece junto, **Private IP ganha em performance**.

9. **Conexão "external app / different project" → Cloud SQL Auth Proxy é a recomendação default.** Não SSL manual (a menos que compliance exija CA própria).

10. **"Unencrypted IP-allowlist" raramente é a resposta certa.** Se o cenário fala em produção, segurança, conformidade — descarte essa opção mesmo se parecer "mais simples".

11. **PITR exige flag explícita.** Não vem ligado em toda instância. Se a pergunta perguntar "como recuperar pra um momento específico?", a resposta inclui que **PITR precisa estar habilitado** (preferencialmente desde a criação, ou os WAL/binlogs precisam ser ativados).

12. **Import/Export usa Cloud Storage como intermediário.** Se a opção "import directly from a local file via gcloud" aparece — está errada. Você sobe o dump pro bucket, daí o Cloud SQL importa.

13. **"Microsecond latency" + cache → Memorystore, não Cloud SQL.** Cloud SQL é "alguns ms", não sub-ms. Se a pergunta exige sub-ms, é Memorystore na frente do Cloud SQL.

14. **Limites duros importam.** Se o cenário fala em **dataset >64 TB** ou **>60.000 IOPS**, Cloud SQL **não atende** sozinho. Vai para Spanner, ou particiona o dado.

15. **Spanner ≠ Cloud SQL Premium.** Spanner não é "uma versão maior" do Cloud SQL — é arquitetura completamente diferente (SQL distribuído proprietário), com **modelo de pricing diferente** (nodes) e **API diferente** (compatível com SQL ANSI, mas extensões específicas).
