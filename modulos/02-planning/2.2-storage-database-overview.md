## Conceitos

### Bloco "Visão geral do módulo"

#### O problema que o módulo resolve

Toda aplicação precisa **armazenar e recuperar dados**. Pode ser:
- dado de negócio (pedidos, usuários, transações),
- mídia para streaming (vídeos, imagens, áudio),
- dado de sensor / telemetria (IoT, logs).

Do ponto de vista da **aplicação**, o que importa não é o nome do produto (banco vs object store), e sim **se o serviço atende os requisitos de armazenamento e recuperação** para as características daquele dado:
- volume,
- padrão de acesso (read-heavy, write-heavy, mixed),
- latência exigida,
- estrutura (estruturado/semi/não-estruturado),
- consistência (eventual ou forte; regional ou global),
- frequência de atualização.

Esse é o **mindset de planejamento** que a prova ACE espera: você lê o cenário, extrai pistas, e mapeia para o serviço.

#### Escopo declarado pelo instrutor (importante)

> *"The purpose of this module is to explain which services are available and when to consider using them from an infrastructure perspective."*

Ou seja, este módulo (e o exame ACE) **NÃO cobrem**:
- design de schema,
- normalização / desnormalização,
- modelagem de row keys do Bigtable,
- particionamento e clustering profundo do BigQuery,
- otimização de queries.

Esses tópicos pertencem aos cursos de **Data Engineering**. O ACE quer que você saiba:
- **escolher** o serviço,
- **provisionar** (Console / `gcloud` / Terraform),
- **conectar** uma aplicação ao serviço,
- entender **billing / lifecycle / storage classes** quando aplicável.

**Armadilha de estudo:** não caia na tentação de aprender modelagem profunda de Bigtable agora. Foque em "quando usar" e "como criar". Sobra tempo pra mergulhar nos casos do dia a dia depois.

### Bloco "Catálogo dos serviços (tabela de visão geral)"

O instrutor apresenta a tabela abaixo logo de cara. Memorizar essa tabela = ganhar boa parte das DQs de planejamento.

| Serviço         | Tipo                              | Bom para                                                                  | Uso típico                                                  |
|-----------------|-----------------------------------|---------------------------------------------------------------------------|-------------------------------------------------------------|
| **Cloud Storage** | Object storage (blob)            | Arquivos imutáveis, qualquer tamanho, acesso por HTTP                     | Imagens, vídeos, backups, logs, data lake "frio"            |
| **Filestore**   | File storage (NFS gerenciado)     | Várias VMs compartilhando um sistema de arquivos POSIX                    | HPC, lift-and-shift de apps que assumem NFS, render farms   |
| **Cloud SQL**   | Relational SQL (MySQL/Postgres/SQL Server) | OLTP regional gerenciado                                       | Apps web tradicionais, e-commerce, ERPs                     |
| **Spanner**     | Relational SQL distribuído (global) | OLTP em **escala global** com consistência forte                       | AdTech, FinTech, gaming global, marketplaces planetários    |
| **AlloyDB**     | Relational Postgres + analytics  | Postgres-compatible com **HTAP** (transacional + analítico no mesmo lugar) | Apps que precisam de BI em tempo real sobre dados operacionais |
| **Firestore**   | NoSQL de documentos              | Sync com clients mobile/web, ACID em documento                            | Mobile apps, perfis de usuário, game state                  |
| **Bigtable**    | NoSQL wide-column                | Petabyte-scale, latência previsível, alto throughput de leitura e escrita | IoT, AdTech, FinTech, séries temporais, fast lookup         |
| **Memorystore** | In-memory (Redis / Memcached)    | Cache de aplicação, sub-milissegundo                                      | Sessões, leaderboards, rate limiting, cache de query        |
| **BigQuery**    | Data warehouse analítico          | Queries SQL ad-hoc em volumes grandes; **dado que não muda muito**       | BI, dashboards, analytics histórico                         |

#### Por que BigQuery aparece "à parte" no slide

O instrutor faz questão de notar: **BigQuery está no limite entre data storage e data processing**. Você pode armazenar dados nele, mas o **propósito declarado** é análise interativa e processamento em massa, não "ser o banco da aplicação".

Detalhe importante mencionado no vídeo:
- BigQuery tem **DML** (data manipulation language): você consegue `UPDATE`, `INSERT`, `DELETE` em tabelas — mas...
- ...ele tem **cache embutido**, então **funciona muito melhor quando os dados não mudam com frequência**.

**Tradução prática:** BigQuery não é o lugar onde você grava cada clique do usuário em tempo real e depois faz UPDATE pra atualizar o estado. Para isso, use Bigtable / Cloud SQL / Firestore. BigQuery é onde você **lê** o histórico desses cliques em lote.

#### Como ler / aplicar a árvore na prova

Cada nó é uma **pergunta binária** que você responde lendo o enunciado:

1. **"Estruturado?"** — se o dado tem schema (linhas e colunas, atributos previsíveis) é estruturado. Se é blob/imagem/vídeo/arquivo solto, é não-estruturado.
2. **"Shared file system?"** — se o enunciado fala em "várias VMs montando o mesmo volume", "NFS", "POSIX", "render farm", "HPC compartilhando arquivos" → **Filestore**. Se é só "armazenar arquivos via API" → **Cloud Storage**.
3. **"Analítico?"** — se a pergunta fala "analyze", "historical", "dashboards", "reporting", "SQL queries by analysts", "big data" → analítico.
4. **"Relacional?"** — se há schema fixo, joins, integridade referencial, transações ACID multi-tabela → relacional.
5. **"HTAP?"** — se a aplicação precisa **OLTP + OLAP no mesmo banco** (sem ETL pra warehouse separado) → **AlloyDB**.
6. **"Escala global?"** — se o enunciado fala "global users", "multiple continents", "horizontal scale", "strong consistency worldwide" → **Spanner**. Senão → **Cloud SQL**.
7. **"Cache?"** — se o cenário fala "in-memory", "sub-millisecond", "session store", "leaderboard", "rate limit", "cache the database" → **Memorystore**.

> **Dica de leitura:** sempre suba a árvore de baixo pra cima. Se o cenário deu **uma pista clara** (ex.: "globally distributed users with SQL"), você pode pular direto pro nó certo (Spanner) sem ir do começo.

### Bloco "Os três serviços novos (não-cobertos no workbook ACE)"

Esses três o workbook ACE não detalhou — entram aqui pela primeira vez.

#### Filestore — file system NFS gerenciado

**O quê:** serviço gerenciado de **NFS (Network File System)**. Você cria uma instância Filestore e várias VMs do GCE (ou pods do GKE) montam aquele "volume" como se fosse um disco compartilhado POSIX.

**Por quê existe:** Cloud Storage não tem **semântica de arquivo POSIX**. Não dá pra fazer `fopen`, `seek`, `lock`, `chmod`, `ln -s`. Muitos workloads legados e científicos **assumem** essa semântica. Filestore preenche esse buraco.

**Quando usar:**
- **Lift-and-shift** de apps on-prem que dependem de NFS compartilhado (ERPs antigos, CMSes, sistemas de imagem médica).
- **HPC / batch jobs** com várias VMs lendo/escrevendo os mesmos arquivos.
- **Render farms** (3D, vídeo) que compartilham assets e geram outputs.
- **CI/CD** com workspace compartilhado entre runners.
- **GKE** com `ReadWriteMany` persistent volumes — Filestore é o backend que aguenta.

**Quando NÃO usar:**
- Servir mídia pela internet → **Cloud Storage** (mais barato, escala melhor, tem CDN).
- Banco de dados → **Cloud SQL / Postgres em VM** (banco no NFS é antipattern).
- Backups → **Cloud Storage** (classes Nearline / Coldline / Archive).

**Analogia mental:** Cloud Storage é "S3 / dropbox". Filestore é "um drive de rede da empresa". Coisas completamente diferentes apesar de ambas guardarem "arquivos".

**Armadilha de prova:** "shared file system" e "NFS" são as pistas léxicas que disparam Filestore. **Nunca confundir** com Cloud Storage — Cloud Storage **não é** um sistema de arquivos.

**Exemplo:**
```bash
# Criar uma instância Filestore (tier BASIC_HDD, 1 TiB, na zone us-central1-a)
gcloud filestore instances create nfs-share \
  --zone=us-central1-a \
  --tier=BASIC_HDD \
  --file-share=name="vol1",capacity=1TB \
  --network=name="default"

# Numa VM Linux, montar:
sudo mount -t nfs <FILESTORE_IP>:/vol1 /mnt/share
```

#### AlloyDB — Postgres + HTAP

**O quê:** banco **Postgres-compatible** gerenciado pelo Google, com arquitetura própria que combina **OLTP** (transacional tradicional) e **OLAP** (analítico) no mesmo lugar.

**HTAP = Hybrid Transactional and Analytical Processing.** É um banco que entrega **as duas coisas simultaneamente**, sem precisar exportar dado pra um warehouse separado.

**Por quê existe:** o padrão clássico é:
- Postgres / MySQL fazendo OLTP (apps gravam pedidos, transações, etc.).
- ETL noturno copia tudo pro warehouse (BigQuery / Snowflake / Redshift).
- Analistas rodam queries no warehouse, sempre com **defasagem** (dado de ontem, no mínimo).

Para muitos casos isso é fino. Mas quando o negócio precisa de **BI em tempo real** ("quantas vendas tivemos nos últimos 5 minutos?", "qual fraude está acontecendo agora?"), a defasagem mata. AlloyDB elimina o ETL: você roda OLTP **e** analítico no mesmo banco, sem perder performance em nenhum dos dois.

**Quando usar:**
- App Postgres-compatible com necessidade de **analytics em tempo real** sobre o dado operacional.
- Migração de Postgres on-prem com requisitos de performance/escala que o Cloud SQL não atende.
- Casos em que você usaria **Cloud SQL + BigQuery sincronizados** — pode considerar consolidar no AlloyDB.

**Quando NÃO usar:**
- App Postgres "comum" sem necessidade analítica → **Cloud SQL Postgres** (mais simples, mais barato).
- Workload puramente analítico, batch, com SQL ad-hoc → **BigQuery**.
- Necessidade de **distribuição global com consistência forte** → **Spanner**.

**Analogia:** AlloyDB é "Postgres com superpoderes analíticos embutidos". Cloud SQL é "Postgres normal gerenciado". Spanner é "outra coisa, não é Postgres por baixo, é SQL distribuído proprietário".

**Armadilha de prova:** se o cenário mencionar **HTAP**, **"analytical and transactional"**, **"real-time analytics on operational data"**, ou **Postgres + escala que Cloud SQL não dá** → AlloyDB.

#### Memorystore — in-memory gerenciado (Redis / Memcached)

**O quê:** serviço gerenciado de **cache in-memory**. Você escolhe o engine: **Redis** (mais comum, suporta estruturas ricas — listas, sets, sorted sets, pub/sub) ou **Memcached** (mais simples, só key-value).

**Por quê existe:** acessar disco (mesmo SSD) é ordens de magnitude mais lento que acessar RAM. Pra workloads sensíveis a latência (sub-milissegundo), você precisa de cache em memória. Subir e operar Redis você mesmo é chato (HA, replicação, backup, failover) — Memorystore gerencia isso.

**Quando usar:**
- **Cache de query de banco:** Postgres demora 50 ms numa query, Redis devolve em 0,5 ms. Cache o resultado.
- **Sessões de usuário:** session store dos seus apps web.
- **Rate limiting:** contadores rápidos por IP / usuário.
- **Leaderboards:** sorted sets do Redis são imbatíveis para "top 100 jogadores".
- **Pub/sub leve** dentro do app.
- **Filas / job queues** simples (com Redis).

**Quando NÃO usar:**
- **Storage primário** — Memorystore é cache. Em alguns tiers o dado é persistido, mas **filosofia** é "se perder, restauro do banco real". Nunca é o source of truth.
- **Dados grandes** — RAM custa caro. Se o conjunto é TB, Memorystore não cabe.

**Armadilha de prova:** pista léxica = "**caching**", "**in-memory**", "**sub-millisecond**", "**session store**", "**leaderboard**". O instrutor menciona explicitamente: *"If caching is a requirement, choose Memorystore."*

**Memorystore for Redis vs Memcached:**

| Característica          | Redis                                        | Memcached                       |
|-------------------------|----------------------------------------------|---------------------------------|
| Estruturas              | Strings, listas, sets, sorted sets, hashes, streams | Apenas key-value (string)       |
| Persistência            | Opcional (snapshot / AOF dependendo do tier) | Nenhuma (puro cache volátil)    |
| HA / replicação         | Sim (Standard tier)                          | Não (multi-node sharding apenas) |
| Caso típico             | Cache rico, leaderboard, sessão, fila        | Cache simples de query, hot keys |

> **Quando bater a dúvida:** comece com Redis. Memcached só faz sentido se você **explicitamente** quer um cache simples e barato, sem nenhuma feature de estrutura ou persistência.

### Bloco "Os serviços que já tinham aparecido — releitura curta"

Apenas o que **este vídeo** acrescenta ou enfatiza. A documentação profunda continua no `2.2-data-storage.md`.

**Cloud Storage** — *"if you don't need a shared file system, choose Cloud Storage."* O instrutor reforça: é a opção default para **dado não-estruturado** que não precisa de NFS.

**Cloud SQL** — opção para **relacional sem HTAP e sem necessidade de escala global**. É a "porta da frente" do mundo relacional gerenciado.

**Spanner** — entra quando precisa de **relacional + escala global**. A pista é "global", "horizontal scale", "strong consistency worldwide".

**Firestore** — entra no ramo **não-relacional sem cache**. É o NoSQL document gerenciado.

**Bigtable** — no ramo **estruturado + analítico**, é a opção quando você precisa de **baixa latência** e **muitas leituras/escritas em escala**. O instrutor reforça o uso em **IoT, AdTech, FinTech**, e também o termo **"fast lookup non-relational database for datasets too large to store in memory"**.

**BigQuery** — no ramo **estruturado + analítico**, é a opção quando você quer **SQL ad-hoc** sobre dado que **não muda muito**. A frase-chave do vídeo: *"BigQuery works really well in cases where the data does not often change."*

> **Bigtable vs BigQuery — pista decisiva:** se o cenário enfatiza **baixa latência por linha** e **alta taxa de updates** → Bigtable. Se enfatiza **queries SQL ad-hoc**, **dashboards**, **dado estável / append-only** → BigQuery.

## Cenário real

**Como esse mindset de decisão se aplica fora da prova:**

- **Portaria / TaxIA hoje:** uso Postgres pra dados transacionais e o filesystem do servidor pra arquivos. Se migrasse pro GCP:
  - Postgres → **Cloud SQL Postgres** (sem HTAP, regional, ok). Se um dia precisasse fazer BI em tempo real sobre as conversas, aí avaliaria **AlloyDB**.
  - Arquivos (imagens vindas do WhatsApp, PDFs) → **Cloud Storage**. Não preciso de NFS — meus serviços acessam por HTTP/SDK, não por `fopen`.
  - Sessões e rate limit → **Memorystore Redis** (hoje uso Redis num container; Memorystore terceiriza HA/backup).

- **Quando eu usaria Filestore?** Se montasse um pipeline de processamento de vídeo com várias VMs renderizando frames compartilhados, ou se migrasse um app legado que escreve em `/var/data` esperando NFS.

- **Quando eu usaria Bigtable?** Telemetria de portaria em escala (milhões de eventos/dia indexados por timestamp) — não cabe em Postgres com performance previsível. Bigtable + BigQuery por cima (Bigtable guarda evento bruto com latência baixa; BigQuery roda análise histórica).

- **Quando eu usaria Spanner?** Difícil justificar no negócio atual. Faria sentido só se tivesse uma operação multi-país com SLA financeiro estrito.

**Armadilha que cometeria no trabalho real:** colocar tudo em Cloud Storage "porque é fácil". Mas se duas VMs precisam compartilhar arquivos com lock, Cloud Storage **não resolve** — eu inventaria um lock manager horrível ao invés de simplesmente usar Filestore.

## Armadilhas de prova

1. **"Shared file system" → Filestore.** Nunca Cloud Storage. Cloud Storage é blob/object, não tem semântica POSIX.
2. **"NFS" no enunciado → Filestore.** Determinístico.
3. **"HTAP" / "real-time analytics on operational data" → AlloyDB.** O workbook não usa esse termo, mas o curso oficial usa — pode cair.
4. **"Caching" / "in-memory" / "sub-millisecond" → Memorystore.** Não é Firestore só porque é "rápido" — Firestore é NoSQL document, não cache.
5. **"Global" + "relacional" → Spanner**, **não Cloud SQL**. Cloud SQL é regional.
6. **"SQL" sozinho não decide nada.** BigQuery, Cloud SQL, Spanner e AlloyDB todos falam SQL. Você precisa de outra pista (analítico? global? HTAP?).
7. **"Data doesn't change often" + "SQL" + "analytics" → BigQuery.** O cache embutido do BigQuery o torna a melhor escolha pra dado append-only.
8. **"Heavy reads and writes at scale" + "low latency" → Bigtable**, mesmo que o enunciado fale em "analytics". Bigtable cobre o lado operacional do analítico; BigQuery cobre o lado ad-hoc.
9. **Bigtable e BigQuery não são intercambiáveis.** Mesmo prefixo, papéis diferentes:
   - Bigtable = NoSQL operacional, baixa latência por row key.
   - BigQuery = warehouse SQL ad-hoc.
10. **Memorystore não é storage primário.** Se o cenário fala "I need to persist user orders" e oferece Memorystore como opção — está errada. Cache nunca é source of truth.
11. **AlloyDB ≠ Cloud SQL Postgres.** Os dois falam Postgres, mas AlloyDB tem arquitetura própria voltada a HTAP e escala. Cloud SQL é Postgres "puro" gerenciado.
12. **Escopo do exame:** se a pergunta entra em **design de schema** ou **modelagem de row keys**, é tema de Data Engineering — **não cai** no ACE com profundidade.
