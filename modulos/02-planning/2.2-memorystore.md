# 2.2 — Memorystore
> **Fonte:** Google Cloud Skills Boost — *Essential Google Cloud Infrastructure: Core Services*, Módulo 3 — vídeo "Memorystore" (overview rápido conforme antecipado no vídeo de abertura do módulo).
> **DQs do workbook cobertas:** nenhuma diretamente. Memorystore aparece quando o cenário enfatiza **cache**, **in-memory**, **sub-millisecond**, **session store**, **leaderboard**, **rate limiting**.
> **Considerações da prova:** Memorystore é o **default in-memory** do GCP. A confusão clássica é confundi-lo com Firestore (Firestore não é cache) ou achar que Memorystore substitui banco primário (não substitui — sempre é cache em frente a um source-of-truth).

---

## Conceitos

### Bloco "O que é Memorystore — definição precisa"

> *"Memorystore for Redis provides a fully managed in-memory datastore service built on scalable, secure, and highly available infrastructure managed by Google."*

Memorystore é o serviço gerenciado de **cache in-memory** do GCP. Ele tem **dois engines**:

| Engine                       | Foco                                                  |
|------------------------------|-------------------------------------------------------|
| **Memorystore for Redis**    | Padrão moderno — estruturas ricas (sorted sets, lists, hashes, streams, pub/sub) |
| **Memorystore for Memcached**| Cache simples key-value, sharding multi-node          |

> *(O vídeo deste módulo foca em Redis. Memcached aparece como opção secundária, tratada com mais detalhe no overview do módulo `2.2-storage-database-overview.md`.)*

O **propósito** de Memorystore: dar a você a **latência de RAM** (sub-ms) sem o esforço operacional de manter um cluster Redis em produção.

#### Por que cache importa

- **Disco SSD** acessa em ~100 µs (microssegundos).
- **Rede** entre data centers acessa em ~1 ms (millissegundos).
- **RAM local** acessa em ~100 ns (nanossegundos) — **1.000x mais rápido** que SSD.
- **Memorystore (RAM em rede)** acessa em ~500 µs — **sub-millisegundo**, ordens de magnitude mais rápido que SSD em rede.

Para workloads que **leem o mesmo dado muitas vezes**, cachear em Memorystore reduz drasticamente latência percebida pelo usuário.

#### Por que usar Memorystore em vez de subir Redis numa VM?

> *"Applications running on Google Cloud can achieve extreme performance by leveraging the highly scalable, available, secure Redis service without the burden of managing complex Redis deployments."*

Subir Redis você mesmo é fácil **até dar problema**. Aí você precisa:
- Configurar **réplica + Sentinel** ou **Redis Cluster** para HA.
- Implementar **failover** (sentinel quórum, promover réplica, redirecionar clients).
- Aplicar **patches** de segurança regularmente.
- Configurar **backup** (RDB/AOF), testar restore.
- Monitorar uso de memória, evictions, latency, conexões.
- Lidar com **out-of-memory** sem corromper o estado.

> *"Memorystore also automates complex tasks like enabling high availability, failover, patching, and monitoring."*

Esses 4 tópicos viram problema do Google. Você foca em **chave-valor + lógica do app**.

### Bloco "Capacidades anunciadas — números do vídeo"

| Métrica                            | Valor                                       |
|------------------------------------|---------------------------------------------|
| **SLA de HA**                      | **99,9%**                                   |
| **Latência**                       | **Sub-millisecond**                         |
| **Capacidade máxima por instância**| **Até 300 GB**                              |
| **Throughput de rede**             | **Até 12 Gbps**                             |
| **HA**                             | Replicação automática entre **2 zonas**     |
| **Failover**                       | Automático                                  |

#### Como ler esses números

- **300 GB de RAM** é muito — cabe sessões de **milhões** de usuários, leaderboards gigantes, cache de query de Postgres com folga.
- **12 Gbps** de rede é o teto antes de saturar. Workloads sensíveis a latência geralmente saturam CPU primeiro.
- **99,9%** = ~8h 45min de downtime/ano permitido. Não é "five nines" — porque cache **pode ficar fora do ar** sem derrubar o app (você cai pra ler do banco primário, mais lento).

**Implicação importante:** Memorystore **não é source of truth**. O app sempre tem que ter um banco primário (Postgres, Firestore, etc.) por trás. Se Memorystore cair, o app continua funcionando, só mais lento.

#### Crescimento gradual

> *"Start with the lowest tier and smallest size, and then grow your instance effortlessly with minimal impact to application availability."*

- Você começa com instância pequena ($X/mês).
- Resize horizontal/vertical com **mínimo impacto** — o Memorystore reconfigura sem perder o cache (na maioria dos casos).
- Estratégia recomendada: provisionar conforme cresce, não overprovisionar de cara.

### Bloco "Tiers — Basic vs Standard"

O vídeo não detalha, mas a documentação separa Memorystore for Redis em dois tiers:

| Característica           | **Basic Tier**                    | **Standard Tier**                  |
|--------------------------|-----------------------------------|------------------------------------|
| HA / replicação          | ❌ (single node)                  | ✅ (réplica em outra zona)         |
| Failover automático      | ❌                                | ✅                                 |
| SLA                      | Sem SLA formal                    | **99,9%**                         |
| Persistência (RDB)       | ✅ (opcional)                     | ✅ (opcional)                     |
| Read replicas            | ❌                                | ✅ (até 5)                         |
| Quando usar              | Dev, staging, cache "descartável" | Produção                           |

**Armadilha de prova:** se o cenário pede "production cache with 99.9% SLA", a resposta inclui **Standard tier** (não Basic).

### Bloco "Compatibilidade — lift-and-shift de Redis open source"

> *"Because Memorystore for Redis is fully compatible with the Redis protocol, you can lift-and-shift your applications from open source Redis to Memorystore without any code changes by using the import/export feature."*

> *"There is no need to learn new tools because all existing tools in client libraries just work."*

**Mensagem-chave:** Memorystore não inventa protocolo nem API novos. Ele fala **Redis protocol RESP3** e é compatível com:

- **Todos os client libraries** (redis-py, ioredis, jedis, lettuce, go-redis, etc.).
- **Todas as ferramentas de CLI** (redis-cli funciona apontando para o endpoint do Memorystore).
- **Todos os comandos Redis** (com algumas exceções de comandos administrativos — você não roda `CONFIG SET` em coisas que o Memorystore gerencia).
- **Formato de export** (RDB) compatível com Redis open source.

**Implicação prática:** se sua app hoje roda Redis num container ou numa VM, migrar para Memorystore é **trocar a string de conexão**. Zero refactoring.

#### Import/Export

- **Export:** dump RDB do estado do Memorystore para um bucket Cloud Storage.
- **Import:** restaurar um RDB (do Redis open source ou outro Memorystore) para uma nova instância.

**Casos de uso:**
- Migração on-prem → Memorystore.
- Backup / DR.
- Clone de produção para staging.
- Mudar de tier (Basic → Standard) preservando dados.

### Bloco "Quando usar Memorystore — casos de uso canônicos"

Memorystore aparece sempre que aparecerem essas pistas léxicas:

| Pista                                | Caso de uso típico                                  |
|--------------------------------------|------------------------------------------------------|
| "sub-millisecond latency"            | Genérico — cache para baixar p50/p99 da aplicação    |
| "session store"                      | Sessões de usuário em apps web (login state, JWT cache) |
| "leaderboard" / "top N"              | Sorted sets do Redis — `ZADD` / `ZRANGE` são imbatíveis |
| "rate limiting"                      | Contador por IP/usuário com TTL                      |
| "real-time analytics"                | Counters, sets, sketches (HyperLogLog)               |
| "cache database queries"             | Cache aside / read-through na frente de Postgres     |
| "pub/sub leve"                       | Notificação entre serviços (não substitui Pub/Sub real) |
| "queue / job queue simples"          | Listas Redis (`LPUSH` / `RPOP`)                      |
| "real-time gaming"                   | State efêmero, ranking, matchmaking                  |
| "lift-and-shift Redis app"           | Migração de Redis on-prem                            |

#### Quando NÃO usar Memorystore

| Pista                                                | Use isso ao invés                              |
|------------------------------------------------------|------------------------------------------------|
| Storage primário durável                             | Cloud SQL / Firestore / Bigtable               |
| Dados grandes (TB+)                                  | Bigtable / BigQuery                            |
| ACID multi-row                                       | Cloud SQL / Spanner / Firestore                |
| Queries SQL complexas                                | Cloud SQL / BigQuery                           |
| Mobile sync                                          | Firestore                                      |
| Time series massivo                                  | Bigtable                                       |
| Pub/Sub robusto entre serviços                       | **Pub/Sub** (serviço dedicado)                 |

### Bloco "Padrões de uso clássicos"

Esses padrões aparecem como cenários em entrevistas/projetos reais — bom entender.

#### 1. Cache aside (lazy loading)

```
1. App lê: GET key
   ├── cache hit → retorna valor (sub-ms)
   └── cache miss → busca no banco primário → escreve no cache → retorna

2. App escreve: invalida ou atualiza a key no cache + escreve no banco
```

**Vantagem:** simples, app controla. **Desvantagem:** primeira request de cada key é lenta.

#### 2. Read-through / Write-through

Camada de cache **transparente** entre app e banco. App não sabe se foi cache ou banco. Mais comum em frameworks ORM.

#### 3. Session store

```
Usuário faz login → app gera session_id → Memorystore.set("sess:" + id, user_data, ttl=1h)
Requests subsequentes → Memorystore.get("sess:" + id) → restaura contexto
```

**Vantagem sobre cookie:** dados grandes (preferences, permissions) sem inflar header HTTP.

#### 4. Leaderboard (sorted set)

```
ZADD leaderboard 12500 player_alice
ZADD leaderboard 9800 player_bob
ZADD leaderboard 15200 player_charlie

ZREVRANGE leaderboard 0 9  → top 10 players, ordenados, em O(log N)
```

**Por que Redis vence:** sorted sets são uma das estruturas mais bem-otimizadas no Redis. Tentar fazer top 10 em Postgres com `ORDER BY ... LIMIT 10` em tabela de milhões é dezenas de ms; sorted set faz em sub-ms.

#### 5. Rate limiting

```
INCR rate:user_42:minute_1234567 ex 60
↓
Se valor > limite (ex: 100 req/min) → bloqueia
```

Atômico (`INCR`), com TTL para expirar automaticamente.

#### 6. Fila de tarefas simples

```
Worker: BRPOP queue:tasks 0  (bloqueia até ter tarefa)
Producer: LPUSH queue:tasks "task_data"
```

Funciona, mas para volume sério use **Pub/Sub** ou **Cloud Tasks** (durabilidade garantida).

### Bloco "Memorystore for Redis vs Memorystore for Memcached"

O vídeo foca em Redis, mas o exame pode pedir comparativo.

| Característica                    | **Redis**                                      | **Memcached**                       |
|-----------------------------------|------------------------------------------------|-------------------------------------|
| Estruturas de dado                | Strings, listas, sets, sorted sets, hashes, streams, bitmaps, HyperLogLog | Apenas key-value (string)           |
| Persistência                      | Opcional (RDB / AOF)                           | Nenhuma (puro cache volátil)        |
| HA / replicação                   | ✅ (Standard tier)                             | ❌ (multi-node sharding, sem replica) |
| Pub/Sub                           | ✅                                             | ❌                                  |
| Transações (MULTI/EXEC)           | ✅                                             | ❌                                  |
| Multi-threading                   | Redis 6+ tem I/O threads; comando ainda single-thread | ✅ verdadeiro multi-thread       |
| Caso típico                       | Cache rico, leaderboard, sessão, fila          | Cache simples de query, hot keys    |
| Quando escolher                   | Default — quase sempre Redis                   | Workload puro de cache horizontal massivo |

**Regra prática:** comece **Redis**. Memcached só faz sentido se você **explicitamente** precisa de cache simples horizontal massivo sem features avançadas.

---

### Bloco "Networking — Private IP only"

Memorystore **só aceita conexão via VPC** (private IP). Não tem IP público.

```
App (GCE / GKE / Cloud Run / etc.)
        │
        │ (mesma VPC, qualquer região)
        ▼
   Memorystore instance (IP privado)
```

**Implicações:**
- App **fora da VPC** (sua máquina local, outra cloud) não consegue conectar diretamente. Você precisa de **VPN, Interconnect, ou bastion host**.
- **Cloud Run** precisa de **Serverless VPC Access** configurado para alcançar Memorystore.
- **Cloud Functions** mesma coisa.

**Por que isso é bom (mesmo sendo chato):** segurança por default. Nenhuma chance de Memorystore ficar exposto à internet por acidente.

### Comandos `gcloud` críticos (referência)

#### Criar uma instância Memorystore for Redis

```bash
# Basic tier, 1 GB, single zone
gcloud redis instances create cache-dev \
  --size=1 \
  --region=us-central1 \
  --redis-version=redis_7_0 \
  --tier=basic

# Standard tier (HA), 5 GB, com auth
gcloud redis instances create cache-prod \
  --size=5 \
  --region=us-central1 \
  --redis-version=redis_7_0 \
  --tier=standard \
  --enable-auth \
  --transit-encryption-mode=SERVER_AUTHENTICATION
```

**Flags importantes:**
- `--tier=basic` (sem HA) vs `--tier=standard` (HA, 99,9% SLA).
- `--size=N` em GB. Redondo de 1 a 300.
- `--enable-auth` exige password no client.
- `--transit-encryption-mode=SERVER_AUTHENTICATION` ativa TLS.

#### Obter endpoint para conectar

```bash
gcloud redis instances describe cache-prod \
  --region=us-central1 \
  --format="value(host,port)"
# Retorna ex: 10.0.0.3   6379
```

#### Conectar de uma VM da mesma VPC

```bash
redis-cli -h 10.0.0.3 -p 6379
> SET hello "world"
> GET hello
```

#### Export para Cloud Storage

```bash
gcloud redis instances export gs://meu-bucket/redis-backups/cache-prod-$(date +%Y%m%d).rdb \
  cache-prod \
  --region=us-central1
```

#### Import de RDB

```bash
gcloud redis instances import gs://meu-bucket/redis-backups/snapshot.rdb \
  cache-prod \
  --region=us-central1
```

## Cenário real

### Onde Memorystore encaixa nos seus projetos

**Portaria / TaxIA — uso atual:** você já roda Redis em container localmente. Migrar pra Memorystore na hora que for pra cloud é **trocar a string de conexão**.

**Casos concretos onde vale a pena nos seus projetos:**

- **Sessão de usuário do TaxIA** — JWT validation cacheada, dados de perfil (plano, créditos, configurações) para evitar query Postgres em cada request.
- **Rate limiting da API** — `INCR rate:user_42 ex 60` para limitar 100 req/min por usuário.
- **Cache de respostas do LLM** — se uma mesma pergunta vem 2x do mesmo contexto, retorna do cache (economiza tokens).
- **Estado efêmero de portaria** — quem está com qual sessão de monitoramento aberta, contador de tentativas de login, etc.

**Onde NÃO usar Memorystore nos seus projetos:**
- **Dados de transações** (Postgres / Cloud SQL).
- **Arquivos de imagem do WhatsApp** (Cloud Storage).
- **Histórico de conversa do TaxIA** (Postgres — precisa durar).

### Pattern "cache aside" para reduzir custo do Postgres

Esse é o pattern que mais economiza dinheiro em SaaS B2B:

```
Request → check Redis → hit  → retorna em sub-ms
                    └→ miss → query Postgres → escreve no Redis (TTL 5 min) → retorna

Resultado: 90%+ dos reads passam pelo Redis.
Postgres pode rodar em instância menor.
Latência p50 cai de 30 ms para 1 ms.
```

### Erro clássico no mundo real

**Tratar Memorystore como banco primário.** Guardar dado importante "só no Redis", sem persistir em banco durável. Funciona até o Redis ter um problema (failover, OOM, drift de timing) e você **perder dado real**. Mesmo com persistência (RDB/AOF) habilitada, Redis **não foi feito** para ser source of truth.

**Regra:** Memorystore sempre tem um banco "verdadeiro" por trás. Cache pode evaporar a qualquer momento — o app tem que sobreviver a isso.

## Armadilhas de prova

1. **Memorystore = cache in-memory, sub-millisecond, NÃO é storage primário.** Memorize.

2. **Pistas léxicas determinísticas:** "in-memory", "sub-millisecond", "caching", "session store", "leaderboard", "rate limiting", "lift-and-shift Redis" → Memorystore.

3. **Memorystore ≠ Firestore.** Firestore é NoSQL document **durável**. Memorystore é cache **efêmero**. Se a pergunta diz "real-time storage with sub-millisecond latency for session data" — **Memorystore**, não Firestore.

4. **Memorystore tem dois engines: Redis e Memcached.** Default = Redis. Memcached só se a pergunta enfatiza **simplicidade pura key-value** + **multi-thread**.

5. **HA = Standard tier, replicação entre 2 zonas, SLA 99,9%.** Basic tier não tem SLA formal.

6. **Capacidade máxima 300 GB, throughput 12 Gbps.** Decorar os números — podem aparecer em "qual o limite de Memorystore?".

7. **Memorystore só tem IP privado.** Não há conexão pela internet pública. Conexão de fora da VPC requer VPN/Interconnect ou Serverless VPC Access.

8. **Migração de Redis OSS é zero-código.** Compatível com Redis protocol — basta trocar string de conexão. RDB import/export para migrar dados.

9. **"Compatible with Redis protocol"** é a frase oficial. Se a pergunta diz "lift-and-shift Redis with zero code changes" — Memorystore.

10. **Memorystore NÃO substitui Pub/Sub.** Redis tem pub/sub, mas é leve, sem garantias de entrega. Para mensageria séria entre serviços, **Pub/Sub** dedicado.

11. **Memorystore NÃO substitui Cloud Tasks / Cloud Scheduler.** Filas Redis funcionam, mas sem durabilidade garantida.

12. **Cache miss não é erro.** Em qualquer pergunta sobre arquitetura, lembrar que app **precisa funcionar mesmo se Memorystore cair**. Sempre tem banco primário atrás.

13. **Memorystore tem persistência opcional (RDB/AOF) — mas ainda não é source of truth.** Persistência ajuda recovery, mas o filosofia continua "cache que pode evaporar".

14. **Read replicas (Standard tier) servem leitura.** Útil para distribuir carga de leitura, mas escrita continua única.

15. **Não suporta "Redis Cluster" puro.** Memorystore tem um único shard por instância (até 300 GB). Para sharding horizontal massivo, o serviço alternativo é **Memorystore for Redis Cluster** (produto separado, mais recente; o curso ainda não menciona).

16. **TLS é opcional mas recomendado.** `--transit-encryption-mode=SERVER_AUTHENTICATION` ativa.

17. **AUTH (password) é opcional mas recomendado.** Sem auth, qualquer um na VPC com IP do Redis conecta. `--enable-auth`.
