# 2.1 — Planning and configuring compute resources

> **Fonte:** Google Cloud — *Preparing for Your Associate Cloud Engineer Journey*, Module 2 Lesson 7 (Section 2)
> **DQs do workbook cobertas:** 01, 02, 03, 04
> **Considerations da prova:** escolher o compute certo para o workload (Compute Engine, GKE, Cloud Run, Cloud Run functions); usar Spot VMs e custom machine types quando fizer sentido.

---

## Conceitos

### O modelo mental: 2 famílias, 5 serviços

O Google divide as opções de compute em **duas grandes famílias**, e o critério para escolher entre elas é simples: **quanto você quer gerenciar versus quanto você quer pagar só pelo tempo de execução**.

| Família | Serviços | O que você paga | O que você gerencia |
|---|---|---|---|
| **Server-based (IaaS)** | Compute Engine, GKE | Infraestrutura provisionada (mesmo ociosa) | Nó/VM, OS (no caso de CE), capacidade |
| **Serverless (PaaS)** | App Engine, Cloud Run, Cloud Run functions | Tempo de execução real | Só o código (Google cuida do resto) |

**Por que importa:** Toda decisão de compute na prova começa por essa divisão. Se a pergunta enfatiza *"foco em código"*, *"velocidade de desenvolvimento"*, *"minimal overhead"* → família serverless. Se enfatiza *"controle do OS"*, *"customização"*, *"lift-and-shift"* → família IaaS.

---

### IaaS — Server-based

#### Compute Engine (CE)

**O que é:** Máquinas virtuais rodando nos datacenters do Google. É o equivalente direto ao seu servidor on-prem, só que na nuvem do Google.

**Quando usar:**
- Controle **total** do OS (kernel, drivers, dependências do sistema)
- Mudar versões de OS ou patches manualmente
- **Mover para a nuvem sem reescrever código** (lift-and-shift)
- Usar **imagens customizadas de VM**
- Qualquer workload que dependa de configuração específica de OS

**Casos típicos:**
- Apps legados projetados para rodar em servidor
- Software on-premises que você quer mover sem reescrever
- Workloads com dependências de OS muito específicas (ex.: Ubuntu altamente customizado)

**Comando-chave (preview):**
```bash
gcloud compute instances create my-vm \
  --machine-type=e2-medium \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud \
  --zone=us-central1-a
```

#### Google Kubernetes Engine (GKE)

**O que é:** Plataforma gerenciada para orquestrar containers usando Kubernetes (projeto open-source). Google cuida do **control plane**; você cuida dos workloads dentro do cluster.

**Quando usar:**
- Você **não tem dependências de OS** (containers abstraem isso)
- Precisa de **velocidade e operabilidade** maior que VMs
- Quer **controle total dos containers em produção** (orquestração, disponibilidade, escala)
- Quer um caminho híbrido: GKE on-prem + GKE no Google Cloud com a mesma stack

**Casos típicos:**
- Workloads **containerizados**
- Sistemas distribuídos **cloud-native**
- Apps **híbridos** (parte on-prem, parte cloud)
- Microsserviços com necessidade de orquestração avançada

**Por que GKE em vez de Cloud Run para containers?** Cloud Run abstrai a orquestração (você só fornece a imagem e ele decide tudo). GKE entrega controle: você define deployments, services, autoscaling policies, network policies, etc. **Trade-off**: mais poder, mais complexidade.

---

### PaaS / Serverless — você só paga o tempo de execução

#### App Engine

**O que é:** Plataforma "zero-ops" para deploy de código. Você manda o código, Google escolhe runtime/infra. Tem dois ambientes:
- **Standard:** sandbox totalmente abstraído (runtimes específicos, limites mais rígidos)
- **Flexible:** mais linguagens, mais runtimes, permite carregar dependências no sistema subjacente

**Quando usar:**
- Foco **puro em escrever código**
- Velocidade de desenvolvedor é prioridade
- Minimizar overhead operacional
- Não há requisito de containerização

**Casos típicos:** websites, apps, backends de jogos, IoT.

**Armadilha de prova:** App Engine **não dá controle sobre o OS**. Se a pergunta menciona "configurar dependências de SO" — App Engine está fora.

#### Cloud Run

**O que é:** Serverless para containers. Você empacota seu código em imagem container, sobe pro Cloud Run, ele executa sob demanda. Internamente roda em **Knative** (plataforma open-source baseada em Kubernetes).

**Quando usar:**
- Quer **escalonar até zero** quando não há tráfego (paga só o que usa)
- Precisa de **API endpoints** ou microsserviços
- Quer a **portabilidade de containers** + simplicidade de serverless
- Configuração mínima

**Casos típicos:** web frameworks, microsserviços, APIs.

**Por que prefirir Cloud Run a App Engine?** Containers são **portáveis** (rodam em qualquer lugar com runtime de container). App Engine te prende mais ao runtime do Google. Se a pergunta menciona **"portabilidade do código"** + **"foco em código"** → Cloud Run.

#### Cloud Run functions

> Nome antigo: **Cloud Functions**. Google renomeou para deixar clara a relação com Cloud Run.

**O que é:** Serverless **event-driven**. Você escreve uma função curta (não um app inteiro nem um container), o Google a executa quando um evento dispara.

**Quando usar:**
- Workloads **disparados por eventos** (upload em bucket, mensagem em Pub/Sub, commit em repo)
- Lógica de **conexão entre serviços** (cola entre sistemas)
- Configuração mínima, código curto

**Casos típicos:**
- Análise estatística disparada por upload
- Geração de thumbnail quando imagem chega no bucket
- Postar comentário no Slack quando há commit no GitHub

**Armadilha:** Cloud Run functions roda **código curto**, não containers. Se a pergunta enfatiza "portabilidade" ou "código containerizado" → não é functions; é Cloud Run.

### Tabela-resumo (essencial para a prova)

| Pista no enunciado | Resposta provável |
|---|---|
| Controle do OS / dependências do SO | **Compute Engine** |
| Lift-and-shift sem mudar código | **Compute Engine** |
| Controle total de containers, sem control plane | **GKE** |
| Microsserviços containerizados em produção | **GKE** |
| Foco em código + portabilidade (containers) | **Cloud Run** |
| Foco em código, sem container (deploy de fonte) | **App Engine** |
| Event-driven, código curto, "trigger" | **Cloud Run functions** |

---

### Spot VMs e custom machine types (citado no exam guide)

> O PDF não detalha esses tópicos, mas o exam guide cita. Resumo rápido:

- **Spot VM:** instância de VM com **preço até 91% menor**, mas o Google pode **encerrar a qualquer momento** se precisar da capacidade. Use para workloads tolerantes a interrupção: batch jobs, CI, fault-tolerant computing.
- **Custom machine type:** em vez de escolher um tipo pré-definido (e2-medium, n2-standard-4...), você define **exatamente quantas vCPUs e quanto de RAM** quer. Usa quando os tipos pré-definidos desperdiçam recursos para seu workload.

> Detalhes profundos virão na seção 3 (Deploying) quando o material cobrir.

## Questionários

### DQ 01 — Dependências específicas de OS
**Cenário:** Cymbal Superstore migra app de supply chain para Google Cloud e precisa configurar **dependências específicas de OS**.

**Resposta correta: D — Compute Engine (VMs)**

**Por quê:** Compute Engine é o **único** que dá controle total sobre escolha e configuração do OS.

**Por que as outras estão erradas:**
- **A (Cloud Run):** abstrai cluster e arquitetura — você não escolhe o OS subjacente
- **B (App Engine):** PaaS gerenciado pelo Google; você não toca no OS
- **C (GKE):** gerencia containers, não dá controle sobre o OS dos nós (e nem precisa, pois containers compartilham kernel)

**Armadilha:** "specific OS dependencies" é o trigger lexical para Compute Engine. Sempre.

### DQ 02 — Foco em código + desenvolvimento rápido + portabilidade
**Cenário:** Piloto de aplicação de PDV. Quer foco em código, desenvolvimento rápido e **código portável**.

**Resposta correta: B — Empacotar em imagem de container e publicar no Cloud Run**

**Por quê:** Cloud Run = serverless **com containers**. Containers garantem portabilidade; serverless garante velocidade e foco em código.

**Por que as outras estão erradas:**
- **A (SSH em VM CE):** não é foco em código — você gerencia VM
- **C (GKE com kubectl):** exige construir e gerenciar cluster — não é "foco em código rapidamente"
- **D (Cloud Run functions):** funções **não são containers** — perde portabilidade

**Armadilha:** "Portable" = container. Cloud Run functions é serverless mas perde nessa palavra-chave.

> Já registrado em [../../flashcards.md](../../flashcards.md) (Quiz 11/05).

### DQ 03 — Ubuntu customizado + menor tempo + mínimas mudanças de código
**Cenário:** App rodando em Ubuntu altamente customizado precisa ir pro Google Cloud com **mínimas mudanças de código** e **menor tempo possível**.

**Resposta correta: A — Compute Engine (lift-and-shift)**

**Por quê:** "Customized Ubuntu" + "minimal code changes" = movimento literal da VM atual para a nuvem. Zero esforço de reengenharia.

**Por que as outras estão erradas:**
- **B (App Engine):** exigiria adaptar o código ao modelo App Engine
- **C (Cloud Run):** exige containerizar (criar Dockerfile, etc.) — esforço extra
- **D (GKE):** exige containerizar **e** gerenciar cluster — esforço máximo

**Armadilha:** "Minimal code changes" + sistema customizado existente = Compute Engine. Sempre que houver OS customizado, GKE/Cloud Run estão fora porque exigem containerização.

> Já registrado em [../../flashcards.md](../../flashcards.md) (Quiz 11/05).

### DQ 04 — Microsserviços + controle total de container + sem gerenciar control plane
**Cenário:** Deploy de aplicação de microsserviços. Precisa de **controle total** de containers, confiabilidade e autoscaling, mas **não quer gerenciar o control plane**.

**Resposta correta: C — Google Kubernetes Engine (GKE)**

**Por quê:** GKE gerencia o control plane **pra você** (Google opera os nós master), mas te dá controle total sobre orquestração via Kubernetes API (deployments, services, autoscalers).

**Por que as outras estão erradas:**
- **A (Cloud Run):** abstrai orquestração — você não tem controle total sobre containers
- **B (App Engine):** não trabalha com containers da mesma forma — não é o caso
- **D (Compute Engine):** você teria que instalar e gerenciar Kubernetes (ou outro orquestrador) na mão

**Armadilha:** "Full control of containers" + "don't manage control plane" = GKE. As duas pistas juntas selam a resposta.

## Armadilhas de prova

1. **"Specific OS / OS dependencies / customized OS"** → sempre **Compute Engine**. Toda outra opção abstrai o OS de alguma forma.
2. **"Portable code"** → o gatilho lexical é **container**, e dentro de container o serverless é **Cloud Run**. Cloud Run functions perde nessa palavra-chave.
3. **"Lift-and-shift / minimal code changes"** → **Compute Engine**. Qualquer migração para serverless/container exige esforço de reengenharia.
4. **"Full control of containers + no control plane"** → **GKE**. Cloud Run abstrai demais; Compute Engine te obriga a instalar Kubernetes.
5. **"Event-driven / trigger / fired by"** → **Cloud Run functions**. Não é Cloud Run (que aceita HTTP requests, mas o destaque "event-driven curto" é functions).
6. **App Engine vs Cloud Run:** se a questão menciona container explicitamente → Cloud Run. Se menciona deploy direto de código sem mencionar container → App Engine. Cloud Run ganha em portabilidade.

---

## Cenário real

**Onde eu usaria isso no trabalho?**

- **Cloud Run** → seria minha escolha default pra qualquer microsserviço novo em Node.js (TaxIA-backend, backend-portaria). Containerizo, faço deploy, esqueço a infra.
- **GKE** → se eu tivesse uma plataforma com **muitos** microsserviços conversando entre si, com necessidade de service mesh, network policies finas, ou rodando em múltiplos clouds — GKE compensa a complexidade.
- **Compute Engine** → se herdasse um sistema legado (ex.: app Windows com COM, ou Ubuntu customizado com dependências de kernel) e precisasse mover pra cloud sem reescrever.
- **Cloud Run functions** → cola entre serviços: "quando chegar arquivo nesse bucket, processa e manda pro Pub/Sub". Não montaria um microsserviço inteiro pra isso.
