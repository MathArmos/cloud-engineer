## Conceitos

### Bloco "O que é Cloud Storage"

**Definição oficial:** serviço de **object storage** do Google Cloud com **armazenamento e recuperação mundial** de qualquer quantidade de dado, a qualquer hora.

**Cenários típicos mencionados no vídeo:**
- Servir **conteúdo de website** (HTML, CSS, JS, imagens estáticas).
- **Backup / disaster recovery / arquivamento**.
- **Distribuir objetos grandes** para usuários via download direto (vídeos, ISOs, datasets).

**Features-chave (memorize as 4):**

| Feature                       | O que significa na prática                                                   |
|-------------------------------|-------------------------------------------------------------------------------|
| **Scalable a exabytes**       | Não há teto prático de armazenamento — você não precisa "provisionar" tamanho |
| **Time-to-first-byte em ms**  | Latência baixa para começar a entregar o arquivo — útil para streaming e web  |
| **High availability em todas as classes** | Mesmo Archive responde em milissegundos (não horas, como AWS Glacier) |
| **API única para todas as classes** | Trocar de Standard pra Archive não exige refatorar o código do app    |

**Armadilha mental:** muita gente pensa em Cloud Storage como "uma pasta no servidor". **Não é.** Não há sistema de arquivos por baixo, não há `inode`, não há `seek`, não há lock. É um **store de objetos** acessado por URL.

#### "Mas eu vejo pastas no Console..."

Quando você usa `gs://meu-bucket/imagens/2024/foto.jpg`, parece haver uma pasta `imagens/2024/`. **Não há.** O que existe é:

- Um **objeto** com a chave literal `imagens/2024/foto.jpg`.
- O Console **simula** pastas agrupando objetos com prefixo comum.

Consequência prática:
- Você **não pode listar** todos os arquivos como faria com `find` ou índice de filesystem nativo.
- Você acessa um objeto pela URL exata dele (`https://storage.googleapis.com/meu-bucket/imagens/2024/foto.jpg`) ou faz **list** com prefixo (`list --prefix=imagens/2024/`).
- Mover um objeto entre "pastas" é, na verdade, **renomear a chave** (operação que envolve copy + delete por baixo).

**Analogia:** Cloud Storage é como um armário gigante onde cada item tem uma etiqueta única. As "pastas" são só substrings da etiqueta — não há gavetas físicas.

### Bloco "Buckets e objetos — anatomia"

#### Bucket

- **Container** que guarda objetos.
- **Nome globalmente único** — não só no seu projeto, no GCP **inteiro**. Se alguém pegou `meu-bucket`, você não pode mais. Por isso convenções com prefixo de empresa/projeto (`acme-prod-logs-2024`).
- **Não pode ser aninhado.** Não existe bucket dentro de bucket. A organização hierárquica acontece **dentro** do bucket via prefixos de objeto.
- Possui **storage class default** e **location type** definidos na criação.

#### Objeto

- Os "arquivos" que vivem dentro do bucket.
- **Herda a storage class do bucket** por default — você pode sobrescrever **por objeto**.
- Pode ser qualquer tipo de arquivo (texto, doc, vídeo, binário, blob arbitrário).
- **Sem tamanho mínimo.** Tamanho máximo por objeto: **5 TiB** (não falado no vídeo, mas é o limite oficial — bom de ter na cabeça).
- Escala **enquanto a quota do projeto permitir** — não há limite por bucket.

#### Diagrama hierárquico

```
Project
  └── Bucket (nome único globalmente, location + default storage class)
       ├── Object: "logs/2024/01/app.log"  (storage class = default ou override)
       ├── Object: "logs/2024/02/app.log"
       └── Object: "imagens/perfil.png"
```

**Por que não há sub-bucket:** o design força você a usar **prefixos** (logs/, imagens/), que escalam infinitamente sem ter que gerenciar hierarquia. Toda lookup é O(1) por chave.

### Bloco "Storage classes — as 4 opções"

Todas as 4 classes têm:
- **Disponibilidade imediata** (milissegundos para começar a baixar).
- **11 nines de durability** (`99.999999999%`).
- **Mesma API**.

O que muda entre elas:
1. **Custo de armazenamento por GB/mês** (quanto mais "fria", mais barato).
2. **Custo de retrieval / operação** (quanto mais "fria", mais caro pra acessar).
3. **Duração mínima de armazenamento** (cobra como se o objeto tivesse ficado X dias mesmo se você deletar antes).
4. **Availability nominal** (todas têm alta, mas as mais frias têm "slightly lower").

#### Tabela comparativa

| Classe       | Acesso esperado     | Min. storage duration | Custo storage | Custo retrieval | Caso típico                                    |
|--------------|---------------------|------------------------|---------------|------------------|------------------------------------------------|
| **Standard** | Frequente (hot)     | Nenhuma                | Mais alto     | **Zero**         | Website, streaming, dado em uso ativo          |
| **Nearline** | ~1x/mês             | 30 dias                | Médio-baixo   | Taxa baixa       | Backup, multimídia long-tail, archive moderado |
| **Coldline** | ~1x/trimestre       | 90 dias                | Baixo         | Taxa média       | DR, dado regulatório frio                      |
| **Archive**  | <1x/ano             | 365 dias               | Mais baixo    | Taxa alta        | Arquivo longo prazo, compliance, DR remoto     |

**Detalhe importante mencionado no vídeo (Standard):**
- **Em region:** ideal pra co-locar com GKE/GCE no mesmo lugar — maximiza performance de computação intensiva e reduz custo de network.
- **Em dual-region:** performance otimizada se a compute estiver numa das duas regiões + melhor availability por redundância geográfica.
- **Em multi-region:** ideal pra **conteúdo servido ao mundo** (web global, streaming, mobile, games).

**Detalhe importante para Nearline/Coldline/Archive:**
> *"slightly lower availability, [X-day] minimum storage duration, and costs for data access are acceptable trade-offs for lowered at-rest storage costs."*

Traduzindo: você troca **3 coisas** por preço mais baixo de armazenamento:
1. Pouca availability a menos.
2. Duração mínima cobrada (deletar antes não livra do custo).
3. Custo de acesso quando lê o dado.

#### Diferencial Archive vs concorrência

> *"Unlike the so-to-speak 'coldest' storage services offered by other Cloud providers, your data is available within milliseconds, not hours or days."*

Esta é uma **vantagem importante** do GCP que pode cair na prova como armadilha:

- **AWS Glacier Deep Archive:** retrieval pode levar **horas** (Standard) a **12 horas** (Bulk).
- **Azure Archive:** retrieval leva **horas**.
- **GCP Archive:** **milissegundos**.

Trade-off no GCP: o custo de **acessar** é mais alto que nas classes quentes — então use Archive pra dado que você quase nunca toca.

**Armadilha de prova:** se a questão sugerir "preciso esperar horas para recuperar dado do Archive", **está errada**. No GCP, Archive é "frio em preço, quente em latência".

### Bloco "Location types — onde o bucket reside"

Toda classe combina com um dos 3 tipos de localização:

#### Region (regional)

- **Uma região específica** (ex.: `us-central1`, `europe-west4`, `southamerica-east1`).
- Dado fica em **uma única região** (replicado em zonas dentro dela, mas não fora).
- **Menor latência** se a sua compute estiver na mesma região.
- **Mais barato** que dual/multi-region.
- **Não tem geo-redundância.** Se a região inteira cair (raro, mas possível), o dado fica indisponível.

#### Dual-region

- **Par específico** de regiões (ex.: `nam4` = Iowa + South Carolina; `eur4` = Finland + Netherlands).
- Dado é **replicado entre as duas regiões** — geo-redundância.
- **Latência baixa** quando você acessa de uma das duas regiões.
- **Mais caro** que regional.
- **Resiliência:** se uma região cair, a outra atende.

#### Multi-region

- **Geografia ampla** (`us`, `eu`, `asia`) — Google escolhe internamente em que regiões dentro daquela geografia o dado vive.
- Dado é distribuído em **várias regiões** dentro daquela geografia.
- **Latência consistente** para usuários **dentro** daquela geografia.
- **Mais caro** que regional, geralmente equivalente a dual-region.
- Ideal para **conteúdo servido globalmente** dentro de uma geografia.

#### Geo-redundância

- **Multi-region** e **dual-region** = **geo-redundant** (cópia em locais geograficamente separados).
- **Region** = **NÃO geo-redundant** (replicação só dentro da região).

**Regra prática:**
- Compute na mesma região → **regional**.
- Quer resiliência e latência baixa em duas regiões → **dual-region**.
- Servir o mundo (web, vídeo) → **multi-region**.

### Bloco "Durability vs Availability — a confusão que cai na prova"

Esta distinção foi feita no vídeo com analogia explícita — vai cair.

#### Durability — "Não vou perder o dado"

- **11 nines (`99.999999999%`)** em **todas** as storage classes.
- Significa: a probabilidade de perder um objeto em um ano é absurdamente baixa (~1 em 100 bilhões).
- Garantido pela replicação interna do Google (várias cópias, em locais diferentes).
- **NÃO depende** da storage class. Standard e Archive são igualmente duráveis.

#### Availability — "Consigo acessar o dado agora?"

- **Varia** entre storage classes e location types.
- Standard multi-region tem availability mais alta que Coldline regional, por exemplo.
- "Slightly lower availability" das classes frias = menos SLA de uptime, não menos durabilidade.

#### A analogia do banco (vídeo)

> *"Your money is in there, it's 11 nines durable. But when the bank is closed we don't have access to it, which is the availability."*

- **Durability** = seu dinheiro está no banco, ninguém vai perder.
- **Availability** = o banco está aberto agora pra você sacar?

**Armadilha clássica de prova:**
- "Preciso garantir que meus dados não serão perdidos" → **durability** (todas as classes empatam, escolha por outro critério).
- "Preciso acessar meus dados a qualquer momento sem interrupção" → **availability** (escolha Standard multi-region).

### Bloco "Como acessar — APIs e CLI"

Três caminhos:

1. **`gcloud storage`** (CLI moderna do GCP) — substitui o velho `gsutil` em quase tudo. Mais rápido, melhor UX.
2. **JSON API** — REST clássico, para integração programática.
3. **XML API** — compatível com S3, útil para portabilidade.

#### Exemplos práticos

```bash
# Criar bucket regional Standard
gcloud storage buckets create gs://acme-prod-logs \
  --location=us-central1 \
  --default-storage-class=STANDARD \
  --uniform-bucket-level-access

# Criar bucket multi-region Standard
gcloud storage buckets create gs://acme-static-website \
  --location=US \
  --default-storage-class=STANDARD

# Upload de arquivo
gcloud storage cp ./relatorio.pdf gs://acme-prod-logs/2026/05/

# Download
gcloud storage cp gs://acme-prod-logs/2026/05/relatorio.pdf ./

# Listar com prefixo
gcloud storage ls gs://acme-prod-logs/2026/

# Mudar storage class de um objeto específico
gcloud storage objects update gs://acme-prod-logs/old-data.tar.gz \
  --storage-class=ARCHIVE
```

**Detalhe sobre `--uniform-bucket-level-access`:** força todo controle de acesso via IAM (sem ACLs por objeto). Recomendado em quase todos os casos novos — simplifica auditoria.

### Bloco "O que dá pra mudar (e o que NÃO dá)"

O vídeo enfatiza um ponto importante: nem tudo é reversível depois de criar o bucket.

| Atributo                                       | Mudável?                                     |
|------------------------------------------------|----------------------------------------------|
| **Default storage class do bucket**            | ✅ Sim                                       |
| **Storage class de um objeto individual**      | ✅ Sim (sem mover de bucket, sem mudar URL)  |
| **Nome do bucket**                             | ❌ Não (precisa criar novo e migrar)         |
| **Location type** (regional ↔ dual ↔ multi)    | ❌ **Não** (precisa criar novo bucket)       |
| **Localização específica** (us-central1 → europe-west4) | ❌ Não                              |

**Armadilha clássica:** uma DQ pode propor "vamos converter o bucket regional em multi-region". **Não dá.** Tem que criar bucket novo na nova location e migrar (`gcloud storage cp` recursivo, ou Storage Transfer Service).

#### Mudar storage class de objeto sem mover

```bash
# Objeto fica no mesmo bucket, mesma URL, mas agora é Archive
gcloud storage objects update gs://acme-prod-logs/log-2022.tar.gz \
  --storage-class=ARCHIVE
```

**Útil quando:** você quer manter o histórico mas pagar menos. Em vez de criar bucket Archive separado, muda a classe in-place.

**Ainda mais útil:** **Object Lifecycle Management** (mencionado no vídeo como "mais depois") automatiza isso. Define regras como "se objeto > 30 dias → mover para Nearline". O Cloud Storage executa em background sem precisar de cron job.

Esboço de lifecycle policy:
```json
{
  "lifecycle": {
    "rule": [
      { "action": {"type": "SetStorageClass", "storageClass": "NEARLINE"},
        "condition": {"age": 30} },
      { "action": {"type": "SetStorageClass", "storageClass": "COLDLINE"},
        "condition": {"age": 90} },
      { "action": {"type": "SetStorageClass", "storageClass": "ARCHIVE"},
        "condition": {"age": 365} },
      { "action": {"type": "Delete"},
        "condition": {"age": 2555} }   // 7 anos
    ]
  }
}
```

### Bloco "Access control — 4 camadas, do grosso ao fino"

O vídeo lista **4 mecanismos** que você pode combinar para controlar quem acessa o quê:

#### 1. IAM (recomendado para a maioria dos casos)

- **Granularidade:** projeto → bucket → objeto, com **herança** (role no projeto vale pro bucket; role no bucket vale pros objetos).
- **Quem usa:** **usuários e service accounts**.
- **Roles típicas:**
  - `roles/storage.objectViewer` — só leitura
  - `roles/storage.objectCreator` — só upload
  - `roles/storage.objectAdmin` — full em objetos
  - `roles/storage.admin` — full em bucket + objetos
- Suficiente para a maioria dos cenários.

#### 2. ACLs (mais granular, legacy)

- **Granularidade:** **por objeto individual** ou por bucket.
- **Limite:** máximo **100 entradas** por bucket ou objeto.
- Cada entrada tem:
  - **Scope:** quem (usuário, grupo, allUsers, allAuthenticatedUsers...).
  - **Permission:** o quê (READER, WRITER, OWNER).
- **Identificadores especiais:**
  - `allUsers` = **qualquer um na internet** (com ou sem conta Google) → cuidado, isso torna o objeto público.
  - `allAuthenticatedUsers` = qualquer um com conta Google autenticado.

**Quando usar ACLs em vez de IAM:**
- Você precisa dar permissão a **um objeto específico** dentro de um bucket sem afetar o resto.
- Caso clássico: bucket privado com **um único arquivo público** (logo, favicon).

**Pegadinha:** se você habilitou `uniform-bucket-level-access`, **ACLs ficam desabilitadas**. Tudo é IAM. Em buckets novos, isso é o default recomendado.

#### 3. Signed URLs (acesso temporário, sem conta Google)

- URL contendo **chave criptográfica** que dá **acesso temporizado** a um objeto.
- Útil quando o cliente final **não tem (e não deveria ter) conta Google**.
- Assinada usando **private key de uma service account**.
- Cloud Storage verifica a assinatura → libera o acesso → confiança delegada ao holder da URL.

**Casos típicos:**
- Upload de arquivo de um usuário sem login Google ("envie sua selfie pelo nosso site").
- Compartilhar download de relatório por X minutos.
- Pré-assinar URLs em massa para uma operação batch.

**Risco:** depois que a URL sai do seu controle, **qualquer um com ela acessa**. Por isso:
- **Sempre defina expiração curta** (minutos, no máximo horas).
- **Não logue a URL** em sistemas terceiros.
- **Para upload, use signed policy documents** (item 4) pra restringir o que pode subir.

**Exemplo conceitual:**
```bash
gcloud storage sign-url gs://acme-uploads/foto.jpg \
  --duration=15m \
  --private-key-file=sa-key.json
# Devolve: https://storage.googleapis.com/...?X-Goog-Signature=...
```

#### 4. Signed policy documents

- Refinam o que **pode ser uploaded** com uma signed URL.
- Define restrições no payload do upload: tamanho máximo, content-type permitido, prefixo de chave obrigatório etc.
- Caso típico: formulário web onde usuário sobe foto **direto** pro bucket — você quer garantir que ele não suba `.exe` de 5 GB no lugar do `.jpg` de até 5 MB.

#### Resumo decisório

| Cenário                                                    | Mecanismo                                  |
|------------------------------------------------------------|--------------------------------------------|
| Usuários internos / service accounts                       | **IAM**                                    |
| Um objeto público num bucket privado                       | **ACL** (ou bucket público + objeto público com IAM, se preferir) |
| Compartilhar download com cliente externo (sem conta Google) | **Signed URL** (expiração curta)         |
| Upload direto do browser do cliente                         | **Signed URL + Signed policy document**   |
| Bucket público de conteúdo estático (CDN)                  | **IAM** com `allUsers:objectViewer`        |

### Bloco "Signed URLs — fluxo completo"

Detalhando o fluxo, porque é o mecanismo mais "novo" pra quem vem de outros stacks:

```
+-----------+      1. pede acesso        +-------------+
| Cliente   | --------------------------> | Seu backend |
| (browser) |                             | (signa URL) |
+-----------+                             +------+------+
      ^                                          |
      |                                          | 2. usa private key
      |                                          |    da Service Account
      |                                          v
      |                                  +-----------------+
      |                                  |   Cloud         |
      | <-- 3. devolve signed URL ------ |   Storage SDK   |
      |                                  +-----------------+
      |
      | 4. acessa o objeto diretamente
      v
+----------------+
| Cloud Storage  |
| valida assin.  |
| e devolve dado |
+----------------+
```

**Pontos-chave do vídeo:**
1. Você quer **autenticação por token**, não por conta Google.
2. A URL "vale" tempo limitado (você define na hora de assinar).
3. A URL é assinada **com a private key da service account** — então a service account precisa ter as permissões mesmo.
4. Após emitir, **fora do seu controle** → defina expiração razoável.

### Bloco "Encryption — chaves Google, CMEK e CSEK"

Toda objeto em Cloud Storage é **encriptado em repouso** por default — você não precisa fazer nada. A pergunta é **quem controla a chave**:

| Modelo            | Quem gera a chave | Onde a chave vive       | Quando usar                                              |
|-------------------|-------------------|--------------------------|----------------------------------------------------------|
| **Google-managed** (default) | Google            | Google (transparente)   | A maioria dos casos — você não tem requisito regulatório forte |
| **CMEK** (Customer-Managed Encryption Keys) | Você, via **Cloud KMS** | Cloud KMS (Google ainda processa) | Você quer **controlar rotação e revogação** sem operar o material criptográfico |
| **CSEK** (Customer-Supplied Encryption Keys) | Você, fora do GCP | **Você mantém** — manda em cada request | Compliance estrito que exige que o Google **nunca** veja a chave persistida |

**CSEK — pontos críticos:**
- Você envia a chave em **toda** request (header). Google usa, encripta, e **descarta** da memória.
- Se você **perder a chave, o dado fica inacessível para sempre** — Google não tem cópia.
- Trade-off: máximo controle, mas operacionalmente arriscado.

**CMEK — pontos críticos:**
- A chave vive em **Cloud KMS** (também GCP, mas isolado).
- Você controla **quem** pode usar a chave (IAM no key ring), **quando rotar**, e pode **revogar**.
- Revogar a chave torna o dado **temporariamente inacessível** mesmo pro Google — útil para "data destruction" lógica.

**Pegadinha de prova:** se a questão mencionar "**we manage our own keys outside of GCP**" → **CSEK**. Se mencionar "**we want to control key rotation and revocation**" → **CMEK**. Se não falar nada → o default é Google-managed.

> Este tópico foi tocado **anteriormente no curso** quando se falou de Persistent Disks. A teoria é a mesma — muda só o produto que consome a chave.

### Bloco "Imutabilidade de objetos e por que isso importa"

**Definição-chave do vídeo:**
> *"In Cloud Storage, objects are immutable, which means that an uploaded object cannot change throughout its storage lifetime."*

Em linguagem direta: você **não consegue editar** um objeto existente. Toda "edição" é:

1. Deletar o objeto atual.
2. Subir um novo objeto com o mesmo nome.

Essa imutabilidade é a base que faz os próximos blocos funcionarem (Versioning, Soft Delete, Retention Lock). Sem imutabilidade, não haveria "versão anterior" para guardar — haveria modificações in-place que se sobrescrevem.

**Analogia:** Cloud Storage é como uma estante de livros onde cada livro é encadernado. Você não escreve mais páginas num livro existente — você imprime um livro novo com as alterações e tira o antigo da estante. As features de versioning/soft delete são "guardar os livros antigos no porão por X tempo antes de queimar".

### Bloco "Object Versioning"

**O que é:** ao habilitar, **cada overwrite ou delete** de um objeto cria uma **versão arquivada** (mantém o mesmo nome, mas ganha um **generation number** único — `g1`, `g2`, ...).

**Por que existe:** proteção contra **delete/overwrite acidental ou malicioso**. Permite "voltar atrás".

**Operações suportadas:**
- Listar versões arquivadas de um objeto.
- Restaurar a live version a um estado antigo.
- Deletar permanentemente uma versão arquivada (caso queira liberar espaço de versões que você sabe que não precisa).

**Liga/desliga a qualquer momento:**
- **Ligar** → começa a acumular versões a cada mudança.
- **Desligar** → **mantém o que já existe** (não apaga histórico), mas para de gerar novas versões.

#### Pontos críticos (vão cair na prova)

1. **Cobrança:** *"You are charged for the versions as if they were multiple files."* Cada versão paga storage como se fosse arquivo independente. Se você tem 100 MB com 10 versões, paga 1 GB.
2. **Generation number** identifica unicamente cada versão (mesmo nome, generation diferente).
3. **Google recomenda Soft Delete em vez de Versioning** para a maioria dos casos. Versioning continua sendo válido (e necessário em alguns cenários), mas Soft Delete é o default melhor para "proteger contra perda".

#### Quando usar Versioning ainda hoje

- Você precisa de **histórico longo** (anos) acessível por chamadas programáticas.
- Você quer expor "tenha sempre as 5 últimas versões deste arquivo" como funcionalidade.
- Você quer combinar com Lifecycle ("mantenha só as 3 versões mais recentes").

Para *somente proteger contra delete acidental*, use Soft Delete (próximo bloco).

### Bloco "Soft Delete — proteção default contra apagamento"

**O que é:** proteção **bucket-level** que **preserva todos os objetos deletados** (incluindo os "sobrescritos", já que sobrescrever é deletar + criar) por um **período configurável**.

**Pontos-chave do vídeo:**
- Buckets novos vêm com Soft Delete **HABILITADO POR DEFAULT**, com retenção de **7 dias**.
- Retenção configurável: **até 90 dias**, ou **0** para desabilitar.
- Durante o período, dá pra **restaurar** o objeto deletado.
- Depois do período, Cloud Storage **permanentemente apaga**.

#### Por que Google recomenda Soft Delete em vez de Versioning

| Critério                  | Object Versioning                              | Soft Delete                                |
|---------------------------|------------------------------------------------|--------------------------------------------|
| Habilitado por default?   | ❌ Não                                         | ✅ Sim, 7 dias                              |
| Custo                     | Paga por cada versão como arquivo separado     | Mesmo modelo (paga retained), mas escopo limitado pela retenção |
| Proteção contra delete    | Sim, mas precisa configurar                    | Sim, automático                            |
| Proteção contra overwrite | Sim                                            | Sim                                        |
| Restauração               | Manual (você escolhe a generation)             | Manual durante o período                   |
| Foco                      | Histórico programático                         | "Lixeira" de segurança                     |

> **Regra prática:** se você só quer proteção contra "alguém deletou sem querer", deixe **Soft Delete habilitado** com retenção adequada. Só ative Versioning se precisar **acessar versões antigas como funcionalidade**.

**Exemplo:**
```bash
# Ver configuração atual
gcloud storage buckets describe gs://meu-bucket --format="value(softDeletePolicy)"

# Aumentar retenção pra 30 dias
gcloud storage buckets update gs://meu-bucket --soft-delete-duration=30d

# Desabilitar
gcloud storage buckets update gs://meu-bucket --soft-delete-duration=0
```

### Bloco "Object Lifecycle Management — automação de custo"

**O que é:** conjunto de **regras** atribuídas ao bucket que o Cloud Storage executa automaticamente quando objetos batem certos critérios.

**Por que existe:** sem isso, gerenciar Standard → Nearline → Coldline → Archive ao longo do tempo viraria um cron job manual. Lifecycle terceiriza isso pro Cloud Storage.

#### Use cases canônicos do vídeo

1. **Downgrade de classe:** "objetos com mais de 365 dias → Coldline".
2. **TTL absoluto:** "deletar objetos criados antes de 2017-01-01".
3. **Manter só N versões:** "com Versioning ON, manter as 3 versões mais recentes" (útil pra controlar o custo de Versioning).

#### Anatomia de uma regra

Cada regra é um par **action + condition**:

| Action                          | O que faz                                          |
|---------------------------------|----------------------------------------------------|
| `SetStorageClass`               | Move o objeto pra outra classe                     |
| `Delete`                        | Apaga o objeto                                     |
| `AbortIncompleteMultipartUpload`| Cancela uploads parciais antigos (limpa lixo)      |

| Condition                       | O que verifica                                     |
|---------------------------------|----------------------------------------------------|
| `age`                           | Idade do objeto em dias                            |
| `createdBefore`                 | Criado antes de data específica                    |
| `numNewerVersions`              | Quantas versões mais novas existem (Versioning)    |
| `matchesStorageClass`           | Classe atual do objeto                             |
| `isLive`                        | Se é a versão live (true) ou arquivada (false)     |

#### Caveats que vão cair na prova

> *"Object inspection occurs in asynchronous batches, so rules may not be applied immediately."*

1. **Execução assíncrona em batches.** Não conte com "vai mover na hora". Cloud Storage roda inspeções periódicas.
2. **Atualizações de config levam até 24h pra entrar em efeito.** Durante esse intervalo, a **regra antiga ainda pode executar**.

Esse segundo ponto é uma **armadilha clássica**: você muda a política, espera "agora não vai mais mover", e mesmo assim por mais 24h o sistema pode estar movendo objetos pela regra anterior. Em código que depende de timing, isso é fonte de bug.

#### Exemplo completo de lifecycle policy

```json
{
  "lifecycle": {
    "rule": [
      {
        "action": {"type": "SetStorageClass", "storageClass": "NEARLINE"},
        "condition": {"age": 30, "matchesStorageClass": ["STANDARD"]}
      },
      {
        "action": {"type": "SetStorageClass", "storageClass": "COLDLINE"},
        "condition": {"age": 90, "matchesStorageClass": ["NEARLINE"]}
      },
      {
        "action": {"type": "SetStorageClass", "storageClass": "ARCHIVE"},
        "condition": {"age": 365}
      },
      {
        "action": {"type": "Delete"},
        "condition": {"age": 2555, "isLive": false}   // 7 anos, versões arquivadas
      },
      {
        "action": {"type": "Delete"},
        "condition": {"numNewerVersions": 3}   // manter só 3 versões
      }
    ]
  }
}
```

```bash
gcloud storage buckets update gs://meu-bucket --lifecycle-file=lifecycle.json
```

### Bloco "Object Retention Lock — imutabilidade para compliance"

**O que é:** configuração que **impede que um objeto seja deletado ou modificado** antes de uma data específica. Pode ser **locked permanentemente** — depois disso, **nem o owner consegue reduzir** o tempo de retenção.

**Por que existe:** atender requisitos regulatórios que exigem **WORM** (Write Once Read Many):
- **FINRA** (Financial Industry Regulatory Authority — EUA)
- **SEC** (Securities and Exchange Commission)
- **CFTC** (Commodity Futures Trading Commission)
- Backups imutáveis (proteção contra ransomware)

**Como funciona:**
1. Habilita o feature no bucket.
2. Para cada objeto (ou bucket inteiro), define um **retention period**.
3. **Opcionalmente** trava (`locked`) — depois de travado, não dá pra reduzir mais.

**Diferença vital pra Soft Delete e Versioning:**

| Feature           | "Apago, mas guardo cópia" | "Não posso apagar, ponto"     |
|-------------------|----------------------------|-------------------------------|
| Soft Delete       | ✅                          | ❌                            |
| Object Versioning | ✅                          | ❌                            |
| Retention Lock    | ❌                          | ✅ (e locked = nem com força) |

**Pegadinha de prova:** se a questão fala em **"regulatory compliance"**, **"FINRA / SEC"**, **"WORM"**, **"imutable backup against ransomware"** → **Retention Lock**. Soft Delete e Versioning **não** atendem isso (porque o owner ainda consegue apagar).

### Bloco "Autoclass — gerenciamento automático de classes (deep dive)"

> Conteúdo expandido após o sub-vídeo de decision tree do M3, que retomou Autoclass em detalhe.

**O que é:** se habilitado no bucket, **Cloud Storage decide automaticamente** a storage class de cada objeto baseado no **padrão de acesso real** observado para aquele objeto.

**Por quê existe:**
- Você nem sempre sabe o padrão de acesso de cada objeto.
- Lifecycle policy estática (`age > 30 → Nearline`) é uma heurística por idade, não por uso — pode ser certa pra alguns objetos, errada pra outros.
- Autoclass observa **cada objeto individualmente**: se for acessado, sobe a classe; se ficar parado, desce.

#### Como Autoclass se comporta (detalhes do vídeo)

1. **Todo objeto entra como Standard.**
   > *"Even if a different storage class is specified in the request, all objects added to the bucket begin in Standard storage."*

   Isso é uma **pegadinha forte de prova**. Mesmo que seu código faça upload pedindo Archive, **o objeto começa em Standard**. Autoclass ignora a classe que você pediu — ele decide.

2. **Não acessado → desce para classes mais frias** automaticamente.

3. **Acessado → volta pra Standard** (se já não estiver).
   > *"When object data is read, the object transitions to Standard storage if it's not already stored in Standard storage."*

4. **Otimiza acessos futuros.** Se o objeto virar hot de novo, ele já estará em Standard na próxima leitura — sem taxa de retrieval.

#### O que torna Autoclass economicamente viável

O vídeo lista **três isenções de cobrança** específicas para buckets com Autoclass ligado:

| Cobrança que **NÃO** existe em Autoclass    | O que isso significa                                                       |
|---------------------------------------------|----------------------------------------------------------------------------|
| **Sem early deletion charges**              | Mesmo se Autoclass moveu o objeto pra Archive e ele for deletado em 30 dias, **não paga** os ~334 dias restantes |
| **Sem retrieval charges**                   | Ler um objeto que Autoclass classificou como Nearline/Coldline/Archive não cobra a taxa de leitura típica daquela classe |
| **Sem charges por storage class transitions** | A movimentação entre classes que o Autoclass faz é **grátis**             |

> **Por que isso é decisivo:** sem essas isenções, Autoclass seria um tiro no pé — o sistema moveria objeto pra Coldline, alguém leria, e você pagaria retrieval. Com as isenções, o pior que pode acontecer é "ficou em Standard quando podia estar em Coldline" — você paga mais storage por um período, mas nunca paga penalidades de movimentação errada.

#### Trade-off real

- **Existe uma taxa de gerenciamento por objeto** (pequena), cobrada por Autoclass administrar a classificação.
- Em troca, você ganha **proteção contra todas as penalidades** das classes frias.

#### Quando faz sentido

- Workloads com **padrão de acesso imprevisível** (ex.: arquivos de usuário, conteúdo gerado por usuário, dados de pesquisa).
- Você não quer gerenciar lifecycle rules manualmente.
- Bucket grande onde alguns objetos são hot, outros não, e você não sabe quais.

#### Quando NÃO faz sentido

- Padrão de acesso **conhecido e estável** → lifecycle rules manuais são mais baratas (sem taxa de gerenciamento).
- Volume muito pequeno → a taxa de gerenciamento por objeto fica desproporcional.
- Você precisa de classe específica desde a criação (Autoclass força Standard inicial — pode contrariar requisitos de compliance/custo).

#### Como ativar

```bash
# Habilitar Autoclass em bucket existente
gcloud storage buckets update gs://meu-bucket --autoclass

# Criar bucket já com Autoclass
gcloud storage buckets create gs://novo-bucket \
  --location=us-central1 \
  --autoclass
```

### Bloco "Object change notifications via Pub/Sub"

**O que é:** Cloud Storage pode **publicar mensagens em um tópico Pub/Sub** sempre que um evento ocorre no bucket.

**Eventos:**
- `OBJECT_FINALIZE` (objeto criado/sobrescrito)
- `OBJECT_METADATA_UPDATE` (metadata mudou)
- `OBJECT_DELETE`
- `OBJECT_ARCHIVE` (versioning: live virou arquivada)

**Casos de uso:**
- Upload → trigger de **Cloud Run / Cloud Functions** para processar (resize de imagem, OCR de PDF, scan antivírus).
- Audit: cada delete vira evento Pub/Sub → grava em BigQuery.
- Replicação cross-region custom (se Storage Transfer Service não basta).

**Exemplo:**
```bash
# Cria tópico
gcloud pubsub topics create bucket-events

# Configura notificações
gcloud storage buckets notifications create gs://meu-bucket \
  --topic=bucket-events \
  --event-types=OBJECT_FINALIZE,OBJECT_DELETE
```

**Pegadinha sutil:** isso é diferente de **Pub/Sub trigger pra Cloud Run** (que pode ser configurado direto pelo serviço Cloud Run). Aqui você cria a **fonte** (bucket notifications); como o destino consome é outra história.

### Bloco "Directory synchronization"

**O que é:** sincronização entre um diretório em VM e um bucket — você ganha "uma pasta local que reflete o bucket".

**Como funciona na prática:**
- `gcloud storage rsync` faz cópia incremental num único comando.
- Cloud Storage FUSE (gcsfuse) **monta** um bucket como filesystem (com várias limitações, já que não é um filesystem de verdade).

**Casos de uso:**
- Backup periódico de diretórios em VMs.
- "Espalhar" configs/artefatos em VMs novas a partir de um bucket central.
- Pipelines de build que publicam artefatos.

**Limitação importante:** gcsfuse **não tem semântica POSIX completa** (sem locks, sem renames atômicos, performance variável). Para apps que precisam de filesystem real compartilhado → **Filestore**, não Cloud Storage.

### Bloco "Transferência em massa — 3 serviços oficiais"

Quando você tem dezenas de TB a PB pra mover, "subir pelo Console" não funciona. O Google oferece **três caminhos**:

| Serviço                         | Natureza        | Volume típico    | Quando usar                                                          |
|---------------------------------|-----------------|------------------|----------------------------------------------------------------------|
| **Transfer Appliance**          | Hardware físico | Centenas de TB até **1 PB** | Sua rede não comporta o upload online. Google envia um appliance, você copia local, devolve o appliance.    |
| **Storage Transfer Service**    | Online          | TB a PB          | Dado já está acessível por rede: outro bucket GCS, **S3 da AWS**, ou HTTP/HTTPS público |
| **Offline Media Import**        | Terceiro físico | Variável          | Você tem HDs / tapes / USB. Manda pra um parceiro 3rd-party autorizado, eles fazem upload |

#### Detalhes que aparecem em prova

**Transfer Appliance:**
- Capacidade até **1 PB** (números do vídeo).
- Indicado quando o ETA de upload via rede passaria semanas/meses.
- Cifrado em trânsito (você não vai mandar um HD desbloqueado pelo correio).

**Storage Transfer Service (STS):**
- **Dado-fonte deve estar online.** As fontes suportadas:
  - Outro bucket Cloud Storage (cross-project, cross-region).
  - **Amazon S3** — caso clássico de migração de cloud.
  - **HTTP/HTTPS** — URLs públicas (listadas em manifesto).
  - Mais tarde: Azure Blob, S3-compatible, on-prem agents.
- Scheduling: pode ser one-shot ou recorrente.
- Filtragem: por prefixo, por idade, por tipo.

**Offline Media Import:**
- Você não usa o Google direto — usa um **provider 3rd party** autorizado.
- Útil quando o volume é grande mas não a ponto de Appliance, ou quando você já tem mídia física empacotada.

**Decisão de prova (memorize):**
- "Migrar AWS S3 → GCS" → **Storage Transfer Service**.
- "Migrar 500 TB on-prem sem rede pra subir" → **Transfer Appliance**.
- "Eu tenho HDs físicos com dados" → **Offline Media Import**.

### Bloco "Strong consistency — global e em todas as operações"

**O que o vídeo afirma (palavra por palavra, traduzido):**

> *"Uploads are strongly consistent — you will never receive a 404 Not Found response or stale data for a read-after-write or read-after-metadata-update operation."*

E ainda:

> *"Strong global consistency also extends to deletion operations. If a deletion request succeeds, an immediate attempt to download the object or its metadata will result in a 404."*

E:

> *"Bucket listing is strongly consistent. Object listing is also strongly consistent."*

#### O que isso significa na prática

| Operação                    | Garantia                                                                  |
|-----------------------------|----------------------------------------------------------------------------|
| Upload novo objeto          | Imediatamente acessível em qualquer região onde Google opere               |
| Overwrite                   | Próxima leitura vê **a nova versão** (nunca a antiga "stale")              |
| Metadata update             | Próxima leitura vê o **novo metadata**                                     |
| Delete                      | Próximo GET retorna **404** imediato — sem janela de "ainda existe"        |
| List buckets após criar     | Bucket recém-criado **aparece** na lista                                   |
| List objects após upload    | Objeto recém-uploaded **aparece** na lista                                 |

#### Por que isso importa (contexto histórico)

- **AWS S3** era **eventually consistent** para overwrites e deletes **até dezembro de 2020**. Antes disso, você podia subir um arquivo e a próxima leitura ainda devolver a versão antiga por minutos. Isso causava bugs sutis em pipelines distribuídos.
- **Cloud Storage sempre foi strongly consistent globalmente**, desde o lançamento.

**Implicação prática:** você pode confiar em **read-after-write** sem implementar "retry com backoff esperando aparecer". Código fica muito mais simples.

**Pegadinha de prova:** se a questão sugerir que você precisa "esperar a replicação" depois de um upload em Cloud Storage — **está errada**. Não há janela de inconsistência.

### Bloco "Decision tree — escolhendo storage class e location (síntese)"

> Este bloco vem direto do sub-vídeo "decision tree" do M3 e fecha o aprendizado sobre Cloud Storage. É a **síntese decisória** dos blocos anteriores — memorizar essa árvore = ganhar quase qualquer DQ de planejamento de storage.

#### Eixo 1 — Storage class por frequência de leitura

**A regra é frequência de READ.** Escrita não entra na decisão — write é write, em qualquer classe. O custo varia com a frequência de leitura porque é nas reads que aparecem retrieval fees, mínimos de duração e penalidades.

```
        Vai LER o dado com que frequência?
                       |
   +-------------------+-------------------+----------+
   |                   |                   |          |
   < 1x / ano    < 1x / 90 dias    < 1x / 30 dias  Mais
        |              |                  |       frequente
        v              v                  v          v
    Archive        Coldline           Nearline    Standard
```

#### Eixo 2 — Location type por padrão de uso

```
                Quem consome o dado e de onde?
                              |
   +--------------------------+--------------------------+
   |                          |                          |
Consumers dentro do      Consumers em duas         Consumers fora da
GCP, agrupados na        regiões + quer            rede Google,
mesma região             alta availability         distribuídos
   |                     geo-redundante            geograficamente
   v                     |                         (OU quer geo-
Region                   v                         redundância máxima)
                      Dual-region                  |
                                                   v
                                                Multi-region
```

#### As 3 perguntas verbatim do vídeo (decoradas, podem cair em DQ)

1. **Region** — *"to help optimize latency and network bandwidth for data consumers, such as analytics pipelines, that are grouped in the same region."*
2. **Dual-region** — *"when you want similar performance advantages as regions, but also want the higher availability that comes with being geo-redundant."*
3. **Multi-region** — *"when you want to serve content to data consumers that are outside of the Google network and distributed across large geographic areas, or when you want the higher data availability that comes with being geo-redundant."*

#### Eixo 3 — E se o padrão de acesso for imprevisível?

Se você responde "**não sei**" à pergunta "com que frequência vai ler?", a decision tree do vídeo aponta para uma resposta diferente:

> *"If your data has a variety of access frequencies, or the access patterns for your data are unknown or unpredictable, you should consider Autoclass."*

Ou seja: **Autoclass não é uma quarta classe** — é uma **alternativa à decisão por classe**. Quando você habilita Autoclass, deixa de escolher Standard/Nearline/Coldline/Archive e deixa o serviço gerenciar.

#### Mapa decisório consolidado

```
                  Conheço o padrão de acesso?
                            |
            +---------------+----------------+
            | NÃO                            | SIM
            v                                v
        Autoclass                Escolho classe por frequência de READ
                                 (Standard / Nearline / Coldline / Archive)
                                            |
                                            v
                                 Escolho location por onde estão os consumers
                                 (Region / Dual-region / Multi-region)
```

#### Tabela "lendo a DQ" — pistas léxicas e o que disparam

| Pista no enunciado                                              | Resposta provável             |
|-----------------------------------------------------------------|-------------------------------|
| "Accessed less than once a year", "compliance archive"          | **Archive**                   |
| "Accessed once a quarter", "DR backup raramente lido"           | **Coldline**                  |
| "Accessed about once a month", "long-tail backup"               | **Nearline**                  |
| "Frequently accessed", "dashboards", "hot data", "in production" | **Standard**                  |
| "Analytics pipeline na mesma região", "co-locate with GCE/GKE"  | **Region**                    |
| "Resilient to region failure", "geo-redundancy"                 | **Dual-region**               |
| "Serve content globally", "users worldwide outside Google"      | **Multi-region**              |
| "Mixed access patterns", "unpredictable", "unknown"             | **Autoclass**                 |


A última linha imprime o JSON da regra confirmando aplicação. Lifecycle aplicado: deletar objetos com mais de 31 dias.

> **Truque:** o `cat > arquivo << 'EOF' ... EOF` cria arquivo direto no shell sem precisar abrir nano. Útil em todo lab que pede pra "criar um JSON".

#### Task 6 — Versioning + recovery

**Ativar versioning:**
```bash
gsutil versioning get gs://$BUCKET_NAME_1     # Suspended
gsutil versioning set on gs://$BUCKET_NAME_1
gsutil versioning get gs://$BUCKET_NAME_1     # Enabled
```

**Criar versões (sem nano, usando `sed`):**
```bash
ls -al setup.html                              # tamanho original (~59386 bytes)
sed -i '1,5d' setup.html                       # deleta 5 primeiras linhas
gcloud storage cp -v setup.html gs://$BUCKET_NAME_1
sed -i '1,5d' setup.html                       # mais 5 linhas
gcloud storage cp -v setup.html gs://$BUCKET_NAME_1
```

A flag `-v` (verbose) faz aparecer o **generation number** de cada versão criada.

**Listar todas as versões:**
```bash
gcloud storage ls -a gs://$BUCKET_NAME_1/setup.html
```

Saída esperada (3 linhas):
```
gs://qwiklabs-.../setup.html#<gen1>   ← original completa
gs://qwiklabs-.../setup.html#<gen2>   ← -5 linhas
gs://qwiklabs-.../setup.html#<gen3>   ← -10 linhas (live)
```

**Recuperar a versão original (primeira linha):**
```bash
export VERSION_NAME=gs://qwiklabs-.../setup.html#<gen1>
gcloud storage cp $VERSION_NAME recovered.txt
ls -al setup.html recovered.txt
```

`recovered.txt` deve ser **maior** que `setup.html` — a diferença é exatamente as linhas deletadas (no meu run: 59386 vs 58954 = 432 bytes = 10 linhas).

> **Pegadinha de prova reforçada:** Versioning cobra **cada versão como arquivo separado**. 3 versões = paga por 3x o storage.

#### Task 7 — Directory sync com `rsync`

```bash
mkdir firstlevel
mkdir ./firstlevel/secondlevel
cp setup.html firstlevel
cp setup.html firstlevel/secondlevel

gsutil rsync -r ./firstlevel gs://$BUCKET_NAME_1/firstlevel
gcloud storage ls -r gs://$BUCKET_NAME_1/firstlevel
```

A flag `-r` é recursive: sincroniza subdiretórios. A última linha lista o que subiu, confirmando que tanto `/firstlevel/setup.html` quanto `/firstlevel/secondlevel/setup.html` foram copiados.

#### Lições operacionais do lab (vão valer pra todos os labs futuros)

1. **`gsutil` mantém estado em `.boto`.** Esse arquivo precisa ser sintaticamente válido — se corromper, **todo** `gsutil` quebra. Recovery: `rm .boto && gsutil config -n`.
2. **Cloud Shell perde variáveis entre sessões.** Sempre que abrir nova aba/voltar do Editor, re-exporte com `export VAR=valor` e confirme com `echo $VAR`. O erro típico é `gs:///` na URL = variável vazia.
3. **`$VAR` ≠ `nome-literal`.** O `$` só prefixa variáveis. Se você digitar `gs://$qwiklabs-...`, o bash expande `$qwiklabs` (vazio) e o resto vira string literal.
4. **Ctrl+W do nano colide com Ctrl+W do navegador (fecha aba).** Alternativas: (a) clica em "Open in new window" no Cloud Shell, (b) usa `sed -i` no terminal sem abrir editor, (c) usa Cloud Shell Editor (VS Code Web).
5. **`sed -i '1,5d' arquivo`** deleta as primeiras 5 linhas in-place — substituto perfeito do nano pra labs que mandam "delete 5 lines".
6. **`cat > arquivo << 'EOF' ... EOF`** cria arquivo de conteúdo conhecido sem abrir editor. Ideal pra JSON do lab.
7. **CSEK pode estar bloqueado** em ambientes gerenciados via Organization Policy — sintoma é `PreconditionException: 412 ... encryption enforcement configuration`. Não é erro seu; é decisão arquitetural da org. Em produção, prefira CMEK (Cloud KMS).
8. **Qwiklabs Check my progress valida via API.** Se a action física não acontece (caso da Task 3 aqui), o check fica vermelho **mesmo seguindo o roteiro à risca**. Quando o ambiente difere do lab, **siga o que dá** e documente o que não deu — vale mais pra feira que o checkmark verde.

- Praticar a rotação completa (`gsutil rewrite -k`) e ver na prática o setup3.html ficar órfão.

## Cenário real

**Como eu uso (ou usaria) Cloud Storage no trabalho:**

- **Mídia do WhatsApp (Portaria / TaxIA):** bucket regional Standard em `southamerica-east1` (perto dos servidores e dos clientes BR). Lifecycle policy:
  - 0–30 dias: Standard (cliente ou operador ainda revisita).
  - 30–90 dias: Nearline.
  - 90–365 dias: Coldline.
  - >365 dias: Archive.
  - >7 anos: delete (se a regulação permitir).
- **Backups do Postgres:** bucket dual-region Coldline (`nam4`) para DR — quero que sobreviva a queda de região inteira.
- **Logs antigos exportados do Cloud Logging:** bucket regional Archive — barato, raramente lido, mas obrigatório por compliance.
- **Site estático (landing):** bucket multi-region Standard em `US` com `allUsers:objectViewer` na frente de Cloud CDN.

**Quando usaria Signed URL no agent WhatsApp:** quando o cliente final precisa fazer upload de um documento (foto de boleto, comprovante) **direto pro bucket** sem passar pelo nosso backend. Backend gera signed URL com expiração de 5 minutos + signed policy document limitando a `image/*` até 5 MB — cliente sobe direto pro Cloud Storage.

**Armadilha que cometeria na vida real:** criar bucket regional pra logs críticos. Se a região cair durante incidente, perco acesso justo aos logs que precisaria pra debugar. Para logs críticos, dual-region.

## Armadilhas de prova

1. **"Pasta" em Cloud Storage não existe.** São prefixos de chave. Pegadinhas que falam em "criar diretório" estão usando linguagem coloquial — não há diretório físico.
2. **Nome do bucket é globalmente único.** Não apenas no projeto, no GCP inteiro.
3. **Bucket não pode ser aninhado.** Não há sub-bucket. Use prefixos.
4. **Location type NÃO pode ser alterado.** Regional ↔ multi-region exige recriar bucket + migrar.
5. **Storage class pode ser alterada** — tanto a default do bucket quanto a de objetos individuais.
6. **11 nines de durability em TODAS as classes.** Não escolha Standard "porque é mais durável" — empatam. Escolha por availability/custo/padrão de acesso.
7. **Archive no GCP é em milissegundos.** Não confunda com Glacier Deep Archive da AWS (horas). Pegadinha clássica.
8. **Standard não tem retrieval cost.** Nearline/Coldline/Archive cobram retrieval — quanto mais fria, mais caro.
9. **Minimum storage duration:** 0 / 30 / 90 / 365 dias. Deletar antes paga como se tivesse ficado o período mínimo.
10. **`allUsers` = público na internet** (sem conta Google). `allAuthenticatedUsers` = qualquer usuário Google logado. NÃO confunda — `allUsers` é o cuidado máximo.
11. **ACL tem limite de 100 entradas** por bucket ou objeto. Se você precisa de mais granularidade que isso, está usando ACL errado.
12. **Uniform bucket-level access desabilita ACLs.** Em buckets novos, é o default recomendado.
13. **Signed URL não envolve conta Google.** Útil pra clientes externos. Service account assina; cliente usa a URL.
14. **Geo-redundância só com multi-region ou dual-region.** Regional não tem.
15. **`gcloud storage`** é a CLI moderna — substituiu `gsutil` para a maior parte das operações. Se a prova mostrar `gsutil`, ainda é válido (ainda existe), mas `gcloud storage` é o preferido.
16. **Objetos são imutáveis.** Editar = deletar + recriar. Todo o resto (Versioning, Soft Delete, Retention Lock) depende disso.
17. **CSEK ≠ CMEK.** CSEK = você guarda a chave fora do GCP (Google nunca persiste). CMEK = chave vive em Cloud KMS. Default = Google-managed.
18. **Versioning cobra como arquivos separados.** 10 versões de 100 MB = paga por 1 GB.
19. **Soft Delete está habilitado por DEFAULT** em buckets novos, com 7 dias. Não precisa ligar — precisa **não desligar** se quer proteção.
20. **Google recomenda Soft Delete em vez de Versioning** quando o objetivo é proteção contra delete. Versioning continua valendo pra histórico programático.
21. **Lifecycle Management roda em batches assíncronos.** Não conte com execução imediata.
22. **Atualizações de lifecycle config levam até 24h pra entrar em efeito.** Pode ainda executar a regra **antiga** nesse intervalo — fonte de bug sutil.
23. **Object Retention Lock locked = imutável de verdade.** Nem o owner consegue reduzir o período. É o único mecanismo que atende **FINRA / SEC / CFTC / WORM / proteção anti-ransomware**.
24. **Autoclass cobra por classificação.** Vale a pena com padrão de acesso imprevisível, não com padrão conhecido.
25. **Pub/Sub é o mecanismo de notificação de eventos** em bucket — não webhook genérico. Configure via `gcloud storage buckets notifications create`.
26. **Cloud Storage tem strong global consistency.** Não há janela de "ainda existe depois de delete" nem "ainda devolve versão antiga depois de overwrite". Diferença histórica vs S3 antigo.
27. **Bucket listing e object listing também são strongly consistent.** Criou bucket → aparece na lista. Subiu objeto → aparece no list.
28. **Transferência em massa — escolha por origem:**
    - Migrar de S3 / HTTPS / outro bucket GCS → **Storage Transfer Service**
    - On-prem com TB-PB sem rede → **Transfer Appliance** (HW físico, até 1 PB)
    - HDs / tapes / USB físicos → **Offline Media Import** (parceiro 3rd-party)
29. **`gcsfuse` (Cloud Storage FUSE) não é Filestore.** Monta bucket como filesystem, mas sem semântica POSIX completa. Pra filesystem compartilhado real → Filestore.
30. **Decision tree de classe é por frequência de READ, não de escrita.** Toda escrita custa o mesmo entre classes; o que varia é o custo de ler depois.
31. **Autoclass ignora a classe pedida no upload.** Mesmo que sua aplicação envie `--storage-class=ARCHIVE`, o objeto **começa em Standard** num bucket com Autoclass. A classe inicial pedida é descartada.
32. **Autoclass = SEM early deletion, SEM retrieval, SEM transition charges.** É isso que torna o feature seguro — as três cobranças que normalmente travam movimentação entre classes ficam isentas.
33. **Autoclass é alternativa à escolha de classe, não uma 5ª classe.** Habilitar Autoclass substitui a decisão "Standard ou Nearline ou Coldline ou Archive".
34. **"Mixed / unknown / unpredictable access pattern" → Autoclass.** Pista léxica oficial do vídeo. Não escolha uma classe específica nesse caso.
35. **Multi-region não é "para usuários globais" genericamente.** A pista exata é **"consumers fora da rede Google distribuídos geograficamente"** ou **"quer geo-redundância máxima"**. Se os consumers estão todos dentro do GCP em uma região, escolha **region** (não multi-region).
36. **Region é a recomendação para analytics pipelines.** O vídeo cita explicitamente: co-locar com GKE/GCE na mesma região otimiza latência e reduz custo de network egress.
